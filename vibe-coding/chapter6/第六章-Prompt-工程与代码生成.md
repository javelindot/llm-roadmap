## 6.3 进阶策略：Few-Shot

### Few-Shot 概念

Few-Shot 是通过提供示例来引导 AI 生成符合预期的代码。

```
┌─────────────────────────────────────────────────────────────┐
│                     Few-Shot 示例                            │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│   Prompt:                                                   │
│   ┌─────────────────────────────────────────────────────┐   │
│   │                                                     │   │
│   │  作为一个 TypeScript 开发者，请为以下 API 端点       │   │
│   │  生成统一的错误处理中间件                            │   │
│   │                                                     │   │
│   │  示例 1:                                            │   │
│   │  输入: POST /api/users (成功)                       │   │
│   │  输出: { success: true, data: user }               │   │
│   │                                                     │   │
│   │  示例 2:                                            │   │
│   │  输入: POST /api/users (验证失败)                   │   │
│   │  输出: { success: false, error: "Invalid input" } │   │
│   │                                                     │   │
│   │  现在请为 GET /api/posts 生成错误处理               │   │
│   │                                                     │   │
│   └─────────────────────────────────────────────────────┘   │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 适用场景

| 场景 | 示例效果 |
|------|----------|
| **代码风格** | 通过示例定义代码风格 |
| **API 格式** | 统一响应和错误格式 |
| **命名规范** | 一致的命名模式 |
| **注释风格** | 统一的注释格式 |

## 6.4 进阶策略：链式思维 (CoT)

### CoT 概念

Chain of Thought 引导 AI 先思考再生成，提高代码质量。

```markdown
# 使用 CoT 的 Prompt

请为以下需求设计一个算法:

## 需求
找出数组中和为目标值的两个数的下标

## 思考步骤（请先思考）
1. 这是一个什么问题？（提示：两数之和）
2. 最直观的解法是什么？（暴力枚举）
3. 暴力解法的时间复杂度是多少？（O(n²)）
4. 如何优化？（使用哈希表）
5. 优化后的时间复杂度是多少？（O(n)）

## 请基于以上分析
1. 写出优化后的代码
2. 说明时间复杂度和空间复杂度
3. 提供测试用例
```

### 输出示例

```markdown
## 分析

1. 问题类型：两数之和（Two Sum）
2. 暴力解法：双重循环 O(n²)
3. 优化思路：使用哈希表，空间换时间
4. 最终复杂度：O(n) 时间，O(n) 空间

## 代码实现

```typescript
function twoSum(nums: number[], target: number): number[] {
  const map = new Map<number, number>();

  for (let i = 0; i < nums.length; i++) {
    const complement = target - nums[i];
    if (map.has(complement)) {
      return [map.get(complement)!, i];
    }
    map.set(nums[i], i);
  }

  return [];
}
```

## 复杂度分析
- 时间复杂度: O(n)
- 空间复杂度: O(n)
```

## 6.5 进阶策略：自洽性 (Self-Consistency)

### 概念

让 AI 生成多个方案，然后选择最佳或综合最优。

```markdown
# 自洽性示例

为这个排序函数添加类型定义，要求：

1. 生成 3 种不同的类型定义方案
2. 分析每种方案的优缺点
3. 推荐最佳方案

## 当前代码
```typescript
function sort(arr) {
  return arr.sort((a, b) => a - b);
}
```
```

### AI 输出

```markdown
## 方案 1: 使用 any
```typescript
function sort(arr: any[]): any[]
```
❌ 丢失类型信息，不推荐

## 方案 2: 泛型 + 约束
```typescript
function sort<T extends number | string>(
  arr: T[]
): T[]
```
✅ 保留类型，⚠️ 限制了元素类型

## 方案 3: 数字专用的泛型
```typescript
function sort<T extends number>(
  arr: readonly T[]
): T[]
```
✅ 最佳：支持只读数组，类型安全

## 推荐
方案 3，理由：[详细说明]
```

```

│                  /   相关代码片段    \                     │
│                 /      底层          \                    │
│                /───────────────────────\                   │
│               /     具体需求细节         \                 │
│              /        基础              \                 │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 上下文注入方法

#### 1. @ 文件引用

```
# Cursor
@auth.ts 请参考这个文件的实现风格

# Claude Code
@file:auth.ts 请参考这个文件的实现风格
```

#### 2. @ 文件夹引用

```
# Cursor
@components 整个组件目录需要遵循的模式

# Claude Code
@folder:src/components 整个组件目录
```

#### 3. 代码块引用

```
# 引用特定代码
```typescript
// 这是现有的用户类型
interface User {
  id: string;
  email: string;
}
// 请基于这个类型扩展其他功能
```
```

### 上下文优化原则

| 原则 | 说明 |
|------|------|
| **相关优先** | 只提供与任务相关的上下文 |
| **简洁有效** | 避免冗余信息 |
| **结构清晰** | 使用分隔符组织 |
| **分层提供** | 从整体到具体 |

## 6.10 避免 AI 幻觉

### 验证策略

```
┌─────────────────────────────────────────────────────────────┐
│                    AI 生成代码验证                           │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│   1. 语法检查                                               │
│      └─> 运行 ESLint / TypeScript 编译器                    │
│                                                             │
│   2. 类型检查                                               │
│      └─> TypeScript 类型是否正确                            │
│                                                             │
│   3. 逻辑验证                                               │
│      └─> 手动审查关键逻辑                                   │
│                                                             │
│   4. 依赖检查                                               │
│      └─> 确认导入的模块/包是否存在                          │
│                                                             │
│   5. 测试验证                                               │
│      └─> 运行生成的测试用例                                 │
│                                                             │
│   6. 集成测试                                               │
│      └─> 实际运行代码验证功能                               │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 边界条件检查

| 检查项 | 说明 |
|--------|------|
| **空值处理** | null/undefined 是否处理 |
| **空数组** | [] 是否正确处理 |
| **边界值** | 0、负数、最大值 |
| **类型错误** | 类型转换是否安全 |
| **异步问题** | Promise/async 处理 |
| **并发问题** | 竞态条件 |

## 6.11 小结

### 关键要点

1. **Prompt 四要素**：角色、任务、要求、上下文
2. **Few-Shot**：通过示例引导输出风格
3. **CoT**：先思考再生成，提高质量
4. **上下文注入**：相关、简洁、结构化
5. **工具专属**：Cursor 和 Claude 有不同的最佳实践

### 最佳实践

1. **清晰具体**：明确的需求获得更好的结果
2. **循序渐进**：复杂任务分步骤执行
3. **验证始终**：AI 生成需要人工审查
4. **迭代优化**：通过反馈改进输出
