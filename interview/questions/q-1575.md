# 设计一个支持日均百万级查询的RAG系统

## 答案

日均 100 万查询的平均 QPS 只有约 `1,000,000 / 86,400 ≈ 11.6`，真正的设计压力来自峰值、长尾延迟、LLM token 吞吐、知识更新、多租户隔离和故障降级。应先确定峰值系数与 SLO，再做容量规划。

### 1. 需求假设

示例：

```text
日请求：1,000,000
峰值系数：10~20
峰值：约 120~230 QPS
可用性：99.9%+
端到端 P95：按业务定义，例如 2~5 秒
知识新鲜度：分钟级或小时级
```

还要明确输入/输出 token、Top-K、模型路由比例、流式响应和多轮请求。若峰值 150 QPS、平均输出 250 token，仅 Decode 就约需要 `37,500 output tokens/s`，实际要按模型和硬件压测。

### 2. 在线架构

```text
API Gateway
-> Auth/Tenant/Rate Limit
-> Query Router
-> Query Rewrite/Embedding
-> Hybrid Retrieval
-> Rerank
-> Context Builder
-> LLM Gateway
-> Citation/Verifier
-> Streaming Response
```

**接入层**

- 鉴权、租户、配额和幂等；
- request/trace ID；
- 限流、WAF、输入大小和安全检查；
- 按用户/商户做稳定灰度。

**检索层**

- BM25/OpenSearch 处理精确词；
- 向量数据库处理语义召回；
- metadata/ACL 前置过滤；
- 索引分片和多副本；
- 多路检索并行，RRF 融合；
- 热 query/embedding 缓存。

**Rerank/上下文**

- GPU 或高性能 CPU batch rerank；
- 动态候选数；
- 去重、版本冲突和 token budget；
- 超时降级为混合召回排序。

**LLM 层**

- 统一模型网关与路由；
- 小模型处理简单请求，强模型处理复杂任务；
- vLLM/SGLang continuous batching；
- Prefix Cache、KV Cache 管理和流式输出；
- 每个模型池独立并发、deadline 和 fallback。

### 3. 离线索引

```text
Source/CDC
-> Parse/OCR
-> Clean/Deduplicate
-> Chunk/Metadata/ACL
-> Embedding
-> Staging Index
-> Validation
-> Atomic Alias Switch
```

使用消息队列削峰、内容指纹幂等、增量更新、删除 tombstone 和蓝绿重建。文档、parser、chunker、embedding 和索引都要版本化。

### 4. 缓存

- query rewrite cache；
- query embedding cache；
- 检索/rerank cache；
- 公共文档片段 cache；
- Prompt/Prefix cache；
- 最终答案缓存仅适用于稳定低风险问题。

缓存键必须包含租户/权限、语言、知识版本、模型和 Prompt 版本，避免越权和陈旧答案。

### 5. 高可用与降级

- 跨可用区副本；
- 检索、rerank、模型服务独立扩缩容；
- 超时和取消传播；
- circuit breaker 与指数退避；
- 队列背压和优先级；
- reranker 故障降级到 RRF；
- 强模型拥塞时路由小模型或只返回检索结果；
- 证据不足时拒答，不让模型凭空补全。

### 6. 可观测性

逐阶段记录：

```text
rewrite/retrieval/rerank/LLM/tool latency
candidate count
context/output tokens
cache hit
index/model/prompt version
fallback reason
```

业务指标包括 Recall@K、faithfulness、引用支持率、无答案识别、任务成功率、安全违规和单次成功成本。

### 7. 容量与压测

- 回放真实长度、语言和 Top-K 分布；
- 分别压测检索、rerank、Prefill 和 Decode；
- 使用峰值加冗余规划，不按日均 QPS；
- 测突发、热点租户、索引更新和下游故障；
- 关注 P99、队列年龄、GPU KV 利用率和 goodput；
- 每次模型/Prompt/索引升级重新做容量测试。

### 8. 跨境支付场景

按商户、国家和角色做检索前 ACL；监管文档保留生效时间；订单、余额、汇率走实时工具。高峰时可降级解释链路，但不能跳过鉴权、账务校验和合规过滤。

### 9. 面试总结

百万日请求不难在平均 QPS，难在 10~20 倍峰值、token 吞吐和质量 SLO。设计重点是分层扩缩容、混合检索、批量推理、权限感知缓存、增量索引、可观测性与可控降级。
