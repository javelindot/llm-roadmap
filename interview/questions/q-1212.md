# 讲一下之前做的sft和dpo

## 答案

SFT 用标准答案教模型“应该如何完成任务”，DPO 用同一问题下的 chosen/rejected 教模型“哪种回答更好”。合理顺序是先建立稳定 SFT，再用 DPO 做偏好对齐。

### 1. SFT 数据

跨境支付场景覆盖：

- 产品、KYC/KYB 和规则问答；
- Query Rewrite；
- 意图和实体抽取；
- 工具选择及参数；
- 订单状态解释；
- 无权限、缺信息、证据冲突和转人工；
- 提示注入与高风险资金动作。

样本来自脱敏工单、专家示范、SOP、工具 trace 和线上 bad case。清理过期规则、错误客服回复、重复模板和实体泄漏。

### 2. SFT 训练

- 使用部署一致的 Chat Template；
- 只对 Assistant 输出计算 loss；
- 工具调用、结果和最终回答角色正确；
- 按时间和会话切分；
- 小数据先 LoRA；
- BF16、packing、checkpointing 和梯度累积；
- 先用小样本 overfit 检查模板和 mask。

训练不能只看 loss，要看格式、工具、引用、安全和分桶业务指标。

### 3. DPO 数据

每条数据是：

```text
(prompt, chosen, rejected)
```

高质量偏好差异应对应：

- chosen 引用正确，rejected 无依据承诺；
- chosen 正确追问订单号，rejected 猜测状态；
- chosen 使用允许工具，rejected 越权；
- chosen 条件完整，rejected 忽略国家或币种；
- chosen 简洁可执行，rejected 冗长但无结论。

chosen/rejected 的长度、格式和来源要平衡，避免模型只学到“长、礼貌、分点多”。

### 4. DPO 训练

策略模型从 SFT checkpoint 初始化，reference 固定。重点调：

- `β`；
- 学习率；
- 有效 batch；
- epoch；
- 最大长度；
- SFT loss 是否混合。

监控 DPO loss、reward margin、pairwise accuracy、KL、输出长度和通用能力。Loss 下降不代表事实性提升。

### 5. 典型问题

**SFT**

- Chat Template 或 label mask 错误；
- 合成数据比例过高；
- 过拟合模板；
- 领域能力提升但通用能力下降；
- 把动态知识写入参数。

**DPO**

- 偏好标签噪声；
- 长度和风格偏置；
- chosen/rejected 差异太弱；
- 过度拒答或转人工；
- 相对 margin 提升但两个答案绝对质量都下降；
- 偏好集与线上 policy 分布不一致。

### 6. 评测和上线

- SFT 前后任务准确率、Tool/Argument Accuracy；
- DPO 人工偏好胜率和 length-controlled win rate；
- faithfulness、引用和严重错误率；
- 拒答、追问和转人工准确率；
- 不同国家、语言、产品和风险分桶；
- P95、token、输出长度和成本；
- 通用能力与安全回归。

上线走 shadow、灰度和 A/B，保留 SFT checkpoint、DPO checkpoint、Prompt 和 Adapter 的独立回滚。

### 7. 使用边界

SFT/DPO 适合学习稳定行为和表达，不适合承担实时事实、权限和资金状态机。付款、退款、冻结和解冻必须由确定性服务执行。
