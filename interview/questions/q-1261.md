# 召回、排序哪个阶段最可能出问题？

## 答案

召回和排序都可能出问题，但影响链路不同：

- 正确候选没有进入召回集，后续排序无法补救；
- 正确候选已召回却排不靠前，才是排序问题；
- 正确结果排在前面但答案仍错，则是上下文或生成问题。

因此不能凭感觉判断“哪一阶段最容易”，要通过候选集和标注证据做错误归因。

### 1. 召回常见问题

- Query 口语化、指代未恢复；
- BM25 漏同义词，Dense 漏精确错误码；
- chunk 切坏条件和结论；
- embedding 领域或语言不匹配；
- Top-K 太小；
- 权限、国家、币种或时间 filter 写错；
- 索引过期、更新未完成；
- ANN 参数过激导致近似召回损失。

判断方式：看黄金相关文档是否出现在初召回 Top-K，计算 Recall@K、HitRate 和各通道独占贡献。

### 2. 排序常见问题

- Reranker 训练负样本太简单；
- Query 与 chunk 被截断；
- 分数未按不同通道校准；
- 否定、版本、权威性和时效没有特征；
- 位置偏差或点击偏差；
- Top-N 候选过多导致延迟，过少导致排序空间不足；
- 相关但不完整的短 chunk 压过完整证据。

判断方式：在“相关文档已召回”的 Query 子集上计算 NDCG、MRR、Precision@K 和 oracle rerank 上限。

### 3. 诊断矩阵

| 正确证据召回 | 最终 Top-K | 结论 |
|---|---|---|
| 否 | 否 | 召回/索引/查询问题 |
| 是 | 否 | 排序或过滤问题 |
| 是 | 是 | 上下文构造/生成问题 |
| 是但版本错误 | 是 | 数据治理和元数据问题 |

### 4. 修复顺序

先保证召回覆盖，再优化精排：

1. 建黄金 Query-Document 标注；
2. 分析 BM25、Dense、FAQ 各路召回；
3. 修 Query、切块、索引和过滤；
4. 从召回候选中挖 hard negative 训练 reranker；
5. 优化去重、冲突和上下文预算；
6. 端到端验证生成。

召回率也不能无限提高。把大量噪声召回后再让 reranker“收拾残局”，会增加成本并可能超过模型能力。

### 5. 跨境支付例子

用户问 `E10027` 错误码：

- 候选集没有该错误码文档，是召回问题；
- 文档已召回但被相似的 `E10072` 排在前面，是排序问题；
- 正确文档排第一但模型解释成退款而非付款失败，是生成问题；
- 召回了旧版本处理方式，是知识版本问题。

生产监控应保留各阶段 trace，才能对症修复，而不是只看最终答案正确率。
