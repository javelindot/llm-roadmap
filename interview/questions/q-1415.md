# 性能优化指标如何获取及实现

## 答案

性能优化首先要建立可重复的测量体系。指标获取通常依靠 metrics、logs、traces 和 profiler，优化流程是“定义目标、建立基线、定位瓶颈、单点修改、回归验证”。

### 1. 定义指标

服务端至少包括：

- QPS/吞吐；
- P50/P95/P99；
- 错误率和超时率；
- CPU、内存、GPU、显存；
- 队列等待；
- DB/缓存/外部 API 延迟；
- 单请求成本。

LLM 服务增加：

- TTFT；
- TPOT；
- input/output tokens；
- tokens/s；
- batch size；
- KV Cache 使用率；
- Prefill/Decode 耗时；
- GPU 利用率和 MFU。

### 2. 如何采集

**Metrics**

在入口、中间步骤和依赖调用处埋点，用 counter、gauge、histogram 记录，常见通过 OpenTelemetry/Prometheus 汇总，再用 Grafana 展示。

**Tracing**

为请求生成 trace id，拆分：

```text
gateway
-> auth
-> retrieve
-> rerank
-> tool call
-> model queue
-> prefill
-> decode
```

这样能判断慢在检索、外部工具还是生成。

**Profiler**

- CPU flame graph；
- PyTorch Profiler；
- Nsight Systems/Compute；
- 数据库 `EXPLAIN` 和慢查询；
- heap/memory profiler；
- 网络和磁盘监控。

### 3. 压测设计

- 固定硬件、版本和数据集；
- 包含真实请求长度分布；
- 分 warm-up 和 steady state；
- 分别测试单请求、目标并发和过载；
- 记录吞吐-延迟曲线；
- 测故障、超时和降级；
- 避免只报最好一次结果。

### 4. 实现优化

定位后再选择：

- 算法和数据结构；
- 缓存；
- 批处理和异步；
- 索引和 SQL；
- 减少序列化与网络往返；
- 模型量化、continuous batching；
- prefix cache、FlashAttention；
- 水平扩容和请求调度。

每次只改一个主要变量，并比较置信区间和资源成本。

### 5. 跨境支付注意

支付链路应记录业务阶段耗时：

```text
风控
-> 路由
-> 通道请求
-> 回调
-> 账务
```

但日志和 trace 不能写完整卡号、证件号或敏感密钥。性能优化也不能通过跳过风控、幂等或账务校验实现。

### 6. 面试总结

性能指标通过埋点、分布式追踪、资源监控和 profiler 获取；实现上先有真实基线，再定位最大瓶颈，做单变量优化和压测回归。最终同时看效果、尾延迟、吞吐、成本和稳定性。
