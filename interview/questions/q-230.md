# 为什么选取top10

## 答案

**一句话**：选 top10 不是固定真理，而是在召回覆盖率、上下文成本、重排效果和生成质量之间做出的经验平衡，最终要靠评测数据验证。

### 1. topK 的作用

RAG 检索时 topK 表示从向量库或检索系统里取前 K 个候选片段。K 太小可能漏掉答案，K 太大又会引入噪音，增加 token 成本，甚至干扰模型。

所以 top10 的含义通常是：先保证召回覆盖，再通过 rerank 或过滤选出最终上下文。

### 2. 为什么常用 top10

常见原因有：

- top3/top5 可能召回不足，尤其用户问题比较模糊时。
- top20/top50 容易噪音太多，重排和模型输入成本变高。
- top10 通常能覆盖多个相近候选，给 rerank 留空间。
- 对客服 FAQ、制度问答、错误码查询这类场景，top10 是比较常见的起点。

但我会强调：这是初始值，不是最终值。

### 3. 怎么确定最终 topK

应该用评测集调参：

- 比较 top5、top10、top20 的召回率。
- 看最终答案准确率和引用命中率。
- 看平均 token 成本和响应延迟。
- 看噪音引入后是否导致模型答偏。
- 区分不同场景设置不同 K。

比如错误码查询可能 top5 就够，复杂制度问答可能需要 top15，再通过 rerank 取前 3-5 段给模型。

### 4. 结合跨境支付项目怎么说（面试口径）

> 在跨境支付知识库里，我会把 top10 当成初始候选数，而不是最终塞给模型的数量。比如用户问“巴西付款为什么失败”，top10 可以覆盖 PIX 通道、国家规则、错误码说明、风控限制、到账时效等多个方向。然后再用 rerank 根据当前国家、币种、通道和错误码排序。
>
> 最终给模型的可能只有前 3 到 5 个高质量片段。这样既不容易漏掉答案，也不会把太多无关规则塞进上下文。
>
> 如果是精确错误码，比如 `PAYOUT_LIMIT_EXCEEDED`，可能 top5 就够；如果是“为什么不到账”这种模糊问题，top10 或 top15 更稳。最终我会用真实工单评测集看命中率和成本，再确定参数。

**钩子**：可以引到“top10 是候选集、最终上下文还要 rerank、不同业务场景动态 topK”。
