# 大模型的工作原理

## 答案

**一句话**：大模型的本质是**自回归 Transformer 在做下一个 token 的概率预测**——所有"理解、推理、对话"都是这个动作的涌现表现。

### 1. 三句话讲透
1. **训练目标**：给定上文，预测下一个 token 的概率分布
2. **架构**：N 层 Transformer Decoder（GPT 系）/ Encoder-Decoder（T5 系）
3. **能力来源**：参数规模 + 数据规模 + 训练算力的暴力 scaling

### 2. 推理流程（一次生成）
```
输入文本 → Tokenizer 切成 token ID → Embedding 查表 → 加位置编码
   → 过 N 层 Transformer（self-attention + MLP）
   → LM head（线性投影到词表）→ softmax → 下一 token 概率分布
   → 采样（greedy/top-p/top-k） → 拼回输入继续 → 重复直到 EOS
```

### 3. 训练三段论（主流路线）
- **预训练（Pretrain）**：海量无监督文本，next-token prediction
- **监督微调（SFT）**：高质量指令数据，学会"听话"
- **对齐（RLHF / DPO）**：偏好数据，学会"说人话"

### 4. 核心组件
| 组件 | 作用 |
|------|------|
| Tokenizer（BPE / SentencePiece） | 文本 → 离散 ID |
| Embedding | ID → 向量 |
| 位置编码（Sinusoidal / RoPE / ALiBi） | 注入位置信息 |
| Self-Attention | 上下文关系建模 |
| FFN / MoE | 知识存储 |
| Norm（Pre/Post LN, RMSNorm） | 训练稳定 |
| LM Head | 向量 → 词表概率 |

### 5. 关键性质
- **自回归**：一次生成一个 token，前文影响后文
- **因果掩码**：训练时一次性算完，掩码保证不偷看未来
- **KV Cache**：推理时缓存历史 K/V，省重复计算
- **涌现**：规模够大才出现的能力（CoT、ICL、Instruction Following）

### 6. 推理时的小白点
- 一次推理 = **prefill（处理输入）** + **decode（逐 token 生成）**
- prefill 算力密集（并行），decode 内存带宽密集（串行）
- 这就是为什么 vLLM 等推理框架要做 **continuous batching、page attention**

### 7. 一句话总结
**LLM = 下一词预测器 + Transformer + 三阶段训练**——简单 setup 通过 scale 涌现出复杂能力。

## 结合项目谈（口语版 · 约 40 秒）

> 我自己的理解是，大模型再花哨它本质就一件事——**给定上文，猜下一个 token 是什么**。所有的"理解""推理""对话"都是这件事被 scale 出来的副产品。
>
> 这个认知对我们跨境支付落地特别重要。举两个例子：
>
> 第一是**客服回复生成**——其实就是给 LLM 一个"客户问 + 历史对话"作为上文，让它续写"客服回答"。我们要做的是把上文设计好（system prompt + few-shot + 工单上下文），让它续写的概率分布偏向"我们品牌的语气"。这就是为什么 SFT 这么有效——直接把"客户问 X → 客服答 Y"的样本喂进去，模型学会的就是"看到 X 类问题，下一 token 应该往 Y 偏"。
>
> 第二是**风控的"为什么"**——LLM 不是真的"理解"洗钱，它是预训练里见过海量金融文本，在那个分布上学到了"出现 A、B、C 几个特征时，下文常常是'可疑'"。所以我们做风控时不指望它真懂业务规则，而是用 SFT 把业务"模式"灌进去，让它的下一 token 概率分布对齐我们的规则。
>
> 想清楚"它本质是下一词预测器"，prompt 工程和数据设计就不会乱抓药——**核心永远是：怎么把上文造好让它续写出我想要的**。

**🪝 顺嘴带出去的钩子**：
- "为什么 prefill 和 decode 对推理性能影响这么大可以聊聊。"
- "涌现能力在我们业务里有没有体现，有几个有趣的观察。"
- "KV Cache 在长上下文推理里的优化我们做过一些工作。"
