# 如何处理特别长的上下文输入？

## 答案

处理特别长的输入，第一原则不是直接换百万上下文模型，而是先判断哪些信息必须进入当前推理。长窗口、RAG、摘要、结构化查询和状态管理应组合使用。

### 1. 先做输入治理

- 去重模板、签名、重复历史；
- 按权限和租户过滤；
- 把图片、表格和附件转成结构化元素；
- 保留标题、页码、时间和来源；
- 删除与当前问题无关的日志和工具返回；
- 对订单号、金额等关键字段禁止摘要改写。

### 2. 检索而非全塞

```text
Question
-> Query Rewrite
-> Metadata Filter
-> Hybrid Retrieval
-> Rerank
-> Context Packing
```

对于交易数据，应按 merchant/order/time 查询数据库；文档知识再走 RAG。长窗口不能代替索引和权限。

### 3. 分层摘要

- Map-Reduce：分段摘要后再汇总；
- Hierarchical Summary：章节 -> 文档 -> 集合；
- Rolling Summary：多轮对话保留最近消息和历史摘要；
- Entity/Fact Memory：把人物、订单、决策和待办结构化保存；
- Extract-then-Generate：先抽关键证据，再生成。

摘要必须保留引用，金额、币种、日期和交易状态不应只保留自然语言概括。

### 4. 长上下文模型

使用原生支持目标长度、且经过对应长度训练的模型。还要验证：

- Needle-in-a-Haystack；
- Lost-in-the-Middle；
- 多跳与跨段引用；
- 不同位置、长度下的准确率；
- TTFT、KV Cache 和成本。

只修改 RoPE 配置不等于获得可靠的长上下文能力。

### 5. 推理工程

- FlashAttention；
- chunked prefill；
- prefix cache；
- PagedAttention；
- KV Cache 量化/压缩；
- Prefill/Decode 分离；
- 超长请求单独队列和限流。

### 6. 上下文预算

可以按优先级分配：

```text
System/Safety
> 当前用户问题
> 实时工具事实
> 直接证据
> 最近对话
> 历史摘要
> 背景材料
```

超预算时优先丢弃低价值重复信息，而不是截掉结尾或随机裁剪。

### 7. 跨境支付

一个 AML 案例可能有多年交易和材料。应先用规则/SQL 聚合异常交易，再检索对应证据，模型只读取聚合、关键样本和引用。原始流水保留在受控系统中，不全部进入 Prompt。

**面试总结**：长上下文问题本质是信息选择、压缩、检索和服务成本问题，窗口长度只是其中一个变量。
