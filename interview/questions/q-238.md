# 技术选型的思路

## 答案

**一句话**：技术选型不是追新，而是围绕业务目标、团队能力、稳定性、成本、可扩展性和合规要求做取舍。

### 1. 选型看哪些维度

我一般会从这些方面判断：

- 业务目标：解决什么问题，是降本、提效、风控还是合规。
- 性能要求：QPS、延迟、吞吐、并发。
- 稳定性要求：是否在核心链路，能不能降级。
- 数据安全：是否涉及敏感数据，能否出域。
- 团队熟悉度：团队能不能维护。
- 生态成熟度：社区、文档、监控、案例是否完善。
- 成本：机器、API、存储、运维、人力成本。
- 可扩展性：后续能否支持更多模型、工具和业务线。

### 2. AI 项目里的选型

AI 项目常见要选：

- 模型：闭源 API 还是开源私有化。
- RAG 框架：LangChain、LlamaIndex、Spring AI 或自研。
- 编排：简单链路、Agent loop、LangGraph。
- 向量库：ES、Milvus、Chroma、pgvector 等。
- 部署：云 API、vLLM、本地推理、混合模式。
- 评估：自研评测脚本、LangSmith、OpenTelemetry 等。

选型时要结合业务阶段。PoC 阶段可以快，生产阶段要稳。

### 3. 面试总结

我会把技术选型分成三步：先明确业务约束，再做候选方案对比，最后用小规模验证数据说话。尤其 AI 项目不能只看 demo 效果，要看线上准确率、幻觉率、成本和运维复杂度。

### 4. 结合跨境支付项目怎么说（面试口径）

> 在跨境支付项目里，我做技术选型会先区分主链路和旁路能力。收付款、账务、清结算这类资金主链路必须稳定、可审计、强一致，不能为了追新把它做得不可控；智能客服、风控解释、知识库问答这类可以作为旁路能力，用大模型提升效率，但要能降级。
>
> 比如模型选型上，如果是客服 FAQ，可以用成本较低、中文效果好的模型加 RAG；如果是合规材料解析，可能要考虑私有化或数据脱敏；如果是复杂风控解释，就需要更强的推理模型，但不能让模型直接给放行结论。
>
> 向量库选型也一样。PoC 阶段可以用 Chroma 快速验证，生产如果需要权限过滤、关键词混合检索和大规模数据，就可能选择 ES 或 Milvus。最终我会用真实工单和知识库评测集比较准确率、延迟和成本。
