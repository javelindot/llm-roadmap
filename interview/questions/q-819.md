# 如果要用 LoRA 做电商推荐场景的微调，你会怎么设计数据和标签？

## 答案

先明确 LoRA 微调的目标：让 LLM 做商品理解、候选重排、解释生成还是直接生成推荐。不同任务不能共用一套标签。若只做高 QPS CTR 排序，传统推荐模型通常比在线 LLM 更合适。

### 1. 数据构造

输入可包含：

- 用户近期和长期行为，按时间排序并截断；
- 当前 query、场景、设备和预算；
- 候选商品标题、类目、属性、价格与库存；
- 曝光、点击、加购、购买、退货等反馈。

严格按时间切分，防止未来行为泄漏；用户 ID、联系方式等先脱敏。只从被曝光商品中构造未点击负例，不能把用户从未见过的商品都视为负例。

### 2. 标签与任务

- 商品理解：类目、属性、query 改写等监督标签；
- pointwise 重排：点击/购买标签，需处理位置与曝光偏差；
- pairwise 偏好：同一曝光集合中的 chosen/rejected；
- listwise：给候选列表学习排序或生成最终列表；
- 解释生成：必须由真实商品证据支持，不把“文案好看”当成购买标签。

购买信号强但稀疏，点击多但噪声大，可做多任务或按价值加权；退货、投诉应作为负向反馈，而不是只优化成交。

### 3. LoRA 训练

使用稳定模板和结构化输出，先从 Q/V 或 Q/K/V/O 注入，复杂任务再覆盖 FFN。扫描 rank、学习率和数据规模，并与 prompt-only、小模型和全参基线比较。训练 loss 只 mask assistant 输出。

### 4. 评估

离线看 Recall/NDCG、pairwise accuracy、格式合法率和长尾切片；线上看 CTR、CVR、GMV、退货、多样性、延迟和成本。资格、库存和广告规则由确定性系统约束。

跨境支付可类比产品推荐：模型生成候选解释，但费率、可用国家和用户资格必须实时校验。

### 面试总结

LoRA 推荐微调要先定义任务，再按时间构造曝光一致的样本和分层反馈，避免把未曝光当负例。LLM 更适合理解、重排和解释，高 QPS 主排序仍需比较轻量模型。
