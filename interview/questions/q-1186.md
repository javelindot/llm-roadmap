# 为什么GRU行为序列建模召回比ItemCF效果更差?

## 答案

GRU 序列召回比 ItemCF 差，不一定说明序列模型能力弱，通常是数据、目标、候选生成或评测口径没有匹配。ItemCF 对“短期共现强、用户序列短、热门 item 稳定”的业务是非常强的基线。

### 1. ItemCF 为什么可能更强

ItemCF 直接统计：

> 与当前 item 共同被用户点击、购买或使用的其他 item。

它具有以下优势：

- 强记忆高频共现，不需要模型从参数中重新学习；
- 对短序列和稀疏用户也能工作；
- 新行为写入共现表后可较快生效；
- 候选解释和调试简单；
- 在“看 A 后常看 B”的局部转移上非常有效。

如果线上主要依赖最近一个或几个行为，复杂 GRU 未必能超过经过时间衰减、热门惩罚和行为加权的 ItemCF。

### 2. 数据与标签问题

GRU 常见问题包括：

- 用户序列太短，大量样本只有 1 到 3 个有效行为；
- 点击、曝光、收藏、购买没有区分权重；
- 序列包含误触、重复曝光和跨场景噪声；
- session 切分不合理，把相隔很久的兴趣强行连接；
- 用未来行为、最终画像或错误时间窗造成泄漏；
- next-item 训练目标与线上 Recall@K 或长期目标不一致；
- 负样本过于随机，模型没有学会区分同类困难候选；
- 热门 item 作为 batch 内负例过多，产生 false negative。

先检查按序列长度、活跃度、时间间隔和新老用户分桶的效果，通常能判断模型在哪类数据上退化。

### 3. 模型表达问题

单个 GRU 最终 hidden state 可能形成信息瓶颈：

- 长期和短期兴趣混在一个向量；
- 多兴趣用户被平均成单一表示；
- 很早的重要行为经过多步后被遗忘；
- 行为类型、时间间隔和上下文没有显式建模；
- hidden size 太小或正则过强造成欠拟合。

可尝试 Attention/Transformer、时间间隔特征、多行为 embedding、多个兴趣向量或“长期塔 + session 塔”。但应先确认收益瓶颈确实来自表达能力。

### 4. 训练与在线检索不一致

双塔式 GRU 召回还可能存在：

- 训练用全量 softmax，线上使用采样 loss；
- 训练相似度与 ANN 索引距离不一致；
- item 和 user 向量的归一化方式不同；
- 模型、特征与索引版本不一致；
- item embedding 更新慢，新 item 无向量；
- ANN 参数过于激进，理论 Top-K 在线没有召回；
- 线上过滤后候选不足，但离线评测没有模拟过滤。

应分别测“精确向量检索 Recall”和“ANN 实际 Recall”，避免把索引损失误判为 GRU 模型差。

### 5. 公平对比与改进

保持相同候选预算、过滤规则、时间切分和标签后，依次做：

1. ItemCF、Popularity 和 GRU 单路离线分桶；
2. 比较独占命中与候选重合；
3. 使用 ItemCF Top 结果作为 GRU hard negative；
4. 加入时间衰减、行为类型和 session 特征；
5. 调整序列长度、hidden size、采样和温度；
6. 检查向量范数、坍缩和 ANN 召回损失；
7. 在线做增量 A/B，而不是只看全局 Recall。

实践中常保留两路并行：ItemCF 提供稳定共现，GRU 补充顺序和个性化独占候选。多路召回的目标不是让单一模型赢过所有基线，而是提供可量化的增量。

### 6. 跨境支付中的迁移

商户产品推荐中，ItemCF 可以统计“开通全球收款账户后常使用哪些 FX 或付款产品”；GRU 则建模商户按时间发生的“收款、换汇、付款、卡消费”序列。

若商户行为低频、产品数量少、流程高度固定，ItemCF 很可能更稳。GRU 更适合行为较丰富、顺序和时间间隔确实影响下一产品需求的商户。无论哪种召回，候选都必须经过 KYB、牌照、额度和风险准入。
