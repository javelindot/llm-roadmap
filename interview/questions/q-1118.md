# 如何选择合适的大模型？

## 答案

模型选型是业务质量、延迟、吞吐、成本、安全和团队维护能力的多目标决策，不应只按参数量或公开榜单排名。

> 明确任务与硬约束 → 建立业务黄金集 → 候选模型初筛 → 效果/性能/成本实测 → 灰度上线 → 分层路由。

### 1. 先按任务拆分能力

不同任务不必共用同一个模型：

- 意图识别、Query 改写：小模型，重点看格式和低延迟；
- RAG 回答：中等模型，重点看忠实性、引用和多语言；
- 复杂风控调查：较强推理模型，但只生成分析建议；
- KYC/KYB 材料解析：视觉语言模型或 OCR 加抽取模型；
- embedding：单独评估检索模型；
- rerank：使用 cross-encoder，不由生成模型兼任；
- 实时风控打分：专用分类/序列模型，不用 LLM 替代。

### 2. 明确硬性约束

候选模型必须满足：

- 数据是否允许发送到外部 API；
- 是否支持目标语言和上下文长度；
- 是否提供结构化输出、工具调用和流式响应；
- 可用部署资源、显存、并发和 SLA；
- 模型许可和商用限制；
- 供应商可用性、版本稳定性和退出方案；
- 敏感数据、本地化和审计要求。

不满足安全、许可或延迟约束的模型，即使效果更高也不进入下一轮。

### 3. 使用真实业务黄金集

公开 benchmark 只用于初筛。最终评测使用脱敏真实问题，按国家、币种、语言、产品、难度和风险分桶：

- 规则问答和多轮客服；
- 工具选择与参数；
- 长文档与多证据；
- 无答案、拒答和转人工；
- 提示注入、越权和敏感信息；
- 结构化输出稳定性。

指标包括答案正确性、faithfulness、工具成功率、格式通过率、安全违规率和稳定性。对非确定性模型同一任务重复运行，不能只看一次最好结果。

### 4. 性能和成本实测

在目标硬件与真实输入长度下压测：

- TTFT、TPOT、P50/P95/P99；
- 单请求输入/输出 token；
- QPS、并发和超时率；
- GPU 显存和利用率；
- 缓存命中与批处理收益；
- 单次请求和单个成功任务成本。

上下文窗口标称很长不代表目标长度下效果和延迟可接受。工具多、多轮长会话和长证据都需要独立测试。

### 5. 开源部署与 API 的取舍

**外部 API**

- 上手快、强模型选择多；
- 需评估数据出境、供应商锁定、限流、版本变化和成本。

**私有化模型**

- 数据和版本可控，可做量化、LoRA 和批处理；
- 需要 GPU、推理优化、容量规划、监控和升级能力。

可以采用混合架构：敏感请求走私有模型，公开低风险任务或极复杂任务在合规条件下路由到外部模型。所有路径使用统一网关、审计和 fallback。

### 6. 是否微调

先用 Prompt、RAG、工具和结构化约束建立基线。以下情况再考虑微调：

- 稳定的领域术语和输出风格；
- 大量重复任务，Prompt 成本高；
- 工具/格式错误有可学习的固定模式；
- 已有高质量、合规且足量的数据；
- 微调收益能通过冻结测试集证明。

规则更新快的知识放 RAG，不写进模型参数；资金权限和状态机放代码，不通过微调“记住”。

### 7. 分层路由和降级

生产中通常不是选一个万能模型：

- 高频简单任务走小模型；
- 复杂多证据问题升级到强模型；
- 高风险问题增加验证器和人工审批；
- 主模型超时或不可用时回退；
- 版本升级先影子评测和小流量灰度。

路由器本身也要评测，防止为了节省成本把困难任务错误发送给小模型。

### 8. 决策方式

建立模型卡矩阵，记录：

> `质量、安全、结构化输出、工具调用、语言、上下文、延迟、吞吐、成本、部署、许可、维护`

先设置硬门槛，再在通过者中比较 Pareto 前沿，而不是随意给指标加权得到一个看似精确的总分。最终选择应附带适用范围、降级方案和重新评估条件。
