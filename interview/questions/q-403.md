# NL2SQL为什么要构建元数据知识库？

## 答案

**一句话**：NL2SQL 需要元数据知识库，是因为模型只理解自然语言，并不知道企业数据库中“有哪些表、字段是什么意思、表之间怎么关联、业务指标怎么算”。元数据知识库负责把用户语言准确地落到真实 Schema 上。

### 1. 元数据知识库里应该有什么

- **物理 Schema**：库、表、字段、类型、主键、外键、索引。
- **业务语义**：字段中文含义、指标口径、枚举值、单位、时间粒度。
- **关系信息**：可连接字段、基数关系、推荐 Join 路径。
- **同义词映射**：如“销售额 / GMV / 成交金额”映射到实际字段。
- **样例与规则**：典型 SQL、常用过滤条件、字段取值样例、权限范围。

### 2. 为什么不能只把建表语句塞给模型

DDL 只能告诉模型字段名和类型，解决不了这些问题：

- `amt` 到底是订单金额、支付金额还是退款金额？
- “本月新增客户”按注册时间还是首单时间统计？
- 订单表和用户表应该通过哪个字段连接？
- 同名字段在不同业务域中含义是否一致？

如果缺少业务语义，模型即使生成了语法正确的 SQL，也可能查询口径错误，这类错误比语法错误更危险。

### 3. 在 NL2SQL 链路中的作用

```text
用户问题
  → 意图与实体识别
  → 检索相关元数据
  → Schema Linking（词语对齐表/字段/指标）
  → 生成 SQL
  → 语法、权限、成本校验
  → 执行与结果纠错
```

元数据通常按需检索，只把相关表和字段放进上下文，既减少 Prompt 长度，也降低模型虚构表名、字段名和 Join 关系的概率。

### 4. 工程注意点

- 元数据必须跟数据库变更同步，避免知识库过期。
- 指标口径要有负责人、版本和生效时间。
- 检索阶段同时使用关键词、向量和关系图信息。
- SQL 执行前做只读限制、行列权限、扫描量和超时校验。
- 用执行正确率和业务口径正确率评估，不能只看 SQL 字符串是否一致。

**面试总结**：元数据知识库的核心价值是完成 **自然语言 → 业务概念 → 数据库对象** 的可靠映射，让 NL2SQL 从“会写 SQL”升级为“理解企业数据”。
