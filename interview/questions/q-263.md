# Agent 的 Plan 是怎么实现的？

## 答案

> Agent 的 Plan 主要有**三种范式**——Plan-and-Execute（先规划后执行）、ReAct（边走边想）、Tree of Thought（分支探索），我们跨境支付里用 Plan-and-Execute 居多。
>
> **Plan-and-Execute 流程**：
>
> ```
> 1. Planner：LLM 看任务 → 输出一个有序步骤列表
>    [
>      {"step": 1, "action": "查订单状态", "tool": "query_order"},
>      {"step": 2, "action": "查关联交易", "tool": "query_related"},
>      {"step": 3, "action": "综合判断", "tool": "llm_summarize"}
>    ]
>
> 2. Executor：按顺序执行每个步骤
>    for step in plan:
>        result = await execute(step)
>        if result.failed: replan()
>
> 3. Replanner：执行中如果偏差大，重新规划剩余步骤
> ```
>
> **关键设计点**：
>
> **1. 计划的结构化输出**
> 用 JSON Schema + Pydantic 强约束 LLM 输出：
> ```python
> class PlanStep(BaseModel):
>     step_id: int
>     description: str
>     tool: str
>     args_template: dict  # 可能依赖前面步骤的结果
>     expected_output: str
>
> class Plan(BaseModel):
>     goal: str
>     steps: list[PlanStep]
> ```
>
> **2. 动态参数依赖**
> 步骤 2 可能用步骤 1 的输出：
> ```python
> args_template = {"order_id": "{step_1.output.order_id}"}
> # 执行时模板填充
> ```
>
> **3. Replanning 触发**
> - 步骤失败 N 次
> - 中途发现新信息
> - 偏离目标太远
> - 用户中途打断
>
> **4. 步数和 token 上限**
> - 步骤 >10 强制结束
> - token >50k 强制结束
> - 防止 Agent "走神"
>
> **我们风控调查 Agent 的具体 Plan 实现**：
>
> ```python
> async def plan_investigation(ticket: Ticket) -> Plan:
>     prompt = f"""
>     工单：{ticket}
>     可用工具：{available_tools}
>     请规划调查步骤，输出 JSON。每步包含 step_id、action、tool、依赖关系。
>     注意：
>     - 必须包含 KYC 查询、关联交易查询、规则匹配三个核心步骤
>     - 涉及大额转账必须加合规校验步骤
>     - 最多 8 步
>     """
>     plan = await llm.invoke(prompt, response_format=Plan)
>     return plan
> ```
>
> **执行 + Replanning**：
> ```python
> async def execute_plan(plan: Plan):
>     context = {}
>     for step in plan.steps:
>         args = render_args(step.args_template, context)
>         try:
>             result = await call_tool(step.tool, args)
>             context[f"step_{step.step_id}"] = result
>         except Exception as e:
>             # 失败 -> 重新规划剩余步骤
>             plan = await replan(plan, current_step=step, error=e)
>             continue
>     return summarize(context)
> ```
>
> **ReAct vs Plan-and-Execute 怎么选**：
>
> | 维度 | ReAct | Plan-and-Execute |
> |------|-------|-----------------|
> | 上手 | 简单 | 中等 |
> | 可追溯 | 一般 | 好 |
> | 适合任务 | 短链路 | 长链路 |
> | 错了能改 | 自然 self-correct | 需要 replan |
> | 合规审计 | 弱 | 强 |
>
> 金融场景**默认 Plan-and-Execute**——可追溯、可审计、可回滚。
>
> **几个工程经验**：
> - 计划要有"为什么这么做"的 reasoning 字段，方便复盘
> - 关键步骤强制 HITL 节点（高风险操作必须人工 confirm）
> - Replan 次数限制（防死循环）
> - Plan 持久化（断点恢复）
> - 监控"plan adherence"——实际执行偏离计划多少
>
> 一句话：**Plan 是把 LLM 自由度装在有边界的框架里**，金融场景必须可追溯，所以 Plan-and-Execute 是首选。

- "Plan 持久化和断点恢复怎么做可以展开。"
