# DPO 的第 0 步 loss 是多少？

## 答案

**一句话**：**log(2) ≈ 0.693**——这是个 sanity check 题，第 0 步 π_θ = π_ref，所有 log ratio 都是 0，sigmoid(0) = 0.5，-log(0.5) = log(2)。

### 1. DPO Loss 公式

$$L_{DPO} = -\log \sigma\left(\beta \left[\log\frac{\pi_\theta(y_w|x)}{\pi_{ref}(y_w|x)} - \log\frac{\pi_\theta(y_l|x)}{\pi_{ref}(y_l|x)}\right]\right)$$

- π_θ：待训模型
- π_ref：参考模型（通常是 SFT 模型，**训练全程冻结**）
- y_w：chosen（好回复）
- y_l：rejected（差回复）
- β：温度系数，控制偏离 ref 的程度（典型 0.1）

### 2. 第 0 步的推导
- 初始化：**π_θ ← π_ref**（拷贝权重）
- 所以 π_θ(y|x) = π_ref(y|x) 对任意 y
- → log(π_θ/π_ref) = log(1) = 0
- → 差值 = 0 - 0 = 0
- → β × 0 = 0
- → σ(0) = 0.5
- → L = -log(0.5) = **log(2) ≈ 0.6931**

### 3. 为什么这是 sanity check
- 训练刚开始 loss 应该精确等于 0.693
- 偏离明显（比如一开始就 0.2 或 1.0）→ **ref model 没冻 / 加载错了 / β 设错了**
- 实际工程里你 print 第一个 batch loss 就能立刻知道有没有配错

### 4. 训练过程中 loss 的方向
- 训练目标是让 chosen 的 log ratio > rejected 的 log ratio
- 差值越来越大 → σ → 1 → -log(σ) → 0
- 所以 loss 是**从 0.693 单调下降**（理想情况）

### 5. 常见踩坑
- ref model 没冻 → loss 一开始就不是 0.693
- chosen 和 rejected 弄反 → loss 不降反升
- β 太大（>1）→ 模型很难偏离 ref，loss 卡在 0.6 附近
- β 太小（<0.05）→ 模型偏离太多，KL 爆炸，胡说八道

### 6. 一句话总结
**DPO 第 0 步 loss = log(2) ≈ 0.693 是个铁律**，能背、能推、能讲为什么是 sanity check，这题就拿满分。

## 结合项目谈（口语版 · 约 40 秒）

> 这道题我自己印象很深，因为我们跨境支付**客服回复对齐**第一次跑 DPO 就因为这个 0.693 救了一次命。
>
> 当时我们用 Qwen-7B 做 base，先 SFT 跑了一轮客服话术，然后准备 DPO 二轮对齐——chosen 是高满意度回复、rejected 是被客户标低分的回复。我们组里实习生写训练脚本，跑起来第一个 batch loss 就给我 print 了个 0.41，我一看就拦下来了。
>
> 推过去看代码——他没把 ref model 单独加载和冻参，是用了同一个 model 实例，相当于 π_θ 和 π_ref 同步更新了，log ratio 就不是 0 了。改完之后第一个 batch 准确显示 0.6931，跑下来效果也对了。
>
> 后来我们把"第 0 步 loss 校验"做成了所有对齐训练的标准 sanity check 步骤，写在 launch script 里——loss 偏离 0.693 超过 1% 直接 abort，不让浪费 H100 时间。
>
> 一个小细节救了大成本，我觉得这种题面试问的不是公式，是问你跑过没跑过。
