# 传统RAG有什么痛点;介绍GraphRAG,GraphRAG的难点是什么;GraphRAG如何应对增量场景

## 答案

传统 RAG 以独立 chunk 为检索单元，适合局部事实问答；GraphRAG 将实体、关系、社区和原文证据组织起来，适合跨文档关联、多跳查询和全局主题总结。两者是互补关系，不是简单替代。

### 1. 传统 RAG 的主要痛点

- **关系被切断**：同一商户、账户或规则的信息分散在多个 chunk；
- **多跳能力弱**：难以回答 A 与 C 如何通过 B 关联；
- **全局问题困难**：Top-K 局部片段不能代表整个语料的主题；
- **实体歧义**：简称、别名和同名实体容易混淆；
- **证据重复**：多个文档复述同一内容，占用上下文；
- **时序和版本弱**：新旧规则、主体关系变化难以表达。

扩大 Top-K 或使用长上下文只能增加候选信息，不能自动恢复缺失的关系结构。

### 2. GraphRAG 的核心结构

离线阶段：

1. 解析文档并生成稳定 chunk；
2. 抽取实体、关系、事件和 claim；
3. 实体消歧与合并；
4. 建立实体到原始 chunk 的证据边；
5. 对图进行社区发现；
6. 为社区生成分层摘要。

在线查询可以选择：

- **Local Search**：从相关实体出发扩展局部子图，回答具体关系问题；
- **Global Search**：基于社区报告回答全局主题和趋势；
- **DRIFT Search**：先利用社区信息形成较宽视角，再逐步下钻到局部证据；
- **Hybrid Search**：图检索与 BM25、向量和结构化查询融合。

最终回答必须能回溯到原始文档或权威业务数据，不能只引用 LLM 生成的社区摘要。

### 3. 跨境支付中的应用边界

文档图适合关联：

- 国家、币种、产品、通道和规则；
- KYC/KYB 材料、限制条件与例外；
- 错误码、原因和处理 SOP。

交易关系图适合关联：

- 商户、账户、收款人、交易、设备、IP 和最终受益人；
- 多笔交易共享主体或资金路径；
- 风险主体与历史调查证据。

两类图的数据来源和权限不同。文档关系可由 LLM 辅助抽取；交易、账户和资金关系应来自业务数据库、CDC 和经过治理的实体解析，不能依赖模型从文本猜测。

### 4. GraphRAG 的难点

**图谱质量**

- 实体边界、别名和同名消歧困难；
- LLM 可能抽错方向、关系类型和时间；
- 错误关系会在多跳检索中放大。

**规模和成本**

- LLM 抽取、社区摘要和重建成本高；
- 高度节点和多跳遍历容易产生子图爆炸；
- 图数据库、向量库和原文存储需要一致性管理。

**检索与生成**

- hop 数、关系类型和时间窗口需要约束；
- 子图序列化会丢失结构或占用大量 token；
- 社区摘要可能过度压缩细节；
- 权限必须传播到节点、边和原文证据。

**评测**

- 不仅要评最终答案，还要评实体、关系、路径和引用；
- 简单 FAQ 上 GraphRAG 可能增加延迟而没有收益。

### 5. 增量更新的基本原则

增量更新不能简单“把新三元组追加进去”，需要维护文档、实体、关系、社区和摘要之间的依赖。

每个对象使用稳定 ID：

> `document_id → chunk_id → entity_id/relation_id → community_id → report_id`

更新流程：

1. 通过内容哈希或 CDC 识别新增、修改和删除；
2. 只解析变化文档，生成 delta chunk；
3. 对 delta 做实体链接，优先匹配已有实体；
4. 对旧文档产生的实体和边做引用计数或 provenance 更新；
5. 删除内容使用 tombstone，避免幽灵关系继续被检索；
6. 找出受影响的局部子图和社区；
7. 仅重算受影响社区的聚类、摘要和 embedding；
8. 在影子索引验证后通过 alias 或版本号原子切换。

实体合并与拆分要保留版本和重定向关系。否则一次消歧修正会让历史引用和缓存全部失效。

### 6. 一致性、时效和降级

- 文档版本、图版本、向量索引版本和摘要版本绑定到同一次发布；
- 在线查询携带 `as_of_time`，只读取当时有效的节点和边；
- 图更新失败时继续使用上一稳定版本，新增文档可暂时走普通向量检索；
- 高频实时交易关系进入 delta graph，周期性与主图合并；
- 权限删除和敏感主体下架优先同步，不能等待批量重建。

微软 GraphRAG 的 update 工作流可以复用旧索引产物更新新文档，但生产系统仍需自行处理删除、实体版本、权限和业务 CDC。

### 7. 评测和选型

- 实体/关系抽取 Precision、Recall、F1；
- 实体消歧准确率；
- Path Recall、路径正确率和证据覆盖；
- Local/Global 问题的答案正确率；
- 增量新鲜度、幽灵边比例和版本一致性；
- P95、建图成本和每次更新成本。

简单规则查询优先使用普通 RAG；只有关系、多跳或全局问题在评测中有明确增益时才启用 GraphRAG。
