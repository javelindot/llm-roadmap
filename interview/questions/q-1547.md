# 怎么构建SFT数据集,数据量多少,微调方式是什么

## 答案

构建 SFT 数据集的核心不是追求条数，而是让样本覆盖目标任务、行为边界和失败模式，并保证答案正确、格式一致、没有泄漏。数据量与任务复杂度、基座能力和训练方式相关。

### 1. 先定义目标能力

把任务拆成可评测类型：

- 指令遵循和输出格式；
- 领域问答；
- 多轮澄清；
- 工具选择与参数；
- 复杂推理或流程；
- 拒答、转人工与越权边界；
- 多语言、长文本和异常输入。

每类先建立独立验证集，再收集训练数据。

### 2. 数据来源

- 经过授权和脱敏的真实历史记录；
- 专家编写的黄金示范；
- SOP、FAQ 和文档转为指令；
- 规则或程序生成的可验证样本；
- 强模型合成后由规则/人工筛选；
- 线上 bad case 修复；
- 公开数据只补通用能力，不替代业务数据。

涉及隐私、版权和许可证的数据必须保留来源、授权和删除能力。

### 3. 数据格式

```json
{
  "messages": [
    {"role": "system", "content": "任务与边界"},
    {"role": "user", "content": "用户请求"},
    {"role": "assistant", "content": "目标回答"}
  ],
  "metadata": {
    "task": "tool_call",
    "language": "zh",
    "source": "expert",
    "quality": 1.0,
    "version": "v3"
  }
}
```

工具数据还要保存 tool schema、调用与 observation。训练时通常只对 assistant token 计算 loss，是否训练 reasoning trace 要根据可验证性和上线输出要求决定。

### 4. 清洗与切分

- 去重和近重复聚类；
- 规则校验 JSON、代码和工具参数；
- 人工抽检与双人仲裁；
- 过滤事实错误、泄漏、模板污染；
- 统一 chat template 与特殊 token；
- 按用户、商户、文档簇和时间切分；
- 防止同源合成样本跨 train/test；
- 控制任务、语言、难度和拒答比例。

### 5. 数据量多少

没有固定答案，可用经验起点：

- 格式/风格适配：数百到数千条高质量样本；
- 单一领域任务：数千到数万；
- 多任务、多语言、工具 Agent：数万到数十万以上；
- 继续预训练则按 token 规模衡量，远大于 SFT。

应画 data scaling curve：

```text
1k -> 3k -> 10k -> 30k ...
```

当新增数据的验证收益趋于饱和，再转向补缺口和提高质量。十万条低质量样本可能不如几千条专家数据。

### 6. 微调方式

- **LoRA**：默认起点，成本低、易迭代；
- **QLoRA**：显存更紧时使用；
- **全参数 SFT**：数据充足、目标变化大、追求上限；
- **部分层/Adapter**：按部署需求选择；
- SFT 后若剩余问题是偏好排序或可验证策略，再用 DPO/GRPO 等。

训练需记录学习率、batch、epoch、sequence length、packing、loss mask 和 checkpoint，并同时做通用能力回归。

### 7. 跨境支付场景

样本应覆盖国家、币种、支付方式、多语言、失败原因、退款与拒付，并加入越权查询、信息不足和转人工示范。订单状态不能从历史回答硬编码，应训练模型调用权威工具。

### 8. 面试总结

可信答案包含：目标能力矩阵、来源、格式、清洗、无泄漏切分、数据 scaling 实验和 LoRA/全参选型。数据量由验证曲线决定，不是背一个数字。
