# 你是怎么设计agent的记忆系统？长期记忆如何存储？如果历史记录量非常大，怎么优化查询效率？

## 答案

Agent 记忆应按用途和可信度分层，不能把全部聊天记录直接塞进向量库，也不能把模型摘要当作权威业务事实。

### 1. 记忆分层

- **工作记忆**：当前任务状态、计划、工具结果，放在 context/checkpoint；
- **情景记忆**：过去任务、事件和处理轨迹；
- **语义记忆**：从多次交互提炼的稳定事实或偏好；
- **程序记忆**：SOP、工具说明和工作流；
- **权威状态**：订单、余额、权限等业务数据库，不属于模型自由记忆。

### 2. 长期存储

一条记忆至少保存：

```text
memory_id
tenant/user scope
type
content
structured fields
embedding
source/evidence
created_at / valid_time
confidence
version
TTL / deletion state
```

结构化字段进关系数据库，语义向量进向量索引，大文本或附件进对象存储。重要事实保留原始证据指针，而不是只存摘要。

### 3. 写入策略

不是每轮对话都写长期记忆。写入前执行：

- 是否对未来任务有用；
- 是否稳定而非一次性状态；
- 是否得到用户同意；
- 是否包含 PII 或敏感数据；
- 是否与已有记忆重复或冲突；
- 是否有来源和置信度；
- 是否应设置 TTL。

可由规则先筛选，再由模型提取候选，最终通过 schema 和权限校验。高风险事实由权威系统写入。

### 4. 大规模查询优化

1. 按租户、用户、类型、时间和权限先做 metadata filter；
2. BM25 + 向量混合召回，避免精确实体漏检；
3. HNSW/IVF 等 ANN，并按分区建立索引；
4. 最近窗口、长期摘要和原始事件做层级检索；
5. 先粗召回，再 rerank、去重和时间衰减；
6. 对高频安全查询做权限感知缓存；
7. 合并重复事件、定期压缩和归档冷数据；
8. 使用热/冷分层与增量索引，避免每次扫全历史。

检索分数可组合：

```text
relevance + recency + importance + confidence
```

但不同任务权重不同，不能固定一个公式。

### 5. 冲突与遗忘

- 使用 valid-time 和版本判断新旧；
- 保留冲突记录，不能静默覆盖；
- 用户可查看、修正和删除；
- TTL 到期自动失效；
- 模型生成摘要时保留来源；
- 删除要同步关系库、向量索引、缓存和备份策略。

### 6. 跨境支付场景

可记住用户语言偏好、常用报表格式等低风险信息；卡号、证件、完整交易明细、风控阈值不应进入自由记忆。订单状态每次查询权威系统，且按商户、角色和地域严格过滤。

### 7. 面试总结

记忆系统的关键是“分层、选择性写入、带来源存储、权限过滤、混合检索和生命周期治理”。历史很大时使用 metadata 分区、ANN、层级摘要、冷热分层和 rerank，而不是扩大上下文。
