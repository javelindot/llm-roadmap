# LLM的参数介绍(temp topk top p等)

## 答案

这些参数作用于“从 logits 选择下一个 token”的解码阶段，不会改变模型权重。不同推理框架的处理顺序和边界语义可能不同，应以具体 API 为准。

### 1. Temperature

```text
p_i = softmax(logit_i / T)
```

- `T < 1`：分布更尖锐，更确定；
- `T > 1`：分布更平坦，更多样；
- `T -> 0`：趋近 greedy argmax。

有些 API 不接受严格的 0，而是提供 greedy/do_sample 开关。

### 2. Top-k

只保留概率最高的 `k` 个 token，再归一化采样。

- `k=1` 等价于 greedy；
- 固定候选数，简单可控；
- 不同上下文的概率分布宽窄不同，固定 k 不总适合。

### 3. Top-p

按概率从高到低选最小集合，使累计概率达到 `p`，再归一化采样：

```text
sum(selected probabilities) >= p
```

分布确定时集合小，分布不确定时集合大，比 top-k 更自适应。top-k 与 top-p 同时设置时通常先后过滤，实际结果取两者约束后的集合。

### 4. 其他常见参数

- `max_new_tokens`：最多生成多少新 token；
- `min_tokens`：达到最小长度前限制 EOS；
- `stop`/stop token IDs：命中后停止；
- `repetition_penalty`：抑制重复 token；
- `frequency_penalty`：按出现次数惩罚；
- `presence_penalty`：出现过即惩罚，鼓励新内容；
- `min_p`：过滤相对最高概率过低的 token；
- `seed`：帮助复现，但并行 kernel/API 未必完全确定；
- `logprobs`：返回候选 token 的对数概率；
- `num_beams`：Beam Search 宽度，常用于翻译和受约束生成；
- `n`/`best_of`：生成多个候选，显著增加计算和 KV Cache；
- `length_penalty`：Beam Search 的长度偏好。

### 5. 参数组合

事实问答/工具参数：

```text
greedy 或低 temperature
受控 max_new_tokens
JSON schema / constrained decoding
```

创意生成：

```text
temperature 适中
top_p 或 top_k
必要时 repetition penalty
```

代码和推理题不能只靠把 temperature 调低保证正确，应使用测试、verifier 或多样本验证。

### 6. 常见误区

- temperature 不等于模型置信度；
- top-p=0.9 不表示答案有 90% 正确率；
- 同时把 temperature 很高、top-k/top-p 很宽会显著增加随机性；
- 过强重复惩罚可能破坏代码、数字和专有名词；
- `max_tokens` 过大增加尾延迟和成本；
- 即使 greedy，相同 Prompt 也可能因模型版本或服务实现变化而不同。

### 7. 跨境支付场景

订单解释和工具调用使用低随机性、结构化约束和字段校验。金额、币种、账户和状态必须来自工具，不能通过调低 temperature 把模型输出变成事实。

### 8. 面试总结

Temperature 调整分布尖锐度，top-k 固定候选数量，top-p 按累计概率动态选择候选。生产选参要结合任务正确性、输出约束、延迟和成本，并使用 verifier 保证关键结果。
