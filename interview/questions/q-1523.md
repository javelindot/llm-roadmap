# 知识蒸馏的理解，出了几个场景问我怎么衡量蒸馏的好坏

## 答案

知识蒸馏的好坏不能只看学生准确率，而要判断它是否在目标资源约束下形成更好的“质量-效率 Pareto 前沿”，并检查学生是否继承教师的错误和偏差。

### 1. 建立三个对照

- 同规模学生直接训练，不使用蒸馏；
- 教师模型上限；
- 蒸馏后的学生。

可计算教师能力保留率：

```text
retention =
(student_distilled - student_baseline)
/ (teacher - student_baseline)
```

具体指标方向需要统一；该比率只作为辅助，不能替代绝对业务指标。

### 2. 质量指标

分类任务：

- accuracy、macro-F1、关键类别 recall；
- calibration、AUROC/PR-AUC；
- 长尾和 OOD。

生成任务：

- 任务成功率、事实正确率、引用支持率；
- exact match、pass@k 或 verifier pass rate；
- 人工偏好、幻觉率和安全违规；
- 多语言、长上下文、工具调用和格式遵循。

不能只看学生拟合教师的 KL，因为“更像教师”不等于对真实任务更正确。

### 3. 效率指标

- 参数量、磁盘大小、峰值显存；
- P50/P95/P99、吞吐、并发；
- TTFT、TPOT；
- CPU/GPU/端侧功耗；
- 单请求或每百万 token 成本；
- 部署冷启动和兼容性。

### 4. 不同场景怎么判断

**端侧分类**

在内存、功耗和 P95 约束下，关键类别 recall 达标才算成功。

**LLM 客服**

比较解决率、事实性、拒答、人工接管、延迟和成本，而不只看 BLEU/ROUGE。

**推理模型**

使用未见题目的 verifier/pass@k，同时观察输出长度和 test-time compute，防止学生只学格式。

**安全模型**

单独评估 jailbreak、隐私、偏见和高风险漏判，平均分不能掩盖严重失败。

### 5. 蒸馏诊断

- 按难度、类别、语言和长度分桶；
- 检查教师正确但学生错误、教师错误且学生复制的样本；
- 消融硬标签、软标签、温度和中间层损失；
- 检查校准和置信度；
- 使用独立、未被教师生成过程污染的测试集；
- 与量化、剪枝和更小基础模型比较成本收益。

### 6. 跨境支付场景

蒸馏客服或风控模型时，必须单测金额、币种、制裁、退款和拒付等高风险切片。学生即使平均准确率接近教师，只要关键风险漏判或事实错误上升，就不能上线。

### 7. 面试总结

蒸馏成功的标准是：相较未蒸馏学生显著提升，在模型大小、时延和成本约束下达到业务门槛，并且没有不可接受的长尾、安全和校准退化。
