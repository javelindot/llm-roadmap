# 上下文压缩的大小怎么评判

## 答案

**一句话**：上下文压缩大小不是拍脑袋定的，要看模型窗口、任务复杂度、关键信息密度、工具结果大小、成本延迟和最终答案质量。

### 1. 为什么要压缩

长上下文会带来几个问题：

- token 成本高。
- 延迟变长。
- 旧信息干扰当前判断。
- 工具返回结果太大，挤占真正有用内容。
- 超出模型上下文窗口。

所以压缩的目标不是越短越好，而是保留决策必须的信息，去掉噪音。

### 2. 压缩大小怎么判断

我会从几方面评估：

- **模型窗口**：比如模型可用上下文是 32K，就要给系统 prompt、用户问题、工具结果、输出预留空间。
- **任务类型**：简单 FAQ 可以压得很短，复杂交易排查要保留更多步骤。
- **信息密度**：订单号、金额、状态、错误码、时间、国家、币种不能丢。
- **答案质量**：压缩后如果引用错误或漏关键条件，就说明压过头。
- **成本延迟**：高频客服场景要更严格控制上下文。

### 3. 常见压缩策略

- 对话历史压成用户目标、关键事实、已确认信息。
- 工具大结果只保留字段摘要和异常点。
- RAG 结果先 rerank，再只保留最相关段落。
- 长文档按标题和段落提炼，不破坏结构。
- 对资金、合规、风控相关字段设置“不可压缩字段”。

### 4. 评估方法

可以用一组真实问题做 A/B：

- 不压缩、轻压缩、强压缩分别跑。
- 比较准确率、引用命中率、漏答率、平均 token、响应时间。
- 对高风险问题人工抽检。

最终找到一个业务可接受的平衡点。

### 5. 结合跨境支付项目怎么说（面试口径）

> 在跨境支付客服场景里，上下文压缩不能把关键交易事实压没。比如一笔付款的国家、币种、金额、通道、当前状态、错误码、风控命中大类、预计到账时间，这些都是不可丢字段。闲聊、重复追问、无关历史可以压缩。
>
> 我会把会话历史压成“用户正在咨询某笔 USD payout 未到账，已确认订单号，当前通道状态为 pending，通道回执显示银行审核中”。这样模型不需要看完整聊天记录，也能继续回答。
>
> 如果工具返回一大段日志，我不会整段塞给模型，而是先提取异常码、时间线和关键状态变化。压缩后再用评测集验证，看是否影响付款失败原因解释、KYC 驳回解释和风控拦截解释的准确率。

**钩子**：可以引到“不可压缩字段、工具结果摘要、用评测集确定压缩阈值”。
