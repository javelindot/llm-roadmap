# 场景题:如何训练一个大模型以实现精确的摘要提取

## 答案

训练精确摘要模型前，先明确是 **抽取式摘要、生成式摘要还是结构化信息提取**。如果业务要求金额、状态、日期等字段零改写，优先抽取或 Schema 输出；如果需要自然语言压缩，再使用受证据约束的生成式摘要。

### 1. 定义任务和验收标准

需要明确：

- 文档类型与长度；
- 摘要目标受众；
- 字数或 token 预算；
- 必须保留的事实与实体；
- 是否允许推断和改写；
- 无法确认时的输出方式；
- 是否需要引用页码或原文 span。

跨境支付工单摘要通常至少保留：商户、订单、金额、币种、时间、状态、通道回执、已执行动作、待办和风险提示。

### 2. 构建高质量数据

每条样本包含原文、摘要、关键事实、证据位置和文档元数据。数据生产流程为：

1. 从真实分布抽样并脱敏；
2. 业务专家编写或修订摘要；
3. 标注必须保留与禁止推断的字段；
4. 多人复核数字、否定、时间和因果关系；
5. 按商户、时间或文档来源切分训练/验证/测试；
6. 去除重复模板和评测泄漏。

可以用强模型生成候选，但只能作为预标注，关键样本必须人工核对。

### 3. 建立非微调基线

先比较：

- 规则与字段抽取；
- Prompt + 长上下文；
- 分块摘要后再汇总；
- RAG 选证据后摘要；
- 小模型与强模型。

如果 Schema 抽取已经满足需求，不应为了“训练大模型”增加生成风险。

### 4. SFT 训练

使用高质量 `(document, instruction, summary)` 做 SFT：

- 只对摘要部分计算 loss；
- 对长度、体裁和字段要求使用显式控制 token；
- 长文档采用结构化截断、层级摘要或长上下文模型；
- 混入短文档和通用指令数据，避免能力退化；
- 对数字、日期和专有名词增加困难样本；
- LoRA 建基线，容量不足再评估全量 SFT。

对于超长文档，可先按章节生成带证据的局部摘要，再由第二阶段仅基于局部摘要与关键原文生成总摘要。中间结果要保留来源，防止错误逐层放大。

### 5. 偏好与事实优化

构造 chosen/rejected pair：

- chosen 完整、简洁且忠实；
- rejected 漏关键事实、混淆主体、虚构因果、改错数字或过度冗长；
- 尽量控制长度和文风，只保留目标质量差异。

DPO 可优化表达偏好，但结构化事实优先使用程序化验证器。金额、币种、日期、订单号和状态可从原文或工具结果中确定性比对。

### 6. 生成约束

- 输出 JSON Schema 或固定章节；
- 要求每个关键字段附证据 span/page；
- 对数字和实体做 copy/constrained decoding；
- 生成后运行字段一致性与 NLI 检查；
- 证据冲突或缺失时标记未知，不自行补齐；
- 高风险摘要进入人工复核。

### 7. 评测

ROUGE 只能衡量词面重合，不能单独代表精确。应组合：

- ROUGE/BERTScore：覆盖与语义相似；
- key fact recall/precision；
- entity、number、date exact match；
- faithfulness 与 contradiction rate；
- 压缩率、长度遵循和重复率；
- 专家盲评：完整、简洁、可读；
- 业务指标：人工修改率、处理时长和错误升级率。

评测按长短文档、语言、表格、OCR、否定、时间线和冲突证据分桶。摘要更流畅但数字错误率上升，不能上线。
