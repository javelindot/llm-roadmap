# prompt调试有做过吗？

## 答案

做过。Prompt 调试不应靠主观“读起来顺不顺”，而应像代码和模型一样版本化、建立评测集、做单变量实验并分析 bad case。

### 1. 先定义目标

明确 Prompt 要优化什么：

- 事实正确率；
- 工具选择和参数准确率；
- JSON/Schema 合法率；
- 引用完整率；
- 拒答和越权率；
- 延迟、token 和成本。

目标冲突时要设优先级，例如支付场景中事实与权限高于措辞自然度。

### 2. 建立评测集

按真实流量分层：

- 常见问题；
- 长尾和歧义问题；
- 多语言；
- 缺字段；
- 工具异常；
- Prompt Injection；
- 高风险合规和资金操作。

每个样本保存预期事实、允许工具、禁止动作和评分规则。开发集用于调试，测试集保持冻结。

### 3. 结构化 Prompt

常用结构：

```text
Role
Business scope
Authoritative data sources
Tool rules
Decision constraints
Output schema
Examples
Fallback / escalation
```

规则尽量具体、可验证，避免“请认真思考”“尽量准确”这类无法执行的要求。

### 4. 调试方法

- 一次只改一个变量；
- 记录 Prompt 版本、模型、参数和评测结果；
- 对失败样本做归因：检索错、工具错、指令冲突、上下文丢失或模型能力不足；
- 少量高质量 few-shot 覆盖边界，不堆几十个相似例子；
- 对结构化输出使用 Schema constrained decoding；
- 用 tracing 查看模型看到的真实 system、history、RAG 和 tool result。

### 5. 跨境支付案例

例如“解释支付失败”Prompt 明确：

- 订单状态以订单工具为准；
- 错误码解释以当前通道文档为准；
- 不猜测余额、风控规则和制裁命中原因；
- 缺少 merchant_id/order_id 时先追问；
- 只读查询可自动执行；
- 退款、冻结、调账必须走审批；
- 输出包含结论、证据、下一步和置信度。

### 6. 防止过度依赖 Prompt

权限、幂等、参数校验、敏感字段脱敏和金额计算必须在代码层完成。模型反复违反同一规则时，应改工作流、工具 schema、检索或训练，而不是无限追加 Prompt。

**面试总结**：Prompt 调试的成熟度体现在“有评测、有版本、有归因、有回归”，不是展示一段很长的提示词。
