# 3.RAG链路的流程。如何评估rag的效果？有哪些指标？

## 答案

RAG 要拆成“数据建库、在线检索、生成校验、持续评测”四段。评估也必须分层，否则端到端答案错了时无法判断是文档没召回、排序不对，还是模型没有忠实使用证据。

### 1. 离线建库

```text
数据接入
-> 解析/OCR/表格抽取
-> 清洗、去重、版本和权限
-> 结构化分块
-> 标题/元数据增强
-> Embedding
-> Dense/Sparse Index
```

支付知识库还需要国家、币种、通道、卡组织、生效时间、商户和租户权限等 metadata。

### 2. 在线链路

```text
Query
-> 意图识别/权限检查
-> Query Rewrite
-> Metadata Filter
-> BM25 + Dense + 结构化查询
-> Fusion
-> Reranker
-> Context Packing
-> LLM Generate
-> Citation/Fact/Policy Check
-> Answer or Escalation
```

交易状态、余额、费率、汇率等实时数据应调用工具，不应只查静态文档。

### 3. 检索层指标

- Recall@K：正确证据是否进入前 K；
- Precision@K：前 K 中相关文档比例；
- MRR：第一个正确结果位置；
- NDCG@K：考虑多级相关性的排序质量；
- MAP：多正例场景的平均精度；
- Hit Rate、路由准确率、过滤正确率。

评测集要有 query、可接受证据列表和版本/权限信息。

### 4. 生成层指标

- Answer Correctness：答案是否正确；
- Faithfulness/Groundedness：结论是否能由上下文支持；
- Citation Precision/Recall：引用是否真的支持结论、是否漏引；
- Completeness：关键要点是否覆盖；
- Refusal Accuracy：证据不足时是否正确拒答；
- 格式、语气和多语言质量。

LLM-as-Judge 可用于规模化评测，但高风险样本要结合规则、专家标注和人工抽检。

### 5. 端到端与系统指标

- 任务成功率、首次解决率、转人工率；
- P50/P95/P99、TTFT、token 成本；
- 检索/模型/工具失败率；
- 缓存命中率；
- 跨租户泄漏、越权、Prompt Injection 成功率。

### 6. 归因实验

至少做：

- Oracle Context：给正确证据，判断生成器上限；
- Oracle Generator：人工读召回结果，判断检索上限；
- 去掉 BM25/Rerank/Rewrite 的消融；
- 按国家、语言、通道、问题类型分桶；
- 时间切分，验证规则更新后的表现。

### 7. 跨境支付案例

对于“某国本地转账失败如何处理”，正确答案不仅要召回错误码，还要匹配国家、通道和当前版本。若订单工具不可用，答案应明确无法核验实时状态并转人工，而不是根据旧文档猜测。

**面试总结**：RAG 的核心指标链是“证据召回正确 -> 排序靠前 -> 模型忠实使用 -> 业务问题解决”。
