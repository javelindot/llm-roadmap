# 是否使用过 Vibe Coding、Claude Code、Cursor 这类开发工具？

## 答案

> 用过，三个都长期在用，分享一下我的体验和选型。
>
> **Cursor**（VSCode fork，最早上手）
> - **优点**：基于 VSCode 生态，上手 0 成本；Composer 多文件改动方便；Tab 补全很顺
> - **缺点**：模型背后是 Anthropic / OpenAI API，国内合规和延迟有问题；订阅贵
> - **我的用法**：本地写脚本、小修小补、看代码很顺手
>
> **Claude Code**（Anthropic 官方 CLI）
> - **优点**：命令行直接跑，跟 git 工作流融合好；Skill / Hooks / MCP 生态强；长任务（重构、调试）特别擅长；可在 SSH 服务器上用
> - **缺点**：纯文本界面，对依赖图形预览的前端开发不友好；模型走 Anthropic 通道
> - **我的用法**：服务器上跑长任务（数据处理、批量重构）、写文档、复杂 debug；我们最近也在用 Claude Code 跑 RAG 索引重建这种长任务
>
> **Vibe Coding**（理念多过工具）
> - 严格说"Vibe Coding"不是某个工具，是一种**"用 LLM 当主力，自己当审阅者"**的工作模式
> - 实践上多通过 Cursor / Claude Code / Aider 体现
> - **特点**：写 prompt 描述需求 → LLM 生成代码 → 自己只读关键节点 → 不读细节
> - **适合**：MVP、原型、个人项目
> - **不适合**：金融生产代码（必须每行读懂、必须有测试覆盖）
>
> **我们团队的使用情况**：
>
> - 算法工程师：80% 时间用 Cursor（写训练脚本、改模型）
> - 后端工程师：50% 用 Cursor + 50% 用 Claude Code（服务器侧长任务）
> - 我自己：日常 Cursor、长任务 Claude Code、紧急小 fix 用 Cursor 内嵌 chat
>
> **几个体会**：
> - **AI 编程提升 30%-50% 效率**——但只对"知道自己在写什么"的人有用，新手反而容易写出无法 debug 的代码
> - **Code review 标准更严格**——人工写的 bug 模式我熟悉，AI 生成的 bug 模式更随机，要更仔细审
> - **Skill 把团队 SOP 沉淀下来**——我们把内部代码规范、PR 模板、风控审查 checklist 都做成 Skill
> - **Vibe Coding 在金融生产线不可取**——我们 PR review 强制每行读懂、强制单元测试覆盖
>
> 一句话：**AI 编程工具不是替代工程师，是放大器**——放大好工程师的效率、也放大坏工程师的灾难。

**🪝 顺嘴带出去的钩子**：
- "团队怎么把 SOP 做成 Claude Code Skill 可以聊。"
- "AI 生成代码的 PR review 我们有专门 checklist 可以展开。"
- "Cursor vs Claude Code 在长任务上的差异可以讲讲。"
