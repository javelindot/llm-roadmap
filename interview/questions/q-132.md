# 谈谈HYDE问题？

## 答案

**一句话**：HyDE 是先让模型生成一个假设答案，再用这个假设答案去做向量检索；问题是它可能引入幻觉和错误方向。

### 核心回答
HyDE 流程：
1. 用户提问。
2. LLM 生成一个 hypothetical document/answer。
3. 对假设答案做 embedding。
4. 用这个向量检索真实文档。
5. 再基于真实文档回答。

优点：
- 对短 query、口语 query 召回更好。
- 能补充语义信息。

问题：
- 假设答案可能编错，导致检索偏。
- 对强事实/合规场景风险较高。
- 成本增加。
- 需要后续真实证据校验。

### 1. HyDE 为什么能提升召回

很多用户 query 很短，信息量不足。比如只问"多久到账"，直接 embedding 可能召回很泛。HyDE 先生成一个假设答案，等于把隐含语义补全，再用这个更丰富的文本去检索。

它适合短 query、口语化问题、用户描述和文档表述差异大的场景。

### 2. HyDE 的风险

HyDE 最大风险是"假设答案带偏检索"。如果模型先猜错原因，检索就会朝错误方向走。尤其在强事实、合规、交易状态场景，不能让模型先编一个原因再查。

所以 HyDE 更适合开放知识问答，不适合直接做交易归因。

### 3. 面试总结

HyDE 是召回增强手段，不是答案生成手段。它生成的假设内容只能帮助检索，最终答案必须回到真实文档。

### 4. 结合跨境支付项目怎么说（面试口径）

> 支付场景我会谨慎用 HyDE。比如用户问"这笔款为什么被退回"，如果模型先假设是 AML 拦截，但真实原因是银行账号格式错误，检索就会被带偏。
>
> 所以我更倾向把 HyDE 用在宽泛知识问答，比如解释某类支付方式的通用规则；但不直接用于强事实归因。涉及交易状态、通道回执、合规限制时，必须先查真实系统和文档，再让模型总结。
>
> 我的原则是：HyDE 可以补语义，但不能补事实。支付场景事实必须来自订单系统、通道回执或合规文档。

**钩子**：HyDE 能补召回，但强事实支付场景要防止假设答案带偏。
