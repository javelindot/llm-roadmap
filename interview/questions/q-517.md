# 如何让多个agent协同工作的？举个具体的协同机制例子。

## 答案

多 Agent 协同的关键不是“让几个模型互相聊天”，而是明确角色、共享状态、通信协议、冲突处理和停止条件。

### 1. 常见协作模式

- **Supervisor-Worker**：主 Agent 分解任务并汇总结果。
- **Pipeline**：上一个 Agent 的结构化输出作为下一个输入。
- **Planner-Executor**：规划与执行分离。
- **Generator-Critic**：一个生成，一个审查并要求修订。
- **Blackboard**：多个 Agent 读写同一任务状态和证据区。
- **Debate / Voting**：多个候选独立推理，再由裁判选择。

### 2. 具体例子：复杂故障排查

```text
Supervisor
  ├─ Log Agent：分析日志和错误时间线
  ├─ Code Agent：定位相关代码与变更
  ├─ Metrics Agent：检查延迟、错误率和资源
  └─ Reviewer Agent：核对证据和修复风险
```

流程：

1. Supervisor 生成唯一 Task ID 和子任务清单。
2. 各 Worker 并行执行，只返回结构化结果：结论、证据、置信度、未解决问题。
3. 结果写入共享状态，而不是依赖聊天记录传递。
4. Supervisor 根据证据决定继续调查、生成修复方案或请求人工确认。
5. Reviewer 检查结论是否有证据、是否遗漏回滚方案。

### 3. 工程保障

- JSON Schema 定义消息协议。
- Task ID、幂等键防止重复执行。
- 每个 Agent 使用最小工具权限。
- 超时、重试、预算和最大轮数。
- 冲突时基于证据或确定性规则裁决。
- 全链路 Trace，记录谁在何时做了什么判断。

### 4. 何时不该用多 Agent

任务简单、子任务高度耦合或一次工具调用就能完成时，多 Agent 只会增加延迟、成本和不确定性。

**面试总结**：多 Agent 的价值来自可并行的专业分工；可靠性来自显式状态机和协议，而不是自由对话。
