# 部署用 Docker 还是 K8s？

## 答案

> 这不是二选一——**Docker 是打包格式、K8s 是编排平台**，生产里两个都用：**Docker 打镜像 + K8s 跑起来**。
>
> **两者关系**：
>
> | 维度 | Docker | K8s |
> |------|--------|-----|
> | 角色 | 容器运行时 + 镜像 | 容器编排平台 |
> | 解决问题 | "环境一致性" | "多容器调度、伸缩、自愈" |
> | 范围 | 单机 | 集群 |
> | 学习成本 | 低 | 高 |
>
> **何时只用 Docker（不上 K8s）**：
> - 单机 / 个人项目
> - POC、demo
> - 边缘部署（资源受限）
> - 简单稳定的小服务
> - 团队不熟 K8s
>
> 工具：`docker-compose` 管多个容器、`docker run` 直接跑。
>
> **何时需要 K8s**：
> - **多副本 + 自动伸缩**（HPA）
> - **滚动发布、灰度**（Deployment、Argo Rollouts）
> - **自愈**（容器挂自动重启 + 调度）
> - **跨节点资源调度**
> - **服务发现 + 负载均衡**
> - **多租户隔离**
> - **GPU 资源调度**（GPU Operator）
>
> **我们跨境支付的部署**：
>
> **生产线**：**全 K8s**
> - 阿里云 ACK + 自建私有云
> - LLM 服务：vLLM Deployment + HPA + GPU 调度
> - RAG / Agent 服务：FastAPI Deployment + ingress
> - 中间件：Milvus / ES / Redis Operator
> - CI/CD：GitLab CI → 镜像构建 → ArgoCD → K8s
> - 监控：Prometheus + Grafana + Loki + OpenTelemetry
>
> **开发/测试**：**Docker / docker-compose**
> - 算法工程师本地用 Docker 跑模型
> - 集成测试用 docker-compose 起 Milvus + ES + LLM mock
>
> **POC / 边缘**：**纯 Docker**
> - 客户现场 POC 部署
> - 资源受限场景（GPU 单卡）
>
> **K8s 生产关键配置（LLM 场景）**：
>
> **1. GPU 资源声明**
> ```yaml
> resources:
>   limits:
>     nvidia.com/gpu: 2  # 单 pod 2 张卡
>     memory: 32Gi
>   requests:
>     memory: 24Gi
> ```
>
> **2. HPA 伸缩**
> ```yaml
> spec:
>   minReplicas: 2
>   maxReplicas: 10
>   metrics:
>     - type: Resource
>       resource:
>         name: cpu
>         target: { type: Utilization, averageUtilization: 70 }
>     - type: External
>       external:
>         metric: { name: gpu_utilization }
>         target: { type: AverageValue, averageValue: 80 }
> ```
>
> **3. PodDisruptionBudget**（防滚动更新影响业务）
> ```yaml
> spec:
>   minAvailable: 50%
> ```
>
> **4. 健康检查**
> ```yaml
> livenessProbe:
>   httpGet: { path: /health, port: 8000 }
>   initialDelaySeconds: 60  # LLM 加载慢，要长一点
> readinessProbe:
>   httpGet: { path: /ready, port: 8000 }
> ```
>
> **5. ConfigMap + Secret**
> 模型路径、API key、向量库地址分开管理
>
> **几个 LLM 部署的特殊问题**：
>
> 1. **模型加载慢**——容器启动几分钟，要长 initialDelaySeconds
> 2. **GPU 调度**——需要 NVIDIA device plugin + GPU Operator
> 3. **镜像大**——LLM 镜像 20G+，要做分层、cache 优化
> 4. **存储**——模型权重通常挂 PVC（NFS 或 OSS）
> 5. **滚动发布**——LLM 服务停一下成本高，要 PDB 保证可用
>
> **不推荐的做法**：
> - 用 Docker Swarm（已死）
> - 直接 docker run 跑生产（无自愈、无监控、无伸缩）
> - K8s 不做监控（生产事故先驱）
>
> 一句话：**Docker 是打包标准，K8s 是生产编排**——LLM 生产环境必须 K8s，开发用 Docker；不是替代关系是组合关系。
