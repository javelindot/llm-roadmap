# 客服项目介绍一下？

## 答案

**一句话**：客服项目可以讲成一个面向业务人员和终端用户的智能问答系统，核心是 RAG 知识库、实时业务查询、Agent 编排和人工客服协同。

### 1. 项目背景

客服系统要解决的不是简单聊天，而是让用户更快得到准确答案，让人工客服减少重复查询。传统客服的问题是：

- FAQ 命中率低。
- 业务规则分散在多个文档和系统。
- 订单、通道、风控状态需要人工多后台查询。
- 回答口径不统一。
- 新人客服培训成本高。

所以项目目标是把知识问答和实时业务数据结合起来。

### 2. 系统能力

主要能力包括：

- 知识库问答：基于 FAQ、SOP、政策制度、操作手册做 RAG。
- 订单查询：通过工具查询交易状态、支付状态、退款状态。
- 异常解释：结合错误码、通道回执、风控状态解释原因。
- 工单辅助：给客服生成建议回复和处理步骤。
- 转人工：置信度低、高风险、投诉类问题转人工。
- 运营闭环：收集用户反馈和人工改写内容，持续优化知识库。

### 3. 技术架构

可以这样介绍：

- 前端接入 Web、客服工作台或 IM 渠道。
- 后端有会话管理、意图识别、Agent 编排。
- RAG 模块负责知识检索和引用。
- Tool 模块对接订单、风控、通道、工单系统。
- 安全模块做权限、脱敏、敏感内容过滤。
- 评测模块做准确率、转人工率、满意度统计。

### 4. 结合跨境支付项目怎么说（面试口径）

> 我做的客服项目放在跨境支付场景里，核心是解决“支付问题解释成本高”的问题。比如商户问“这笔付款为什么失败”，以前客服要去订单系统看状态、去通道系统看回执、去风控系统看拦截原因，再翻知识库找解释口径。智能客服可以把这些步骤串起来。
>
> 技术上，制度、FAQ、错误码、国家通道规则走 RAG；实时交易状态、通道回执、风控结果走 Function Call 工具查询；Agent 负责判断下一步先查什么。最终回答会给出原因、当前状态、预计处理时间和下一步建议。
>
> 我们对高风险问题做了边界控制，比如风控拦截只解释大类原因，不暴露具体规则；资金操作不由模型直接执行，必须转人工或走审批。

**钩子**：可以引到“RAG + Function Call、错误码解释、人工客服协同、风控信息脱敏”。
