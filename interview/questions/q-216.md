# 项目中的上下文信息如何处理与管理？

## 答案

**一句话**：上下文管理的核心是把“模型当前必须知道的信息”留下，把历史噪音、工具大结果和无关内容压缩或外置，保证模型既能连续对话，又不被长上下文拖垮。

### 1. 上下文包括什么

项目里的上下文一般分几类：

- 用户当前问题和对话历史。
- 用户身份、角色、权限和业务线。
- 任务状态，比如当前走到哪一步、调用过哪些工具。
- RAG 检索结果。
- 工具调用结果。
- 业务实体信息，比如订单、商户、交易、工单。
- 系统约束，比如不能泄露风控规则、不能执行资金操作。

这些信息不应该都无脑塞给模型，要按任务需要选择。

### 2. 管理方式

常见做法有：

- **短期上下文**：保留最近几轮对话和当前任务信息。
- **摘要压缩**：长对话压缩成关键事实、用户目标、已完成步骤。
- **外部状态存储**：把交易信息、工具结果、流程状态存 Redis/DB，只在需要时取。
- **RAG 动态注入**：根据当前问题检索知识，不把整个知识库放进上下文。
- **上下文隔离**：不同 SubAgent 只拿自己需要的信息。
- **敏感信息脱敏**：银行卡、证件号、完整风控规则不能直接暴露给模型。

### 3. 工程注意点

上下文管理要平衡三件事：准确性、成本和安全。

保留太少，模型容易答非所问；保留太多，成本高、延迟高，还容易被旧信息干扰。更好的做法是把上下文分层：会话摘要在模型侧，业务事实在数据库侧，知识内容通过 RAG 动态取，敏感数据通过工具按权限读取。

### 4. 结合跨境支付项目怎么说（面试口径）

> 在跨境支付客服场景里，上下文不能简单等同于聊天记录。比如用户连续问“这笔付款为什么失败”“那我怎么处理”“会不会影响下一笔”，模型必须知道当前讨论的是哪笔交易、交易状态是什么、风控命中了什么、通道返回了什么，但不一定需要保留用户前面所有闲聊内容。
>
> 我的做法会是：会话里保留用户目标和关键摘要；订单、商户、风控、通道回执这些结构化数据存在业务系统里，通过工具按需读取；知识库内容通过 RAG 注入；敏感字段做脱敏。比如只给模型展示银行卡后四位、风险原因的大类，不给完整规则和评分细节。
>
> 如果用了 SubAgent，我还会做上下文隔离。风控 SubAgent 只看风险相关特征，通道 SubAgent 只看通道状态和回执，客服主 Agent 最后汇总成用户能理解的说明。这样既降低 token，也减少敏感信息扩散。

**钩子**：可以引到“上下文分层、敏感信息脱敏、SubAgent 隔离、RAG 动态注入”。
