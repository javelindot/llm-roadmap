# DPO训练时，chosen和rejected的reward一起下降的可能

## 答案

**一句话**：DPO 里 chosen 和 rejected 的隐式 reward 一起下降不一定代表训练失败，关键要看二者的差值是否变大，也就是模型是否更偏向 chosen。

### 1. 先看 DPO 的隐式 reward

DPO 里常看的 reward 通常是相对 reference model 的隐式 reward：

```text
r(x, y) = beta * (log pi_theta(y|x) - log pi_ref(y|x))
```

它不是一个绝对奖励，而是看当前模型相对参考模型对某个回答的偏移。

### 2. 为什么会一起下降
- **只要差值变大就可能正常**：DPO 优化的是 chosen 相对 rejected 更好，不要求两个 reward 绝对都上升。
- **reference 约束影响**：如果新模型整体比 reference 更保守，两个隐式 reward 都可能下降。
- **长度影响**：长回答的 token logprob 累加更容易变低，导致 reward 看起来下降。
- **数据噪声**：chosen/rejected 标注质量差，模型学到混乱偏好。
- **beta 或学习率不合适**：beta 太大、学习率太高都可能导致训练不稳。
- **概率质量转移**：模型可能降低了两个候选的绝对概率，但更大幅度降低 rejected。

### 3. 怎么判断和处理

重点看：
- `chosen_reward - rejected_reward` 是否变大。
- preference accuracy 是否提升。
- DPO loss 是否下降。
- chosen logprob 和 rejected logprob 的相对差。
- 离线人工评估是否变好。

处理方式：
- 清洗偏好数据，去掉标注不一致样本。
- 做长度归一化或控制回答长度。
- 调小 learning rate 或 beta。
- 加 SFT loss 混合，防止模型漂移太远。
- 增加稳定评估集，不只看训练曲线。

## 结合项目谈（口语版 · 面试现场约 50 秒）

> 我会先判断这是不是假问题。DPO 优化的不是 chosen reward 必须绝对上升，而是 chosen 相比 rejected 的优势要变大。比如我们训练一个支付风控解释模型，chosen 是"给出风险原因并建议补材料"，rejected 是"直接拒绝商户且不给原因"。如果训练后两个隐式 reward 都下降，但 rejected 降得更多，模型实际还是更偏向 chosen，这可能是正常的。
>
> 但如果差值也没变大，或者 preference accuracy 不升，那就要排查数据和参数。支付场景里偏好数据很容易有噪声，比如有些人工审核偏保守，有些偏业务通过；还有回答长度问题，长的合规解释可能 logprob 更低，曲线看起来不漂亮。
>
> 我的处理会是先看 reward gap 和验证集，不只看单条 reward；然后清洗偏好对、控制长度、调小 beta 或学习率，必要时混一点 SFT，保证模型别为了偏好优化把原来的业务表达能力丢掉。
