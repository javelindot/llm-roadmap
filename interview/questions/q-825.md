# 10. 4.31 模型在SFT后会出现“复读机”情况该如何debug，以及出现的原因是什么？

## 答案

“复读机”可能是输出内部循环，也可能是重复 prompt、固定模板或训练语料。应先区分训练问题与解码问题。

### 1. 常见原因

- 数据中重复文档、模板回答或爬虫循环；
- chat template、BOS/EOS、角色 token 配置错误；
- label mask 错，把 system/user/padding 也作为目标；
- 截断后丢失 EOS，模型学不到停止；
- 学习率过大、epoch 太多、数据太小导致过拟合或遗忘；
- 同类样本连续排列，packing 边界处理错误；
- 推理时 temperature 过低、greedy、重复惩罚缺失；
- KV Cache、position id 或 attention mask 实现错误；
- LoRA scale/rank 过强导致行为偏移。

### 2. Debug 顺序

1. 用基座模型和同一推理配置对比；
2. 抽查 tokenized 样本、labels、mask、EOS 和 packing；
3. 统计 exact/near duplicate、n-gram 重复和模板占比；
4. 在训练样本、验证样本、全新样本分别复现；
5. 查看出现复读的 checkpoint、学习率和梯度范数；
6. 比较 greedy 与采样，关闭/开启 KV Cache；
7. 观察 next-token entropy 是否异常塌缩；
8. 回退 checkpoint，做单变量消融。

### 3. 修复

去重并增加数据多样性，修正模板与 EOS/mask，降低学习率和 epoch，混入通用 SFT 数据，减小 LoRA 强度。解码侧可设置 `repetition_penalty`、`no_repeat_ngram_size` 和合理温度，但这只是缓解，不能掩盖训练数据或实现错误。

### 跨境支付场景

客服重复回复可能导致用户多次提交操作。对退款或转账 Agent 还应设置最大步骤、幂等键和循环检测，不能只靠文本重复惩罚。

### 面试总结

先用基座对照判断训练还是解码，再检查数据重复、chat template、label mask、EOS、过拟合和 KV/position 实现。解码惩罚只能止症，根因通常在数据或训练管线。
