# 问腾讯IEG这边的工作，deepresearch的deep是如何体现的（介绍了DMA的tool调用，感觉可以再了解一下多少个tool，SFT的流程

## 答案

这道题涉及腾讯 IEG 的具体内部工作和 DMA 工具，公开信息不足，不能假定内部工具数量或训练细节。面试中应先基于对方介绍复述，再从通用 Deep Research 架构解释“deep”体现在哪里，并追问真实实现。

### 1. Deep 不等于搜索次数多

Deep Research 的“deep”通常体现在：

- 对复杂目标进行多层任务分解；
- 搜索、阅读、推理交错，而不是一次检索；
- 根据证据缺口动态生成下一轮 query；
- 比较多个来源并处理冲突；
- 调用代码、数据、文档等不同工具；
- 维护长时程状态和研究预算；
- 对关键结论做验证；
- 输出结构化报告与逐条引用。

流程可画成：

```text
目标理解
-> 研究计划
-> 子问题分解
-> 工具选择
-> 检索/阅读/分析
-> 证据评估
-> 发现缺口
-> 重规划
-> 交叉验证
-> 报告与引用
```

### 2. 工具数量不是核心指标

工具可以包括：

- Web Search；
- Browser/Reader；
- 内部知识库；
- PDF/文档解析；
- SQL/数据查询；
- Python/代码执行；
- 多媒体理解；
- Citation verifier；
- 业务 API。

工具越多，路由、权限、描述冲突和失败率也越高。应该问：

- 工具按原子能力还是领域封装；
- 哪些工具模型可自主选择；
- 最大调用深度和预算；
- 是否有并行子任务；
- 工具失败如何重试或降级；
- 如何评估 tool selection 与参数正确率。

### 3. DMA Tool Calling 可以怎么理解

若 DMA 是面试官介绍的内部 Agent/编排模块，建议现场确认缩写和边界，再追问：

```text
Planner 是否直接选择 DMA tools？
Tool schema 如何设计？
是否有检索、执行、验证三类工具？
轨迹如何记录？
权限和敏感数据如何隔离？
```

不要擅自给 DMA 扩写或编造内部架构。

### 4. SFT 数据构造

Deep Research SFT 数据不能只有最终报告，应包含可监督的研究轨迹：

```text
user goal
research plan
subtasks
tool call name
tool arguments
tool observation
evidence selection
replanning
final answer
citations
```

来源可以是：

- 专家人工轨迹；
- 高质量线上成功轨迹脱敏；
- 强模型生成后由 verifier 和人工筛选；
- 失败轨迹修正；
- 对比不同工具和计划的偏好数据。

### 5. 清洗和训练

- 删除不可复现和来源不支持的轨迹；
- 工具参数做 schema 校验；
- 过滤泄露敏感数据的样本；
- 按任务和来源去重；
- 对 tool result 做 loss mask；
- assistant 的计划、调用和回答 token 作为监督目标；
- 先学格式和基础工具使用，再加入长链路复杂任务；
- 后续可用可验证 reward、偏好优化或 RL 提升规划。

### 6. 评测

- 任务完成率；
- 工具选择和参数准确率；
- 证据 Recall/Precision；
- 引用支持率；
- 多源一致性；
- 研究深度和无效调用率；
- 长链路成功率；
- 延迟、Token 和工具成本；
- 注入攻击、权限和隐私安全。

### 7. 跨境支付场景

研究“某国家监管变化对商户收款的影响”时，需要查监管原文、内部政策、产品配置和历史案例，核对生效日期与适用主体，再形成影响和行动建议。最终规则修改仍应由合规与产品审批。

### 8. 面试总结

Deep Research 的深度来自多轮规划、证据驱动的重规划、工具协同和交叉验证，而不是简单增加工具数量。涉及腾讯内部 DMA 时应主动澄清定义；SFT 则要覆盖计划、工具调用、观察、重规划、引用和最终报告的完整轨迹。
