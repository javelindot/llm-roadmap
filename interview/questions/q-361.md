# RAG 优化？

## 答案

> RAG 优化是个**端到端工程问题**——查询、索引、检索、重排、生成、评测每一层都能优化。详见 q-321 准确率优化、q-123 索引质量、q-241 延迟分析；这里给个**综合总结**。
>
> **优化的三大方向**：
>
> 1. **准确率提升**（详见 q-321）
> 2. **延迟降低**（详见 q-241）
> 3. **成本下降**
>
> **准确率优化（最重要）**：
>
> ```
> 查询侧：
>   + Query 改写（多轮、扩展、HyDE）→ +10
>   + 意图分类 + 路由 → +5
>
> 索引侧：
>   + Embedding 业务精调 → +10-15
>   + 元数据富化（摘要、假设问题）→ +5
>   + 结构感知分块 → +5
>   + 多 Index → +5
>
> 检索侧：
>   + 混合检索（向量 + BM25）→ +8
>   + 元数据过滤 → +3
>
> 重排侧：
>   + Cross-Encoder Rerank → +10-15
>   + LLM Rerank（关键场景）→ +3-5
>
> 生成侧：
>   + 引用强约束 → 幻觉 -50%
>   + NLI 事实校验 → +3
>   + 拒答机制 → 已答准确率 +5
>
> 评测 + 飞轮：
>   + 黄金集 + RAGAS
>   + 失败 case 回流
>   + 月度迭代
> ```
>
> **延迟优化**（详见 q-241）：
>
> - 流式输出（首 token 800ms）
> - 检索并行（向量 + BM25 async）
> - vLLM 加速
> - Speculative Decoding
> - Prompt Cache
> - 路由跳过非必要步骤
>
> **成本优化**：
>
> - 业务精调小模型替代大模型
> - 量化（AWQ INT4 推理快 3x）
> - 路由分级（80% 走小模型）
> - prompt cache 减少 prefill
> - 答案缓存（高频 query）
> - 知识库定期清理过期
>
> **我们跨境支付的优化路径**：
>
> 基线 → 现在：
> - 召回率 62% → 92%
> - 端到端 accuracy 65% → 88%
> - p95 延迟 5s → 1.5s
> - 单 query 成本 0.5 元 → 0.01 元
>
> 关键优化（按收益排序）：
>
> 1. **Embedding 业务精调** —— 单点收益最大
> 2. **混合检索 + Rerank** —— 组合杠杆
> 3. **业务路由 + 多 Index** —— 大幅提精度
> 4. **vLLM 部署** —— 延迟和成本双降
> 5. **失败 case 飞轮** —— 持续提升
>
> **优化优先级判断**：
>
> 看当前瓶颈：
> - 召回失败多 → 查询侧 + 索引侧
> - 召回到但生成错 → 生成侧 + 评测
> - 延迟高 → vLLM + 流式 + 并行
> - 成本高 → 蒸馏 + 量化 + 路由
>
> **几个反直觉经验**：
>
> 1. **业务精调 embedding 比换大模型便宜得多**
>
> 2. **Rerank 比换 LLM 收益大**
>
> 3. **路由分级 ROI 极高**
>
> 4. **流式输出比换模型更能改 UX**
>
> 5. **失败 case 飞轮是长期复利**
>
> **不优化的反例**：
>
> - 没评测就调参（瞎搞）
> - 无脑加召回路（成本翻倍收益递减）
> - 追新方法不看 ROI（HyDE 在简单场景反而引入噪声）
> - 不分场景路由（所有 query 走最强 pipeline）
>
> 一句话：**RAG 优化是滚雪球**——查询、索引、检索、重排、生成、评测六层每层 3-15 个点，组合起来 30+ 个点的质变。
