# 解决tokens不够的问题应该怎么办

## 答案

“Token 不够”可能指上下文窗口放不下、输出预算不足、训练 token 不足或 API 配额不足。面试时应先确认是哪一种。

### 1. 输入超过上下文窗口

先做 token 预算：

```text
system + user + history + evidence + tool results
+ reserved output <= context window
```

可采用：

- 删除重复、寒暄和低价值历史；
- 保留系统约束、当前任务和关键实体；
- RAG 只召回相关证据；
- rerank、去重和 query-aware compression；
- rolling summary + 原文指针；
- 超长文档分块、层级 map-reduce；
- 工具结果结构化，只保留必要字段；
- 长任务状态放外部存储，按需读取；
- 最后才考虑更长上下文模型。

不能简单从头截断，因为 system policy 或早期关键事实可能丢失。

### 2. 输出 token 不够

- 提高 `max_new_tokens`，同时预留上下文空间；
- 要求先输出核心结论，再输出细节；
- 分页/分阶段生成；
- 代码和报告按章节生成后再校验合并；
- 使用 stop 条件避免无效展开；
- 检查是否因 EOS、长度限制或服务端上限提前终止。

长输出会增加 Decode 延迟、成本和中途失败概率。

### 3. 训练 token 不足

- 提高真实数据质量和覆盖；
- 去重，避免重复 token 浪费；
- 使用公开许可数据、专家数据和可验证合成数据；
- curriculum 与数据混合配比；
- 继续预训练时监控 data scaling；
- 小数据优先 PEFT/RAG，而不是从零训练；
- 合成数据要防模型坍缩和错误循环放大。

### 4. API 配额或成本不足

- Prompt/Prefix cache；
- 模型路由，小任务走小模型；
- 批处理和异步队列；
- 限制无效重试和 Agent 最大步骤；
- 统计单次成功任务成本，而不只是每次调用；
- 申请配额或私有化部署前先做容量评估。

### 5. 跨境支付场景

长工单应保留订单号、金额币种、工具事实、待办和权限，旧对话摘要后可检索。实时订单状态不能为了省 token 被摘要成长期事实，仍应重新查询权威系统。

### 6. 面试总结

Token 不够不是单一问题。上下文不足用检索、压缩和外部状态；输出不足用预算与分段；训练数据不足用高质量数据和 PEFT；配额不足用缓存、路由和流程优化。
