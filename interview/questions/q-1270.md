# 生成式推荐与传统DLRM有什么区别？

## 答案

传统 DLRM 把推荐主要建模为“给定 User、Item 和 Context 特征，预测交互概率”；生成式推荐把用户行为与候选表示为 token/ID 序列，通过自回归或序列建模直接生成下一物品、Semantic ID 或推荐序列。

### 1. DLRM

经典 DLRM 包含：

- 稠密特征 MLP；
- 大规模类别特征 Embedding Table；
- 特征交互；
- Top MLP 输出 CTR/CVR 等概率。

```text
score = P(action | user, item, context)
```

它主要用于排序或逐候选打分。大规模部署中常将 Embedding Table 做模型并行，MLP 做数据并行。

### 2. 生成式推荐

一种代表路线是把每个物品量化为 Semantic ID，然后训练 Transformer：

```text
P(item_id_tokens | user_history, context)
```

模型自回归生成候选 ID，不必对所有 Item 逐一打分。另一类大型序列推荐模型直接对长用户行为序列做 next-item prediction。

“生成式”不是让 LLM 生成一段推荐文案，而是把推荐目标本身转成序列生成或生成式检索。

### 3. 主要差异

| 维度 | DLRM | 生成式推荐 |
|---|---|---|
| 目标 | 逐候选概率预测 | 生成候选 ID/序列 |
| 物品表示 | 独立 ID Embedding + 特征 | Token/Semantic ID |
| 用户历史 | 聚合或专门序列模块 | 原生序列上下文 |
| 候选生成 | 另有召回系统 | 可直接生成召回候选 |
| 新物品 | 依赖特征与训练 | Semantic ID 有潜力迁移，但仍需编码 |
| 推理 | 大量候选可并行打分 | 自回归存在串行解码 |
| 约束 | 候选集天然有效 | 需约束无效 ID 与重复 |

### 4. 优势与挑战

生成式推荐的潜在优势：

- 统一召回与序列建模；
- 利用长行为历史；
- Semantic ID 表达物品语义与层次；
- 多任务可统一成 token 预测。

挑战：

- ID 量化与码本碰撞；
- 新增、删除物品后的 ID 稳定性；
- 无效/重复候选和约束解码；
- 自回归延迟；
- 流行度偏差；
- 线上增量和可解释性；
- 与精排、库存和业务规则衔接。

### 5. 不是二选一

可以使用生成式模型召回候选，再由 DLRM/精排模型融合实时特征打分；也可以让 DLRM 系统保留主链路，生成式召回作为新增通道。应比较 Recall@K、NDCG、覆盖、长尾、冷启动、P99 和成本。

跨境支付若采用类似方法做产品候选生成，仍需在精排/规则层加入国家、币种、牌照、限额、实时成本和风险约束。生成 ID 不能直接决定资金通道。
