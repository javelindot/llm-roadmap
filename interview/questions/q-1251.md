# 比较Match、DeepQA和文心一言在文本相关性及问答效果提升方面的表现

## 答案

`Match` 和 `DeepQA` 可能是项目内部模块名，不同团队的含义并不统一；如果没有给出模型版本、训练数据和候选集，不能直接断言谁的准确率更高。更稳妥的比较是把它们还原为三类问答范式：

1. Match：文本匹配或 FAQ 相似度；
2. DeepQA：检索、阅读理解与答案抽取/排序流水线；
3. 文心一言类 LLM：生成式问答，可结合 RAG 和工具。

### 1. Match 类方法

输入通常是 Query 与候选问题/文档，输出相关性分数。可以是 BM25、双塔 embedding、Cross-Encoder 或它们的组合。

**优势**

- 延迟低、吞吐高；
- 排序分数和候选来源可追踪；
- FAQ 高覆盖场景稳定；
- 不会自由生成候选库之外的事实。

**局限**

- 只能返回或改写已有答案；
- 多轮、复杂推理和答案融合能力弱；
- 效果依赖候选库覆盖与负样本质量。

### 2. DeepQA 类流水线

通常包含 Query 理解、文档召回、段落排序、阅读理解、答案打分和置信度。与纯 Match 相比，它能在文档中定位答案 span 或组合证据。

**优势**

- 比 FAQ Match 更适合开放文档问答；
- 可以输出证据位置和置信度；
- 抽取式答案事实边界相对可控。

**局限**

- 多模块误差会级联；
- 跨文档、多跳和自然语言组织能力有限；
- 每个模块都需要单独训练与维护。

### 3. 文心一言类生成模型

LLM 可以理解多轮问题、综合多个证据、生成自然答案并调用工具。结合 RAG 后，能够覆盖 Match 和抽取式 QA 难处理的复杂表达。

**优势**

- 多轮理解、摘要、推理和答案组织强；
- 能统一多种任务与输出格式；
- 可结合检索、知识图谱和业务工具。

**局限**

- 可能幻觉或忽略证据；
- 延迟和成本较高；
- 同一输入不一定完全确定；
- 权限、实时事实和高风险动作仍需外部系统控制。

### 4. 公平比较方式

使用同一知识版本、同一候选集合和冻结测试集，按问题类型分桶：

- FAQ 同义问；
- 精确实体与错误码；
- 文档抽取；
- 多文档综合；
- 多轮指代；
- 无答案与冲突证据；
- 实时工具问题；
- 越权和敏感请求。

指标至少包括：

- Match/检索：Recall@K、MRR、NDCG；
- 抽取 QA：Exact Match、Token F1、证据命中；
- 生成 QA：正确性、faithfulness、完整性、引用；
- 系统：P95、成本、稳定性、拒答和安全。

### 5. 推荐组合

这三类不是互斥关系：

```text
Match/BM25/Dense 负责高效召回
→ DeepQA/Reranker 负责证据定位和排序
→ LLM 负责多轮理解、融合与表达
→ Rule/Tool 负责实时事实、权限和执行
```

跨境支付中，高频标准 FAQ 可由 Match 直接返回；错误码和规则文档可走检索与抽取；“为什么这笔付款没到账”需要订单工具、规则检索和 LLM 综合。应按任务路由，而不是让生成模型替换所有已有模块。
