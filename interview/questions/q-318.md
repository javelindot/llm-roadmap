# 为什么选择Chroma？

## 答案

**一句话**：选择 Chroma 通常是因为它轻量、上手快、适合 PoC 和中小规模 RAG 验证，但生产大规模场景要评估性能、权限和运维要求。

### 1. Chroma 的优点

Chroma 常见优势有：

- 接入简单，开发成本低。
- 本地部署方便。
- 和 LangChain 等生态集成好。
- 适合快速验证 embedding、chunk、检索效果。
- 对中小规模知识库比较友好。

所以在 PoC 或项目初期，用 Chroma 能很快把 RAG 链路跑起来。

### 2. 局限性

如果进入生产大规模场景，要考虑：

- 数据量很大时性能和扩展性。
- 多租户权限隔离。
- 高可用和备份恢复。
- 监控和运维能力。
- 混合检索能力是否满足要求。

如果需要亿级数据、复杂过滤、关键词检索和企业级运维，可能会评估 ES、Milvus、Vespa、pgvector 等。

### 3. 选型逻辑

我的选择逻辑是：

- PoC：优先 Chroma，快速验证效果。
- 中小规模内部知识库：Chroma 可以继续使用。
- 大规模生产：根据数据量、QPS、过滤需求和团队运维能力重新评估。

### 4. 结合跨境支付项目怎么说（面试口径）

> 在跨境支付客服项目初期，我会选择 Chroma 做 RAG PoC，因为它接入快，能快速验证文档切分、embedding、topK、rerank 这些策略是否有效。比如先把 FAQ、错误码、通道规则导入，跑一批真实工单看召回效果。
>
> 但我不会说 Chroma 一定适合所有生产场景。如果后续知识库数据量很大，或者需要按商户、国家、通道做复杂权限过滤，还需要关键词和向量混合检索，那我会评估 ES 或 Milvus。
>
> 所以我的回答是：选择 Chroma 是为了快速验证和降低初期复杂度，生产选型要看规模和权限要求。
