# 有做什么部署优化吗

## 答案

部署优化不能只说量化或换推理框架，而要先确定业务 SLO，再沿着“请求入口、检索与工具、模型推理、资源调度、降级容灾”逐层定位瓶颈。以跨境支付商户助手为例，我会同时关注答案正确性、资金与合规边界、P95/P99、吞吐和单次成功任务成本。

### 1. 先建立可比较的基线

将请求按场景分桶：

- 短 FAQ；
- 长文档 RAG；
- 订单、账户、汇率等工具调用；
- 多轮问题；
- 长输入、短输出和短输入、长输出。

每类记录：

- TTFT、TPOT、端到端 P50/P95/P99；
- 输入输出 token、并发、QPS；
- GPU 利用率、显存、KV Cache 命中率；
- 检索、rerank、工具、排队和模型耗时；
- 超时率、降级率、正确率和单位成本。

这样才能判断瓶颈在 prefill、decode、检索、外部接口还是调度，而不是凭 GPU 利用率猜测。

### 2. 模型和推理引擎优化

- 按任务复杂度路由模型：意图识别、FAQ 用小模型，复杂规则解释和多文档推理走强模型；
- 在业务黄金集上比较 BF16、FP8、INT8、AWQ/GPTQ，重点回归金额、币种、状态码、结构化输出和拒答；
- 使用 continuous batching、Paged KV Cache、FlashAttention、prefix cache、chunked prefill；
- 对稳定系统提示和公共政策前缀使用 prefix cache；
- 长短请求分队列，避免长 prefill 阻塞短请求；
- 限制异常上下文与最大输出，设置 admission control，避免高并发 OOM；
- 对固定 shape 或稳定 decode 路径评估 CUDA Graph，对高并发短输出评估 speculative decoding。

这些能力是否有效取决于模型结构、硬件、batch 和长度分布，必须用线上相近流量压测。

### 3. 优化 RAG 和工具链路

跨境支付问答的总延迟经常不只在 LLM：

- BM25 与 Dense 并行召回，先做租户、国家、产品和有效期过滤；
- 控制初召回数量，只对高价值候选 rerank；
- 缓存公共知识查询，缓存键包含租户、权限、知识版本和语言；
- 订单、账户、汇率等独立工具并行调用，并给每个工具设置 deadline；
- 将“业务无数据”和“接口超时”分开处理；
- 对高频确定性问题使用模板或小模型，不必进入完整 Agent 链路。

涉及余额、交易状态、实时汇率等动态事实时，以授权工具结果为准，不缓存成长期答案。

### 4. 服务稳定性和弹性

- 模型预热、健康检查、异步流式响应；
- 限流、超时、取消、熔断和背压；
- 多副本负载均衡与自动扩缩容；
- 主模型不可用时降级到小模型、静态 FAQ 或人工；
- Prompt、模型、向量索引和 Adapter 全部版本化；
- 灰度发布，保留一键回滚；
- 用 trace_id 串起网关、检索、工具和模型日志。

支付场景还要做字段脱敏、租户隔离、最小权限、审计留痕，不能为了缓存命中率跨商户复用敏感上下文。

### 5. 验收方式

每次只改变一个主要变量，比较：

- 业务正确率、事实一致性、工具成功率；
- 高风险字段错误率和安全拒答；
- P95/P99、吞吐、峰值显存；
- 单个成功任务成本；
- 不同国家、币种、语言和上下文长度分桶结果。

最终成果应描述为“在哪类负载下，哪一层是瓶颈，采用了什么优化，质量守护指标是否持平”。没有压测记录时不报提升比例，但要能给出压测矩阵、上线门槛和回滚条件。
