# 为什么不用微调

## 答案

不使用微调通常不是因为微调无效，而是当前问题更适合通过 Prompt、RAG、工具和规则解决。应先判断缺的是 **知识、行为、格式还是基础能力**，再决定是否修改模型参数。

### 1. 知识更新快，不适合写入参数

跨境支付的费率、支持国家、通道状态、KYC/KYB 材料和合规规则会持续变化。微调后的知识：

- 无法保证准确记住；
- 很难按生效时间和权限检索；
- 更新需要重新训练和发布；
- 不能提供可靠引用；
- 新旧规则容易混淆。

这类信息应放在带版本、有效期和 ACL 的知识库或业务 API 中。RAG 提供规则证据，工具返回订单、余额、报价和通道实时状态。

### 2. 数据量和质量不足

少量历史客服对话往往含有：

- 人工口径不一致；
- 过期规则；
- 用户与账户敏感信息；
- 处理结果与原因错配；
- 同一模板大量重复。

直接微调会放大数据噪声。若没有合法授权、脱敏、去重、版本治理和冻结评测集，先做微调很可能比 Prompt/RAG 基线更差。

### 3. 缺陷不一定来自模型

很多问题其实位于其他模块：

- 事实错误：检索或业务工具缺失；
- 错用国家规则：元数据过滤错误；
- JSON 不稳定：Schema、解码约束或重试不足；
- 工具调用失败：接口定义和权限设计错误；
- 长文档答非所问：切块、召回或重排问题；
- 资金动作风险：必须由确定性权限和状态机控制。

微调无法修复错误数据源、越权接口或过期索引，也不应承担业务状态机职责。

### 4. 成本和运维复杂度不划算

微调会增加数据生产、训练、评测、模型仓库、灰度、回滚和持续回归成本。全量 SFT 还涉及多卡训练和大 checkpoint；LoRA 虽然便宜，也要管理 base revision、adapter、tokenizer 和推理兼容性。

如果 Prompt + RAG + Tool 已满足质量、延迟和成本目标，微调带来的边际收益可能不足以覆盖长期维护。

### 5. 可能产生能力与安全回归

低质量或分布过窄的 SFT/DPO 可能导致：

- 通用能力下降和灾难性遗忘；
- 过度模板化或过度拒答；
- 多语言能力退化；
- 学到错误的人工操作捷径；
- 安全边界和工具选择发生漂移。

因此微调前要有 SFT-only、旧模型和 Prompt/RAG 基线，微调后按语言、国家、产品和风险分桶回归。

### 6. 什么时候应该微调

以下目标更适合微调：

- 稳定的领域术语和输出风格；
- 大量重复任务中，Prompt 很长且成本高；
- 工具调用格式、分类或抽取错误具有稳定模式；
- 已有足量、高质量、合规的数据；
- LoRA/SFT/DPO 能在冻结测试集上证明增益；
- 基础模型能力不足，而不是知识源缺失。

决策顺序可以是：

```text
Prompt baseline
→ RAG / Tool / 结构化约束
→ 小规模 LoRA-SFT
→ 偏好优化
→ 确有必要时全量微调
```

每一步都以业务正确率、安全、延迟和成本为门槛。对跨境支付而言，微调主要塑造模型行为；动态事实、资金权限和审计证据仍留在模型外部。
