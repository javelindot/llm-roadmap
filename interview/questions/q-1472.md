# 哈希表的底层实现

## 答案

哈希表通过哈希函数将 key 映射到数组桶位置，目标是在平均 O(1) 时间完成插入、查询和删除。

### 1. 核心结构

```text
key
-> hash(key)
-> index
-> bucket
-> compare key
```

桶数组容量为 `m` 时：

```text
index = hash % m
```

若容量是 2 的幂，可使用：

```text
index = hash & (m - 1)
```

前提是 hash 扰动和低位分布合理。

### 2. 冲突处理

不同 key 可能映射到同一桶。

**Separate Chaining**

每个桶保存链表、动态数组或树。Java HashMap 使用数组+链表，并在特定条件下树化为红黑树。

**Open Addressing**

所有元素都在数组中，冲突后探测其他位置：

- 线性探测；
- 二次探测；
- 双重哈希；
- Robin Hood Hashing。

### 3. 查询与删除

查询先定位桶，再比较完整 key，不能只比较 hash。

开放寻址删除通常设置 tombstone，不能直接清空位置，否则会截断后续探测链。

### 4. 扩容

负载因子：

```text
load_factor = size / capacity
```

超过阈值后分配更大数组并 rehash/搬迁元素。扩容一般是 O(n)，但摊销后插入平均 O(1)。

部分实现使用 incremental rehash，分多次请求迁移，降低单次延迟尖峰。

### 5. 哈希函数要求

- 快；
- 均匀；
- 相同 key 稳定；
- 低碰撞。

安全场景还要防 Hash DoS，可使用随机种子或抗碰撞策略。密码存储中的 cryptographic hash 与容器哈希用途不同。

### 6. 复杂度

平均：

```text
insert/get/delete = O(1)
```

最坏可能 O(n)，树化桶可降到 O(logn)。实际性能也受 cache locality、负载因子和 key 比较成本影响。

### 7. 并发

普通哈希表通常非线程安全。可使用锁分段、CAS、ConcurrentHashMap 或 immutable snapshot。复合操作需原子 API，不能简单 `contains` 后 `put`。

### 8. 跨境支付场景

哈希表可做请求内缓存和去重索引，但订单幂等和账务不能只依赖进程内 Map，应使用数据库唯一键或一致性存储。

### 9. 面试总结

哈希表是“数组定位 + 冲突处理 + 负载因子扩容”。平均 O(1) 来自均匀哈希和摊销，回答时还应讲清链地址/开放寻址、删除 tombstone、扩容与并发。
