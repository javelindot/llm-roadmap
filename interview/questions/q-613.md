# 8.beamsearch 怎么做的

## 答案

Beam Search 在每一步保留累计得分最高的 $B$ 条候选序列，而不是像贪心搜索只保留一条。

### 算法流程

假设 beam size 为 $B$：

1. 用起始 token 初始化一个候选，累计对数概率为 0。
2. 对每个未结束候选运行模型，得到下一个 token 的 log probability。
3. 将当前累计分数与各 token 的 log probability 相加，形成扩展候选。
4. 从所有扩展候选中选出分数最高的 $B$ 条。
5. 生成 EOS 的候选放入完成集合，其他候选继续扩展。
6. 达到最大长度，或已经有足够完成序列且未完成候选不可能超过当前最优结果时停止。

为避免模型偏爱短序列，通常使用长度惩罚，例如：

$$
score(y)=\frac{\log P(y\mid x)}{length(y)^\alpha}
$$

实现时不必真的枚举完整词表再排序，可以对形状为 `B × V` 的分数取 top-k。还要正确维护每条候选对应的 KV Cache、父 beam 索引、EOS 状态和重复惩罚。

### 优缺点

优点是比贪心搜索探索更多路径，在机器翻译、语音识别等目标较确定的任务中常能提升序列概率与准确率。缺点是计算和显存大致随 $B$ 增长，候选之间容易相似，而且最大似然分数不一定等于人类偏好。

开放式对话和创作通常更常用 temperature、top-k、top-p 等采样，因为普通 Beam Search 容易产生安全但重复、缺乏多样性的文本。若仍需多样性，可使用 diverse beam search、词级约束或 n-gram 重复限制。
