# Reflection 机制里，如何判断一个 Agent 的失败是由知识缺失还是工具调用错误引起的？

## 答案

不能只让同一个 LLM 自我反思后主观归因，应依靠结构化 trace、分阶段指标和可重放实验判断。

### 1. 定义错误分类

- 知识缺失：检索库不存在所需事实，或正确文档未召回；
- 知识使用错误：证据存在且召回，但模型忽略或误读；
- 工具选择错误：应调用 A 却调用 B，或不应调用；
- 参数错误：schema、实体、金额或时间不合法；
- 工具执行错误：超时、权限、下游 5xx；
- 规划错误：调用都成功，但顺序或停止条件错误。

### 2. 诊断信号

为每步记录计划、检索 query、候选文档、引用、工具名、参数、返回码、状态变化和最终验证结果。

- oracle 文档加入上下文后成功，说明偏检索/知识问题；
- oracle 工具调用成功而原轨迹失败，说明工具选择或参数问题；
- 原工具返回成功且包含答案，最终仍错，说明推理/证据使用问题；
- 知识库搜索无结果，但外部权威源存在答案，属于知识覆盖；
- 工具返回权限或超时，属于系统执行，不应归因模型知识。

### 3. Counterfactual Replay

固定其余变量，只替换一个环节重放：换成正确文档、正确参数或 mock 成功结果。比较任务是否恢复，可定位因果贡献。再用规则分类器或独立 Judge 汇总，低置信样本人工审核。

### 跨境支付场景

例如退款失败，应区分“模型不知道退款规则”“选错订单”“退款接口参数错误”“订单状态不允许退款”。资金系统返回的状态码优先于模型反思，所有重放在沙箱或只读环境完成。

### 面试总结

Reflection 归因要依赖可观测 trace 和单变量重放：给 oracle 知识能否恢复、给 oracle 工具调用能否恢复。把知识覆盖、检索、工具选择、执行和推理错误分开，才能选择正确修复手段。
