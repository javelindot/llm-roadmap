# 你了解你在美团的BI系统的整个架构吗？ 原始数据从哪里来的？全部导入数据库吗？

## 答案

BI 系统不是把所有原始数据复制到一个业务数据库。合理架构是数据源、采集与流处理、湖仓/数仓、语义指标层、查询服务和 BI 应用分层；不同数据按实时性、结构和合规要求走不同链路。

### 1. 跨境支付 BI 的数据来源

**业务 OLTP**

- 收款、付款、退款、拒付订单；
- 商户、账户、余额和产品配置；
- 清分、清算、账务分录；
- 换汇报价与成交；
- 通道路由和银行回执；
- KYC/KYB、风控和合规结果。

**事件与日志**

- Kafka 业务事件；
- 网关、服务和工具调用日志；
- 埋点、客服与工单；
- 任务调度和数据质量事件。

**外部数据**

- 银行/通道文件；
- 汇率和日历；
- 制裁、国家与行业配置；
- CRM、财务或 SaaS 数据。

### 2. 数据怎么进入分析层

```text
OLTP / API / File / Log
-> CDC、Kafka、批量同步
-> ODS / Raw Layer
-> 清洗、去重、统一主键
-> DWD 明细层
-> DWS 汇总与主题层
-> Metric / Semantic Layer
-> BI Dashboard / NL2SQL / API
```

实时指标通过 CDC/Kafka/Flink 等链路更新；财务对账、历史回溯可走离线批处理。对象存储或湖仓保留原始和明细数据，OLAP/数仓承担交互式聚合查询。

### 3. 为什么不全部导入一个数据库

- OLTP 为交易写入和一致性优化，复杂分析会影响主链路；
- 交易、日志、文件和文档结构不同；
- 原始日志量大，关系数据库存储成本高；
- 不同国家数据有驻留和访问要求；
- BI 需要列式扫描、分区裁剪和大规模聚合；
- 同一份数据需要原始、清洗、汇总和审计版本；
- 实时流和离线历史的时效要求不同。

因此通常采用：

- MySQL/TiDB 等承载业务交易；
- Kafka/Flink 承载事件与实时加工；
- 对象存储 + Iceberg/Hudi/Delta 等承载湖仓明细；
- ClickHouse/Doris/StarRocks/数仓承载 OLAP；
- Redis 只缓存热点结果；
- Elasticsearch 用于日志和文本检索。

具体组件由公司现有技术栈决定，架构原则比产品名称更重要。

### 4. 数据一致性和建模

支付 BI 最难的不是接入，而是口径：

- 订单事件可能乱序、重复或迟到；
- 通道成功不等于账务已入账；
- 退款、拒付会改变净收入；
- 汇率、时区和自然日定义不同；
- 商户、国家、产品维度会缓慢变化；
- 财务与运营指标的确认时点不同。

需要：

- 业务主键和幂等去重；
- event time、watermark 和迟到补偿；
- CDC offset/checkpoint；
- 拉链表或 SCD 管理维度历史；
- 数据血缘、质量规则和对账；
- 指标中心统一成功率、交易额、成本、收入口径。

### 5. BI 与 NL2SQL

NL2SQL 不直接开放所有物理表。先经过：

- 身份、租户和行列权限；
- 语义层中的指标、维度和 join 关系；
- SQL AST 校验；
- 只读账户、超时和扫描量限制；
- 脱敏和结果审计。

例如“上周巴西付款失败率”应映射到经过定义的付款成功/失败口径，而不是让模型自行选择状态字段。

### 6. 原始数据是否长期保留

不是所有数据都无限期保留。需要按数据分类设定：

- 热、温、冷存储；
- 分区和生命周期；
- 加密与密钥；
- 脱敏或 tokenization；
- 地域隔离；
- 删除和监管留存；
- 可重放与审计需求。

核心原则是交易主库与分析负载隔离、原始数据可追溯、指标口径统一、权限贯穿查询全链路。
