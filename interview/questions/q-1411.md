# 针对一个优化点，如何快速上线一个模型？

## 答案

快速上线不是跳过验证，而是把需求、实验、工程和灰度链路标准化，让一个明确优化点以最小风险进入生产。

### 1. 把优化点定义成可验收假设

例如：

```text
问题：支付失败工单意图召回不足
改动：加入错误码特征并重训分类模型
主指标：目标类别 Recall 提升
护栏：Precision、P95、误转人工率不恶化
```

先限定单一变量，避免同时改数据、特征、模型和规则而无法归因。

### 2. 建立可复现 baseline

- 固定训练数据版本；
- 固定特征 schema；
- 固定评测集和脚本；
- 记录代码、配置、随机种子和模型 artifact；
- 对比旧模型、规则基线和新模型。

### 3. 离线验收

- 主指标；
- 关键切片；
- bad case；
- 安全和合规；
- 延迟、吞吐、显存和成本；
- train-serving skew 检查；
- 模型可加载、可回滚。

跨境支付必须覆盖高金额、高风险国家、KYC、退款和拒付等关键切片。

### 4. 工程化上线

- 模型注册和版本管理；
- 容器化或标准推理服务；
- 自动化单测、集成测试和回放；
- shadow traffic；
- feature flag；
- canary/灰度发布；
- 限流、超时、熔断和 fallback；
- 一键回滚。

### 5. 分阶段放量

```text
离线回放
-> shadow
-> 1% canary
-> 小流量 A/B
-> 分层扩量
-> 全量
```

每阶段设置停止条件，例如错误率、P99、风险指标超过阈值立即回滚。

### 6. 上线后闭环

- 对比预测分布和特征分布；
- 监控服务状态与业务指标；
- 记录模型版本和决策链；
- 抽样人工复核；
- 收集误例进入下一轮数据；
- 检查长期指标和标签成熟结果。

### 7. 面试总结

快速上线靠的是标准化流水线：明确单点假设、复现 baseline、离线和安全验收、shadow/canary/A-B、监控和回滚。支付场景不能用“快速”作为省略合规、账务验证和人工审批的理由。
