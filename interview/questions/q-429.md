# 🔥 你这个 prompt 是怎么调的？

## 答案

**一句话**：Prompt 调优不是凭感觉改措辞，而是先建立评测集和基线，按错误类型定位问题，每次只改变一个关键变量，再做回归测试。

### 1. 先定义目标

先明确 Prompt 要优化什么：

- 事实正确率
- 格式遵循率
- 工具调用成功率
- 拒答 / 幻觉率
- 延迟与 Token 成本
- 业务指标，如工单解决率

没有可量化目标，“感觉回答更好了”不能作为结论。

### 2. 建立评测集

从真实请求中抽取代表性样本，至少覆盖：

- 高频正常案例
- 长尾和歧义问题
- 边界输入
- 对抗输入
- 历史 Bad Case

为每条样本定义期望结果或评分 Rubric，先跑出 Baseline。

### 3. 做错误归因

常见问题与对应手段：

| 问题 | 优先手段 |
|------|----------|
| 不理解任务 | 补角色、目标、术语定义 |
| 步骤遗漏 | 明确工作流和检查清单 |
| 输出不稳定 | 提供 JSON Schema / 模板 |
| 风格不一致 | 增加高质量 Few-shot |
| 缺少事实 | 接 RAG 或工具，不靠 Prompt 硬记 |
| 指令冲突 | 调整指令层级，删除冗余约束 |
| 上下文太长 | 检索、摘要、结构化裁剪 |

### 4. 迭代方法

1. 固定模型、温度和评测数据。
2. 每轮只改一个主要变量。
3. 比较总体分数和分场景分数。
4. 检查新 Prompt 是否修好旧问题却引入新回归。
5. 记录版本、改动原因和评测结果。

### 5. 一个推荐结构

```text
角色与目标
→ 输入说明
→ 执行步骤
→ 约束与禁止项
→ 工具 / 知识使用规则
→ 输出格式
→ Few-shot 示例
→ 提交前自检
```

**面试总结**：我会把 Prompt 当成代码管理：有数据集、有指标、有版本、有 A/B 和回归测试。Prompt 解决不了的知识与能力问题，应交给 RAG、工具或微调，而不是无限堆指令。
