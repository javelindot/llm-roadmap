# Agent的自我修正机制设计与优化

## 答案

Agent 自我修正不应只是追加一句“请反思”。可靠设计是用外部信号定位错误，只重做受影响步骤，并限制修正次数和风险。

### 1. 错误来源

- 计划缺步骤或依赖顺序错误；
- 选错工具；
- 参数格式、实体或权限错误；
- 工具超时、限流或返回冲突；
- 检索证据不足、过期或互相矛盾；
- 最终答案不满足 schema、事实或安全规则；
- 环境状态在执行中变化。

不同错误需要不同修正器，不能统一让 LLM 重写整条轨迹。

### 2. 修正闭环

```text
plan/action
-> execute
-> observe
-> verifier/classifier
-> classify error
-> local repair or replan
-> execute again
-> stop/escalate
```

Verifier 优先使用：

- JSON Schema、类型和业务规则；
- 代码测试、SQL 执行和环境状态；
- 引用/事实对齐；
- 工具错误码；
- 人工审批；
- 开放式任务最后才使用经校准的 LLM judge。

### 3. 分层修正

1. **格式修复**：解析失败时只修 JSON/参数；
2. **参数修复**：缺字段则追问或重新抽取；
3. **工具重试**：仅对可恢复错误指数退避；
4. **证据补充**：检索不足时改写 query 或换数据源；
5. **局部重规划**：前提失效时重做当前子计划；
6. **全局重规划**：目标或主要约束改变才使用；
7. **转人工**：达到预算、冲突无法解决或风险过高。

### 4. 优化方式

- 使用结构化错误码代替长自然语言报错；
- 保存已验证事实，避免每次从头推理；
- 对重复状态做 cycle detection；
- 每类错误有独立最大重试；
- 简单修复用小模型或规则；
- 只读、无依赖修正可并行；
- 记录首次成功率、修正后成功率与额外成本；
- 将高频失败转为新工具、规则或训练数据。

反思次数增加通常有边际递减，甚至会把正确答案改错，因此应由 verifier 触发，而非固定每次反思。

### 5. 安全边界

- 不可逆动作失败后不能盲目重试；
- 每次重试复用幂等键；
- 权限失败不得通过改写参数绕过；
- 高风险计划变更重新审批；
- 修正模块不能看到超出任务范围的敏感数据。

### 6. 跨境支付场景

订单查询超时可有限重试；订单号缺失应追问；规则冲突需按司法辖区和生效时间重新检索；退款工具返回“已处理”时必须停止，不能再次调用。任何金额或收款账户变化都重新确认和审批。

### 7. 面试总结

可靠自我修正是“外部验证 -> 错误分类 -> 局部修复 -> 有界重试 -> 必要时转人工”。重点是 verifier、幂等和停止条件，而不是无限 reflection。
