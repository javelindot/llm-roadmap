# 针对最后一个问题（agent 之间的关系和流程），您是怎么思考的？

## 答案

Agent 之间的关系不应只是“主 Agent 随意调用子 Agent”，而要围绕职责边界、任务契约、共享状态和失败恢复设计。广告场景可以采用主 Agent 编排、领域 Agent 执行的层级结构。

### 1. 角色划分

**Main Agent**

- 理解用户最终目标；
- 制定和修改全局计划；
- 分配预算、时间和工具调用次数；
- 处理跨领域依赖；
- 汇总结果并对最终交付负责；
- 发起用户确认和高风险审批。

**Subject Agent**

- 在一个领域内完成任务，例如素材、受众、投放、分析；
- 维护本领域状态；
- 调用本领域工具；
- 返回结果、证据、置信度和待解决依赖；
- 不越权修改其他领域状态。

### 2. 使用结构化任务契约

主 Agent 下发：

```text
task_id
goal
input_assets
constraints
budget
deadline
allowed_tools
expected_output_schema
approval_policy
```

Subject Agent 返回：

```text
status
artifacts
evidence
cost
confidence
dependencies
errors
next_action_proposal
```

### 3. Subject Agent 发现需要新素材时

不建议它绕过主 Agent 随意调用所有 Agent。更稳妥流程：

1. 投放 Agent 发现现有素材不满足版位或质量要求；
2. 返回 `need_new_creative` 事件，说明尺寸、渠道、受众和截止时间；
3. Main Agent 判断是否符合预算和用户目标；
4. Main Agent 创建素材子任务给 Creative Agent；
5. Creative Agent 产出带版本的素材；
6. 安全/品牌检查；
7. 投放 Agent 从 checkpoint 恢复；
8. 上线前根据风险级别请求用户确认。

### 4. 状态和调度

- 使用共享任务状态库，而不是靠自然语言记忆；
- 任务表示成 DAG/状态机；
- 每个动作带幂等键；
- Artifact 使用版本和 lineage；
- 支持 timeout、retry、compensation 和人工接管；
- 防止 Agent 循环调用，设置深度、预算和终止条件。

### 5. Agent 之间何时可直接通信

高频、低风险、协议稳定的依赖可以点对点调用；跨预算、跨权限或改变全局计划的动作应经过 orchestrator。否则难以审计，也容易出现重复执行和局部最优。

### 6. 跨境支付类比

支付调查 Agent 可调用交易、账务、风控和 KYC Agent 收集证据，但退款、出款和规则修改必须回到主流程进行权限检查和人工审批。

### 7. 面试总结

Main Agent 负责目标、计划和全局约束，Subject Agent 负责边界清晰的领域执行。Agent 间通过结构化 contract 和共享状态协作，依赖变化以事件返回主编排器，高风险动作必须审批。
