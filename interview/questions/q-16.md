# on-policy和off-policy有什么区别？

## 答案

**一句话**：on-policy 用当前策略采样的数据来更新当前策略，off-policy 可以用历史策略或其他策略产生的数据来训练当前策略。

### 1. 核心区别

| 对比点 | On-policy | Off-policy |
|------|-----------|------------|
| 数据来源 | 当前策略自己采样 | 历史策略、行为策略、经验池 |
| 代表算法 | SARSA、A2C、PPO | Q-learning、DQN、DDPG、SAC |
| 样本效率 | 较低，需要不断新采样 | 较高，可以复用历史数据 |
| 稳定性 | 分布一致，训练更稳 | 分布不一致，需要校正 |
| 风险 | 成本高、探索慢 | 容易有分布偏移和估计偏差 |

### 2. 放到 RLHF 里理解

PPO 类 RLHF 更接近 on-policy：当前模型生成回答，再由 reward model 打分，然后更新当前模型。因为模型一变，旧回答的数据分布就不完全匹配了。

DPO 不是传统意义上的在线 RL，它更像离线偏好优化：直接用已有的 chosen/rejected 偏好数据训练，不需要模型在线探索采样。

### 3. 面试总结

on-policy 的优点是更新和采样策略一致，缺点是贵；off-policy 的优点是能复用历史经验，缺点是要处理数据分布不一致。

## 结合项目谈（口语版 · 面试现场约 45 秒）

> 我会用业务数据来理解这个区别。比如跨境支付里，如果我们让当前风控 Agent 在线处理一批新交易，再根据人工复核结果更新它，这更像 on-policy，因为数据来自当前策略的行为。好处是反馈很贴近线上真实表现，但成本和风险都高，尤其支付场景不能随便在线探索。
>
> off-policy 更像用历史工单、历史交易、历史人工审核结论来训练模型。比如过去半年哪些交易被人工判成可疑、哪些商户最后通过了 KYB，这些都可以作为离线经验。它成本低、样本多，但问题是历史策略可能已经过时，比如某个通道、国家或监管规则变了，数据分布会偏。
>
> 所以在支付业务里，我更倾向先用 off-policy 的历史数据做离线训练和评估，再通过小流量灰度或 shadow mode 收集当前策略反馈，逐步接近 on-policy。
