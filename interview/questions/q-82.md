# 什么是迭代式微调？

## 答案

**一句话**：迭代式微调是模型上线或评估后，不断收集错误案例和新数据，清洗后再次微调，让模型逐轮变好。

### 核心回答
典型流程：
1. 用初始数据做 SFT/LoRA。
2. 离线评估和小流量验证。
3. 收集 bad case：错误回答、格式错误、拒答、幻觉。
4. 人工修正或构造偏好对。
5. 再训练新版本。
6. 对比评估后灰度上线。

优点：贴近业务真实问题，持续提升。

风险：如果只追线上反馈，可能过拟合近期样本；如果数据没清洗，会把错误偏好带进模型。

### 1. 为什么需要迭代式微调

业务模型上线后一定会遇到训练集没覆盖的问题，比如新业务、新规则、新表达、新错误类型。如果模型一直停留在初版，就会逐渐和真实业务脱节。

迭代式微调的核心是形成闭环：

- 线上收集 bad case。
- 按错误类型归因。
- 补数据或修知识库。
- 再训练和评估。
- 灰度上线。

### 2. 不是所有 bad case 都应该训练

要先分类：

- **知识缺失**：优先补 RAG，不一定微调。
- **表达格式错**：适合 SFT。
- **偏好不符合业务**：适合 DPO/KTO。
- **工具调用错**：优先修工具描述、schema 或流程。
- **规则过期**：更新知识库和规则，不要让模型死记。

### 3. 面试总结

迭代式微调的关键是 bad case 归因，而不是把所有线上失败样本都扔进训练集。

### 4. 结合跨境支付项目怎么说（面试口径）

> 跨境支付规则变化很快，所以我觉得模型不是训一次就完事。比如风控策略、国家限制、通道规则变了，客服和审核助手就会出现旧答案。
>
> 我的做法是把线上工单里的 bad case 收集起来，先分类：如果是规则没召回，补知识库和检索；如果是回答格式不符合运营习惯，做 SFT；如果是模型总是过度拒绝或过度放行，就构造偏好对做 DPO。
>
> 这样每轮迭代都有明确目标，而不是盲目加数据。支付场景里尤其要注意，规则过期的问题不能靠模型记忆解决，必须更新 RAG 或规则系统。

**钩子**：迭代微调要区分知识问题和能力问题，不是所有 bad case 都该拿去训练。
