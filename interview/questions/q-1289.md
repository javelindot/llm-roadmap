# 你做过AI相关的开发是吧？介绍一下你在美团的agent的开发的场景和技术难点？

## 答案

Agent 项目面向跨境支付商户服务和内部客服，典型任务是解释“付款为什么失败”“款项为什么未到账”“KYC 材料为何被退回”。这些问题不能只靠模型知识，需要串联订单、通道回执、风控结果、账务状态和业务 SOP。

### 1. 业务流程

以付款未到账为例：

```text
用户问题
-> 身份与商户权限校验
-> 意图、订单号和必要槽位识别
-> 查询订单主状态
-> 按状态查询通道/账务/风控
-> 检索对应国家、币种和产品 SOP
-> 基于证据生成解释
-> 低置信、高风险或工具异常时转人工
```

动态事实通过工具获取，规则解释通过 RAG 获取。模型不能凭参数记忆猜测余额、订单状态、实时汇率或预计到账时间。

### 2. 技术架构

- API 网关：认证、租户、限流和 trace_id；
- Agent 编排：LangGraph/状态机维护步骤和路由；
- Tool Gateway：统一订单、账务、通道、风控、工单接口；
- RAG：解析 FAQ、错误码、通道规则、合规 SOP，混合召回加 rerank；
- Model Gateway：模型路由、超时、重试、降级和版本管理；
- Guardrail：字段脱敏、权限、敏感规则隐藏、操作确认；
- Observability：记录节点、工具参数摘要、证据、token、延迟和失败原因；
- Evaluation：离线任务集、trace 回放、灰度和人工反馈闭环。

状态中只保留必要字段，例如：

```text
merchant_id, order_id, intent,
order_status, channel_code,
risk_category, retrieved_evidence,
tool_errors, need_human, trace_id
```

### 3. 技术难点一：模型规划不稳定

同一问题可能被模型选择不同工具，甚至循环调用。处理方式：

- 高频流程使用显式状态图，不完全依赖自由 ReAct；
- Tool schema 使用枚举、必填字段和业务校验；
- 每个节点设置最大重试、deadline 和终止条件；
- 将“无数据、权限不足、系统超时”定义为不同状态；
- 工具返回结构化结果，不把整段日志直接塞给模型。

### 4. 技术难点二：静态知识与实时事实冲突

SOP 可能说某通道通常 T+1 到账，但实时订单可能处于银行审核。答案生成时建立证据优先级：

```text
授权实时工具结果
> 当前有效业务配置
> 带版本和生效时间的知识文档
> 模型参数知识
```

回答必须引用证据，并避免把经验时效承诺为确定时间。

### 5. 技术难点三：权限和资金安全

- 不同商户、客服、风控人员可见字段不同；
- 风控只返回可解释的大类，不暴露规则阈值；
- 查询工具默认只读；
- 退款、重试付款、账户冻结等写操作进入确定性工作流；
- 校验幂等键、订单状态和审批；
- 模型输出不能直接成为资金指令。

### 6. 技术难点四：上下文、延迟与失败恢复

- 独立工具并行调用；
- 长回执先结构化摘要；
- RAG 只保留与国家、币种、产品匹配的证据；
- 公共前缀和低风险 FAQ 缓存；
- 工具失败时局部重试，不从头重复整条链路；
- 超时后给出明确状态并转人工，不编造结果。

### 7. 评估

不能只看最终文本：

- 意图与槽位准确率；
- 工具选择和参数准确率；
- trace 步骤正确率；
- 任务完成率、转人工率、人工改写率；
- 引用、事实一致性；
- 越权、敏感信息和资金操作违规；
- P95/P99、调用次数和单位任务成本。

项目价值来自把分散查询步骤变成可审计流程，而不是让模型拥有更大权限。
