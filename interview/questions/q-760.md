# 服务的容错，docker和k8s

## 答案

服务容错要覆盖实例、依赖、节点、发布和数据五类故障。Docker 提供可重复的进程隔离与镜像，Kubernetes 负责调度和期望状态，但业务容错仍需应用自身设计。

### 应用层

- 请求设置超时、取消传播和有界重试，重试采用指数退避与抖动。
- 写操作使用幂等键，避免重试造成重复副作用。
- 对下游使用连接池、限流、熔断、隔离舱和降级。
- 无状态服务，状态外置到高可用数据库/对象存储。
- 健康检查、结构化日志、metrics、trace 和告警。

### Docker

使用固定基础镜像和依赖版本，多阶段构建减小镜像；非 root 用户、只读文件系统和资源限制提升安全。正确处理 PID 1 与 SIGTERM，应用在终止前停止接流、完成请求并保存状态。镜像不等于高可用，单容器故障仍需编排恢复。

### Kubernetes

- Deployment 多副本、滚动更新和回滚。
- readiness 控制是否接流，liveness 只用于真正卡死，startupProbe 保护慢启动。
- requests/limits、HPA、PodDisruptionBudget 和拓扑分散。
- Service 负载均衡，节点故障后重新调度。
- ConfigMap/Secret、持久卷和 NetworkPolicy。

发布采用 canary/blue-green，设置 `preStop` 和足够 `terminationGracePeriodSeconds`。还要做依赖故障、节点宕机、网络延迟和容量突增的演练。K8s 能重启 Pod，但无法修复错误重试、数据不一致或级联雪崩。
