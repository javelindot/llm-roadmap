# LoRA 微调秩设置的是多少

## 答案

LoRA rank 不能脱离项目直接报一个数字。应回答真实设置、候选范围、目标层、显存预算和消融结果；如果只试过一个 rank，就明确说明尚未证明它最优。

### 1. rank 控制什么

对原权重 `W in R^(d_out x d_in)`：

```text
Delta_W = B * A
A in R^(r x d_in)
B in R^(d_out x r)
```

新增参数量约为：

```text
r * (d_in + d_out)
```

`r` 越大，更新子空间容量越高，同时参数、optimizer state、通信和 Adapter 大小近似线性增加。

### 2. 常见选择

实践中可从以下范围做搜索：

- `r=4/8`：格式、风格或简单任务；
- `r=16/32`：常见领域 SFT 起点；
- `r=64/128`：任务复杂、数据充足或目标模块较少时；
- 更高 rank：必须用消融证明收益，不能默认越大越好。

这只是经验起点，不是固定标准。Attention 的 q/k/v/o、MLP 的 gate/up/down 是否都训练，往往与 rank 同样重要。

### 3. 面试回答示例

> 项目最终使用 `r=32`、`alpha=64`，目标层是 q/v 和部分 MLP。我们比较过 8、16、32、64；32 相比 16 在领域验证集有稳定收益，64 的提升很小但显存和训练时间继续增加，所以选 32。这个结论只适用于当时的数据和模型。

数字必须替换成真实实验。

### 4. 如何评测

- 相同 base model、数据顺序和训练 token；
- 比较 held-out 质量、通用能力回归和长尾切片；
- 记录显存、tokens/s、checkpoint 大小；
- 同时搜索 alpha、dropout、学习率和 target modules；
- 检查 rank 增大后是否只是更快过拟合。

可使用 AdaLoRA 动态分配 rank，或 DoRA 改善低秩表达，但仍需验证推理框架支持。

### 5. 跨境支付场景

若只是学习客服格式和工具 schema，较低 rank 可能足够；若同时覆盖多语言、复杂合规推理和文档理解，可能需要更高 rank 或扩大目标层。事实正确性与资金安全不能通过提高 rank 保证。

### 6. 面试总结

可信回答是“rank 数值 + target modules + alpha + 对比实验 + 资源权衡”。rank 决定低秩更新容量，但数据质量和训练模块范围通常更关键。
