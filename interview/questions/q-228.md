# 如果不用springai 调用api怎么实现agent调用

## 答案

**一句话**：不用 Spring AI 也可以实现 Agent，本质上就是自己封装模型调用、工具注册、工具选择、工具执行、状态管理和循环控制。

### 1. Agent 调用的本质

Agent 并不依赖某个框架才能存在。最小闭环是：

1. 接收用户输入。
2. 拼接 system prompt、历史上下文和工具定义。
3. 调用 LLM API。
4. 解析模型返回，看是直接回答还是要调用工具。
5. 如果要调用工具，校验参数并执行业务方法。
6. 把工具结果追加回 messages。
7. 再次调用模型，直到得到最终答案或达到最大步数。

这就是一个基础 Agent loop。

### 2. 自己实现需要哪些模块

如果不用 Spring AI，我会自己封装：

- **ModelClient**：统一不同模型 API。
- **ToolRegistry**：注册工具名称、描述、参数 schema 和执行器。
- **PromptBuilder**：构造系统提示和上下文。
- **AgentExecutor**：控制推理-工具调用循环。
- **StateStore**：保存会话状态、工具结果和执行步骤。
- **Guardrail**：做权限、参数、敏感动作校验。
- **TraceLogger**：记录每次模型调用和工具调用。

这样即使底层换模型或换框架，上层业务逻辑也比较稳定。

### 3. 关键控制点

- 最大工具调用轮数，防止死循环。
- 工具白名单，防止模型乱调。
- 参数 schema 校验，防止脏数据进入业务系统。
- 超时和熔断，防止模型或工具拖垮服务。
- 幂等和审计，尤其是写操作。
- 兜底回答，避免模型失败时用户无响应。

### 4. 结合跨境支付项目怎么说（面试口径）

> 如果不用 Spring AI，我会在 Java 服务里自己封装一个 AgentExecutor。比如用户问“这笔付款为什么失败”，Executor 先把可用工具列表给模型，包括查交易、查通道回执、查风控原因、查知识库。模型返回要调用 `queryPaymentStatus`，我们校验订单号和商户权限后执行工具，再把结果交回模型继续判断是否需要查风控或通道。
>
> 对支付系统来说，我一定会把工具分级。查询类工具可以开放给客服 Agent；资金变更类工具不直接给模型，必须走人工确认和审批。Agent loop 里还要设置最大步数，避免模型一直查下去。
>
> 所以不用框架也能做，框架只是帮我们封装一部分通用能力。真正重要的是工具注册、状态控制、权限校验和可观测性。
