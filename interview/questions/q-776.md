# 询问Agent工具的设计情况，是否采用workflow形式。

## 答案

Agent 工具应设计成小而明确、参数强类型、权限最小化的能力单元。是否采用 workflow 取决于流程确定性和错误成本。

### 工具设计

每个工具包含唯一名称、清晰用途、JSON Schema、返回结构、超时、幂等与错误码。相似工具避免描述重叠；参数由服务端再次校验，模型不能直接拼 SQL 或任意 URL。

对读取工具可自动执行；支付、删除、发信等副作用工具要求权限和人工确认。调用记录包含输入、输出、耗时、版本和 trace，便于回放。

### Workflow 形式

固定或高风险流程优先显式 workflow：节点代表检索、审批、执行和验证，边由确定性条件控制，状态持久化并支持 checkpoint。模型只在需要语义判断的节点决策。

开放探索任务可使用 ReAct 循环，但仍要设置允许工具、最大步数、预算、停止和失败分支。常见方案是 workflow 外壳加局部 Agent，而不是完全自由循环。

### 工具选择

工具多时先按权限和领域过滤，再用 embedding/router 选 top-k schema 给 LLM。执行结果返回结构化事实，Verifier 检查是否满足目标；失败时只重试可恢复错误。

评估包括工具选择准确率、参数通过率、执行成功率、无效调用率、任务完成率和安全违规。若一个函数调用就能完成任务，不应引入多轮 Agent。
