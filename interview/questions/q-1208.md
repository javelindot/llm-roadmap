# agent 服务高可用、稳健性是怎么保证的？

## 答案

Agent 高可用和稳健性需要同时覆盖服务可用、状态一致、工具可靠、模型不确定性和业务安全。核心原则是：

> Agent 可以失败，但失败必须可检测、可恢复、可降级，且不能产生重复或越权业务动作。

### 1. 服务架构

- 编排 Worker 尽量无状态，支持水平扩容；
- 会话、计划和工具结果写入持久状态存储；
- 使用队列削峰，设置背压和最大并发；
- 多可用区部署，依赖服务健康检查；
- 模型、检索、工具分别设置超时和连接池；
- 配置、Prompt、模型和工具 Schema 版本化；
- 发布使用 canary、blue-green 和自动回滚。

### 2. 持久化状态与恢复

每个任务记录：

```text
trace_id、session_id、tenant_id、current_step、
tool_calls、tool_results、state_version、budgets
```

关键步骤 checkpoint。Worker 重启后从已确认状态继续，而不是重新执行全部计划。状态更新使用乐观锁或版本号，避免并发会话覆盖。

### 3. 工具调用可靠性

- 每个调用设置 timeout、retry budget 和 circuit breaker；
- 只对明确可重试错误退避重试；
- 写操作必须有幂等键；
- 超时后先查询业务状态，不能盲目重发；
- 限制最大步数、token、费用和工具调用数；
- 工具结果做 Schema、权限和新鲜度校验；
- 连续失败、循环或结果冲突时转人工。

支付动作还要校验订单状态、金额、币种、报价有效期和审批。

### 4. 降级策略

- 强模型不可用：路由到备用模型或模板；
- 向量检索失败：退回 BM25/FAQ；
- Reranker 超时：使用融合召回排序；
- 实时工具失败：明确说明暂时无法确认，不猜测状态；
- 编排异常：保存上下文并转人工；
- 写服务异常：切换只读模式。

降级输出必须降低承诺，不应在数据不可用时生成看似确定的答案。

### 5. 模型稳健性

- 结构化输出和约束解码；
- 低置信度追问、拒答或转人工；
- 多证据冲突检测；
- 提示注入隔离；
- 工具和文档作为不可信输入；
- 高风险动作由代码审批；
- 关键 claim 用规则、工具或 verifier 核验。

多次采样或多个模型一致并不能证明事实正确，仍要以权威数据为准。

### 6. 数据和权限

- 每次请求绑定不可被模型修改的身份和租户；
- 检索、缓存、记忆和工具都执行服务端 ACL；
- 日志和 Trace 脱敏；
- 长期记忆设置来源、有效期和删除机制；
- 密钥使用短期凭证，不能出现在 Prompt；
- 跨租户缓存键包含 tenant 和权限版本。

### 7. 可观测性

为每次任务记录：

- 路由、Prompt 和模型版本；
- 检索候选和最终证据；
- 工具参数、结果、重试和耗时；
- 状态迁移和人工审批；
- token、成本和降级原因；
- 最终业务状态。

指标包括成功率、P95/P99、超时、重试、熔断、循环、降级、转人工、错误动作和敏感信息泄漏率。

### 8. 测试与容灾

- 单元测试每个工具和状态转移；
- 契约测试 Tool Schema；
- 历史流量回放；
- 依赖超时、重复回调和乱序事件注入；
- 提示注入与越权红队；
- 峰值压测、节点故障和网络分区演练；
- 定期验证备份恢复与 checkpoint 兼容性。

跨境支付中，系统目标不是“永不报错”，而是模型或依赖故障时不重复付款、不泄漏数据、不错误承诺，并能把任务安全地交给人工。
