# 资源有限的情况，不能让一个vllm把资源吃完。

## 答案

**一句话**：资源有限时，不能让一个 vLLM 服务把 GPU、显存和请求队列全部吃满，要通过资源隔离、并发限制、模型分层、调度和降级来控制。

### 1. vLLM 为什么容易吃资源

vLLM 为了提高吞吐，会尽量利用 GPU 显存做模型权重、KV Cache 和 batch 调度。如果不控制：

- 单个模型会占满显存。
- 长上下文请求会挤占 KV Cache。
- 高并发会导致排队和 OOM。
- 一个业务流量可能影响其它业务。

所以必须做资源治理。

### 2. 控制手段

常见方式包括：

- 设置 `gpu_memory_utilization`，不要让进程占满全部显存。
- 控制 `max_model_len`，避免长上下文拖垮并发。
- 控制 `max_num_seqs` 和 `max_num_batched_tokens`。
- 在模型网关做限流和配额。
- 按业务线拆服务，避免互相影响。
- Kubernetes 里用 GPU 资源限制和节点隔离。
- 有条件时用 MIG 或多卡隔离。
- 简单任务走小模型，复杂任务走大模型。
- 超时后降级、缓存或转人工。

### 3. 运维监控

要监控：

- GPU 显存占用。
- GPU 利用率。
- 请求队列长度。
- 首 token 延迟。
- 总响应时间。
- OOM 和错误率。
- 不同业务的调用量和成本。

没有监控，就不知道是模型慢、检索慢还是排队慢。

### 4. 结合跨境支付项目怎么说（面试口径）

> 在跨境支付项目里，客服问答、风控解释、文档抽取可能都想用同一个本地大模型。如果不做限制，一个长文档解析任务就可能把 vLLM 显存和队列打满，导致客服实时问答变慢。
>
> 我会通过模型网关做资源隔离。比如客服实时问答有独立实例和并发配额，离线文档抽取走另一个实例或低优先级队列；简单 FAQ 走小模型或缓存，复杂疑难工单再走大模型。vLLM 本身设置显存利用率、最大上下文和最大并发，防止 OOM。
>
> 对支付业务来说，模型服务不能拖垮核心系统。模型超时后要返回兜底话术或转人工，而不是让用户付款查询一直卡住。

**钩子**：可以引到“gpu_memory_utilization、KV Cache、模型网关配额、实时和离线任务隔离”。
