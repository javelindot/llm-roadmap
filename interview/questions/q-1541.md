# 如何实现 RAG 的增量索引更新？

## 答案

RAG 增量索引的目标是只处理新增、修改和删除的内容，同时保证文档库、chunk、向量、关键词索引和权限元数据最终一致。

### 1. 事件来源

通过 CDC、消息队列、对象存储事件或定时扫描生成：

```text
CREATE(document_id, version)
UPDATE(document_id, old_version, new_version)
DELETE(document_id, version)
```

每个事件带 event ID、时间、租户和来源版本，消费者使用幂等键处理重复投递。

### 2. 稳定标识与指纹

- `document_id` 标识逻辑文档；
- `version` 标识内容版本；
- 内容 hash 判断是否真的变化；
- chunk ID 由 `document_id + chunk position/content hash + parser version` 决定；
- 记录 embedding model、chunker 和 schema 版本。

修改时重新解析文档，与旧 chunk 集合做 diff：

```text
unchanged -> 复用向量
added/changed -> 重新 embedding 并 upsert
removed -> tombstone/delete
```

### 3. 写入一致性

推荐使用版本化两阶段流程：

1. 在 staging 写入新 chunk、向量和倒排索引；
2. 校验数量、权限、向量维度和可检索性；
3. 原子更新文档的 active version；
4. 旧版本延迟删除，便于回滚；
5. 使相关缓存失效。

查询只读取 active version，避免更新中同时看到新旧 chunk。

### 4. 何时不能只做增量

以下变更通常需要后台全量重建或蓝绿索引：

- 更换 embedding 模型或向量维度；
- 修改 chunking 规则；
- metadata/ACL schema 大改；
- 距离度量或 ANN 索引类型变化；
- 大规模解析器升级。

可双写新旧索引、回放流量验证，再原子切换 alias。

### 5. 删除与合规

删除必须覆盖：

- 原文与对象存储；
- chunk 和向量索引；
- BM25/图索引；
- query/context cache；
- 派生摘要；
- 按合规要求处理备份。

使用 tombstone 防止乱序 UPDATE 把已删除文档重新写回，并定期物理清理。

### 6. 监控

- 事件积压和更新延迟；
- 源文档与 active chunk 数对账；
- embedding 失败、重复和孤儿 chunk；
- 删除完成时间；
- 新旧索引检索差异；
- Recall、P95 和存储增长；
- 权限与版本泄漏。

### 7. 跨境支付场景

监管规则要保存司法辖区、生效/失效时间和来源，不能用新文档静默覆盖历史版本。商户权限变更应优先更新 ACL；订单、余额和汇率等实时事实不应进入文档向量索引，而应查询业务系统。

### 8. 面试总结

增量索引依赖 CDC、稳定 ID、内容指纹、chunk diff、幂等 upsert、版本切换和删除传播。模型或切分规则大改时使用蓝绿全量重建，不能混用不兼容向量。
