# RAG技术的性能瓶颈主要体现在哪些环节?

## 答案

RAG 的性能瓶颈既包括延迟和吞吐，也包括检索质量与知识新鲜度。应通过端到端 trace 拆分每个阶段，不能只说“慢在大模型”。

### 1. 文档解析与索引

- PDF 版面、表格、OCR 和图片解析耗时；
- 大量重复与低质量文档增加索引；
- embedding 批处理吞吐不足；
- 全量重建导致更新时间长；
- 文档更新后旧 chunk 未删除，影响正确性；
- 多语言、多个 embedding 版本占用存储。

优化方式包括增量 CDC、内容指纹、幂等 upsert、批量 embedding、异步索引和按版本原子切换。

### 2. 查询理解

意图识别、指代消解、Query rewrite 和问题分解若全部串行调用大模型，会显著增加 TTFT。简单请求可以用规则或小模型，复杂多跳请求才启用 LLM planner，并设置超时和最大步骤。

### 3. 检索

- 向量库规模、索引类型和参数不合理；
- ACL、国家、币种等过滤选择性差；
- Top-K 过大增加网络与重排成本；
- BM25、Dense、FAQ 多路串行；
- 冷缓存、跨区域网络和热点分片。

可以并行多路召回、前置强过滤、使用合适的 HNSW/IVF 参数、缓存高频安全 Query，并监控 Recall 与延迟的 Pareto 曲线。

### 4. Rerank

Cross-Encoder 对每个 Query-Document pair 做前向，候选多、chunk 长时成本明显。优化包括：

- 初召回先去重和规则过滤；
- 控制候选数与最大长度；
- 动态 rerank，只对模糊 Query 启用；
- batch 推理和轻量 reranker；
- 超时后降级到 RRF 排序。

### 5. 上下文构造

重复 chunk、父子块重叠和过长历史会增加 token、稀释证据并拖慢模型。应做去重、冲突检测、token budget、动态打包和证据覆盖优化，而不是把 Top-K 全部拼接。

### 6. LLM 生成

通常是在线延迟与成本的大头：

- Prompt 和证据过长导致 prefill 慢；
- 输出过长导致 decode 慢；
- 并发时 KV Cache 和显存不足；
- 模型尺寸、量化和 batching 不匹配；
- 外部 API 限流或网络抖动。

可采用模型路由、streaming、prefix cache、continuous batching、GQA/KV 优化、量化和输出长度控制。

### 7. 工具与下游系统

订单、余额、通道和权限 API 可能比检索更慢。应设置每个工具的 deadline、并行调用、熔断、幂等和降级，并区分“工具超时”和“没有数据”，避免模型把超时解释为业务失败。

### 8. 质量瓶颈

- chunk 边界切坏条件和结论；
- embedding 不适合领域或语言；
- 精确错误码被 dense retrieval 漏掉；
- 旧版本和越权文档被召回；
- reranker 只看语义，忽略否定和时间；
- 模型不遵循证据或引用错配。

因此性能优化必须守住 Recall@K、Context Precision、faithfulness 和安全指标，不能为了降低延迟直接减少关键证据。

### 9. 定位方法

每次请求记录：

```text
parse/rewrite/retrieval/rerank/tool/LLM latency,
candidate count, context tokens, output tokens,
cache hit, model/index/version, fallback reason
```

同时观察 P50/P95/P99、QPS、超时率、成本和分桶质量。跨境支付高峰可能来自结算日或通道异常，容量规划要按突发并发、商户隔离和关键查询优先级进行。
