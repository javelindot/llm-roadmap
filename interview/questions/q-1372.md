# VLLM prefix cache实现过吗?

## 答案

可以这样回答：我了解并能配置 vLLM 的 Automatic Prefix Caching。它的作用是缓存相同前缀的 KV blocks，下次请求命中同一前缀时可以跳过这部分 prefill 计算，从而降低 TTFT。它加速的是共享 prompt 前缀，不是复用最终输出。

### 1. Prefix cache 解决什么问题

很多线上请求有相同前缀：

- system prompt；
- 工具调用 schema；
- few-shot examples；
- 固定业务规则；
- RAG 中稳定的公共说明；
- 多轮对话前面不变的历史。

如果每次都从头 prefill，会重复计算。Prefix cache 把前缀 token 对应的 KV cache block 存起来，后续相同前缀直接复用。

### 2. vLLM 的基本实现思路

vLLM 把 KV cache 分成 block。每个 block 会根据以下信息计算 hash：

```text
parent block hash
current block tokens
extra hashes
```

extra hashes 可以包含 LoRA ID、多模态输入 hash、cache salt 等。这样只有完整前缀一致时才会命中缓存，避免不同请求误复用。

请求进来后：

1. tokenizer 得到 token ids；
2. 按 block size 切分；
3. 从前往后查 block hash；
4. 命中的 block 复用 KV；
5. 从第一个未命中的 block 开始继续 prefill；
6. decode 阶段照常逐 token 生成。

### 3. 工程使用要点

通常需要开启 prefix caching，例如服务参数中启用 automatic prefix caching。实际部署时还要关注：

- prompt 必须 token 级别一致，空格、chat template、工具顺序变化都可能导致 miss；
- system prompt 和工具 schema 版本要稳定；
- RAG 文档如果每次排序不同，会降低命中率；
- LoRA、多模态输入、多租户场景要隔离；
- 需要监控 cache hit rate、TTFT、GPU KV cache 占用、eviction；
- 对不同用户共享缓存时要考虑隐私风险，可使用 cache salt 做隔离。

### 4. 它不能解决什么

- 不能减少完全不同 prompt 的 prefill；
- 不能复用 decode 生成结果；
- 不能替代语义缓存，语义相似但 token 不同也不会命中；
- 长上下文若每轮都在前面插入新内容，命中率会下降；
- cache 占用显存，需要和并发、block eviction 做平衡。

### 5. 跨境支付场景

适合缓存：

```text
system prompt
合规边界说明
工具 schema
客服话术规范
固定 few-shot
```

不适合直接缓存或共享：

```text
商户敏感信息
订单明细
KYC 材料
风控命中原因
```

对商户客服助手来说，prefix cache 可以明显优化“同一系统提示 + 同一工具说明 + 不同用户问题”的 TTFT，但订单状态和账务数据仍要实时查工具。

### 6. 面试总结

vLLM prefix cache 的核心是按 token block 缓存相同前缀的 KV，命中后跳过重复 prefill。面试要讲清楚它只对完全相同前缀有效，收益主要体现在 TTFT；线上要关注 chat template 稳定性、命中率、显存占用、多租户隔离和 cache salt。
