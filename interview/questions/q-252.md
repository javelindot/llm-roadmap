# vllm部署遇到的坑？

## 答案

**一句话**：vLLM 部署最大的坑通常不在“能不能跑起来”，而在显存、并发、KV Cache、模型格式、CUDA 环境、多卡通信、限流和线上稳定性。

### 1. 常见部署坑

我遇到或重点关注的坑主要有：

- **CUDA/驱动版本不匹配**：PyTorch、CUDA、显卡驱动、vLLM 版本不兼容会导致启动失败或性能异常。
- **显存估算不准**：模型权重、KV Cache、batch、上下文长度都会吃显存，不能只按模型参数量估。
- **max_model_len 设置过大**：上下文越长，KV Cache 越大，并发能力会明显下降。
- **并发参数不合理**：`max_num_seqs`、`max_num_batched_tokens` 设置过高容易 OOM，过低吞吐又上不去。
- **多卡通信问题**：tensor parallel 涉及 NCCL、网络、CUDA_VISIBLE_DEVICES，配置不对会卡住。
- **模型 chat template 不一致**：同一个模型如果 prompt 模板不对，回答质量会明显下降。
- **量化兼容问题**：AWQ、GPTQ、FP8 等量化方式不是所有模型和显卡都适合。
- **OpenAI 接口兼容细节**：流式输出、tool call、stop words、错误码需要适配调用方。

### 2. 优化思路

部署前我会先做容量评估：

- 模型大小和精度。
- GPU 显存和卡数。
- 目标 QPS、并发、平均输入输出长度。
- 是否需要长上下文。
- 是否需要流式返回。

然后通过压测调整 batch、并发、上下文长度和限流策略。线上要加监控，包括显存、吞吐、首 token 延迟、总延迟、队列长度、错误率和 OOM 次数。

### 3. 面试总结

vLLM 的优势是 PagedAttention 和高吞吐推理，但它不是开箱即用就能稳定上生产。生产环境要重点关注容量规划、请求隔离、超时降级、日志监控和灰度发布。

### 4. 结合跨境支付项目怎么说（面试口径）

> 在跨境支付项目里，如果我们用 vLLM 部署本地模型，通常不会把它直接挂到收付款主链路上，而是用于客服问答、文档抽取、风控解释这类旁路能力。部署时我会先根据平均问题长度、知识库上下文长度、并发量估算显存，避免一上来把 `max_model_len` 设得很大，导致并发能力掉下来。
>
> 比如客服高峰期大量用户问付款状态，如果模型服务排队严重，就会影响体验。所以我会做限流、超时降级和模型分层：简单 FAQ 可以走小模型或缓存，复杂交易解释再走大模型。vLLM 侧监控首 token 延迟、总延迟和 OOM，调用方也要有兜底答案和转人工策略。
>
> 还有一个细节是模型输出不能直接信。涉及风控、合规、资金状态时，模型只负责解释，真实结论来自订单系统、风控系统和通道回执。

**钩子**：可以引到“KV Cache 显存估算、max_model_len 和并发权衡、AI 服务旁路化部署”。
