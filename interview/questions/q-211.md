# 请解释 Query 重写的定义、作用，以及在 RAG 系统中的实现逻辑？

## 答案

> **定义**：Query 重写就是**在检索前把用户的原始 query 改写成"更检索友好"的版本**——填补"用户表达"和"知识库表达"之间的语义鸿沟。
>
> **为什么需要**：
> - 用户问得短、口语化（"退款多久"）
> - 知识库写得长、专业化（"跨境信用卡退款的到账时效……"）
> - 多轮对话用户用代词（"那个怎么办"）
> - 业务术语不一致（用户："退款"、文档："退付 / refund"）
>
> 不重写直接 embed 检索，召回率经常掉 10+ 个点。
>
> **作用**：
> - 提升召回率（最直接）
> - 处理多轮对话上下文
> - 提取真实查询意图（剥离客套话、口水话）
> - 适配不同检索器（BM25 关键词版 vs 向量版）
>
> **我们在跨境支付 RAG 里的 5 种重写形态**：
>
> **1. 多轮上下文消解（必上）**
> ```
> 历史：U: 我的订单 #20260601-XX 怎么样了？  A: 已完成清算
> 当前：U: 那退款呢？
> 重写：U: 订单 #20260601-XX 的退款流程是怎么样的？
> ```
> 用 Qwen-7B 做改写器，0.3 秒延迟，召回率 +12 个点。
>
> **2. 规范化重写**
> ```
> 用户：钱多久到？
> 重写：跨境支付到账时效是多少
> ```
> 把口语转成"知识库语言风格"。
>
> **3. 业务术语扩展**
> ```
> 用户：怎么退款
> 重写：怎么退款 / 退付 / refund / 退回原账户
> ```
> 配合 BM25 关键词检索特别有效。
>
> **4. 复杂问题拆解**
> ```
> 用户：我跨境付款被拒了，是为啥，要怎么办
> 拆解：
>   - 跨境支付被拒的常见原因
>   - 跨境支付被拒后的处理流程
> ```
> 每个子问题独立检索后聚合。
>
> **5. HyDE 风格"假设答案"**（详见 q-131）
> 让 LLM 先假装回答，用假设答案 embed 检索。
>
> **实现逻辑**（代码示意）：
> ```python
> async def rewrite_query(query: str, history: list[Msg]) -> RewrittenQuery:
>     # 1. 检测是否需要重写（短 query / 有代词 / 业务术语）
>     if not needs_rewrite(query): return RewrittenQuery(query)
>
>     # 2. 调小模型改写
>     prompt = REWRITE_PROMPT.format(history=history, query=query)
>     rewritten = await llm.invoke(prompt)
>
>     # 3. 输出多个变体（标准、扩展、HyDE）
>     return RewrittenQuery(
>         standard=rewritten.standard,
>         keyword_variants=rewritten.expanded,
>         hyde=rewritten.hyde,
>     )
> ```
>
> **几个工程经验**：
> - **不全开**：简单 query 跳过重写，省延迟
> - **小模型够用**：Qwen-7B 改写器准确率 96%，不需要大模型
> - **效果监控**：每月对比"开重写 vs 不开重写"的召回率，下降立刻回滚
> - **配合 cache**：重写后的 query 也可以缓存
>
> 上线后 RAG 端到端准确率提升 8 个点，主要功劳给多轮消解和业务术语扩展两块。
