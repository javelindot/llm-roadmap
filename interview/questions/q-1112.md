# 什么是 RAG (Retrieval-Augmented Generation)？

## 答案

RAG（Retrieval-Augmented Generation，检索增强生成）是在大模型生成前，先从外部知识源检索与问题相关的证据，再让模型基于证据回答。

> 用户问题 → 查询理解 → 检索与重排 → 构造证据上下文 → LLM 生成 → 引用与事实校验。

它解决的是“模型参数知识不够新、不够私有、不可追溯”的问题，但不等于把所有业务数据都放进向量库。

### 1. 离线知识索引

离线链路包括：

1. 采集 PDF、网页、Word、表格、数据库规则等知识；
2. 解析正文、标题、表格、图片和页码；
3. 清洗、去重、版本和权限标注；
4. 按结构与业务语义分块；
5. 生成 embedding；
6. 建立向量、BM25 和元数据索引。

每个 chunk 应保留：

> `source、section、page、country、currency、product、channel、version、effective_time、visibility`

这样在线检索才能判断内容来自哪里、何时生效以及谁可以看到。

### 2. 在线检索与生成

在线阶段先对用户问题做：

- 多轮指代消解；
- 意图识别和实体抽取；
- 知识库、实时工具或人工服务路由；
- 国家、币种、产品、通道和权限过滤。

检索通常使用 BM25 与 dense retrieval 两路召回，通过 RRF 融合，再用 cross-encoder reranker 精排。最终只将少量高质量证据放入模型上下文，并要求每个关键结论引用来源。

### 3. 跨境支付中的使用方式

适合进入 RAG 的内容：

- 全球收款、付款、换汇和收单产品说明；
- KYC/KYB 材料与合规规则；
- 国家、币种和通道支持范围；
- 错误码、退款、拒付和结算 SOP；
- 客服与运营操作手册。

不适合仅靠 RAG 的内容：

- 当前订单、余额和账务状态；
- 实时报价和通道健康；
- 风控审核进度；
- 付款、退款、冻结等资金操作。

这些动态事实必须通过有权限的业务工具查询。RAG 负责解释规则，工具负责提供当前状态。

例如“这笔 USD 付款为什么没到账”，系统先查询订单、通道回执和清算状态，再检索该通道的到账规则，最后将事实与规则合并说明。

### 4. RAG 的优势

- 知识更新不需要重新训练大模型；
- 可以接入企业私有知识；
- 回答能够引用和审计；
- 通过权限和元数据控制检索范围；
- 相比把全部文档塞入长上下文，成本和噪声更可控。

### 5. RAG 的局限

- 正确知识未召回时，模型无法可靠回答；
- 文档过期、冲突或解析错误会直接污染结果；
- 多跳关系和全局总结能力有限；
- 模型仍可能忽略证据或产生幻觉；
- 实时业务事实不能由静态知识库保证；
- 检索、重排和大模型会增加延迟与成本。

因此生产系统还需要知识治理、混合检索、重排、拒答、工具调用、权限、安全、评测、灰度和降级机制。

### 6. 如何评估

- 检索：Recall@K、MRR、NDCG、Context Precision/Recall；
- 生成：Faithfulness、答案正确性、完整性和引用正确率；
- 业务：首次解决率、人工改写率、转人工率和投诉率；
- 系统：P95/P99、超时率、成本和索引新鲜度；
- 安全：越权召回、敏感信息泄漏和错误资金承诺。

RAG 的核心价值不是让模型“知道更多”，而是让模型在回答时能找到当前适用、具备权限且可以追溯的证据。
