# 强化学习状态、动作、奖励如何设置？

## 答案

强化学习中的状态、动作和奖励应从业务决策过程定义，而不是先选算法再硬套。好的设计要满足信息充分、动作可执行、奖励与真实目标一致，并避免模型利用漏洞。

### 1. State

状态 `s_t` 应包含做当前决策需要的信息，并尽量满足马尔可夫性：

- 当前观察；
- 历史的必要摘要；
- 环境状态；
- 资源和预算；
- 已执行动作与结果；
- 权限与约束；
- 时间。

不能直接包含未来信息或标签，否则会泄漏。若观察不完整，实际更接近 POMDP，可用 belief、RNN/Transformer memory 或外部状态。

### 2. Action

动作空间应：

- 明确且可执行；
- 粒度适中；
- 有合法性 mask；
- 区分可逆和不可逆；
- 设置最大步数；
- 支持结束、追问和转人工。

Agent 场景可包括：

```text
search
retrieve
call_tool
ask_user
respond
escalate
stop
```

工具参数也属于动作的一部分，需要 schema 校验。

### 3. Reward

奖励应优先接近最终业务目标：

```text
R =
task_success
+ correctness
+ efficient_process
- unsafe_action
- unnecessary_cost
- timeout
```

设计原则：

- 可执行 verifier 优先；
- 结果奖励与过程奖励结合；
- 稀疏奖励可用分阶段 shaping；
- reward scale 做归一化；
- 延迟奖励要做 credit assignment；
- 安全硬约束不能被其他正奖励抵消；
- 防止长度、重复调用和格式投机。

### 4. 跨境支付调查 Agent 示例

**状态**

```text
用户意图
授权商户
订单 id
订单/通道/账务状态
已查证据
剩余工具预算
风险等级
```

**动作**

```text
查询订单
查询通道
查询账务
检索 SOP
要求补材料
生成解释
转人工
结束
```

**奖励**

```text
+ 正确解决问题
+ 状态和工具一致
+ 正确引用
+ 必要时转人工
- 编造状态
- 重复无效调用
- 泄露信息
- 越权退款/出款
```

真实资金系统不能作为自由探索环境，应使用沙箱、历史回放或仿真。

### 5. 如何验证设计

- 随机策略和规则 baseline；
- reward 与人工/业务 KPI 相关性；
- adversarial 测试；
- reward 高但任务失败的样本；
- 动作分布和循环率；
- OOD 状态；
- safety violation；
- 消融每个 reward 分量。

### 6. 面试总结

状态回答“模型知道什么”，动作回答“模型能做什么”，奖励回答“什么才算好”。三者必须与真实环境、权限和业务目标一致，尤其要防信息泄漏、不可执行动作和 reward hacking。
