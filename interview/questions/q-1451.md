# 介绍GRPO、PPO、DPO，分别需要几个模型，需要训练的是哪些

## 答案

模型数量应区分“逻辑角色”和“是否独立占一份完整权重”。通过参数共享、LoRA、vLLM rollout、预计算 log-prob 等方式，物理显存不一定等于逻辑模型数。

### 1. PPO

经典 RLHF PPO 有四个逻辑角色：

| 角色 | 是否训练 |
|---|---|
| Policy/Actor | 是 |
| Reference Policy | 否，冻结 |
| Reward Model | 否，PPO 阶段冻结 |
| Value/Critic | 是 |

流程：

```text
Policy rollout
-> Reward Model 打分
-> Critic 估计 value
-> GAE/advantage
-> PPO clipped objective 更新 Actor
-> value loss 更新 Critic
-> KL 约束 Reference
```

Actor 和 Critic 有时共享 backbone 加不同 head，但逻辑职责仍不同。

### 2. GRPO

典型 GRPO：

| 角色 | 是否训练 |
|---|---|
| Policy | 是 |
| Reference Policy | 否，冻结 |
| Reward Model/Verifier | 通常否，可能是规则 |
| Critic | 不需要 |

同一 prompt 采样一组回答，根据组内 reward 均值和标准差计算相对 advantage，因此省去独立 value model。

若使用程序 verifier，reward 不一定是一个神经网络模型。部分实现也可使用不同 KL 估计或无独立 reference 配置，应以具体算法为准。

### 3. DPO

标准 DPO 有：

| 角色 | 是否训练 |
|---|---|
| Policy | 是 |
| Reference Policy | 否，冻结 |

输入是：

```text
prompt, chosen, rejected
```

DPO 直接比较 Policy 相对 Reference 对 chosen/rejected 的 log-prob，不需要显式 Reward Model、Critic 或在线 rollout。

Reference log-prob 可预先计算，训练时不一定持续驻留一份完整模型；也有 reference-free 变体，但标准 DPO 逻辑上使用 Reference。

### 4. 对比

| 算法 | 逻辑角色 | 训练对象 | 是否在线采样 |
|---|---|---|---|
| PPO | Actor、Ref、RM、Critic | Actor+Critic | 是 |
| GRPO | Policy、Ref、Reward/Verifier | Policy | 是 |
| DPO | Policy、Ref | Policy | 否 |

### 5. 选择

- PPO：控制精细，但系统复杂、显存和训练成本高；
- GRPO：可验证任务和多样本相对奖励，省 Critic，但 rollout 仍贵；
- DPO：偏好数据充足时简单稳定，但无法在线探索新轨迹。

### 6. 跨境支付场景

工具调用可用规则 verifier 做 GRPO；客服表达偏好可用 DPO；高风险资金场景不适合无约束在线探索。所有算法都必须配合权限、工具沙箱和独立安全评测。

### 7. 面试总结

经典 PPO 四个角色、训练 Actor 和 Critic；GRPO 去掉 Critic，训练 Policy；DPO 只需 Policy 与冻结 Reference，训练 Policy。回答时要说明逻辑模型和物理部署可通过共享与预计算优化。
