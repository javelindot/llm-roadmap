# LLM推理的时候，如何从prompt到response？

## 答案

LLM 从 prompt 到 response 可以分为请求处理、prefill、逐 token decode 和后处理四个阶段。

### 请求与分词

服务端先应用 chat template，将 system、user、assistant 消息拼成模型要求的格式，然后 tokenizer 将文本编码为 token id。系统会检查上下文窗口、截断策略、最大输出长度和停止条件，并将请求加入调度队列。

### Prefill

所有 prompt token 一次送入模型：

1. token id 查表得到 embedding。
2. 每层通过归一化、带因果 mask 的自注意力和 FFN。
3. RoPE 等位置编码作用于 Q/K。
4. 每层历史 token 的 K、V 保存到 KV Cache。
5. 最后一个位置经 LM head 得到下一 token 的词表 logits。

Prefill 计算量随输入长度增长，矩阵计算并行度高，通常偏算力受限。长 prompt 会同时增加首 token 延迟和 KV Cache 占用。

### Decode

对 logits 应用温度、重复惩罚、top-k/top-p 或贪心策略，选出一个 token。随后只把这个新 token 输入模型，并读取历史 KV Cache 计算它对所有历史位置的注意力，再产生下一个 token。该循环持续到生成 EOS、命中 stop sequence、达到长度限制或请求被取消。

Decode 每步只处理一个新 token，但要读取各层 KV Cache，通常更受显存带宽影响。推理框架会用 continuous batching、PagedAttention、prefix cache 和量化等方法提高并发。

### 后处理

tokenizer 将 token 增量解码为文本，服务端过滤特殊 token、检测停止字符串并流式返回。最后记录 token 数、延迟、完成原因和缓存占用，并释放 KV Cache。生产系统还会在前后增加鉴权、内容安全、超时、限流和可观测性逻辑。
