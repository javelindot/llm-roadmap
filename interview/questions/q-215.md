# 自主抉择 Agent 的工作流具体如何实现？

## 答案

> 我们用 **LangGraph 状态图 + 条件边 + LLM 决策**实现，以风控调查 Agent 为例。
>
> **整体架构**：
> ```
>      ┌──────────────┐
>      │  收到工单     │
>      └──────┬───────┘
>             ▼
>      ┌──────────────┐
>      │ 工单分类      │ ← LLM 决策
>      └──────┬───────┘
>             ▼
>     ┌───────┴────────┬─────────────┐
>     ▼                ▼             ▼
>  [简单查询]      [可疑交易]     [其他]
>     │                │             │
>     ▼                ▼             ▼
>  [查订单]        [拉链路]      [转人工]
>     │                │
>     │                ▼
>     │           [检查证据是否充分] ← LLM 决策
>     │                │
>     │           ┌────┴─────┐
>     │           ▼          ▼
>     │       [继续查]    [出报告]
>     │           ↑          │
>     │           └──────────┘
>     │                       │
>     └──────────► [输出最终结果]
> ```
>
> **核心代码框架**（LangGraph 风格）：
> ```python
> from langgraph.graph import StateGraph, END
>
> class State(TypedDict):
>     ticket: dict
>     evidence: list
>     iteration: int
>     conclusion: Optional[str]
>
> def classify(state: State) -> dict:
>     # LLM 决策：工单类型
>     return {"ticket_type": llm.classify(state["ticket"])}
>
> def gather_evidence(state: State) -> dict:
>     # 拉链路 + 查关联
>     new_evidence = tool.query_related(state["ticket"])
>     return {"evidence": state["evidence"] + new_evidence,
>             "iteration": state["iteration"] + 1}
>
> def is_evidence_enough(state: State) -> str:
>     # LLM 自主决策
>     if state["iteration"] >= 5:
>         return "give_up_to_human"
>     decision = llm.evaluate_evidence(state["evidence"])
>     return "conclude" if decision.is_enough else "gather_more"
>
> graph = StateGraph(State)
> graph.add_node("classify", classify)
> graph.add_node("gather", gather_evidence)
> graph.add_node("conclude", make_conclusion)
> graph.add_conditional_edges(
>     "gather", is_evidence_enough,
>     {"conclude": "conclude", "gather_more": "gather", "give_up_to_human": END}
> )
> ```
>
> **关键设计点**：
>
> **1. 状态可序列化**
> 整个 state 是 TypedDict，能 dump 到 Redis，崩了能从中间恢复。
>
> **2. 决策点显式 LLM 调用**
> 不是"让 LLM 想想再说"，而是显式 prompt："以下证据是否足以判断是否可疑？输出 yes/no/need_more"。
>
> **3. 步数限制**
> `iteration` 字段 hard limit 5 次，否则强转人工。
>
> **4. HITL 节点**
> 关键决策（最终判定为可疑）必须有人工 review 节点，不能完全自主。
>
> **5. trace 全留**
> 每个节点的输入输出 + LLM 决策原文都进 trace 系统，供监管审计。
>
> **典型一次跑通**：
> ```
> Tick: 工单 #20260601-RISK-001 进入
> Node: classify     → 类型 = 可疑交易（信心 0.92）
> Node: gather       → 拉一跳关联，发现共享设备 ID
> Node: is_enough?   → LLM 决策：need_more
> Node: gather       → 拉二跳关联，发现共同收款方
> Node: is_enough?   → LLM 决策：yes
> Node: conclude     → 结论：疑似多账户洗钱
> Node: hitl         → 等待人工确认
> ```
>
> **几个工程经验**：
> - **决策点用小模型够用**：Qwen-7B 判断"证据充分否"准确率 91%，省成本
> - **关键路径冗余**：评估证据必须用两次不同 prompt 双独立 LLM 判定，一致才走下去
> - **每个 trace 入数据库**，方便 case study
> - **失败 case 月度复盘**，迭代决策 prompt
>
> 上线后单工单调查时间从 30 分钟降到 3 分钟，人工只复核 LLM 结论。
