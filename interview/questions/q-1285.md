# 训练时为什么要mask,推理时也需要吗?

## 答案

Mask 不是一种单一机制。训练时常见的 causal mask、padding mask、loss mask 解决不同问题；推理时是否需要，要看模型结构、批处理方式和所处阶段。

### 1. Causal Mask：防止看到未来 token

Decoder-only 模型训练时一次并行输入整段序列。如果位置 `t` 能看到 `t+1` 之后的 token，就会发生标签泄漏，所以 self-attention 只能关注当前位置及之前：

```text
M[i,j] = 0       if j <= i
M[i,j] = -inf    if j > i

Attention(Q,K,V)
= softmax(QK^T / sqrt(d) + M) V
```

训练虽是并行计算，语义上仍保持自回归：

```text
p(x) = product_t p(x_t | x_<t)
```

**推理阶段仍需要因果约束。**

- Prefill 一次处理整个 Prompt，必须使用 causal mask；
- Decode 每次只生成一个新 token，未来 token 尚不存在，因果性天然成立；
- 但实现中仍会通过 attention kernel、KV Cache 长度或显式 mask 保证新 query 只能访问有效历史位置。

因此不能简单说“推理不需要 mask”，更准确的说法是 decode 阶段常把约束隐含在缓存位置和 kernel 中。

### 2. Padding Mask：屏蔽补齐位置

不同样本长度不同时，batch 通常补到统一长度。Padding token 不应参与注意力：

```text
attention_mask = 1 for valid token
attention_mask = 0 for padding token
```

训练和批量推理都可能需要 padding mask。若单请求无 padding，或推理框架使用 paged/ragged batch 只记录真实长度，可以不构造完整二维 mask，但等价的长度约束仍存在。

要特别注意 left padding 与 right padding、position_ids 和 KV Cache 的对应关系，否则批量生成可能发生位置偏移。

### 3. Loss Mask：决定哪些 token 参与监督

SFT 常把 Prompt 和 Response 拼在同一序列中，但只训练回答部分：

```text
labels:
system/user/padding -> -100
assistant response  -> target token id
```

`-100` 通常表示交叉熵忽略该位置。这样模型不会因为重复预测用户输入而稀释回答目标。

推理没有训练 loss，所以不需要 loss mask。它与 attention mask 不能混淆：

- attention mask 决定 token 能看见谁；
- loss mask 决定哪些位置计算损失。

### 4. Packed Sequence 和文档边界

为了提高训练利用率，可能把多个短样本拼进一个 sequence。此时仅有 causal mask 不够，后一个样本会看到前一个无关样本，需要 block-diagonal/document mask，或使用支持 sequence boundary 的 packed attention。

跨境支付数据中，不能让不同商户工单在 packing 后互相可见。即使只是训练样本，也要先脱敏并保证样本边界不会造成跨租户上下文污染。

### 5. 不同模型的区别

- GPT/Decoder-only：causal mask + 可选 padding mask；
- BERT/Encoder：双向注意力，一般只有 padding mask，MLM 的 `[MASK]` 是输入破坏策略，不等同 attention mask；
- Encoder-Decoder：Encoder 通常双向，Decoder 使用 causal mask，Cross-Attention 再屏蔽 source padding；
- Prefix LM：前缀内部可双向，生成区保持因果，需要定制 mask。

### 6. 常见故障

- mask shape 广播错误；
- FP16 中用不合适的极小值导致 NaN；
- padding token 同时被当成有效标签；
- FlashAttention 的 boolean mask 语义与上层框架相反；
- KV Cache 后 position_ids 未递增；
- packed sample 未隔离；
- 训练模板和推理模板不同。

验证时应做小矩阵可视化，并比较单条与批量、无 padding 与有 padding、全量计算与 KV Cache 增量生成的 logits。
