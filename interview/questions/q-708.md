# Prompt engineering的方法？

## 答案

Prompt Engineering 的本质是明确任务、提供必要上下文并约束可验证输出。应从简单稳定的方法开始，再根据失败类型增加复杂度。

### 常用方法

- 清晰指令：说明目标、受众、边界、禁止项和成功标准。
- 角色与上下文：给出必要领域背景，但不要用角色描述替代具体要求。
- Few-shot：提供代表性正例、边界例和格式示例，避免示例偏置。
- 分解任务：先抽取、再分析、最后生成，或使用显式工作流。
- Structured Output：JSON Schema、枚举、模板与 constrained decoding。
- 检索增强：只注入相关证据，要求引用并在证据不足时拒答。
- Self-consistency、reflection、critic：适用于高价值复杂任务，但增加成本，且自我检查不保证正确。
- Tool use：让模型输出结构化函数调用，由代码执行和校验。

### 优化流程

先建立固定评测集和基线，按错误类型分析是信息不足、指令冲突、示例不当、检索失败还是模型能力不足。每次只改一类因素，版本化 prompt，并比较任务成功率、格式通过率、事实性、延迟和 token 成本。

### 安全与边界

系统约束与用户内容分隔，对外部文档按不可信输入处理。权限、业务规则和高风险操作必须由代码层保证，不能只写在 prompt 中。

Prompt 不能补足缺失知识或突破模型能力上限。当模板越来越复杂仍不稳定时，应考虑 RAG、工具、微调、约束解码或更明确的状态机。
