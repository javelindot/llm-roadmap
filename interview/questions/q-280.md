# 你做的Agent工作？LangGraph项目？简单介绍下具体实现。

## 答案

**一句话**：我在 Agent 工作里主要负责把大模型能力工程化，包括流程编排、工具调用、状态管理、RAG 接入、保护节点和效果评估。

### 1. 可以怎么介绍 LangGraph 项目

LangGraph 项目可以这么讲：

- 用图结构把 Agent 流程显式化。
- 每个节点负责一个明确动作，比如意图识别、检索、工具调用、答案生成、校验。
- 用 state 在节点之间传递信息。
- 用条件边决定下一步走向。
- 用 checkpoint 保存中间状态，支持恢复和排查。

相比简单 chain，Graph 更适合有分支、有循环、有工具、有人工节点的复杂流程。

### 2. 具体实现

可以描述一个典型流程：

1. 用户提问进入入口节点。
2. 意图识别节点判断是知识问答、订单查询、异常解释还是投诉。
3. 如果需要知识，走 RAG 检索节点。
4. 如果需要实时数据，走工具调用节点。
5. 决策节点判断信息是否足够。
6. 答案生成节点生成回复。
7. 安全校验节点检查敏感信息和事实一致性。
8. 低置信度或高风险问题转人工。

### 3. 我的职责怎么说

可以说：

- 设计 Agent 状态结构。
- 封装工具调用和参数校验。
- 设计图节点和条件路由。
- 接入 RAG 检索。
- 做日志 trace 和评测集。
- 优化延迟、准确率和安全边界。

### 4. 结合跨境支付项目怎么说（面试口径）

> 在跨境支付客服项目里，我做的 Agent 工作主要是把用户问题处理流程图形化。比如用户问“这笔付款为什么失败”，入口节点先识别这是交易异常；然后订单查询节点查交易状态；如果状态是通道失败，就查通道回执和错误码；如果状态是风控拦截，就查风控大类和处理 SOP；最后答案节点生成用户能理解的解释。
>
> 我用 LangGraph 的原因是这类流程不是简单一问一答，而是有分支、有工具、有保护节点。比如查不到订单要追问订单号，命中高风险要转人工，工具超时要走兜底回复。
>
> 我个人更关注的是状态设计和安全边界。Agent state 里会记录当前问题、订单信息、工具结果、风险标记、已走节点和置信度，方便后续回放和排查。
