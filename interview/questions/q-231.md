# 线上频繁 Full GC 怎么办？

## 答案

> Full GC 频繁是个**症状**，背后原因有十几种。我按"**先止血、再定位、最后根治**"三步走。
>
> **第一步：先止血（保业务稳定）**
> - 临时扩容（多副本）分流
> - 触发滚动重启（重启完前几分钟通常 Full GC 少）
> - 限流非核心接口
> - 把可疑的大对象生成接口暂时关闭
>
> **第二步：定位根因**
>
> 收集证据：
> - `jstat -gcutil <pid> 1000`：实时看各代占用、GC 频次
> - `jmap -histo:live <pid>`：看堆里对象分布
> - **dump 堆**：`jmap -dump:format=b,file=heap.bin <pid>`（先扩容再 dump，dump 期间服务卡顿）
> - GC 日志：`-XX:+PrintGCDetails -Xloggc:gc.log`
> - **MAT / VisualVM** 分析 dump，找 retained heap 大的对象
>
> 常见根因分类：
>
> **1. 内存泄漏**
> - 业务对象进入老年代后无法释放
> - 典型：缓存（如 Caffeine 没设过期）、ThreadLocal、监听器没移除、ClassLoader 泄漏
> - dump 看 dominator tree，找 retained 异常大的实例
>
> **2. 大对象分配**
> - 一次性加载大文件、大 List
> - 直接进老年代，导致老年代快速填满
> - 优化：分批处理、流式读取
>
> **3. 元空间满**
> - 动态生成 class 过多（反射、CGLIB、热部署）
> - JDK 8+ 是 Metaspace，无限制时会用满
> - 设 `-XX:MaxMetaspaceSize`
>
> **4. GC 参数不合理**
> - 新生代太小，对象提前晋升老年代
> - Survivor 区比例不对
> - GC 算法选错（CMS 已废弃，用 G1 / ZGC）
>
> **5. 并发流量陡增**
> - 短时请求量翻倍，对象堆积速度超过 Minor GC 处理能力
> - 优化：限流 + 异步化
>
> **6. JVM bug**
> - 极少但有，看 JDK release notes
>
> **第三步：根治**
>
> 按定位的根因做：
>
> - **泄漏类**：修代码，加单元测试覆盖
> - **大对象类**：流式处理、分批、外存
> - **参数类**：调 GC（G1 默认 ok，超大堆用 ZGC）
> - **流量类**：扩容 + 限流 + 缓存
>
> **JVM 参数推荐（生产）**：
> ```
> -Xmx8g -Xms8g                 # 堆固定大小
> -XX:+UseG1GC                  # G1 收集器
> -XX:MaxGCPauseMillis=200      # 目标 STW
> -XX:G1HeapRegionSize=16m
> -XX:MetaspaceSize=256m
> -XX:MaxMetaspaceSize=512m
> -XX:+HeapDumpOnOutOfMemoryError
> -XX:HeapDumpPath=/tmp/heap.bin
> -Xloggc:/var/log/gc.log
> -XX:+PrintGCDetails
> -XX:+PrintGCDateStamps
> -XX:+UseGCLogFileRotation
> -XX:NumberOfGCLogFiles=10
> -XX:GCLogFileSize=100m
> ```
>
> **我们跨境支付的实际案例**：
>
> 我们 Java 风控网关跑了一段时间后频繁 Full GC，5 分钟一次。
>
> 定位：dump 后发现一个 `Map<String, RuleConfig>` 几十万条，是从配置中心拉的规则配置，**每次更新都新增不删旧**，老年代被吃满。
>
> 解决：换成 Caffeine 缓存（带过期 + size limit），Full GC 间隔从 5 分钟拉到 1 天，业务也不感知。
>
> 一句话：**Full GC 频繁是症状，根因 90% 在代码（泄漏 / 大对象），10% 在参数**——别一上来就调 GC，先 dump 看对象分布。
