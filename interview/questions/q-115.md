# LLM 有哪些缺点

## 答案

**一句话**：LLM 缺点可归纳为**七宗罪 + 三个时代痛点**——幻觉、贵、慢、私、偏、忘、笨；外加可解释性差、长期记忆缺、自洽性差。

### 1. 七宗罪（详见 q-55）
1. **幻觉**：自信地胡说
2. **贵**：训练/推理成本高
3. **慢**：首 token + decode 速度
4. **私**：数据安全
5. **偏**：训练数据偏见
6. **忘**：上下文长度限制
7. **笨**：多步推理不稳

### 2. 三个时代新痛点（2024-2026 暴露的）

**可解释性差**
- 黑盒决策，业务/监管要求"为什么这么说"
- 风控、医疗、法务无法接受
- 治理：CoT 显示推理 + reference citation + audit log

**长期记忆缺失**
- 每次会话独立，上次说过的它不记得
- 治理：外部记忆系统（向量库 + 摘要 + 检索）

**自洽性差**
- 同一问题问两次答案可能不一样（temperature）
- 业务侧"为什么昨天答这个今天答那个"
- 治理：cache + 固定 seed + 评测稳定性

### 3. 工业落地额外的实际痛点
- **难评测**：开放生成无客观指标，靠 human eval / LLM judge
- **难复现**：模型版本、prompt 微调、temperature 都影响
- **法律风险**：训练数据版权、生成内容责任
- **依赖关键供应商**：闭源 API 厂商一改价就被动
- **对齐脆弱**：jailbreak 攻击层出不穷

### 4. 几条治理对策（系统化）
- **RAG**：治幻觉、补私域知识
- **工具调用**：治计算/查询能力短板
- **Constitutional AI**：治偏见和对齐脆弱
- **蒸馏 + 量化**：治贵和慢
- **本地部署 + 脱敏**：治私
- **多模型 fallback**：治依赖单一供应商

### 5. LLM 不适合做什么（重要的反向认知）
- 强可解释决策（用规则引擎）
- 简单结构化任务（用传统 ML）
- 实时低延迟（<10ms）
- 严格确定性输出（数学/财务计算）
- 高频小量请求（成本扛不住）

### 6. 一句话总结
**LLM 缺点本质是"概率模型 + 数据驱动"的副作用**——治理路径只有两条：**接外部约束**（RAG/工具/规则）和**对齐少犯错**（SFT/RLHF/Constitutional）。

## 结合项目谈（口语版 · 约 30 秒）

> 这道题我会和 q-55 一起谈，但侧重不同——q-55 谈"什么缺点"，这里更侧重"哪些缺点让我们在跨境支付里痛苦"。
>
> 三个最痛的：
>
> **第一是可解释性**。监管要求每条风控决策可追溯——为什么把这笔交易判为可疑、依据哪条规则。LLM 直接给"这笔交易疑似洗钱"我们没法用。后来我们强制让 LLM **输出完整 CoT + 引用具体字段证据**，每个结论都对应一段推理 + 证据链。这是把"可解释"做成硬要求。
>
> **第二是自洽性**。客服模型同一个问题问两次回答不一样，客户拿截图来投诉"你昨天明明答应给我退款的"。我们做了三件事：高频问题答案 cache、temperature=0.1（不是 0 因为 0 会复读）、关键政策类回答必须从知识库引用。这才把投诉降下来。
>
> **第三是法律和合规**。我们不能用 GPT-4 处理客户隐私（数据出境合规），不能蒸馏 GPT-4 训商业模型（版权风险），不能让 LLM 自主决策大额操作（责任问题）。这些**不是技术缺点，是工业落地的政治正确**——所有方案都得绕过这些。
>
> 一句话：**LLM 在工业里被"业务/合规/客户预期"三道墙夹击**，技术缺点反而是最容易解的。

**🪝 顺嘴带出去的钩子**：
- "可解释性怎么做成硬要求我们有方案可以聊。"
- "客服自洽性问题的工程治理可以展开。"
- "金融行业 LLM 合规链路是个大话题可以讲讲。"
