# 抽取式摘要和生成式摘要存在哪些问题

## 答案

**一句话**：抽取式摘要"**精准但呆板**"、生成式摘要"**流畅但易幻觉**"——两者各有痛点，工业里常用"**抽取定边界 + 生成做润色**"的混合方案。

### 1. 抽取式摘要（Extractive）
**思路**：从原文挑句子直接组合

**优点**：
- 内容 100% 来自原文，**无幻觉**
- 可解释（直接看挑了哪句）
- 算法简单（TextRank、LexRank、BERT 排序）

**问题**：
- **连贯性差**：拼接的句子缺乏过渡
- **冗余**：相邻句子可能讲同一事
- **覆盖不全**：散布在多段的信息难整合
- **信息密度有限**：受原文句子表达能力约束
- **多文档难处理**：句子来自不同文档难融合
- **代词指代失效**："这家公司"脱离上下文不知道指谁

### 2. 生成式摘要（Abstractive）
**思路**：用模型重新生成新句子

**优点**：
- **连贯流畅**
- **信息整合能力强**：可以跨段融合
- **可定制风格、长度**
- 多文档天然支持

**问题**：
- **幻觉**：可能编造原文没有的信息
- **事实性错误**：数字、人名、时间张冠李戴
- **可解释差**：不知道每句来自哪
- **难评测**：客观指标缺
- **格式不稳定**：偶尔输出不符合要求
- **依赖训练数据**：风格被训练集固化

### 3. 评测难点（公共）
- ROUGE / BLEU 只看 n-gram 重合
- 业务实际看"事实正确 + 关键点覆盖"
- 人工评 + LLM-as-judge 是主流

### 4. 工业混合方案
- **抽取定关键句** → 生成润色重写
- **生成 + 引用 enforcement**：每句话必须带 citation
- **生成 + 后处理 verification**：用 NLI 模型检查每句是否被原文支持

### 5. LLM 时代摘要的新挑战
- 长文档（百页报告）：超出上下文
- 多文档融合：观点冲突怎么处理
- 实时性：每天新增 doc 增量摘要
- 多语言：跨语言摘要

### 6. 工程经验值
- 摘要长度：原文的 10-30%
- 关键点保留：人工标 5-10 个 must-have
- 评测：ROUGE + LLM-judge + 人工抽检
- 生成温度：0-0.3（事实性优先）

### 7. 一句话总结
**抽取式不幻觉但不灵活、生成式灵活但易幻觉**——LLM 时代生成式占主流，但事实性敏感场景必须配 citation + verification。

## 结合项目谈（口语版 · 约 25 秒）

> 我们跨境支付里**摘要场景挺多**：客服会话摘要、风控调查报告、合规变更摘要、对账差异摘要。两种方案我们都用过。
>
> **场景一：客服会话摘要（生成式）**——客户和客服 10-20 轮对话，业务侧要一句"这次客服解决了什么问题、处理结果"。用 Qwen-14B 生成式摘要，加了"必须包含订单号、问题类型、处理结果三个字段"的结构化约束，效果稳定。
>
> **场景二：风控调查报告（抽取 + 生成 混合）**——监管要求每条结论可追溯到原始证据。我们的做法是：抽取式找出关键证据句（带原始来源）+ 生成式做总结性结论，**结论部分必须 cite 证据 ID**。这是抽取保事实、生成保流畅的混合。
>
> **场景三：合规政策变更摘要（生成式 + Verification）**——每月各国监管发布新政策，我们用 Qwen-72B 生成摘要给业务团队。**生成完用 NLI 模型逐句检查**是否被原文支持，不被支持的句子标红人工审核。这是"生成 + 事实校验"的工程模式。
>
> 几个工业坑：
> - 生成式默认 temperature 太高会幻觉，要 0-0.2；
> - 多文档摘要要先抽取关键点再融合，否则信号被稀释；
> - LLM-judge 评摘要质量比 ROUGE 准得多。
>
> 一句话：**事实性敏感场景生成式必须配 citation + verification**，不能"裸用"。

**🪝 顺嘴带出去的钩子**：
- "NLI 事实校验在摘要里的具体落地可以聊。"
- "抽取 + 生成混合方案在金融报告生成上的设计。"
- "多文档摘要的观点冲突处理可以展开。"
