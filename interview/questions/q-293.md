# 调过 RAG 吗？还是直接使用市面上的 API？

## 答案

> **从头自研，不用现成 API**——金融场景数据合规决定了我们**不能用 OpenAI / Anthropic / 阿里云百炼等 API**。我详细讲一下选型理由。
>
> **市面上 RAG API 的几大类**：
>
> 1. **国外 SaaS**：OpenAI Assistants API、Claude RAG API
> 2. **国内云厂**：阿里云百炼、华为云盘古、火山方舟
> 3. **垂直产品**：Pinecone（向量库）、Cohere（embedding + rerank）
> 4. **开源开箱**：RAGFlow、Dify、Coze
>
> **为什么我们没用**：
>
> **1. 数据合规（最致命）**
> - 客户数据不能出境（不能用 OpenAI / Anthropic）
> - 不能给第三方厂商（即使是阿里云 / 华为云这种"国内"API，跨境支付数据也要在自家私有云）
> - 监管要求模型可审计 / 可解释 / 可追溯
>
> **2. 业务定制**
> - 我们有 5 个独立知识库 + 业务路由
> - 多轮 query 改写、length-balanced DPO、引用强约束……都是业务特有
> - 现成 API 提供"通用 RAG"，调不出这些
>
> **3. 成本**
> - 闭源 API：每千 token 几分钱，我们每天千万级 token，月烧百万
> - 自建：GPU 折旧 + 人力 ~ 月 50w，长期划算
>
> **4. 可控性**
> - API 厂商改价、改规则、改模型版本，我们被动
> - 自建可以版本锁定、A/B 测试、灰度
>
> **5. 数据沉淀**
> - 用 API 等于把所有业务 prompt 和反馈数据交给第三方
> - 自建数据沉淀在自家，是核心资产
>
> **我们自研用了哪些开源组件**：
>
> 详见 q-173，简单说：
> - 检索层：Milvus + Elasticsearch
> - Embedding：bge 系列（业务精调）
> - Rerank：bge-reranker
> - LLM：Qwen-14B（本地 vLLM）
> - 编排：LangGraph
> - 评测：RAGAS（魔改）
>
> **业务层 70% 自研**：路由、引用强约束、多轮改写、失败回流、HITL、监控……都是自己写。
>
> **什么情况会考虑 API**：
>
> 1. **POC / 原型**——快速验证想法，可以先调 API
> 2. **非核心业务**——比如内部工具，不涉及敏感数据
> 3. **小公司 + 无算力**——自建门槛高，API 起步快
> 4. **国际化业务**——海外节点可能要用当地 SaaS
>
> 但对**金融生产业务来说，自研几乎是唯一选项**。
>
> **我们的"自研但不重复造轮子"原则**：
>
> - 通用组件用开源（Milvus、bge、vLLM）
> - 业务逻辑自写（路由、评测、回流）
> - 关键能力可控（base 模型、知识库、prompt）
> - 监控 / 治理自建
>
> 30% 时间在选 / 调开源、70% 时间在写业务逻辑。
>
> **如果非要用 API，我会怎么评估**：
>
> 1. 数据合规过不过得了？
> 2. 业务定制空间够不够？
> 3. SLA 承诺？
> 4. 厂商锁定风险？
> 5. 三年总成本对比自建？
>
> 一句话：**金融场景必须自研，自研不等于从零造**——开源组件用足、业务逻辑写透、合规与成本双 win。
