# RequestBody 调用 AI 过长了怎么办？

## 答案

> 先分清楚"过长"是什么意思——是**网络层（HTTP body 过大）**、**模型层（超出 context window）** 还是**业务层（token 成本太贵）**，三者解法不同。
>
> **1. 网络层过长（HTTP 413 / 414）**
>
> - 调整反向代理（nginx）`client_max_body_size`
> - 调整 FastAPI `limit_concurrency`、`max_request_size`
> - 用 **流式上传** 而非一次性大 body
> - 大文本上传走 **预签名 OSS URL**，POST 只传 object_key
>
> **2. 模型 context window 超长（model 报错或质量下降）**
>
> **首选：截断 + 摘要**
> - 旧消息滚动摘要（详见 q-225）
> - 长文档先切块再 RAG 检索，只传 top-K
> - 保留头尾 + 摘要中段（lost-in-the-middle 缓解）
>
> **进阶：分段调用**
> - Map-Reduce：每段独立调用 + 最后聚合
> - 适合摘要、信息抽取、批量分类
>
> **结构性压缩**
> - 删冗余字段（page_url、created_at 这种）
> - 表格转 CSV（比 Markdown 表节省）
> - JSON → 紧凑形式
>
> **升级长上下文模型**
> - Qwen-14B 默认 32k，配 YaRN 可外推到 128k
> - 切到 Qwen-72B / DeepSeek-V3 这种原生 128k 模型
> - 但**成本翻倍**，慎用
>
> **3. 业务层成本过贵**
>
> - prompt cache：固定 system prompt 走 Anthropic / OpenAI 的 cache 折扣
> - few-shot 改 fine-tune：把示例训进模型，prompt 短了上限不掉
> - 蒸馏到小模型：详见 q-81
> - 分级模型：简单查询走小模型、复杂走大模型（路由）
>
> **我们跨境支付的具体处理**：
>
> **风控调查 Agent**：原始 prompt 经常 30k+
> - 关联交易压缩到摘要态（LLMLingua）
> - 历史只保留近 5 笔原文
> - 用 Qwen-14B + YaRN 外推到 32k 兜底
> - 最终 prompt 控制在 8k-12k
>
> **客服多轮**：超长对话
> - 滚动摘要前面对话，最近 3 轮保留
> - system prompt 用 prompt cache
> - 控制在 4k 以内
>
> **长 PDF 文档 RAG**：
> - 离线切块、不直接进 prompt
> - 用户查询时 RAG 召回 top-5
> - 控制在 2k 以内
>
> **几个实战经验**：
> - **不能盲目截断**——关键信息（订单号、金额、规则）要硬保留
> - **报错优先级**：先解决"能跑通"（截断），再优化"跑得好"（压缩）
> - **监控 token 分布**：p95 token 数 + 超限率，设阈值告警
> - **业务侧降级**：长 prompt 触阈值时拒绝服务比硬塞强（OOM 比慢更糟）
>
> 一句话：**Body 过长不是单一问题**——网络层调配置、模型层做压缩、业务层算 ROI；优先把"能跑通"做好，再考虑质量。
