# 处理长文档PDF时，采用了哪些长上下文扩展方法或技术？

## 答案

处理长 PDF 的首要问题通常不是模型窗口，而是版面解析、表格/OCR、层级结构、版本和权限。可靠方案是“结构化解析 + 分层索引/RAG + 按需长上下文”，而不是把整份 PDF 原文直接塞给模型。

### 1. 解析层

先判断 PDF 类型：

- 文本型：提取文字、字体、坐标和阅读顺序；
- 扫描型：OCR + 版面检测；
- 图文混排：图片、图注与邻近段落关联；
- 表格：保留行列、合并单元格和表头；
- 多栏：按版面重建阅读顺序；
- 页眉页脚：去重但保留页码。

输出结构化节点：

```text
Document
-> Section
-> Paragraph/Table/Figure
-> Page/BBox/Source
```

### 2. 结构化分块

- 按标题层级、条款、段落和表格切分；
- 小 chunk 用于召回，大 parent 用于生成；
- 注入文档名、章节路径、页码和版本；
- 表格按业务行组切，重复表头；
- 跨页段落合并；
- overlap 只用于自然边界不可靠的文本。

### 3. 分层检索

```text
Query
-> 文档/章节路由
-> BM25 + Dense
-> Metadata Filter
-> Rerank
-> Parent Expansion
-> Context Packing
```

复杂问题可先检索章节摘要，再进入局部细节；多跳问题拆为子问题并合并证据。

### 4. 长上下文技术

当问题确实需要跨很多章节时，可使用：

- 原生长上下文模型；
- RoPE scaling/YaRN 等模型配套能力；
- FlashAttention；
- chunked prefill；
- prefix cache；
- PagedAttention/KV Cache 量化；
- context/sequence parallel；
- Map-Reduce/Tree Summary。

必须验证 Lost-in-the-Middle、不同位置证据、跨页引用和 TTFT。只修改最大长度配置没有意义。

### 5. 上下文压缩

- 先抽取与问题相关句；
- 去重重复定义；
- 表格转结构化 JSON/Markdown；
- 章节摘要保留引用；
- 对多个证据按时间和权威性排序；
- 关键金额、日期和条款原文保留。

### 6. 跨境支付案例

KYC 手册、通道规则和卡组织拒付指南可能数百页。应先按国家、产品、规则版本和生效日期过滤，再召回具体条款。模型输出必须引用页码；费率、订单和实时状态仍从系统工具获取。

### 7. 评测

- OCR/表格字段准确率；
- 章节与证据 Recall@K；
- 引用页码正确率；
- 跨页/多跳问题准确率；
- 长度分桶和证据位置分桶；
- P95 TTFT、token 和成本；
- 过期版本与越权文档拦截率。

**面试总结**：长 PDF 是文档工程问题优先、模型窗口问题其次。先把内容解析成可检索、可引用的证据单元，再按需使用长上下文。
