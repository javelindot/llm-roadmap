# SFT 模型出现各种复读问题如何解决？

## 答案

**一句话**：复读 90% 是**数据问题**，10% 是解码问题——所以解法要从数据 → 训练 → 解码 → 对齐**四层一起治**，光改 temperature 是治标。

### 1. 数据层（根本解）
- **n-gram 去重**：训练集做 3-gram / 5-gram 去重，模板化样本砍掉
- **多样性增强**：长度分布、风格分布要均衡
- **清洗低质量样本**：爬虫数据里 "点击查看更多 点击查看更多 …" 这种直接 drop
- **避免格式偏置**：所有样本"很高兴为您服务！"开头 → 模型学到这是必须的

### 2. 训练层
- **学习率**：过小容易塌陷到记忆模式，过大不收敛——典型 2e-5
- **Early stopping**：训过头会过拟合到训练数据的复读模式
- **样本打包（packing）**：长序列拼接时 attention mask 要正确，否则模型学到 "结束后继续重复"
- **Loss masking**：只对 assistant token 算 loss，user/system 不算

### 3. 解码层（应急解）
| 参数 | 作用 | 建议值 |
|------|------|--------|
| `temperature` | 采样温度 | 0.7（贪心 = 0 几乎必复读） |
| `top_p` | 核采样 | 0.9 |
| `top_k` | 截断 | 40-50 |
| `repetition_penalty` | 重复词惩罚 | 1.1-1.3 |
| `no_repeat_ngram_size` | n-gram 禁止 | 3-4 |
| `frequency_penalty` | 频率惩罚 | 0.5-1.0 |
| `presence_penalty` | 出现惩罚 | 0.3-0.6 |

### 4. 对齐层（DPO / RLHF）
- 把"复读回答"作为 rejected 样本喂 DPO
- RLHF 阶段 reward 里加入"是否复读"的惩罚项
- 效果最持久但成本最高

### 5. 复读类型 → 对应解
| 现象 | 主因 | 解 |
|------|------|----|
| 一句话内 token 复读 | 解码 temperature 太低 | 加 repetition_penalty |
| 段落级模板复读 | 训练数据模板化 | 数据去重 + 多样化 |
| 长生成尾部退化 | 长上下文外推不稳 | 限长 + 截断标点 |
| 同一问不同问都复读 | 过拟合 / 数据污染 | 减 epoch + 数据清洗 |

### 6. 一句话总结
**先看复读类型再开方**——token 级看解码、模板级看数据、退化级看训练。光调 temperature 是退烧不治病。

## 结合项目谈（口语版 · 约 40 秒）

> 这个我们跨境支付太有体会了。我们做的是**智能客服回复生成**，第一版 SFT 上线之后客户和运营都来吐槽——"每条回复都是'尊敬的客户，您好！'开头"、"末尾都是'如有疑问请联系客服'"、严重的还有"很高兴为您服务！很高兴为您服务！"句内复读。
>
> 我们一开始也是先调解码——repetition_penalty 拉到 1.2、temperature 从 0 调到 0.7、加 no_repeat_ngram_size=3，句内复读基本止住了，但开头结尾的模板化还在。
>
> 然后回头去查数据，发现训练集 5w 条客服话术里，**90% 是历史模板回复**，几乎每条都是同一种格式开头。模型学得太"好"了。我们做了两件事：一是按 3-gram 去重，砍掉 30% 高度重复样本；二是增加多样化样本——口语化回复、短回复、带表情的回复，混进去 1w 条。
>
> 重新训之后模板化消失，加上 DPO 二轮（把模板复读作为 rejected），最终模型回复多样性大幅提升，运营那边客户满意度提了 8 个点。
>
> 总结一句：**复读基本都是数据病，解码只能压不能治。**

**🪝 顺嘴带出去的钩子**：
- "n-gram 去重的具体做法和阈值选择我们有经验。"
- "DPO 治模板化复读的样本怎么构造可以聊聊。"
- "客服回复多样性怎么量化评测，我们做了一套指标。"
