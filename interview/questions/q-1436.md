# 为什么在参数高效微调中使用低秩矩阵分解？

## 答案

参数高效微调使用低秩矩阵分解，是基于一个经验假设：预训练模型虽然参数很多，但适配具体下游任务所需的权重更新往往位于较低维的子空间，不需要自由更新整个大矩阵。

### 1. LoRA 形式

原权重：

```text
W in R^(d_out x d_in)
```

冻结 `W`，只学习：

```text
Delta W = B A
A in R^(r x d_in)
B in R^(d_out x r)
r << min(d_in, d_out)
```

前向：

```text
y = W*x + (alpha/r) * B*A*x
```

### 2. 参数为什么大幅减少

全量更新需要：

```text
d_out * d_in
```

LoRA 只需：

```text
r * (d_in + d_out)
```

例如方阵维度很大而 rank 较小时，可训练参数、梯度和优化器状态都会显著减少。

### 3. 为什么低秩可能有效

- 预训练模型已具备大部分通用能力；
- 下游任务常是调整行为、格式和领域偏好；
- 不同参数更新存在相关性；
- 实际有效更新具有较低 intrinsic dimension；
- 低秩约束也形成一定正则，减少小数据过拟合。

这是一种经验假设，不保证所有任务都严格低秩。

### 4. 工程优势

- 显存低；
- 训练快；
- checkpoint 小；
- 多个任务可保存不同 adapter；
- 基座权重共享；
- 部署时可动态挂载；
- 合并到权重后通常不增加额外推理层。

### 5. 局限

- rank 太小可能欠拟合；
- 大规模知识注入和能力改变可能需要更高 rank 或全参训练；
- target modules 选择影响很大；
- 多 adapter 管理复杂；
- 量化基座的训练稳定性需要验证；
- LoRA 并不自动避免灾难性遗忘和数据问题。

### 6. 跨境支付场景

LoRA 适合学习客服格式、领域术语、工具调用和安全话术；实时汇率、费率、订单状态和监管更新不应通过增大 rank 写进模型，而应由 RAG、规则和工具提供。

### 7. 面试总结

低秩分解用小矩阵近似大权重更新，利用下游适配更新的低维结构，以少量参数换取接近全参微调的效果。它是资源和效果折中，不是所有任务都能无损替代全量微调。
