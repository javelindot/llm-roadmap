# 如何看待各种pporlhf的平替算法dpo/kto/rrhf/slic/orpo/simpo

## 答案

**一句话**：DPO、KTO、RRHF、SLiC、ORPO、SimPO 这些方法本质上都是为了降低 PPO/RLHF 的工程复杂度，用离线偏好数据或更简单的目标函数做对齐，但它们不是完全替代 PPO，而是适合不同数据条件和业务场景。

### 1. PPO/RLHF 的问题

传统 PPO/RLHF 通常需要 policy model、reference model、reward model、value model，训练链路复杂、成本高、调参难，还容易 reward hacking。

所以后面很多方法都在解决一个问题：能不能不用在线 RL，直接用偏好数据把模型调好？

### 2. 常见平替算法

| 方法 | 核心思路 | 适合数据 |
|------|----------|----------|
| DPO | 直接优化 chosen 相对 rejected 的概率偏好 | 成对偏好数据 |
| KTO | 不要求成对，只需要知道回答是 desirable 还是 undesirable | 单条好/坏标签 |
| RRHF | 对多个候选回答按排序学习 | 多候选排序数据 |
| SLiC | 用监督学习 + 对比约束拉开好坏回答 | 偏好对或排序数据 |
| ORPO | 把 SFT 和偏好优化合在一个目标里，不依赖 reference model | SFT 数据 + 偏好信号 |
| SimPO | 不用 reference model，直接用平均 logprob 和 margin 做偏好优化 | 成对偏好数据 |

### 3. 怎么看待

- **DPO**：最常用，简单稳定，但依赖高质量偏好对。
- **KTO**：数据要求更低，适合只有正负反馈的场景。
- **RRHF**：适合一个 query 有多个候选排序的场景。
- **ORPO/SimPO**：进一步简化训练链路，但要注意长度偏置和泛化。
- **PPO**：仍然适合需要在线探索、工具使用、多轮决策、复杂 reward 的场景。

### 4. 面试总结

这些方法的方向都是"用更简单的离线优化替代复杂在线 RL"。选哪个不是看论文名，而是看你手里有什么数据、业务是否需要探索、能不能接受训练复杂度。

## 结合项目谈（口语版 · 面试现场约 55 秒）

> 我会从数据形态来选这些算法。比如跨境支付里，如果我们有人工审核留下来的成对样本，像一个回答能清楚解释风控命中原因，另一个回答只会模板化拒绝，那 DPO 就很合适。如果只有单条反馈，比如运营点了"有帮助"或"没帮助"，没有严格 chosen/rejected，那 KTO 可能更自然。
>
> 如果一个商户审核问题下，我们让模型生成多个候选解释，再由风控同学排序，这种就可以考虑 RRHF。ORPO 和 SimPO 的吸引力在于链路更简单，不用像 PPO 那样维护 reward model、value model，工程成本低很多。
>
> 但我不会说这些一定完全替代 PPO。支付场景如果只是优化风控解释、审核话术、合规问答，我会优先用 SFT + DPO/ORPO 这类离线方法；如果要优化多步工具调用、动态策略选择，甚至模拟交易处置流程，PPO 或类似在线强化学习才更有价值。
