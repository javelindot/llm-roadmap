# 项目细节:证明Cold-Start SFT后再RL的模型比只RL效果好

## 答案

严格来说，不能靠一句“论文里说有效”来证明。应该做受控消融实验：同一个基座模型、同一批训练 prompt、同一套 reward、同样训练预算，只改变是否经过 Cold-Start SFT。

### 1. 实验分组

至少设计两组：

```text
A: Base Model -> RL
B: Base Model -> Cold-Start SFT -> RL
```

如果资源允许，再加：

```text
C: Base Model -> SFT only
D: Base Model -> random/low-quality SFT -> RL
```

这样可以区分“确实是高质量冷启动有效”，还是“多训练了一段所以有效”。

### 2. 保持变量一致

需要控制：

- 基座模型 checkpoint；
- RL 训练 prompt 分布；
- rollout 数量和最大 token；
- reward model/verifier；
- 学习率、batch size、KL 系数、clip 参数；
- 训练步数或总 token 预算；
- 评测集和解码参数；
- 随机种子，最好多 seed 重复。

否则结论不可信。

### 3. 看哪些指标

不只看最终分数，还要看训练过程：

- **样本效率**：达到同一 reward 需要多少 rollout；
- **最终效果**：held-out pass rate、任务成功率、人工偏好胜率；
- **稳定性**：reward 曲线是否震荡、KL 是否爆炸、entropy 是否过早崩塌；
- **格式和可读性**：是否能稳定输出业务需要的结构；
- **安全性**：是否出现越权承诺、编造状态、绕过规则；
- **泛化性**：新国家、新币种、新通道、新错误码上的表现。

跨境支付场景里，我会特别看：

```text
订单状态解释准确率
工具调用参数准确率
错误码归因准确率
合规拒答率
敏感信息泄露率
人工复核通过率
```

### 4. 为什么 Cold-Start SFT 可能有效

RL 的探索空间很大。如果一开始模型连基本格式、业务术语、工具调用 schema 都不稳定，RL 很容易把奖励浪费在无效输出上。Cold-Start SFT 的作用是：

- 给模型一个可读、可解析的初始策略；
- 降低无效 rollout 比例；
- 让模型先学会业务边界；
- 减少 RL 初期 reward hacking；
- 让后续偏好优化更像“精修”，而不是从零摸索。

例如支付助手如果没经过 SFT，可能连 `payment_order_id`、`payout_order_id`、`chargeback_id` 都混用；先做 SFT 可以让 RL 更关注“解释是否准确、是否合规、是否更有帮助”。

### 5. 不能绝对化

也要承认边界：Cold-Start SFT 不一定总优于纯 RL。如果基座模型已经很强、任务 reward 很可验证、采样预算足够，纯 RL 也可能有效；如果 SFT 数据质量差，还可能把模型带偏。因此面试里应说“通过消融证明在我的任务分布上有效”，不要说“永远更好”。

### 6. 面试总结

证明方式是受控实验和消融：Base+RL 对比 Base+Cold-Start SFT+RL，统一 reward、预算和评测集，看最终效果、训练稳定性、样本效率和安全指标。跨境支付中，Cold-Start SFT 的价值主要是让模型先学会业务术语、工具格式和合规边界，再用 RL 优化偏好和复杂场景决策。
