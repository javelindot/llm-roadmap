# 大模型sft解决不了业务问题吗?为什么还要引入强化学习。强化学习主要解决业务中的什么问题。

## 答案

SFT 能解决大量业务问题，尤其是格式、风格、工具协议和标准流程。引入强化学习不是因为 SFT“没用”，而是当目标难以写成唯一标准答案、需要在线探索，或最终结果可验证但优秀过程不唯一时，RL 才可能提供额外收益。

### 1. SFT 擅长什么

SFT 最大化示范答案的 token likelihood：

```text
L_SFT
= -sum_t log pi_theta(y_t | x, y_<t)
```

适合：

- 学习对话和报告格式；
- 工具名、参数 Schema；
- 标准客服流程；
- 领域术语和话术；
- 已有高质量专家轨迹的任务。

它稳定、可控、成本相对低，通常是第一选择。

### 2. SFT 的局限

- 只模仿数据中的一条或少量路径；
- 对“多个答案都可以但优劣不同”利用不足；
- 错误示范会被直接学习；
- 无法主动探索比示范更好的长程策略；
- token loss 与最终任务成功不完全一致；
- 长 CoT 中每个 token 等权，不知道哪一步真正导致结果正确。

### 3. RL 主要解决什么

RL 直接优化序列或任务奖励：

- 数学答案、代码测试等可验证正确性；
- Agent 最终任务完成率；
- 多步工具顺序和成本；
- 多目标偏好，如正确、简洁、安全；
- 从当前策略采样并探索新的有效轨迹。

例如同一个对账问题可以有多种查询顺序，只要金额、币种和账期最终核验正确，就可以给正奖励，不要求复制唯一参考答案。

### 4. 业务中的正确顺序

```text
Prompt/Workflow 基线
-> SFT 学基本行为
-> DPO 学离线偏好
-> RL 优化可验证任务
```

不要一开始就上 RL。模型连工具格式都不会时，rollout 大多无效，奖励信号很差。

### 5. 跨境支付

适合 RL 的任务：

- SQL/代码执行结果；
- 订单查询工具参数；
- 对账金额是否一致；
- 流程是否按规则完成；
- 在保证成功的前提下减少冗余调用。

不适合只靠 RL 的任务：

- AML 最终定性；
- 制裁误报豁免；
- 冻结、退款和调账审批；
- 开放式监管解释。

这些需要确定性规则、证据和人工责任链。

### 6. RL 的代价

- rollout 成本高；
- reward hacking；
- 训练不稳定；
- OOD 能力退化；
- 奖励定义可能固化错误业务目标。

**面试总结**：SFT 教“怎么做”，偏好优化教“哪种更好”，RL 在可验证环境里优化“最终是否成功”。多数业务先把前两层做好。
