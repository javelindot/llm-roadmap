# 如何理解稠密模型&混合专家模型

## 答案

**一句话**：稠密模型每次推理都会激活全部参数，MoE 模型有多个专家，每个 token 只路由到部分专家，参数大但计算可控。

### 核心回答
- **稠密模型 Dense**：
  - 所有参数都参与计算。
  - 结构简单、训练稳定、部署相对容易。
  - 推理成本随参数量直接增加。
- **MoE（Mixture of Experts）**：
  - 有多个专家网络和路由器。
  - 每个 token 只激活 top-k 专家。
  - 总参数量大，但每次计算只用一部分。
  - 难点是路由均衡、通信成本、训练稳定和部署复杂。

MoE 的价值是用更大的参数容量换更低的单次计算量。

### 1. 稠密模型和 MoE 的核心差异

稠密模型每个 token 都会经过同一套参数，所以部署和训练相对直接。MoE 则有多个专家，每个 token 通过 router 选择少数几个专家参与计算。

可以这样理解：

| 模型类型 | 特点 | 优势 | 难点 |
|----------|------|------|------|
| 稠密模型 | 每次激活全部参数 | 稳定、简单、易部署 | 参数越大计算越贵 |
| MoE | 每次只激活部分专家 | 总容量大，单次计算可控 | 路由、负载均衡、通信复杂 |

### 2. MoE 的工程问题

MoE 看起来很美，但上线不简单：

- router 可能把流量打到少数专家，造成负载不均。
- 多机部署时专家通信复杂。
- 小 batch 下专家利用率可能不稳定。
- 监控和排障比稠密模型难。

### 3. 面试总结

MoE 的本质是"稀疏激活的大容量模型"。它不是免费变强，而是在模型容量、计算成本和工程复杂度之间做交换。

### 4. 结合跨境支付项目怎么说（面试口径）

> 在跨境支付业务里，如果只是做单一垂直助手，比如内部客服知识库或风控解释，稠密模型更容易部署和稳定调优。它简单、可控，出了问题也容易定位。
>
> 但如果公司有多业务、多语言、多国家合规场景，MoE 的思路就很有吸引力。比如有些专家更擅长风控，有些专家更擅长合规条款，有些专家更擅长多语言客服。
>
> 只是我会谨慎落地 MoE，因为支付系统重稳定性。MoE 的路由、专家负载、跨机通信都会影响延迟和可用性。除非容量收益明显，否则我会优先用稳定稠密模型加 RAG 和工具。

**钩子**：MoE 适合多领域大容量，但业务落地要权衡部署复杂度和稳定性。
