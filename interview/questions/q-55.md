# 大模型（LLMs）的缺点？

## 答案

**一句话**：LLM 的核心缺点可以归到**七宗罪**——幻觉、贵、慢、私、偏、忘、笨——每一条都对应明确的工程治理手段。

### 1. 幻觉（Hallucination）
- 自信地胡说，把不知道的也编出来
- 治理：RAG 接外部知识、引用要求、置信度校准、对齐阶段加 reject

### 2. 贵（Cost）
- 训练贵：百万到亿美金
- 推理贵：长上下文 + 高 QPS 烧钱
- 治理：模型蒸馏、量化、KV Cache、prompt cache、混合架构（小模型兜底 + 大模型托底）

### 3. 慢（Latency）
- 首 token 慢（prefill）、生成慢（decode 串行）
- 治理：vLLM、speculative decoding、量化、prompt 压缩

### 4. 私（Privacy）
- 上下文里塞的数据可能被记忆、被泄露
- 治理：PII 脱敏、本地部署、不送外部 API、审计日志

### 5. 偏（Bias）
- 训练数据偏见 → 输出偏见（性别、种族、文化、语言）
- 治理：数据清洗、对齐 + Constitutional AI、安全测试

### 6. 忘（Forget / 上下文限制）
- 上下文窗口限制（虽然变长，但仍有上限）
- 长对话遗忘前文
- 治理：摘要、向量检索、记忆系统、Context Engineering

### 7. 笨（Reasoning 局限）
- 多步推理脆弱、数学算错、计数失败
- 不会真的"理解"，靠模式匹配
- 治理：CoT、工具调用（calculator、code interpreter）、Self-Consistency、verifier

### 8. 工业落地的"潜在第八宗罪"
- **不可预测**：温度、prompt 微小变化引发输出剧变
- **难评测**：开放生成的好坏没有客观指标
- **法律风险**：版权、合规、可追溯性

### 9. 一句话总结
**LLM 缺点的本质是"概率模型 + 数据驱动"两个性质带来的**——所以治理路径只有两条：**接外部约束**（RAG/工具/规则）和**对齐"少犯错"**（SFT/RLHF/Constitutional）。

## 结合项目谈（口语版 · 约 40 秒）

> 这道题我会挑我们跨境支付里最真实踩过的三个坑讲。
>
> **第一是幻觉**——我们客服模型早期会编造规则。客户问"信用卡退款多久到账"，模型回答"3-5 个工作日"——但我们业务实际是 7-15 天。客户拿这个答案去投诉，我们差点担责。后来我们硬性要求所有回答**必须从知识库引用**（RAG），不在知识库里的问题模型必须说"我帮您转人工"。这是把"不胡说"做成了硬约束。
>
> **第二是贵和慢**——风控调查 Agent 一开始用 GPT-4 跑全流程，单工单成本 0.5 元、p95 延迟 8 秒。一天几十万工单根本顶不住，运营也吐槽慢。我们做了三件事：一是 7B 蒸馏取代 GPT-4，二是 prompt cache + KV cache 优化，三是 speculative decoding。最后单工单 0.001 元、p95 1.5 秒，才上得了生产。
>
> **第三是私**——客户数据绝对不能送外部 API，我们一律本地部署 + 私有化模型。这意味着我们没法直接用 GPT-4，只能用开源 + 自训。这是合规和能力的痛苦权衡。
>
> 我面试会强调一句：**LLM 缺点不可怕，可怕的是不知道哪条缺点会卡死你的业务**——每条缺点都对应明确的工程化方案，但 ROI 各不相同，要算清账再上。

**🪝 顺嘴带出去的钩子**：
- "幻觉怎么治我们有 RAG + 引用强约束的方案可以聊。"
- "GPT-4 → 7B 蒸馏的成本/质量 trade-off 我们有数据。"
- "合规和能力怎么权衡是个产品决策可以展开。"
