# 训练完验证使用的是什么数据集

## 答案

这是项目真实性与评测规范题。必须回答实际使用的数据集名称、规模、来源、切分方式和指标；不能只说“用了内部数据和公开数据”。

### 1. 推荐回答结构

```text
公开 benchmark
+ 业务自建集
+ 时间外测试集
+ 安全/对抗集
```

示例：

> 训练过程中使用独立 validation set 做超参数选择和 early stopping，最终结果只在从未参与调参的 test set 上报告。业务数据按时间切分，训练截止到 T1，验证使用 T1-T2，测试使用 T2 以后数据；同一用户/商户/文档簇不会跨集合泄漏。公开集只用于衡量通用能力，核心结论来自与线上分布一致的自建评测集。

数字和数据集名称必须替换成真实内容。

### 2. 数据集如何设计

- **Validation**：调超参、选 checkpoint、early stopping；
- **Test**：训练方案冻结后只做最终评估；
- **Temporal holdout**：验证概念漂移和未来数据；
- **OOD set**：新国家、新语言、新产品或异常长度；
- **Adversarial/safety set**：越权、提示注入、隐私和边界案例；
- **Regression set**：线上历史高风险故障，版本迭代必须通过。

若是 RL，还应把训练 prompt、reward 开发集和最终评测 prompt 隔离，防止 verifier 或 reward 过拟合。

### 3. 防止数据泄漏

- 文档、近重复样本和同源模板去重；
- 按用户、商户或会话分组切分；
- 训练数据不能包含测试答案或未来标签；
- 合成数据要追踪生成模型和来源；
- RAG 测试要冻结知识库版本，或明确测试更新策略；
- 不用 test set 反复选模型。

### 4. 跨境支付场景

业务验证集应覆盖国家、币种、支付方式、商户规模、多语言和异常链路，并按时间切分。主指标之外还应单测金额/订单号准确性、引用支持率、权限泄漏、制裁与高风险拒答。

订单状态等动态事实应通过快照或模拟权威系统评测，不能让模型凭参数记忆答案。

### 5. 面试总结

高质量回答至少包含“数据集名称与来源、样本规模、切分方法、泄漏控制、主要指标和长尾切片”。Validation 用于选择，Test 用于最终报告，两者不能混用。
