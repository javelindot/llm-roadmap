# 多Agent系统，怎么设置失败回滚机制？大概4、5个方面。

## 答案

答：5层保障
状态快照：每步执行前保存完整状态；
写前日志：记录所有操作，可追踪可恢复；
补偿事务：对已执行操作做反向撤销；
重试与降级：短暂失败自动重试，严重失败回滚；
原子步骤：保证要么全成功，要么全回滚。

### 补充说明

多 Agent 回滚要区分“读操作”和“写操作”。读操作失败一般可以重试、降级或换工具；写操作必须有事务、幂等、补偿和人工确认。

比较完整的机制包括：

- 每一步保存状态快照和 trace。
- 写操作前记录 write-ahead log。
- 工具调用必须幂等。
- 外部副作用用补偿动作处理。
- 超过重试次数进入降级或人工。
- 主 Agent 汇总子 Agent 状态，统一决定是否继续。

### 结合跨境支付项目怎么说（面试口径）

> 在跨境支付多 Agent 场景里，回滚机制非常关键。比如主 Agent 调度风控 Agent、通道 Agent、账务 Agent 处理一笔异常付款。查询类操作失败可以重试或降级，但如果涉及重新发起付款、退款、冻结解冻，就必须严格控制。
>
> 我会让每个关键步骤执行前保存状态快照，写操作前记录日志，并要求工具幂等。比如退款请求不能因为 Agent 重试就退两次；通道路由建议不能直接变成真实资金动作，必须经过保护节点和人工确认。
>
> 如果某个子 Agent 失败，主 Agent 根据状态决定回滚到上一个安全点、换备用工具、转人工或结束流程。支付系统里不能让一个子 Agent 的异常影响全局资金状态。

**钩子**：可以引到“读写分离、幂等、补偿事务、资金动作人工确认”。
