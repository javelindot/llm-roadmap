# XGBoost和LightGBM的区别及其并行化方法

## 答案

XGBoost 和 LightGBM 都是 GBDT：每一轮拟合当前残差/梯度，因此 Boosting 轮次具有顺序依赖，不能把所有树完全并行训练。二者主要差异在分裂搜索、树生长策略、采样与分布式通信。

### 1. 分裂算法

**XGBoost**

- 支持 `exact、approx、hist` 等 tree method；
- 现代版本默认主要使用 histogram 路线；
- `hist` 先把连续值量化为 bins，再累计梯度/二阶梯度直方图；
- 支持稀疏感知、外存、CPU/GPU 和分布式训练。

**LightGBM**

- 原生以 histogram 为核心；
- 使用 histogram subtraction，只构建较小叶子的直方图，另一叶由父直方图相减得到；
- GOSS 对大梯度样本保留更多、小梯度样本采样；
- EFB 将互斥稀疏特征绑定，降低有效特征数。

GOSS/EFB 是 LightGBM 论文的重要设计，但实际是否启用和收益取决于版本、接口与参数。

### 2. 树生长

- XGBoost 默认常用 depth-wise，逐层扩展；也支持 `lossguide`；
- LightGBM 使用 leaf-wise/best-first，每次分裂增益最大的叶子；
- 相同叶子数下 leaf-wise 通常更快降低 loss；
- 小数据上更容易过拟合，需要 `num_leaves、max_depth、min_data_in_leaf` 等约束。

### 3. 类别与缺失值

两者都能学习缺失值默认方向。LightGBM 长期支持类别特征的原生处理；XGBoost 现代版本也支持 categorical data，因此不能再绝对地说“XGBoost 必须 one-hot”。仍需检查数据接口、版本和生产导出兼容性。

### 4. 单机并行

树轮次串行，但一棵树内部可并行：

- 多线程构建不同特征/数据分片的 histogram；
- 并行评估 split gain；
- 并行数据加载、量化和预测；
- GPU 并行处理直方图与分裂。

XGBoost 使用 `nthread/n_jobs` 控制线程；LightGBM 使用 `num_threads`。线程数过高会因 NUMA、内存带宽和过度订阅变慢，需要实测。

### 5. 分布式并行

**数据并行**

- 各 worker 持有不同样本；
- 本地构建梯度直方图；
- AllReduce/ReduceScatter 聚合全局直方图；
- 同步最佳分裂。

**特征并行**

- worker 负责不同特征；
- 各自寻找局部最佳 split；
- 通信得到全局最佳 split。

**Voting Parallel**

LightGBM 使用两阶段投票减少候选特征与直方图通信，适合数据和特征都很大的场景。

XGBoost 可通过 Dask、Spark 等运行分布式 `hist`，GPU 场景常使用 NCCL 通信。`exact` 不适合分布式扩展。

### 6. 如何选

| 场景 | 倾向 |
|---|---|
| 大规模稀疏、高维、追求训练速度 | 优先实测 LightGBM |
| 需要丰富约束、生态和跨平台部署 | 优先实测 XGBoost |
| 小数据 | 两者都需强正则，不能只看训练分 |
| GPU/分布式 | 按数据格式、集群和通信压测 |

不能用“LightGBM 一定更快、XGBoost 一定更准”作结论。应在相同特征、时间预算和验证切分下比较 AUC/PR-AUC、校准、训练时间、峰值内存、推理延迟和稳定性。
