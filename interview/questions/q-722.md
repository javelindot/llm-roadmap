# 介绍一下MoE和极化问题的解决方法

## 答案

MoE 将 Transformer 的稠密 FFN 替换为多个 Expert，并用 Router 为每个 token 只激活 top-k 个专家。这样总参数容量可以很大，但单 token 计算量接近少数专家。

### 基本计算

Router 输出：

$$
p(x)=softmax(W_rx)
$$

选择 top-k 专家后：

$$
y=\sum_{e\in TopK(x)}p_e(x)f_e(x)
$$

训练和部署还需 all-to-all 通信、expert parallel、容量管理和 token dispatch。

### 极化/专家坍缩

若 Router 早期偏向少数专家，热门专家得到更多数据、训练更好，进一步吸引 token，形成正反馈。结果是负载不均、溢出丢 token、专家利用率低，甚至不同专家学不到差异。

### 缓解方法

- Load-balancing auxiliary loss：约束专家接收 token 比例和 router probability 更均衡。
- Router z-loss：限制 router logits 过大，提高数值稳定性。
- Noisy routing、温度或随机探索：训练早期避免过早确定。
- Capacity factor 与 token dropping/rerouting：限制单专家容量，溢出 token 转发或保留残差路径。
- 合理初始化、较小 router 学习率、expert dropout。
- Shared experts：承载通用知识，routed experts 专注差异，减轻所有 token 争抢少数专家。
- 无辅助损失的动态 bias 等负载策略，降低 auxiliary loss 对主任务的干扰。

监控每个专家 token 数、负载 CV、丢弃率、router entropy 和领域分布。完全均匀也不一定最优，目标是在不堵塞通信的前提下形成有意义的专家专门化。
