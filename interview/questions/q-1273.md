# 讲实习中的部署优化

## 答案

部署优化应先用 profiling 找瓶颈，再从模型、服务、推理 Kernel 和业务架构逐层处理。跨境支付应用的目标不是单请求最快，而是在质量和安全不回退的前提下满足并发、P95/P99 和成本。

### 1. 建立基线

固定模型、硬件和数据分布，记录：

- TTFT、TPOT、总延迟；
- P50/P95/P99；
- 输入/输出 token；
- QPS 和并发；
- GPU 利用率、显存和功耗；
- prefill、decode、调度、网络和工具耗时；
- 单个成功任务成本。

使用真实的短问答、长 RAG、工具调用和多轮请求分桶，不能只跑一个固定 Prompt。

### 2. 模型与精度

- 按任务路由 7B/14B/更强模型；
- BF16、FP8、INT8、GPTQ/AWQ 做量化对比；
- LoRA Adapter 合并或高效批量管理；
- 控制上下文与输出长度；
- 蒸馏高频分类、Query rewrite 和 rerank 到小模型。

所有变更必须回归业务黄金集和安全集。

### 3. 推理引擎

使用 vLLM/TensorRT-LLM 等引擎获得：

- continuous batching；
- Paged KV Cache；
- FlashAttention/fused attention；
- tensor/pipeline parallel；
- prefix caching；
- chunked prefill；
- CUDA Graph；
- speculative decoding；
- 量化 Kernel。

是否有效取决于模型、硬件、batch 和输入输出长度，不能只根据功能列表判断。

### 4. KV Cache 与并发

- 按最长上下文和并发计算 KV 预算；
- 限制异常长请求；
- 系统 Prompt 使用 prefix cache；
- GQA/MLA 模型减少 KV；
- 必要时 KV FP8/INT8；
- 长短请求分队列，避免长 prefill 阻塞短请求；
- 设置 admission control，防止显存 OOM。

### 5. 服务层

- 异步 API 与流式返回；
- 动态 batching；
- 模型预热和健康检查；
- 限流、超时、取消和熔断；
- 多副本负载均衡与容量弹性；
- 主模型失败时降级小模型/FAQ/人工；
- Prompt、模型和 Adapter 版本化回滚。

### 6. RAG 与工具链路

支付请求的延迟不只来自 LLM：

- BM25/Dense 并行召回；
- 初召回先过滤再 rerank；
- 缓存公共知识，但缓存键包含租户、权限和知识版本；
- 订单、通道工具并行并设置 deadline；
- 区分工具超时和业务无数据；
- 低风险标准问题直接走模板或小模型。

### 7. 观测与验收

每个请求记录 trace：路由、检索、工具、模型、token、cache hit 和降级原因。优化后同时比较质量、事实性、工具成功、安全、P99、吞吐和成本。

部署优化的核心成果应报告“什么瓶颈、做了什么、在何种负载下改善多少”。没有真实测量时不报数字，但要说明压测方法和上线门槛。
