# Agent 如何定时工作？

## 答案

**一句话**：Agent 自己不会"定时"，定时是**外部调度系统把 Agent 唤起来**——所以这题考的是"调度怎么接、上下文怎么传、幂等怎么保"。

### 1. 三种触发模型
| 触发方式 | 工具 | 场景 |
|---------|------|------|
| 时间触发 | cron、APScheduler、Airflow、Celery beat、K8s CronJob | 每天对账、每 5min 风扫 |
| 事件触发 | Kafka、webhook、DB CDC | 新交易落库就跑 |
| 条件触发 | Prometheus 报警、阈值监控 | 异常率超 5% 拉起调查 Agent |

### 2. 实现形态
- **一次性 Run**：调度器拉起一个无状态 Agent，跑完即销毁，最简单。
- **长驻 Agent + 心跳**：Agent 自己挂着，被定时事件唤醒（Claude Code 的 `ScheduleWakeup`、`CronCreate` 就是这种）。
- **状态机式**：用 LangGraph / Temporal 把"等 5 分钟"做成一个节点，可中断、可恢复。

### 3. 关键工程问题
- **幂等**：定时任务最大的坑就是重复执行——同一条交易扫两遍、对账单发两次。务必带业务唯一键去重。
- **超时与重试**：策略要写在 Agent 外部（调度器层），别让 Agent 自己重试。
- **上下文注入**：定时 Run 没有人工对话，必须在系统提示里说清楚"你现在为什么被叫起来、要干什么、产出是什么"。
- **观测**：每次 run 留 trace、token 消耗、产出，方便追责和复盘。
- **冷启动**：定时跑没有用户在场，输出失败也要有告警，不能默默死。

### 4. 选型建议
- 简单：cron + 脚本拉起 Agent。
- 复杂工作流：Airflow / Temporal / Prefect。
- 想用 Claude Code 自己内置的：`CronCreate`（远程定时 agent）/ `ScheduleWakeup`（dynamic loop 自己回唤）。

### 5. 一句话总结
**定时 Agent 真正难的不是"怎么定时"，是"被叫起来这一次知道自己要干啥、不会干重、出问题能被发现"**。

## 结合项目谈（口语版 · 约 40 秒）

> 定时这块我们跨境支付里用得挺多。最典型的几个场景：每天 T+1 早上 6 点跑全量清算对账、每 5 分钟扫一遍可疑交易、商户 KYB 证件过期前 7 天自动提醒续签、还有夜里 2 点跑反洗钱规则的批量回溯。
>
> 我们一开始图省事直接 cron + 脚本调 Agent，跑着跑着出过一次事故——网络抖动 cron 重跑了一次，结果同一批可疑工单被生成了两遍，运营那边炸了。后来加了两个东西：一是任务唯一键（业务日期 + 任务名）去重，二是给每个定时 Agent 配一个 trace_id 落库，谁、什么时候、消耗多少 token、产出什么，全留底。
>
> 还有一个坑是上下文——定时跑没人陪聊，系统提示必须把"你现在为啥被叫起来"写清楚。我们专门做了个"定时任务上下文模板"，把任务背景、时间窗、产出格式都注进去，Agent 才不会跑偏。
>
> 复杂一点的我们后来切到 Temporal，因为有些任务要"跑一步等 30 分钟看结果再决定下一步"，cron 玩不转。

**🪝 顺嘴带出去的钩子**：
- "定时 Agent 幂等这块我们踩过坑，可以展开聊业务唯一键设计。"
- "我们从 cron 切到 Temporal 的过程也挺有意思的，选型可以聊。"
- "定时任务的 trace 我们和业务系统打通做了一个 Agent 行为审计大盘。"
