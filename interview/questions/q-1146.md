# 你说你会做上下文记忆优化，怎么做的？

## 答案

上下文记忆优化的目标不是保存更多历史，而是在当前 token、延迟和隐私预算内，选择真正有助于本轮任务的信息。

> 工作记忆 → 历史压缩 → 长期记忆检索 → 权威事实查询 → 上下文编排 → 写入与遗忘。

### 1. 先区分不同类型的记忆

**短期工作记忆**

当前会话的最近消息、意图、实体、工具结果和任务状态，通过 thread/checkpoint 保存。

**语义记忆**

稳定事实和偏好，例如商户常用币种、语言偏好和已开通产品。

**情景记忆**

过去一次问题如何解决、哪些步骤失败，用于相似案例和流程优化。

**程序记忆**

SOP、Prompt、Skill 和工具使用规则，由版本化配置或知识库管理。

**权威业务事实**

余额、订单、汇率、通道和风控状态不属于自由记忆，必须实时查询业务系统。

### 2. 短期上下文压缩

使用“最近窗口 + 结构化状态 + 历史摘要”：

- 保留最近几轮原始消息；
- 提取订单号、币种、意图、已确认事实和未解决问题；
- 较早对话压缩成结构化摘要；
- 工具结果只保留必要字段和引用；
- 长文档证据保存 chunk 引用，需要时重新检索。

摘要必须区分“用户陈述”“系统确认事实”和“模型推断”，防止推断被压缩后变成永久事实。

### 3. 动态 token 预算

为不同区域分配预算：

> 系统策略、当前问题、最近对话、任务状态、长期记忆、RAG 证据、工具结果、输出空间。

按任务动态调整。简单 FAQ 不加载完整历史；付款失败调查保留订单和工具状态；复杂报告为证据留更多 token。

超预算时优先删除重复闲聊、过期工具结果和低相关历史，不能删除权限、风险提示和关键业务状态。

### 4. 长期记忆的写入门禁

不是每轮对话都写长期记忆。写入前判断：

- 内容是否稳定、未来是否有用；
- 来源是否可信；
- 是否包含 PII 或敏感交易数据；
- 是否已有重复或冲突记忆；
- 用户是否允许保存；
- TTL 和删除条件是什么。

保存：

> `namespace、memory_type、content、source、confidence、created_at、expires_at、version`

跨境支付中不保存完整卡号、证件、账户凭证、具体风控规则和一次性交易状态。

### 5. 长期记忆检索和重排

先用 tenant/user namespace 做硬过滤，再按：

> `relevance + recency + importance + confidence - conflict_penalty`

进行召回和重排。只取少量记忆，并明确标记为“用户偏好”或“历史经验”，不能覆盖实时业务系统。

发生冲突时，新记忆不能直接覆盖旧记忆，应保存版本、来源并根据事实类型决定更新或人工确认。

### 6. 防止记忆污染和串台

- 用户输入、网页和工具文本不自动写入长期记忆；
- 提示注入内容不得改变写入策略；
- tenant、merchant 和 user 多级隔离；
- 记忆写入、读取和删除都有审计；
- 用户切换角色或商户后重新建立上下文；
- 支持按用户请求删除和过期清理；
- 使用 canary 和越权测试验证跨租户隔离。

### 7. 缓存与记忆分开

缓存解决重复计算，记忆保存对未来任务有用的信息：

- embedding、公共 RAG 结果和 Prompt 前缀可缓存；
- 订单、余额和实时报价使用短 TTL 或不缓存；
- 公共缓存键包含权限和知识版本；
- 缓存失效不等于删除长期记忆。

### 8. 评测

- 指代消解和多轮任务成功率；
- 记忆召回 Precision/Recall；
- 摘要事实保持率和实体一致性；
- 无关记忆进入上下文的比例；
- 跨租户泄漏率和敏感信息写入率；
- token、TTFT、P95 和单次成本；
- 旧信息导致错误回答的比例；
- 用户删除请求的完整执行率。

LangGraph 当前文档将短期记忆作为 thread state/checkpoint，将长期记忆组织在 namespace/key 的 store 中，并支持语义搜索。这种分层方式适合工程实现，但具体写入策略、隐私边界和事实优先级仍需业务系统定义。
