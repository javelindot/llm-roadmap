# 当输入文本长度超出模型的上下文窗口时，有哪些主流的处理方案或模型架构来应对？

## 答案

输入超过上下文窗口时，不能直接全部送入模型。应根据任务选择截断、分块检索、摘要、层级处理、外部记忆或长上下文模型，并保留关键证据与状态。

### 1. 智能截断

- 保留 system 和最新用户问题；
- 保留任务约束、实体和未完成事项；
- 删除重复、寒暄和低价值日志；
- head+tail；
- 按优先级分配 Token。

简单，但可能丢失中间关键信息。

### 2. RAG

将长文档：

```text
parse -> chunk -> index -> retrieve -> rerank
```

只把相关片段放入上下文。适合知识查询，但需要解决 chunk、召回、权限、版本和无答案问题。

### 3. 摘要与压缩

- rolling summary；
- extractive compression；
- map-reduce summary；
- query-aware compression；
- 结构化状态；
- 保留引用和原文指针。

摘要是有损的，重要结论需可回溯原文。

### 4. 层级处理

长文档分块并行处理：

```text
chunk analysis
-> section summary
-> document summary
-> final synthesis
```

适合报告、代码库和多文档研究。跨 chunk 关系需在上层融合。

### 5. Sliding Window / Chunked Processing

对分类、抽取、序列标注可用滑窗加 overlap，再聚合结果。视频和长音频也可按时间窗口处理并维护状态。

### 6. 外部记忆和工具

将会话事实、任务状态、历史事件存入数据库/向量库，按需检索。订单、余额和流程状态应查询权威系统，而不是长期塞在 Prompt。

### 7. 更长上下文模型

可选支持更长窗口的模型，并结合：

- RoPE scaling；
- FlashAttention；
- sparse/sliding-window attention；
- recurrent memory；
- state-space/hybrid architecture；
- context/sequence parallel。

但标称窗口大不代表能有效利用全部信息，仍要评测 lost-in-the-middle。

### 8. 工程策略

- 先估算 Token；
- 超预算按优先级裁剪；
- 检索和摘要缓存；
- 分块 prefill；
- 限制用户上传和工具输出；
- 记录被丢弃内容；
- 证据不足时追问或拒答。

### 9. 跨境支付场景

长商户工单应保留订单号、已确认事实、工具结果和待处理事项；历史原文可检索。支付状态每次实时查询，不能依赖旧摘要。

### 10. 面试总结

超长输入的主流方案是“截断/压缩 + RAG + 层级处理 + 外部记忆 + 长上下文架构”。生产中通常组合使用，目标不是塞进最多 Token，而是让关键证据可找到、可验证。
