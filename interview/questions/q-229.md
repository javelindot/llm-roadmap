# 模型怎么评判答案是否准确？

## 答案

> 分两个维度看准确——**事实性（factual accuracy）** 和 **任务符合度（task alignment）**，两者用不同方法。
>
> **事实性评判**：
>
> **1. 引用对比（RAG 场景必上）**
> - 答案里的每个事实点必须能在检索到的文档中找到出处
> - 用 NLI 模型（natural language inference）逐句校验答案是否被检索内容 entail
> - 不被支持的句子标红 → 转人工 / 重生成
>
> **2. 黄金集对比**
> - 准备 (query, 标准答案) 对
> - LLM-as-judge 比较模型答案和标准答案的语义相似度
> - 三档：完全正确 / 部分正确 / 错误
>
> **3. 多模型投票**
> - 用 3 个独立 judge 模型
> - 一致认为对才算对
> - 不一致转人工
>
> **4. 规则校验**
> - 业务硬规则（金额必须是数字、订单号格式、时间合理性）
> - 不需要 LLM，正则 / 类型检查即可
>
> **任务符合度评判**：
>
> **5. LLM-as-judge 多维度打分**
> 让 judge 给"信息完整 / 术语准确 / 简洁度 / 礼貌度"分别 1-5 分。
>
> **6. 用户反馈代理指标**
> - 点赞 / 点踩率
> - 转人工率
> - 后续投诉率
>
> **我们跨境支付里的具体落地**：
>
> **RAG 答案评判 pipeline**：
> ```python
> async def judge_answer(query, answer, retrieved_docs):
>     # 1. 引用校验（必过）
>     unsupported = await nli_check(answer, retrieved_docs)
>     if unsupported:
>         return JudgeResult(faithful=False, action="regenerate")
>
>     # 2. 任务符合度（LLM judge）
>     scores = await llm_judge(query, answer, dimensions=[
>         "事实性", "完整性", "简洁性", "礼貌度"
>     ])
>
>     # 3. 业务规则
>     if not pass_business_rules(answer):
>         return JudgeResult(rule_violated=True, action="human")
>
>     return JudgeResult(scores=scores, faithful=True)
> ```
>
> **黄金集评测**：
> - 500 条人工标 (query, 期望答案, 期望引用 chunk_id)
> - 离线跑 RAGAS 框架，输出 Faithfulness、Answer Relevance、Context Precision 三个指标
> - 每周一次基线对比
>
> **线上监控**：
> - 每天 sample 100 条人工 spot check
> - 主要看"幻觉率"和"任务不符合率"两个指标
> - 不达标的回流补数据 / 调 prompt / DPO 迭代
>
> **几个工程经验**：
>
> 1. **judge 模型要比被评模型强**——Qwen-14B 答的用 Qwen-72B 判，效果好
> 2. **judge 也会犯错**——3 judge 一致才算准
> 3. **事实性 > 任务符合度**——金融场景宁可拒答也不能编
> 4. **黄金集会过时**——业务规则变了要更新黄金集
> 5. **LLM judge 成本**：Qwen-72B 当 judge，1w 条评估 ~50 块，可接受
>
> 一句话：**没有 single judge 能搞定一切**——事实性靠 NLI + 引用比对，任务符合度靠多维 LLM judge，业务规则靠硬校验，三者组合才能在金融场景立得住。
