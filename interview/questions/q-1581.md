# 为什么大模型使用GQA（Group Query Attention）

## 答案

GQA（Grouped-Query Attention）让多个 Query heads 共享一个 KV head，在 MHA 的质量与 MQA 的推理效率之间折中。大模型使用它的主要原因是减少 KV Cache 和 Decode 内存带宽。

### 1. 三种结构

设：

```text
H_q = Query head 数
H_kv = KV head 数
```

- MHA：`H_kv = H_q`；
- MQA：`H_kv = 1`；
- GQA：`1 < H_kv < H_q`。

GQA 中每组：

```text
group_size = H_q / H_kv
```

个 Query heads 共享同一 K/V head。

### 2. KV Cache 收益

KV Cache 近似为：

```text
2 * layers * sequence_length
* H_kv * head_dim * bytes
```

相对 MHA 的比例为：

```text
GQA cache / MHA cache
= H_kv / H_q
```

例如 `H_q=32、H_kv=8`，KV Cache 约降到 MHA 的 `1/4`。高并发、长上下文时可显著增加可服务序列数。

### 3. 为什么 Decode 更快

Decode 每步 Query 长度为 1，需要从显存读取全部历史 K/V，常受内存带宽而非算力限制。减少 KV heads 可降低：

- KV Cache 容量；
- 每 token 读取带宽；
- 跨卡 KV 通信；
- Beam/parallel sampling 的缓存成本。

Prefill 仍有大矩阵计算，GQA 的相对收益通常没有 Decode 那么突出。

### 4. 为什么不全部用 MQA

MQA 只保留一个 KV head，压缩最大，但所有 Query heads 共享同一 K/V 表示，可能影响模型质量和容量。GQA 保留多个 KV 组，在质量和成本间取得更稳妥的折中。

### 5. 工程注意

- `H_q` 通常应能被 `H_kv` 整除；
- RoPE 作用于 Q/K；
- 不应物理复制 K/V 到每个 Query head，kernel 可按组读取；
- 训练与推理权重形状需一致，不能部署时随意把 MHA 改成 GQA；
- GQA 与 FlashAttention、PagedAttention 是不同层次，可组合使用；
- MLA 是 latent 压缩，不等同于 GQA。

### 6. 跨境支付场景

长政策 RAG 与高并发客服可从 GQA 降 KV 中受益，但上下文治理仍是首要措施，不能用更大 Cache 容纳无限历史。

### 7. 面试总结

GQA 通过减少 KV heads，将 KV Cache 按 `H_kv/H_q` 比例压缩，同时保留多组 K/V 以维持比 MQA 更好的质量，是现代 LLM Decode 优化的常见选择。
