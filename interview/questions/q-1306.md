# 刚刚提到 MLA ，那 MLA 是怎么对 KV Cache 做优化的

## 答案

MLA（Multi-head Latent Attention）的核心是：不为每个 token 缓存完整的多头 K/V，而是先把 K/V 压缩到低维 latent，推理时缓存这个低维表示，再通过投影恢复各个 head 所需的信息，从而显著降低 KV Cache。

### 1. MHA 的 KV Cache

标准多头注意力中，每个 token 需要保存：

```text
K: [n_heads, d_head]
V: [n_heads, d_head]
```

每层每 token 的缓存量近似：

```text
2 * n_heads * d_head
```

长上下文和高并发下，Decode 往往受显存容量和读取 KV Cache 的带宽限制。

### 2. MLA 的低秩压缩

MLA 先对 hidden state 做下投影：

```text
c_KV = W_DKV * h
```

其中 `c_KV` 维度远小于完整 K/V。再通过上投影得到各 head 的 K/V：

```text
K_C = W_UK * c_KV
V_C = W_UV * c_KV
```

推理时主要缓存：

```text
c_KV + 位置相关的小部分 K_R
```

而不是缓存展开后的所有 K/V。DeepSeek-V2 论文给出的典型配置中，缓存维度可从 MHA 的大规模 K/V 降到一个较小的 latent 加 RoPE key 分量。

### 3. 为什么不能直接把 RoPE 全加在压缩 K 上

RoPE 是位置相关旋转。若把它直接施加到完整 K 后再希望吸收到低秩投影中，位置相关操作与投影矩阵无法像普通线性变换一样统一合并，破坏 MLA 在 Decode 时通过矩阵吸收减少重建计算的优化。

DeepSeek 采用 decoupled RoPE：

- 内容部分 `K_C` 走低秩压缩；
- 单独生成较小的 RoPE key `K_R`；
- Query 也拆为内容部分和 RoPE 部分；
- 注意力分数由两部分共同计算。

这样兼顾位置编码与低维缓存。

### 4. 能带来什么收益

- KV Cache 显著下降，可容纳更长上下文和更多并发请求；
- Decode 读取的数据更少，缓解显存带宽瓶颈；
- 服务端连续批处理可放入更多序列，提高吞吐；
- 代价是实现、算子融合、量化和并行适配更复杂。

MLA 不等于所有阶段都同比加速。Prefill 仍有大量矩阵计算；实际收益取决于 batch、上下文长度、硬件、kernel 和推理框架是否原生优化。

### 5. 跨境支付场景

长商户历史、合规材料和多轮调查会占用大量 KV Cache。MLA 可降低长上下文客服或调查 Agent 的服务成本，但更优先的手段仍是检索、摘要和结构化工具查询，不能把全部交易流水无筛选地塞进上下文。
