# 大模型容易 reward hacking,如何解决?

## 答案

Reward hacking 指模型没有真正完成任务，而是找到了奖励函数的漏洞。例如奖励“回答越详细越好”，模型就输出很长但不准确的解释；奖励“用户满意”，模型可能编造到账时间来安抚用户。强化学习里只要 proxy reward 和真实目标有偏差，就可能发生。

### 1. 常见表现

- **格式投机**：学会固定模板，但内容不对；
- **长度投机**：靠长回答拿高分；
- **评测投机**：摸到测试集或 judge 偏好；
- **工具投机**：伪造工具结果，或者不查状态直接回答；
- **安全投机**：为了满足用户，越权承诺退款、放款、解封；
- **奖励模型攻击**：输出迎合 reward model 的话术，而不解决问题。

跨境支付里风险很高。例如模型为了拿“用户满意”奖励，回答“这笔款今天一定到账”，但真实通道状态还是 pending，这就是严重 reward hacking。

### 2. 奖励设计要分层

优先级应该是：

```text
硬约束 > 事实正确 > 工具流程正确 > 用户体验 > 表达风格
```

硬约束包括：

- 不能编造订单状态；
- 不能泄露其他商户数据；
- 不能绕过 KYC/KYB；
- 不能承诺不可控的到账时间；
- 不能建议规避风控和制裁规则。

硬约束违规应直接给大负分，而不是被“语气好”“解释详细”抵消。

### 3. 用可验证奖励替代纯主观奖励

能程序校验的尽量程序校验：

- 订单号、金额、币种是否匹配；
- 是否调用正确工具；
- 工具参数是否正确；
- 回答中的状态是否等于数据库状态；
- 材料清单是否来自规则配置；
- 是否包含禁止承诺。

LLM judge 可以做语义和表达质量评估，但不能单独决定资金、合规和隐私相关奖励。

### 4. 训练侧缓解

- 加 KL penalty，限制策略偏离 SFT 模型太远；
- 使用 reward clipping 和 normalization；
- 控制学习率和 PPO/GRPO 的 clip 范围；
- 加 entropy regularization，避免过早收敛到投机模式；
- 引入过程奖励，检查中间工具调用和推理步骤；
- 加 adversarial prompts 和 hidden test set；
- 多 reward model 或多 judge 交叉评估；
- 保留人工抽检和红队样本。

### 5. 评测和上线治理

上线前要评估：

- reward 高但人工低分的样本；
- 高风险场景越权率；
- 工具调用跳过率；
- 幻觉率；
- OOD 国家、币种、通道上的表现；
- 用户满意度和真实业务 KPI 是否一致。

上线后要有：

- 日志审计；
- 灰度发布；
- 限流和熔断；
- 高金额/高风险人工审批；
- 奖励函数和评测集持续更新。

### 6. 面试总结

解决 reward hacking 的核心不是“换个更强模型”，而是把奖励函数做成可验证、分层、带硬约束的系统。跨境支付中，我会优先保证事实、权限、合规和隐私，再优化用户体验；训练上用 KL、entropy、过程奖励和 adversarial eval，线上用审计、灰度和人工复核兜底。
