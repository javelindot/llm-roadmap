# 哪些因素会影响内存使用？

## 答案

**一句话**：大模型内存使用主要由参数、梯度、优化器状态、激活值、KV Cache、batch size、序列长度和并行策略决定。

### 核心回答
- **训练阶段**：
  - 模型参数：参数量越大显存越高。
  - 梯度：反向传播要保存梯度。
  - 优化器状态：Adam 通常还要保存一阶、二阶动量，显存压力很大。
  - 激活值：为了反传要保存中间激活，和 batch size、seq length 强相关。
  - 精度：FP32、FP16、BF16、INT8 占用不同。
- **推理阶段**：
  - 参数权重是基础占用。
  - KV Cache 随 batch size、上下文长度、层数、hidden size 增长。
  - 并发越高，KV Cache 压力越大。

优化方式包括量化、梯度检查点、ZeRO、张量并行、流水线并行、PagedAttention、限制上下文长度等。

### 1. 训练和推理的内存瓶颈不同

训练时显存最重的不只是模型权重，还包括梯度、优化器状态和激活值。以 Adam 为例，除了参数本身，还要保存一阶、二阶动量，所以全量训练的显存压力会非常大。

推理时通常没有梯度和优化器状态，但会有 KV Cache。上下文越长、并发越高，KV Cache 越容易成为瓶颈。很多线上系统不是权重装不下，而是并发请求的缓存把显存吃满。

### 2. 排查内存问题的顺序

我会先判断是哪类内存占用：

- 如果模型刚加载就 OOM，多半是权重太大，需要量化、分片或换小模型。
- 如果训练反向传播时 OOM，多半是激活值、梯度或优化器状态，需要梯度检查点、LoRA、ZeRO/FSDP。
- 如果推理并发上来后 OOM，多半是 KV Cache，需要限制上下文、使用 vLLM/PagedAttention、做前缀缓存或降低并发。
- 如果长文档 RAG 请求特别慢，要检查是不是上下文拼得太长。

### 3. 面试总结

内存优化不能只说"量化"。要先区分权重、激活、优化器状态和 KV Cache 分别占多少，再对症处理。

### 4. 结合跨境支付项目怎么说（面试口径）

> 在支付业务里，内存问题最常见的是智能客服或风控助手并发起来以后，显存不是只被模型参数吃掉，而是被长上下文和 KV Cache 撑爆。比如一次商户审核，如果把营业资料、历史交易、风控规则、对话记录全塞进去，单个请求就会特别长。
>
> 我的处理顺序会比较工程化：先做 RAG 过滤，只给必要证据；再做历史对话摘要，避免无限拼上下文；推理框架上用 vLLM 这类支持 PagedAttention 的方案；模型本身可以做 4bit/8bit 量化。只有这些还不够，才考虑多卡并行或加机器。
>
> 这也是我做模型部署时比较关注的点：线上瓶颈往往不是"模型能不能答"，而是高并发、长上下文、低延迟同时满足时，显存和吞吐能不能扛住。

**钩子**：线上推理显存很多时候瓶颈在 KV Cache，不只是模型参数。
