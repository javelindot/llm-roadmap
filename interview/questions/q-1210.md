# RAG系统的架构，每一个模块的作用，文本截断的长度是多少？

## 答案

生产级 RAG 的完整架构是：

> 数据接入与治理 → 解析与切分 → 多路索引 → 查询理解 → 混合召回 → 重排过滤 → 上下文构造 → 受约束生成 → 校验、监控与反馈。

### 1. 数据接入与治理

接收产品文档、SOP、FAQ、错误码、表格和 PDF，完成权限、来源、版本、生效时间、语言和租户标记。文档删除或更新时同步清理旧索引。

### 2. 解析与切分

识别标题、段落、列表、表格、图片和页码，按业务语义切分。跨境支付规则的国家、币种、条件、例外和结论应保持完整。

可使用父子块：

- 子块用于精确召回；
- 父块用于补齐上下文；
- 命中后按需扩展相邻块；
- 重叠块在送入模型前去重。

### 3. 索引

- BM25/倒排：错误码、国家码、币种和产品名；
- Dense Vector：口语、同义和多语言；
- 结构化索引：表格、状态和实体字段；
- 图索引：确有多跳关系时使用。

每个 chunk 携带 `tenant、country、currency、product、channel、version、effective_time、ACL`。

### 4. 查询理解

- 多轮指代消解；
- 意图和实体识别；
- Query Rewrite；
- 支付术语归一化；
- 区分知识检索与实时工具；
- 信息不足时追问；
- 绑定当前用户、商户和权限。

### 5. 混合召回

BM25、Dense、FAQ 和结构化通道并行，使用 RRF 或校准分数融合。强权限、租户和生效时间过滤尽量在检索前执行。

### 6. Rerank 与过滤

Cross-Encoder 对候选精排，并执行：

- 去重；
- 权限与版本过滤；
- 国家、币种和产品一致性；
- 冲突检测；
- 证据充分性判断。

证据不足时不进入自由生成。

### 7. 上下文构造与生成

工具事实、知识证据和对话历史分区放置，每条证据保留来源和版本。Prompt 要求输出：

1. 可确认的状态；
2. 依据；
3. 下一步建议；
4. 引用；
5. 是否需要人工。

金额、状态、时效和风控原因不能在证据之外补全。

### 8. 校验与监控

- Claim 与引用一致性；
- 工具结果和文档冲突；
- 敏感信息与越权；
- 结构化输出；
- 完整 Trace；
- Recall、NDCG、Faithfulness、引用、FCR、P95 和成本。

### 9. 文本截断长度

不存在通用固定值。需要区分：

- 原始文档解析长度；
- Embedding 模型最大输入；
- Reranker 最大输入；
- 生成模型上下文预算；
- 单个 chunk 的语义边界。

可将以下值作为实验起点，而不是固定答案：

- 普通说明文：约 300 到 800 tokens；
- FAQ/错误码：一条完整问答或规则一个 chunk；
- 长小节：结构切分后必要时父子块；
- overlap：约 chunk 的 10% 到 15%，仅在跨边界时使用；
- 初召回：Top 20 到 50；
- Rerank 后：保留约 5 到 10 个高质量证据。

最终上下文长度按模型窗口反推：

```text
证据预算 =
模型上下文
- 系统指令
- 会话历史
- 工具结果
- 预留输出
- 安全余量
```

如果模型支持 128K，也不代表应塞满 128K。过长上下文会增加延迟、成本和干扰。应在真实问题集上比较 Recall@K、Context Precision/Recall、答案完整性、P95 和 token 成本，选择业务最优点。
