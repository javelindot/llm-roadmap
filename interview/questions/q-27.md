# 讲一下 DPO 和 PPO，DPO 和 PPO 有什么区别？

## 答案

**一句话**：**PPO 是在线强化学习，需要 4 个模型；DPO 是离线偏好优化，只要 2 个模型**——上限 PPO 高，但 DPO 简单稳定，工业界基本以 DPO 起家。

### 1. PPO 流程（4 个模型）
```
Prompt → Actor 采样 → RM 打分 → GAE 算 Advantage → PPO clip 更新 Actor + value loss 更新 Critic
                                         ↑
                                  Ref Model 约束 KL
```
**必备 4 个模型**：
- **Actor**（待训）
- **Critic**（价值函数，估计 V(s)）
- **Reward Model**（评分）
- **Reference Model**（KL 约束，冻结）

**PPO Loss**：
$$L_{PPO} = \min(r_t A_t, \text{clip}(r_t, 1-\epsilon, 1+\epsilon) A_t) - c_1 \cdot L_{value} + c_2 \cdot H(\pi)$$

### 2. DPO 流程（2 个模型）
```
偏好数据 (x, y_w, y_l) → 直接闭式 loss 更新 Actor
                              ↑
                         Ref Model 约束 KL
```
**只需 2 个模型**：Actor + Ref Model

**DPO Loss**：
$$L_{DPO} = -\log\sigma\left(\beta \left[\log\frac{\pi_\theta(y_w|x)}{\pi_{ref}(y_w|x)} - \log\frac{\pi_\theta(y_l|x)}{\pi_{ref}(y_l|x)}\right]\right)$$

DPO 的数学推导：从 KL 约束的最优策略反推，把 RM 这一步消掉。

### 3. 核心区别对比表

| 维度 | PPO | DPO |
|------|-----|-----|
| 模型数 | 4 个 | 2 个 |
| 数据形式 | 在线采样 + RM 打分 | 离线偏好对 (x, y_w, y_l) |
| 是否需要 RM | 必需 | 不需要 |
| 是否需要 Critic | 必需 | 不需要 |
| 训练稳定性 | 差（要调 KL、clip） | 好 |
| 显存压力 | 大（4 模型并存） | 小 |
| 训练速度 | 慢 | 快 |
| 性能上限 | 高 | 中等 |
| 调参难度 | 高（β KL、clip ε、value loss） | 低（主要 β） |
| 应对长尾偏好 | 好（在线探索） | 一般（受数据覆盖限制） |
| Reward Hacking 风险 | 高 | 低（无显式 RM） |

### 4. 选型建议
- **原型 / 小团队 / 算力有限** → **DPO**
- **数据离线就够覆盖** → **DPO**
- **追求最高对齐质量 / 有大算力** → **PPO**
- **业务领域强，需要在线探索** → **PPO**（或 Online DPO）
- **快速迭代多轮对齐** → **DPO**（重训快）

### 5. 业界共识
- **Llama 2 / Llama 3 用 PPO**，因为算力充足、追求上限
- **大多数开源项目用 DPO**（Zephyr、Tulu、OpenChat）
- **新趋势**：KTO（无需 pair 的偏好优化）、IPO（解 DPO 过拟合）、SimPO（无 ref）、Online DPO

### 6. 一句话总结
**PPO 是开车手动挡（猛但难开），DPO 是自动挡（稳但上限有限）**——工业界 80% 场景 DPO 够用，追上限再上 PPO。

## 结合项目谈（口语版 · 约 40 秒）

> 我们跨境支付**客服回复对齐**这条线，PPO 和 DPO 两个都跑过，最后定 DPO。
>
> 先说为啥试 PPO——理论上限高嘛。但跑通之后发现几个工程现实：
>
> 一是 **4 个模型同时占显存**，actor + critic + RM + ref，单机 8 卡 H100 只能跑 7B 模型，再大就 OOM。生产链路想上 14B 直接没戏；
>
> 二是 **训练抖**——KL 系数 β 难调，调大不学、调小飞掉，团队里 RLHF 经验有限，三天两头训挂；
>
> 三是 **Reward Hacking**——RM 偏好长且礼貌，actor 学了几轮所有回复都"尊敬的客户"开头，业务侧吐槽。
>
> 切回 **DPO** 之后整个世界清净了：只要 actor + ref 两个模型，14B 单机能跑；β 一个超参数搞定；偏好数据离线准备，5w 条客服历史标注扔进去训一天就出结果；上线后客户满意度提了 8 个点。
>
> 当然 DPO 也有局限——它**吃数据覆盖**，长尾偏好（比如刁难客户的复杂场景）数据少就学不动。我们的解法是 **DPO 跑稳基线 + 偶尔做一次 reject sampling 数据增强**，效果接近 Online DPO。
>
> 我现在的看法：**业务工程团队默认上 DPO，PPO 留给追求 SOTA 的算法团队**。
