# 为什么需要大模型应用开发框架？

## 答案

**一句话**：大模型应用开发框架是为了把模型、提示词、RAG、工具调用、记忆、评估和部署这些通用能力模块化，降低工程复杂度。

### 核心回答
为什么需要：
- 统一接入不同模型。
- 管理 prompt 模板和变量。
- 封装 RAG 检索、切块、向量库。
- 支持工具调用和 Agent 流程。
- 管理 memory、callback、trace。
- 方便评估、监控和调试。
- 降低重复造轮子成本。

常见框架：LangChain、LlamaIndex、Haystack、Dify、Semantic Kernel 等。

注意：框架提升效率，但复杂业务仍要理解底层逻辑，不能完全依赖封装。

### 1. 框架解决什么问题

大模型应用通常不只是调用一次模型，而是要串起 prompt、RAG、工具、记忆、评估、日志和部署。框架把这些通用能力封装起来，可以减少重复造轮子。

### 2. 框架不能替你解决什么

框架通常不能直接解决业务核心问题：

- 权限怎么控。
- 高风险工具怎么审批。
- 数据怎么脱敏。
- 错误怎么回滚。
- 审计日志怎么留。
- 业务指标怎么评估。

这些仍然要自己设计。

### 3. 面试总结

框架适合提效和快速验证，但生产系统要掌握底层链路，不能被框架绑架。

### 4. 结合跨境支付项目怎么说（面试口径）

> 在跨境支付项目里，框架的价值是快速把 RAG、工具调用和流程编排搭起来。比如接知识库、接交易查询 API、接工单系统，如果全手写会很慢。
>
> 但我不会把核心业务逻辑完全绑死在框架里。高风险动作、权限控制、审计日志、幂等重试、人工审批这些必须自己设计。比如退款、冻结、拒付处理，不能因为框架支持 tool calling 就让模型直接执行。
>
> 所以框架对我来说是提效工具，业务可靠性还是要靠自己的工程边界。

**钩子**：框架解决通用编排，支付场景的权限、审计和幂等必须自己把控。
