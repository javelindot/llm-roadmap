# 用 LangGraph 实现多轮对话 Agent,相比手写 prompt 流程有哪些工程和效果优势?

## 答案

LangGraph 的核心价值不是让模型本身突然更聪明，而是把多轮 Agent 从隐式 Prompt 协议变成显式状态图：节点、路由、状态、持久化、中断和恢复都有可检查的工程语义。效果提升主要来自流程约束、上下文治理和错误恢复，而不是框架名称。

### 1. 手写 Prompt 流程的典型问题

简单实现往往是：

```text
拼接历史对话
-> 让模型决定下一步
-> 解析文本
-> 调工具
-> 再拼回 Prompt
```

随着轮次和工具增加，会出现：

- 状态散落在 Prompt 和代码变量中；
- 路由依赖自然语言解析；
- 工具失败后难以从中间恢复；
- 循环、重试和终止条件不清晰；
- 人工审批需要大量特例代码；
- 历史越来越长，污染上下文；
- 并行工具结果合并困难；
- 单步不可重放，bad case 难定位。

### 2. LangGraph 的工程优势

**显式状态**

用 TypedDict/Pydantic 定义：

```text
messages, intent, merchant_id, order_id,
tool_results, retrieved_docs,
retry_count, need_human, trace_id
```

节点只读写明确字段，减少靠文本猜状态。

**图式控制流**

- 条件边决定下一节点；
- 高频确定性流程固定化；
- ReAct 只在真正需要开放决策的位置使用；
- 最大步数、重试和终止条件显式化。

**持久化与恢复**

- Checkpointer 保存线程状态；
- 工具或模型异常后从最近节点恢复；
- 长任务可暂停；
- 多轮对话通过 thread/session 恢复上下文；
- 需保证外部写操作本身幂等。

**Human-in-the-loop**

在退款、付款重试、账户处置等高风险节点 interrupt，人工确认后继续，而不是让模型直接执行。

**并行和子图**

订单、通道、账务等独立查询可并行；复杂能力封装为子图，主图只接收结构化结果。

### 3. 效果优势来自哪里

- 工具选择空间被状态和路由约束，减少乱调用；
- 只把当前节点需要的信息送给模型，降低上下文噪声；
- 实时事实、检索证据和对话历史分字段管理；
- 对工具无数据、超时、权限不足分别处理；
- 循环次数和 fallback 明确；
- 人工反馈可回到对应节点。

因此通常会改善工具成功率、流程完成率和稳定性，但不会自动提高基座模型知识或推理上限。

### 4. 跨境支付示例

```text
authenticate
-> classify_intent
-> extract_or_ask_order_id
-> query_order
-> route_by_status
   -> query_channel
   -> query_ledger
   -> query_risk_summary
-> retrieve_sop
-> generate_grounded_answer
-> safety_check
-> finish / human_review
```

每个工具使用最小权限。风控节点只返回允许解释的大类，写操作进入独立审批子图并要求幂等键。

### 5. 评估方式

比较手写流程与 LangGraph 版本时，控制模型、Prompt、工具和数据集：

- 任务完成率；
- 工具选择、参数准确率；
- 平均步骤和循环率；
- 故障恢复率；
- 人工介入准确率；
- 事实与引用；
- P95/P99、token 和成本；
- 安全违规和重复写入。

### 6. 何时不需要 LangGraph

- 单轮 FAQ；
- 一个固定工具调用；
- 无状态分类或抽取；
- 简单线性链路且没有恢复需求。

此时普通函数或小型状态机更简单。LangGraph 适合状态丰富、分支多、长时间运行且需要可恢复和人工介入的 Agent。
