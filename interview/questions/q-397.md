# 独立微调，强化每个子 Agent 的垂域边界

## 答案

> 独立微调是 Agent 边界划分的**模型层硬约束**——通过给每个 Agent 训独立的模型 / LoRA，让模型本身只擅长它的垂域，**根本不会做别的领域**。
>
> **为什么独立微调有效**：
>
> - **prompt 约束是软的**——模型 base 能力还在，可以被诱导
> - **数据权限是边界的**——但不影响模型行为
> - **独立微调是模型层硬约束**——它只学过自己领域的数据
>
> **微调方式（按成本）**：
>
> **方式 1：独立 LoRA（最常用）**
> - 同一个 base 模型，每个 Agent 一个 LoRA adapter
> - 切换 Agent 时换 LoRA
> - 优点：base 共享、显存省、训练快
> - 缺点：base 能力影响所有 Agent
>
> ```python
> # 推理时按 Agent 加载对应 LoRA
> base_model = load_base("qwen-14b")
> cs_lora = load_lora("customer_service_v3")
> risk_lora = load_lora("risk_v2")
> kyb_lora = load_lora("kyb_v1")
>
> async def serve_agent(agent_name, query):
>     lora = {
>         "customer_service": cs_lora,
>         "risk": risk_lora,
>         "kyb": kyb_lora,
>     }[agent_name]
>     return await base_model.invoke(query, lora=lora)
> ```
>
> **方式 2：独立全参微调（深度领域）**
> - 每个 Agent 一个完整模型
> - 优点：垂域能力强、模型本身就是边界
> - 缺点：显存爆、训练成本高
>
> 适用：风控、合规等高价值领域
>
> **方式 3：MoE Expert Routing（前沿）**
> - 一个 MoE 大模型，每个 expert 对应一个 Agent
> - Router 决定激活哪些 expert
> - 优点：参数共享 + 垂域分工
> - 缺点：训练复杂
>
> **我们跨境支付的具体落地**：
>
> | Agent | Base | 微调方式 | 数据量 |
> |-------|------|---------|--------|
> | 客服 | Qwen-14B | SFT + DPO LoRA | 5w SFT + 5w 偏好对 |
> | 风控调查 | Qwen-14B | SFT + GRPO（长 CoT） | 5k 冷启动 + RL |
> | KYB 多模态 | Qwen2.5-VL-7B | LoRA + 投影层 | 5w 证件 |
> | 路由 / 改写 | Qwen-7B | 轻 LoRA | 1w 路由样本 |
> | 报表 | Qwen-14B | 共享客服 LoRA | 共享 |
>
> **垂域微调的关键 trick**：
>
> **1. 训练数据严格限定领域**
> - 客服模型不喂风控数据
> - 防止跨域能力溢出
>
> **2. 添加"拒绝越界"样本**
> ```
> Query: 这个客户的风控分是多少？
> （客服模型）Answer: 抱歉，这个问题不在我的服务范围，请联系风控团队。
> ```
> 让模型学会**主动说"我做不了"**。
>
> **3. 系统 prompt 双重保险**
> ```
> 你是客服 Agent，只能回答 FAQ、订单、汇率相关问题。
> 风控、合规、商户准入等问题请回复"超出服务范围"。
> ```
> Prompt + 微调双保险。
>
> **4. 评测拒答率**
> - 测试集包含 20% 越界 query
> - 评估模型"主动拒答"的准确率
> - 拒答率 > 95% 才能上生产
>
> **5. 持续监控**
> - 上线后看是否真的没越界
> - 失败 case 回流增强
>
> **几个工程经验**：
>
> 1. **LoRA 切换是主流**——base 共享省显存，每 Agent 一个 LoRA
>
> 2. **deploy as separate service** + **共享 base**：vLLM 支持 multi-LoRA serving
>
> 3. **base 模型升级时全员重训**——LoRA 兼容性是个问题
>
> 4. **垂域不等于隔离**——LoRA 是个"调整旋钮"，base 能力还在，prompt + 权限 + 工具白名单要一起上
>
> 5. **小模型够用就别 14B**——简单 Agent 用 7B + 轻 LoRA 即可
>
> **不推荐的做法**：
>
> - **共享 LoRA 多 Agent**——LoRA 是为特定 Agent 训的，混用降效果
> - **一个万能模型干所有**——容易跨域、调试难、风险大
> - **不做拒答样本**——模型没学会"我不会"，会强行答
>
> **代价**：
>
> - 训练成本：每个 Agent 一个微调任务
> - 部署复杂：multi-LoRA 切换、版本管理
> - 评测复杂：每个 Agent 独立黄金集
>
> **vLLM Multi-LoRA Serving 示例**：
>
> ```python
> from vllm import LLM, SamplingParams
>
> llm = LLM(model="qwen-14b", enable_lora=True, max_loras=4)
>
> response = llm.generate(
>     prompts=[query],
>     lora_request=LoRARequest("customer_service_v3", 1, "/path/to/cs.lora")
> )
> ```
>
> 一句话：**独立微调让 Agent 边界从"约束"变成"能力边界"**——模型本身就不会越界，配合 prompt + 权限 + 工具白名单，是金融 Agent 系统的模型层硬约束。
