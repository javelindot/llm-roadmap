# 你知道 lovart 吗？你认为他是 agent 还是 Workflow？你怎么判断的？

## 答案

截至 2026 年，Lovart 的公开产品定位是面向设计工作的 AI Design Agent。若从系统行为判断，我会把它称为“Agent 化设计产品”，但其内部一定也包含大量确定性 Workflow；Agent 和 Workflow 不是非黑即白。

### 1. 判断标准

**Workflow**

- 步骤预先定义；
- 输入输出 schema 明确；
- 分支有限；
- 相同输入路径较稳定；
- 失败后按固定规则重试或回退。

**Agent**

- 根据用户目标自主拆解任务；
- 动态选择和组合工具；
- 根据中间结果重新规划；
- 能处理开放式输入和不确定环境；
- 有循环、反思、验证和终止判断。

### 2. 为什么更偏 Agent

设计任务通常不是单次模板填充，而是：

```text
理解 brief
-> 拆解品牌/受众/渠道需求
-> 搜集或生成素材
-> 调用图像、视频、排版等工具
-> 生成多个方案
-> 根据反馈继续修改
-> 交付设计资产
```

如果产品能根据目标动态决定步骤、跨多种设计工具执行，并依据结果迭代，它就具备 Agent 特征。

### 3. 为什么又包含 Workflow

具体环节通常需要确定性编排：

- 文件解析；
- 图片生成；
- 抠图和扩图；
- 尺寸适配；
- 品牌检查；
- 导出；
- 失败重试；
- 内容安全检查。

这些适合做可观测、可重试、可审计的 Workflow。生产系统常见架构是：

```text
Agent 负责规划与决策
Workflow 负责可靠执行
```

### 4. 如何进一步验证

我会观察：

- 用户是否只需描述目标，而不是手动选择固定流程；
- 同类任务是否会产生不同计划；
- 是否根据工具结果动态追加步骤；
- 是否有长期任务状态；
- 是否能解释当前计划；
- 用户是否可在中间节点干预；
- 失败时是重新规划还是固定报错。

不能仅因为产品用了多个模型或调用工具就称为 Agent。

### 5. 跨境支付类比

支付客服中，固定的“查订单 -> 返回状态”是 Workflow；面对“为什么某地区本周成功率下降”时，系统需要拆解国家、币种、通道、错误码和时间窗口，调用多个分析工具并迭代验证，更接近 Agent。资金操作仍应由审批工作流执行。

### 6. 面试总结

我的判断是 Lovart 公开定位和交互形态都偏 Agent，但底层应是 Agent 与 Workflow 的混合：Agent 负责理解目标、规划和动态工具选择，Workflow 保证生成、审核和导出的稳定执行。判断依据是自主性和反馈闭环，不是营销名称。
