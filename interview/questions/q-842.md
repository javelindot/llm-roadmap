# LLM与推荐系统结合的可落地方案有哪些

## 答案

LLM 与推荐结合应按离线增强、在线理解、生成式推荐和交互 Agent 分层，避免让大模型直接承担所有高 QPS 排序。

### 1. 内容与特征增强

- 从标题、图片、评论生成类目、属性、主题和质量标签；
- 用 LLM/VLM 生成 item/user embedding，改善冷启动和长尾；
- 生成 query、hard negative 和弱标签，蒸馏到小模型；
- 对用户行为总结长期/短期兴趣，但保留原始行为供排序。

这类多为离线任务，落地快且成本可控。

### 2. Query 与意图理解

LLM 做搜索改写、实体抽取、约束识别和对话上下文补全，再由传统召回与排序执行。适合复杂自然语言需求，如“适合雨天通勤且轻便的鞋”。

### 3. 重排与解释

对第一阶段少量候选做 pointwise/pairwise/listwise rerank；根据真实商品属性生成推荐理由。库存、价格和资格必须实时校验，解释不能编造。

### 4. 生成式推荐

P5、TIGER、HSTU、OneRec 等尝试把推荐改写为 token/semantic ID 生成，统一任务、召回或列表优化。挑战包括超大 item 空间、无效 ID、新品更新、延迟、列表约束与反馈偏差。

### 5. 推荐 Agent

多轮追问预算、偏好和使用场景，调用搜索、库存、评价和比较工具完成决策。固定交易流程用 workflow，LLM 负责语义判断。

### 6. 落地顺序

通常先离线标签/蒸馏，再 query 理解与解释，之后尝试少量候选 rerank，最后才评估生成式主链。指标同时包含 CTR/CVR、长期留存、退货、多样性、P99 和单位成本。

跨境支付可迁移为产品或通道推荐，但费率、国家资格和风险边界由实时规则控制。

### 面试总结

最可落地的是离线内容理解、query 改写、轻量模型蒸馏和少量候选重排；生成式推荐与 Agent 潜力更大，但必须解决约束、实时性、延迟和反馈偏差。
