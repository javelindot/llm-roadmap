# 如何让模型知道代码生成的对不对（代码评估）

## 答案

判断代码是否正确，最可靠的是在隔离环境中执行可验证测试，而不是让同一个模型自评。

### 1. 静态检查

- 解析 AST，确认语法和禁止 API；
- 编译、类型检查、lint 和格式检查；
- 静态安全扫描、依赖与许可证检查；
- 复杂度、死代码和潜在资源泄漏分析。

静态通过只代表“可能运行”，不代表功能正确。

### 2. 动态执行

- 公共单元测试验证基本样例；
- 隐藏测试覆盖边界和反例；
- property-based testing 随机生成满足约束的输入；
- differential testing 与可信参考实现比较；
- mutation testing 检查测试是否真的能发现错误；
- fuzzing 检查崩溃、安全和异常输入；
- 性能测试验证时间、内存和并发约束。

执行必须在容器/沙箱中限制 CPU、内存、时间、网络、文件和系统调用，并清理副作用。

### 3. 指标

- `pass@1`：一次生成通过率；
- `pass@k`：k 个候选中至少一个通过的概率；
- compile rate、test pass rate、代码质量、安全漏洞和性能；
- repo-level 任务还需运行项目测试、构建和集成检查。

[HumanEval](https://arxiv.org/abs/2107.03374)奠定了执行式 `pass@k` 评估，但测试覆盖不足会高估正确率；HumanEval+、LiveCodeBench 等进一步增加测试或降低污染风险。

### 4. 训练时如何反馈

可用测试通过作为 rejection sampling、SFT 数据过滤、过程验证或 RL 可验证奖励。应保留隐藏测试，避免模型针对公开用例投机；生成代码经人工 review 后才能进入生产。

跨境支付代码还需测试金额精度、幂等、重试、权限和账务一致性，普通功能测试通过仍不足以上线。

### 面试总结

代码正确性以“受限沙箱中的编译和隐藏测试”为主，LLM Judge 只能补充可读性。结合属性测试、差分测试、安全扫描和性能约束，才能避免只对样例过拟合。
