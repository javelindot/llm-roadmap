# MOE架构原理

## 答案

MoE（Mixture of Experts）通过稀疏路由让模型拥有大量总参数，但每个 token 只计算少量专家，从而解耦模型容量与单 token FLOPs。

### 1. 基本结构

通常用多个 Expert FFN 替换 Dense FFN。Router 计算：

\[
g(x)=\operatorname{softmax}(W_rx)
\]

选择 Top-k 专家：

\[
y=\sum_{i\in\operatorname{TopK}(g(x))}
\tilde g_i(x)E_i(x)
\]

Attention 往往仍共享。总参数随专家数增长，而激活参数只与 top-k 有关。

### 2. 训练难点

- **负载不均**：少数专家过热，其他专家学不到；
- **容量溢出**：单专家接收 token 超过 capacity，可能丢弃或重路由；
- **专家塌缩**：Router 持续偏向同一专家；
- **通信**：Expert Parallel 需要 all-to-all；
- **路由稳定**：噪声、精度和 batch 组成会影响 token 分配。

常用 load-balancing auxiliary loss、router z-loss、capacity factor、shared expert 或无辅助损失的偏置均衡。

### 3. 并行与推理

Expert Parallel 将不同专家放到不同设备，Tensor/Data/Pipeline Parallel 可组合。推理时总权重仍需存储，batch 小或流量不均会导致专家 kernel 利用率差；MoE 的“激活参数少”不等于低内存或低延迟。

### 4. 代表模型

Switch Transformer 使用 Top-1；Mixtral 使用 8 专家 Top-2；DeepSeekMoE/V3 使用细粒度 routed experts 与 shared experts；Qwen3-MoE 使用更大专家池和稀疏激活。

### 面试总结

MoE 用 Router 为每个 token 选择少量 FFN 专家，以稀疏计算扩大容量。核心工程挑战是负载均衡、容量、all-to-all 和小 batch 利用率，并要区分总参数、激活参数和实际部署内存。
