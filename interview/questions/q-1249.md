# 面试中被要求讲解RAG项目时，如何应对面试官引导到模型相关技术点的提问？

## 答案

讲解 RAG 项目时，不能只背“切块、向量库、Prompt”流程，也不能被模型问题带离项目主线。回答应始终围绕：

> 业务问题 → 为什么需要模型 → 模型放在哪一层 → 如何训练或选型 → 如何评测 → 失败时如何降级。

### 1. 先讲清模型在 RAG 中的角色

一个完整 RAG 中可能有多类模型：

- Embedding：Query 与 chunk 双塔编码；
- Reranker：Cross-Encoder 联合判断相关性；
- LLM：Query rewrite、问题分解和答案生成；
- NER/分类模型：意图、实体与路由；
- OCR/VLM：解析扫描件、表格和图片；
- Judge/Verifier：评测事实性、引用和格式。

说明每个模型的输入、输出和不可替代性，比笼统说“用了大模型”更有说服力。

### 2. Embedding 追问

需要能回答：

- 为什么不能直接使用生成模型 hidden state；
- cosine、dot product 和 L2 的区别；
- Query/Document 是否使用相同前缀或编码器；
- 如何构造正样本、hard negative 和 in-batch negative；
- 为什么 BM25 与 Dense 互补；
- 如何用 Recall@K、MRR 和长尾分桶选模型。

支付领域的错误码、币种和通道名适合 BM25，口语化与多语言问法依赖 Dense Retrieval。

### 3. Reranker 追问

说明 Cross-Encoder 将 Query 与候选一起编码，质量高但成本大，因此只处理初召回 Top-N。重点参数包括候选数、最大长度、batch、阈值和超时降级。评测 NDCG、Precision@K 和关键证据保留率。

### 4. 生成模型追问

回答模型选型时比较：

- 业务黄金集上的正确性与 faithfulness；
- 多语言、长上下文和结构化输出；
- 工具调用和拒答；
- TTFT、TPOT、吞吐与成本；
- 私有化、许可和数据合规。

模型有证据仍答错是生成层问题，正确证据未召回则先修检索，不能一律通过换更大模型解决。

### 5. 是否微调

先建立 Prompt、RAG 和工具基线。Embedding 领域偏移明显时，可用 Query-Positive-Hard Negative 做对比学习；生成模型只有在稳定格式、工具行为或表达偏好无法靠 Prompt 解决时，再考虑 LoRA-SFT/DPO。

实时费率、订单状态和规则版本不通过微调记忆。

### 6. 用实验支撑技术选择

项目回答至少准备一组可复现消融：

| 实验 | 说明 |
|---|---|
| BM25 vs Dense vs Hybrid | 召回方式贡献 |
| 无 Rerank vs Rerank | 排序收益与延迟 |
| 不同 chunk/父子块 | 证据完整性 |
| 小模型 vs 强模型 | 质量、成本和时延 |
| RAG-only vs RAG+Tool | 静态规则与实时事实 |

没有真实数据时不编造百分比，可以说明使用了哪些指标、如何分桶、什么结果会触发方案切换。

### 7. 跨境支付项目主线

以“付款未到账”为例：

1. 识别订单、币种和产品；
2. 鉴权后调用订单与通道工具；
3. BM25 + Dense 检索到账、失败和退汇规则；
4. Rerank 并过滤过期或其他国家内容；
5. LLM 基于工具事实与证据生成解释；
6. 校验金额、状态、引用和敏感信息；
7. 冲突或低置信时转人工。

无论面试官问 Transformer、Embedding、微调还是推理部署，都应落回这一链路中该技术解决的具体问题，以及如何证明它有效。
