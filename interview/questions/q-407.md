# 图谱的使用场景？

## 答案

**一句话**：知识图谱适合处理“实体多、关系复杂、需要多跳推理或可解释路径”的问题；如果只是简单聚合查询或纯语义相似度检索，关系型数据库或向量库通常更合适。

### 1. 典型使用场景

| 场景 | 图谱解决的问题 |
|------|----------------|
| 搜索与推荐 | 利用用户、内容、作者、主题之间的关系做关联推荐和解释 |
| 金融风控 | 发现账户、设备、手机号、商户之间的团伙和资金网络 |
| 企业知识管理 | 连接人员、组织、项目、文档、产品和术语 |
| GraphRAG | 通过实体关系做多跳检索，补足纯向量检索的全局关联能力 |
| 客户 360 | 合并不同系统中的同一客户及其交易、服务和风险记录 |
| 运维与供应链 | 分析服务依赖、故障传播、零部件和供应商影响范围 |
| 医疗科研 | 表达疾病、症状、药物、靶点和文献之间的关系 |

### 2. 图谱最有价值的能力

- **多跳查询**：回答“某客户关联的商户中，哪些又与高风险设备有关”。
- **关系推理**：不仅找相似内容，还能沿着明确关系寻找证据。
- **可解释性**：结果可以展示 `用户 → 设备 → 商户 → 风险事件` 的路径。
- **异构数据整合**：用统一实体层连接多个业务系统。
- **图算法**：社区发现、中心性、最短路径、相似节点和链路预测。

### 3. 什么时候不应该用

- 数据主要是规则整齐的表格，查询以过滤、聚合为主。
- 关系很少，或者关系不会参与查询和决策。
- 只需要文本语义召回，向量数据库已经足够。
- 数据治理和实体对齐能力不足，建图成本高于收益。

### 4. 常见组合架构

生产系统往往不是“图谱替代一切”，而是：

```text
关系数据库：保存交易事实
向量数据库：召回语义相似文档
知识图谱：连接实体关系与多跳证据
LLM：理解问题、编排检索并生成答案
```

**面试总结**：选择图谱的判断标准不是“数据量大不大”，而是 **关系是否是一等公民，以及问题是否依赖关系路径**。
