# LoRA微调中如何选择rank值？合并adapter权重时是否遇到梯度爆炸？

## 答案

Rank 应通过任务复杂度和受控实验选择，不能固定认为 8、32 或 64 最好。另一个表述要纠正：合并 Adapter 是前向权重运算，没有梯度，因此合并时不会发生“梯度爆炸”；可能出现的是权重尺度、激活或输出异常。

### 1. Rank 的作用

```text
Delta W = (alpha / r) * B A
```

新增参数：

```text
r * (d_in + d_out)
```

- 小 rank：便宜、正则强，可能欠拟合；
- 大 rank：容量高，显存和 checkpoint 增长，收益会递减；
- 任务越复杂、target modules 越少，通常越需要更高 rank。

### 2. 选择方法

从 `r=8/16/32/64` 做 sweep，固定：

- 数据、训练 token、global batch；
- target modules；
- 训练轮次和随机种子；
- 合理的学习率搜索；
- alpha/scaling 方案；
- 同一业务评测集。

同时比较质量、峰值显存、吞吐、过拟合和通用回归。不同层可以使用 rank pattern 或 AdaLoRA，不必全层相同。

### 3. Rank 与 Alpha

标准 LoRA scaling：

```text
alpha / r
```

增大 rank 而 alpha 不变会降低单方向更新尺度。rsLoRA 使用：

```text
alpha / sqrt(r)
```

比较不同 rank 时必须明确 scaling，否则实验不公平。

### 4. 合并过程

```text
W_merged
= W_base
  + scale * B A
```

常见异常：

- base model revision 不匹配；
- Adapter 被重复合并；
- alpha/r 读取错误；
- FP16 中矩阵乘或写回溢出；
- 量化基座未先正确反量化；
- 合并后又重复加载 Adapter；
- tied embedding/LM Head 处理不一致；
- 多 Adapter 直接相加造成冲突。

### 5. 如何排查

- 合并前检查 `||Delta W|| / ||W||`；
- 用 FP32/BF16 完成 merge，再转换目标 dtype；
- 比较 merge 前动态 Adapter 与 merge 后 logits；
- 检查权重 NaN/Inf、最大值和层级范数；
- 固定一组输入做生成回归；
- 保留未合并 checkpoint，不原地破坏基座；
- 量化应在合并后重新校准，除非框架明确支持量化权重合并。

训练阶段如果出现真正梯度爆炸，则检查学习率、alpha、初始化、梯度裁剪、低精度和异常样本，与 merge 是两个问题。

### 6. 跨境支付

低 rank 可用于话术和格式；复杂工具调用或多语言可逐步提高 rank/覆盖 FFN。规则、费率和实时状态不应靠提高 rank 写入模型。
