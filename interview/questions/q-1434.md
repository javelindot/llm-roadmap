# cot微调数据怎么构造的？

## 答案

CoT 微调数据通常包含“问题、可验证的中间推理或步骤、最终答案”。构造重点不是让文字越长越好，而是保证步骤与答案正确、过程对任务有帮助，并防止教师模型生成看似合理的错误推理。

### 1. 数据来源

- 专家人工编写；
- 业务 SOP 转成步骤；
- 强教师模型生成；
- 多模型或多次采样；
- 正确解题轨迹；
- 工具执行日志；
- 失败样本经专家修正；
- 可验证环境中的成功轨迹。

### 2. 基本构造流程

```text
收集问题
-> 定义标准答案/verifier
-> 生成多个候选推理
-> 验证最终答案
-> 检查中间步骤
-> 去重和质量评分
-> 人工抽检
-> 切分训练与评测
```

### 3. Rejection Sampling

对同一问题采样多条路径：

1. 用规则、单测、数据库或标准答案验证结果；
2. 删除答案错误、步骤自相矛盾、引用不存在的样本；
3. 在正确路径中按清晰度、长度和多样性选样本；
4. 保留多种正确解法，避免模式塌缩。

只检查最终答案还不够，模型可能通过错误过程碰巧得到正确结论。

### 4. 过程验证

可使用：

- 数学/代码 verifier；
- 工具参数和返回校验；
- Process Reward Model；
- LLM judge + rubric；
- 专家审核；
- 关键步骤的结构化约束。

LLM judge 不能单独验证高风险事实，应结合程序和人工。

### 5. 数据格式

```text
system
user question
assistant analysis/structured steps
assistant final answer
```

训练时通常对 user 和 tool result 做 loss mask，只监督 assistant token。也可以将过程和最终答案分成结构化字段，便于分别评测。

对于生产系统，不一定要展示完整内部思维过程。更稳妥的是训练和输出可验证的简短依据、工具证据和决策步骤，而不是不可审计的冗长自由推理。

### 6. 跨境支付示例

问题：“为什么这笔 USD 出款没有到账？”

高质量轨迹应包括：

```text
识别 payout_order_id
-> 查询订单
-> 查询通道回执
-> 对照错误码和材料规则
-> 输出已验证原因和下一步
```

订单状态和错误码必须来自工具，不允许教师模型编造。资金或合规决定不应作为自由 CoT 监督。

### 7. 评测

- 最终答案正确率；
- 步骤正确率；
- 工具调用成功率；
- 推理长度和冗余；
- OOD 泛化；
- 多路径一致性；
- 幻觉和越权率；
- 训练后通用能力回归。

### 8. 面试总结

CoT 数据构造是“多路径生成 + 结果验证 + 过程验证 + 人工抽检”。质量和可验证性比推理长度更重要，业务场景中应优先训练结构化、可审计的决策依据。
