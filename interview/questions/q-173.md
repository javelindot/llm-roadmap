# AI RAG 项目是基于开源项目做的吗？

## 答案

> 我们是**开源框架 + 自研业务层**的混合方案，不是纯套壳也不是从头造轮子。
>
> **开源用到的**：
> - **检索层**：Milvus（向量库）、Elasticsearch（BM25）
> - **Embedding**：bge-large-zh-v1.5（业务精调）
> - **Rerank**：bge-reranker-large
> - **LLM**：Qwen-14B（本地部署 + vLLM）
> - **解析**：MinerU + Marker（PDF）、PaddleOCR（扫描页）
> - **编排**：早期 LangChain，后来 LangGraph
> - **评测**：RAGAS（魔改版）
>
> **自研部分**（也是工业的关键）：
> - **业务路由器**：用户问题先分类到 5 个知识库之一（客服 / 政策 / 商户 / 风控 / 操作手册）
> - **多轮对话改写器**：处理代词指代
> - **引用强约束生成层**：强制 LLM 答案必须 cite 召回文档，不在召回里的回答转人工
> - **失败 case 回流飞轮**：每天 sample 100 条人工标，回流补数据 / 调 prompt
> - **业务评测黄金集**：500 条人工标场景题
> - **运维监控**：trace、token 消耗、p95 延迟、点踩率
>
> 开源框架解决"怎么跑通"，业务层解决"怎么跑稳跑准"——这是工业 RAG 跟 demo 的根本差距。
>
> 简单说：**我们 30% 时间在选 / 调开源组件，70% 时间在写业务逻辑、做评测、踩坑修 bug**。

**🪝 顺嘴带出去的钩子**：
- "LangChain 切到 LangGraph 的迁移原因可以聊。"
- "业务路由器我们怎么避免选错可以展开。"
- "为什么不直接用 RAGFlow / Dify 这种开箱方案可以讲讲。"
