# 上下文隔离，分发任务时只给子Agent必要的最小的上下文；

## 答案

**一句话**：上下文隔离就是分发任务时只给 SubAgent 完成任务所需的最小信息，既降低 token 成本，也减少敏感信息泄露和任务干扰。

### 1. 为什么要上下文隔离

如果把完整对话、全部工具结果和所有业务数据都给每个 SubAgent，会带来问题：

- token 成本高。
- 子 Agent 被无关信息干扰。
- 敏感信息扩散。
- 权限边界不清。
- 审计时难以判断谁看过什么数据。

所以要按任务最小化上下文。

### 2. 怎么做

可以从几个方面控制：

- 主 Agent 只传递任务目标和必要字段。
- 子 Agent 只拿自己需要的工具结果。
- 敏感字段脱敏。
- 不同 SubAgent 使用独立上下文窗口。
- 子 Agent 只返回摘要和依据，不返回大量原始数据。
- 通过 trace 记录上下文传递内容。

### 3. 和权限的关系

上下文隔离不是只省 token，它也是权限控制的一部分。即使某个子 Agent 没有写工具，如果给它过多敏感数据，也可能造成泄露。所以数据最小化和工具最小化要一起做。

### 4. 结合跨境支付项目怎么说（面试口径）

> 在跨境支付项目里，主 Agent 处理一笔交易异常时，可能会涉及订单、风控、合规、通道、账务多个信息源。但我不会把全部信息都给每个 SubAgent。
>
> 比如风控 SubAgent 只需要交易金额、国家、设备风险、历史频次、风险命中大类，不需要看到完整银行卡号和客服聊天记录；通道 SubAgent 只需要通道名、错误码、回执和状态，不需要看到风控规则；合规 SubAgent 只需要国家、主体、材料状态和名单结果。
>
> 最后主 Agent 汇总各子 Agent 的结论，生成给用户的解释。这样上下文更干净，成本更低，也符合支付行业的数据最小化原则。

**钩子**：可以引到“最小必要上下文、敏感字段脱敏、SubAgent 独立上下文、审计追踪”。
