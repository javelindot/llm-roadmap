# RL tool 的 loss 有什么区别？

## 答案

这道题表述有歧义，面试时应先确认是在比较“工具调用 SFT 与 Tool-RL 的 loss”，还是比较“结果奖励与过程奖励”。通常可以从监督粒度、奖励归因和约束项三个层面回答。

### 1. Tool SFT 的 Loss

工具调用 SFT 仍是 token-level 交叉熵：

```text
L_SFT = -sum_t m_t log pi_theta(y_t | x, y_<t)
```

`m_t` 可只监督 assistant 和 tool-call token，屏蔽 system/user/tool result。它学习模仿标准轨迹：

```text
选择工具 -> 生成参数 -> 读取结果 -> 输出答案
```

优点是稳定，缺点是只能模仿数据中已有动作，无法直接优化真实任务成功率。

### 2. Tool RL 的 Policy Loss

Tool-RL 把一整条多步轨迹视为策略采样：

```text
state -> tool call -> observation -> next action -> final answer
```

PPO/GRPO 类目标使用 advantage：

```text
L_policy
= -E[min(r_t A_t, clip(r_t) A_t)]
  + beta * KL
```

奖励可以来自：

- 最终任务是否完成；
- 工具是否选对；
- 参数是否合法；
- 调用顺序和次数；
- 最终答案是否与工具结果一致；
- 是否触发越权或高危动作。

### 3. Outcome 与 Process Reward

**Outcome reward**

- 只评价最终结果；
- 简单、可用程序验证；
- 多步轨迹中信用分配困难。

**Process reward**

- 对每次工具选择、参数和中间状态评分；
- 信号更密集；
- 标注和 Reward Model 更复杂，也更容易把错误规则固化。

实践中常组合：

```text
R = w1 * task_success
  + w2 * tool_validity
  + w3 * evidence_consistency
  - w4 * invalid_call
  - w5 * excessive_steps
  - w6 * policy_violation
```

### 4. 与普通对话 RL 的区别

工具 RL 更容易获得可验证奖励，例如 SQL 是否执行成功、API schema 是否正确、最终数字是否匹配；但也存在环境非确定性、工具超时、错误传播和奖励作弊。训练时需要 sandbox、幂等工具和回放日志。

### 5. 跨境支付约束

- 查询类工具可在沙箱自动验证；
- 退款、冻结、调账等写操作不能让训练 Agent 直接调用生产系统；
- 奖励中必须加入权限、限额、幂等和审批约束；
- “少调用工具”不能压过“正确核验事实”，否则模型会猜交易状态。

**面试总结**：SFT loss 教模型“像标准答案一样调用”，RL loss 优化“整条轨迹是否真正完成任务”；Tool-RL 的关键难点是长程信用分配和安全环境。
