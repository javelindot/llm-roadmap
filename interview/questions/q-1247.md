# 大模型的超长上下文是怎么实现的,比如说 KIMI

## 答案

超长上下文不是把配置文件中的 `max_position_embeddings` 改大，而是训练数据、位置编码、注意力计算、推理缓存和评测共同扩展。Kimi 商业模型的所有细节没有完整公开，以下以 Moonshot 已公开研究和通用技术说明。

### 1. 位置表示能够外推

现代 LLM 常使用 RoPE。超过训练长度后，可通过：

- Position Interpolation；
- 调整 RoPE base/NTK scaling；
- YARN、LongRoPE 等频率缩放；
- 在目标长度上继续训练；

扩展位置范围。只做数学外推通常会损失远距离检索和顺序能力，因此仍需长上下文数据训练。

### 2. 长上下文训练

训练数据要覆盖：

- 长书、论文、代码仓库和多文档；
- Needle retrieval 与 key-value binding；
- 跨章节、多跳和时间顺序；
- 长输入中的无答案与冲突证据；
- 不同长度分布，而不是只训练最大长度。

常见课程学习从短序列逐步扩展到长序列，配合 sequence parallel、context parallel、FlashAttention、packing 和激活检查点控制显存。

Moonshot 公开的 Kimi k1.5 把 RL 上下文扩展到 128K，并用 partial rollout 复用已有轨迹的一大段，减少重新生成长轨迹的成本。这是 reasoning 训练中的长上下文扩展，不代表所有 Kimi 产品都使用完全相同配方。

### 3. 降低注意力复杂度

标准 full attention 的计算和注意力矩阵随长度近似二次增长。常见方案包括：

- FlashAttention：结果仍是精确注意力，主要降低 HBM I/O；
- sliding-window/local attention；
- block-sparse attention；
- global token + local block；
- linear/recurrent attention；
- 检索或 token pruning，只关注相关上下文。

Moonshot 公开的 MoBA 使用 Mixture of Block Attention，让 Query 只路由到部分相关 KV block；Kimi Linear 则采用 Kimi Delta Attention 与全局 MLA 的混合结构，公开模型支持 1M 上下文，并重点降低长序列 KV Cache 和 decode 成本。

### 4. KV Cache 与服务系统

长上下文推理的瓶颈经常是 KV Cache、prefill 和跨节点数据传输：

```text
KV memory ≈
2 * layers * sequence_length
* num_kv_heads * head_dim * bytes
```

工程优化包括：

- GQA/MQA/MLA 或线性注意力减少缓存；
- Paged KV、KV 量化和分层缓存；
- prefix cache 复用公共前缀；
- prefill/decode 分离；
- chunked prefill；
- context parallel；
- 按请求长度调度。

Mooncake 是 Moonshot 为 Kimi 公开的 KVCache-centric 分离式服务架构，重点优化长上下文场景中的 KV Cache 传输、缓存和调度。

### 5. 超长上下文不等于有效使用

模型宣称支持 128K、256K 或 1M，只说明输入接口和训练/外推范围，还要检查：

- 长度内 Needle 能否取回；
- 多个 Needle 是否混淆；
- 开头、中间、结尾位置是否均衡；
- 跨文档推理与顺序是否正确；
- 长上下文是否压低短任务质量；
- TTFT、TPOT、吞吐和成本；
- 不同语言、表格与代码表现。

“Lost in the Middle”说明模型即使窗口装得下，也可能忽视中间证据。

### 6. 跨境支付中的使用方式

长上下文可以处理长合规手册、多轮工单和多份商户材料，但不应把整个知识库和全部交易历史塞入模型。先按权限检索相关国家、币种、产品和订单，再将必要证据放入上下文，通常更准确且可审计。订单状态、余额和实时费率仍由工具提供。
