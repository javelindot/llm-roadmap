# dpo 是在其他指标都优化的情况下漏检率上升在可接受范围内，当时的话你针对这个事情是怎么调整或者说去解决并落地的？

## 答案

这是典型的指标权衡题。即使漏检率上升“在可接受范围”，也不能只看总指标批准上线，需要确认漏掉的是什么样本、业务损失多大，以及是否存在高风险切片恶化。回答应使用真实项目数据；下面给出处理框架。

### 1. 先确认指标可信

- 统一 DPO 前后的数据和解码配置；
- 检查置信区间和多随机种子；
- 按用户、语言、场景、风险等级切片；
- 人工复核漏检样本；
- 检查偏好标注是否偏向“更友好”而牺牲检测；
- 排除阈值和评测脚本变化。

### 2. 定义可接受边界

不是凭主观说可接受，而是：

```text
主指标提升 >= 目标
整体漏检率增加 <= 阈值
高风险漏检率不得上升
人工审核量在容量内
预期损失下降
```

支付欺诈、制裁和隐私泄露类漏检通常是硬护栏，不能被语气或满意度提升抵消。

### 3. 数据层调整

- 补充漏检 hard cases；
- 对高风险 chosen/rejected 提高采样权重；
- 修正偏好对中错误或含糊标注；
- 加入边界样本和反事实样本；
- 保留 SFT 安全数据 replay；
- 分阶段训练，避免 DPO 覆盖基础检测能力。

### 4. 目标函数与训练

- 调整 DPO `beta`；
- 降低学习率和 epoch；
- 使用更强 reference；
- 对关键样本增加权重；
- 多目标 loss：偏好 loss + 安全/检测 auxiliary loss；
- 使用 constrained optimization 或在训练后再做安全适配；
- 对生成任务增加可验证 reward。

不能只靠无限提高某类样本权重，否则可能造成误检或过度拒答。

### 5. 推理与系统补偿

- 重新校准阈值；
- 高风险场景使用级联检测器；
- 低置信度转人工；
- 规则和关键词作为独立护栏；
- 对关键动作二次检查；
- 灰度和实时监控；
- 一键回滚。

模型优化和系统护栏应共同工作。

### 6. 落地流程

```text
离线分层评测
-> bad case 修复
-> shadow
-> 小流量灰度
-> 主指标和漏检护栏
-> 人工复核
-> 分阶段扩量
```

### 7. 回答模板

> DPO 后主指标提升，但漏检率轻微上升。我先确认变化集中在哪些切片，并把高风险漏检设为硬护栏。随后补 hard negative、调整偏好样本权重和 beta，同时对低置信度样本使用二级检测/人工复核。上线采用灰度并监控预期损失，而不是只看整体漏检率。具体数值按真实项目实验说明。

### 8. 面试总结

解决方式是“分层风险评估 + 数据和 DPO 调整 + 阈值/级联补偿 + 灰度监控”。可接受范围必须由业务损失和关键切片定义，而不是用整体平均掩盖高风险漏检。
