# AgentExecutor 的创建方式

## 答案

**一句话**：`AgentExecutor` 是 LangChain 旧版串起 "Agent + 工具 + 循环控制" 的运行器，新版趋势是用 **LangGraph** 替代——所以答这题要既会用，也得知道为什么要替代。

### 1. LangChain 旧版三种主流创建方式

```python
from langchain.agents import AgentExecutor, create_react_agent

# 方式 1: ReAct（最经典）
agent = create_react_agent(llm, tools, prompt)
executor = AgentExecutor(agent=agent, tools=tools, verbose=True)

# 方式 2: OpenAI Tools / Function Calling
from langchain.agents import create_openai_tools_agent
agent = create_openai_tools_agent(llm, tools, prompt)

# 方式 3: Structured Chat（结构化输入）
from langchain.agents import create_structured_chat_agent
agent = create_structured_chat_agent(llm, tools, prompt)
```

### 2. AgentExecutor 关键参数
| 参数 | 作用 | 建议值 |
|------|------|--------|
| `agent` | Agent 实例 | 必填 |
| `tools` | 工具列表 | 必填 |
| `max_iterations` | 防死循环 | 10-15 |
| `max_execution_time` | 总超时 | 60s+ |
| `memory` | 对话记忆 | 短任务用 Buffer |
| `handle_parsing_errors` | 容错 | True（生产必开） |
| `return_intermediate_steps` | 保留中间步骤 | True（要 trace） |
| `early_stopping_method` | 提前停止策略 | "force" 或 "generate" |

### 3. 为什么社区在切到 LangGraph
- AgentExecutor 是黑盒 while 循环，**分支、并行、回滚、人工介入都不好做**。
- LangGraph 用**状态图（节点 + 边 + 状态）**显式建模工作流：
  - 可以画出来
  - 可以中断 / 恢复 / 回放
  - 可以加 HITL 节点
  - 状态序列化方便做持久化
- 官方文档现在主推 LangGraph，AgentExecutor 进入维护期。

### 4. 自己手写一个 mini AgentExecutor
```python
def run_agent(llm, tools, prompt, max_iter=10):
    state = {"messages": [prompt], "steps": []}
    for _ in range(max_iter):
        out = llm.invoke(state["messages"], tools=tools)
        if out.tool_calls:
            for call in out.tool_calls:
                result = dispatch(call, tools)
                state["messages"].append(result)
                state["steps"].append((call, result))
        else:
            return out.content, state["steps"]
    raise RuntimeError("max_iter reached")
```
不到 20 行——这也是为什么很多团队不用框架直接手写。

### 5. 一句话总结
**AgentExecutor 旧版能跑、新项目建议直接上 LangGraph**。能讲出"为什么切"比讲出"怎么用"更值钱。

## 结合项目谈（口语版 · 约 40 秒）

> 我们跨境支付一开始风控调查 Agent 就是 LangChain `AgentExecutor` 起的家，`create_react_agent` 配一组只读工具，跑得起来。
>
> 但跑着跑着遇到两个坎。一个是 HITL——商户准入到批准这一步必须停下来等运营点，AgentExecutor 那个 while 循环没法优雅地"暂停-恢复"，我们硬塞了一个"等待工具"返回 special token、外层 catch 这个 token 再回写，能跑但很别扭。第二个是分支和回滚——对账 Agent 跑到一半发现假设错了要回滚，AgentExecutor 一旦进了循环只能跑完，回滚得全靠自己维护状态。
>
> 后来我们就切到 **LangGraph**。状态图建模之后，HITL 是一个标准节点，暂停-恢复是框架原生支持的；分支用条件边表达；状态可以 dump 到 Redis，崩了能从中间步恢复。代码量虽然多了点，但可维护性和可观测性提升非常大。
>
> 现在新项目我们基本不再用 `AgentExecutor` 了，老项目维护性能改也只改不重写。如果是小工具脚本，反而我会直接手写一个 20 行的循环，框架反而是负担。

**🪝 顺嘴带出去的钩子**：
- "我们 AgentExecutor 切 LangGraph 的迁移成本和收益可以聊聊。"
- "LangGraph 的 HITL 节点在金融场景怎么落地的我们有具体设计。"
- "Agent 状态持久化我们用 Redis + Checkpoint 的方案可以展开。"
