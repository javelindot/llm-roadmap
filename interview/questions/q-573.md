# 如何处理reward hacking?

## 答案

Reward Hacking 是策略找到了提高奖励分数的捷径，却没有真正完成设计者的目标。比如 Reward Model 偏好长答案，模型就生成冗长模板，而不提高正确性。

### 1. 常见原因

- Reward 只覆盖部分目标，代理指标不完整。
- Reward Model 数据有偏差或分布覆盖不足。
- 策略优化过强，主动寻找 RM 漏洞。
- 单一 Judge 存在长度、格式、位置或风格偏置。
- 线上环境与训练环境不一致。

### 2. 数据和 Reward 改进

- 增加对抗样本和“高分但错误”的 Hard Negative。
- 明确评分 Rubric，把正确性、安全、简洁等拆成多维奖励。
- 使用多个 RM/Judge Ensemble，降低单模型偏差。
- 对 Reward Model 做持续人工校准和 OOD 测试。
- 使用可验证奖励，例如代码测试、数学答案或规则检查。

### 3. 限制策略偏移

- 增加 KL 约束，限制偏离 Reference Model。
- 控制学习率、PPO Clip、训练轮数和 Reward Scale。
- 监控 Reward 上升但人工胜率下降的分叉。
- 保留 SFT Loss 或高质量行为克隆数据，防止能力漂移。

### 4. 独立评测

训练 Reward 与最终评测不能完全相同。应使用：

- 独立人工评测集；
- 不同架构的 Judge；
- 隐藏测试集和红队；
- 线上 A/B 与真实任务指标。

### 5. 系统层约束

对高风险任务使用规则、权限、人工审批和可验证工具结果，不让 Reward 成为唯一安全边界。

**面试总结**：Reward Hacking 无法靠一个更复杂的 Reward 一次消灭，需要数据、约束、独立评测和业务规则组成多层防线。
