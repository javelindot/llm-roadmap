# 如果客户要求至少要 90% 的准确率，你有什么解决方案？

## 答案

> 我们现在是 88%，要做到 90%+，我会从**召回、生成、拒答、回流**四块同时下手：
>
> **第一块：召回侧（最有空间）**
> - 扩 embedding 微调数据从 5w 到 10w，重点补充长尾业务术语
> - 升级 embedding 到 Qwen3-Embedding-8B 离线索引（在线仍用 bge）
> - 多 query Fusion 在低分召回场景启用
> - 文档侧：补充缺失内容（line up 失败 case 分析的高频 missing 主题）
>
> **第二块：生成侧**
> - 客服模型 DPO 加强——把"答非所问""信息编造"作为 rejected 样本
> - prompt 工程：强化"必须从召回结果引用，否则拒答"约束
> - 多模型投票：关键场景上 Qwen-72B + Qwen-14B 双模型一致性检查
>
> **第三块：拒答机制（关键！）**
> - **设阈值**：rerank score < 0.4 直接转人工
> - **不确定性检测**：让 LLM 输出自信度，低于 0.7 也转人工
> - **降低 coverage 换 accuracy**：宁可拒答 15%，剩下 85% 准确率到 95%
> - 这是金融场景常用 trick——**准确率指标算"已回答里的对错"**
>
> **第四块：持续优化飞轮**
> - 每天 sample 200 条人工标（现在 100 条）
> - 失败 case 月度复盘，定位"召回失败 / 生成失败 / 知识缺失"三类
> - 月度迭代 embedding + DPO 数据
>
> **预估收益**：
> - 召回优化 +2 个点 → 90%
> - 生成优化 +1.5 个点 → 91.5%
> - 拒答机制 +2 个点（已答里的准确率） → 93.5%
> - 飞轮三个月内再 +1 个点 → 94%+
>
> **诚实说**：90% 在客户问题分布稳定的场景能达到，但**新业务长尾问题**短期还是会拖累整体数字。所以我会跟客户先对齐"90% 是怎么算的"——按已回答算还是按总流量算，差别很大。
