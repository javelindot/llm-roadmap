# Function Call 工具过多导致准确率下降，你是如何解决的？

## 答案

Function Call 工具数量增加后，准确率下降通常来自四个原因：

- 模型需要在大量相似描述中选择，决策空间过大；
- 工具命名和职责重叠，例如 `query_payment`、`get_payment`、`check_payment`；
- JSON Schema 过于宽松，缺少必填、枚举和边界；
- 所有工具无差别暴露，既增加 token，也扩大越权和误操作风险。

解决方案不是单纯换更大的模型，而是：

> 权限过滤 → 工具分组路由 → 动态检索少量工具 → 结构化参数校验 → 执行后验证。

### 1. 建立统一 Tool Registry

每个工具登记：

> `tool_id、namespace、capability、intent、risk_level、roles、input_schema、output_schema、side_effect、owner、version`

工具名称使用稳定命名空间，例如：

- `order.get_status`
- `channel.get_receipt`
- `knowledge.search_rule`
- `ticket.create_draft`

避免语义相近的多个别名。描述不仅写“做什么”，还要写“什么时候用、什么时候不用、需要哪些前置条件、是否有副作用”。

### 2. 先按权限和场景裁剪

模型看到工具前，后端根据用户角色、租户、商户、业务线和风险等级过滤：

- 客服 Agent 只看到订单查询、规则检索和工单草稿；
- 风控 Agent 只看到调查所需的只读风险工具；
- 通道 Agent 只看到回执和健康状态；
- 退款、付款、冻结等写工具不出现在普通对话 Agent 的工具列表。

最小权限既提高选择准确率，也减少提示注入后的攻击面。

### 3. 使用分层路由而不是一次选择几十个工具

第一层选择能力域：

> `knowledge、order、channel、accounting、risk、ticket`

第二层只在该域内选择具体工具。高频、边界明确的意图可使用规则或小分类器直接路由；长尾问题再由 LLM 判断。

当工具仍然很多时，对工具描述建立检索索引，根据用户问题召回 Top-K 工具，再将少量 schema 注入模型。检索条件同时包含语义、权限、风险和前置状态，不能只做向量相似。

例如“USD 付款未到账”只需要暴露：

- `order.get_status`
- `channel.get_receipt`
- `settlement.get_batch_status`
- `knowledge.search_payout_rule`

没有必要同时暴露换汇、商户准入和退款执行工具。

### 4. 优化工具定义和参数 Schema

工具定义应做到互斥、具体和机器可校验：

- 参数使用明确类型、枚举、格式、长度和范围；
- 必填字段与可选字段分开；
- 金额使用整数最小货币单位或明确 decimal 规则；
- 币种使用 ISO 代码枚举；
- 订单号、商户号和国家码提供格式约束；
- 输入和输出都定义 schema；
- 标注只读、破坏性、幂等和开放世界属性。

避免使用一个包含大量可选字段的“万能工具”。如果查询订单和退款是不同权限、不同副作用，应拆为独立工具。

模型缺少必填参数时应先追问，不能猜测订单号、金额或币种。

### 5. 将复杂任务拆成计划和原子调用

“查失败原因并重新付款”不能一步映射到写工具。应拆为：

1. 查询订单状态；
2. 查询通道回执；
3. 检索失败码和重试规则；
4. 判断是否允许重试；
5. 生成重试建议；
6. 需要执行时进入人工审批。

每一步只开放当前状态允许的工具，工具集合随状态变化。这样比始终把全部工具交给模型更准确、更安全。

### 6. 执行层必须二次校验

Function Call 只是模型提出的调用意图。工具网关仍要验证：

- 当前身份是否有权限；
- 对象是否属于当前商户；
- 参数、状态、限额和风险是否满足；
- 幂等键是否存在；
- 工具版本是否兼容；
- 是否需要人工确认。

工具结果使用结构化输出，并明确 `success、error_code、retryable、data_version`。模型不能根据一段模糊错误文本自行判断是否重试。

### 7. 训练与评测

评测集需要覆盖：

- 工具选择正确率和 Top-K Recall；
- 参数完整率、类型正确率和实体绑定正确率；
- `no_tool`、追问和转人工判断；
- 相似工具、工具不可用和权限不足；
- 多工具顺序与任务最终成功率；
- 未授权调用率、重复调用率和错误写操作率；
- token、延迟和单任务调用数量。

针对高频混淆对加入 hard negative，例如区分“查询退款状态”和“发起退款”。工具新增、删除或描述变更后必须跑回归集。

MCP 2025-11-25 规范支持工具输入与输出 Schema、能力协商和工具列表变更通知，这些有助于标准化动态工具目录；但具体的权限过滤、审批和业务状态校验仍由应用实现。
