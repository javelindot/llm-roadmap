# 介绍一下了解的其他基于 MoE 的模型架构

## 答案

MoE 通常把 Transformer 的 Dense FFN 替换成多个专家，由 Router 为每个 token 选择少量专家：

\[
y=\sum_{i\in \operatorname{TopK}(g(x))}
p_iE_i(x)
\]

这样总参数可以很大，但每个 token 只激活一部分参数。代表架构可按演进介绍。

### 1. 经典稀疏 MoE

- **Sparsely-Gated MoE**：使用可学习门控选择少量专家，奠定现代稀疏专家基础；
- **GShard**：把 MoE 与大规模自动分片结合，使用 Top-2 routing；
- **Switch Transformer**：每个 token 只路由到 Top-1 专家，简化通信和训练，并使用 load-balancing loss；
- **GLaM**：大规模稀疏语言模型，每 token 激活少量专家。

### 2. 开源 LLM 中的 MoE

- [Mixtral 8x7B](https://arxiv.org/abs/2401.04088)：每层 8 个 FFN 专家，每 token 选 2 个；约 47B 总参数、13B 激活参数；
- **Qwen1.5-MoE**：使用细粒度专家和共享专家思想，提高专家组合灵活性；
- [DeepSeekMoE/DeepSeek-V3](https://arxiv.org/abs/2412.19437)：细粒度路由专家加共享专家；V3 为 671B 总参数、每 token 激活 37B，并采用无辅助损失负载均衡；
- [Qwen3-MoE](https://qwenlm.github.io/blog/qwen3/)：例如 30B-A3B 和 235B-A22B，均有 128 个专家、每 token 激活 8 个；
- DBRX 等模型也采用细粒度专家与 Top-k routing。

### 3. 核心难点

- Router 容易专家负载不均、token drop 或专家塌缩；
- Expert Parallel 需要 all-to-all，通信可能超过计算收益；
- 总权重仍需存储，激活参数少不等于部署内存小；
- 小 batch、长尾 token 和多机拓扑会降低专家利用率；
- 微调时部分专家可能收不到足够数据，需监控路由分布。

### 面试总结

MoE 用稀疏路由把模型容量与每 token 计算解耦。Switch 强调 Top-1 简化，Mixtral 使用 Top-2，DeepSeekMoE 使用细粒度路由与共享专家，Qwen3 则提供大、小规模 MoE。真正瓶颈常在负载均衡、all-to-all 通信和总权重存储。
