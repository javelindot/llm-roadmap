# 项目中的状态数据采用何种方式存储？存储方案的设计思路是什么？

## 答案

**一句话**：项目状态数据一般不会只放在模型上下文里，而是根据生命周期和一致性要求，分别放在内存、Redis、数据库、消息队列或工作流状态表中。

### 1. 状态数据有哪些

Agent 或业务流程中的状态包括：

- 当前会话状态。
- 当前任务执行到哪一步。
- 已调用过哪些工具。
- 工具返回结果。
- 用户身份和权限。
- 风险标记和校验结果。
- 异步任务进度。
- 失败次数、重试次数和异常信息。
- 最终输出和审计记录。

这些状态的保存方式要看是否需要持久化、是否需要恢复、是否高频读写。

### 2. 存储方案选择

常见设计是：

- **内存**：适合单次请求内的临时状态。
- **Redis**：适合会话态、短期任务状态、分布式锁、幂等 token。
- **关系型数据库**：适合交易、订单、工单、审核记录等强一致和可审计数据。
- **消息队列**：适合异步状态流转，比如文档解析完成、风控评估完成。
- **对象存储**：适合大文件、工具大结果、日志附件。
- **向量库/ES**：适合知识检索和文本召回，不适合承载核心交易状态。

Agent 的流程状态如果需要恢复和回放，可以单独设计 state 表或使用 LangGraph checkpoint 这类机制。

### 3. 设计原则

- 核心业务状态必须持久化，不能只靠模型记忆。
- 短期会话状态可以放 Redis，但要设置过期时间。
- 涉及资金和合规的状态要有审计记录。
- 状态流转要有幂等设计，防止重复回调或重复执行。
- Agent state 和业务 state 要区分，不能混在一起。

### 4. 结合跨境支付项目怎么说（面试口径）

> 在跨境支付项目里，状态数据我会分层存。交易订单、付款状态、清结算状态这类核心数据必须落数据库，因为它们涉及资金和审计；客服会话、Agent 当前步骤、临时工具结果可以放 Redis 或 Agent checkpoint；PDF 解析结果、大模型工具输出这类大内容可以放对象存储，再在状态里保存引用。
>
> 比如一笔付款从创建、风控审核、通道处理中、银行成功、入账完成，中间每一步都要可追踪。Agent 可以解释这个过程，但不能成为交易状态的唯一存储。真正的状态源一定是支付系统和账务系统。
>
> 我会特别关注幂等和回放。跨境支付里三方回调可能重复、延迟、乱序，所以状态流转要校验前置状态和版本号，避免一条旧回调把新状态覆盖掉。

**钩子**：可以引到“Agent state 和业务 state 分离”“支付状态机”“幂等与乱序回调处理”。
