# RAG 各块有哪些优化策略？

## 答案

**一句话**：RAG 优化要分阶段看：文档解析、切块、索引、召回、重排、上下文构造、生成和评估，每一层都可能影响最终效果。

### 核心回答
- **解析优化**：保留标题、表格、图片、页码和元数据。
- **切块优化**：语义切分、overlap、父子 chunk。
- **索引优化**：向量索引、关键词索引、元数据过滤。
- **召回优化**：混合检索、多路召回、query rewrite、HyDE。
- **重排优化**：reranker、cross-encoder、业务规则加权。
- **上下文优化**：去重、压缩、排序、引用来源。
- **生成优化**：约束模型只基于资料回答，无法确定时拒答。
- **评估优化**：看 answer correctness、faithfulness、context recall。

### 1. RAG 优化要先定位瓶颈

RAG 效果差时不能直接怪大模型。要先定位是哪一层出问题：

- 文档没解析出来：解析层问题。
- 正确内容没召回：检索层问题。
- 召回了但排在后面：重排问题。
- 上下文太乱：拼接和压缩问题。
- 有证据但答错：prompt 或模型问题。
- 没证据还硬答：拒答和安全策略问题。

### 2. 常见组合优化

比较实用的组合是：

- 结构化解析 + 语义切块。
- BM25 + 向量混合召回。
- 元数据过滤。
- reranker 重排。
- prompt 强制引用。
- 低置信拒答。
- bad case 驱动迭代。

### 3. 面试总结

RAG 优化是系统工程，不是单点换 embedding 或换模型。要分层诊断、分层优化。

### 4. 结合跨境支付项目怎么说（面试口径）

> 跨境支付 RAG 我会重点优化三块。第一是文档解析，费率表、国家限制表、错误码表不能乱；第二是混合检索，因为用户可能问"巴西 PIX"这种关键词，也可能问"为什么这笔本地转账失败"这种语义问题。
>
> 第三是引用和拒答，合规场景不能让模型编。比如没有检索到某国家的明确规则，就应该提示未找到依据，而不是根据相似国家规则猜一个答案。
>
> 我的目标不是让答案更花哨，而是让运营能追溯到具体规则来源，并减少查文档时间。

**钩子**：支付 RAG 的核心是准确、可追溯、可拒答，不是单纯生成流畅。
