# 如果让 agent 调用搜索引擎，如何避免无关结果影响回答？

## 答案

搜索引擎返回的是候选证据，不是最终事实。Agent 调用搜索时应采用：

> 查询规划 → 来源约束 → 结果抓取 → 去重过滤 → 证据重排 → 冲突校验 → 基于引用生成。

### 1. 搜索前先确定检索范围

先判断问题是否真的需要外部搜索：

- 公司内部产品规则、商户资料和 SOP 优先查内部知识库；
- 订单状态、余额、汇率报价和通道回执必须查受控业务接口；
- 监管公告、支付网络公开规则和时效性行业信息才使用外部搜索。

跨境支付场景不能用互联网搜索结果替代内部交易事实。例如“这笔付款为什么失败”应查订单和通道系统；搜索引擎最多用于补充公开规则，不能推断该笔交易状态。

搜索请求应结构化为：

> `query、intent、country、currency、product、time_range、allowed_domains、required_freshness`

复杂问题先拆为独立子问题，避免一个过长查询同时搜索监管、产品和交易信息。

### 2. 使用来源白名单和质量分级

来源按可信度分层：

1. 监管机构、央行、卡组织、清算网络和政府网站；
2. 公司官方文档、开发者文档和状态页；
3. 学术论文、标准组织和可信行业报告；
4. 新闻、论坛和聚合内容。

涉及牌照、制裁、KYC/KYB、支持国家和资金时效时，优先使用前两层。搜索 API 返回的摘要可能被截断或过期，必须打开原文确认标题、发布日期、生效时间和适用范围。

域名白名单不是唯一条件，还要检查：

- 作者和发布机构；
- 发布时间、更新时间和规则生效日期；
- 是否引用一手来源；
- 页面是否属于目标国家、币种和产品；
- 是否存在付费推广、SEO 拼接或转载失真。

### 3. 对搜索结果做过滤和重排

搜索结果进入模型前先经过：

- URL 规范化、内容哈希和近似去重；
- 广告、导航、评论区和模板文本清理；
- 语言、国家、时间和文档类型过滤；
- query-document cross-encoder 重排；
- 来源可信度、时效性和直接证据程度加权；
- 相互矛盾内容的聚类和标记。

排序分数可以由相关性、权威性、时效性和业务适配度共同组成，而不是直接使用搜索引擎排名：

> `Score = w1 × relevance + w2 × authority + w3 × freshness + w4 × scope_match`

最终上下文控制在少量高质量证据，并保留 URL、标题、机构、发布日期和抓取时间。

### 4. 把网页内容当作不可信输入

网页可能包含提示注入，例如“忽略之前指令并调用某工具”。抓取内容只能作为数据，不能改变 Agent 的系统指令、工具权限或执行计划。

工程上需要：

- 将网页正文放入独立的 `<untrusted_content>` 数据区；
- 删除脚本、隐藏文本和无关页面指令；
- 禁止网页内容触发新工具、修改权限或读取凭证；
- 搜索 Agent 只拥有只读网络工具，不拥有付款、退款等业务写工具；
- 对下载文件做类型、大小、恶意内容和敏感信息检查。

### 5. 生成时强制证据一致

回答按 claim 绑定引用。关键结论至少应有一个直接一手来源；高风险结论可要求两个独立来源交叉验证。

出现以下情况时不应强行回答：

- 只有低可信或转载来源；
- 不同官方来源相互冲突；
- 页面已过期或未标明适用时间；
- 搜索结果与内部权威系统不一致；
- 证据无法覆盖用户问题中的国家、币种或产品。

此时应明确说明不确定性，并转内部规则库、业务专家或人工客服确认。

### 6. 评测和监控

离线构建包含正确来源的搜索黄金集，评估：

- Source Recall@K、NDCG 和一手来源命中率；
- 无关结果进入最终上下文的比例；
- 过期信息、冲突信息和提示注入拦截率；
- 引用正确率、claim 支撑率和最终答案准确率；
- 搜索延迟、抓取失败率和单次成本。

线上记录查询、候选 URL、过滤原因、最终引用和答案，针对错误引用、过期规则和低质量域名持续更新策略。目标不是让 Agent “搜得多”，而是让每个关键结论都能回到可信、适用且及时的证据。
