# 为什么需要对文本分块？

## 答案

**一句话**：文本分块是为了让长文档变成适合 embedding、检索和放入模型上下文的小片段。

### 核心回答
为什么需要分块：
- embedding 模型和 LLM 都有长度限制。
- 太长的 chunk 召回不精准。
- 太短的 chunk 语义不完整。
- 分块能提高检索粒度，减少无关上下文。

常见策略：
- 固定长度切分。
- 按标题、段落、列表、表格切分。
- 滑动窗口 overlap。
- 语义切分。
- 父子 chunk：小块召回，大块提供上下文。

关键是平衡"召回精准"和"语义完整"。

### 1. 分块太大和太小的问题

chunk 太大：

- embedding 表示会变得模糊。
- 检索命中不精准。
- 放进 prompt 成本高。
- 无关信息容易干扰生成。

chunk 太小：

- 语义不完整。
- 规则条件和例外条款可能被拆开。
- 模型回答时缺上下文。

### 2. 好的分块策略

好的分块通常结合：

- 文档结构：标题、章节、段落。
- 业务语义：一条规则、一张表、一个流程。
- overlap：保留上下文衔接。
- metadata：国家、币种、通道、版本。
- 父子 chunk：小块检索，大块生成。

### 3. 面试总结

分块是 RAG 的底层质量工程。它决定了检索能不能既准又完整。

### 4. 结合跨境支付项目怎么说（面试口径）

> 支付文档切片不能只按固定 token 切。比如一个通道的国家限制、币种、手续费和结算周期通常是一张表或一个章节，切断了就会答错。
>
> 我会优先按标题和业务语义切，比如"入网要求""拒付规则""结算周期"分开；表格作为完整单元保留；必要时加 overlap。每个 chunk 还会带国家、币种、通道、文档版本等元数据。
>
> 这样运营问具体规则时，可以先按业务字段过滤，再检索，召回内容更完整，也减少模型混用不同国家规则的风险。

**钩子**：RAG 不是切得越碎越好，支付规则要保持业务语义完整。
