# 召回索引怎么优化

## 答案

召回索引优化应从质量、延迟、内存、构建更新和稳定性共同考虑。先根据向量规模、维度、查询分布和 Recall-SLO 选择索引，而不是只调一个 ANN 参数。

### 索引结构

- HNSW：召回高、低延迟，但图内存大，构建与删除较复杂；调 `M`、`efConstruction`、查询 `efSearch`。
- IVF：先聚类到倒排桶，查询探测 `nprobe` 个桶；调 `nlist/nprobe` 平衡速度和召回。
- PQ/OPQ：压缩向量降低内存和带宽，代价是距离近似误差。
- DiskANN/分层冷热索引：超大规模时使用 SSD 和内存导航结构。

### 数据与向量

embedding 做归一化并保证线上线下同版本；必要时降维、量化和去除异常向量。按租户、地域、类目或时间分片，先路由再检索。热门 item、长尾和重复向量需单独分析。

### 更新与服务

主索引加增量 delta index，周期性合并；使用蓝绿版本或 alias 原子切换。删除、下架和权限变化需及时进入过滤层，不能等待全量重建。副本、缓存和批量查询提升吞吐。

### 查询优化

根据 query 难度动态设置 `efSearch/nprobe`；先做元数据过滤，或使用支持 filtered ANN 的结构。与 BM25 多路召回并由 reranker 融合，弥补量化和语义召回缺陷。

压测要使用真实 query 和正例，画 Recall@K、P95/P99、QPS、内存与构建时间曲线。索引参数的最佳点由业务 SLO 决定。
