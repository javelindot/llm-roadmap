# 主 Agent 和子 Agent 怎么划分边界？除了提示词，还有其他方法吗？

## 答案

> 划分边界的方法**远不止 prompt**——prompt 只是软约束，工程上靠**独立微服务、权限隔离、独立微调、工具白名单、上下文隔离**五个硬约束才稳。
>
> **边界划分的五个层次**：
>
> **1. Prompt 约束（最弱，软约束）**
>
> 在 system prompt 里写"你只能做 X，不要做 Y"。
>
> 优点：实现简单
> 缺点：LLM 可能漏 prompt 偷越界（jailbreak）
>
> **2. 独立微服务（架构层）**
>
> 详见 q-393，每个 Agent 是一个独立微服务：
> - 独立部署、独立 release
> - 独立的进程 / 容器 / 资源
> - 通过 HTTP / gRPC 通信
> - 不共享内存、不共享上下文
>
> 这是**最硬的边界**——子 Agent 想越界，**根本不知道其他 Agent 的存在**。
>
> **3. 数据权限（数据层）**
>
> 详见 q-395，每个子 Agent 只能访问授权数据：
> - 数据库账号分级（只读 / 业务线只读 / admin）
> - Agent 的工具调用走代理，代理做权限校验
> - 越权调用直接拒绝 + 告警
>
> 例：
> - 客服 Agent：只能读"客户 FAQ 库 + 政策库"
> - 风控 Agent：只能读"风控规则库 + 客户档案"，不能读"商户合同库"
> - KYB Agent：只能读"商户准入库"
>
> **4. 独立微调（模型层）**
>
> 详见 q-397，每个子 Agent 用独立微调的 LoRA / 全参：
> - 模型本身只会做"它擅长的事"
> - 不让一个万能模型干所有
> - 改一个 Agent 不影响其他
>
> 例：
> - 客服 Agent：Qwen-7B + 客服话术 DPO
> - 风控 Agent：Qwen-14B + 风控判定 SFT + 长 CoT
> - KYB Agent：Qwen-VL-7B + 证件识别 SFT
>
> **5. 工具白名单（执行层）**
>
> 每个 Agent 注册时声明可用工具列表：
>
> ```python
> @register_agent("customer_service")
> class CustomerServiceAgent:
>     allowed_tools = ["query_order", "query_kyc", "query_rate"]
>     # 不能调 freeze_account / send_alert 等高危工具
> ```
>
> 工具调用层强制校验：
> ```python
> async def call_tool(agent_name, tool_name, args):
>     if tool_name not in get_agent(agent_name).allowed_tools:
>         raise PermissionError(f"{agent_name} cannot use {tool_name}")
>     ...
> ```
>
> **6. 上下文隔离（信息层）**
>
> 主 Agent 调子 Agent 时，**只传必要信息**：
>
> ```python
> async def main_agent(query):
>     # 子 Agent 只看到自己需要的信息
>     subagent_input = {
>         "task": "extract_kyb_fields",
>         "doc_path": "/uploads/abc.pdf"
>     }
>     # 不传整个 conversation history
>     result = await subagent.invoke(subagent_input)
>     return main_summarize(result)
> ```
>
> 这是 Claude Code 的 Subagent 设计哲学——**上下文隔离**比 prompt 约束更可靠。
>
> **我们跨境支付的具体设计**：
>
> ```
> 主 Agent（客服调度）
>   ↓ 路由
> ├── 客服 Subagent（只读 FAQ + 政策）
> ├── 订单查询 Subagent（只读订单库 + read-only API）
> ├── KYB Subagent（只读商户库 + VLM 模型）
> └── 风控 Subagent（独立微服务 + 高权限工具，但调用要 audit log）
> ```
>
> 五个层次同时上：
> - 独立微服务：每个 Subagent 一个 pod
> - 数据权限：每个 Agent 有专属数据库账号
> - 独立微调：客服 / 风控 / KYB 用不同模型
> - 工具白名单：工具注册时绑定 Agent
> - 上下文隔离：主 Agent 调子 Agent 只传摘要
>
> **几个工程经验**：
>
> 1. **prompt 约束只是第一层，必须有硬约束兜底**
>
> 2. **权限要在工具层而非 Agent 层校验**——Agent 跳过了工具层防护就失守
>
> 3. **审计日志**：每个跨 Agent 调用都入 trace，可追溯
>
> 4. **失败隔离**：一个子 Agent 挂不影响其他
>
> 5. **资源隔离**：CPU / 内存 / GPU 配额，防止一个 Agent 占满资源
>
> **不推荐的设计**：
>
> - 一个 Agent 类内 if-else 分支当多 Agent（没隔离）
> - 共享 LLM 实例 + 不同 prompt（context 容易混）
> - 只用 prompt 约束没有工具白名单（必被 jailbreak）
>
> 一句话：**Agent 边界 = prompt + 微服务 + 权限 + 微调 + 工具 + 上下文**——五层硬约束才能在工业上稳定，prompt 只是第一层。
