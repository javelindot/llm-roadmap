# 文搜图这个任务，几个人开发？多大数据量？

## 答案

**一句话**：文搜图项目可以介绍成“文本 query 通过多模态 embedding 检索图片库”，核心是数据清洗、图文向量化、索引构建、召回排序和评估。

### 1. 任务怎么理解

文搜图就是用户输入一段文字，比如“蓝色背景的付款成功截图”或“包含银行回单的图片”，系统从图片库里找到语义最匹配的图片。

核心流程是：

1. 图片入库，提取图片特征和 OCR 文本。
2. 用 CLIP 或类似多模态模型生成图向量。
3. 用户 query 生成文本向量。
4. 在向量库里做相似度检索。
5. 结合 OCR、标签、业务规则做重排。
6. 返回相似图片和命中原因。

### 2. 人员和数据量怎么说

面试时可以给一个可信的项目口径：

> 这个项目一般 3 到 5 个人比较合理，包括后端/算法/前端/测试或数据标注。PoC 阶段可能先用几万到几十万张图片验证效果，生产阶段如果接入业务素材库，数据可以到百万级甚至千万级。

如果你没有真实精确数字，可以说“我参与的部分主要负责检索链路和工程接入，数据量按阶段逐步扩展”，避免把话说死。

### 3. 技术难点

- 图片质量参差不齐。
- 文本描述和图片内容存在语义鸿沟。
- OCR 内容、图片语义和人工标签需要融合。
- 相似图片很多，需要去重和重排。
- 向量库规模变大后要考虑性能和成本。

### 4. 结合跨境支付项目怎么说（面试口径）

> 如果结合跨境支付场景，文搜图可以用于搜索回单、错误截图、KYC 材料截图或运营素材。比如客服输入“某通道付款失败截图”或者“包含 SWIFT 回单的图片”，系统能从历史附件里找到类似图片，辅助客服判断问题。
>
> 技术上我会把图片本身的视觉向量、OCR 识别出来的文字、业务标签一起用于检索。比如银行回单图片里 OCR 到了 bank reference、currency、amount，这些文本信息对支付场景非常关键，不能只靠图片向量。
>
> 项目人员上，一个小团队就能先做 PoC：算法负责模型和向量效果，后端负责入库检索服务，前端负责检索页面和标注反馈。数据量可以先从几万张脱敏附件做验证，再逐步扩展。

**钩子**：可以引到“多模态 embedding、OCR 融合、支付附件检索、从 PoC 到百万级扩展”。
