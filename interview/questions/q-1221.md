# 在不同的项目中，都使用了DPO，有哪些区别？

## 答案

DPO 的数学形式可以相同，但不同项目的 **偏好定义、数据来源、负样本难度、风险约束和评测指标** 往往完全不同，不能把一套数据模板和超参数直接复制到所有场景。

标准偏好样本为 `(x, y_w, y_l)`，DPO 让策略相对参考模型提高 chosen 的隐式奖励、降低 rejected 的隐式奖励：

```text
L_DPO = -log sigmoid(
    beta * (
        log pi_theta(y_w | x) - log pi_ref(y_w | x)
        - log pi_theta(y_l | x) + log pi_ref(y_l | x)
    )
)
```

### 1. 跨境支付客服回答

目标是答案准确、合规、清晰并给出可执行下一步。

**偏好对构造：**

- chosen：引用正确规则，区分“处理中、失败、退汇、拒付”，不承诺未确认到账时间；
- rejected：业务状态混淆、虚构费率、遗漏关键材料、泄露敏感字段或表达过度冗长；
- 对事实型问题先用交易状态和知识库验证，再由业务人员判断表达质量。

**训练特点：**

- 以 response-level DPO 为主；
- 控制 chosen/rejected 的长度和风格差异，避免模型只学会“更长”；
- 对多语言和国家规则分桶采样；
- 评测正确率、faithfulness、拒答边界、转人工准确率和重复工单率。

### 2. 支付工具调用与 Agent

目标从“回答更好”变成“动作序列正确且安全”。

**偏好对构造：**

- 根据 API Schema、沙箱执行和最终状态自动形成部分偏好；
- chosen 使用正确工具、参数、调用顺序和幂等键；
- rejected 包括工具选错、币种或金额单位错误、跳过鉴权、重复提交、未确认即执行；
- 既比较最终回答，也比较中间 action trajectory。

**训练特点：**

- 程序化验证器和执行反馈占比更高；
- 高风险错误不能仅靠相对偏好降低概率，还要有 hard constraint；
- 评测工具成功率、参数准确率、任务完成率、调用次数、越权率和副作用。

DPO 只学习离线偏好，不会自动解决长程 credit assignment。多步任务还需要过程监督、环境评测或在线 RL。

### 3. KYC/KYB 文档与多模态任务

输入同时包含图片、OCR、表格或 PDF，目标是回答必须由视觉证据支持。

**偏好对构造：**

- 同一文档、同一问题下比较 chosen/rejected；
- chosen 正确抽取公司名、证件有效期、地址或字段状态；
- rejected 使用相似但错误的实体、漏读表格、把别页信息串入当前页，或根据语言先验猜测；
- 增加 image-contrast pair，要求模型区分视觉内容变化，而不是只学文本措辞。

**训练特点：**

- 标准 DPO 可能忽略图像条件，只学习语言侧偏好；
- 要检查移除或替换图像后模型是否仍给出相同答案；
- 可使用 image preference、视觉对比负样本或 MDPO 类目标强化条件依赖；
- 评测字段级 F1、OCR 鲁棒性、视觉 grounding、幻觉率和跨页一致性。

### 4. 超参数不会完全相同

`beta` 控制策略相对 reference 的偏离强度，但有效范围受数据 margin、回答长度、模型规模和 reference 质量影响。不同项目需要分别确定：

- SFT checkpoint 和 reference model；
- beta、学习率、batch、最大长度；
- pair 去重、难度分布和 chosen/rejected 长度差；
- 是否只训练 LoRA、是否更新视觉投影层；
- 多个目标的采样权重；
- 防止灾难性遗忘的通用数据或正则。

### 5. 统一的验收原则

所有项目都应与 SFT-only、旧版本和规则基线比较，并使用冻结测试集、人工盲评和安全集。总胜率不能掩盖关键分桶退化：客服回答变流畅但事实性下降，Agent 成功率上升但越权增加，或者文档抽取平均分提升但证件号幻觉变多，都不能上线。

资金划转、风控放行和账户冻结始终由确定性权限、规则与人工审批控制，DPO 优化的是模型行为，不替代支付核心状态机。
