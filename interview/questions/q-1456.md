# 3.为什么现在大模型能处理动辄一两万 token 的长序列信息？

## 答案

大模型能处理一两万甚至更长 token，不是单一技术突破，而是位置编码、长上下文训练、注意力算子、KV Cache、并行和数据共同改进的结果。

### 1. Transformer 本身可建模任意位置

Self-Attention 允许每个 token 与其他 token 建立依赖：

```text
Attention(Q,K,V) =
softmax(QK^T / sqrt(d)) V
```

但模型需要位置编码区分顺序，并且标准 attention 计算/显存随长度近似 `O(n^2)`。

### 2. 位置编码扩展

现代模型常使用 RoPE。扩展上下文的方法包括：

- 在更长序列上继续训练；
- Position Interpolation；
- NTK-aware scaling；
- YaRN 等 RoPE scaling；
- 调整频率和训练长度分布；
- ALiBi 等替代位置方案。

只修改配置中的 max length 不代表模型真正具备长上下文能力，必须训练和评测。

### 3. 高效 Attention Kernel

FlashAttention 通过分块计算、减少 HBM 读写，显著降低 attention 中间显存并提高速度。它仍是精确 attention，通常没有改变理论 `O(n^2)` 计算复杂度，但让长序列更可运行。

### 4. 稀疏和混合注意力

- sliding-window attention；
- local + global attention；
- block sparse attention；
- chunk/recurrent memory；
- linear attention；
- state-space/hybrid models。

这些方法减少所有 token 两两交互，但可能牺牲跨块信息，需要专门训练。

### 5. KV Cache 和架构

推理时缓存历史 K/V，避免每生成一个 token 都重算整个前缀。GQA/MQA 减少 KV head 数，KV quantization、PagedAttention 和 prefix cache 提高并发与内存利用率。

KV Cache 只优化自回归重复计算，不能消除长 Prefill 的成本。

### 6. 分布式与工程

- sequence/context parallel；
- tensor parallel；
- ring attention 类通信；
- 长度分桶；
- sequence packing；
- gradient checkpointing；
- 分块 prefill；
- 限制并发最大 token 预算。

### 7. 长上下文训练数据

模型要见过：

- 长文档；
- 多文档问答；
- 中间位置关键信息；
- 长程引用；
- 代码仓库；
- 多轮对话；
- 合成 needle 和真实任务。

否则会出现“支持 32K，但只会利用开头和结尾”的 lost-in-the-middle。

### 8. 上下文窗口不等于记忆和准确率

窗口更长仍可能：

- 检索不到关键内容；
- 注意力稀释；
- 成本和延迟上升；
- 被无关/恶意内容干扰；
- 多轮状态冲突。

生产系统仍需要 RAG、摘要、外部状态和证据选择。

### 9. 面试总结

长上下文来自位置外推与训练、FlashAttention、稀疏/混合注意力、GQA/KV 优化和分布式工程。标称长度只是容量上限，真正要看长文检索、位置鲁棒性、延迟和成本。
