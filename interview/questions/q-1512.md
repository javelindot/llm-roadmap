# KV Cache的进化之路，详细讲

## 答案

KV Cache 的演进主线是：避免重复计算、减少每 token 占用、解决动态内存碎片、复用公共前缀，再扩展到量化、淘汰、分层和跨节点缓存。

### 1. 从无 Cache 到标准 KV Cache

自回归生成第 `t` 个 token 时，历史 token 的 Key/Value 不会改变。若每步重算全部历史，会重复执行投影和 Attention 前序计算。标准 KV Cache 保存各层历史 K/V，新一步只计算当前 token 的 Q/K/V，再让新 Q 查询历史 K/V。

单请求缓存大小近似为：

```text
2 * layers * sequence_length * num_kv_heads * head_dim * bytes
```

其中 2 表示 K 和 V。它显著减少重复计算，但显存随层数、序列长度和并发线性增长。

### 2. 从 MHA 到 MQA/GQA/MLA

- **MHA**：每个 query head 都有独立 K/V，缓存最大；
- **MQA**：所有 query head 共享一组 K/V，缓存最小，但可能损失容量；
- **GQA**：多组 query head 共享一组 K/V，在质量与缓存间折中；
- **MLA**：把 K/V 压缩到低维 latent 表示，进一步降低缓存和带宽，但依赖专门架构与 kernel。

这是从模型架构层减少 `num_kv_heads` 或缓存表示维度。

### 3. PagedAttention

早期服务为每个序列预留连续显存，输出长度未知会造成内部/外部碎片。PagedAttention 把 KV Cache 切成固定大小逻辑块，通过 block table 映射到非连续物理块，类似虚拟内存。

收益包括：

- 按需分配，减少预留浪费；
- 支持 continuous batching；
- 序列结束后快速回收；
- beam/parallel sampling 可通过 copy-on-write 共享前缀块；
- 更容易做抢占和交换。

### 4. 前缀复用

Automatic Prefix Caching、radix tree/trie cache 可让相同 system prompt、模板或文档前缀复用已计算 KV。收益取决于前缀长度和重复率；它降低重复 Prefill，不会让后续 Decode 本身变快。

缓存键必须包含模型、Adapter、token 序列和影响结果的配置，敏感多租户场景还要防止跨租户侧信道与数据泄漏。

### 5. 量化、淘汰与窗口化

- FP8/INT8 KV Cache：降低容量和带宽，需验证长上下文质量；
- sliding-window/local attention：只保留最近窗口；
- attention sink、heavy-hitter 等策略：保留重要 token、淘汰低价值缓存；
- KV offload：把冷块放到 CPU、SSD 或远端存储；
- 分层缓存：GPU HBM、主存、NVMe、远端 KV 池分级管理。

压缩和淘汰会改变模型可见历史，必须按任务评测，而非只测 perplexity。

### 6. 分布式与 PD 分离

在 Prefill/Decode 分离架构中，Prefill 节点生成 KV 后传给 Decode 节点。Mooncake 等 KV-centric 架构进一步把 KV Cache 视为可调度资源。瓶颈从“是否能算”转变为网络带宽、缓存位置、路由、生命周期和故障恢复。

### 7. 未来方向

- 更强的架构级压缩，如 GQA/MLA 和混合状态模型；
- KV-aware 调度与跨请求语义/前缀复用；
- 训练时让模型适应量化或受限缓存；
- 跨节点、跨层级缓存解耦；
- 将长上下文从“全部保留”转向检索、记忆和缓存联合管理。

### 8. 跨境支付场景

政策问答可复用固定 system prompt 和公共合规文档前缀，但商户交易上下文必须按租户隔离。支付状态应实时查权威系统，不能因为 KV Cache 命中而复用旧状态。

### 9. 面试总结

KV Cache 从计算优化演化为完整的内存与分布式系统问题：标准缓存消除重算，MQA/GQA/MLA 降低大小，PagedAttention 管理碎片，Prefix Cache 复用前缀，量化/淘汰/分层缓存继续扩展容量。
