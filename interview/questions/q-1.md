# SKILL 是如何工作的？

## 答案

**一句话**：SKILL 是 Claude Code 的"可发现的能力包"，本质是一组 Markdown + 脚本 + 资源的文件夹，通过**渐进式加载**（progressive disclosure）按需注入到模型上下文，从而把"领域知识 / 工作流程 / 工具用法"沉淀成可复用的 Agent 能力。

### 1. 是什么（What）

- 一个 Skill = 一个文件夹，里面有：
  - `SKILL.md`：核心文件，含 frontmatter（name、description、allowed-tools 等）+ 正文指令。
  - 可选的脚本（Python / Bash）、模板、参考文档、子目录。
- 来源有三类：**built-in**（官方）、**plugin**（插件市场）、**project/user**（`.claude/skills/` 或 `~/.claude/skills/`）。

### 2. 怎么被发现（Discover）

- 启动时，Claude Code 扫描各 skill 目录，只把每个 SKILL.md 的 **frontmatter（name + description）** 注入系统提示，**正文不加载**。
- 模型看到的只是一份"能力清单"——这就是渐进式加载第一层，目的是**省 token**。

### 3. 怎么被触发（Trigger）

- 用户显式：输入 `/skill-name` 或在消息里点名。
- 模型隐式：根据 description 判断当前任务匹配某个 skill，主动调用 `Skill` 工具。
- 触发后，Claude Code 才把该 SKILL.md 的正文（以及它显式引用的子文件）读入上下文。

### 4. 怎么执行（Execute）

- SKILL.md 正文是给模型看的"操作手册"，告诉它：什么时候用、用什么工具、按什么步骤、注意什么坑。
- 模型按手册一步步调用工具（Bash、Read、Edit、WebFetch……）；脚本类资源由模型用 Bash 跑，不是"自动执行"。
- `allowed-tools` 可限制 skill 内能用的工具，做最小权限。

### 5. 关键设计点

- **渐进式披露**：清单 → 正文 → 子文件，三层按需加载，避免上下文爆炸。
- **可组合**：一个 skill 可在正文里 `@reference` 其他文件，模型按需打开。
- **可移植**：纯文件，纳入 git 即可团队共享；打包成 plugin 可跨项目分发。
- **声明式 > 代码**：skill 描述"做什么/怎么做"，真正动手的是模型 + 工具，不需要自己写调度器。

### 6. 和相关概念的区别


| 概念            | 本质             | 谁执行                |
| ------------- | -------------- | ------------------ |
| Slash Command | 一段 prompt 模板   | 模型按 prompt 走       |
| Subagent      | 独立上下文的子任务      | 新开一个 Claude 实例     |
| **Skill**     | **可发现的指令+资源包** | **当前 Claude，按需加载** |
| MCP Server    | 外部工具协议         | 外部进程提供工具           |


### 7. 典型用法

- 沉淀团队 SOP：发版流程、Code Review 规范、事故排查手册。
- 封装复杂工具链：Terraform 部署、数据库迁移、性能压测。
- 领域知识库：把 RAG / 微调 / 评测的最佳实践写成 skill，团队共享。

**一句话总结**：Skill = "Markdown 写的 Agent 微程序"，靠 frontmatter 被发现、被模型按需加载、由模型驱动工具执行，是 Claude Code 把"人类经验"工程化为"可复用 Agent 能力"的最小单元。

## 结合项目谈（口语版 · 面试现场约 40 秒）

> 我们公司是做跨境支付的嘛，业务里其实有挺多那种"流程一大堆、规则还老变"的活儿。你比如说反洗钱里的可疑交易要排查吧、新商户进来要做 KYB 审核、还有跨币种对账对不平这种，以前全靠运营写在飞书文档里，新人来了得师傅带，挺依赖人的。
>
> 后来我们想接 AI Agent 进去做这些事，一上来就踩坑——你要是把规则一股脑全塞到系统提示里，token 直接就爆了；而且业务那边规则一改，prompt 整个得重写，谁都不敢动。
>
> Skill 这个东西对我们来说就挺对路的。说白了它就是一个文件夹，里面写清楚：这是干啥用的、什么时候触发、查哪几个字段、调哪个接口、什么时候得转人工。Agent 平时就看一眼描述，真命中了才把正文加载进来。
>
> 我之前落地过一个叫"可疑交易初筛"的 Skill，工单一进来 Agent 自己判断要不要用，token 算下来省了得有七成。更爽的是运营后面想加新规则，加个 md 文件提个 PR 就上线了，主 prompt 根本不用动。
>
> 现在我在推清算对账、商户准入这些场景也都 Skill 化，慢慢把那种"只有老员工知道"的流程沉到代码里去。

**🪝 自然带出去的钩子**（任选一句收尾，引面试官追问）：

- "对了，我们刚开始还踩过一个坑——几个 Skill 描述写得太像，模型老误触发，后来怎么治的可以再聊。"
- "Skill 和 Subagent 在我们风控链路里其实是搭着用的，一个管流程、一个管上下文隔离，这块选型我们也纠结过一阵。"
- "权限这块我们也专门做过设计，不然 Agent 万一乱调高危接口就麻烦了。"
