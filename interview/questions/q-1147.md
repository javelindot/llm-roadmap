# RAG 技术的核心缺陷及优化方案

## 答案

RAG 的核心缺陷是最终效果受知识源、解析、分块、检索、上下文和生成整条链路共同约束。它降低幻觉风险，但不能保证事实正确。

### 1. 检索依赖和召回遗漏

正确证据没有进入候选，生成模型无法可靠补救。原因包括 query 与文档表达不一致、长尾实体、跨语言、错误分块和 embedding 领域不匹配。

优化：

- Query 改写、术语归一化和多意图拆分；
- BM25 与 dense hybrid retrieval；
- metadata/权限过滤；
- 领域 hard negative 微调；
- 多查询或 HyDE 按需启用；
- 用 Recall@K 和分桶评测定位。

### 2. 上下文噪声和冲突

扩大 Top-K 会召回重复、过期或不同国家规则，模型可能混合矛盾证据。

优化：

- 去重、版本和生效时间治理；
- cross-encoder rerank；
- Context Precision 和证据覆盖优化；
- 冲突检测与权威来源排序；
- 动态 Top-K 和 token budget；
- 证据不足时追问或拒答。

### 3. 文档切片破坏完整语义

固定长度分块会切断表头、限制条件和例外条款。

优化：

- 文档结构与业务语义分块；
- parent-child chunk；
- 表格、流程图和图片专门解析；
- 为 chunk 添加章节上下文；
- 保留页码、来源和原文定位。

### 4. 静态知识无法表达实时状态

订单、余额、汇率和通道健康不断变化，向量库只能反映索引时状态。

优化：

- 将实时事实路由到业务工具或 SQL；
- RAG 负责解释规则，工具负责当前状态；
- 建立事实优先级和冲突处理；
- 文档使用增量索引、删除和蓝绿版本切换。

### 5. 多跳关系和全局问题能力弱

独立 chunk 难以表达商户、账户、交易、收款人和规则之间的关系，也难回答整个语料的全局主题。

优化：

- 子问题分解和多步检索；
- GraphRAG/local-global search；
- 结构化图或 SQL 查询；
- 图结果与原文证据混合；
- 只在关系型问题上启用，简单 FAQ 保持普通 RAG。

### 6. 生成模型仍可能幻觉

模型可能忽略证据、错误引用、补全金额和时效，或把相似规则混在一起。

优化：

- 证据与指令分区；
- 强制引用和 claim-evidence 对齐；
- 金额、币种、日期和状态程序校验；
- 无答案训练与拒答；
- 高风险输出模板化并人工审批；
- 使用 Self-RAG/CRAG 类检索评价和纠正思想，但验证模块本身也必须评测。

### 7. 权限和提示注入风险

RAG 可能把越权文档或恶意指令带进上下文。

优化：

- 检索前执行租户、商户和角色 ACL；
- 文档与工具返回标记为不可信数据；
- 外部内容不能修改系统指令和工具权限；
- PII 脱敏、审计和数据生命周期治理；
- 缓存键包含权限和知识版本。

### 8. 延迟和成本

多路召回、rerank、长上下文和强模型会增加 P95 和成本。

优化：

- 独立检索并行；
- 分级路由，简单任务走轻链路；
- embedding、rerank 和 LLM 批处理；
- 版本化缓存和 Prompt Cache；
- 控制候选、上下文和工具预算；
- 超时、熔断与降级。

### 9. 评测和反馈不充分

只看最终答案总分无法定位故障，也容易对固定测试集过拟合。

优化：

- 分别评查询、召回、重排、生成和工具；
- 使用自然流量集、长尾集、安全集和事故回放集；
- 分国家、币种、语言、产品和风险等级；
- LLM judge 用人工黄金集校准；
- 影子流量、线上 A/B、成熟标签和 bad case 回流；
- 所有结果关联组件与知识版本。

### 10. 技术选型原则

Self-RAG、CRAG、GraphRAG、长上下文和 Agentic RAG 都是在解决特定缺陷：

- Self-RAG：学习何时检索并对结果反思；
- CRAG：评价检索质量并触发纠正；
- GraphRAG：增强关系和全局检索；
- 长上下文：减少切片，但增加成本和注意力风险；
- Agentic RAG：动态规划检索，但增加延迟和控制复杂度。

生产中应根据 bad case 选择最小有效方案，并通过消融证明收益。跨境支付的底线是：RAG 可以辅助解释和调查，不能替代权限、状态机、账务事实、风控专模和人工审批。
