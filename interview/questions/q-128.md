# 什么是RAG Fusion？工作流程？

## 答案

**一句话**：RAG Fusion 是先把用户问题生成多个改写查询，多路检索后再融合排序，提升召回覆盖率。

### 工作流程
1. 用户提出原始问题。
2. LLM 生成多个语义等价或不同角度的查询。
3. 每个查询分别去知识库检索。
4. 合并多个检索结果。
5. 用 RRF 或 reranker 重新排序。
6. 选出 top-k 内容给 LLM 生成答案。

价值：
- 覆盖用户表达不清的问题。
- 提高多意图问题的召回率。
- 减少单次 query 没命中的风险。

缺点：检索成本更高，改写质量差会引入噪声。

### 1. RAG Fusion 解决什么问题

用户的一个问题可能有多种表达方式，也可能包含多个子意图。单 query 检索如果没命中关键表达，结果就会偏。RAG Fusion 用多个 query 从不同角度召回，再融合结果，提高覆盖率。

### 2. 关键点

- query 改写要围绕原问题，不能发散太远。
- 多路召回要去重。
- 融合时常用 RRF。
- 最好配合元数据过滤，避免召回噪声。
- 最后仍需要 rerank。

### 3. 面试总结

RAG Fusion 的价值是提升召回覆盖，代价是成本和噪声上升。适合复杂、多意图、表达不清的问题。

### 4. 结合跨境支付项目怎么说（面试口径）

> 比如运营问"巴西这笔收款为什么没到"，这个问题可能涉及巴西本地支付、通道回执、清算周期、风控拦截。RAG Fusion 可以把它改写成几个查询："巴西收款清算周期""巴西本地支付失败原因""PIX 入账延迟规则""收款风控拦截处理"。
>
> 多路召回后融合，答案更完整。但我会加业务过滤，比如国家必须是巴西，币种必须是 BRL，通道必须匹配当前订单，否则多 query 很容易把其他国家或其他产品的规则召回来。
>
> 所以 RAG Fusion 在支付场景里有用，但一定要配合元数据和 rerank 控制噪声。

**钩子**：RAG Fusion 适合多意图问题，但要配合元数据过滤控噪声。
