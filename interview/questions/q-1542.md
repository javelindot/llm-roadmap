# 如何为特定任务构建高质量上下文（动态打包、向量索引、信息检索）？

## 答案

高质量上下文不是把检索 Top-K 全部拼接，而是在 token、时延和权限预算下，选择最能支持当前任务的指令、事实、历史和工具结果。

### 1. 先定义任务所需信息

把上下文分为：

- 不可丢的 system policy 和安全约束；
- 当前问题、用户/租户权限和结构化实体；
- 权威工具结果；
- 检索证据；
- 必要会话历史和任务状态；
- 输出 schema 与少量示例。

先判断任务是精确查找、语义问答、多跳推理、总结还是执行，不同任务使用不同检索器和预算。

### 2. 检索

- BM25 处理错误码、名称、数字和精确短语；
- Dense retrieval 处理语义改写和跨语言；
- SQL/图查询处理结构化事实和关系；
- metadata 过滤权限、国家、时间和版本；
- multi-query/query rewrite 只对复杂问题启用；
- parent-child retrieval 保留章节语境。

多路结果可用 RRF 或学习融合，再用 Cross-Encoder rerank。

### 3. 动态打包

给每个候选估计：

```text
utility =
relevance
+ authority
+ freshness
+ coverage
- redundancy
- conflict risk
```

在 token budget 下按 utility/token 选择，但对必须证据、反例和不同子问题保留配额。实际打包步骤：

1. 去重和近重复合并；
2. 检测版本冲突，优先权威且当前有效来源；
3. 按子问题保证证据覆盖；
4. MMR 等方法增加多样性；
5. 超长 chunk 做 query-aware 抽取；
6. 保留标题、日期、页码和来源；
7. 超预算时先删低价值历史，不删安全指令。

### 4. 上下文结构

```text
[System policy]
[User and permission context]
[Current task]
[Authoritative tool results]
[Evidence 1..N with source/version]
[Output schema]
```

把外部文档标记为不可信数据，防止文档中的提示注入覆盖系统指令。工具事实与知识文档分区，冲突时由确定性优先级处理。

### 5. 效率优化

- 简单任务使用小 Top-K，复杂任务动态增加；
- 并行混合召回；
- 缓存权限安全的 query embedding、rerank 和公共前缀；
- 使用层级摘要保存长任务状态；
- 根据 lost-in-the-middle 将关键证据放在显著位置；
- 记录每段上下文是否被引用，用于后续裁剪；
- 建立 token、P95 和单次成功成本预算。

### 6. 评测

- Recall@K、NDCG 和多跳覆盖；
- Context Precision/Recall；
- 引用支持率、faithfulness 和无答案识别；
- 上下文 token、P95 和成本；
- 权限泄漏、过期信息和提示注入；
- 移除某类上下文后的消融。

### 7. 跨境支付场景

回答付款失败问题时，当前订单和通道回执来自授权工具，国家/币种规则来自当前有效文档，历史对话只保留必要实体。旧订单状态和其他商户文档即使语义相关也不能打包。

### 8. 面试总结

高质量上下文构建是一个受约束的信息选择问题：任务路由、多路检索、权限与版本过滤、rerank、去重、覆盖优化和 token budget 缺一不可。
