# 什么是Scaling Laws?它揭示了模型性能、计算量和数据量之间的什么关系?这对LLM的研发有什么指导意义?

## 答案

Scaling Laws 指模型损失或能力随参数量、数据量和计算量扩大呈现可预测的幂律趋势。它不是“模型越大一定越好”的口号，而是用于预测收益、分配训练预算和判断瓶颈。

### 1. 基本形式

经验上测试损失可近似写为：

```text
L(N, D)
≈ L_infinite
+ A / N^alpha
+ B / D^beta
```

- `N`：非 Embedding 参数量或模型规模；
- `D`：训练 token 数；
- `L_infinite`：数据分布下不可约损失；
- `alpha、beta`：通过小规模实验拟合。

固定其他因素时，扩大模型或数据带来的损失下降通常是平滑幂律，边际收益逐渐减小。

### 2. 计算量关系

Dense Transformer 训练 FLOPs 常粗略估为：

```text
C ≈ 6 * N * D
```

常数会随架构、稀疏性、序列和实现变化。给定计算预算 `C`，模型太大但数据不足会欠训练；模型太小但重复喂过多数据又会浪费模型容量。

Kaplan 等早期工作更偏向在固定计算下扩大模型。Chinchilla 重新研究 compute-optimal 分配，表明参数量与训练 token 应更均衡增长；其特定实验中的约 20 token/parameter 是经验结果，不是所有模型、数据和训练目标的永恒常数。

### 3. 对研发的指导

1. 用小模型、小数据实验拟合曲线，预测大训练收益；
2. 在参数、token 和 FLOPs 间选择 compute-optimal 配置；
3. 判断当前是模型受限、数据受限还是计算受限；
4. 规划数据采集、训练时间和硬件；
5. 在训练前做 stop/go 决策，避免盲目扩大规模；
6. 比较架构或数据方案时，尽量在等计算预算下比较。

### 4. 现代扩展

- 数据质量、去重和课程学习会改变曲线；
- MoE 要区分总参数与每 token 激活参数；
- 数据受限时需要研究重复训练和合成数据退化；
- 长上下文、多模态和领域数据有各自 scaling 行为；
- test-time scaling 研究推理 token、采样数、搜索和 verifier 如何换取能力；
- benchmark 饱和后，loss 下降不一定等比例转化为业务收益。

因此 Scaling Laws 是局部经验模型，需要在目标数据、架构和训练范围内验证，不能无条件外推几个数量级。

### 5. 跨境支付场景

垂域模型的数据量有限，盲目扩大参数常不如提高支付术语、多语言、长尾国家和高质量流程数据。可通过小规模 scaling 实验判断继续训练、扩模型、做 RAG，还是使用更强基座更划算。

### 6. 面试总结

Scaling Laws 揭示损失随参数、数据和计算按幂律平滑改善，并指导 compute-optimal 预算分配。核心不是追求最大模型，而是在给定资源下找到模型规模、数据 token 和质量的最佳组合。
