# 项目中的指令数据是你自己构造的吗？是否自己尝试过对指令数据做修改？

## 答案

指令数据不是全部逐条手写，也不是直接把历史聊天原样送去训练。自己负责的是数据标准、构造流水线、难例补充、质量校验和版本迭代；业务专家负责确认支付口径、合规边界和答案是否可执行。

### 1. 数据来源

- 脱敏后的历史客服问答和工单；
- FAQ、SOP、错误码和产品文档；
- 工具调用日志中的成功轨迹；
- 人工整理的高风险拒答、澄清和转人工案例；
- 基于规则或强模型生成的合成样本；
- 线上 bad case 经人工修订后的样本。

实时余额、订单状态、汇率、通道可用性不能做成固定事实答案。这类数据只用于训练“应该调用什么工具、参数如何填写、如何解释返回结果”。

### 2. 指令样本结构

一条工具型 SFT 数据可包含：

```json
{
  "messages": [
    {"role": "system", "content": "权限、工具和安全规则"},
    {"role": "user", "content": "商户问题"},
    {"role": "assistant", "tool_call": "query_payment_order"},
    {"role": "tool", "content": "结构化脱敏结果"},
    {"role": "assistant", "content": "基于结果的解释"}
  ],
  "metadata": {
    "scenario": "payout_failed",
    "language": "zh-CN",
    "risk_level": "medium",
    "source_version": "..."
  }
}
```

metadata 不一定进入模型输入，但用于采样、分桶评测和追溯。

### 3. 做过的修改方向

**统一输入输出**

- 统一角色、Prompt 模板和工具 schema；
- 规范金额、币种、日期和状态枚举；
- 去掉客服个人口头禅和无效上下文；
- 输出拆成结论、依据、下一步和引用。

**增加任务覆盖**

- 正常路径、边界样本和工具异常；
- 信息不足时先追问；
- 权限不足和跨商户查询拒绝；
- 风控原因只解释允许的大类；
- 中英文和支付术语混合；
- 多轮改口、订单号错误和时间歧义。

**控制数据分布**

- 避免高频 FAQ 淹没少数高风险场景；
- 对长度、语言、国家、产品和风险分层采样；
- 降低近重复模板比例；
- 保留一部分无工具、单工具和多工具任务；
- 区分 SFT、验证和测试来源。

### 4. 合成数据不能直接使用

生成流程应是：

```text
业务 schema 和规则
-> 生成候选
-> 规则校验
-> 去重与泄漏检查
-> 强模型或人工评审
-> 小规模训练
-> 独立评测
-> 通过后扩量
```

自动检查包括 JSON schema、工具是否存在、参数类型、金额币种一致性、引用来源、敏感字段、答案与工具结果是否矛盾。

### 5. 防止数据泄漏和污染

- PII、账户号、证件号 tokenization 或脱敏；
- 训练/测试按用户、模板、文档和时间切分；
- MinHash/embedding 去重；
- 不把评测问题改写后混入训练；
- 记录 source_id、构造方法、审核人和版本；
- 删除过期政策，避免模型学习旧规则；
- 检查合成模型是否把测试答案带回训练集。

### 6. 如何判断修改有效

每个数据版本只改变少数因素，比较：

- 任务完成率；
- 工具选择与参数准确率；
- JSON/schema 通过率；
- 事实一致性和引用；
- 澄清、拒答、转人工；
- 分国家、语言、产品和风险指标；
- 通用能力与过拟合回退。

数据修改的价值不是“条数更多”，而是错误类型减少、覆盖更完整、标签一致且来源可追溯。
