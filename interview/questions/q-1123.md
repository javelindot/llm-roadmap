# 检索策略是什么？

## 答案

检索策略不是固定的“向量 Top-K”，而是根据问题类型、数据源、权限、时效和风险动态选择检索路径。

> 查询理解 → 数据源路由 → 前置过滤 → 多路召回 → 融合重排 → 证据覆盖 → 充分性判断。

### 1. 先区分数据源

- 产品规则、KYC/KYB、错误码和 SOP：知识库检索；
- 订单、余额、汇率报价和通道状态：实时工具；
- 商户、账户、交易和收款人关系：图或结构化查询；
- 最新监管公开信息：受控外部搜索；
- 无权限或高风险问题：拒答或转人工。

一个问题可以同时走多条路径，但事实来源的优先级必须明确。

### 2. 查询理解和路由

抽取：

> `intent、country、currency、product、channel、error_code、time_range、merchant_id`

多轮问题先做指代消解，多意图问题拆成子任务。路由采用规则、小分类器和 LLM 分级组合；低置信度允许多路候选或先追问。

例如“巴西收款没到账”既需要查 BRL/PIX 相关知识，也可能需要订单和通道回执。缺少订单号时先确认用户是在问规则还是具体交易。

### 3. 前置过滤

检索前使用：

- 租户、商户、角色和数据权限；
- 国家、币种、产品和通道；
- 文档状态、生效时间和版本；
- 语言和文档类型；
- 时间范围与敏感等级。

过滤尽量在索引层完成，避免先召回越权数据再在 Prompt 中删除。

### 4. 多路召回

**BM25**

处理错误码、币种、国家码、产品名和法规编号。

**Dense Retrieval**

处理口语、同义表达、跨语言和语义相似。

**FAQ/案例**

处理高频标准问题，案例需要脱敏、去重和标明适用范围。

**Graph/SQL**

处理多跳关系和结构化条件，不使用向量相似替代精确账务状态。

**Multi-Query/HyDE**

只对复杂、短且表达鸿沟明显的问题启用；精确查询保持原词。

### 5. 融合和重排

BM25 与向量结果使用 RRF 或学习型融合器合并。保留来源、通道排名、分数和命中字段。

候选经过：

- 去重和相邻 chunk 合并；
- cross-encoder rerank；
- 来源权威性、新鲜度和范围匹配；
- 否定词、版本和冲突检查；
- 子问题证据覆盖。

Top-K 动态决定。简单 FAQ 只需少量证据，多条件规则或多跳问题需要覆盖多个证据单元，但不应把所有候选直接放入上下文。

### 6. 充分性和降级

检索后判断：

- 必要事实和限制条件是否齐全；
- 是否存在版本冲突；
- 实时状态与文档规则是否一致；
- 是否需要补充订单、商户或国家信息；
- 是否含不可披露内容。

证据不足时追问、只返回已确认部分或转人工。向量服务失败可退回 BM25 和 FAQ；reranker 超时使用融合排名；工具失败不能用文档推测实时状态。

### 7. 缓存和时效

可缓存 query embedding、公共文档召回和高频 FAQ，但缓存键必须包含租户、权限、国家、语言和知识版本。汇率、余额、通道状态和个性化交易结果不做长期共享缓存。

### 8. 评测

- 各路 Recall@K、MRR、独占贡献和重合率；
- NDCG、Context Precision/Recall；
- 权限违规和过期文档命中率；
- 多语言、长尾和无答案分桶；
- 端到端正确率、引用和任务成功率；
- P95、成本与降级率。

检索策略的目标是在满足权限和时效的前提下，用最小的延迟与上下文成本覆盖回答所需证据。
