# 基于llm的智能队友中，asr-&gt;意图识别-&gt;对话生成-&gt;tts，里面需要用的模型规模以及优化方法

## 答案

智能队友的关键指标是低延迟、可打断、自然对话和游戏状态一致性。模型规模不能统一给一个数字，应按 ASR、意图、对话和 TTS 的任务复杂度、端云位置和延迟预算分别设计。

### 1. 整体链路

```text
VAD/降噪
-> Streaming ASR
-> 意图/槽位/游戏状态
-> Dialogue Policy / LLM
-> 安全与动作校验
-> Streaming TTS
```

游戏事件、队友状态和技能冷却应通过结构化接口提供，不能只从语音猜。

### 2. ASR

模型选择：

- 端侧或实时场景：几十 M 到数百 M 的流式模型；
- 云端高精度：数百 M 到 1B+；
- 专用词表处理角色名、地图、技能和俚语。

优化：

- VAD；
- streaming chunk；
- hotword bias；
- endpointing；
- INT8/FP16；
- distillation；
- 噪声和多人说话增强；
- partial result 提前送入下游。

指标：WER、首字延迟、末字延迟、实时率和噪声切片。

### 3. 意图识别

高频意图可用几十 M 到数百 M 的 Encoder，甚至规则+小模型：

```text
进攻
撤退
请求支援
报点
闲聊
系统控制
```

同时抽取位置、目标、时间和优先级。小模型延迟低、可解释，没必要所有意图都调用 LLM。

### 4. 对话生成

- 端侧：约 1B 到 4B 量化模型，适合短响应和隐私；
- 云端实时：约 3B 到 8B 可获得较好性价比；
- 复杂策略或开放闲聊：更大模型按需路由。

不要只看参数量，还要看 active parameters、context、量化、硬件和 tokens/s。

优化：

- 短 prompt 和结构化游戏状态；
- KV/prefix cache；
- speculative decoding；
- continuous batching；
- INT4/INT8；
- 蒸馏；
- 限制输出长度；
- 小模型处理意图，大模型处理复杂对话；
- 将动作决策与自然语言生成分离。

### 5. TTS

使用流式声学/生成模型，目标是尽早产出首段音频：

- 文本分句增量合成；
- speaker/style embedding 缓存；
- 声码器量化和蒸馏；
- 常用短句缓存；
- 支持被用户打断；
- 控制情绪但避免影响可懂度。

指标：首包延迟、实时率、MOS、音色一致性和稳定性。

### 6. 端到端延迟

可以并行和流水化：

```text
ASR partial
-> 提前识别意图
-> LLM 提前规划
-> 首句生成后立即 TTS
```

不要等完整文本全部生成后才开始语音合成。还需要 barge-in、取消旧请求和过期状态检查。

### 7. 安全与业务

- 动作白名单；
- 敏感内容过滤；
- 游戏状态校验；
- 防提示注入；
- 儿童和语音隐私；
- 记录模型版本和执行动作。

如果迁移到跨境支付语音客服，账户和订单信息要先鉴权，资金动作仍需二次确认和服务端审批。

### 8. 面试总结

推荐架构是小型流式 ASR + 小型意图模型 + 按难度路由的 LLM + 流式 TTS。优化重点是流水并行、量化蒸馏、缓存、短输出和可打断，而不是四个环节都堆最大模型。
