# 介绍一下双塔模型，结构、样本

## 答案

双塔模型把匹配双方独立编码到同一个向量空间，典型结构是 User/Query Tower 和 Item/Document Tower：

```text
用户特征、行为序列、场景特征 → User Tower → user embedding
物品属性、内容、统计特征     → Item Tower → item embedding
                                      ↓
                       dot product / cosine similarity
```

它主要用于大规模召回或轻量预排序。Item Tower 的向量可以离线计算并写入 ANN 索引，在线只计算用户向量，就能从百万级候选中快速检索 Top-K。

### 1. 模型结构

User Tower 常用的输入包括：

- 用户 ID、国家、语言、设备和渠道等离散特征；
- 最近点击、搜索、收藏、购买等行为序列；
- 当前时间、入口、query 和 session 上下文；
- 长期兴趣与短期兴趣的聚合表示。

Item Tower 常用的输入包括：

- item ID、类目、品牌、价格带等结构化属性；
- 标题、描述、图片等内容 embedding；
- 新鲜度、质量和历史统计特征；
- 可离线计算且不会因当前用户变化的特征。

两侧分别经过 Embedding、MLP 或序列编码器，投影到相同维度。若线上用 cosine similarity，训练和建索引时应统一做 L2 normalization；若用 dot product，则要防止向量模长无约束增长。

双塔的关键限制是双方在最终打分前没有细粒度交互，表达能力弱于 Cross-Encoder。生产中通常让双塔负责高效召回，再由能够使用交叉特征的粗排和精排模型处理复杂关系。SparCode、IntTower 等工作尝试在保留检索效率的同时增强交互，但是否替换经典双塔仍要看收益、延迟和索引成本。

### 2. 正样本构造

正样本不能把所有行为等价处理。通常根据业务漏斗设置行为权重：

> 曝光 < 点击 < 收藏/加购 < 下单 < 支付成功或有效消费。

可使用 `(user_context, positive_item)` 作为训练对，并注意：

- 用户特征必须截断到行为发生时，不能使用未来信息；
- 同一用户的连续重复行为要去重或降权，避免少数重度用户主导训练；
- 标签成熟窗口要覆盖退款、取消等延迟反馈；
- 训练、验证、测试按时间切分，并隔离同一 session 或订单；
- 多目标场景可分别预测点击、使用或支付成功，再定义召回目标。

### 3. 负样本决定模型上限

未发生行为不等于用户不喜欢，因此负样本需要分层：

- **随机负样本**：覆盖全局 item，训练稳定但通常过于简单；
- **In-batch negatives**：同一 batch 中其他样本的正 item 作为负例，计算高效；
- **曝光未点击样本**：比随机负例更接近线上分布，但存在位置偏差；
- **困难负样本**：从 BM25、旧模型或 ANN Top-K 中选择“很相似但未被选择”的 item；
- **同类困难负样本**：同类目、同国家、同价格带或同产品类型，迫使模型学习细粒度区别。

要排除 false negative。例如用户没有点击某个 item，可能只是没有看到；同一个问题也可能有多个相关文档。热门 item 在 In-batch 中出现频率更高，还需通过采样概率校正、logQ correction 或损失权重减少流行度偏差。

常用训练目标包括 sampled softmax、InfoNCE 和 pairwise ranking loss。以 batch 内对比学习为例：

```text
L = -log exp(sim(u, i+) / τ)
         / Σj exp(sim(u, ij) / τ)
```

其中 `τ` 是温度系数。Batch 越大，负样本通常越丰富，但也会增加 false negative 和分布偏差，需要结合业务数据调节。

### 4. 在线服务

线上链路通常是：

1. 离线或准实时计算 item embedding；
2. 构建带版本号的 ANN 索引；
3. 在线读取用户与 session 特征并计算 user embedding；
4. ANN 检索 Top-K；
5. 执行权限、库存、地域和状态等硬过滤；
6. 将候选交给粗排、精排和重排。

模型版本、特征版本和索引版本必须一致。新 item 可先使用内容特征生成向量解决冷启动，待积累行为后再融合协同信号。

### 5. 跨境支付中的应用边界

双塔可以用于向商户召回适用的收款、付款、换汇、卡或收单产品：

- Merchant Tower 编码注册地、行业、交易国家、币种、金额区间和历史产品使用；
- Product Tower 编码产品覆盖国家、币种、成本、时效、接入条件和能力标签；
- ANN 召回候选产品后，再由确定性服务校验 KYB、牌照、额度、风险等级和实时可用性。

也可以为通道路由生成候选，但双塔分数只能表达匹配程度，不能绕过合规、资金安全和通道硬约束。跨商户数据必须满足授权、租户隔离和最小化原则，不能直接用 A 商户的用户级交易行为为 B 商户建模。

### 6. 评测指标

- 召回效果：Recall@K、HitRate@K、MRR、NDCG；
- 业务覆盖：item 覆盖率、长尾覆盖、新 item 覆盖；
- 分桶效果：新老用户、国家、语言、产品和活跃度；
- 系统指标：ANN P95/P99、QPS、索引大小和更新时延；
- 下游效果：候选进入精排后的转化、支付成功、净收益和风险指标。

离线 Recall 提升不代表线上一定提升，还要通过消融和 A/B 实验验证它是否给后续排序提供了真正新增的高质量候选。
