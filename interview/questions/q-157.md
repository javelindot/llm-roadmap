# BLEU 指标

## 答案

**一句话**：BLEU（Bilingual Evaluation Understudy）= **n-gram 精确率 + brevity penalty** 的组合，衡量"机器译文和参考译文的 n-gram 重合度"——简单、自动、可比较，但只看表面 n-gram、不看语义。

### 1. 公式
$$BLEU = BP \cdot \exp\left(\sum_{n=1}^{N} w_n \log p_n\right)$$
- p_n：n-gram 精确率
- w_n：权重（通常 1/N，N=4）
- BP：brevity penalty（短译文惩罚）

### 2. n-gram 精确率
$$p_n = \frac{\sum_{\text{candidate}} \text{Count}_{\text{clip}}(\text{ngram})}{\sum_{\text{candidate}} \text{Count}(\text{ngram})}$$
- Count_clip：截断到参考译文中出现的最大次数
- 避免"this this this this"刷词作弊

### 3. Brevity Penalty（BP）
$$BP = \begin{cases} 1 & \text{if } c > r \\ e^{1-r/c} & \text{if } c \leq r \end{cases}$$
- c：候选译文长度
- r：参考译文长度
- 防止短译文用极少高频词刷分

### 4. 常见变种
- **BLEU-1**：只看 unigram
- **BLEU-4**：1-4 gram 几何平均（最常用）
- **SacreBLEU**：标准化版本，可重复比较
- **chrF**：character-level F-score（对形态丰富语言更好）

### 5. BLEU 的局限
- **不看语义**：同义改写得分低
- **依赖参考译文**：只有 1 个参考时偏差大
- **不擅长生成评估**：摘要、对话 BLEU 很差
- **对短句敏感**：长句更鲁棒
- **跨语言不可比**：BLEU 0.4 在英德 ≠ 英中

### 6. BLEU 的代替方案
| 指标 | 用例 |
|------|------|
| **ROUGE** | 摘要（看召回） |
| **METEOR** | 翻译（考虑同义词） |
| **chrF** | 形态丰富语言 |
| **BERTScore** | 语义相似度 |
| **BLEURT** | 学习型，更接近人类 |
| **COMET** | 多语言翻译 SOTA |
| **LLM-as-judge** | 任意生成任务 |

### 7. 现代 LLM 时代的位置
- 翻译：仍用 BLEU + chrF + COMET
- 摘要：ROUGE > BLEU
- 对话：BLEU 几乎无用，靠 LLM judge 和人工
- 开放生成：人工评 + 排行榜（MT-Bench 等）

### 8. 一句话总结
**BLEU = n-gram 重合 + 长度惩罚**，翻译领域的工业标准，但只看表面、不看语义，新一代评估靠 BERTScore/COMET/LLM-judge 补强。

## 结合项目谈（口语版 · 约 20 秒）

> 我们跨境支付里**BLEU 还在用，但已经不是主力**。
>
> **场景一：客服话术多语言翻译**——客服回复需要中→英→日→泰多语言版本，原文版本人工写。我们的翻译模型用 BLEU + chrF + COMET 三个指标评估，BLEU 是 baseline 但**最终决策看 COMET**（更接近人类感受）。BLEU 在我们东南亚小语种翻译里偏低且不稳，主要因为参考译文只有 1 份。
>
> **场景二：客服意图改写**——把客户口语化提问改写成知识库的标准表达。BLEU 在这里几乎没用——同样意图的"退款多久"和"退款时效是多久"BLEU 接近 0，但语义完全一样。我们改用 **BERTScore + LLM judge** 评估。
>
> **场景三：风控告警文案生成**——给运营生成"为什么这笔交易可疑"的解释文本。BLEU 完全不适合（开放生成），我们用 **GPT-4-as-judge** 评 4 个维度（信息完整、术语准确、清晰度、长度合理），加上**事实性 NLI 校验**。
>
> 一句话：**BLEU 在翻译领域仍是基线**，但 LLM 时代的开放生成任务里它越来越只是个参考——主战场已经是 LLM-as-judge 和人工评。

- "LLM-as-judge 在生成任务评估里的具体落地。"
- "评估指标怎么对齐业务效果可以展开。"
