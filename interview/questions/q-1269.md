# 为什么用精排

## 答案

召回阶段强调从海量候选中“不要漏”，受延迟限制通常只使用可离线索引的向量、关键词或轻量特征；精排面对较小候选集，可以使用更丰富的交叉特征和复杂模型，回答“候选之间最终怎么排”。

### 1. 召回分数表达能力有限

双塔的 User/Item 在打分前独立编码，点积计算快，但用户与候选之间的细粒度交互不足。BM25 和向量相似度也难同时处理：

- 否定和条件；
- 用户短期意图；
- 候选质量与权威性；
- 时间、价格、库存和风险；
- 多目标权衡；
- 候选之间的重复与多样性。

### 2. 精排可以使用丰富特征

精排输入通常包括：

- User、Item、Context 特征；
- 用户历史与候选的交叉；
- 召回通道和原始分数；
- 实时统计；
- 类目、时间和位置；
- 多任务目标；
- 业务规则和校准。

模型可使用 Wide&Deep、DeepFM、DCN、DIN/DIEN、Transformer、GBDT/LambdaMART 或 Cross-Encoder。

### 3. 精排优化目标更接近业务

召回常优化 Recall 或对比学习；精排可以预测 CTR、CVR、停留、满意度或相关性，并按业务价值组合：

```text
final_score =
P(click)
* P(convert | click)
* value
- risk_penalty
```

实际目标要处理校准、样本选择偏差和多任务冲突，不能直接把概率相乘而不验证。

### 4. RAG 中的精排

BM25/Dense 召回 Top-N 后，Cross-Encoder 联合编码 Query 与 chunk，能识别：

- 同词不同义；
- 否定和数字差异；
- 条件是否完整；
- Query 与证据是否真正对应。

同时加入版本、权限、来源和时效过滤，避免只看语义相似度。

### 5. 成本和降级

精排不是越复杂越好。需要控制：

- 候选 N；
- 输入最大长度；
- batch 与模型大小；
- P95/P99；
- 超时和缓存；
- 精排失败时的融合排序降级。

如果轻量召回已经能满足单一 FAQ，精排收益可能不抵成本。是否使用由 `NDCG/Precision@K/端到端正确率` 相对延迟和费用的增量决定。

跨境支付 RAG 中，精排尤其用于区分不同国家、币种、通道和版本的相似规则；实时订单事实仍来自工具，不由精排模型决定。
