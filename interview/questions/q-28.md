# PPO的缺点是什么？

## 答案

**一句话**：PPO 的缺点是工程链路复杂、训练成本高、调参敏感、稳定性差，并且容易受 reward model 偏差影响产生 reward hacking。

### 1. 主要缺点
- **模型多**：通常要维护 policy、reference、reward、value/critic 多个模型。
- **训练贵**：需要在线采样生成回答，样本效率不高。
- **调参难**：学习率、KL、clip range、reward scale、GAE 都会影响稳定性。
- **不稳定**：KL 可能突然飙升，模型可能崩或输出退化。
- **依赖 reward model**：reward model 有偏差时，PPO 会放大这个偏差。
- **容易 reward hacking**：模型找到讨好 reward model 的方式，不一定人类满意。
- **复现困难**：分布式训练、采样随机性和超参组合都会影响结果。
- **业务风险高**：如果在线探索接近真实业务，安全和合规成本很高。

### 2. 和 DPO 相比

DPO 更简单，直接用离线偏好对训练，不需要 reward model 和 value model，工程成本低很多。

但 PPO 也不是没价值。它更适合：
- 多步决策。
- 工具调用。
- 需要在线反馈。
- reward 不能简单写成 chosen/rejected 的场景。

### 3. 面试总结

PPO 强在能做复杂策略优化，弱在训练链路太重。实际项目里通常先用 SFT/DPO 这类简单方法，只有离线偏好优化不够时才考虑 PPO。

## 结合项目谈（口语版 · 面试现场约 50 秒）

> 放到跨境支付里，我会对 PPO 比较谨慎。因为支付系统不像游戏环境，可以随便探索。比如让模型学习怎么处理可疑交易，如果 PPO 在线探索太激进，可能带来误拦正常交易、漏放风险交易，甚至影响资金链路，这个成本很高。
>
> 另外 PPO 依赖 reward model。风控场景里的 reward 很难设计，一个回答到底好不好，不只看安全，还要看证据是否充分、是否影响商户体验、是否符合合规要求。如果 reward model 偏保守，PPO 可能把模型训成什么都转人工；如果 reward model 偏业务通过，又可能漏风险。
>
> 所以我实际选型会先用 SFT 加 DPO，把风控解释、审核建议、客服合规问答这些离线任务做好。只有到多步工具调用、策略路由、模拟处置流程这种 DPO 不太好表达的场景，我才会考虑 PPO，而且会先在离线模拟环境和 shadow mode 里验证。
