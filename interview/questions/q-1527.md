# 如何设计 Agent 的灰度发布策略？

## 答案

Agent 灰度发布比普通模型更复杂，因为它包含模型、Prompt、工具、知识、记忆和工作流状态，且一次请求可能执行多个有副作用的动作。应按“能力风险分级 + 分阶段流量 + 自动止损”设计。

### 1. 先定义发布单元

每次发布要明确变更了什么：

- 模型与采样参数；
- system prompt 或子 Prompt；
- planner/workflow；
- 工具 schema、权限和后端版本；
- 知识库、索引和 reranker；
- 记忆读写策略；
- 安全规则和 verifier。

版本必须绑定为一个可回滚 release，避免 Agent 版本回滚后仍读取新 schema 或新记忆。

### 2. 按动作风险分级

```text
L0：纯问答，无工具
L1：只读检索/查询
L2：创建草稿、写入低风险系统
L3：外部发送、账户修改、资金动作
```

新版本先开放 L0/L1；L2 要预览和确认；L3 默认 shadow 或人工审批，不能直接随流量比例自动放开。

### 3. 分阶段流程

1. 离线黄金集、事故回放和对抗测试；
2. 仿真环境执行，验证工具参数和状态迁移；
3. shadow traffic，只生成计划，不执行副作用；
4. 内部员工和白名单租户；
5. 1% canary；
6. 5%/10%/25%/50% 分层扩量；
7. A/B 或全量，并保留长期 holdout。

流量应按稳定用户、商户或会话 ID 分桶，保证一个任务全程固定版本。

### 4. 放量指标

- 任务成功率和首次解决率；
- 工具选择、参数和执行成功率；
- 无效循环、重试和人工接管率；
- 事实性、引用和安全违规；
- P95/P99、token、工具次数和成本；
- 不可逆动作、投诉和业务损失；
- 新旧版本分群差异。

每一阶段都设置进入条件、观察窗口和停止阈值。

### 5. 回滚与状态处理

- feature flag 一键切回旧 workflow；
- 工具服务端保持鉴权和幂等，不能信任 Agent；
- 新旧状态 schema 做向后兼容或迁移；
- 队列中的长任务可固定旧版本完成，或从 checkpoint 安全重启；
- 记忆写入带版本、来源和 TTL，错误记忆支持删除；
- 达到安全、错误率或尾延迟阈值自动熔断。

### 6. 跨境支付场景

客服解释可先灰度，订单查询只读开放；退款、出款、冻结和收款账户修改即使模型效果很好，也必须经过用户确认、服务端重新校验和人工审批。支付成功率之外，还要看重复操作、金额币种错误和账务一致性。

### 7. 面试总结

Agent 灰度的核心不是只调流量百分比，而是把版本、状态和动作风险一起管理：离线 -> 仿真 -> shadow -> 白名单 -> canary -> 分层扩量，并为高风险动作保留硬权限和人工审批。
