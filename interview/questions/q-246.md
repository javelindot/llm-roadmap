# 怎么做压测的？

## 答案

vllm自带的benchmark

### 补充说明

压测不能只跑一个工具命令，要先定义目标：测 RAG 端到端、测 LLM 推理服务、测接口网关，还是测整个客服链路。不同目标指标不一样。

常用方式：

- vLLM 自带 benchmark 测模型吞吐、首 token 延迟、输出 token/s。
- Locust、JMeter、k6 测 HTTP 接口并发和稳定性。
- 自定义压测脚本模拟不同 prompt 长度、输出长度、并发数。
- 真实业务样本回放，覆盖 FAQ、订单查询、风控解释、文档问答。

核心指标包括 QPS、P50/P95/P99、首 token 延迟、总耗时、错误率、超时率、GPU 显存、队列长度、CPU/内存、下游接口耗时。

### 结合跨境支付项目怎么说（面试口径）

> 在跨境支付客服项目里，我会分层压测。先单独压 vLLM，看不同输入长度和并发下的吞吐和显存；再压 RAG 服务，看检索、rerank、生成的端到端耗时；最后用真实客服问题回放，比如付款未到账、KYC 驳回、退款时效、风控拦截。
>
> 压测数据不能只看平均值，我更关注 P95/P99，因为客服高峰时慢请求会明显影响体验。还要看下游订单系统和风控系统是否被工具调用打爆，所以工具接口也要限流。
>
> 最后根据压测结果做限流、缓存、模型分层和降级。比如高频错误码问题走缓存，复杂问题走大模型，超时后转人工。

**钩子**：可以引到“vLLM benchmark、真实工单回放、P95/P99、下游工具限流”。
