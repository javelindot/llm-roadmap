# DeepSeek 有了解吗， DeepSeek 用到的 MLA 注意力是怎么做的？它可以直接用 RoPE 吗？为什么不能，它做了哪些优化

## 答案

DeepSeek 的 MLA 用低秩 latent 压缩 K/V Cache，但不能把标准 RoPE 简单作用在全部压缩表示上，否则会影响 Decode 阶段的矩阵吸收优化。因此它采用 decoupled RoPE，把内容注意力和位置注意力拆开。

### 1. MLA 主体

对 hidden state `h_t`：

```text
c_t^KV = W_DKV * h_t
K_t^C  = W_UK  * c_t^KV
V_t^C  = W_UV  * c_t^KV
```

`c_t^KV` 是低维表示。推理时缓存 `c_t^KV`，而不是各 head 展开后的完整 K/V。

Query 也可做低秩投影：

```text
c_t^Q = W_DQ * h_t
Q_t^C = W_UQ * c_t^Q
```

### 2. 为什么标准 RoPE 不能直接套用

无位置旋转时，线性投影可以重排：

```text
q^T * (W_UK * c)
= (W_UK^T * q)^T * c
```

这样 Decode 时可以直接让变换后的 Query 与缓存 latent 做计算，不必逐 token 恢复完整 K。

但 RoPE 矩阵 `R_t` 随位置 `t` 变化：

```text
(R_i q)^T * (R_j W_UK c_j)
```

位置相关旋转夹在投影之间，无法统一吸收到固定的 `W_UK` 中。若强行这样做，就会失去 latent cache 的主要计算和带宽优势。

### 3. Decoupled RoPE

DeepSeek 将每个 head 拆成：

- 内容 Query/Key：`Q^C`、`K^C`，走 MLA 压缩；
- 位置 Query/Key：`Q^R`、`K^R`，维度较小并应用 RoPE。

注意力分数近似：

```text
score(i,j)
= (Q_i^C)^T K_j^C
  + (RoPE(Q_i^R))^T RoPE(K_j^R)
```

缓存只需保存低维 `c_j^KV` 和共享/较小的 `K_j^R`。

### 4. 与 GQA/MQA 的区别

- MQA：所有 Query head 共享一组 K/V；
- GQA：若干 Query head 共享 K/V；
- MLA：缓存一个低秩 latent，各 head 通过投影读取不同内容。

MLA 保留更强的多头表达能力，同时把缓存压得更小，但实现复杂度和 kernel 适配成本更高。

### 5. 工程结论

MLA 的主要收益在长上下文、高并发 Decode，因为这里更受 KV Cache 容量和显存带宽限制。是否加速要通过目标框架和硬件实测，不能仅凭理论缓存下降比例推断端到端吞吐。
