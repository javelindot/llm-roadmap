# 整个项目采用了什么技术框架？为何选择 Python 语言来开发 RAG 相关模块？

## 答案

> 我们整个跨境支付 AI 平台是 **Python + Java 双语言架构**——RAG / 模型层用 Python，业务网关 / 交易主链路用 Java。
>
> **完整技术栈**：
>
> - **Python 部分**（AI 核心）：
>   - 框架：FastAPI（同步 + 异步 endpoint）+ asyncio
>   - LLM：HuggingFace transformers、vLLM（推理）、PEFT（微调）
>   - RAG：LangGraph、Milvus、Elasticsearch、bge 系列
>   - 多模态：Qwen-VL、PaddleOCR、Marker、MinerU
>   - 任务调度：Celery + Redis、APScheduler
>   - 监控：OpenTelemetry + Prometheus
> - **Java 部分**（业务主线）：
>   - 网关：Spring Cloud Gateway
>   - 微服务：Spring Boot + Dubbo
>   - 消息：Kafka
>   - 数据库：MySQL + TiDB + Redis
> - **基础设施**：
>   - K8s（私有云 + 阿里云）
>   - GitLab CI + ArgoCD
>   - MinIO（模型仓库）
>
> **为什么 RAG 模块用 Python**：
>
> **1. 生态绝对碾压**
> 整个 LLM / RAG / Agent 生态——HuggingFace、LangChain、LangGraph、LlamaIndex、Milvus、vLLM、bge——全部是 Python first。Java 也有 LangChain4j 但功能滞后 3-6 个月。
>
> **2. 数据处理优势**
> NumPy / pandas / scikit-learn 处理 embedding 向量、batch 数据天生快、API 友好。Java 等价方案（ND4J）很难用。
>
> **3. 快速迭代**
> RAG / Agent 业务变化频繁，Python 改完即跑、没编译期。试错速度比 Java 快几倍。
>
> **4. 模型生态**
> PyTorch、transformers 都是 Python。Java 推理只能用 ONNX / Triton，灵活性差。
>
> **5. 团队习惯**
> AI 工程师对 Python 熟悉度远超 Java。
>
> **Python 的代价**（也要诚实说）：
> - 性能：GIL 限制，并发用 asyncio 或多进程
> - 生产稳定性：弱类型 → 我们强制 mypy + Pydantic
> - 部署：Java 单 jar 简单，Python 要管虚拟环境 + 依赖
>
> **怎么解决 Python 性能问题**：
> - **vLLM / TGI 做推理**，本身就是 C++ / CUDA 实现
> - **FastAPI + uvicorn[standard] + asyncio** 高并发
> - **关键热路径用 Rust 写**（部分嵌入预处理）
> - **CPU 密集任务交给 Java 微服务**
>
> 一句话：**Python 在 AI 层是不可替代**，Java 在业务主链路是稳定基础，两者通过 HTTP / gRPC 通信。
