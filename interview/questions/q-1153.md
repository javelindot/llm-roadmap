# 谈谈传统搜索算法和RAG的理解

## 答案

传统搜索和 RAG 不是替代关系。传统搜索负责从大规模语料中找出并排序相关证据；RAG 在检索结果之上增加上下文组织和生成能力，把多份证据综合成答案。生产级 RAG 的底座仍然是搜索，检索失败后，生成模型通常只会把错误包装得更流畅。

### 1. 传统搜索解决“哪些文档相关”

传统搜索通常包括：

- 倒排索引和 BM25 等词项匹配；
- 字段过滤、分面、同义词、拼写纠错和查询扩展；
- 规则打分或 Learning to Rank；
- 权限、时效、地域和文档类型过滤；
- 高亮、分页、结果来源和可解释排序。

它对错误码、订单号、国家码、币种、通道名和产品名等精确标识符非常可靠，延迟低、可缓存、结果稳定，也容易执行 ACL 和版本过滤。缺点是对口语化、多语言、上下文省略和语义改写的理解有限。

Dense Retrieval 使用向量相似度补足语义匹配。例如“款一直没到”和“付款处于银行处理中”可能没有明显关键词重合，但语义相近。它的风险是把“支持 USD”与“不支持 USD”、不同国家的相似规则排得很近，因此不能替代精确词项、结构化过滤和业务约束。

### 2. RAG 解决“如何基于证据回答”

RAG 通常增加以下环节：

> 查询理解 → 混合召回 → 融合与重排 → 上下文构造 → 受约束生成 → 引用与校验。

它的价值主要是：

- 将多篇规则整理为直接、连贯的回答；
- 结合对话上下文解释用户的真实问题；
- 提供来源和版本，使知识可以更新和追踪；
- 在证据不足时追问、拒答或转人工；
- 将知识解释与实时工具查询编排到同一流程中。

RAG 的成本和风险也更高，包括模型延迟、token 成本、上下文污染、引用错配和生成幻觉。因此高风险答案不能只给自然语言结论，还要展示依据并设置可验证边界。

### 3. 跨境支付场景应按数据类型分流

不同问题适合不同链路：

- “错误码 `PAYOUT_1027` 是什么”优先走精确搜索和结构化错误码库；
- “英国 GBP 本地付款需要哪些收款人信息”适合混合检索后由 RAG 汇总规则并引用来源；
- “这笔付款为什么没有到账”必须调用订单、通道回执和清结算系统，RAG 只负责解释对应规则；
- “帮我重新发起付款”属于资金操作，必须走确定性工作流、鉴权、幂等和人工审批，不能由生成模型直接执行；
- 合规或风控原因无法公开时，系统应返回受控说明，不能通过检索内部材料绕过披露政策。

因此更合理的架构是：

```text
用户问题
  ├─ 文档与规则：Search + RAG
  ├─ 实时交易状态：Tool / API
  ├─ 精确实体查询：SQL / Search / KV
  └─ 资金操作：受控业务工作流
```

### 4. 生产方案通常采用多阶段混合搜索

第一阶段追求高召回率，组合 BM25、Dense Retrieval、字段查询和 FAQ；第二阶段用 RRF 或校准分数融合；第三阶段由 Cross-Encoder Reranker 判断 query 与候选的细粒度相关性；最后执行权限、时效、冲突和证据充分性检查。

生成上下文中每条证据要携带 `source、version、effective_time、country、currency、product`。如果工具数据和文档冲突，订单或账务等实时事实以权威业务系统为准，同时触发知识更新任务。

### 5. 搜索和 RAG 要分层评测

- **搜索层**：Recall@K、MRR、NDCG、Precision@K、过滤正确率；
- **上下文层**：关键证据覆盖率、重复率、冲突率和过期文档进入率；
- **生成层**：faithfulness、答案完整性、引用正确率和拒答准确率；
- **业务层**：一次解决率、转人工率、人工改写率、错误承诺率；
- **系统层**：P95/P99、QPS、超时率和单次成本。

只看最终答案分数会掩盖根因。必须保留查询改写、候选列表、融合分数、过滤原因、最终上下文和模型输出的 trace，才能判断问题来自搜索、重排、上下文构造还是生成。
