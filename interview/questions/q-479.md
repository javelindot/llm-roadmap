# 项目:如何在有限算力下做大模型微调?常用方法有哪些?

## 答案

有限算力下的核心原则是：**先减少需要训练的参数，再减少每一步的显存和计算，最后用高质量数据提升单位算力收益**。

### 1. 优先选择 PEFT

- **LoRA**：冻结基座模型，只训练低秩矩阵，通用性最好。
- **QLoRA**：基座权重量化到 4bit，计算时反量化，进一步降低显存。
- **Adapter**：在层间插入小模块。
- **Prefix / Prompt Tuning**：只训练虚拟 Token，成本更低但能力上限通常较低。
- **部分层微调**：只解冻顶部层、Norm 或 LM Head。

大多数业务指令微调会先从 LoRA/QLoRA 开始。

### 2. 降低训练显存

- BF16/FP16 混合精度。
- Gradient Checkpointing：用重算换显存。
- Gradient Accumulation：小 Micro Batch 模拟大 Batch。
- FlashAttention / SDPA。
- 8bit Optimizer。
- ZeRO、CPU/NVMe Offload。
- 缩短序列长度或按长度分桶，减少 Padding。

### 3. 降低任务规模

- 选择更合适的小模型，而不是默认追大模型。
- 先做领域继续预训练还是 SFT，要看知识和行为缺口。
- 只对关键模块注入 LoRA，例如 Attention 的 q/v，必要时再扩展到 FFN。
- 使用蒸馏，把大模型生成或评审的高质量数据用于训练小模型。

### 4. 数据比盲目加步数更重要

- 去重、纠错、统一格式。
- 保留真实难例和高价值长尾。
- 控制类别、长度和场景分布。
- 先用小样本跑通并做 Learning Curve，再决定是否扩数据。

### 5. 一个可执行方案

```text
7B/8B 基座
→ 4bit QLoRA
→ r=16 或 32
→ BF16 计算
→ Gradient Checkpointing
→ 小 Micro Batch + 梯度累积
→ 1–3 Epoch
→ 真实业务集评测
```

具体参数要根据数据规模、序列长度和任务复杂度调节。

**面试总结**：有限算力时，最优解通常不是把所有参数都训一遍，而是用 PEFT、量化和高质量数据把算力集中在任务真正需要改变的能力上。
