# 问code llm中的训练难点

## 答案

Code LLM 的训练难点不只是收集更多代码，而是处理许可、重复、仓库上下文、可执行性、安全和评测污染。

### 1. 数据来源与许可

- 明确开源许可证和商用边界；
- 保留仓库、文件、commit 和 license provenance；
- 删除密钥、Token、个人信息和内部域名；
- 避免把生产私有代码无授权送入训练；
- 支持按仓库或权利请求删除数据；
- 区分人工代码、生成代码和低质量自动提交。

代码可公开访问不等于可无条件用于任何训练和分发。

### 2. 去重与污染

代码存在 fork、镜像、复制粘贴和模板：

- 文件级精确哈希；
- token/AST/MinHash 近重复；
- 仓库级切分；
- benchmark 与训练数据去污染；
- 同一题解的变量改名不能视为独立样本；
- 训练、验证、测试按仓库和时间隔离。

不严格去重会让离线 benchmark 虚高。

### 3. 代码解析与格式

- 使用语言解析器识别函数、类、注释和依赖；
- 保留缩进、换行和特殊 token；
- Tokenizer 要兼顾标识符、空白和多语言代码；
- 处理截断的文件和不完整 AST；
- 保留文件路径、语言、依赖和仓库结构；
- Notebook、配置、SQL 和脚本需要不同处理。

### 4. 训练目标

除了 Left-to-Right，还可使用 Fill-in-the-Middle：

```text
prefix + suffix → middle
```

它更接近 IDE 补全。仓库级任务还需要：

- 跨文件依赖；
- import、类型和 API；
- 修改已有代码而非从零生成；
- Issue、测试和 patch 对齐；
- Tool use，如搜索、编译和测试。

### 5. 长上下文和检索

完整仓库远超上下文窗口。需要：

- 文件和符号索引；
- import/call graph；
- 代码 Embedding 与 BM25；
- 当前编辑文件、相关定义和测试的动态检索；
- Context Packing 和去重；
- 防止检索第三方无关或越权代码。

上下文越长不代表越好，错误依赖会增加幻觉。

### 6. 可执行反馈

自然语言相似度不足以评估代码。训练和筛选可使用：

- 编译、静态类型和 lint；
- 单元测试、属性测试；
- 沙箱执行和资源限制；
- Patch 是否正确应用；
- 安全扫描；
- 多轮修复后的最终仓库状态。

生成代码必须在隔离沙箱中运行，限制网络、文件、CPU、内存和时间，避免执行恶意样本。

### 7. SFT 与偏好数据

SFT 样本包括需求、仓库上下文、修改 patch、测试结果和最终修复。偏好对应该比较功能正确、安全、可维护和最小改动，而不是只比较代码长度或注释数量。

自动 Judge 必须结合编译和测试结果，不能仅凭代码外观打分。

### 8. 评测

- HumanEval/MBPP 只覆盖函数级生成；
- Repo-level 使用 SWE-bench 类真实 Issue；
- pass@k、编译率和测试通过率；
- Patch 正确率、无关改动和回归；
- 多语言、框架和依赖版本分桶；
- 安全漏洞、许可证和秘密泄漏；
- 延迟、上下文 token 和工具调用成本。

### 9. 跨境支付代码场景

支付代码包含密钥、账户、商户逻辑和合规规则，不能直接进入通用训练。更稳妥的是在权限隔离的私有环境中使用代码检索和小规模适配，并要求生成的账务、幂等、金额和币种逻辑通过确定性测试及人工 Review。
