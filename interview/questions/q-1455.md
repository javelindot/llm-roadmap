# 8. 在微调 Qwen 的时候，数据是怎么构造的

## 答案

微调 Qwen 的数据构造与其他 Decoder-only 指令模型类似，但必须使用与具体 Qwen 版本匹配的 chat template，并正确处理 system、user、assistant、tool 和多模态内容。不能只把问答字符串随意拼接。

### 1. 先定义任务

根据目标构造不同样本：

- 领域问答；
- 多轮客服；
- 信息抽取；
- JSON/结构化输出；
- Function Calling；
- RAG 引用；
- 推理与规划；
- 安全拒答；
- 多语言；
- 图片/音频等多模态任务。

### 2. 对话格式

逻辑数据可写成：

```json
{
  "messages": [
    {"role": "system", "content": "你是跨境支付助手"},
    {"role": "user", "content": "为什么出款未到账？"},
    {"role": "assistant", "content": "请提供出款订单号。"}
  ]
}
```

训练前用对应 tokenizer 的：

```python
tokenizer.apply_chat_template(...)
```

生成真实 token 序列。不同 Qwen 版本的特殊 token 和工具模板可能不同，必须固定 tokenizer revision。

### 3. Loss Mask

通常只监督 assistant 输出：

```text
system/user/tool input labels = -100
assistant target labels = token ids
```

工具调用样本中，可以监督 assistant 的 tool call 和最终回答；tool result 通常作为条件输入，不作为模型生成目标，除非任务设计另有目的。

### 4. 数据来源

- 专家示范；
- 脱敏真实工单；
- SOP/知识库改写；
- 强模型合成后筛选；
- 工具成功轨迹；
- 线上 bad case；
- 安全和拒答样本；
- 多语言平行或本地化数据。

### 5. 清洗

- PII 脱敏；
- 去重和近重复；
- 删除错误、过期、无来源答案；
- 统一角色和格式；
- 校验 JSON/tool schema；
- 控制长度和截断；
- 过滤答案泄漏；
- 按时间、商户或任务切分；
- 保留困难和关键长尾。

### 6. Packing 与截断

短样本可以 packing 提高利用率，但要确保边界和 loss mask 正确。长样本应：

- 保留 system 和关键用户约束；
- 避免从中间截断 JSON 或 tool call；
- 对超长文档改用 RAG；
- 按长度分桶降低 padding。

### 7. 数据配比

领域样本、通用指令、安全和工具数据需要平衡。真实高频样本不应完全淹没高风险长尾；合成数据也不能占比过高而导致语言单一和教师偏差。

### 8. 跨境支付示例

```text
输入：用户问题 + 授权后的订单工具结果
目标：解释状态、给下一步、禁止承诺未知到账时间
```

实时费率、订单和规则不应作为固定答案背进参数，而应训练模型正确调用工具。

### 9. 面试总结

Qwen 微调数据构造的关键是：任务 taxonomy、正确 chat template、assistant-only loss mask、工具 schema、质量清洗和分层评测。模型名称不是重点，数据是否可验证且与线上输入一致才是重点。
