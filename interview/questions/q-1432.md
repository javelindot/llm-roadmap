# 介绍实习时自动化如何实现(基于什么框架和工具实现)

## 答案

这题要按真实项目回答“触发、编排、执行、状态、监控和审批”六个部分，而不是只报框架名。下面给出一个可替换的自动化架构。

### 1. 自动化链路

```text
Scheduler / Webhook / Message Event
-> Workflow Orchestrator
-> Task Queue
-> Workers / Agent
-> Internal APIs and Tools
-> State Store
-> Notification / Human Approval
```

### 2. 可选框架

按任务特征选：

- Airflow：定时批处理和有向无环数据任务；
- Temporal：长流程、重试、补偿、状态恢复；
- Celery/RQ：异步任务队列；
- Kafka/RabbitMQ：事件和消息；
- LangGraph：带状态的 Agent 图编排；
- FastAPI：工具和服务接口；
- Playwright/RPA：没有 API 的页面自动化，优先级低于正式 API；
- PostgreSQL/Redis：状态、锁和缓存；
- Prometheus/Grafana/OpenTelemetry：监控和追踪。

不要因为使用大模型就把所有流程写成 Agent。确定性步骤更适合 Workflow。

### 3. 工程关键点

- 每个任务有 `task_id` 和幂等键；
- 状态持久化，进程重启后可恢复；
- retry 使用退避和最大次数；
- 不可逆动作设计 compensation；
- 工具 schema 和错误码统一；
- 密钥放 Secret Manager；
- 权限最小化；
- 日志脱敏；
- 高风险动作人工审批；
- 失败进入 dead-letter queue。

### 4. 如果使用 LLM/Agent

LLM 适合：

- 理解非结构化输入；
- 分类和抽取；
- 生成计划；
- 在受限工具集合中路由；
- 总结执行结果。

确定性系统负责：

- 状态机；
- 调度和重试；
- 数据写入；
- 权限；
- 审批；
- 审计。

### 5. 跨境支付示例

自动化处理商户支付失败工单：

```text
工单事件
-> 抽取订单号和意图
-> 查订单/通道/账务
-> 根据规则生成原因
-> LLM 生成解释草稿
-> 高风险场景转人工
-> 回复并记录审计
```

退款、调账、出款等资金动作不能由浏览器 RPA 或自由 Agent 无审批执行。

### 6. 如何证明做过

准备说明：

- 任务量和触发方式；
- 使用的框架及选型原因；
- 失败率、重试和 SLA；
- 自己写了哪些模块；
- 遇到过的故障；
- 上线前后的人工时长；
- 监控与回滚。

### 7. 面试总结

自动化的核心是可靠编排，不是工具数量。回答时讲清触发、状态、幂等、重试、权限、人工审批和观测，再说明框架如何服务这些要求。
