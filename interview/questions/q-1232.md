# RAG怎么用的，有没有更好的改进方法，为什么套用大模型结合BM25算法，文档怎么处理的，文档都有什么类型，如何整入数据库并提取

## 答案

RAG 的作用是让模型在生成前读取当前有效、具备权限且可追溯的外部证据。生产流程不是“把文件丢进向量库”，而是：

> 数据接入 → 解析清洗 → 结构切块 → 稀疏/稠密索引 → 查询理解 → 混合检索 → 重排过滤 → 生成引用 → 评测与增量更新。

### 1. 为什么同时使用 BM25 与大模型

大模型负责理解问题、改写 Query、组织答案和判断证据是否充分；BM25 是检索器，不与大模型互相替代。

**BM25 的优势**

- 精确命中错误码、国家码、币种、API 字段和产品名；
- 无需训练，行为可解释；
- 对新出现的专有名词和长尾标识友好；
- 索引和过滤能力成熟。

**Dense Retrieval 的优势**

- 处理同义词、口语和多语言表达；
- “款没到”可以匹配“付款处理中”；
- 对字面不同但语义接近的 Query 更稳。

两路候选使用 RRF 或校准分数融合，再由 cross-encoder reranker 精排。仅使用 BM25 会漏掉语义改写，仅使用向量检索又容易错过精确标识。

### 2. 文档类型和解析策略

| 文档类型 | 解析重点 |
|---|---|
| HTML/Markdown | 标题层级、链接、代码块 |
| PDF/Word | 版面、页码、段落、页眉页脚 |
| 扫描件/图片 | OCR、区域、置信度和人工复核 |
| Excel/CSV | 表头、单位、合并单元格、主键 |
| FAQ | 问题、答案、同义问和适用范围 |
| API 文档 | endpoint、字段、类型、错误码、示例 |
| 数据库 | 结构化字段、时间、主键和权限 |

跨境支付规则必须保留国家、币种、产品、通道、客户类型、条件、例外和生效时间，不能只保留一段脱离上下文的正文。

### 3. 如何切块

- 先按文档结构切分，再对超长段落递归切；
- FAQ 一个完整问答为原子块；
- 错误码与原因、处置方式放在同一块；
- 表格按业务实体或行组切，并重复必要表头；
- 操作流程保留步骤顺序和异常分支；
- 使用父子块兼顾小块召回精度和大块上下文完整性；
- overlap 只用于被迫跨边界的长段落，召回后去重。

块大小从真实评测确定，而不是固定 500 字。需要同时观察 Recall@K、Context Precision、答案完整性、延迟和 token 成本。

### 4. 如何进入数据库

通常至少维护三类存储：

**对象存储或文档库**

- 保存原文件、解析结果和内容指纹；
- 支持审计、重放和重新解析。

**关系型/元数据存储**

```text
document_id, chunk_id, tenant_id, source,
title_path, page, country, currency, product,
version, effective_from, effective_to, ACL,
parser_version, embedding_version, content_hash
```

**检索索引**

- 倒排索引存正文 token 与 BM25 字段；
- 向量索引存 embedding 和 filter metadata；
- 必要时为结构化表格建立 SQL/列式索引。

写入采用幂等 upsert。文档更新时按 `document_id + version + content_hash` 生成变更集，新增或更新 chunk，同时删除旧版本索引，避免新旧规则同时召回。索引与元数据提交失败时需要补偿或重试。

### 5. 在线提取和检索

1. 鉴权并绑定当前商户、角色和租户；
2. 恢复多轮指代，提取国家、币种、产品、错误码和时间；
3. 判断应查知识、实时工具还是两者都查；
4. 前置 ACL、生效时间和业务元数据过滤；
5. BM25 与 dense 多路召回；
6. RRF 融合、去重和 rerank；
7. 冲突检测与证据充分性判断；
8. 将少量证据按来源、版本和页码打包给模型；
9. 对金额、状态、日期和引用做生成后校验。

实时订单、余额、报价、通道健康和审核状态必须由 API/SQL 工具返回，不能从静态向量库推断。

### 6. 更好的改进方法

- Query rewrite、multi-query 和问题分解提升复杂问法召回；
- Contextual chunk、父子块和 late chunking减少切块后的语境丢失；
- 领域 embedding 与 cross-encoder reranker；
- 结构化过滤、时间衰减和版本冲突处理；
- HyDE 只在验证有效时使用，避免生成假设文档带偏；
- Agentic RAG 用于需要多步检索和工具的任务，但限制最大步骤与权限；
- 对高频 Query 做安全缓存，缓存键包含租户、权限和知识版本；
- 建立 no-answer 与低置信度转人工机制；
- 使用 bad case 回流优化具体模块，而不是只调整 Prompt。

### 7. 评测与上线

- 检索：Recall@K、MRR、NDCG、长尾实体覆盖；
- 重排：Precision@K、关键证据保留率；
- 生成：faithfulness、正确性、引用和拒答；
- 安全：越权召回、PII 泄露、错误资金承诺；
- 系统：P95/P99、索引新鲜度、超时、成本和降级率。

每次 embedding、切块、reranker 或模型升级都要使用同一冻结黄金集回归，并通过影子流量与小流量灰度上线。
