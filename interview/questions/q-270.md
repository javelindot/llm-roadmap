# 检索阶段怎么优化RAG？

## 答案

**一句话**：检索阶段优化 RAG，重点是让“正确内容能被召回并排在前面”，所以要从 query、chunk、索引、召回、过滤和重排一起优化。

### 1. Query 优化

用户问题往往表达不标准，可以做：

- 查询改写，把口语化问题改成业务术语。
- 多路查询，从不同角度召回。
- 同义词扩展，比如 payout、付款、出款。
- 错误码、币种、国家、通道名等关键词增强。
- 意图识别后进入不同检索策略。

### 2. Chunk 和索引优化

- 按标题、段落、表格语义切分。
- 保留标题路径，避免语义丢失。
- 给 chunk 加业务元数据，比如国家、币种、通道、版本。
- 对 FAQ、错误码、制度文档分别建索引或字段。
- 清理过期和重复文档。

### 3. 召回和重排优化

- 向量召回处理语义相似。
- BM25 处理关键词精确匹配。
- 混合召回兼顾语义和精确。
- Rerank 模型提高排序质量。
- 业务规则加权，比如最新版本、官方文档优先。

### 4. 结合跨境支付项目怎么说（面试口径）

> 在跨境支付 RAG 里，检索优化非常关键。比如用户问“美金一直没到账”，如果只用向量检索，可能召回到通用到账时效 FAQ，但真正原因可能是 USD payout 某个通道的中间行审核规则。所以我会先做 query rewrite，把问题扩展成“美元付款未到账、USD payout、银行处理中、到账时效、通道回执”等表达。
>
> 然后做混合召回。支付场景里错误码、币种、国家、通道名非常重要，必须支持精确匹配。召回后再用 rerank，结合订单里的国家、币种、通道和当前状态，把最相关的知识排到前面。
>
> 我还会给知识 chunk 加元数据，比如“国家=巴西、支付方式=PIX、生效时间=2026-01”。这样系统不会拿别的国家规则回答当前问题。

**钩子**：可以引到“query rewrite、混合召回、元数据过滤、支付错误码精确匹配”。
