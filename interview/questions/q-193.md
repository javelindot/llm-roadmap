# ReAct 的思路是怎样的？跑过 Demo 吗？

## 答案

> ReAct 跑过、还落地了生产场景。先说思路，再说我们的实战。
>
> **ReAct 思路**：
>
> 名字是 **Reasoning + Acting** 的缩写。核心是个循环：
>
> ```
> Thought（思考下一步要干啥）
>   → Action（调一个工具）
>   → Observation（看工具返回结果）
>   → Thought（基于结果想下一步）
>   → ... 直到能给 Final Answer
> ```
>
> 模型每一步显式输出 "Thought: xxx, Action: tool_name, Action Input: xxx"，框架解析后调工具，再把 Observation 拼回上下文让模型继续。
>
> **为什么有效**：
> - 思考过程显式 → 可调试可观测
> - 工具调用解决 LLM 的能力短板（查数据、算数、搜索）
> - 错了能 self-correct（在下一个 Thought 里反思）
>
> **我们的实战**：
>
> 跨境支付里我们**客服意图分类 + 工单分流**就是 ReAct 风格：
> - 工单进来 → Agent 思考：这是查询？投诉？还是异常？
> - Action：根据猜测调对应工具（查订单 / 查商户 / 查风控）
> - Observation：拿到数据后继续判断
> - 最终给出"工单类型 + 处理建议"
>
> 我们用 LangChain 的 `create_react_agent` 跑通的，后来切到 LangGraph（详见 q-13）。
>
> **几个踩坑**：
> - **格式不稳**：Qwen-7B 早期版本经常不输出标准 `Thought/Action` 格式，要 prompt 强化 + 失败重试
> - **死循环**：Agent 在工具结果不理想时反复 retry 同一工具，加 `max_iterations=10` 兜底
> - **过度思考**：简单工单也写一大段 Thought，token 成本高。后来加 "简单问题直接答" 的 prompt 约束
>
> **我对 ReAct 的看法**：
> - 适合**短链路、低风险**任务（意图分类、信息查询）
> - 长链路 / 高风险（风控调查）用 **Plan-and-Execute** 更合适（可追溯）
> - 是 Agent 入门最朴素的范式，是其他范式的基础

**🪝 顺嘴带出去的钩子**：
- "ReAct vs Plan-and-Execute 在我们业务里的选型可以聊。"
- "Qwen 早期 ReAct 格式不稳的解法可以展开。"
- "max_iterations 怎么设和死循环治理。"
