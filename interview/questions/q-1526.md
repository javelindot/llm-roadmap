# 对其他分布式工具是否有所了解？

## 答案

了解分布式工具时，应按“单机多卡、数据并行分片、模型并行、任务编排和推理服务”分层回答，而不是简单罗列框架。

### 1. 训练侧工具

| 工具 | 主要能力 | 适用场景 |
|---|---|---|
| PyTorch DDP | 梯度同步的数据并行 | 模型单卡可放下，追求简单稳定 |
| PyTorch FSDP/FSDP2 | 参数、梯度、优化器状态分片 | 大模型训练，偏原生 PyTorch |
| DeepSpeed | ZeRO、Offload、流水线等 | 显存紧张、已有 DeepSpeed 生态 |
| Megatron-Core | TP、PP、CP、EP 等大规模模型并行 | 超大模型、高性能集群 |
| Colossal-AI | 多种并行与显存优化 | 研究和特定训练栈 |
| JAX distributed | `pjit/shard_map` 等显式切分 | JAX/TPU 生态 |

模型并行维度包括：

- TP：层内矩阵切分；
- PP：按层切流水线；
- CP/SP：序列或上下文切分；
- EP：MoE 专家切分；
- DP/FSDP：样本和模型状态切分。

### 2. 调度与编排

- **Ray Train/Ray Core**：分布式任务、actor、容错和资源调度；
- **Kubernetes**：容器编排、弹性、作业生命周期；
- **Slurm**：HPC 集群队列和资源管理；
- **Kubeflow/Argo**：训练流水线和工作流；
- **verl**：在 Ray 上组织 actor、rollout、reference、critic 等 RL worker。

这些工具解决的不是同一层问题。例如 Ray 可以启动 FSDP 训练任务，但不能替代 FSDP 的张量分片。

### 3. 推理侧

- vLLM、SGLang：continuous batching、KV Cache 管理和高并发服务；
- TensorRT-LLM：NVIDIA 平台 kernel 与图优化；
- Triton Inference Server：多模型服务、动态批处理与标准部署；
- llama.cpp：CPU/边缘和量化部署。

推理框架与训练框架的权重格式、并行布局和数值 kernel 可能不同，RL rollout 场景还需处理训练推理一致性。

### 4. 如何选型

```text
模型能否单卡放下
-> 集群网络和 GPU 拓扑
-> 需要哪种并行维度
-> 吞吐/显存目标
-> checkpoint 与容错
-> 团队维护成本
```

模型单卡可放下时优先 DDP；需要状态分片可选 FSDP/ZeRO；超大 Dense/MoE 模型再组合 Megatron 的 TP/PP/CP/EP。先用 profiler 找到计算、显存还是通信瓶颈，不能为了“分布式”叠加全部工具。

### 5. 跨境支付场景

模型训练和推理可使用上述框架，但敏感数据还要满足地域调度、租户隔离、加密、审计和 checkpoint 生命周期管理。不能让通用调度器把数据或缓存自动迁移到不合规区域。

### 6. 面试总结

DDP/FSDP/DeepSpeed/Megatron 解决训练并行，Ray/K8s/Slurm 解决资源编排，vLLM/SGLang/TensorRT-LLM 解决推理。选型关键是瓶颈、拓扑、容错和维护成本，而不是框架数量。
