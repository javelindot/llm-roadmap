# 如何通过 混合检索 提升 RAG 效果？

## 答案

**一句话**：混合检索是把关键词检索和向量检索结合起来，兼顾精确匹配和语义匹配，从而提升 RAG 召回效果。

### 核心回答
- **关键词检索 BM25**：适合编号、术语、国家、币种、产品名等精确匹配。
- **向量检索**：适合同义表达、口语问题、语义相似问题。
- **混合方式**：
  - 两路召回后合并。
  - 分数归一化加权。
  - RRF 融合排名。
  - 再用 reranker 重排。
  - 加元数据过滤，如国家、业务线、文档类型。

混合检索能减少单一路径的盲区。

### 1. 为什么单纯向量检索不够

向量检索擅长语义相似，但对精确词不一定稳。比如错误码、国家码、币种、产品名、通道名，这些需要精确匹配。BM25 擅长关键词匹配，但对口语表达和同义改写不够好。

混合检索把两者结合：

- BM25 保证专有名词和错误码命中。
- 向量检索覆盖语义相似表达。
- reranker 负责最终相关性排序。

### 2. 融合方式

常见做法：

- 两路召回后去重。
- 用 RRF 做排名融合。
- 按业务权重调整分数。
- 使用元数据过滤缩小范围。
- 最后用 cross-encoder rerank。

### 3. 面试总结

混合检索能同时覆盖"精确词"和"语义相似"，是生产 RAG 里非常常用的方案。

### 4. 结合跨境支付项目怎么说（面试口径）

> 支付 RAG 里我一定会用混合检索。因为很多查询必须精确命中，比如 SWIFT、SEPA、PIX、USD、BRL、国家代码、错误码，这些向量检索不一定稳。
>
> 但用户描述失败原因又很口语，比如"款没到""银行没回""是不是被风控拦了"，这类问题向量检索更合适。所以我会 BM25 和向量两路召回，再按国家、币种、通道过滤，最后 rerank。
>
> 这样比单纯向量库可靠得多，尤其适合支付这种专有名词多、规则精确度要求高的场景。

**钩子**：支付文档有大量专有名词和错误码，混合检索比纯向量更稳。
