# vLLM部署中如何实现2k tokens/s的吞吐?

## 答案

“2k tokens/s”必须先定义口径：是单请求生成速度、服务器聚合 output tokens/s，还是包含 Prefill 的 total tokens/s。还必须说明模型、精度、GPU、TP 数、输入输出长度和并发，否则这个数字不可复现。

### 1. 先建立基线

记录：

```text
Model/Revision
Quantization/Dtype
GPU x Count
Tensor Parallel
Prompt Length Distribution
Output Length Distribution
Concurrency
TTFT / TPOT / Output tokens/s
```

使用 vLLM 官方 benchmark serving/throughput 工具，并用真实流量分布，不只测固定短 Prompt。

### 2. 选择合适模型和精度

- 模型越小，吞吐通常越高；
- BF16/FP16 是基线；
- 硬件支持时评估 FP8；
- INT8/INT4 weight-only 可省显存，但是否提速取决于 kernel；
- GQA/MLA 和 KV 量化可提高并发；
- 限制无必要的长 thinking 和输出。

质量回归必须覆盖数字、JSON、工具调用和多语言。

### 3. vLLM 调度参数

重点调：

- `--gpu-memory-utilization`：为 KV Cache 分配更多显存，但保留 OOM 余量；
- `--max-num-batched-tokens`：每轮调度 token 预算；
- `--max-num-seqs`：并发序列上限；
- `--max-model-len`：不要无条件开最大窗口；
- chunked prefill：减少长 Prompt 阻塞；
- prefix caching：复用 system prompt/工具定义；
- scheduler/async output 等版本支持能力。

参数要用网格压测，不能照抄固定值。

### 4. 提高有效 Batch

聚合吞吐主要来自 continuous batching：

- 让足够并发请求填满 GPU；
- 按输入/输出长度分桶；
- 长请求单独队列；
- 避免一个超长 Prefill 挤压所有 Decode；
- 控制最大输出，及时回收完成序列；
- 上游连接池和 SSE 不成为瓶颈。

单请求 TPOT 和聚合 throughput 是不同目标，提高 batch 可能增加排队延迟。

### 5. 并行与硬件

- 单卡能放下时先测单卡，避免 TP 通信；
- 放不下或矩阵足够大时再用 Tensor Parallel；
- 多副本 Data Parallel 扩总吞吐；
- MoE 关注 Expert Parallel；
- 多机检查 NVLink/NVSwitch/IB 和 NCCL；
- GPU 利用率低时用 profiler 判断 CPU、通信或采样瓶颈。

### 6. Kernel 和缓存

- FlashAttention/FlashInfer；
- CUDA Graph；
- fused sampling/RMSNorm/RoPE；
- Paged KV Cache；
- prefix cache；
- speculative decoding 对低并发延迟更有利，未必提升高 batch 总吞吐。

### 7. 达标方法

逐步实验：

```text
Baseline
-> 提高并发/continuous batching
-> 调 batched tokens 与 max seqs
-> prefix cache/chunked prefill
-> 量化或更小模型
-> 并行和多副本
-> kernel/profile
```

每一步报告质量、TTFT、TPOT、P95、吞吐、显存和错误率。若目标硬件和模型理论上达不到 2k output tokens/s，应扩容或换模型，而不是修改统计口径。

### 8. 跨境支付

公共 system prompt 和工具定义可做 prefix cache，但 cache key 必须包含权限、租户、模型和模板版本，防止跨租户复用敏感上下文。
