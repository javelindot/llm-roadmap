# Attention 优化方向

## 答案

**一句话**：Attention 优化的痛点是**O(n²) 复杂度 + 内存带宽瓶颈 + KV Cache 显存**，对应三大方向——**算子优化（Flash-Attn）、结构优化（MQA/GQA/Sliding Window）、稀疏化（Sparse/Linear/Local）**。

### 1. 算子级优化（IO-aware）
- **Flash-Attention v1/v2/v3**
  - 不存中间 attention 矩阵
  - tiling + softmax 在线归一化
  - HBM 读写减少 ~5x
  - v2 提升并行度，v3 用 H100 FP8
- **Paged Attention（vLLM）**
  - KV Cache 按 page 管理（类虚拟内存）
  - 减少显存碎片
  - 支持高 batch 并发

### 2. 结构级优化（头数）
| 方法 | KV 头数 | 显存 | 质量 |
|------|---------|------|------|
| **MHA**（多头） | = Q 头数 | 大 | 高 |
| **MQA**（多 Q 共 KV） | 1 | 极小 | 略降 |
| **GQA**（分组共享） | 1 < g < 头数 | 中 | 接近 MHA |
| **MLA**（DeepSeek-V2/V3） | 低秩压缩 | 极小 | 高 |

业界主流：**Llama-3/Qwen-3 用 GQA**，DeepSeek 用 MLA。

### 3. 稀疏化方向
- **Sliding Window Attention**（Mistral）：只看最近 W 个 token
- **Sparse Attention**（Longformer / BigBird）：固定 + 全局 + 滑窗稀疏
- **Local + Global**：Streaming-LLM 思路
- **Linformer / Performer**：把 O(n²) 压成 O(n)，质量损失

### 4. 长上下文专用
- **Ring Attention**：跨设备循环算注意力，省单卡显存
- **Sequence Parallel**：序列维度切分
- **Sink Token + Sliding Window**：StreamingLLM
- **KV Cache 压缩**：H2O、Scissorhands 丢弃低重要 KV

### 5. 推理侧专门优化
- **Continuous Batching**（vLLM）：批级动态调度
- **Speculative Decoding**：小模型先猜大模型校验
- **Multi-Query / Group-Query**：推理省 KV Cache
- **Prefill chunking**：长 prompt 分块算
- **Prompt Cache**：复用 KV Cache

### 6. 选型经验
- **训练优化首选 Flash-Attention 2/3**：免费 5x 加速
- **模型结构默认 GQA**：质量基本不损，KV 显存省 4x
- **长上下文**：YaRN（位置） + Flash-Attn + KV 压缩 + Ring Attention
- **高并发推理**：vLLM + GQA + speculative decoding

### 7. 一句话总结
**Attention 优化 = IO 优化（Flash） + 结构优化（GQA/MLA） + 稀疏化（SWA/Linear）**——工程上 Flash + GQA 是免费午餐，必选。

## 结合项目谈（口语版 · 约 40 秒）

> 我们跨境支付推理侧的 Attention 优化基本是"能开都开"，分享几个落地经验。
>
> **第一块是训练侧 Flash-Attention 2**。我们 Qwen-14B SFT 一开始没装 FA2，单步 3.2 秒，开了之后 1.5 秒——**直接快了 2 倍多，显存还省**，这是真正免费的午餐。后来又升级 FA3（H100 支持 FP8 attention），单步又压到 1.1 秒。
>
> **第二块是推理侧 vLLM + Paged Attention + GQA**。客服对话场景 QPS 高，朴素 HuggingFace generate 单机能跑 5 QPS、上 vLLM 能跑 80 QPS——单 Paged Attention 这一项就让显存碎片化降下来很多，配合 continuous batching 把吞吐拉到极致。
>
> **第三块是 Sliding Window 的取舍**。我们风控 Agent 有时上下文很长（30k+），原本想试 Mistral 的 SWA，但 SWA 对"前文重要证据要被忆起"这种场景不友好，容易丢早期信息。最后还是用 RoPE + YaRN + Flash-Attn 这套，没上 SWA。
>
> **第四块是 KV Cache 压缩**——我们试过 H2O 把长上下文 KV 砍掉 50%，质量基本不掉、延迟下降明显，长 Agent 链路用得上。
>
> 我对 Attention 优化的总结：**Flash + GQA 是基线必开，Paged Attention + Continuous Batching 是推理必上，长上下文加 KV 压缩**——剩下的稀疏化方案看业务。
