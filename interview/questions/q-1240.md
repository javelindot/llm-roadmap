# DPO 除了长度问题还有其他的问题嘛，与问题2对应， reward hacking ？都没有奖励模型了

## 答案

DPO 没有单独训练显式 Reward Model，但不代表没有 reward。它使用 policy 与 reference 的 log-ratio 定义隐式 reward：

```text
r_theta(x, y) =
    beta * (log pi_theta(y | x) - log pi_ref(y | x))
```

训练通过偏好对让 chosen 的隐式 reward 高于 rejected。因此仍可能出现广义的 reward hacking：模型利用偏好数据或评审流程中的代理特征提高目标，而不是真正提升用户关心的质量。

### 1. 长度和风格捷径

如果 chosen 普遍更长、更礼貌或有固定模板，模型会学习这些表面特征。离线 loss 和 pair accuracy 上升，但事实性未必提高。

### 2. 偏好标签噪声

人工标注会分歧，AI feedback 会有位置、长度和自偏好。DPO 把每个 pair 当作确定偏好，矛盾样本会产生不稳定梯度，低质量 rejected 还会让任务过于简单。

### 3. chosen likelihood 下降

DPO 只优化 chosen 与 rejected 的相对 margin，不保证 chosen 绝对概率上升。二者可能同时下降，只是 rejected 下降更多。可监控平均 token log-prob，并考虑 reward anchor 或 SFT/NLL 辅助项。

### 4. Reference model 敏感

隐式 reward 依赖 reference：

- reference 太弱，偏好差异可能失真；
- reference 与 policy/chat template 不一致会破坏概率比；
- beta 不合适会导致更新过弱或过度漂移；
- base revision、tokenizer 和 mask 错误会造成隐蔽问题。

### 5. 离线分布限制

DPO 只看到固定 chosen/rejected，无法自动探索数据中没有的更好答案，也难处理长程 Agent 的 credit assignment。训练策略分布变化后，原偏好对可能不再代表当前常见输出。

### 6. 多目标冲突

helpfulness、safety、简洁、事实性和风格可能互相冲突。把它们压成一个二元标签会丢失原因，导致过度拒答、模板化或某些语言退化。

### 7. 过拟合与通用能力回退

数据量小、epoch 多或学习率高时，模型可能记住偏好格式，损害知识、多语言、代码和长上下文能力。Train margin 继续扩大并不代表泛化继续改善。

### 8. 缓解方式

- 构造长度与风格匹配的 hard pair；
- 保存偏好维度、置信度和标注者分歧；
- 过滤矛盾与重复数据；
- 使用高质量 SFT reference；
- 调整 beta、学习率和 epoch；
- 混合 SFT loss 或通用 replay data；
- 监控 chosen/rejected log-prob、KL、长度和分桶指标；
- 在线采样新策略输出并重新标注；
- 对可验证事实使用程序规则，而不是偏好判断。

跨境支付中，“回答更受欢迎”不能覆盖错误金额、虚构到账时间或越权操作。安全和资金规则应作为硬门禁，DPO 仅优化门禁内的表达与行为排序。
