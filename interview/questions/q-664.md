# 介绍agent相关项目经验

## 答案

介绍 Agent 项目不能只说使用了 LangChain 或调用了大模型，应围绕任务闭环、状态管理、工具可靠性和评测展开。下面是可替换的回答结构。

### 项目概述

先说明用户、场景和为什么需要 Agent：

> 项目面向某类用户，目标是完成跨数据源查询、分析并生成结果。普通单轮 RAG 无法处理任务拆解和多次工具调用，因此采用有状态工作流。

然后说明自己的职责边界，例如工作流设计、工具接口、检索、服务端或评测，不要把团队成果全部归为个人。

### 技术架构

典型链路包括意图识别、计划、工具选择、执行、结果校验和最终回答。使用显式状态图管理分支与循环，将会话和中间结果持久化到数据库或 checkpoint。工具使用 schema 约束参数，并配置权限、超时、重试、幂等和熔断。知识问答节点使用 hybrid retrieval 与 reranker，答案绑定引用。

对高风险操作增加人工确认；设置最大步骤、token 和费用预算，避免无限循环。模型不可用时可降级到固定工作流或传统搜索。

### 难点与评测

选择一个真实难点深入，例如工具选错、长任务恢复、检索证据不足或 prompt injection。说明如何通过 trace 和失败样本分类定位，并采用候选工具检索、状态约束或权限隔离修复。

评测至少包括任务完成率、工具选择准确率、参数正确率、引用正确率、平均步骤、P95 延迟和成本；上线后从 bad case 回流构建回归集。最终给出真实结果及局限。没有 Agent 项目经验时，可以介绍设计方案和做过的相关 RAG/工作流经验，但不要虚构上线规模。
