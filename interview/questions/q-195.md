# 了解 SKILLS 吗？

## 答案

> 了解，我们跨境支付里也落地了。简单说一下我的理解和实战。
>
> **是什么**：
> SKILL 是 Claude Code（Anthropic）提出的"**可发现的 Agent 能力包**"——本质就是一个文件夹，里面 `SKILL.md` 写清楚：什么场景用、查哪些字段、调哪些工具、什么时候转人工。
>
> **核心机制是渐进式加载**：
> - 启动时只把 SKILL.md 的 frontmatter（name + description）注入系统提示
> - 模型看到的是"能力清单"，正文不加载
> - 命中时才把正文加载进来，省 token
>
> **和工具、Subagent 的区别**：
> - 工具 = 单步动作（"做一件事"）
> - **Skill = 多步流程包**（"做一类事的 SOP"）
> - Subagent = 独立上下文的子任务（"隔离地干一件事"）
>
> 三者在生产里**配合使用**——Skill 编排工具、Subagent 隔离长流程上下文。
>
> **我们的实战**：
>
> 我把跨境支付的几个"流程性强、规则又老变"的场景做成了 Skill：
>
> - **"可疑交易初筛 Skill"**：风控工单进来 Agent 自动命中，描述如何查链路、调哪些只读接口、什么时候转人工
> - **"商户准入 KYB 初审 Skill"**：商户上传证件后调用，描述如何提取字段、校验、风险标签打分
> - **"清算对账差异定位 Skill"**：对账失败时 Agent 调用，按 SOP 排查
>
> **关键收益**：
> - token 省 70%（相比所有规则塞系统提示）
> - 业务侧改规则只改 md 文件 + PR review，不动主 prompt
> - 运营自己都能改 SOP
> - 权限通过 `allowed-tools` 做最小化
>
> **踩过的坑**：
> - description 写得太像 → 模型误触发，加 "description 评测集" 治理
> - Skill 之间互相调用容易死循环 → 加调用栈深度限制
>
> 我对 Skill 的看法：**它是把"团队隐性流程经验工程化"的最小单元**，比 RAG 更结构化、比 fine-tune 更轻量。是 Agent 工程化的下一个增长点。

**🪝 顺嘴带出去的钩子**：
- "Skill 误触发的治理方法可以聊。"
- "Skill 和 Subagent 的协作链路设计可以展开。"
- "Skill 权限准入流程怎么做可以讲讲。"
