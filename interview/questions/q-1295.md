# 项目有做过流量压测吗？单服务还是多服务部署？

## 答案

项目做过分层流量压测，生产采用多服务部署，不把网关、Agent、检索、工具和模型全部塞进一个进程。PoC 可以单服务快速验证，但跨境支付生产场景需要独立扩缩容、故障隔离、权限边界和可观测性。

### 1. 为什么采用多服务

典型链路：

```text
API Gateway
-> Conversation / Agent Service
-> Retrieval + Rerank Service
-> Tool Gateway
-> Model Gateway
-> vLLM Model Serving
```

辅助服务包括：

- Redis：会话、限流、短期缓存；
- Milvus/Elasticsearch：向量与关键词检索；
- Kafka/任务队列：离线解析、索引和评测；
- Prometheus/OpenTelemetry：指标和 trace；
- 对象存储：文档、模型和评测产物。

拆分原因：

- LLM 服务依赖 GPU，检索和业务工具主要消耗 CPU、网络和数据库；
- 模型可按 GPU 容量扩容，Agent 服务可按 QPS 扩容；
- 检索故障不应拖垮所有模型副本；
- Tool Gateway 需要更严格的认证、审计和超时；
- 实时客服与离线文档任务需要资源隔离；
- 不同服务可以独立灰度和回滚。

### 2. 分层压测

**模型服务基准**

使用 vLLM benchmark 或自定义脚本，构造不同：

- 输入长度、输出长度；
- 并发数、请求速率；
- batch、量化和模型副本；
- prefill-heavy 与 decode-heavy 流量。

记录 TTFT、TPOT、tokens/s、P95/P99、队列长度、GPU 利用率、显存和 OOM。

**RAG 服务**

- 文档过滤、BM25/Dense、rerank 各阶段耗时；
- Top-K、候选长度和并发；
- 向量库冷/热缓存；
- 索引更新期间的查询性能。

**Tool 服务**

- 订单、账务、通道和风控接口的 P95/P99；
- 连接池、超时、重试和熔断；
- Agent 放大下游请求的倍数；
- 依赖限流和故障注入。

**端到端压测**

用脱敏真实工单回放：

- 高频 FAQ；
- 单工具订单查询；
- 多工具付款异常；
- 长文档问答；
- 工具超时和转人工。

### 3. 负载模型

先用 Little's Law 做容量起点：

```text
concurrency ≈ arrival_rate * average_latency
```

如果目标峰值为 `R` requests/s，平均每请求触发 `k` 次工具调用，则下游理论请求率约为：

```text
tool_qps ≈ R * k
```

还要加入重试、突发和副本故障余量。不能只压模型 QPS，而忽略 Agent 把一次用户请求放大为多次检索和工具调用。

### 4. 压测曲线

按阶梯逐步增加负载：

```text
warm-up
-> steady load
-> step load
-> target peak
-> burst
-> soak test
-> dependency failure
```

目标是找到：

- latency 开始非线性上升的饱和点；
- 最大稳定吞吐；
- 单副本故障后的剩余容量；
- 限流和降级是否生效；
- 长时间运行是否有显存、连接或缓存泄漏。

### 5. 生产验收

除性能外同时验证：

- 业务正确率、工具成功率和引用；
- 高风险问题转人工；
- 跨租户数据隔离；
- 超时不编造订单结果；
- 写操作不重复执行；
- 扩缩容期间无请求雪崩；
- 模型、检索或工具失败时可降级。

没有真实压测报告支撑时不报固定 QPS 提升数字，应说明负载分布、硬件、模型版本、SLO、瓶颈和容量余量，否则数字不可比较。
