# 为什么大模型需要外挂（向量）知识库？

## 答案

**一句话**：LLM 自身有**知识陈旧、私域缺失、幻觉风险、上下文有限、可追溯弱** 五大短板——外挂向量知识库（RAG）是工业落地里最便宜、最可控的补救方案。

### 1. 五大根本原因
**1) 知识陈旧**
- 训练数据有截止时间
- 业务规则、产品信息、行情每天变
- 重新训练 base 成本几十万到百万

**2) 私域知识缺失**
- 公开训练集没有公司内部数据
- 客户信息、内部 SOP、产品手册都是私域
- 不可能把所有私域塞进 prompt

**3) 幻觉风险**
- LLM 不知道就编
- 业务上"编错"=生产事故
- 外挂知识库让回答必须"有据可查"

**4) 上下文有限**
- 即使 128k 也装不下整个企业知识库
- 实时筛选相关内容比一次性塞强得多

**5) 可追溯性**
- 监管要求"为什么这么答"
- RAG 自然带 citation
- LLM 单跑没有来源

### 2. RAG vs 微调 vs Long-Context（三选一）
| 方案 | 知识更新 | 成本 | 可追溯 | 上限 |
|------|---------|------|--------|------|
| **微调** | 慢、贵 | 高 | 弱 | 高 |
| **Long-Context** | 即时 | 中（推理贵） | 中 | 受窗口限制 |
| **RAG** | 极快（更新文档即可） | 低 | 强 | 高 |

工业 80% 场景选 RAG。

### 3. 向量库 vs 倒排索引
- **向量库**（Faiss/Milvus/Qdrant）：捕捉语义
- **倒排（ES/BM25）**：捕捉关键词精确匹配
- 实际生产是 **混合检索（hybrid search）**：vector + keyword + 重排

### 4. RAG 落地的三层防线
- **第一层（索引）**：高质量切块 + embedding + 元数据
- **第二层（检索）**：multi-query + 混合检索 + rerank
- **第三层（生成）**：strict 引用 + 拒绝 fallback + 评测监控

### 5. RAG 的局限（也要诚实讲）
- 检索质量决定生成质量上限
- 长文档分块策略难
- 跨文档推理弱
- 实时性还是有几百毫秒延迟
- 多语言/术语不一致难处理

### 6. 一句话总结
**RAG = 给 LLM 装一个"开卷考试"系统**——便宜、可控、可追溯，是工业 LLM 落地的事实标准。

## 结合项目谈（口语版 · 约 30 秒）

> 我们跨境支付**客服 RAG** 是整个 AI 系统的基础，分享下我们落地的逻辑链。
>
> 为什么不微调？早期我们试过把客服知识库全微调进去——结果三个问题：一是**业务规则一周变三次**，微调一次几天，跟不上；二是**模型仍然会编**，把客户问题答成"和真实政策很像但又有差异"的回答；三是**合规要求引用**，微调后的模型没有 citation 能力。
>
> 为什么不 Long-Context？我们试过把整个知识库 50w 字塞 128k 上下文——推理成本翻 20 倍、首 token 延迟 4 秒、不相关内容稀释了关键信息，效果反而比 RAG 差。
>
> RAG 是怎么落地的？
> - 知识库 5w 条 FAQ + 1w 条政策文档
> - bge-large + 业务精调
> - 混合检索：向量 top-20 + BM25 top-20 → 重排 top-5
> - 生成时强制要求"答案必须来自检索结果，否则说不知道"
> - 每个答案附带 citation
>
> 上线后客服查询准确率 88%、平均响应 1.2 秒、客户投诉降了 40%。
>
> 一句话：**LLM 是大脑、RAG 是它的开卷资料**——业务知识必须外挂，这是工业 LLM 的铁律。
