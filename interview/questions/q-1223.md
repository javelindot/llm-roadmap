# 介绍下KV Cache，GQA，MQA，MHA

## 答案

KV Cache 是自回归推理的缓存机制，MHA、MQA 和 GQA 则决定 Query head 与 Key/Value head 如何组织。三者直接影响模型表达能力、KV Cache 大小、显存带宽和解码吞吐。

### 1. KV Cache

生成第 $t$ 个 token 时，当前 Query 需要与此前所有 token 的 Key、Value 做注意力。历史 token 的 K/V 在每一步都相同，因此首轮 prefill 后把它们按层缓存，后续 decode 只计算新 token 的 Q/K/V，并将新 K/V 追加到缓存。

缓存的是 **K 和 V，不是 Q**。忽略对齐和框架开销，KV Cache 大小近似为：

```text
M_KV ≈ 2 * B * L * T * H_kv * D_h * S
```

其中：

- `2` 表示 K 和 V；
- `B` 是并发 batch；
- `L` 是层数；
- `T` 是已缓存序列长度；
- `H_kv` 是 KV head 数；
- `D_h` 是每个 head 的维度；
- `S` 是每个元素字节数。

所以并发和上下文长度增长时，KV Cache 可能比模型权重更先耗尽显存。

### 2. MHA

Multi-Head Attention 中每个 Query head 都有独立的 K/V head：

```text
Hq = Hkv
```

不同 head 可以学习不同关系，表达能力强，也是原始 Transformer 的标准结构。但 K/V 投影和缓存数量最多，自回归 decode 的显存带宽压力较大。

### 3. MQA

Multi-Query Attention 保留多个 Query head，但所有 Query head 共享一组 K/V：

```text
Hq > 1, Hkv = 1
```

KV Cache 相比同 Query head 数的 MHA 可大幅缩小，读取 K/V 的带宽也更低，因此解码吞吐更高。代价是所有 Query head 共用 K/V 表示，容量压缩过强时可能影响质量。

### 4. GQA

Grouped-Query Attention 是 MHA 与 MQA 的折中：多个 Query head 分组，每组共享一个 K/V head。

```text
1 < Hkv < Hq
```

例如 32 个 Query head、8 个 KV head，表示每 4 个 Query head 共用一组 K/V。它通常能保留接近 MHA 的质量，同时显著降低 KV Cache 和内存带宽，因此现代大模型中非常常见。

| 结构 | Query head | KV head | KV Cache | 典型取舍 |
|---|---:|---:|---:|---|
| MHA | `H_q` | `H_q` | 最大 | 表达能力强、解码成本高 |
| GQA | `H_q` | 若干组 | 中等 | 质量与吞吐折中 |
| MQA | `H_q` | 1 | 最小 | 吞吐高、共享最强 |

MHA 和 MQA 都可以看作 GQA 的边界情况：每组一个 Query head 时是 MHA，所有 Query head 一组时是 MQA。

### 5. Prefill 与 Decode 的瓶颈

- **Prefill**：一次处理完整 Prompt，矩阵计算密集，更容易受算力限制；
- **Decode**：每步通常只生成一个 token，需要反复读取权重和 KV Cache，更容易受显存带宽限制；
- **长上下文高并发**：KV Cache 分配、碎片和调度成为核心问题。

因此服务端还会配合：

- PagedAttention 或块式 KV 管理，减少碎片；
- continuous batching，提高动态请求吞吐；
- KV Cache FP8/INT8 量化；
- prefix caching，共享相同系统 Prompt 或文档前缀；
- sliding window、分层缓存或上下文压缩；
- tensor parallel 下按 KV head 合理切分。

优化后必须同时回归精度、TTFT、TPOT、P95/P99、吞吐和峰值显存，不能只看单请求 tokens/s。

### 6. 跨境支付部署

跨境支付客服在活动、结算日或通道异常时会出现突发并发，同时系统 Prompt、合规条款和历史工单可能很长。GQA 能降低单会话 KV 成本，prefix cache 可复用固定合规前缀；但交易详情应按需检索和工具查询，不能把所有商户数据长期塞进上下文。缓存还要按租户隔离并及时清理，避免跨商户信息泄露。
