# Agent 如何使用工具、SKILL

## 答案

**一句话**：工具是**单步动作**（"做一件事"），Skill 是**多步流程包**（"怎么做一类事"），Agent 用工具靠 function calling 协议、用 Skill 靠渐进式加载。

### 1. Agent 使用工具的循环（ReAct）
```
思考 → 选工具 (tool_use) → 执行 → 拿到 tool_result → 继续思考 → ...
```
- 模型看到的是 **工具清单 + 描述**，不会看到代码。
- 选工具靠 description 匹配，所以 description 比代码还重要。
- 工具结果回填后模型才知道下一步。

### 2. Agent 使用 Skill 的流程（渐进披露）
- **启动阶段**：只加载所有 Skill 的 frontmatter（name + description），几十行。
- **命中阶段**：用户 `/skill-name` 或模型判断匹配 → 调用 `Skill` 工具 → 加载该 SKILL.md 正文。
- **执行阶段**：模型按 SKILL.md 的 SOP 一步步调用底层工具（Bash、Read、API 调用……）。
- **资源引用**：SKILL.md 里可以 `@reference` 子文件，模型按需打开。

### 3. 工具 vs Skill 怎么选
| 维度 | 工具 | Skill |
|------|------|-------|
| 粒度 | 单步动作 | 多步流程 |
| 内容 | 函数 + Schema | Markdown SOP + 可引用脚本 |
| 加载 | 启动就在系统提示 | 命中时才加载正文 |
| 维护 | 改代码、发版 | 改 md、走 PR |
| 谁写 | 工程师 | 业务运营也能写 |

### 4. 组合关系（真实生产里）
- **Skill 编排工具**：一个 Skill 内调用 N 个工具，工具是手，Skill 是脑。
- **Subagent 隔离上下文**：流程太长就丢 Subagent，主 Agent 上下文不被污染。
- **MCP 提供工具源**：第三方系统的工具走 MCP，Skill/Agent 透明使用。

### 5. 一句话总结
**工具回答"我能做什么"，Skill 回答"做这件事我该怎么走流程"**。Agent 用工具是反射，用 Skill 是按手册办事。

## 结合项目谈（口语版 · 约 40 秒）

> 这块我们跨境支付落地下来感受挺深的。一开始全堆工具——查链路、查商户、查黑名单、查规则……二十多个工具丢给 Agent，结果发现两个问题：一是 Agent 在简单工单上调来调去绕远路，二是新人想加一个"商户准入初审"的流程，发现得改 prompt、改代码、还得发版，运营一句"我要改规则"我们就头疼一周。
>
> 后来我们把"流程性的活儿"沉到 Skill 里——比如"可疑交易初筛""商户准入初审""清算差异定位"，这些都是有固定 SOP 的。工具就只保留原子操作，查数据、调接口这种。Agent 拿到工单后先匹配 Skill，Skill 里告诉它走哪几步、调哪些工具、什么时候转人工。
>
> 这么拆完之后最大的好处是边界清晰——工程师维护工具集，运营维护 Skill。运营自己改 SOP 提 PR 就行，不用排我们排期。token 也省，因为 Skill 平时不加载正文，只在命中时展开。
>
> 现在我们工具大概 30 个、Skill 大概 15 个，配合 Subagent 做长流程隔离，整个风控 Agent 链路跑得挺稳。

**🪝 顺嘴带出去的钩子**：
- "工具和 Skill 边界怎么划其实也踩过坑，我们有几条经验法则可以聊。"
- "运营自己改 Skill 我们怎么做评测和准入审计，这块也搭了一套。"
- "Skill + Subagent 组合在长流程里的拆分逻辑可以展开。"
