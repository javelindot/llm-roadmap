# 机器学习-简述验证集和测试集的区别

## 答案

**一句话**：验证集用于训练过程中选模型和调参，测试集用于最终评估模型泛化能力，测试集不能参与调参。

### 核心回答
- **训练集**：用于拟合模型参数。
- **验证集**：用于调超参、选模型、早停、做方案对比。
- **测试集**：用于最终报告效果，模拟模型面对未见数据的表现。

区别：
- 验证集可以被反复查看，因此会被间接过拟合。
- 测试集应该尽量只在最终使用，保证评估客观。

如果数据少，可以用交叉验证，但最终仍最好保留独立测试集。

### 1. 为什么验证集不能当测试集

验证集会被反复用于调参、选模型、改特征、改 prompt 或改检索策略。虽然模型没有直接用验证集训练参数，但人会根据验证集结果不断调整方案，所以验证集会被"间接过拟合"。

测试集应该模拟最终上线后遇到的新数据，用来回答一个问题：这个方案在没见过的数据上到底怎么样。

### 2. 数据切分要避免泄漏

实际项目里，随机切分不一定可靠。常见泄漏包括：

- 同一个用户或商户出现在训练集和测试集。
- 同一事件的多条记录被拆到不同集合。
- 未来数据特征被用于训练。
- 文档问答里答案原文泄漏到 prompt 或训练数据。

### 3. 面试总结

验证集用于开发阶段决策，测试集用于最终客观评估。高风险业务里，切分方式比指标本身还重要。

### 4. 结合跨境支付项目怎么说（面试口径）

> 在支付风控里，验证集和测试集划分要特别小心，不能随机一切就完事。比如同一个商户的多笔交易同时出现在训练和测试里，就会数据泄漏，效果看起来很高，但线上遇到新商户就不行。
>
> 我会按时间、商户、国家或通道做分层切分。验证集用于调模型和调阈值，测试集尽量模拟未来月份的数据，尤其要覆盖新商户、新通道、新国家和新风险模式。
>
> 对 RAG 项目也一样，不能让同一批问题的标准答案泄漏到开发过程里。否则离线指标很好看，上线后业务方一问新规则就答错。

**钩子**：支付场景最怕数据泄漏，测试集最好按时间或主体隔离。
