# 介绍检索做的优化,具体追问子问题分解怎么做,有没有做意图识别

## 答案

检索优化应先建立可诊断基线，再从查询理解、索引、召回、重排和上下文构造逐层优化。完整链路是：

> 意图识别 → 实体与约束抽取 → 子问题分解 → 分路检索/工具查询 → 融合重排 → 证据充分性检查。

### 1. 意图识别决定去哪里找答案

跨境支付问题先按业务动作分类：

- 全球收款、付款、换汇、收单、退款、拒付；
- KYC/KYB、合规、风控、通道和清结算；
- 知识规则查询、实时交易查询、操作请求或投诉。

采用分层路由：

1. 错误码、币种和标准命令优先走规则；
2. 高频单意图使用小模型或 embedding classifier；
3. 长尾、多意图和多轮问题由 LLM 输出结构化结果；
4. 低置信度允许 `unknown`，追问或转人工。

路由结果包含：

> `intents、entities、knowledge_bases、tools、risk_level、confidence`

分类器不直接决定资金动作，只决定后续检索和工具范围。

### 2. 子问题分解不是把一句话随意拆开

适合分解的情况包括：

- 用户同时提出多个独立任务；
- 答案依赖多个数据源；
- 问题存在明确的多跳依赖；
- 一个查询包含规则和实时状态。

例如：

> “这笔 EUR 付款为什么失败，重新发起需要什么材料？”

可以分为：

1. 查询订单当前状态；
2. 查询通道回执和失败码；
3. 检索该国家、币种和通道的失败码说明；
4. 根据失败原因检索补充材料和重试规则；
5. 汇总证据并判断能否重试。

子问题表示为带依赖关系的 DAG，而不是全部并行。订单状态和通道回执可以并行，重试规则必须等待失败原因明确后再检索。

每个子问题保留 `subtask_id、depends_on、source、filters、expected_output`，防止结果混淆。

### 3. 查询改写与约束抽取

查询侧主要优化：

- 多轮指代消解；
- 支付术语和缩写归一化；
- 拼写、OCR 和错误码格式纠正；
- 中英文及本地语言扩展；
- 国家、币种、产品、通道、时间和文档版本抽取；
- 为不同召回器生成不同查询。

BM25 查询保留 SWIFT、SEPA、PIX、MCC、错误码等精确词；向量查询可以改写成自然语言语义。HyDE 和多查询扩展只用于语义缺口明显的复杂问题，精确错误码问题不使用，避免引入虚构术语。

### 4. 索引和召回优化

索引侧采用：

- 文档结构与业务语义分块；
- parent-child chunk；
- 标题、章节路径和适用范围上下文化；
- 国家、币种、产品、通道、权限、版本和有效期元数据；
- BM25 与向量双索引；
- 稳定 chunk ID 和增量更新机制。

召回侧采用：

- BM25 处理专有名词和错误码；
- dense retrieval 处理口语和跨语言表达；
- FAQ/案例处理高频问题；
- 图或 SQL 处理结构化关系；
- metadata filter 前置缩小候选；
- RRF 融合并统计各路独占贡献。

召回通道不是越多越好。没有独占正确结果、只增加重合候选和延迟的通道应删除。

### 5. 重排与证据覆盖

使用 cross-encoder 对候选精排，并加入：

- 来源权威性和文档生效时间；
- 国家、币种和产品匹配；
- 否定词及限制条件；
- 多个子问题的证据覆盖；
- 权限和敏感等级。

复杂问题不能只选相似度最高的几个 chunk。可以使用 set-cover 思路，在 token 预算内覆盖不同子问题和必要条件，同时去掉重复内容。

### 6. 生成前做充分性判断

检索完成后判断：

- 每个子问题是否都有证据；
- 工具状态与文档规则是否一致；
- 是否混入其他国家、币种或历史版本；
- 是否需要用户补充订单号、商户或材料；
- 是否存在不可公开的风控信息。

证据不足时不进入自由生成，直接追问、返回已确认部分或转人工。

### 7. 评测与消融

- 意图识别：Macro-F1、多意图召回、错误路由率；
- 子问题分解：分解完整率、依赖正确率、冗余子任务率；
- 召回：Recall@K、MRR、独占贡献；
- 重排：NDCG、Context Precision、关键证据保留率；
- 端到端：任务成功率、引用正确率、人工改写率和转人工率；
- 系统：P95、工具调用数、token 和单次成本。

优化时一次只改变一个主要变量并做分桶消融。若正确证据根本没召回，应修查询和索引；若已召回但排序靠后，应修重排；若证据完整仍答错，才处理生成和 Prompt。
