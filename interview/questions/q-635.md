# 大模型如何根据场景调用不同的工具类

## 答案

大模型调用工具通常采用 function calling：把每个工具的名称、用途和结构化参数 schema 提供给模型，模型根据当前意图输出工具名与参数，由宿主程序校验并执行，再把结果返回给模型继续推理。

### 工具定义

工具描述要区分清楚适用边界，参数使用 JSON Schema 等强类型约束。例如天气工具需要城市和日期，数据库查询工具只能接受受控查询条件。相似工具如果描述重叠，模型容易选错，应合并、重命名或增加反例。

### 选择流程

1. 根据用户请求、会话状态和权限，先由代码筛出当前允许的工具集合。
2. 将候选工具 schema 与上下文交给模型。
3. 模型输出 `tool_name` 和参数，而不是自由文本调用代码。
4. 服务端做 schema、权限、业务规则和风险校验。
5. 执行工具，将结构化结果和错误信息返回模型。
6. 模型决定继续调用、换工具、追问缺失参数或生成最终答案。

工具数量很大时，不应把全部 schema 塞入上下文。可以先用规则、分类器或 embedding 检索出少量候选，再让 LLM 精选；还可以采用领域路由器，将请求分配给搜索、办公、数据等子 Agent。

### 可靠性设计

- 给每个工具设置超时、重试、幂等键、并发限制和熔断。
- 对删除、支付、发信等有副作用的操作要求人工确认，模型不能直接执行。
- 参数值使用白名单或业务校验，防止 prompt injection 越权调用。
- 限制最大工具调用次数和 token/费用预算，检测循环。
- 记录选择理由、参数、结果、延迟和错误，便于回放评测。

评估不能只看最终回答，还应分别统计工具选择准确率、参数正确率、任务完成率和不必要调用率。对于规则明确的场景，优先由代码路由；只有语义判断复杂时才让 LLM 做选择。
