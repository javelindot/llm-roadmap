# 如何扩展闭源的benchmark，或者如何从大量的指令数据里面提取能力项，构建扩展已有的benchmark

## 答案

扩展闭源 Benchmark 不应反向猜测或复刻私有题目，而应基于其公开能力定义和真实业务失败，建立独立、可审计的新测试集。

### 1. 建立能力 taxonomy

从公开说明、评测报告、业务需求和 bad case 提取能力层级，例如：

```text
推理
├─ 算术
├─ 约束满足
├─ 多跳
└─ 不确定性表达
```

每项定义输入条件、预期行为、难度、评分方式和安全边界。

### 2. 从指令数据提取能力项

- 用 embedding 聚类得到主题与任务簇；
- 用 LLM 对样本生成候选能力标签；
- 多标签分类，因为一题可同时涉及检索、推理和格式；
- 人工合并同义标签、拆分过宽标签；
- 统计覆盖矩阵，发现长尾和空白能力；
- 使用主动学习优先审核低置信样本。

### 3. 生成扩展题

- 专家原创与真实日志脱敏改写；
- 参数化模板和程序生成；
- 改实体、语言、长度、位置和约束做 stress test；
- 从真实错误构造最小反例；
- 使用 LLM 生成候选，但由规则、执行器或专家验收；
- 保留 easy/medium/hard 和 OOD 切片。

### 4. 评分与质量

能执行的用 exact match、代码测试、SQL 执行或工具结果；开放题用 rubric、双盲人工和校准后的 Judge。测标注一致性、题目区分度和模型间稳定排序。

### 5. 防污染

新题私有保存，限制访问；与训练库做 n-gram、MinHash 和 embedding 去重；滚动加入新时间窗口。Benchmark 版本化，测试集与开发集分离。

### 面试总结

先定义能力树，再用聚类和 LLM 多标签抽取覆盖，从业务错误和专家原创生成独立题目，最后用可执行评分、人工验收和污染检测保证质量。不要试图复制闭源题库。
