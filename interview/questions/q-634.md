# DPO数据如何构建

## 答案

DPO 数据的基本单元是同一 prompt 下的一对回答：

```json
{
  "prompt": "...",
  "chosen": "...",
  "rejected": "..."
}
```

`chosen` 应比 `rejected` 更符合目标偏好，而且两者必须针对同一个输入，否则偏好差异无法归因。

### 候选生成

从目标业务 prompt 池抽样，覆盖任务类型、难度、长度、安全边界和长尾场景。使用当前 SFT 模型、不同 checkpoint、温度或多个模型生成若干候选。候选既要有差异，又不能全是明显错误，否则模型学不到细粒度偏好。

也可加入人工答案，但要避免让“是否为人工写作”成为格式捷径。生产迭代中应增加线上 bad case 和模型不确定样本。

### 偏好标注

制定明确 rubric，例如正确性、帮助性、指令遵循、安全性、简洁性和引用可靠性，并规定冲突时的优先级。标注者进行成对选择，可允许 tie 或 invalid；高风险和高分歧样本需要复审。记录标注者、理由和置信度，便于质量分析。

一个 prompt 有 $K$ 个排序候选时，可构造多对数据，但不要机械生成全部 $K(K-1)/2$ 对导致少数 prompt 权重过大。可采样相邻等级或难度适中的 pair。

### 清洗与切分

- 去除空回答、截断异常、模板泄漏、重复和 chosen/rejected 几乎相同的样本。
- 平衡回答长度和格式，避免模型只学到“更长”“有列表”就是更好。
- 按 prompt 或语义簇切分训练、验证、测试集，不能让同一问题的改写跨集合泄漏。
- 保证 chat template、EOS 和 system prompt 与训练时一致。

### 质量验证

统计领域、长度、偏好强度、标注一致率和模型来源分布。训练前用基线模型检查 chosen/rejected 的 log-probability 差，训练后在独立 pairwise 集和人工盲测上评估。高质量、与目标策略分布接近且偏好边界清晰的数据，通常比单纯扩大 pair 数量更重要。
