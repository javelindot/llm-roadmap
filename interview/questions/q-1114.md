# Multi-Agent 通讯和协作的难点是什么？

## 答案

Multi-Agent 的难点不在于让多个模型互相发送消息，而在于任务边界、状态一致性、权限、失败恢复和结果责任。Agent 数量越多，通信成本和错误传播路径也越多。

### 1. 先判断是否真的需要多 Agent

只有出现以下情况才值得拆分：

- 不同任务需要独立上下文或专业知识；
- 工具和数据权限必须隔离；
- 多个子任务可以并行；
- 需要 Planner、Executor、Reviewer 等不同视角；
- 单 Agent 上下文过长或流程难以维护。

简单 FAQ、单次工具调用和固定流程优先使用单 Agent 加状态机。为了形式拆成多个角色，通常只会增加延迟和调试难度。

### 2. 任务分解和职责边界

主 Agent 需要把目标拆成有依赖关系的任务 DAG，并为每个子任务定义：

> `task_id、parent_id、target_agent、objective、input_refs、expected_output、depends_on、deadline、risk_level`

目标必须可验证。例如“调查付款失败原因”应拆为订单状态、通道回执、账务状态和规则证据，而不是给多个 Agent 一个模糊的“帮我分析”。

每个 Agent 只负责自己的领域：

- 客服 Agent 负责用户意图和回复；
- 订单 Agent 读取交易状态；
- 通道 Agent 解释回执；
- 风控 Agent 返回可披露的风险大类；
- 合规 Agent 检查材料和国家限制；
- Reviewer 检查证据完整性和输出边界。

### 3. 通讯必须使用结构化协议

Agent 间消息不能只靠自然语言，应使用版本化 schema：

```json
{
  "message_id": "msg-...",
  "task_id": "task-...",
  "sender": "supervisor",
  "receiver": "channel_agent",
  "type": "task_request",
  "schema_version": "1.0",
  "input_refs": ["order_snapshot:v12"],
  "deadline": "...",
  "trace_id": "...",
  "idempotency_key": "..."
}
```

返回结果包含状态、结构化数据、证据引用、置信度、错误码和是否可重试。大对象通过受控存储传引用，不在消息中复制完整交易、证件或长文档。

### 4. 共享状态和并发一致性

多个 Agent 同时修改一个共享字典容易出现覆盖、乱序和脏读。更稳妥的方式是：

- 权威状态由 orchestrator 或状态服务维护；
- Agent 返回 state delta，不直接覆盖全局状态；
- 使用版本号、乐观锁和事件日志；
- 每个步骤带幂等键；
- 并行任务写入各自命名空间，汇总节点统一合并；
- 关键快照持久化，支持暂停、恢复和回放。

支付业务状态仍以订单、账务和通道系统为准。Agent checkpoint 只是编排状态，不能作为资金事实。

### 5. 上下文隔离和最小权限

跨 Agent 只传递完成任务所需的最小信息：

- 使用数据引用代替完整原文；
- PII 和敏感字段按接收方权限脱敏；
- 风控 Agent 的内部规则不能回传客服 Agent；
- 子 Agent 不能继承主 Agent 的全部工具；
- 每次工具调用在服务端重新鉴权。

需要防止间接提示注入沿消息链传播。网页、文档和工具结果标记为不可信数据，不能成为新的系统指令。

### 6. 冲突处理和结果聚合

多个 Agent 可能给出不同结论。不能简单让最后一个结果覆盖前面结果，应：

- 区分事实、推断和建议；
- 记录每个结论的来源、时间和版本；
- 按权威数据源优先级处理；
- 对冲突证据触发专门仲裁节点；
- 高风险冲突转人工。

例如通道 Agent 返回“处理中”，账务 Agent 返回“未入账”，两者不一定矛盾。聚合器应结合状态机解释为“通道仍处理中，账务尚未入账”，而不是投票选一个状态。

### 7. 失败、超时和级联风险

多 Agent 系统需要处理：

- 子 Agent 超时、重复消息和丢消息；
- 部分成功后的补偿；
- Agent 循环委派；
- 上游错误被下游放大；
- 并行任务长期不收敛；
- 成本和 token 失控。

每个任务设置 deadline、最大重试、最大委派深度和预算。重试只用于可重试的只读操作；资金写操作先查询最终状态，不能盲目重放。关键子任务失败时返回部分结果、降级到单 Agent 或转人工。

### 8. 可观测性和评测

统一 trace 记录：

- 任务 DAG 和消息因果关系；
- 每个 Agent 的输入引用、输出和版本；
- 工具调用、状态迁移和失败原因；
- 权限决策和人工审批；
- 延迟、token、成本和重试。

评测分三层：

- 单 Agent：领域任务、工具和安全能力；
- 协作链路：路由、消息、依赖和冲突处理；
- 端到端：任务成功率、稳定性、P95、成本和人工接管率。

还要与“单 Agent + 工具”的基线做消融。若成功率没有明显提升，却增加了延迟和故障点，就不应使用多 Agent。

### 9. 标准协议如何使用

MCP 更适合 Agent 与工具、资源和 Prompt 之间的标准化连接；A2A 面向独立 Agent 系统之间的能力发现、任务管理和 artifact 交换。A2A 0.3.0 支持不同框架和供应商的 Agent 在不暴露内部状态、记忆和工具的情况下协作。

协议只能解决互操作格式，不能自动解决业务权限、事实一致性和资金安全。跨境支付内部仍需在协议外增加租户、商户、数据范围、风险等级、审计和审批信息。
