# 在 PPO 中，如何防止模型在微调数据集以外的问题上泛化能力下降？如何防止模型收敛到单一类型的高奖励回答

## 答案

PPO 后泛化下降和回答模式坍缩，本质上都是策略过度追逐有限奖励信号：训练分布太窄、Reward Model 有漏洞、更新步幅过大，都会让模型忘掉原有能力或学会单一高奖励模板。

### 1. 防止分布外能力下降

**限制策略漂移**

```text
L_total = L_PPO + beta * KL(pi_theta || pi_ref)
```

- 保留冻结的 SFT/reference model；
- 自适应调整 KL 系数；
- PPO ratio clipping；
- 较小学习率、梯度裁剪、早停。

**混入通用数据**

- PPO batch 不只包含单一业务题；
- 混入通用指令、推理、安全和多语言样本；
- 加 auxiliary pretraining/SFT loss 做能力回放；
- 定期在训练外 benchmark 上评估，不能只看 reward。

**扩大训练分布**

- 对 Prompt 做主题、长度、语言和难度分层；
- 对真实 bad case 做聚类后补齐长尾；
- 避免重复样本和单一模板占据 batch。

### 2. 防止单一高奖励回答

- 对同一 Prompt 采样多条 response，增加组内多样性；
- 调整 temperature/top-p，避免采样数据本身过早坍缩；
- Reward Model 同时看正确性、完整性、格式、安全和简洁度；
- 对重复短语、固定开头、过长/过短回答增加惩罚；
- 使用 ensemble reward 或 rule + model 混合奖励；
- 保持适量 entropy bonus，但不要让随机性压过正确性；
- 监控 distinct-n、self-BLEU、长度分布和语义聚类。

### 3. 防止 Reward Hacking

Reward Model 必须持续用对抗样本校准。若模型发现“多写免责声明”“套固定格式”就能拿高分，需要加入反例并把可验证任务交给程序判分。

跨境支付中，交易状态、金额匹配、币种、规则引用和工具调用结果可以使用确定性 verifier；服务态度和解释质量再交给模型评分。奖励函数不应鼓励模型为了“回答完整”猜测缺失的清算或合规事实。

### 4. 训练监控

至少同时看：

- train/eval reward；
- KL、clip fraction、entropy；
- value loss、advantage 分布；
- 通用能力与安全回归集；
- 回答长度和模板重复率；
- 业务事实正确率、工具成功率和越权率。

**面试总结**：KL 和 clip 只负责“别走太远”，不能替代多样数据、可靠奖励和分布外评测。防坍缩要从采样、奖励、优化和监控四层一起做。
