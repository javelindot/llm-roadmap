# 如何确保一个 Agent 的行为是安全、可控且符合人类意图的？在 Agent 的设计中，有哪些保障对齐方法？

## 答案

Agent 的安全不能只依赖系统提示词或模型“自觉”，而要采用纵深防御：

> 明确目标与边界 → 身份和权限控制 → 受约束规划 → 工具执行网关 → 状态与数据保护 → 人工审批 → 全链路审计与持续评测。

模型负责理解、规划和生成建议，代码系统负责决定“能看什么、能调用什么、是否允许执行”。

### 1. 先建立威胁模型和动作分级

上线前先按业务识别资产、攻击面和最坏后果。跨境支付 Agent 需要重点保护：

- 商户、用户和交易的敏感数据；
- 账户余额、订单、汇率、通道和账务状态；
- 付款、退款、冻结、解冻和调额等资金能力；
- KYC/KYB、AML、制裁筛查和风控规则；
- Agent 的长期记忆、工具凭证和审计日志。

动作按风险分级：

1. **低风险只读**：查询公开规则、产品说明；
2. **受限只读**：查询当前商户自己的订单和通道回执；
3. **可逆写操作**：创建工单、保存草稿；
4. **资金或合规动作**：付款、退款、冻结、解冻、修改收款账户和调整风控策略。

前三类也要鉴权和审计；第四类默认不允许模型自主执行，必须经过确定性校验和人工审批。

### 2. 模型层：对齐目标，但不承担授权

模型层可以通过以下方式降低偏离人类意图的概率：

- 系统指令明确职责、禁止事项、信息披露边界和转人工条件；
- SFT 学习支付术语、业务 SOP、结构化输出和合规话术；
- DPO/RLAIF 优化有帮助、忠实、合规的回复偏好；
- 对抗数据覆盖提示注入、越权请求、诱导泄密和错误工具参数；
- 输出固定 schema，并对枚举、金额、币种、订单号和置信度做校验；
- 低置信度、需求不明确或证据冲突时主动追问、拒答或转人工。

模型对齐只能说明模型更可能遵守规则，不能等价于权限控制。即使模型输出“允许退款”，执行层仍然必须独立判断该用户是否有权限、交易状态是否允许以及金额是否合法。

### 3. 身份、租户和最小权限

每个请求都绑定不可由模型修改的安全上下文：

> `user_id、role、tenant_id、merchant_id、session_id、data_scope、request_id`

Agent 只接收完成任务所需的最小数据。客服 Agent、风控 Agent、通道 Agent 和账务 Agent 分别绑定独立工具白名单：

- 客服 Agent 可查订单和知识库，不能退款或解冻；
- 风控 Agent 可读风险命中大类，不能修改规则；
- 通道 Agent 可查回执和健康状态，不能自行切换资金路由；
- 写操作 Agent 与只读 Agent 隔离，使用单独身份和审批链。

权限校验必须在工具服务端再次执行，不能只在 Prompt 或 Agent 编排器中判断。检索知识库也要做行级、字段级和文档级权限过滤，避免跨商户、跨区域泄漏。

### 4. 规划和工具调用必须受约束

Agent 只输出结构化计划和工具参数，不直接拼接任意代码或 URL。执行前由工具网关检查：

- 工具是否在当前 Agent 白名单；
- 参数类型、金额、币种、国家和对象归属是否正确；
- 当前交易状态是否允许该动作；
- 是否存在重复请求和有效幂等键；
- 汇率报价、授权令牌和审批是否过期；
- 是否触发限额、频控、风控或合规规则。

外部工具返回内容按“不可信数据”处理，不能把网页、邮件、文件或工具结果中的指令提升为系统指令。工具响应只进入数据字段，不能覆盖 Agent 的目标和安全策略。

循环调用设置最大步数、时间、token、费用和工具调用预算；连续失败、状态不收敛或出现重复动作时熔断并转人工。

### 5. 状态机、幂等和人工审批

支付状态由业务状态机维护，Agent 不能自由改写。例如付款从 `PROCESSING` 不能因为用户重复询问就回退到 `PENDING`，旧回调也不能覆盖更新后的状态。

所有写操作需要：

- 幂等键和唯一业务流水号；
- 乐观锁或状态版本校验；
- 执行前后的不可篡改审计记录；
- 超时后的查询确认，而不是盲目重试；
- 必要时使用补偿流程，而不是让模型自行“撤销”。

人工审批界面应展示目标动作、对象、金额、币种、证据、风险提示和参数差异，避免操作者在信息不足时机械点击确认。审批后也要由后端重新校验，不直接复用模型生成的结果。

### 6. 记忆、数据和输出安全

短期会话、长期记忆和业务事实分开存储：

- 长期记忆只保存经过筛选的用户偏好，不保存完整卡号、证件和敏感风控细节；
- 写入记忆前做来源验证、PII 脱敏、租户隔离和有效期控制；
- 检索到的历史内容不能改变当前权限和系统策略；
- 日志对账户、证件、银行卡、密钥和原始材料做脱敏；
- 对外回答禁止暴露具体风控阈值、黑名单来源和内部通道策略。

输出层检查事实引用、敏感信息、错误资金承诺和违规建议。对于“为什么被冻结”这类问题，可以说明需要补充材料或人工复核，但不能披露可被攻击者利用的具体规则。

### 7. 监控、红队和上线门禁

安全评测既看 Agent 最终回复，也看完整 trajectory：

- 是否选择了正确工具和正确参数；
- 是否发生越权读取或写操作；
- 是否泄露敏感信息或内部策略；
- 是否能识别提示注入和恶意工具返回；
- 是否在需要时正确拒绝、追问和转人工；
- 是否存在循环、重复扣款、级联失败和成本失控。

核心指标包括策略违规率、未授权工具调用率、敏感信息泄漏率、安全任务完成率、拒答精确率/召回率、误转人工率和高风险动作拦截率。上线采用离线对抗集、影子流量、小流量灰度和自动回滚；安全 guardrail 触发时优先降级到只读查询或人工服务。

OWASP 2026 Agentic Top 10 将目标劫持、工具滥用、身份权限滥用、供应链风险、意外代码执行、记忆污染、Agent 间通信风险、级联故障、人机信任利用和失控 Agent 纳入统一威胁模型。NIST AI RMF 的 Govern、Map、Measure、Manage 可以用于把这些控制落实到组织责任、风险评测和上线治理中。
