# 什么时候用元数据表？

## 答案

> "元数据表"在不同场景含义不同——**RAG 里指给 chunk 加属性**，**NL2SQL 里指描述数据库 schema 的表**。我两个都讲。
>
> **场景一：RAG 中的元数据**
>
> 给每个 chunk 加结构化属性，用于：
>
> 1. **过滤检索**：用户问"东南亚汇率"，先按 country = "SG"/"ID"/"TH" 过滤再做向量
> 2. **路由**：按业务线（客服 / 风控 / KYB）路由到对应索引
> 3. **可解释**：答案附带"来自第 X 章第 Y 节"
> 4. **权限**：商户只能看自己授权的内容
> 5. **时效性**：按 created_at 排序，旧文档降权
>
> 元数据示例：
> ```python
> {
>     "doc_id": "policy_2026_sg",
>     "source": "monetary_authority_sg",
>     "country": "SG",
>     "currency": "SGD",
>     "business_line": "cross_border",
>     "section_path": "第三章 > 3.2 跨境汇款时效",
>     "page": 12,
>     "version": "2026.05",
>     "language": "zh-CN",
>     "created_at": "2026-05-15",
>     "confidence_level": "high",
> }
> ```
>
> 在 Milvus / Qdrant 里用 `filter expr` 直接过滤：
> ```python
> milvus.search(
>     query_vector,
>     expr="country == 'SG' && version >= '2026.01'",
>     limit=20
> )
> ```
>
> **场景二：NL2SQL 中的元数据表**
>
> 让 LLM 写正确 SQL 必须告诉它表结构，但**直接把 DDL 全塞 prompt 太长**——这就是元数据表的作用。
>
> 元数据表内容：
> ```yaml
> # table_metadata.yaml
> tables:
>   - name: orders
>     description: "跨境支付订单主表"
>     columns:
>       - name: order_id
>         type: varchar(32)
>         description: "订单唯一标识"
>         primary_key: true
>       - name: status
>         type: varchar(16)
>         description: "订单状态"
>         enum: ["pending", "completed", "refunded", "failed"]
>       - name: amount
>         type: decimal(18,2)
>         description: "订单金额（分）"
>       - name: country
>         type: varchar(8)
>         description: "国家代码 ISO-3166"
>     example_queries:
>       - "查询近 7 天完成订单总额"
>       - "查询某国家退款率"
> ```
>
> NL2SQL 流程：
> ```python
> async def nl2sql(user_query: str):
>     # 1. 元数据库找相关表
>     relevant_tables = await meta_db.find_relevant(user_query)
>
>     # 2. 拼 prompt（只塞相关表的元数据）
>     schema_prompt = render_schema(relevant_tables)
>
>     # 3. LLM 生成 SQL
>     sql = await llm.generate(user_query, schema_prompt)
>
>     # 4. SQL 校验 + 执行
>     return await sql_executor.run(sql)
> ```
>
> **什么时候必须用元数据表**：
>
> 1. **表多到塞不下 prompt**——超过 20 张表必须分级
> 2. **多业务线 schema 不同**——按业务路由元数据
> 3. **频繁 schema 变化**——元数据库可配置
> 4. **需要权限控制**——元数据带 ACL，LLM 只看授权字段
> 5. **跨方言**（MySQL / Postgres / Hive）——元数据规范化方言
>
> **什么时候不需要**：
>
> - 单表 / 少量字段 → 直接 DDL 塞 prompt
> - 业务简单稳定 → schema 写死
> - POC 阶段 → 元数据维护成本高
>
> **我们的实际用法**：
>
> **客服 RAG**：
> - chunk 元数据：source、section_path、country、language、version
> - 用于多 Index 路由 + 元数据过滤
>
> **业务报表 NL2SQL**：
> - 元数据表 + 业务术语映射表
> - 200+ 表的 schema 文档化
> - 维护成本：每月 1 人天
>
> **几个工程经验**：
>
> 1. **元数据要可视化 + 可编辑**——别全代码里硬编码
>
> 2. **元数据本身做向量检索**——表多了找相关表也是检索问题
>
> 3. **元数据带 few-shot 例子**——LLM 看 example 比看 schema 更准
>
> 4. **元数据版本管理**——schema 变化要追溯
>
> 5. **元数据规模化时上 dbt + metadata layer**——dbt-llm 这种工具
>
> 一句话：**元数据表是"让 LLM 看得懂数据"的中间层**——RAG 用它做过滤路由、NL2SQL 用它喂 schema，本质都是"结构化描述数据形态"。
