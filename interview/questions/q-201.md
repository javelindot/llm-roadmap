# 请详细介绍 ReAct 框架以及自主抉择 Agent 的核心概念、工作原理与应用场景？

## 答案

> 我从 ReAct 和"自主抉择"两块分开讲，再说怎么组合。
>
> **ReAct（Reasoning + Acting）核心**：
>
> 一个"思考-动作-观察"的循环——
> ```
> Thought → Action → Observation → Thought → ... → Final Answer
> ```
> 每一步显式输出思考过程，再调一个工具，看返回结果，再继续思考。优点是过程可观测、错了能 self-correct。
>
> **自主抉择 Agent**：
>
> 这其实是 ReAct 思想的进阶——让 Agent **自己决定下一步该走哪条分支**，而不是按固定流程跑。常见三种实现方式：
> - **基于 prompt 的 LLM 决策**：Agent 自己在 Thought 里输出 "决定走 A 路径还是 B 路径"
> - **基于 LangGraph 的条件边**：每个节点输出后由 condition function 决定下一节点
> - **基于策略模式 + 状态机**：每个状态对应一组可用动作，LLM 选一个
>
> **工作原理**（结合我们落地）：
> ```
> 1. Agent 收到工单
> 2. 思考：这是什么类型的工单（分类）
> 3. 决策：选哪个 SOP / Skill 处理
> 4. 执行：按 SOP 调一系列工具
> 5. 反思：结果对吗？要不要换路径？
> 6. 输出：最终答案 + 引用 + trace
> ```
>
> **应用场景**：
>
> 我们跨境支付里用了三个：
> - **客服意图分流**（ReAct，短链路）：问询、投诉、异常分类后路由
> - **风控调查 Agent**（自主抉择，长链路）：根据查到的证据决定继续调查还是结案，可回溯
> - **对账差异定位**（Reflexion 风格）：猜原因 → 验证 → 错了换假设
>
> **几个工程经验**：
> - 简单任务用 ReAct，复杂高风险用 Plan-and-Execute + HITL
> - 自主抉择 Agent 必须加 max_iterations + token 上限，防死循环
> - 关键决策点的 reasoning trace 留存，方便审计

**🪝 顺嘴带出去的钩子**：
- "ReAct vs Plan-and-Execute 在我们业务里的选型可以聊。"
- "自主抉择 Agent 的死循环治理可以展开。"
- "LangGraph 状态图实现自主抉择的细节可以讲讲。"
