# RAG 的查询改写和混合检索是如何实现的？

## 答案

查询改写解决“用户表达与知识库表达不一致”，混合检索解决“精确匹配和语义匹配各有盲区”。典型链路是：

> 原始问题与会话 → 结构化改写 → BM25/向量并行召回 → 过滤与融合 → rerank → 证据构造。

### 1. 查询改写输入

改写器不仅接收当前一句话，还接收：

- 最近几轮必要会话；
- 当前用户、商户和业务对象；
- 已提取的订单号、国家、币种、产品和通道；
- 允许访问的知识库和工具；
- 当前时间与文档版本要求。

输出使用结构化 Schema：

```json
{
  "standalone_query": "...",
  "intents": ["..."],
  "entities": {
    "country": "...",
    "currency": "...",
    "channel": "...",
    "error_code": "..."
  },
  "bm25_query": "...",
  "semantic_queries": ["..."],
  "needs_realtime_tool": true,
  "needs_clarification": false
}
```

原始 query 始终保留，并与改写结果一起检索，防止模型改写偏离。

### 2. 常见改写策略

**多轮消歧**

将“那退款呢”改写为“订单 X 的退款状态和预计到账时间是什么”。

**术语归一化**

统一 `付款/payout/出款`、`拒付/chargeback`、产品别名和错误码格式。

**多意图拆分**

“查付款失败原因并问重新提交材料”拆为实时交易查询和规则检索两个任务。

**Multi-Query**

对复杂语义生成少量不同角度查询，提高召回覆盖；每个改写必须受原问题实体和范围约束。

**HyDE**

对短、口语化且与正式文档存在表达鸿沟的问题生成假设文档用于向量检索。错误码、法规编号和产品名等精确问题不使用 HyDE，避免引入错误术语。

### 3. BM25 与向量检索

BM25 擅长：

- SWIFT、SEPA、PIX；
- USD、BRL 等币种；
- 国家码、错误码、产品名；
- 法规编号和固定术语。

向量检索擅长：

- “款没到”“银行没有回”等口语表达；
- 同义词和跨语言语义；
- 用户描述与正式文档之间的风格差异。

两路召回前先做租户、权限、国家、币种、产品、版本和有效期过滤。

### 4. 融合和重排

不同召回器分数不可直接相加。常用 RRF：

> `RRF(d) = Σ 1 / (k + rank_i(d))`

RRF 不依赖不同分数空间的绝对值，适合合并 BM25、向量和多查询结果。也可以在有标注数据时训练融合模型，将召回来源、排名、匹配字段和新鲜度作为特征。

融合后执行：

- URL/chunk 去重；
- parent-child 合并；
- cross-encoder rerank；
- 来源权威性和时效加权；
- 否定词、适用范围和版本校验；
- 子问题证据覆盖。

### 5. 失败控制

- 改写置信度低或关键实体缺失时先追问；
- 改写不得改变金额、币种、订单号和用户目标；
- 实时状态问题必须走工具，不能只改写后查文档；
- 多查询结果发散时退回原 query 和精确过滤；
- 无有效证据时拒答或转人工。

### 6. 评测

- 改写：意图保持率、实体保持率、指代恢复正确率；
- 路由：知识库/工具选择准确率；
- 召回：Recall@K、MRR、各路独占贡献；
- 融合与重排：NDCG、Context Precision；
- 端到端：答案正确率、引用正确率和任务成功率；
- 工程：延迟、查询扩展数量和单次成本。

需要做消融对比原 query、只改写、混合检索、改写加混合检索和 rerank，确认提升来自哪一层。
