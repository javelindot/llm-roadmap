# 大模型微调全流程，SFT 数据构建经验

## 答案

大模型微调应从目标能力和数据闭环出发，而不是先选 LoRA 参数。完整流程是：

> 任务定义 → 基线与模型选型 → 数据构建 → 格式化和切分 → 训练方案 → 离线评测 → 灰度上线 → Bad Case 回流。

### 1. 定义任务和边界

先明确模型需要学习什么：

- 支付意图识别和实体抽取；
- RAG 证据整合与引用；
- Query Rewrite；
- 工具选择和参数生成；
- 客服回复、材料摘要；
- 拒答、追问和转人工。

实时汇率、订单状态、国家规则和权限不写入模型参数，分别交给工具、RAG 和确定性服务。

### 2. 建立未微调基线

使用基础模型加 Prompt/RAG 跑冻结黄金集，记录：

- 正确性、faithfulness、格式通过率；
- 工具和参数准确率；
- 不同国家、语言、产品分桶；
- P95/P99、token 和成本；
- 安全、越权和拒答表现。

基线能判断问题是否真的需要微调。知识缺失优先补 RAG，权限错误优先修代码。

### 3. SFT 数据 Schema

按任务保存：

```text
messages
available_tools
expected_tool_call
tool_result
final_response
source
scenario
risk_level
data_version
```

数据来源包括脱敏客服会话、业务专家示范、SOP、线上 bad case、工具 trace 和对抗样本。

### 4. 数据治理

- 删除姓名、证件、银行卡、账户和真实交易标识；
- 同一会话内实体替换保持一致；
- 清理过期规则、错误回复、乱码和模板残留；
- 精确哈希与语义近重复去重；
- 多轮会话不能丢失必要前文；
- 按国家、币种、语言、产品和风险分桶；
- 高频任务按流量覆盖，长尾和高风险适度过采样；
- 合成数据限制比例，并由规则或专家抽检。

高风险样本双人标注和仲裁，标注指南明确应回答、追问、拒答或转人工的条件。

### 5. 数据切分

按时间、会话、商户或业务事件分组切分，避免同一 FAQ 改写、同一订单或相邻文档片段跨训练与测试集。测试集在模型选择前冻结。

每个版本记录数据来源、时间范围、清洗规则、分桶数量、Prompt、Tokenizer 和标注规范。

### 6. 格式化和 Label Mask

- 使用与部署一致的 Chat Template；
- 检查 BOS/EOS、role token 和 padding；
- 通常只对 Assistant 输出计算 loss；
- 工具调用与工具结果角色必须正确；
- Packing 时防止跨样本 Attention 和标签串扰；
- 用几十条样本先做过拟合测试，确认 loss、mask 和模板正确。

很多“微调没效果”实际来自模板或标签错位。

### 7. 训练方案

数据较少时先使用 LoRA/QLoRA：

- 选择 target modules；
- sweep rank、alpha、学习率和 epoch；
- BF16、Gradient Checkpointing 和长度分桶；
- 按有效 token 设置 batch 与 scheduler；
- 保存训练配置、代码、数据和 base model revision。

只有数据足够、任务变化大且资源允许时再评估全参微调。训练中监控 train/validation loss、梯度范数、吞吐、显存和分桶任务指标。

### 8. 评测与上线

离线同时评估：

- 任务正确率、Macro-F1；
- RAG faithfulness、引用正确率；
- Tool/Argument Accuracy；
- Schema 一次通过率；
- 拒答与转人工准确率；
- 通用能力和安全回归；
- 输出长度、重复和幻觉；
- P95/P99 与成本。

通过后依次做历史回放、shadow、小流量灰度和 A/B。上线保留模型、Prompt、数据和 Adapter 回滚版本。

### 9. 持续迭代

线上 bad case 先归因：

```text
知识缺失 → 更新 RAG
工具或状态错误 → 修业务服务
固定格式/行为模式错误 → 补 SFT
偏好和表达问题 → 构造 DPO
权限与资金安全 → 修确定性控制
```

只有错误属于可学习的稳定模式时才回流训练，避免用微调掩盖系统缺陷。
