# 如果商品知识库实时更新，你怎么保证 RAG 的召回结果和库存一致？

## 答案

核心原则是：**语义知识索引负责找候选，库存服务是实时权威源**。不要把高频变化的库存数量只写入向量库并依赖索引实时一致。

### 1. 数据分层

- 标题、描述、属性等低频信息进入全文和向量索引；
- 库存、价格、上下架、限购等高频状态保存在交易型商品/库存服务；
- chunk 保存稳定商品 ID、版本和更新时间，不把 embedding 当业务主键。

### 2. 更新链路

- 商品变更通过 CDC 或 Transactional Outbox 发送事件；
- 消费者幂等更新倒排、向量索引和缓存；
- 使用版本号或事件序列号拒绝乱序旧更新；
- 下架与删除发送 tombstone，并清理各索引及缓存；
- 大版本重建采用蓝绿索引和原子 alias 切换。

### 3. 查询时保证

1. RAG 从语义索引召回商品 ID；
2. 批量查询实时库存/价格服务；
3. 过滤无货、下架或无资格商品；
4. 将校验后的结构化事实交给生成模型；
5. 下单时再次做原子库存校验与扣减。

缓存键包含商品版本，库存缓存使用短 TTL 或事件失效。若库存服务超时，宁可显示“库存待确认”或降级传统搜索，不让模型声称有货。

### 4. 监控

监控 CDC lag、索引版本差、删除残留、无货召回率、库存二次校验失败率和缓存陈旧时间，并支持对账修复。

跨境支付同理：知识库可召回产品说明，余额、费率和通道状态必须查询实时权威系统，执行前再校验一次。

### 面试总结

用 CDC 和版本号让索引最终一致，用查询时实时过滤保证展示正确，用交易时原子校验保证执行正确。RAG 负责语义候选，不承担库存账本的一致性。
