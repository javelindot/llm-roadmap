# 大模型解决了什么问题以及为什么引入大模型

## 答案

引入大模型不是为了替换所有传统系统，而是解决过去需要为每个自然语言任务单独建模、规则和接口的问题。它提供统一的语言理解、生成、少样本迁移和工具编排层，但精确状态和高风险决策仍需确定性系统。

### 1. 传统方案的痛点

- 每个意图、语言和流程单独开发；
- 规则维护成本高；
- 非结构化文档难利用；
- 新任务需要重新标注和训练；
- 多轮、长文本和开放式表达覆盖差；
- 不同系统缺少统一自然语言入口。

### 2. 大模型解决的核心问题

**通用自然语言接口**

同一模型覆盖问答、摘要、抽取、改写、分类和代码/SQL。

**少样本迁移**

通过 Prompt/Few-shot 快速适配新任务，降低冷启动成本。

**非结构化信息处理**

理解文档、邮件、工单、图片和对话，并组织结果。

**知识与工具编排**

通过 RAG 查文档，通过 Function Calling 调用订单、账务和风控系统。

**多语言与生成**

统一处理跨境业务中的多语言咨询和报告。

### 3. 为什么现在可用

- 模型规模、数据和 Post-training 提升；
- 长上下文与多模态能力增强；
- 推理框架、量化和硬件降低成本；
- RAG、Agent、Schema 和评测生态成熟；
- 可通过私有化/网关做数据与权限治理。

### 4. 不能解决什么

- 不能天然保证事实正确；
- 不能替代数据库和 Ledger；
- 不能保证实时知识；
- 不能仅靠 Prompt 实现权限和合规；
- 对确定性计算、超低延迟分类可能不如规则/小模型；
- 高风险结论仍需责任主体。

### 5. 跨境支付价值

- 多语言商户客服；
- KYC/KYB 材料抽取与审核辅助；
- AML 案例证据摘要；
- 清结算差异解释；
- 拒付材料整理；
- 内部规则 RAG；
- Agent 查询订单和生成工单。

但余额、费率、汇率、交易状态、名单命中和资金操作必须走真实系统。

### 6. 是否值得引入

用业务指标判断：

- 首次解决率；
- 人工处理时长；
- 工具任务成功率；
- 事实和引用准确率；
- P95 延迟和单次成本；
- 安全、越权和人工接管率。

如果规则或小模型已能稳定解决，不应为了“使用大模型”增加复杂度。

**面试总结**：大模型的价值是把复杂非结构化任务统一成可编排的语言接口，而不是成为新的账务核心。
