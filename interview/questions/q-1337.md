# 大模型推理加速方案在工业界的落地经验

## 答案

工业推理优化不能从“换 vLLM”开始，而要先判断瓶颈在 Prefill、Decode、调度、通信还是外部工具。目标也不是单一 tokens/s，而是在质量不退化的前提下同时优化 TTFT、TPOT、吞吐、P95/P99、显存和单请求成本。

### 1. 先区分 Prefill 与 Decode

**Prefill**

- 一次处理整个 Prompt；
- 计算密集；
- 长上下文 TTFT 高；
- 重点是 FlashAttention、chunked prefill、prefix cache 和并行。

**Decode**

- 每步生成一个 token；
- 受 KV Cache 读取和显存带宽影响；
- 重点是 continuous batching、KV 管理、量化和 speculative decoding。

### 2. 模型级优化

- 选择更小但满足质量的模型；
- 蒸馏和任务路由；
- FP8/INT8/INT4/FP4 权重量化；
- KV Cache 量化；
- GQA/MQA/MLA 降低 KV Cache；
- 剪枝和稀疏/MoE；
- 控制最大输入、输出和 thinking budget。

量化必须按业务集回归数字、JSON、工具参数和多语言，不能只看通用 benchmark。

### 3. Kernel 与图优化

- FlashAttention/FlashInfer；
- fused RMSNorm、RoPE、SwiGLU、采样；
- CUDA Graph；
- TensorRT-LLM 等编译和算子融合；
- 合适的 BF16/FP8 accumulate；
- 减少 CPU-GPU 同步和 Python 调度开销。

FlashAttention 优化的是精确 Attention 的 IO，不改变模型数学结果。

### 4. KV Cache 与调度

- PagedAttention/分页 KV，降低碎片；
- continuous/in-flight batching；
- prefix/radix cache 复用公共前缀；
- chunked prefill 避免长 Prompt 阻塞 Decode；
- preemption 和优先级队列；
- 按输入/输出长度分桶；
- 限制超长和异常请求。

跨境支付客服的 system prompt、工具定义和公共政策前缀高度重复，prefix cache 通常很有价值；但缓存 key 必须包含模型、模板、权限和知识版本。

### 5. 并行与集群

- Tensor Parallel：单层矩阵切多卡；
- Pipeline Parallel：层间切分；
- Expert Parallel：MoE 专家分布；
- Data Parallel：多副本扩吞吐；
- Prefill/Decode Disaggregation：两阶段使用不同资源池；
- 请求路由考虑 cache affinity 和负载。

小模型盲目多卡可能被通信拖慢，应先单卡压测。

### 6. Speculative Decoding

Draft Model 或 MTP/EAGLE 类方法先生成候选，Target Model 并行验证。适合低 batch、单请求延迟敏感场景；大 batch 或 draft 命中率低时收益可能有限。

### 7. 生产落地步骤

1. 固定质量基线和代表性流量；
2. 采集输入/输出长度分布；
3. 分别压测 TTFT、TPOT、throughput；
4. 找到 GPU 利用率、KV、通信或调度瓶颈；
5. 每次只改一类变量；
6. 做灰度、回滚和容量保护；
7. 监控队列、OOM、cache hit、超时和降级率。

### 8. 未来方向

截至 2026 年，重点正在转向低精度软硬件协同、P/D 解耦、KV Cache 压缩、MoE 专家并行、多 token prediction/speculation，以及根据任务动态分配推理计算。

**面试总结**：工业推理优化是工作负载、模型、kernel、缓存、调度和集群的联合优化，必须用端到端指标证明收益。
