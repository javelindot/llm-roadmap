# 推荐项目里的图检索具体是如何实现的

## 答案

推荐中的图检索，是从用户、item 和业务实体构成的图上，以当前用户或最近行为为种子做邻居扩展、路径打分或图向量召回。它适合建模协同关系、互补关系和多跳关联。

### 1. 图 Schema

典型节点包括：

- User、Item、Category、Brand、Author、Query；
- 跨境支付场景中的 Merchant、Product、Country、Currency、Channel。

边包括：

- 点击、收藏、购买、关注、搜索；
- 同类、相似、搭配、替代；
- 作者发布、商品属于类目；
- 商户使用产品、产品支持币种、通道覆盖国家。

每条边携带 `type、timestamp、weight、source、version`，行为边按类型和时间衰减。

### 2. 离线图构建

从行为日志和主数据生成边，完成：

- 去重、异常账号和机器人过滤；
- 行为权重和时间衰减；
- 热门节点降权；
- 权限、下架和无效边清理；
- 构建邻接表或 CSR；
- 为高频节点预计算 Top 邻居；
- 训练 Node2Vec、GraphSAGE、LightGCN 等图 embedding。

生产检索不一定直接查询通用图数据库。大规模低延迟场景通常把预计算邻居放在 KV/倒排服务，把图 embedding 放 ANN 索引；图数据库更多用于图探索、规则和离线分析。

### 3. 在线召回

请求到来后：

1. 选择种子：用户、最近点击 item、Query 实体；
2. 一跳扩展：相似 item、关注作者、同类目；
3. 二跳扩展：`User → Item → User → Item` 或互补关系；
4. 控制每跳 fan-out、路径类型和最大深度；
5. 按路径权重、时间、热度和 embedding 相似度打分；
6. 聚合同一候选的多条路径；
7. 去重并执行权限、状态和业务硬过滤；
8. 交给粗排和精排。

候选得分可以表示为：

```text
score(u, i) =
Σ_path path_weight
× time_decay
× edge_confidence
× popularity_penalty
```

多路径命中本身也可以作为后续排序特征。

### 4. 控制图爆炸

- 限制跳数，推荐召回通常不超过 2 到 3 跳；
- 每种关系设置 fan-out；
- 对超级热门节点降权或截断；
- 路径白名单，避免无业务含义的连接；
- 缓存高频种子的邻居；
- 在线只做轻量扩展，复杂 GNN 离线计算；
- 为每路设置超时、最低返回数和降级。

跳数越多不一定越好，噪声、延迟和隐私风险都会快速增长。

### 5. 冷启动与更新

- 新 item 用内容、类目和作者边补充；
- 新用户用地域、入口和 session 种子；
- 实时行为写入增量邻接表；
- 图 embedding 周期训练，关键状态实时过滤；
- 删除用户或下架 item 时同步清理边、缓存和索引。

### 6. 评测

- Recall@K、HitRate、MRR、NDCG；
- 图召回独占贡献和与 ItemCF/双塔的重合；
- 路径覆盖、平均跳数和热门度；
- 新 item、长尾和稀疏用户分桶；
- P95/P99、缓存命中、图更新延迟；
- 隐私、权限和无效候选率；
- 线上转化和长期价值。

### 7. 跨境支付中的实现

可构建：

```text
Merchant → Currency → Country → Product → Channel
```

根据商户注册地、交易走廊、历史产品和供应商国家召回收款账户、本地付款、SWIFT、FX 或卡产品。跨商户行为必须匿名聚合并满足最小群组规模，不能暴露其他商户或收款人的交易关系。

模型召回后仍需由 KYB、牌照、制裁、额度和实时通道状态过滤。图路径可以解释“为什么候选相关”，但不能作为合规准入依据。
