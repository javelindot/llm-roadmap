# 响应阶段，分段生成、全文生成的优缺点

## 答案

> 两种各有适用场景，**我们跨境支付里"流式分段 + 全文回流"组合用**，按场景选。
>
> **全文生成（一次性返回）**：
>
> 优点：
> - 实现简单
> - **生成完整后才返回 → 可校验、可改写**
> - 适合需要"全文一致性"的场景（报告、摘要）
> - 引用校验、NLI 事实校验都容易加
>
> 缺点：
> - **用户等待时间长**：必须等所有 token 生成完
> - **延迟感差**：1-3 秒空白，用户以为坏了
> - **响应窗口超时风险**：长生成可能超出 HTTP timeout
> - **失败时全丢**：生成到一半挂了，前面 token 全浪费
>
> **分段生成 / 流式生成**：
>
> 优点：
> - **首 token 延迟低**：用户 100-500ms 就看到第一个字
> - **体感快**：边读边等
> - **部分内容也能用**：长文本边出现边看
> - **可中断**：用户觉得跑偏可以停
>
> 缺点：
> - **没法做全文校验**：边生成边返回，事实性校验来不及
> - **引用难注入**：流式输出时引用还在 sourcing
> - **错误恢复难**：中途断开重连复杂
> - **前后端协议复杂**：SSE / WebSocket、增量帧、错误处理
> - **某些场景反而慢**：因为不能批处理
>
> **我们的实际选型**：
>
> | 场景 | 选型 | 原因 |
> |------|------|------|
> | 客服对话 | **流式** | 用户期望"打字感"、UX 优先 |
> | 风控调查报告 | **全文** | 必须全文校验、引用、事实性 |
> | 月度业务 PPT | **分段持久化** | 长任务，每章节存盘 |
> | KYB 字段提取 | **全文** | JSON 输出，必须完整结构 |
> | 简单查询 | **全文**（够快了）| 500ms 内出全文，不值得流式 |
>
> **流式的工程实现**：
>
> 用 SSE（Server-Sent Events）：
> ```python
> from fastapi import FastAPI
> from sse_starlette.sse import EventSourceResponse
>
> @app.get("/chat/stream")
> async def chat_stream(query: str):
>     async def event_gen():
>         async for token in llm.stream(query):
>             yield {"data": token}
>         yield {"data": "[DONE]"}
>     return EventSourceResponse(event_gen())
> ```
>
> **流式 + 引用注入怎么做**：
>
> 我们的做法是**两阶段**：
> 1. 第一阶段：流式生成正文（用户体感快）
> 2. 第二阶段：生成完成后单独发"引用列表"事件，前端附加显示
>
> ```
> SSE event 1: {"type": "token", "data": "您的"}
> SSE event 2: {"type": "token", "data": "订单"}
> ...
> SSE event N: {"type": "done"}
> SSE event N+1: {"type": "citations", "data": [...]}
> ```
>
> **分段生成的另一种含义：分段生成长文档**
>
> 比如生成 30 页 PPT，不是流式 token 而是按章节生成：
> - 优点：每章节可校验、可持久化、可断点恢复
> - 缺点：章节间一致性可能差
> - 我们 PPT 生成 + 报告生成都用这个范式
>
> **几个工程经验**：
>
> 1. **流式必须做反压**——前端接不动会拖累后端
>
> 2. **超时设计**：流式连接要长（5-10 分钟），但每 token 间隔有 timeout（30 秒无 token 断）
>
> 3. **首 token 延迟是关键 KPI**——用户感受比总延迟更直接
>
> 4. **流式 nginx 配置**：`proxy_buffering off`、`proxy_request_buffering off`
>
> 5. **错误处理**：SSE error event 比 HTTP error code 复杂，要前端协议化
>
> **不适合流式的场景**：
> - 结构化输出（JSON）——token 级输出可能破坏 JSON 结构
> - 强校验场景——校验只能在全文完成后
> - 短回复——< 200ms 就出完，流式没意义
>
> 一句话：**流式优化 UX，全文保证质量**——客服对话流式、报告生成全文、长任务分段持久化。
