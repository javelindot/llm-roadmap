# Qwen 2.5 VL的图片输入处理与Loss设计

## 答案

Qwen2.5-VL 的图片处理可以概括为：图片先经过视觉编码器变成视觉 token，再通过投影/连接模块对齐到语言模型的 token 空间，最后和文本 token 一起送入大语言模型做自回归生成。官方技术报告和仓库强调了动态分辨率、动态 FPS、窗口注意力、SwiGLU/RMSNorm 和 2D-RoPE 等设计。

### 1. 图片输入怎么处理

典型流程是：

```text
image
-> processor 做 resize / normalize / patchify
-> vision encoder 提取视觉特征
-> projector/merger 压缩并映射到 LLM hidden size
-> 与文本 token 拼接
-> decoder-only LLM 生成答案
```

和固定 224x224 的老式视觉模型不同，Qwen2.5-VL 使用动态分辨率思路：不同大小的图片会对应不同数量的视觉 token。这样对 OCR、票据、表格、截图、支付凭证等高分辨率场景更友好，因为小字和布局信息不容易被过度压缩。

跨境支付业务中，常见输入包括：

- 护照、营业执照、地址证明；
- 银行回单、SWIFT MT103、支付凭证；
- 后台截图、报错截图；
- 拒付材料和物流证明。

模型可以做 OCR、字段抽取、版面理解和材料完整性检查，但最终 KYC/KYB、账务和合规结论仍要走规则系统或人工复核。

### 2. 位置编码和多模态对齐

多模态模型不能只知道“有哪些视觉 token”，还要知道它们在图片中的位置。Qwen2.5-VL 使用和多模态位置相关的设计，把文本、图像高宽位置、视频时间维度对齐起来。对于图片，可以理解为需要保留二维空间关系；对于视频，还要保留时间关系。

这对文档理解很关键。例如收款人姓名、银行账号、币种和金额在票据中的相对位置，会影响字段归属。

### 3. Loss 怎么设计

在指令微调阶段，常见 loss 仍是语言模型的自回归交叉熵：

```text
loss = CrossEntropy(logits, target_tokens)
```

但通常只对 assistant 需要生成的 token 计算 loss：

```text
labels[system_tokens] = -100
labels[user_tokens] = -100
labels[image_tokens] = -100
labels[assistant_tokens] = target_ids
```

也就是说，图片 token 是条件输入，不是让模型“预测图片 token”。如果任务是 OCR、目标定位、表格理解或 grounding，本质上也是把目标文本、坐标、结构化字段序列化成 token 后做交叉熵监督。

### 4. 训练任务类型

多模态训练数据可以包括：

- 图像描述；
- OCR 和文档问答；
- 表格/票据字段抽取；
- 视觉推理；
- 坐标定位和 grounding；
- 多图对比；
- 视频理解。

跨境支付可设计为：

```text
输入：付款凭证图片 + 问题
输出：抽取金额、币种、付款人、收款人、交易时间，并说明哪些字段缺失
```

### 5. 面试容易错的点

- 不要说图片直接变成普通文字，实际需要视觉编码器；
- 不要把 CLIP 对比学习 loss 说成所有 VL 指令微调的主 loss；
- 不要忘记 loss mask，prompt 和图片 token 通常不作为生成目标；
- 不要只讲模型结构，还要讲高分辨率文档、OCR、票据理解这些业务价值；
- 不要让模型直接给最终合规结论。

### 6. 面试总结

Qwen2.5-VL 的核心是把动态分辨率图片编码成视觉 token，再和文本 token 一起进入 LLM；SFT 阶段主要用 assistant token 的 next-token cross entropy。放到跨境支付里，它适合做凭证 OCR、材料抽取、截图理解和工单辅助，但最终资金、风控、KYC/KYB 结论必须由确定性系统和人工审核兜底。
