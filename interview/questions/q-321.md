# RAG 怎么提高检索准确率？

## 答案

> RAG 准确率优化分**查询侧、索引侧、检索侧、重排侧、评测侧** 5 层，每层都有明确动作，组合下来 RAG 召回率能从 60% 提到 90%+。
>
> **1. 查询侧优化**
>
> - **Query 改写**：多轮消解 + 业务术语扩展（详见 q-211）→ +10 个点
> - **HyDE 假设答案 embed**（详见 q-131）→ +5 个点
> - **多 query Fusion**（详见 q-129）→ 关键场景 +5 个点
> - **Step-back prompting**：抽象上位概念再检索 → 长尾 +3 个点
> - **意图分类 + 路由**（详见 q-189）→ +5 个点
>
> **2. 索引侧优化（提升上限）**
>
> - **更好的 embedding 模型**：bge-large > m3e-base，业务精调再 +10 个点（详见 q-103）
> - **业务精调**：用 5w 业务 (query, doc) 对做 InfoNCE → +10-15 个点
> - **更细的分块**：结构感知 + 合理大小 + overlap（详见 q-121）→ +5 个点
> - **元数据富化**：摘要、假设问题、关键词（详见 q-123）→ +5 个点
> - **多 Index**：按业务线分库 + 路由 → +5 个点
>
> **3. 检索侧优化**
>
> - **混合检索（向量 + BM25）**：捕捉语义 + 关键词（详见 q-213）→ +8 个点
> - **元数据预过滤**：缩小候选集 → +3 个点（精度）
> - **多路并行**：向量 + BM25 + HyDE async → 延迟不变
> - **加大 top-K**：召回 30-50 → 给 rerank 留空间
>
> **4. 重排侧优化（精度放大器）**
>
> - **Cross-Encoder Rerank**：bge-reranker top-30 → top-5（详见 q-127）→ +10-15 个点
> - **LLM Rerank**：关键场景上 Qwen-72B 二次精排 → +3-5 个点
> - **多模型 Rerank ensemble**：多个 rerank 投票 → 关键场景 +2 个点
> - **threshold 过滤**：rerank score < 0.3 直接丢
>
> **5. 评测 + 飞轮**
>
> - **黄金集评测**（详见 q-185、q-239）：知道是哪类失败
> - **失败 case 分析**：召回失败 vs 生成失败定位
> - **数据回流**：失败 case 补到 embedding 训练集
> - **持续迭代**：每月重训 embedding、补知识库
>
> **我们的实际优化路径（数据）**：
>
> 起点 Naive RAG：召回率 62%
> + 业务精调 embedding：72%（+10）
> + 混合检索（BM25）：80%（+8）
> + Rerank：88%（+8）
> + Query 改写：90%（+2）
> + 多 Index + 路由：92%（+2）
> + 失败回流飞轮（3 个月）：94%
>
> **几个反直觉经验**：
>
> 1. **embedding 业务精调是 ROI 最高的一步**——比换大模型便宜得多
>
> 2. **不要无脑加 top-K**——召回越多噪声越多、rerank 越累，30-50 是甜点
>
> 3. **失败 case 比成功 case 重要**——召回成功的下次还成功，失败的需要补
>
> 4. **rerank 的提升比直接换大模型大**——便宜十倍效果一样
>
> 5. **Query 改写的成本要算账**——简单 query 不上改写省 200ms
>
> **不要做的事**：
>
> - 盲目加召回路（每加一路成本翻倍但收益递减）
> - 太细的分块（200 token 以下，语义信号不足）
> - 用单一指标判定（看 Recall@5 不看 Faithfulness 会被骗）
> - 不做评测就调参（瞎调）
>
> 一句话：**RAG 准确率优化是渐进式工程**——查询、索引、检索、重排、评测五层组合拳，每层 3-15 个点，加起来 30+ 个点的提升。
