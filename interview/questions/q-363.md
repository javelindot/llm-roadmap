# Chroma 的检索原理？

## 答案

> **Chroma 本质是 SQLite + 向量索引（HNSW）的轻量级嵌入式向量库**——单进程内存索引 + 持久化到本地文件，适合中小规模 RAG，**不适合生产高并发**。
>
> **架构**：
>
> ```
> Python API
>   ↓
> Chroma Core (Python)
>   ↓
> ┌──────────────┐  ┌─────────────┐
> │  HNSW Index  │  │  SQLite DB  │
> │  (内存向量)   │  │  (元数据)    │
> └──────────────┘  └─────────────┘
> ```
>
> **检索流程**：
>
> 1. **入库阶段**
>    - 计算 embedding（用户提供或 Chroma 内置 sentence-transformers）
>    - 向量存入 HNSW 索引（内存）
>    - 元数据 + 原文存 SQLite
>    - 持久化到 `./chroma` 目录
>
> 2. **查询阶段**
>    - Query 转 embedding
>    - HNSW 索引算最近邻（默认 cosine 距离）
>    - 根据 metadata filter 过滤（SQLite where）
>    - 返回 top-K
>
> **核心组件**：
>
> **1. HNSW（Hierarchical Navigable Small World）**
>    - 多层近似图索引
>    - 查询复杂度 O(log N)
>    - 默认参数：M=16，ef_construction=200
>
> **2. SQLite 存元数据**
>    - 轻量、嵌入式
>    - 不需要单独部署
>    - 适合中小规模
>
> **3. Collection**
>    - 类似 Milvus 的 collection
>    - 一个 collection 一个 HNSW 索引
>
> **代码示例**：
>
> ```python
> import chromadb
>
> client = chromadb.PersistentClient(path="./chroma_db")
> coll = client.get_or_create_collection("docs")
>
> # 入库
> coll.add(
>     documents=["doc1 text", "doc2 text"],
>     embeddings=[[0.1, 0.2, ...], [0.3, 0.4, ...]],
>     metadatas=[{"source": "a"}, {"source": "b"}],
>     ids=["1", "2"]
> )
>
> # 查询
> result = coll.query(
>     query_embeddings=[[0.15, 0.25, ...]],
>     n_results=5,
>     where={"source": "a"}  # 元数据过滤
> )
> ```
>
> **Chroma vs 其他向量库**：
>
> | 维度 | Chroma | Milvus | Qdrant | FAISS |
> |------|--------|--------|--------|-------|
> | 部署 | 嵌入式 | 集群 | 单机 / 集群 | 嵌入式 |
> | 持久化 | SQLite | 持久 | RocksDB | 自管 |
> | 性能 | 中 | 高 | 高 | 高 |
> | 并发 | 低 | 高 | 高 | 中 |
> | 元数据过滤 | 支持 | 强 | 强 | 不支持 |
> | 学习成本 | 极低 | 中 | 中 | 低 |
> | 适用规模 | 100w 级 | 亿级 | 千万级 | 看实现 |
>
> **何时用 Chroma**：
>
> 1. **POC / Demo**——快速跑通，单机
> 2. **个人项目**——不要复杂部署
> 3. **教学 / 学习**——理解向量检索
> 4. **小规模生产**（< 100w 文档）
> 5. **LangChain 默认 demo**
>
> **何时不用 Chroma**：
>
> - 高并发生产（QPS 100+）
> - 数据量 >100w
> - 需要分布式 / 多副本
> - 需要严格的可用性 SLA
>
> 我们跨境支付**没用 Chroma 上生产**——选 Milvus 因为：
> - 千万级数据
> - 100+ QPS 并发
> - 多副本高可用
> - 元数据 filter 性能
>
> **Chroma 的局限**：
>
> 1. **单进程内存索引**——重启加载慢、内存吃满
> 2. **无原生分布式**——加副本要靠 K8s
> 3. **写入并发差**——SQLite 单写
> 4. **元数据复杂查询弱**——SQLite 简单 where，不支持复杂 expr
> 5. **大规模时性能急剧下降**
>
> **几个工程经验**：
>
> 1. **POC 用 Chroma，生产换 Milvus / Qdrant**——迁移成本低
>
> 2. **Chroma client 模式 vs 持久化模式**——前者纯内存，后者落盘
>
> 3. **Chroma + LangChain** 是新人入门组合
>
> 4. **Chroma 重启慢**——大数据集加载几分钟
>
> 一句话：**Chroma = 嵌入式 HNSW + SQLite**——POC / 学习 / 小规模生产神器，但工业级需求请用 Milvus / Qdrant。
