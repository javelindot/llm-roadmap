# MLA是怎么做的？为什么它比LoRA快？同样是低秩分解，为啥推理时LoRA慢而MLA快？

## 答案

先纠正前提：MLA（Multi-head Latent Attention）是 Attention 架构，目标是压缩 KV Cache；LoRA 是参数高效微调方法，目标是用低秩增量训练权重。两者都出现低秩分解，但优化对象不同，不能简单比较“谁更快”。

### 1. MLA 怎么做

标准 MHA 对每个历史 token 缓存所有头的 K、V。MLA 先把隐藏状态压缩为低维潜变量：

\[
c_t^{KV}=W^{DKV}h_t
\]

再从潜变量上投影出各头的 K、V：

\[
k_t=W^{UK}c_t^{KV},\qquad
v_t=W^{UV}c_t^{KV}
\]

推理时主要缓存低维 \(c_t^{KV}\)，而不是完整 K/V。通过矩阵吸收，可避免每一步显式恢复全部 key 投影。RoPE 与低秩吸收不完全兼容，因此 MLA 将 key/query 分为可吸收的内容部分和较小的解耦 RoPE 部分。

[DeepSeek-V2](https://arxiv.org/abs/2405.04434) 报告 MLA 将每 token KV Cache 压缩为很小的 latent 表示，从而降低自回归 decode 时反复读取历史缓存的 HBM 带宽。

### 2. LoRA 为什么可能慢

LoRA 前向为：

\[
y=Wx+\frac{\alpha}{r}BAx
\]

若在线保留独立 adapter，会增加两次小矩阵乘、kernel launch 和多 adapter 调度开销。但部署单一 adapter 时，可先合并：

\[
W'=W+\frac{\alpha}{r}BA
\]

此后前向与原线性层相同，通常没有额外 LoRA 算子。因此“LoRA 推理一定更慢”是错误的。

### 3. MLA 为什么能加速

LLM decode 经常受 KV Cache 显存容量和带宽限制。MLA 从架构上减少每个 token 必须存储与读取的状态，所以可提高并发和 decode 吞吐。收益不是因为“低秩计算总比全秩快”，而是因为**把低秩结构设计成了可缓存、可吸收的推理表示**。

需要注意，MLA 的单 latent head 在高 TP 场景存在分片瓶颈，2026 年的 [MLRA](https://arxiv.org/abs/2603.02188) 等工作正尝试改善这一点，说明 MLA 也不是所有硬件拓扑下都自动最快。

### 面试总结

MLA 低秩压缩的是每 token 的 K/V 状态，主要节省 KV Cache 和 HBM 带宽；LoRA 低秩表示的是权重增量，合并后可无额外前向开销。MLA 更快的根本原因是推理数据流优化，而不是低秩分解这个名字。
