# MLA具体是怎么做的？可以加快推理速度吗？

## 答案

MLA（Multi-head Latent Attention）通过低秩压缩把每个 token 的 K/V 表示存成较小 latent，降低 KV Cache 容量和 Decode 读取带宽。它通常能提高长上下文和高并发推理效率，但“缓存减少多少”不等于“端到端速度同比提升”。

### 1. 标准 MHA

设有 `n_h` 个 head，每个 head 维度为 `d_h`，标准 MHA 每层每 token 需要缓存：

```text
K + V = 2 * n_h * d_h
```

FP16/BF16 下再乘 2 bytes。请求数、层数和上下文长度都线性放大缓存。

### 2. MLA

先把 hidden state 下投影到低维：

```text
c_t^KV = W_DKV h_t
```

需要时再上投影：

```text
K_t^C = W_UK c_t^KV
V_t^C = W_UV c_t^KV
```

同时将位置部分拆出：

```text
K_t = [K_t^C ; RoPE(K_t^R)]
Q_t = [Q_t^C ; RoPE(Q_t^R)]
```

缓存主要变为：

```text
c_t^KV + K_t^R
```

其维度远小于展开后的多头 K/V。

### 3. 矩阵吸收

在 Decode 阶段，可把部分上投影矩阵吸收到 Query 或输出投影中，使计算直接围绕 latent 展开，避免为所有历史 token 显式恢复完整 K/V。Decoupled RoPE 用较小独立分量保留位置信息。

### 4. 是否加速

**更可能明显收益的场景**

- 长上下文；
- 高并发连续批处理；
- Decode 占主导；
- 服务受显存容量或带宽限制；
- 推理框架有原生 MLA/fused kernel。

**收益可能有限的场景**

- 短 Prompt、低 batch；
- Prefill 计算占主导；
- kernel 未优化，频繁做额外 reshape/投影；
- 通信、采样或外部工具才是瓶颈。

因此要分别测 TTFT、TPOT、tokens/s、最大并发、显存占用和 P95 延迟。

### 5. 与其他方案比较

- MQA/GQA：通过共享 K/V head 降缓存，生态成熟；
- MLA：低秩 latent，压缩更激进、表达更灵活；
- KV 量化：降低每个缓存元素的字节数，可与架构优化组合；
- PagedAttention：解决缓存分配和碎片，不改变单 token 表示维度。
