# SFT+DPO训练怎么组织这部分数据的?是自己构造还是用公开数据?

## 答案

SFT 与 DPO 数据目标不同：SFT 教模型“怎样完成任务”，DPO 教模型“两个可行回答中更偏好哪一个”。生产数据通常是公开数据打底、业务数据决定方向、合成数据扩覆盖，三者都需验证。

### 1. SFT 数据

格式：

```json
{
  "prompt": "用户请求和上下文",
  "response": "高质量示范"
}
```

来源：

- 专家编写；
- 脱敏历史优质对话；
- 文档/SOP 转换；
- 可执行工具轨迹；
- 规则可验证的合成样本；
- 有明确许可证的公开 instruction 数据。

SFT 应覆盖正常任务、澄清、无答案、拒答、工具和边界场景。

### 2. DPO 数据

格式：

```json
{
  "prompt": "同一个输入",
  "chosen": "更好的回答",
  "rejected": "较差但有区分价值的回答"
}
```

偏好对可来自：

- 同一模型多次采样后由人工排序；
- 新旧模型或不同策略的回答；
- 线上人工改写前后；
- 规则/verifier 选出正确与错误候选；
- 强模型生成候选，再由独立验证筛选；
- 公开偏好数据用于通用对齐。

Rejected 不能全是明显垃圾，否则模型学不到细粒度偏好。应包含事实、帮助性、安全、格式、工具和风格等具体差异。

### 3. 推荐组织方式

```text
公开 SFT
+ 高质量领域 SFT
+ 边界/工具数据
-> SFT checkpoint

同分布 prompt
+ hard chosen/rejected
+ 安全偏好
-> DPO
```

SFT 与 DPO 的 prompt 分布应接近上线任务，但最终测试集完全隔离。DPO 的 chosen/rejected 长度、模板和来源要平衡，避免模型只学习“更长”或某种措辞。

### 4. 自构造还是公开数据

**公开数据**

- 优点：规模大、覆盖通用能力；
- 风险：许可证、污染、风格不一致和业务不匹配。

**自建数据**

- 优点：贴近业务、可覆盖工具与边界；
- 风险：数量小、标注成本高、组织偏差。

合理方式不是二选一：公开数据保持通用能力，自建数据决定业务行为，合成数据补长尾；按质量权重、采样比例和 curriculum 混合。

### 5. 数据治理

- 用户、商户、文档簇和时间分组切分；
- 去重与 benchmark contamination；
- PII 脱敏、授权和删除；
- 记录生成模型、标注者、规则和版本；
- chosen/rejected 多标注者一致性；
- 工具轨迹在沙箱重放；
- 对金额、代码、JSON 和引用做程序校验。

### 6. 评测

- SFT 前后任务与格式能力；
- DPO win rate、事实、安全和长度偏差；
- 通用能力回归；
- 多语言、长尾、越权和无答案；
- 线上 shadow/A-B；
- KL、输出长度、拒答率和多样性。

### 7. 跨境支付场景

SFT 示范如何查询订单、引用规则和转人工；DPO 可偏好“事实准确、合规、给出下一步”的回答，拒绝虚构到账时间、泄露风控阈值或跳过审批的回答。资金动作仍由系统硬约束。

### 8. 面试总结

SFT 用单条黄金回答，DPO 用同 prompt 的 chosen/rejected。公开数据负责通用覆盖，自建数据负责业务匹配，合成数据负责扩展；最终效果取决于验证、无泄漏切分和困难偏好对。
