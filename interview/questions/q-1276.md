# deepspeed 的每一段的通信比较,zero3分别是0和2的多少倍

## 答案

在经典 ZeRO 通信量分析中，如果用 `P` 表示完整模型参数或梯度的数据量，并忽略拓扑、协议常数和通信计算重叠，那么：

```text
ZeRO-0 / 传统数据并行：约 2P
ZeRO-1：              约 2P
ZeRO-2：              约 2P
ZeRO-3：              约 3P
```

因此：

```text
ZeRO-3 / ZeRO-0 ≈ 3P / 2P = 1.5
ZeRO-3 / ZeRO-2 ≈ 3P / 2P = 1.5
```

结论是：在标准假设下，ZeRO-3 的通信量约为 ZeRO-0 和 ZeRO-2 的 **1.5 倍**，不是 3 倍。

### 1. 为什么 ZeRO-0 是 2P

传统数据并行中，每张卡保留完整参数、梯度和优化器状态。反向结束后对完整梯度做 AllReduce。Ring AllReduce 可以理解为：

```text
ReduceScatter：约 P
AllGather：    约 P
合计：         约 2P
```

更精确地考虑数据并行度 `N` 时，每个 rank 的流量约为：

```text
2 * (N - 1) / N * P
```

### 2. ZeRO-1 和 ZeRO-2 为什么仍约为 2P

**ZeRO-1**

- 只切分 optimizer states；
- 梯度仍需要跨卡聚合；
- 各 rank 更新自己负责的参数分片后，再同步完整参数。

**ZeRO-2**

- 切分 optimizer states 和 gradients；
- 梯度用 ReduceScatter 直接落到负责该分片的 rank；
- 更新完成后 AllGather 参数。

所以二者的主要通信仍可近似为：

```text
梯度 ReduceScatter + 参数 AllGather ≈ 2P
```

ZeRO-2 比 ZeRO-1 更省梯度显存，但不意味着通信量自动减半。

### 3. ZeRO-3 为什么是 3P

ZeRO-3 进一步切分 parameters。某层计算前需要临时收集该层参数：

```text
前向参数 AllGather：约 P
反向参数 AllGather：约 P
梯度 ReduceScatter：约 P
合计：               约 3P
```

反向阶段通常还要再次 AllGather 参数，因为前向用完后参数可能已经释放，不能假设永久保留完整参数。

### 4. 为什么实测不一定正好 1.5 倍

`1.5x` 是通信字节量的理论近似，不等于训练耗时也是 `1.5x`。实测还受以下因素影响：

- bucket 大小和参数持久化阈值；
- 通信与计算是否有效 overlap；
- gradient accumulation；
- activation checkpointing 导致的重计算；
- 参数是否在多个算子间复用；
- 节点内 NVLink 与跨节点网络拓扑；
- CPU/NVMe offload；
- 小张量通信和 latency；
- profiler 统计的是逻辑数据量还是链路实际字节。

ZeRO-3 的价值主要是显著降低单卡模型状态显存，使更大模型或更大 batch 可以训练；是否值得增加通信，要结合模型是否能放下、网络带宽、吞吐和成本判断。
