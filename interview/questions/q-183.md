# 召回阶段时的召回策略是怎样的？

## 答案

> 我们用**多路召回 + RRF 融合 + Rerank** 的三段式策略：
>
> **第一段：多路并行召回**
> - **向量检索**：bge-large-zh-v1.5（业务精调）→ top-30
> - **BM25 关键词检索**：ES 倒排 → top-30
> - **元数据过滤**：用户的国家、币种、业务线先过滤候选集
> - 同时还有一路 **HyDE 假设回答检索**（关键场景才上）
>
> **第二段：融合**
> - 用 **RRF（Reciprocal Rank Fusion）** 合并向量和 BM25 的结果
> - 公式：score = Σ 1/(k + rank)，k=60
> - 不需要分数归一化，跨检索器友好
> - 融合后取 top-30
>
> **第三段：Rerank**
> - **bge-reranker-large** 对 top-30 做精排
> - 取最终 **top-5** 喂 LLM
> - threshold 0.3 以下直接丢，避免低质 doc 污染
>
> **为什么这样设计**：
> - 向量召回语义、BM25 召回精确关键词（业务术语命中）—— **互补**
> - 元数据预过滤把候选集缩小，提精度
> - rerank 是精度的关键放大器（+10 个点）
>
> **几个工程参数**：
> - 向量 top-30 + BM25 top-30 → 融合 → top-30 → rerank → top-5
> - 总召回延迟约 300ms（向量 80ms + BM25 50ms + rerank 150ms + 余下）
> - 单 query embed 缓存 60s，重复 query 直接 hit
>
> **失败 case 类型**：
> - 召回失败（query 太短 / 知识库没有）→ LLM 回 "我不知道"
> - 召回到但 rerank score 太低 → 丢，避免编
>
> 整体召回率 92%（500 条黄金集 + 1000 条线上抽样）。
