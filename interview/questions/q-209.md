# 实际使用过程中若大语言模型出现异常，你会采取切换模型还是其他优化处理方式？

## 答案

> 不是单纯切换模型，是**多层降级**——异常的原因不同，治理动作也不同。
>
> **先分清楚"异常"是什么**：
>
> | 异常类型 | 表现 | 治理 |
> |---------|------|------|
> | 服务不可用 | 504 / 502 / OOM | 切备用模型 + 重试 |
> | 输出格式错 | JSON 解析失败 | 重试 + format prompt 重写 |
> | 输出内容错 | 幻觉 / 拒答 | RAG 兜底 + 多模型投票 |
> | 性能下降 | p95 飙升 | 限流 + 缩短上下文 |
> | 余额 / 鉴权 | API key 错 | 立即告警 + 切备用 |
>
> **我们的多层降级方案**：
>
> **L0：主模型**
> - Qwen-14B 本地 vLLM 部署
> - 正常情况承担 100% 流量
>
> **L1：同模型重试**
> - 单次超时（>10s）或 5xx → 重试 1 次，指数退避
> - 重试还失败 → 走 L2
>
> **L2：切备用模型**
> - 主模型挂了 → 切到 Qwen-7B（小模型兜底）
> - 优势：质量稍降但延迟更低，业务能继续
> - 通过 service mesh 切流量
>
> **L3：模板回复 / 转人工**
> - 备用模型也挂 → 走预设模板
> - 客服场景："您的问题已收到，正在为您转人工"
> - 风控场景：保守拒绝（safe-fail）
>
> **L4：完全静默**
> - 极端情况（基础设施级故障）→ 接入老的规则引擎 / 旧 API
> - 业务侧不感知 AI 故障
>
> **几个工程细节**：
>
> 1. **限流保护**：单租户 QPS 100 上限，防一个用户打爆全局
>
> 2. **熔断器**：主模型连续 5 次失败 → 自动切备用 30 秒
>
> 3. **多模型投票**：关键场景（风控决策）主备同时跑，结果不一致转人工
>
> 4. **trace + 告警**：所有异常都进 OpenTelemetry，p95 超阈值企微告警
>
> 5. **每月演练**：手动关掉主模型，验证整套降级链路可用
>
> **不切模型的常见优化**：
> - prompt 缩短 / cache 命中
> - temperature 调低
> - 上下文截断（保留关键部分）
> - 流式输出（首字节延迟感知）
>
> 一句话：**异常治理是多层冗余 + 业务无损**——切模型只是其中一环，不能只靠这个。

**🪝 顺嘴带出去的钩子**：
- "熔断器在我们 LLM 服务里的具体阈值可以聊。"
- "主备模型一致性检查的设计可以展开。"
- "演练流程怎么不影响生产可以讲讲。"
