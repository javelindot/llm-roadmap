# 传统Attention存在哪些问题？

## 答案

**一句话**：传统 Attention 的主要问题是计算和显存复杂度高，长序列下成本随序列长度平方增长。

### 核心回答
传统 self-attention 要计算 `QK^T`，如果序列长度是 `n`，注意力矩阵大小是 `n x n`。

问题包括：
- **时间复杂度高**：`O(n^2)`。
- **显存占用高**：注意力矩阵很大。
- **长上下文成本高**：上下文翻倍，attention 成本接近平方增长。
- **推理 KV Cache 压力大**：生成越长缓存越大。
- **对无关 token 也计算**：很多 token 之间其实没必要全连接。

优化方向：FlashAttention、稀疏注意力、滑动窗口注意力、线性注意力、MQA/GQA、上下文压缩等。

### 1. 为什么传统 Attention 成本高

Self-Attention 要计算所有 token 两两之间的相关性。如果序列长度是 `n`，注意力矩阵就是 `n x n`。这意味着上下文长度翻倍，attention 计算和显存开销接近四倍增长。

这对长文档、长对话、代码、日志分析、RAG 拼接上下文都很不友好。

### 2. 常见优化方向

- **FlashAttention**：通过 IO-aware 方式减少显存读写，提高 attention 计算效率。
- **稀疏/滑窗注意力**：只关注局部或部分关键 token。
- **MQA/GQA**：减少 KV head，降低 KV Cache。
- **上下文压缩**：从业务侧减少输入 token。
- **RAG 精准召回**：只把相关证据放进上下文。

### 3. 面试总结

传统 Attention 的问题不是不能用，而是在长上下文和高并发下成本太高。工程上要模型优化和业务侧上下文治理一起做。

### 4. 结合跨境支付项目怎么说（面试口径）

> 在跨境支付的 RAG 或风控助手里，这个问题很现实。业务方经常希望把所有交易流水、规则文档、历史对话都塞给模型，但 attention 成本会很高，而且无关信息多了还会干扰判断。
>
> 我的做法是业务侧先控上下文：交易流水先做统计和异常筛选，规则文档通过 RAG 召回 top 证据，历史对话做摘要。模型侧再用 FlashAttention、GQA、vLLM 这些优化。
>
> 所以我会说，长上下文不是单纯靠模型硬吃，支付场景更需要先把证据筛干净，再交给模型推理。

**钩子**：长上下文成本不只是模型问题，也是检索和上下文工程问题。
