# Agent整体流程是怎么做的？包括哪些模块？

## 答案

生产级 Agent 可以分为入口、理解规划、知识与工具、执行控制、状态记忆、结果校验和治理观测七个模块。典型流程是：

> 请求接入 → 身份鉴权 → 意图与实体识别 → 任务规划/路由 → 知识检索与工具执行 → 结果校验 → 生成回复或人工审批 → 状态持久化与反馈。

### 1. 接入和安全上下文

入口层负责：

- 用户身份、角色、租户和商户鉴权；
- 限流、幂等键、trace ID 和会话 ID；
- PII 脱敏、内容安全和提示注入初筛；
- 加载当前用户允许访问的数据范围和工具权限。

安全上下文由后端注入，不能让模型从用户文本中生成或修改：

> `user_id、role、tenant_id、merchant_id、session_id、data_scope、request_id`

### 2. 意图理解与任务拆分

理解层完成：

- 多轮指代消解；
- 意图分类和多意图拆分；
- 订单号、商户、国家、币种、通道、产品和错误码抽取；
- 判断信息是否完整，缺失时先追问；
- 区分知识问答、实时查询、复杂调查和资金动作。

例如“这笔 USD 付款为什么失败，退款多久到账”应拆为交易失败调查和退款规则查询两个子任务，分别走工具和 RAG。

### 3. Planner、Router 和状态图

简单流程使用规则或分类器路由，复杂流程使用显式状态图。状态至少包含：

> `messages、intent、entities、plan、current_step、evidence、tool_results、risk_level、retry_count、approval_state、final_answer`

LangGraph 可用节点和条件边表达：

1. 解析请求；
2. 选择知识库或工具；
3. 并行收集证据；
4. 判断证据是否充分；
5. 继续调查、追问、生成或转人工；
6. 输出后写入 trace 和反馈。

固定业务规则放在代码节点中，开放式判断才交给模型。这样流程可回放、可测试，也能限制循环次数和错误路径。

### 4. 知识与工具层

知识层负责产品文档、国家/币种规则、KYC/KYB、错误码、拒付和结算 SOP，使用混合检索、元数据过滤和 rerank。

工具层负责实时事实：

- 订单、账户和账务查询；
- 通道回执与健康状态；
- 汇率报价和结算批次；
- 工单、风控与合规状态。

每个工具使用明确 JSON Schema、超时、重试、熔断和版本；工具服务端再次做权限与参数校验。Agent 只能看见当前角色所需的最小工具集。

### 5. 执行控制和人工介入

只读、互不依赖的任务可以并行执行；存在数据依赖和资金状态变化的步骤必须串行。

执行层负责：

- 工具参数校验和最小权限；
- 超时、有限重试和 fallback；
- 幂等、状态版本和重复调用拦截；
- 最大步骤、token、时间和费用预算；
- 失败补偿和人工接管。

查询与解释可以自动完成。付款、退款、冻结、解冻、修改收款账户等高风险动作必须暂停流程，展示对象、金额、币种、依据和风险，等待人工审批；审批后后端仍需重新校验。

### 6. 结果校验与回复生成

生成前检查：

- 证据是否覆盖问题；
- 实时工具与知识文档是否冲突；
- 订单、金额、币种和状态是否一致；
- 是否包含不可披露的风控和客户信息；
- 是否应追问、拒答或转人工。

最终回复采用结构化内容：当前状态、依据、原因、下一步、引用和人工处理状态。工具事实和知识解释分开呈现，禁止模型自行补全资金状态或到账时间。

### 7. 状态、记忆和持久化

短期会话状态、长期偏好记忆和权威业务事实分开保存。长期记忆只写入经过筛选的稳定偏好，不保存卡号、证件、完整交易明细和风控阈值。

复杂任务使用 checkpoint 保存每一步状态。LangGraph 的持久化支持 thread、checkpoint、失败恢复、人工中断和历史回放；持久化强度需要在性能与一致性之间选择。支付写操作还要依赖业务流水和状态机，不能把 checkpoint 当作资金事实来源。

### 8. 可观测性与评测

全链路记录：

- 意图、计划、节点和状态迁移；
- 检索候选与最终证据；
- 工具参数、结果、耗时和错误；
- 模型、Prompt、知识和工具版本；
- 拒答、转人工和审批原因；
- token、成本和 P95/P99。

离线评测任务成功率、工具选择、参数正确率、trajectory、安全和稳定性；线上观察首次解决率、人工改写率、转人工率、投诉率、错误资金承诺和单次成功解决成本。

Agent 的核心不是让模型自由循环，而是把模型能力嵌入可鉴权、可恢复、可审批、可观测的业务工作流。
