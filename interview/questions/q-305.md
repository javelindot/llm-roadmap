# RAG 的范式？如何搭建？

## 答案

> **三大范式**：Naive RAG（朴素）→ Advanced RAG（高级）→ Modular RAG（模块化）。我们跨境支付从 Naive 演进到 Modular，搭建过程也是这个演进。
>
> **范式 1：Naive RAG（最早期）**
>
> ```
> 文档 → 切块 → embed 入库
> 用户 query → embed → 检索 top-K → 拼 prompt → LLM 生成
> ```
>
> 优点：最简单、能跑通
> 缺点：召回烂、生成幻觉、无评测
>
> **范式 2：Advanced RAG（主流）**
>
> 在 Naive 基础上加各种优化：
> - **检索前**：Query 改写、HyDE、多 query Fusion
> - **检索**：混合检索（向量 + BM25）、元数据过滤
> - **检索后**：Rerank、压缩、过滤
> - **生成**：引用强约束、Self-Critique
> - **评测**：RAGAS、黄金集
>
> 这是大多数生产 RAG 系统的状态。
>
> **范式 3：Modular RAG（前沿）**
>
> 把 RAG 拆成可插拔模块，按场景组合：
> - **路由模块**：多知识库选择
> - **改写模块**：Query 改写、扩展
> - **检索模块**：向量 / BM25 / Graph / SQL
> - **重排模块**：Cross-Encoder / LLM Judge
> - **生成模块**：直接生成 / Self-RAG / Active Retrieval
> - **后处理模块**：引用注入、事实校验
> - **记忆模块**：长期记忆、对话历史
>
> Modular 让 RAG 系统像微服务一样可组合、可替换。
>
> **怎么搭建（实战路径）**：
>
> **Phase 1：跑通基线（1-2 周）**
>
> 选型 + POC：
> - LLM：Qwen-14B
> - Embedding：bge-large-zh
> - 向量库：Milvus
> - 编排：LangChain（早期）/ LangGraph（推荐）
>
> 最小可用流程：
> ```python
> async def naive_rag(query: str) -> str:
>     query_emb = await embed(query)
>     chunks = await milvus.search(query_emb, top_k=5)
>     prompt = f"基于以下内容回答：{chunks}\n问题：{query}"
>     return await llm.invoke(prompt)
> ```
>
> **Phase 2：检索质量提升（2-4 周）**
>
> - 混合检索（向量 + BM25）
> - Rerank 精排
> - Query 改写
> - 元数据 + 多 Index
> - 召回率从 65% → 85%
>
> **Phase 3：生成质量 + 评测（2-4 周）**
>
> - 引用强约束 prompt
> - 拒答机制
> - RAGAS 评测
> - 黄金集 + 线上监控
>
> **Phase 4：业务定制（4-8 周）**
>
> - Embedding 业务精调
> - 多语言支持
> - 多模态（PDF 图片）
> - 失败 case 飞轮
>
> **Phase 5：模块化重构（持续）**
>
> 把每个模块拆成独立服务 / 库：
> - QueryRewriter Service
> - HybridRetriever Service
> - Reranker Service
> - Generator Service
> - 业务编排只调用模块
>
> **我们跨境支付的最终架构**：
>
> ```
> 用户 query
>   ↓
> [路由分类]（5 知识库选 1-3）
>   ↓
> [多轮改写]（消解代词）
>   ↓
> [混合检索]（向量 + BM25 并行 → RRF 融合）
>   ↓
> [Rerank]（bge-reranker top-30 → top-5）
>   ↓
> [Prompt 拼装]（含引用约束）
>   ↓
> [Qwen-14B 生成]
>   ↓
> [事实校验]（NLI）
>   ↓
> [失败回流] + [线上监控]
> ```
>
> **几个搭建经验**：
>
> 1. **不要一上来追求 Modular**——先 Naive 跑通，再渐进式优化
>
> 2. **检索质量是上限**——80% 时间应该花在召回 / 重排，不是 prompt
>
> 3. **必须有评测闭环**——黄金集 + RAGAS 是基础设施，不是奢侈品
>
> 4. **失败回流飞轮**——上线后才是真正的迭代开始
>
> 5. **业务路由 + 多 Index**：一个 index 装一切 → 召回精度差，分开装才对
>
> 一句话：**RAG 搭建是渐进式工程**——Naive → Advanced → Modular，每阶段对应不同业务成熟度。
