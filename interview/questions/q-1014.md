# 如何证明 Agent 比人工客服更好？设计评估方案

## 答案

不能用单一准确率证明 Agent 比人工客服“全面更好”。更合理的结论是：在明确的咨询类型、风险边界和服务水平下，Agent 是否能以更低成本取得不劣于人工的质量，并在响应速度、稳定性或覆盖时间上产生增量价值。

评估采用：

> 离线黄金集 → 影子流量 → 三组随机对照实验 → 延迟结果回收 → 分场景决策。

### 1. 先定义可比较的任务范围

跨境支付客服可以按风险和复杂度分层：

- **L1 高频知识问答**：产品支持范围、材料要求、退款时效、费用说明；
- **L2 只读交易查询**：订单状态、通道回执、到账进度；
- **L3 复杂异常处理**：付款失败归因、账务差异、投诉；
- **L4 资金与合规动作**：退款、重新付款、冻结、解冻、风控申诉。

L1、L2 可以评估全自动 Agent；L3 更适合 Agent 辅助人工；L4 默认由人工审批。把所有场景混在一起比较，会因为任务难度和风险分布不同得到误导性结论。

每类任务先定义“解决成功”：

- 答案事实正确且引用了有效规则；
- 查询到了正确的订单、商户和通道状态；
- 没有错误承诺到账时间、汇率或处理结果；
- 必要信息不足时能正确追问；
- 超出权限或证据不足时正确转人工；
- 对话结束后业务状态与预期目标状态一致。

### 2. 建立离线黄金集

从真实客服会话中分层抽样并脱敏，覆盖：

- 国家、币种、语言、支付产品和商户类型；
- 高频问题与长尾问题；
- 多轮指代、信息缺失和多意图请求；
- 过期规则、冲突文档和实时状态变化；
- 提示注入、越权查询、诱导泄密和重复资金动作；
- 投诉、拒付、风控和其他高风险边界。

由支付业务专家和客服共同标注：

- 标准答案或允许的答案要点；
- 必须引用的知识和允许调用的工具；
- 期望的最终业务状态；
- 应追问、拒答或转人工的条件；
- 严重错误等级。

评测不能只看最终文本，还要检查 Agent 的完整 trajectory，包括工具选择、调用顺序、参数、状态变化和转人工判断。同一用例重复运行多次，使用 task success 和 `pass^k` 观察稳定性，避免一次成功掩盖随机性。

### 3. 离线指标分为五组

**质量指标**

- 任务成功率、事实准确率、引用正确率；
- 工具选择和参数准确率；
- 首次解决率 FCR、重复咨询率；
- 人工对答案的改写率。

**用户体验指标**

- 首字时间、平均响应时间、完整解决时间；
- CSAT、投诉率、会话中断率；
- 多语言表达质量和用户追问次数。

**效率与成本指标**

- Agent 自动承接率、转人工率；
- 人工平均处理时长 AHT、并发处理能力；
- 单次会话成本和单个成功解决成本。

**安全与合规指标**

- 敏感数据泄漏率、未授权工具调用率；
- 错误资金承诺率、错误执行率；
- 应转人工未转人工率；
- 合规拒答准确率和误拒绝率。

**系统指标**

- P95/P99 延迟、超时率、工具失败率；
- 降级率、模型和知识版本稳定性；
- 峰值流量下的可用性。

金融场景应设置硬性 guardrail。即使 Agent 降低了成本，只要严重错误、数据泄漏或未授权资金动作超过阈值，也不能扩大流量。

### 4. 影子流量验证真实分布

离线通过后，Agent 接收真实请求并完成推理，但结果不展示给用户，也不执行写操作。将 Agent 的回答、工具轨迹和转人工决策与人工客服实际处理结果比较。

影子阶段重点发现：

- 黄金集未覆盖的新意图和表达；
- 线上权限、数据缺失和工具超时；
- 知识版本与业务系统状态冲突；
- Agent 在真实多轮会话中的循环和错误累积；
- 不同国家、币种和语言上的质量差异。

只有影子评测达到预设门槛，才进入对用户可见的实验。

### 5. 线上采用三组随机对照

实验设置三组：

1. **A 组：纯人工客服**，作为当前基线；
2. **B 组：Agent 辅助人工**，Agent 提供检索、摘要和回复草稿，由客服确认；
3. **C 组：全自动 Agent**，只处理通过准入的低风险场景。

随机单元优先使用用户或商户，而不是单条消息，避免同一用户在不同组之间切换造成污染。按意图、国家、语言、商户等级、新老用户和问题难度分层随机，保证三组任务结构可比。

实验前明确：

- 主要指标，例如 FCR 或单个成功解决成本；
- 非劣效界值，例如质量最多允许下降多少；
- 安全、投诉和合规 guardrail；
- 最小可检测效应、统计功效和所需样本量；
- 固定观察窗口和停止规则。

结果报告效应量和置信区间，不只报告“提升百分比”。有用户历史数据时可使用 CUPED 等方差缩减方法提高实验灵敏度，但不能在看到结果后临时修改指标或人群。

### 6. 处理跨境支付的延迟标签

部分回答在会话结束时无法判断是否正确：

- 退款是否在承诺窗口内到账；
- 付款失败解释是否与最终通道回执一致；
- 用户是否再次咨询或投诉；
- 错误指导是否造成资金或合规损失。

因此需要设置即时、短期和成熟标签窗口：

- 即时：回答、引用、工具调用、响应时间；
- 7 天或业务适配窗口：重复咨询、投诉、退款进展；
- 更长窗口：拒付、资金损失和合规事件。

实验不能只看当天 CSAT 就宣布成功。

### 7. 人工评审和自动评审结合

规则校验适合验证订单状态、金额、币种、工具参数和最终数据库状态。LLM judge 可以扩大语义评测规模，但必须先与业务专家评分做一致性校准，并定期抽检漂移。

评审人员应尽量不知道答案来自人工还是 Agent，减少品牌和表达风格偏差。多人标注时计算一致性，并对“事实错误、遗漏条件、错误承诺、合规风险”分别打分，不能只给一个主观总分。

2024 年的 τ-bench 强调通过最终环境状态和 `pass^k` 评估工具型 Agent；2025 年的 τ²-bench 进一步覆盖用户和 Agent 共同操作环境的双向协作；2026 年 AJ-Bench 则继续验证基于环境取证的 Agent-as-a-Judge。它们共同说明：客服 Agent 评估应检查任务过程和真实状态，不能只看回复像不像人工。

### 8. 计算真实业务价值

成本使用全成本口径：

> `Cost_per_resolved = (模型 + 检索 + 工具 + 基础设施 + 人工复核 + 风险损失) / 成功解决数`

同时计算：

- 高峰期排队和放弃率是否下降；
- 夜间及多语言服务覆盖是否提升；
- 人工是否从重复咨询转向复杂工单；
- Agent 辅助是否降低新人培训成本；
- 节省的工时是否真的转化为产能，而不是只发生系统成本转移。

最终不要求 C 组在所有场景替代人工。常见的合理结果是：L1/L2 由 Agent 自动处理，L3 使用 Agent 辅助人工，L4 保持人工审批。只有当质量达到非劣、效率或体验显著改善，并且安全 guardrail 全部满足时，才能证明 Agent 在对应场景具有上线价值。
