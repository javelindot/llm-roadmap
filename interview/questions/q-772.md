# 6.RAG有哪些可优化的地方

## 答案

RAG 优化要按“数据、索引、查询、召回、重排、上下文、生成、评测”逐层定位，不能只换 embedding 模型。

### 数据与分块

提高解析/OCR、去重和元数据质量；按标题、段落、表格结构分块，保留 parent-child 与来源。chunk size 和 overlap 通过任务评测调节。建立增量更新、删除和版本机制。

### 召回与重排

- query rewrite、意图/时间抽取和 multi-query；
- dense + BM25 hybrid retrieval；
- metadata/权限过滤和领域路由；
- hard-negative 训练 embedding；
- cross-encoder/LLM reranker；
- MMR、去重和 parent document expansion。

### 上下文与生成

按证据价值而非相似度直接拼接，控制 token 预算和 lost-in-the-middle；对长文档做层级检索或 contextual compression。Prompt 要求引用、区分证据与指令，并在证据不足时拒答。数学、SQL 和实时信息使用工具。

### 性能与成本

缓存 query embedding、热点检索和公共前缀；向量索引分片、量化和批处理；小 reranker 处理大部分请求，复杂 query 才升级 LLM。各阶段设置超时和降级。

### 评测

分别测 retrieval Recall@K/nDCG、reranker、上下文利用、答案正确性、引用支持率和拒答。用真实 query 与 bad case 分桶，记录失败属于解析、召回还是生成。

只有端到端评测能判断优化是否有效，例如召回率提高但加入噪声导致答案下降，就需要重排而不是继续扩大 top-k。
