# 你们的产品是否使用大模型？如果使用，是如何训练的？是从头开始训练还是进行微调？

## 答案

产品使用大模型，但没有从零预训练。跨境支付公司通常没有必要承担从零训练所需的数据、算力、人才和合规成本，更合理的是“选择合适基座 + RAG/工具 + 按需 SFT/LoRA + 评测与安全治理”。

### 1. 为什么不从头训练

- 通用语言和代码能力已由成熟基座提供；
- 从零训练需要海量高质量 token 和集群；
- tokenizer、数据清洗、并行和稳定性风险高；
- 支付私域数据规模不足以支撑通用预训练；
- 训练数据涉及隐私、PCI、KYC/KYB 和数据驻留。

除非公司目标是建设基础模型平台，否则 ROI 通常不成立。

### 2. 产品分层

```text
轻量分类/Embedding
-> 工单路由、召回、Rerank

通用 Instruct LLM
-> 多语言问答、摘要、报告、Agent

RAG/Tools
-> 规则、订单、账务、费率、汇率和状态

Workflow/Policy
-> 权限、幂等、审批和审计
```

### 3. 什么时候微调

先用 Prompt + RAG + Tool 做基线。以下情况再微调：

- 工具格式长期不稳定；
- 领域术语和多语言理解不足；
- 报告结构、语气和拒答边界明确；
- 有足够高质量、可脱敏数据；
- 评测证明瓶颈在模型行为而非数据源。

通常先 LoRA/QLoRA，容量不足且数据足够时再考虑全参或继续预训练。

### 4. 训练流程

1. 数据脱敏、权限和去重；
2. 按商户/时间/问题簇切分；
3. 构造正常、异常、拒答和工具轨迹；
4. SFT/LoRA；
5. 必要时 DPO 或可验证任务 RL；
6. 事实、工具、安全、延迟回归；
7. 灰度和人工兜底；
8. bad case 回流。

### 5. 不能写进模型的内容

- 实时订单状态；
- 余额和 Ledger；
- 当前汇率/费率；
- 制裁名单；
- 经常变化的国家规则；
- 密钥、卡号和敏感身份信息。

这些由受控系统实时提供。

### 6. 面试表达

不要虚构“自研基础模型”。可以说：

> 我们的核心工作不是从零训模型，而是模型选型、领域数据治理、RAG/Agent、LoRA 和线上评测。这样既保留通用能力，又能让支付事实可更新、可审计和可回滚。
