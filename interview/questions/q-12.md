# Agent创建方式

## 答案

**一句话**：Agent 的创建方式可以从简单到复杂分为 prompt Agent、工具调用 Agent、RAG Agent、Workflow Agent 和 Multi-Agent。

### 1. 常见创建方式
- **Prompt Agent**：只靠角色提示和上下文完成任务，适合简单问答和文本处理。
- **Tool Agent**：给模型绑定函数调用或 API，让它能查数据、执行命令、调用业务系统。
- **RAG Agent**：接入知识库和检索能力，适合企业文档、制度、规则查询。
- **Workflow Agent**：把主流程固化成状态机或工作流，关键节点由 Agent 判断。
- **Multi-Agent**：多个 Agent 分工协作，比如规划、执行、审查、总结。
- **SubAgent**：主 Agent 把专项任务委派给独立上下文的子 Agent。

### 2. 选择原则

选型要看任务复杂度：
- 只是回答知识问题，用 Prompt 或 RAG 就够。
- 需要查实时数据，用 Tool Agent。
- 需要多步骤、可恢复、可审计，用 Workflow Agent。
- 需要角色分工和并行处理，用 Multi-Agent / SubAgent。

不要一上来就做复杂多 Agent，先用最简单可靠的方式覆盖核心流程。

### 3. 面试总结

创建 Agent 的关键不是形式，而是任务需要什么能力：是否需要工具、是否需要记忆、是否需要工作流、是否需要多角色协作。

## 结合项目谈（口语版 · 面试现场约 50 秒）

> 结合跨境支付，我不会一上来就说要做一个很复杂的多 Agent 系统。我会先看业务场景。比如只是问运营制度、拒付处理流程，那用 RAG Agent 接知识库就够；如果要查一笔付款为什么失败，就需要 Tool Agent 去查订单、账务、通道回执；如果是商户准入或可疑交易排查，就更适合 Workflow Agent，因为它有固定阶段，也需要留审计。
>
> 真正复杂的地方，比如风控、合规、通道路由关注点不一样，我才会拆成多个 SubAgent。风控看交易模式和历史行为，合规看国家、名单和监管限制，路由看成本和成功率，最后由主 Agent 汇总。
>
> 所以我理解 Agent 创建方式不是越高级越好，而是从业务复杂度出发。支付场景里我会优先保证流程稳定、数据可查、动作可审，再逐步引入多 Agent 分工。

- "我会按场景从 RAG、Tool、Workflow 再到 Multi-Agent 逐步演进。"
