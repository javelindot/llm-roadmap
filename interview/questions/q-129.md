# 为什么 RAG Fusion 需要生成多个查询？

## 答案

**一句话**：单一 query 容易被"措辞偏置"卡死——RAG Fusion 让 LLM 生成 N 个语义近但表达不同的 query，**多角度检索 + RRF 融合**，让真正相关的文档更容易被召回到 top-k。

### 1. 单查询的痛点
- 用户的表达只是"一个角度"
- 知识库可能用完全不同的措辞写同一件事
- 单 embedding 命中受限于这一对"措辞"的语义距离
- 长尾、多义、术语差异都会让单 query 失败

### 2. RAG Fusion 的流程
```
Query "退款多久"
   ↓ LLM 生成 N 个变体
   ├── "跨境退款到账时间"
   ├── "信用卡退款几个工作日"
   ├── "退付时效"
   └── "钱什么时候退回来"
   ↓ 每个变体独立检索 top-k
   ↓ Reciprocal Rank Fusion 融合
   → 最终 top-5
```

### 3. RRF（Reciprocal Rank Fusion）
$$\text{score}(d) = \sum_q \frac{1}{k + \text{rank}_q(d)}$$
- k 通常 60
- 多 query 的 rank 倒数加权和
- 不需要分数标准化、跨检索器友好

### 4. 为什么"多查询"更好
- **覆盖措辞多样性**：同义、同义但术语不同
- **降低单 embedding 偏差**：多次采样平均化
- **rank 维度叠加信号**：多 query 都命中的 doc 一定相关
- **召回率系统性提升 5-10%**

### 5. 工程取舍
- 成本：每多一个 query 多调一次检索 + LLM 改写一次
- 推荐 3-5 个 query（再多收益递减）
- 简单 query 不需要 fusion
- 关键场景才上

### 6. RAG Fusion vs HyDE vs Query Decomposition
| 方法 | 思路 | 适用 |
|------|------|------|
| **RAG Fusion** | 同一意图多种措辞 | 措辞偏置严重 |
| **HyDE** | 用假设答案 embed | query/doc 语义差距大 |
| **Decomposition** | 拆子问题 | 多步复杂问题 |

### 7. 落地经验
- LLM 生成 query 时 prompt 要求"语义近但措辞多样"
- 生成的 query 要质量过滤（避免 LLM 生成奇怪东西）
- 融合后再过 rerank 进一步提精
- A/B 测试看是否值得多消耗的 token

### 8. 一句话总结
**RAG Fusion = 用多 query 覆盖措辞偏置 + RRF 融合排名**——是"小 LLM 成本换大召回提升"的高 ROI 技巧。

## 结合项目谈（口语版 · 约 25 秒）

> RAG Fusion 我们跨境支付**只在关键场景上**，不是默认开。
>
> **客服 RAG**：不用 Fusion。客户问题大多直接，向量召回 + 多轮改写 + rerank 已经够，再上 Fusion 提升不到 1%，但 LLM 调用次数和延迟翻倍，ROI 不划算。
>
> **风控调查 Agent**：上 Fusion。这种场景每条 query 单价高（要支撑监管决策）、措辞专业难匹配（如"同设备 ID 短时间高频跨境"业务术语和知识库表达可能差很远）。我们让 Qwen-7B 生成 4 个变体 query，分别检索 top-30，RRF 融合后 rerank → top-5。召回率从 79% 涨到 88%（**+9**），延迟从 600ms 涨到 1.2s——风控场景能接受这点延迟。
>
> **KYB 准入审核**：也上 Fusion。审核要查商户类别相关的所有政策、限额、风险提示，多 query 覆盖多角度。
>
> **几个工程坑**：
> - 一开始 LLM 生成的 query 太相似（全都是改个标点的"退款多久了" / "退款时间多久"），没起到多角度作用。prompt 加约束"必须用不同术语和角度"才解决。
> - RRF 的 k=60 是默认，我们试过 k=30 / k=100，差异不大，保持默认就行。
>
> 一句话：**RAG Fusion 是奢侈品不是必需品**——在高价值、低延迟容忍的场景里用，否则单 query + 改写 + rerank 已经够。
