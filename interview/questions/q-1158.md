# 意图识别准确率如何计算或评测？

## 答案

意图识别不能只报一个 Accuracy。评测方案取决于单标签、多标签、层级意图还是开放集识别，并且要把不同误判造成的业务损失纳入指标。

### 1. 单标签意图分类

假设每个样本只有一个真实意图，最基础的准确率为：

```text
Accuracy = 预测正确的样本数 / 总样本数
```

但类别不均衡时，Accuracy 很容易被高频意图主导。例如 80% 的问题都是“查询付款状态”，模型全部预测成该类也能得到 80% 准确率，却无法处理退款、拒付或账户冻结。

因此还应查看混淆矩阵以及每类的 Precision、Recall 和 F1：

```text
Precision = TP / (TP + FP)
Recall    = TP / (TP + FN)
F1        = 2 × Precision × Recall / (Precision + Recall)
```

- **Macro-F1**：每个意图等权，能反映长尾类别表现；
- **Micro-F1**：汇总所有样本，更接近整体流量表现；
- **Weighted-F1**：按类别样本量加权，但仍可能掩盖低频高风险意图。

生产报告至少同时给出 Accuracy、Macro-F1、各类 Recall 和混淆矩阵。

### 2. 多标签与层级意图

一个问题可能同时包含多个意图，例如“这笔付款为什么失败，手续费能退吗”同时涉及付款失败和费用退款。多标签场景可使用：

- Micro/Macro Precision、Recall、F1；
- Exact Match / Subset Accuracy：整组标签完全一致才算正确；
- Hamming Loss：标签位预测错误的比例；
- Jaccard Score：预测标签集合和真实集合的交并比。

如果意图是层级结构，例如：

```text
付款
  ├─ 创建付款
  ├─ 查询状态
  ├─ 付款失败
  └─ 退汇
```

则应分别统计一级、二级准确率以及整条路径完全正确的 Path Accuracy。只预测对“付款”大类但路由错了子意图，不能算完整成功。

### 3. 未知意图和置信度评测

线上一定会出现训练集没有覆盖的问题。模型除了分类，还应能输出 `unknown`、澄清或转人工。开放集评测应加入：

- AUROC、AUPR；
- 指定 TPR 下的 FPR；
- 不同阈值下的 coverage-risk 曲线；
- Expected Calibration Error 或 Brier Score；
- 已知意图误拒率与未知意图误接收率。

置信度不能只取大模型口头给出的百分比。更可靠的做法是使用校准过的分类概率、多个信号融合，或在验证集上按业务成本选择阈值。

### 4. 对话系统还要评测实体和路由结果

意图正确但关键实体错误，任务仍然会失败。例如识别为“查询付款状态”，却提取了错误订单号或币种。因此还要统计：

- 国家、币种、产品、订单号、错误码等 slot/entity 的 Precision、Recall、F1；
- 意图和全部关键实体同时正确的 Joint Goal Accuracy；
- 工具选择正确率、参数完整率和参数准确率；
- 需要追问时的澄清触发准确率；
- 需要人工处理时的转人工 Recall。

尤其要关注成本不对称：把“查询状态”误判为“创建退款”，风险远高于两个知识查询意图之间的误判。可以建立成本矩阵：

```text
Cost-sensitive Score
= 1 - Σ(某类误判数量 × 该误判业务成本) / 归一化基准
```

高风险意图还应设置确定性规则、强鉴权和二次确认，不能仅依赖分类准确率。

### 5. 测试集必须代表真实流量

测试集应从真实线上问题中按渠道、语言、国家、产品、商户类型和意图分层抽样，并满足：

- 同一会话、同一工单及其改写不能跨训练集和测试集；
- 使用时间切分检验新产品、新规则和表达漂移；
- 保留拼写错误、中英混输、缩写、口语、省略和多轮指代；
- 单独建立长尾、高风险、未知意图和对抗样本集；
- 对标签定义模糊的样本进行双人标注和业务仲裁；
- 记录标注一致性，标签体系本身不稳定时不能把问题归咎于模型。

### 6. 离线通过后再验证线上效果

上线前先做历史流量回放和 shadow test，再通过灰度或 A/B 实验观察：

- 下游任务成功率和一次解决率；
- 用户改写、追问和重新打开会话的比例；
- 人工纠正率、转人工率和错误路由率；
- 工具调用失败率与高风险误操作拦截率；
- 各语言、国家和产品分桶是否出现性能退化。

最终目标不是让分类指标看起来漂亮，而是把问题稳定路由到正确的知识、工具或人工流程。离线 Macro-F1 用于发现类别短板，Joint Goal Accuracy 验证端到端理解，成本敏感指标和线上任务成功率决定系统是否真正可用。
