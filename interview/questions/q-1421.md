# 你觉得有代表性/有特色/有难度的是哪一个项目？

## 答案

这题不应选择“技术最多”的项目，而应选择最能证明自己解决过真实问题的项目。推荐结构是：为什么有代表性、难点是什么、自己负责什么、如何验证、结果与反思。

### 1. 参考项目口径

> 我认为最有代表性的是跨境支付智能调查/客服 Agent 项目。它不是简单接一个大模型 API，而是把支付订单、账务、风控、KYC、知识库和工单系统连接起来，让模型完成问题理解、证据检索、工具编排和解释，同时保证资金与合规边界。

### 2. 特色

- 结构化系统与非结构化知识结合；
- 多轮状态管理；
- RAG 与实时工具调用结合；
- 按商户和权限隔离；
- 高风险动作进入人工审批；
- 答案带证据和可追溯日志。

### 3. 难点

**事实一致性**

订单状态随时间变化，不能从静态知识库回答，必须查询实时系统。

**长链路可靠性**

检索、工具、模型和外部通道任何一步都可能超时，需要状态机、重试、幂等和断点恢复。

**安全合规**

不同角色能查看的账户、交易和风控信息不同；模型不能直接修改账务、放款或绕过 KYC。

**评测**

不能只看回答流畅度，要评估工具调用、事实一致、引用、任务完成、风险和延迟成本。

### 4. 自己负责什么

必须按真实经历替换，例如：

- 数据清洗和评测集；
- RAG 分块、召回和 Rerank；
- Agent 状态机和工具 schema；
- SFT/LoRA；
- 推理服务与监控；
- 线上 bad case 分析。

不要把团队工作全部归到自己名下。

### 5. 结果怎么讲

真实数据可按：

```text
baseline
-> 改动
-> 离线指标
-> 线上指标
-> 样本量/周期
-> 风险护栏
```

没有真实线上数字时，应说完成了离线验证、原型或灰度，不要编造百分比。

### 6. 反思

可以说：

- 早期过度依赖单一向量召回；
- 没有区分实时状态与知识文档；
- 只看整体准确率，忽略高风险切片；
- 后续增加结构化过滤、工具验证和人工审批。

### 7. 面试总结

一个有代表性的项目要能展示业务理解、技术深度、工程落地和风险意识。回答中最重要的是说清自己的决策、证据和取舍，而不是堆框架名。
