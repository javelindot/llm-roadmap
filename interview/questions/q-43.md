# 什么是 DeepSpeed ZeRO & DeepSpeed Offload？

## 答案

**一句话**：**ZeRO 是把训练状态在 GPU 之间切分，Offload 是把状态搬到 CPU/NVMe**——本质都是用通信换显存，工业里 ZeRO-2 是甜点、Offload 是兜底。

### 1. ZeRO 三个 stage

| Stage | 切什么 | 显存节省 | 通信增量 |
|-------|--------|---------|---------|
| **ZeRO-1** | Optimizer state（Adam 的 m、v、FP32 master） | 4x | 几乎 0 |
| **ZeRO-2** | + Gradients | 8x | 略增（reduce-scatter） |
| **ZeRO-3** | + Parameters | N x（N 卡数） | 大（前后向都要 all-gather） |

显存账（7B 模型 + Adam + FP16 训练）：
- 朴素 DP：~112 GB
- ZeRO-1：~50 GB
- ZeRO-2：~28 GB
- ZeRO-3：~14 GB（8 卡）

### 2. Offload 两种
- **ZeRO-Offload**（CPU offload）：optimizer state + gradients 放 CPU 内存
- **ZeRO-Infinity**（NVMe offload）：连参数也能放 NVMe
- 代价：每步 GPU↔CPU/NVMe 传输巨慢，**1.5-3x 减速**

### 3. ZeRO 工作原理（以 stage 3 为例）
1. 模型参数切分到 N 张卡
2. 前向：要算第 i 层时 all-gather 参数 → 算完释放
3. 反向：再 all-gather 一次算梯度 → reduce-scatter 回切分态
4. Optimizer step：每张卡只更新自己持有的那部分

### 4. 选型经验
| 场景 | 推荐 |
|------|------|
| 单节点 8 卡 + 模型能塞下 | **ZeRO-2**（甜点） |
| 模型刚好塞不下单卡 | **ZeRO-2 + Gradient Checkpointing** |
| 模型大但跨节点慢 | **TP + PP + ZeRO-1** |
| 单机想跑超大 LoRA 微调 | **ZeRO-2 + Offload** |
| 实在没卡 | **ZeRO-Infinity**（接受慢） |

### 5. ZeRO vs FSDP（PyTorch 原生）
- 思路几乎一样
- FSDP 是 PyTorch 官方实现、API 更"Torch 原生"
- DeepSpeed 配置文件方式、生态成熟
- 业界两者都用，HuggingFace Accelerate 都支持

### 6. 常见坑
- ZeRO-3 跨节点慢得离谱，能避就避
- Offload 慢到训练时间翻倍，最后一招
- DeepSpeed 配置文件难调，要慢慢摸甜点参数
- ZeRO + Gradient Checkpointing 顺序要对，否则报错

### 7. 一句话总结
**ZeRO 是 GPU 内切分、Offload 是出 GPU 搬运**——拿不动模型时先 ZeRO-2，拿不动了再 ZeRO-3，实在不行才 Offload。

## 结合项目谈（口语版 · 约 40 秒）

> ZeRO 是我们跨境支付训练流水线里默认开的东西。**14B 客服 SFT** 我们就是 ZeRO-2 + BF16 + Gradient Checkpointing 单机 8 卡 H100 跑稳的，显存占 60G 左右，batch 32。
>
> 一开始我们也想"既然 ZeRO-3 更省显存，是不是直接上 ZeRO-3 能塞更大 batch"——上了之后训练速度反而慢了 40%。复盘下来是 ZeRO-3 每层前向反向都要 all-gather 参数，单节点也是有通信开销的，14B 没大到必须 ZeRO-3 的程度，ZeRO-2 才是甜点。
>
> Offload 我们专门为一个场景上过：**实习生想在自己单卡 RTX 4090（24G）上微调 13B 模型 LoRA**。这种情况 ZeRO-Offload + QLoRA 是唯一办法，把 optimizer state 和部分参数 offload 到 CPU 内存，跑得起来但训练慢 2-3 倍。实习生说"反正我下班挂着跑一晚上"，挺好。
>
> 现在我们的默认策略：**生产训练 ZeRO-2，实验环境 ZeRO-2 + Offload，70B 调研用 TP+PP+ZeRO-1**。
