# 介绍下问数项目，包括人员和时间

## 答案

**一句话**：问数项目本质是让业务人员用自然语言查询数据，系统把问题转成指标理解、SQL 生成、权限校验、结果解释和可视化。

### 1. 项目背景

很多业务同学不会写 SQL，但经常要问：

- 昨天收款成功率是多少？
- 某个国家付款失败原因分布是什么？
- 本月拒付率有没有升高？
- 哪些通道成本最高？
- 某商户交易量变化如何？

问数项目的目标是降低取数门槛，让业务人员能用自然语言拿到可信的数据结果。

### 2. 核心流程

典型流程是：

1. 用户输入自然语言问题。
2. 识别指标、维度、时间范围和过滤条件。
3. 结合数据字典和表结构生成 SQL。
4. 做权限校验和 SQL 安全检查。
5. 执行查询并返回结果。
6. 生成解释或图表。
7. 用户追问时基于上下文继续分析。

### 3. 人员和周期怎么说

可以按这个口径回答：

> 这类项目一般 4 到 6 人比较合适：后端负责数据服务和权限，算法/AI 负责 NL2SQL 和 RAG，数据同学负责指标口径和数仓表，前端负责问数页面，测试负责用例和回归。PoC 可能 3 到 4 周，生产可用版本通常需要 2 到 3 个月，因为指标口径、权限和 SQL 安全都要打磨。

### 4. 结合跨境支付项目怎么说（面试口径）

> 在跨境支付业务里，问数项目很有价值。比如运营问“上周巴西付款失败率为什么升高”，系统要识别时间是上周，国家是巴西，指标是付款失败率，还要关联失败原因、通道、币种和商户维度。然后生成安全 SQL 查询数仓，最后给出结果和解释。
>
> 我会特别强调权限和指标口径。支付数据涉及商户、金额和交易明细，不同角色能看的范围不同；同时成功率、拒付率、到账时效这些指标必须有统一口径，不能让模型自由发挥。
>
> 所以我会把数据字典、指标定义、表关系放进知识库，模型生成 SQL 前先检索相关口径，生成后再过 SQL 校验和权限过滤。这样业务同学用起来方便，数据团队也能接受。

**钩子**：可以引到“NL2SQL、指标口径、SQL 安全、支付数据权限”。
