# 独立入口，每个子 Agent 是一个独立的微服务，不共享内存、不共享上下文

## 答案

> 这是 Agent 边界划分里最硬的一道防线，**微服务化是工业级 Agent 系统的标准做法**。我们跨境支付的 Agent 架构就是这样。
>
> **微服务化的核心思想**：
>
> 每个 Agent = 一个独立的微服务：
> - 独立的代码仓库 / 部署 / 资源
> - 独立的进程 / 容器
> - 独立的状态存储
> - 通过 HTTP / gRPC API 通信
> - 不共享内存、不共享上下文
>
> **为什么这么做**：
>
> 1. **故障隔离**：一个 Agent 挂不影响其他
> 2. **独立伸缩**：各 Agent 流量不同、独立 HPA
> 3. **独立发布**：改一个 Agent 不影响其他
> 4. **权限隔离**：每个 Agent 独立账号、独立网络
> 5. **资源隔离**：CPU / 内存 / GPU 配额清晰
> 6. **审计清晰**：跨服务调用走标准 RPC 链路，trace 完整
> 7. **团队解耦**：不同团队负责不同 Agent
>
> **架构图**：
>
> ```
>                   ┌──────────────┐
>                   │ API Gateway  │
>                   └──────┬───────┘
>                          ▼
>             ┌─────── Router Agent ───────┐
>             │  (主 Agent，统一入口)         │
>             └──┬──────┬──────┬──────┬─────┘
>                │      │      │      │
>                ▼      ▼      ▼      ▼
>           ┌────────┐┌────────┐┌────────┐┌────────┐
>           │ 客服   ││ 风控   ││ KYB    ││ 报表   │
>           │ Agent ││ Agent ││ Agent ││ Agent │
>           │(独立) ││(独立) ││(独立) ││(独立) │
>           └───┬───┘└───┬───┘└───┬───┘└───┬───┘
>               │       │       │       │
>           ┌───▼──────────────────────▼───┐
>           │     共享基础设施（中间件）       │
>           │ Milvus / ES / Redis / MQ      │
>           └────────────────────────────────┘
> ```
>
> **通信方式**：
>
> **方式 1：HTTP / REST**
> ```python
> # 主 Agent 调子 Agent
> response = await httpx.post(
>     "http://kyb-agent.internal/process",
>     json={"task": "extract_fields", "doc_path": "/uploads/abc.pdf"},
>     timeout=30,
>     headers={"trace_id": current_trace_id}
> )
> ```
>
> **方式 2：gRPC**
> 性能更好、类型安全、双向流。
>
> **方式 3：消息队列（异步）**
> ```python
> # 长任务用 Kafka
> await kafka.publish("kyb_tasks", {"task_id": "xxx", ...})
> # KYB Agent 订阅 + 处理 + 回写结果
> ```
>
> **状态管理**：
>
> 各 Agent 独立状态：
> - Redis：会话上下文
> - PG：长期状态
> - Milvus：向量检索
>
> 不能共享内存对象，**所有状态走中间件**。
>
> **上下文不共享的具体含义**：
>
> 主 Agent 调子 Agent，**只传必要的、最小的信息**：
>
> 错误的做法（共享上下文）：
> ```python
> # 把整个对话历史传给子 Agent
> subagent.invoke({
>     "history": full_conversation,  # ❌
>     "session": session_obj,        # ❌
>     "user_info": user_data         # ❌
> })
> ```
>
> 正确的做法（隔离上下文）：
> ```python
> # 只传子 Agent 需要的
> subagent.invoke({
>     "task": "extract_kyb_fields",   # ✅ 明确任务
>     "doc_path": "/uploads/abc.pdf"  # ✅ 必要参数
> })
> ```
>
> 子 Agent 完成后**只返回结果摘要**：
> ```python
> {
>     "status": "success",
>     "fields": {"company_name": "...", "tax_id": "..."},
>     "confidence": 0.95
> }
> ```
>
> 主 Agent 收到结果后再决定下一步，子 Agent 的中间过程不污染主上下文。
>
> **工程实现**（K8s 风格）：
>
> 每个 Agent 一个 Deployment：
>
> ```yaml
> apiVersion: apps/v1
> kind: Deployment
> metadata:
>   name: kyb-agent
> spec:
>   replicas: 3
>   template:
>     spec:
>       containers:
>         - name: kyb-agent
>           image: registry/kyb-agent:v1.2
>           resources:
>             limits:
>               nvidia.com/gpu: 1
>               memory: 16Gi
>           env:
>             - name: DB_USER
>               valueFrom: { secretKeyRef: ... }
> ```
>
> Service + Ingress 暴露内部地址。
>
> **跨 Agent 调用的 trace**：
>
> 必须传 `trace_id` 头：
> ```python
> @app.post("/process")
> async def process(req, trace_id: str = Header(...)):
>     with tracer.start_span("kyb_process", parent_trace_id=trace_id):
>         result = await ...
>     return result
> ```
>
> 在 Grafana 看完整调用链。
>
> **几个工程经验**：
>
> 1. **失败重试**：跨服务调用 timeout + retry + circuit breaker
>
> 2. **超时控制**：每个 RPC 设置严格 timeout
>
> 3. **版本兼容**：API 加版本号，平滑升级
>
> 4. **服务发现**：用 K8s service 名 + DNS
>
> 5. **配额限制**：服务间调用要限流
>
> 6. **降级策略**：子 Agent 不可用时主 Agent 走兜底
>
> **代价**：
>
> - 部署复杂（K8s + ArgoCD + 监控）
> - 调试复杂（分布式 trace）
> - 性能损失（RPC 延迟）
> - 团队成本（要懂微服务）
>
> 但收益**碾压代价**，工业 Agent 系统必上微服务。
>
> 一句话：**独立微服务是 Agent 边界的硬约束**——故障 / 资源 / 权限 / 上下文 / 发布全部隔离，是工业级 Agent 系统不可妥协的架构选择。

**🪝 顺嘴带出去的钩子**：
- "跨 Agent trace 怎么串起来可以聊。"
- "微服务化的隐性成本可以展开。"
- "分布式 Agent 的失败模式可以讲讲。"
