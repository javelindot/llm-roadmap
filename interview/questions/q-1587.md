# SFT后会出现哪些问题

## 答案

SFT 能让基础模型快速学习任务格式和示范行为，但它本质上仍是对训练答案做 token 级最大似然学习，因此训练完成不代表模型已经可靠对齐。

### 1. 常见问题

- **过拟合与模板化**：模型记住固定措辞、长度和结构，对同义表达或新场景泛化较差，输出多样性下降。
- **灾难性遗忘**：学习率过大、数据过窄或训练过久，会损伤原有通用知识、推理和多语言能力。
- **暴露偏差**：训练时每一步看到真实上文，推理时只能看到自己生成的上文，一次错误可能在长回答或多步 Agent 中不断累积。
- **训练目标错位**：交叉熵奖励“像示范答案”，却不直接优化事实正确率、用户偏好、工具执行成功率或业务收益。
- **过度自信与幻觉**：SFT 不会自动补充缺失知识，也可能使模型对训练风格之外的问题给出高置信度错误答案。
- **偏好无法充分表达**：单个标准答案不能说明两个都正确的回答中哪个更简洁、更安全或更符合用户意图。
- **工具调用脆弱**：可能出现 JSON 不合法、参数缺失、接口顺序错误、训练 schema 与线上 schema 不一致等问题。
- **安全边界漂移**：安全数据比例和质量不当，可能造成该拒绝时不拒绝，或正常问题也过度拒绝。
- **数据风险继承**：训练集中的偏见、PII、错误事实和数据泄漏会被模型学习，推理链数据还可能教会模型模仿错误步骤。

### 2. 如何定位

不能只看 SFT validation loss，还应建立多维评测：

- 保留任务集与通用能力回归集；
- 对国家、语言、币种、支付方式等长尾切片评测；
- 进行自由生成和多步 rollout，而不只做 teacher-forcing 评估；
- 实际执行工具调用，统计 schema 合法率、参数正确率和任务成功率；
- 检查 NLL、Brier Score、ECE 等校准指标；
- 评估幻觉、安全拒绝、输出长度和与基座模型的 KL 漂移；
- 使用时间隔离测试集排查数据泄漏和线上分布漂移。

### 3. 改进方法

- 清洗、去重并平衡数据，补充困难负例、长尾语言和真实失败样本；
- 降低学习率、早停，并混入一定比例的通用数据以减少遗忘；
- 对偏好问题使用 DPO 等离线偏好优化，确有序列级奖励时再考虑 RL；
- 对知识问题使用 RAG，对工具参数使用 constrained decoding、schema 校验和执行验证器；
- 通过拒答、升级人工、权限控制和审计机制处理模型无法可靠承担的风险；
- 按离线评测、影子流量、灰度、回滚的顺序上线。

### 4. 跨境支付示例

例如模型在 SFT 数据里大量见过“银行卡退款”，可能把同一流程错误套用到本地钱包或银行转账；在英文数据上表现良好，却误解小语种付款凭证；还可能编造到账时间或把退款接口参数填错。

因此，跨境支付模型应按国家、渠道、币种和合规风险分层评测。到账状态、费率和监管要求由实时系统或知识库提供，资金操作由确定性工作流和审批控制，模型主要负责理解、规划与解释。

### 面试总结

SFT 后的核心风险是过拟合与遗忘、暴露偏差、目标错位、幻觉和过度自信，以及工具与安全边界不可靠。解决思路不是盲目增加 SFT 轮数，而是先用分层评测定位问题，再分别采用数据治理、偏好优化、RAG、验证器、约束解码和运行时权限控制。
