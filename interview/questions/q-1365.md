# 多轮对话中如何解决模型遗忘历史信息的问题

## 答案

多轮对话的“遗忘”本质上不是模型权重真的忘了，而是当前输入窗口里缺少可用历史、历史被压缩得太粗、检索不到关键状态，或者模型没有区分“用户说过的话”和“业务系统的真实状态”。解决时不能只说“加大上下文”，要做上下文工程和外部记忆。

### 1. 分层管理上下文

- **短期上下文**：保留最近几轮原文，处理指代、省略和情绪语气；
- **滚动摘要**：把长历史压缩成结构化摘要，例如目标、已确认事实、待办、约束、风险点；
- **长期记忆**：把用户偏好、商户配置、常见问题、历史工单等写入外部存储；
- **实时状态**：订单状态、清结算结果、KYC/KYB、汇率、费用、风控命中必须从工具或数据库取，不能依赖模型记忆。

跨境支付场景中，可以把会话状态拆成：

```text
merchant_id
payment_order_id
payout_order_id
currency_pair
country_or_region
kyc_status
risk_case_id
current_intent
confirmed_facts
pending_questions
```

这样用户问“那笔美元提现为什么还没到”时，模型先解析“那笔”对应的订单，再查支付/出款/清算系统，而不是凭历史猜。

### 2. 记忆写入要有策略

不是所有对话都写长期记忆。通常只写：

- 稳定偏好：默认币种、常用国家、语言偏好；
- 业务实体：商户、店铺、渠道、订单、工单；
- 已确认事实：用户明确确认过的收款账户、合同条款、问题原因；
- 任务进度：已完成、待审批、待补材料。

要避免写入：

- 一次性闲聊；
- 未确认猜测；
- 敏感信息明文；
- 已过期状态，例如“昨天风控审核中”。

### 3. 检索增强和状态机

长对话中常用做法是：

1. 对当前问题做意图识别和实体抽取；
2. 从会话摘要、向量记忆、结构化数据库中检索相关历史；
3. 拼接最近上下文、关键记忆和实时工具结果；
4. 用状态机约束下一步动作，例如“补材料 -> 等待审核 -> 复核通过 -> 放款”。

在支付业务里，状态机尤其重要。模型可以解释“为什么需要补充法人证件”，但不能跳过合规审核直接承诺到账。

### 4. 历史压缩要可追溯

摘要不能只写“用户遇到支付问题”，而要写：

```text
用户 A 询问订单 P123 的 EUR->USD 出款延迟。
已查到：渠道返回 pending，原因是收款行需要补充 beneficiary address。
用户已确认愿意补材料。
下一步：引导上传地址证明，并说明预计审核 SLA。
```

如果摘要中包含结论，最好保留来源：来自哪轮对话、哪个工具、哪个订单状态。这样后续模型回答时可以减少幻觉。

### 5. 评测指标

可以从这些维度评估：

- 历史事实召回率；
- 指代消解准确率；
- 与实时系统状态的一致性；
- 摘要压缩后的任务完成率；
- 错误继承率，即旧错误是否被后续反复引用；
- 隐私泄露和跨租户串记忆风险；
- Token 成本、延迟和检索命中率。

### 6. 面试总结

我的回答会强调：多轮记忆不是简单扩大 context window，而是“近期上下文 + 结构化摘要 + 外部记忆 + 实时工具 + 状态机”的组合。跨境支付里尤其要把模型记住的会话事实和支付系统的真实状态分开，高风险结论必须以订单、账务、合规和风控系统为准。
