# 怎么防止客服LLM串台问题

## 答案

客服 LLM “串台”通常包括三类问题：

- A 用户的会话或数据进入 B 用户回答；
- 不同商户、租户或业务线的知识被混用；
- 并发流式响应、工具结果或缓存被错误关联。

防护必须覆盖身份、存储、检索、缓存、模型、工具和流式链路。

### 1. 请求级身份绑定

每个请求由网关注入不可修改的安全上下文：

> `tenant_id、merchant_id、user_id、session_id、request_id、role、data_scope`

不能信任用户在 Prompt 中自报身份。后端所有数据查询、知识检索、缓存和日志都使用该上下文。

会话键至少包含租户、用户和 session：

> `conversation:{tenant_id}:{user_id}:{session_id}`

不能只使用前端传来的会话名称或一个全局 `thread_id`。

### 2. 请求状态隔离

- 服务保持无状态或使用 request-scoped context；
- 禁止全局可变的 messages、tool_result 和 current_user；
- 异步任务显式传递 context，不依赖线程局部变量误复用；
- 每个子任务携带 request_id 和 trace_id；
- 并行结果按 task_id 聚合，不能按返回顺序拼接；
- LangGraph checkpoint 使用租户隔离的 thread namespace。

连接池可以共享，业务上下文和游标不能共享。

### 3. 知识库与向量检索隔离

检索前必须使用：

- tenant/merchant ACL；
- 角色、地区和产品权限；
- 文档可见性和敏感等级；
- 国家、币种、版本和有效期。

过滤在数据库或索引层执行，而不是先把其他商户内容召回后再要求模型忽略。对于高敏感租户，可以使用物理独立 index；共享 index 时必须支持强制 server-side filter 并做越权测试。

### 4. 缓存隔离

缓存键包含：

> `tenant、role、language、country、knowledge_version、model_version、normalized_query`

个性化订单、余额、汇率报价和工具结果不进入公共答案缓存。缓存值记录权限范围和失效时间，读取时再次校验。

Prompt Cache/KV Cache 可以在推理服务内部复用公共前缀，但不能把某个用户的私有消息作为其他请求的可见前缀。日志和调试缓存同样需要租户隔离。

### 5. 工具和数据层二次鉴权

Agent 调用订单、账户、通道或工单工具时，工具服务端校验：

- 用户与商户关系；
- 订单是否属于当前商户；
- 角色是否允许读取字段；
- 请求中的实体是否与安全上下文一致。

模型生成的 `merchant_id` 不能覆盖后端绑定值。返回结果使用最小字段和脱敏内容，避免后续节点传播不必要数据。

### 6. Prompt、记忆和角色隔离

- 系统 Prompt 按业务线版本化，不在同一会话中动态拼接多个角色；
- 短期记忆只属于当前 session；
- 长期记忆按 tenant/user namespace 保存；
- 写入长期记忆前做 PII、来源和稳定性检查；
- 不把一次交易状态写成永久用户偏好；
- 切换商户、角色或业务线时创建新上下文，不复用旧摘要。

摘要器也可能串台，生成摘要前后都要保留用户和会话边界并校验实体归属。

### 7. 流式响应隔离

SSE/WebSocket 连接绑定 user、session 和 request_id：

- 每个事件带单调递增 event_id；
- 服务端发送前核对连接所有者；
- 断线续传只读取该连接的事件流；
- 取消请求后停止对应生成任务；
- 多请求并发时不复用同一个响应队列；
- 日志中记录连接、请求和模型任务映射。

### 8. 输出检测和审计

生成后检查回答中的订单号、商户名、账户和币种是否属于当前上下文。可在测试环境注入租户 canary token，若出现在其他租户回答中立即阻断上线。

持续监控：

- 跨租户检索命中率，应为零；
- 错误会话绑定和缓存串扰；
- 输出中未知实体比例；
- SSE 事件错投；
- 权限拒绝和异常工具调用。

通过并发压测、会话切换、缓存碰撞、断线重连和恶意 ID 注入做专项测试。支付场景中串台属于数据泄漏事故，不能依赖模型发现和自我纠正，边界必须由基础设施和服务端权限保证。
