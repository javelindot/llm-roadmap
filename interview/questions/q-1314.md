# Reranker模型如何构造数据微调

## 答案

Reranker 微调的核心不是堆大量随机负样本，而是构造能代表线上召回错误的 `query-positive-hard negative` 数据，并保持业务分布、难度和标注口径一致。

### 1. 先定义训练形式

**Pointwise**

```text
(query, document) -> relevance score/label
```

可用 BCE、回归或多分类损失。

**Pairwise**

```text
(query, positive, negative)
```

目标是 `score(q, pos) > score(q, neg)`，可用 margin ranking 或 pairwise logistic loss。

**Listwise**

```text
query + [d1, d2, ... dn]
```

直接优化一组候选的排序分布，更贴近 NDCG/MRR，但训练成本更高。

### 2. 正样本来源

- 人工标注的相关文档；
- 用户点击、采纳、问题解决等行为日志；
- 工单最终引用的 SOP、错误码和规则；
- QA 与其证据段落；
- 同一文档中可回答问题的具体 chunk。

行为日志有位置偏置和曝光偏置，不能把所有点击都当真值。

### 3. 负样本分层

- Random negative：容易，主要用于训练初期；
- In-batch negative：成本低，但可能出现假负例；
- BM25 hard negative：关键词相似但答案不对；
- Dense hard negative：语义相似但事实或条件不匹配；
- Cross-encoder/LLM mining：挖更难样本；
- 线上 bad case：最有业务价值。

跨境支付中，好的 hard negative 包括：

- 同一错误码但不同国家或支付通道；
- 同一商户但不同结算周期；
- Visa 和 Mastercard 相似拒付规则；
- 同币种但交易日、清算日不同的汇率文档；
- 已过期和当前版本的合规规则。

### 4. 防止数据问题

- 按 query 或业务实体切分 train/dev/test，防止近重复泄漏；
- 处理多正例，不把另一篇正确文档标成负例；
- 加版本、国家、币种、通道、时间等 metadata；
- 控制不同业务线和难度比例；
- 对 LLM 合成标签做人工抽检和一致性校验；
- 高风险合规数据必须脱敏、隔离和审计。

### 5. 训练与评估

训练可从现有 cross-encoder 或专用 reranker 继续微调。先用容易负样本稳定训练，再逐步加入 hard negatives；可用教师模型蒸馏分数，但不要只学习教师偏差。

离线看：

- MRR@k、NDCG@k、MAP；
- Recall@k 之后的重排增益；
- pair accuracy；
- 不同国家、语言、通道的分桶指标。

线上看回答正确率、首次解决率、人工转接率、P95 延迟和成本。Reranker 再强也无法找回召回阶段完全漏掉的文档。
