# 实际项目中用过langchain吗？

## 答案

**一句话**：LangChain 我用过或了解过，主要用于快速搭建 RAG、工具调用和 Agent 编排，但生产落地时会谨慎封装核心业务逻辑。

### 核心回答
LangChain 常用能力：
- PromptTemplate：管理提示词模板。
- LLM/ChatModel：统一模型调用。
- Retriever：接入向量库和检索器。
- Chains：串联多步骤调用。
- Agents/Tools：工具调用和 ReAct 流程。
- Memory：对话记忆。
- Callback/Tracing：调试和观测。

优点：
- 快速原型。
- 生态丰富。
- 接各种模型、向量库、工具方便。

缺点：
- 抽象层较厚，复杂场景调试成本高。
- 版本变化快。
- 生产核心流程可能需要自己实现更可控的编排。

### 1. LangChain 适合做什么

LangChain 适合快速验证和组织大模型应用链路，比如 prompt、retriever、tool、agent、memory、callback。它能帮你少写很多 glue code。

### 2. 生产里为什么要谨慎

生产系统要关注可控性：

- 每一步输入输出是否可观测。
- 工具调用是否可审计。
- 错误是否容易定位。
- 版本升级是否影响链路。
- 核心业务逻辑是否被框架抽象遮住。

复杂场景里，我更倾向把关键流程显式写清楚。

### 3. 面试总结

LangChain 是提效工具，不是业务架构本身。会用框架，更要理解框架下面的 RAG、Agent、tool calling 原理。

### 4. 结合跨境支付项目怎么说（面试口径）

> 我在 RAG 原型里会用 LangChain，比如接 embedding、向量库、retriever、prompt 和工具调用，能很快跑通支付知识助手。
>
> 但真到生产，我会把关键链路自己掌控，比如权限校验、知识库路由、引用格式、工具幂等、审计日志。因为支付场景里，某个工具误调、某条规则引用错，都可能带来业务风险。
>
> 所以 LangChain 对我来说是提效工具，不是业务架构本身。可靠性、可追溯和权限边界，比框架用得多花更重要。

**钩子**：LangChain 适合快速验证，生产要沉淀自己的检索、权限和观测体系。
