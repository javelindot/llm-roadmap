# FunctionCall工具调用的完整流程是什么？

## 答案

**一句话**：Function Call 的完整流程是：模型理解用户意图，选择合适工具并生成结构化参数，业务侧执行工具，把结果返回给模型，模型再基于结果生成最终答复。

### 1. 基本流程

完整链路通常是：

1. 开发者定义工具名称、描述和参数 schema。
2. 用户提出问题。
3. 模型判断是否需要调用工具。
4. 如果需要，模型输出工具名和结构化参数。
5. 应用层校验参数、权限和业务规则。
6. 应用层执行真实函数或调用后端 API。
7. 工具结果返回给模型。
8. 模型结合工具结果生成自然语言答案。
9. 记录日志、trace、工具入参出参和最终结果。

### 2. 关键点

Function Call 不是让模型直接访问数据库，而是让模型“提出调用意图”，真正执行的是业务系统。因此中间一定要有校验层。

重点包括：

- 参数 schema 要清晰，字段含义不能模糊。
- 工具描述要准确，否则模型会选错工具。
- 工具权限要按场景开放，不能所有工具都暴露。
- 参数要做二次校验，不能相信模型生成的内容。
- 涉及写操作要有确认、幂等和审计。
- 工具异常要转成模型可理解的错误信息。

### 3. 常见问题

- 模型选错工具：需要优化工具描述和路由。
- 参数缺失：可以追问用户或根据上下文补全。
- 参数幻觉：必须校验 ID、金额、币种、国家等关键字段。
- 工具结果太大：需要摘要、分页或上下文压缩。
- 多工具调用顺序混乱：可以用流程图或状态机约束。

### 4. 结合跨境支付项目怎么说（面试口径）

> 在跨境支付客服场景里，Function Call 很适合做订单查询、到账状态查询、风控原因查询、通道回执查询。比如用户问“这笔付款为什么还没到账”，模型先识别需要查交易状态，然后生成 `transactionId` 或商户订单号，调用查询工具。后端查到交易状态、通道回执、预计到账时间后，再把结构化结果返回给模型，让模型组织成用户能听懂的话。
>
> 但我不会让模型直接操作资金类接口。比如重新发起付款、解冻账户、修改收款人信息，这些都属于高风险动作，即使模型能识别意图，也必须经过权限校验、人工确认和幂等控制。模型只负责理解和生成参数，业务系统负责执行和兜底。
>
> 我认为 Function Call 落地的关键是工具边界。查询类工具可以比较开放，资金变更类工具必须严格管控，并且所有调用都要有 traceId 和审计日志。

**钩子**：可以引到“模型不直接执行业务、工具白名单、资金操作保护节点”。
