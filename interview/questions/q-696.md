# 有没有尝试过动态prompt，训练数据如何构造，如何保证policy正确？

## 答案

动态 prompt 指根据用户、任务、上下文、检索结果或策略状态实时选择模板、示例和工具说明。回答时应区分规则路由、检索式 prompt、可学习 prompt 和由模型生成 prompt。

### 数据构造

先定义场景 taxonomy 和每类 prompt policy，例如必须包含的 system 约束、允许工具、输出 schema 和示例。训练样本应覆盖：

- 不同意图、领域、难度、语言和上下文长度；
- 边界、歧义、缺参、冲突指令和 prompt injection；
- 正确路由以及容易混淆的 hard negative；
- 工具成功、超时、空结果和权限不足等状态。

样本可以来自线上去隐私日志、专家编写和模型辅助生成。合成样本需去重并通过规则、执行器或人工审核。若训练的是 policy/router，标签应是模板 ID、工具集合或结构化决策，而不是自由文本描述。

### 保证 Policy 正确

把不可违反的权限、合规和参数约束放在代码层，不依赖 prompt 自觉。Router 输出使用 schema 校验，只能选择白名单策略；高风险动作要求人工确认。为每个策略建立版本、适用范围、回退模板和灰度机制。

离线评估路由准确率、任务完成率、工具参数正确率、安全违规率、格式通过率和成本；用对抗集检查越权与注入。线上 shadow/A-B 测试并监控分桶指标，失败时回退静态 prompt。

动态 prompt 的收益必须通过相对固定 prompt 的消融证明。若场景规则明确，代码路由通常比让 LLM 自由生成 system prompt 更可靠。
