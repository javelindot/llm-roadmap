# 水平分表按什么来进行？从理论层面分析达到什么量级分表，什么量级分库，什么量级分集群？分库分表的场景和方式？

## 答案

水平分表是把同一逻辑表的行按分片键分散到多个物理表或节点。不存在“达到一千万行一定分表、一个亿一定分库”的通用阈值；是否拆分由单分片容量、读写吞吐、热点、索引、备份恢复窗口和业务 SLO 共同决定。

### 1. 分片键怎么选

好的分片键应满足：

- 高基数，数据分布相对均匀；
- 大部分查询能携带该键，避免 scatter-gather；
- 写入不会集中到单个分片；
- 生命周期稳定，不频繁修改；
- 支持租户隔离和数据驻留；
- 未来可扩容、迁移和重分片。

跨境支付常见选择：

- `merchant_id`：商户维度查询多，天然租户隔离；
- `account_id`：账务流水围绕账户访问；
- `order_id` 哈希：全局分布均匀，适合按单号点查；
- `region + hash(merchant_id)`：兼顾数据驻留和均衡；
- 时间通常用于分区或归档，不宜单独作为在线写分片键，否则最新分片成为热点。

### 2. 常见方式

**Range**

```text
id/time in [a,b) -> shard N
```

优点是范围查询和归档方便；缺点是递增键容易产生热点。

**Hash/Modulo**

```text
shard = hash(key) mod N
```

分布均匀、点查简单；扩容改变 `N` 时迁移量大，可使用一致性哈希或虚拟分片缓解。

**Directory/Lookup**

维护 `key -> shard` 映射，灵活支持迁移和大客户独立分片，但多一层元数据服务。

**复合策略**

先按地域或合规域分库，再按商户哈希分片，表内按月分区。真实系统通常是多级组合，不是单一取模。

### 3. 理论容量怎么估算

设：

```text
daily_rows = peak_qps * write_ratio * 86400
retained_rows = daily_rows * retention_days
data_size = retained_rows * avg_row_bytes
```

再加索引、MVCC、多副本和预留空间：

```text
physical_size
≈ data_size
  * index_amplification
  * replica_factor
  / target_utilization
```

所需分片数至少满足：

```text
shards >= max(
  total_size / safe_capacity_per_shard,
  peak_read_qps / safe_read_qps_per_shard,
  peak_write_qps / safe_write_qps_per_shard
)
```

这里的安全容量来自目标硬件上的压测，不是互联网流传的固定行数。

### 4. 何时分表、分库、分集群

**先做单表优化**

- 索引、SQL、冷热分离；
- 按时间分区；
- 读副本、缓存；
- 归档历史数据；
- 批处理和连接池优化。

**分表**

当单表索引和维护成本过高、DDL/备份窗口不可接受，或按时间管理数据更方便时采用。数据库内分区表也可能已经足够。

**分库/分片节点**

当单实例 CPU、IOPS、存储、连接数或写吞吐成为瓶颈，且垂直扩容或读写分离无法满足 SLO 时采用。它解决资源边界，但引入路由、扩容和跨分片事务复杂度。

**分集群**

常用于：

- 地域和监管隔离；
- 业务域隔离；
- 超大租户隔离；
- 故障域与爆炸半径控制；
- 单集群元数据、节点数或运维窗口达到上限。

因此“分集群”更多是可靠性、合规和组织边界，不只是数据行数。

### 5. 分库分表代价

- 跨分片 join、聚合和分页困难；
- 分布式事务与一致性复杂；
- 全局唯一 ID；
- 二级索引和非分片键查询；
- 扩容、双写、回填和校验；
- 热点大商户；
- 运维与数据修复成本。

报表和复杂聚合应通过 CDC 进入数仓/OLAP，而不是在线跨所有交易分片执行。

### 6. 跨境支付设计

订单表可按 `region + merchant_id hash` 路由，`order_id` 中编码或映射分片信息；账务分录围绕账户分片，保证同账户记账尽量在一个事务域。热点商户可以迁到独立 shard。

支付系统还要保证：

- 幂等键全局有效；
- 资金流水可审计；
- 分片迁移期间不漏单、不重单；
- 对账能跨分片汇总；
- 数据驻留、加密和删除要求；
- 扩容通过影子写、校验和可回滚切流。
