# 介绍 HyDE 思路

## 答案

**一句话**：HyDE（Hypothetical Document Embeddings）= **让 LLM 先"假装回答"用户问题，用这个假设回答的 embedding 去检索**——因为"答案 vs 文档"在语义空间里通常比"问题 vs 文档"更接近。

### 1. 核心 idea
传统检索：
```
query embed → 向量库找最近的 doc
```
HyDE：
```
query → LLM 生成假设回答 → 假设回答 embed → 向量库找最近的 doc
```

### 2. 为什么有效
- 用户**问题**和**答案**在 embedding 空间里距离大
- 知识库存的是**答案性内容**
- 用"假设答案"匹配"真实答案"，**同质对同质**，距离小
- 即使假设答案有幻觉、不全对，**只要风格/术语相近**就能召回正确 doc

### 3. 经典 prompt 范式
```
Please write a passage to answer the question.
Question: {query}
Passage:
```

例：
- query: "退款多久"
- LLM: "跨境信用卡退款的到账时效通常为 7-15 个工作日，具体取决于发卡行处理速度和清算渠道……"
- 用这段 embedding 去检索

### 4. HyDE 的两个关键
- **风格匹配**：LLM 生成要像知识库的口吻
- **术语覆盖**：假设答案要带相关术语词

### 5. HyDE 的局限
- 多调一次 LLM（延迟 + 成本）
- 长尾问题 LLM 可能编出错误方向
- 简单 query 可能 HyDE 反而引入噪声

### 6. HyDE 的变体
- **Multi-HyDE**：生成多个假设答案，每个分别检索
- **HyDE + 原 query 融合**：双重 embedding 加权
- **LLM-only HyDE**：用 LLM 直接生成 query + answer pair

### 7. 何时上 HyDE
- query 很短、答案应该很长（长尾 QA）
- 用户用口语化表达、知识库是正式文档
- 高价值低延迟容忍场景

### 8. 一句话总结
**HyDE = "答案对答案"代替"问题对答案"**——一招便宜的语义对齐，对 query/doc 风格差距大的场景特别有效。

## 结合项目谈（口语版 · 约 25 秒）

> HyDE 在我们跨境支付 RAG 里**针对性使用**，不是全场景开。
>
> **客服 RAG 上 HyDE**：客户问"我的钱怎么还没到"，知识库写的是"跨境结算时效根据清算渠道和发卡行处理速度，一般 24h 到 7 个工作日……"——口语 vs 正式文本距离大。我们让 Qwen-7B 先生成假设回答，再用这段 embedding 检索，召回率从 78% 涨到 84%（**+6**）。
>
> **风控 RAG 不上 HyDE**：风控查询大多是结构化术语（"商户 MCC 5816 跨境美元结算"），术语已经和知识库高度对齐，HyDE 反而引入噪声。
>
> **几个工程坑**：
> - HyDE 生成要短（50-100 字），太长引入幻觉；
> - LLM 用业务领域微调过的版本，避免生成"错误方向"的假设；
> - 我们做了 **HyDE + 原 query 双 embedding 加权融合**（0.7:0.3），比纯 HyDE 稳一点；
> - 简单 query 不上 HyDE，做个 query 长度判定（< 10 字才上）。
>
> 还有一个有意思的：**HyDE 经常生成"错答案但召回正确文档"**——比如它假设回答"30 个工作日"，实际答案是"7-15 个工作日"，但 embedding 已经命中正确章节了。这正好说明 HyDE 的本质——它要的不是"回答正确"，是"语义靠近"。
>
> 一句话：**HyDE 是个聪明 trick**，但要分场景用，不是默认开。

**🪝 顺嘴带出去的钩子**：
- "HyDE 的 prompt 怎么设计能减少幻觉。"
- "HyDE + 原 query 融合的权重怎么定可以聊。"
- "用领域微调小模型当 HyDE 生成器的考虑可以展开。"
