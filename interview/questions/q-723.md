# 引导：那你描述一下Epoll编程我怎么去监听一个事件？wait你要监听哪些东西，有哪些东西我其实可以不监听？（这个没答出来，对epoll的理解还是太浅了）

## 答案

epoll 的基本流程是创建实例、把文件描述符及关心的事件注册到 interest list，再用 `epoll_wait` 获取 ready list。

### 编程流程

```c
int epfd = epoll_create1(EPOLL_CLOEXEC);

struct epoll_event ev = {0};
ev.events = EPOLLIN | EPOLLRDHUP;
ev.data.fd = listenfd;
epoll_ctl(epfd, EPOLL_CTL_ADD, listenfd, &ev);

for (;;) {
    struct epoll_event events[1024];
    int n = epoll_wait(epfd, events, 1024, -1);
    for (int i = 0; i < n; ++i) {
        // 根据 events[i].events 和 data 处理
    }
}
```

监听 socket 的 `EPOLLIN` 表示可 `accept`；连接 socket 的 `EPOLLIN` 表示可读。非阻塞 `connect` 或输出缓冲区还有数据时才注册 `EPOLLOUT`。

### 哪些需要监听

- `EPOLLIN`：需要读数据或接受连接。
- `EPOLLOUT`：连接建立中，或之前写到 `EAGAIN`、仍有待发送数据。
- `EPOLLRDHUP`：对端关闭写侧，便于发现半关闭。
- `EPOLLERR`、`EPOLLHUP` 即使未显式注册也会被报告，但事件循环仍必须处理。
- `EPOLLET` 选择 edge-triggered；`EPOLLONESHOT` 多线程处理同一 fd 时常用，处理后需 `MOD` 重新激活。

### 不该一直监听什么

不要对所有连接永久监听 `EPOLLOUT`。可写状态大部分时间成立，会让 LT 模式不断唤醒形成 busy loop。发送缓冲清空后应通过 `EPOLL_CTL_MOD` 移除 `EPOLLOUT`。

ET 模式必须使用 nonblocking fd，并循环 `accept/read/write` 直到返回 `EAGAIN`；只处理一次会丢失后续通知。LT 模式只要条件仍成立就会重复通知，逻辑更简单。关闭 fd 前要处理生命周期，防止 fd 复用造成陈旧事件误关联。
