# Qwen是怎么做长度外推的

## 答案

Qwen 的长上下文方案随代际演进，不能只回答“用了某一种 RoPE 插值”。早期 Qwen 主要依赖 NTK-aware RoPE 插值与 attention scaling；Qwen2/2.5 使用 RoPE、GQA 和长上下文训练；Qwen2.5-1M 又加入长短混合训练、长度外推和稀疏注意力工程。

### 1. 长度外推为什么困难

模型在训练长度之外推理时会遇到：

- RoPE 位置角度超出训练分布；
- 注意力分数随长度变化；
- 模型没有学过超长距离依赖；
- 全量注意力的计算和显存成本呈平方增长。

所以只把 `max_position_embeddings` 改大并不能可靠获得长上下文能力。

### 2. RoPE 缩放

Qwen 系列使用过 NTK-aware interpolation、动态 NTK 或 YaRN 类缩放思路，本质是调整 RoPE 频率，使更长位置映射到模型可处理的频率范围。

常见形式是在配置中设置 `rope_scaling`。但缩放因子应匹配目标长度，过度缩放可能损害短文本效果，因此新版本更强调动态选择或按场景配置。

### 3. 训练数据和课程

真正稳定的长上下文能力还需要训练：

- 混合短、中、长序列，避免只训长文本导致短任务退化；
- 合成长依赖任务，如 needle-in-a-haystack、多跳检索、跨段引用；
- 长文档 QA、代码仓库、表格和多文档数据；
- SFT 后继续验证指令遵循和位置鲁棒性。

Qwen2.5-1M 公布了从较短上下文预训练逐步扩展到更长上下文的路线，而不是一步拉到百万 token。

### 4. 注意力与推理工程

百万级上下文不能只靠位置编码：

- 使用 FlashAttention 等高效 kernel；
- Prefill 采用分块和并行；
- Qwen2.5-1M 引入稀疏注意力路线降低超长序列成本；
- KV Cache 使用分页管理、量化或低精度；
- 服务层做 prefix cache、chunked prefill 和请求调度。

### 5. 面试中的准确说法

> Qwen 的长度扩展是“位置编码缩放 + 长上下文继续训练 + 长短数据混合 + 注意力/推理系统优化”的组合。老版本可重点讲 NTK-aware RoPE 和 logN attention scaling；Qwen2.5-1M 则要讲百万上下文训练与稀疏注意力，不能只说修改 RoPE 参数。

### 6. 跨境支付落地

支付调查可能涉及合同、KYC 材料、订单、清算和沟通记录，但生产上应先按商户、交易、时间和权限检索，再把证据送入模型。长上下文是兜底能力，不是替代数据治理；关键金额、汇率和交易状态仍从工具实时读取。
