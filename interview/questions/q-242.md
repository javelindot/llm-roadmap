# 大模型推理的优化

## 答案

答：标配是vllm + AWQ + FlashAttention2 + FastAPI + PD分离
（1）模型量化方法AWQ，新一代INT4量化，与GPTQ相比，保留重要权重的精度；
（2）Vllm核心是PagedAttention，将KVCache按页管理，解决内存碎片，提升GPU利用率；
（3）FlashAttention2，高性能算子，把QKV切成小块，只在高速缓存里算，不再来回搬；
（4）动态批处理，不用等整句生成完毕；
（5）3D并行，数据并行是多份模型算一份数据，张量并行是一层计算分多张卡，流水线并
行是一层计算分段执行；
（6）提示词缓存，PD分离；
（7）异步服务，FastAPI+async。

### 补充说明

大模型推理优化一般分四层：模型层、算子层、服务层和业务层。模型层包括量化、蒸馏、小模型路由；算子层包括 FlashAttention、TensorRT-LLM；服务层包括 vLLM、动态批处理、KV Cache 管理、流式输出；业务层包括缓存、限流、降级和按任务选择模型。

推理优化不能只追求单次最快，还要看吞吐、并发、首 token 延迟、平均延迟、显存占用和稳定性。线上服务最怕 OOM、排队过长和一个业务把资源打满。

### 结合跨境支付项目怎么说（面试口径）

> 在跨境支付项目里，我会把推理优化和业务优先级结合起来。比如客服 FAQ 量大，适合用小模型、缓存和流式返回；复杂风控解释调用更强模型，但要限流。vLLM 侧通过 PagedAttention 和动态批处理提高吞吐，同时控制最大上下文和并发，避免显存被长问题吃满。
>
> 如果是文档抽取这类离线任务，可以放到低优先级队列，不和实时客服抢 GPU。线上还会监控首 token 延迟、总延迟、队列长度、OOM 和错误率。
>
> 我的理解是，推理优化最终服务的是业务稳定性。支付场景不能因为模型生成慢，影响用户查询付款状态或客服处理工单。

**钩子**：可以引到“PagedAttention、动态批处理、KV Cache、实时和离线任务隔离”。
