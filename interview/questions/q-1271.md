# MmoE的全流程,怎么确定专家数

## 答案

MMoE 用多个共享 Expert 学习不同表示，每个任务通过独立 Gate 组合 Expert，再进入任务 Tower。它主要解决多任务共享参数时的任务冲突和负迁移。

### 1. 模型流程

输入特征 `x` 进入 E 个 Expert：

```text
e_i = expert_i(x), i = 1...E
```

任务 `t` 的 Gate：

```text
g_t = softmax(W_t x)
```

任务表示与输出：

```text
h_t = Σ_i g_t[i] * e_i
y_t = tower_t(h_t)
```

每个任务有独立 Gate 与 Tower，Expert 在任务之间共享。

### 2. 数据与 Loss

样本需要包含各任务标签及 label mask：

```text
L_total = Σ_t lambda_t * mask_t * L_t
```

例如点击标签多、转化标签稀疏，不能把缺失转化标签当负样本。需要处理：

- loss 尺度；
- 任务采样比例；
- CVR 的选择偏差；
- 正负样本不均；
- 延迟标签；
- 多任务校准。

`lambda_t` 可手工设定，也可使用 uncertainty weighting、GradNorm 等方法，但最终以主任务与 guardrail 指标决定。

### 3. 怎么确定专家数

没有解析公式直接给出最优 E。采用受控搜索：

```text
Shared-bottom baseline
→ MMoE E=2
→ E=4
→ E=8
```

保持 Expert 总宽度或总 FLOPs可比，否则专家更多同时参数更多，无法归因。

评估：

- 各任务 AUC/PR-AUC/LogLoss；
- 主任务与长尾分桶；
- 任务 Pareto 改善；
- Gate entropy 和 Expert 使用率；
- Expert 输出相关性；
- 训练、P95/P99 和内存；
- 参数量和吞吐。

当增加 Expert 后收益饱和、Gate collapse 或延迟超预算，就停止增加。

### 4. 任务相关性

- 任务高度相关：少量 Expert 或 Shared Bottom 可能足够；
- 任务部分相关：MMoE 更有价值；
- 任务冲突严重：可增加 Expert，但也可使用 PLE、任务独占 Expert 或分开模型；
- 数据量不足：专家过多会欠训练和过拟合。

专家数不是任务数的固定倍数。两个任务也可能需要四个专家，多个高度相关任务也可能只需少量专家。

### 5. 防止 Expert Collapse

- 监控每个任务 Gate 的均值、方差和熵；
- 比较 Expert 表示的 cosine；
- 检查 Expert 梯度和消融；
- 调整初始化、容量和正则；
- 使用 shared + task-specific expert；
- 必要时增加多样性或负载均衡约束。

### 6. 部署

经典 MMoE 是 dense experts，所有 Expert 都执行。可将多个小 Expert 合并成 batched GEMM，融合 Gate 与加权，减少 launch 和中间张量；还要优化 embedding lookup 和特征服务。最终专家数由质量与线上 P99/成本共同决定。

跨境支付多目标路由可预测成功、成本、时效和风险，但硬合规条件在模型外过滤，任务权重不能让成本收益抵消安全风险。
