# ReAct、自主抉择模式、状态机、策略模式在项目中分别如何落地使用？

## 答案

> 我把这四个模式按"**复杂度从低到高**"在项目里分层用，分别对应不同场景，不是替代关系。
>
> **1. ReAct（最朴素）**
>
> 用在**短链路、低风险**任务：
> - 客服意图分类
> - 工单类型分流
> - 简单工具调用（查订单状态）
>
> 落地：LangChain `create_react_agent` → Qwen-7B → 加 max_iterations=8 兜底。原因是上手快、调试简单。
>
> **2. 策略模式（Strategy Pattern）**
>
> 用在**"同类任务多种实现"**：
> - 不同币种的汇率查询（A 渠道 / B 渠道 / 缓存）
> - 不同国家的 KYB 审核规则（中国 / 新加坡 / 印尼）
> - 不同业务线的退款流程
>
> 落地：Python 抽象基类 + 多个 concrete 子类 + factory 注册。Agent 决定调哪个 strategy，strategy 内部封装具体实现。
>
> ```python
> class RefundStrategy(ABC):
>     def execute(self, order): ...
>
> class CrossBorderCardRefund(RefundStrategy): ...
> class LocalWalletRefund(RefundStrategy): ...
> ```
>
> 原因：**业务规则按维度变化**时，策略模式比 if-else 链可维护。
>
> **3. 状态机（State Machine）**
>
> 用在**有明确流转的长流程**：
> - 商户准入流程：草稿 → 资料审核 → 风控审核 → 通过 / 拒绝
> - 风控调查：初查 → 关联分析 → 证据收集 → 结论 → 复核
> - 客服工单：新建 → 处理中 → 等待 → 已完成
>
> 落地：用 LangGraph 的 StateGraph 实现，每个状态是一个节点，条件边定义流转规则。每次 Agent 动作只改一个状态字段。
>
> 原因：**可中断、可恢复、可审计**——监管要求每个状态变更有 trace。
>
> **4. 自主抉择 Agent（最灵活）**
>
> 用在**开放式任务**：
> - 风控调查（不确定要查几跳关联）
> - 复杂客诉处理（不确定要不要转人工）
>
> 落地：状态机骨架 + 关键决策点让 LLM 自主选下一节点（LangGraph 条件边里调 LLM）。
>
> 原因：**业务路径多变**，写死流程不现实，让 LLM 在框架内自由决策但有边界。
>
> **为什么这么分层**：
> - 越简单的模式越可控、越便宜——能用就别上复杂的
> - 复杂模式只在收益大于工程成本时上
> - **不能纯 LLM 自主抉择**——金融场景需要可审计、可回滚，纯 LLM 黑盒不行
>
> 这其实是"**用工程模式给 LLM 的自由度加边界**"——LLM 决策、框架保证稳定。
