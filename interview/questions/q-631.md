# decoder预测时i位置预测出来了，然后预测i+1位置，那么是用i位置的token结果还是概率结果？

## 答案

自回归 decoder 在预测 $i+1$ 位置时，输入的是第 $i$ 位置选出的 token id 对应的 embedding，而不是把整个概率分布直接当作下一个位置的输入。

### 推理阶段

模型先根据已有序列计算词表 logits：

$$
z_i=f_\theta(x_{\le i})
$$

再经过 softmax 得到概率分布 $p_i$。解码器根据策略从中选择一个离散 token：

- greedy：取概率最大的 token；
- sampling：按 temperature、top-k、top-p 处理后的分布采样；
- beam search：保留多条候选序列。

选出的 token id 会被追加到序列，查 embedding 表后用于下一步：

$$
\hat{x}_{i+1}\sim p_i,\qquad
z_{i+1}=f_\theta(x_{\le i},\hat{x}_{i+1})
$$

实际实现会复用历史 KV Cache，只对新增 token 做一次 decode，而不是重新计算整段序列。

### 训练阶段

标准 teacher forcing 不使用模型在第 $i$ 步预测出的 token，而是把数据中的真实 token 作为下一位置输入。这样训练可并行计算所有位置，也更稳定，但会带来 exposure bias：训练时总看到正确历史，推理时却要面对自己的错误。

Scheduled sampling 等方法会在训练时混合真实 token 与模型生成 token，但离散采样不可直接普通反向传播，且训练目标可能不一致，因此大语言模型预训练通常仍采用 teacher forcing。

### 概率分布能否直接输入

可以构造“期望 embedding”：

$$
e=\sum_v p_i(v)E_v
$$

它是可微的，但会把多个互斥 token 混成一个训练时存在、真实生成时不存在的软表示，容易产生 train-inference mismatch。Gumbel-Softmax、straight-through estimator 等方法会用于特定可微采样任务，但不是常规 LLM 解码流程。
