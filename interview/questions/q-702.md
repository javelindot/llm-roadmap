# vllm推理过程中哪些参数可以调优，结合经验来说。

## 答案

vLLM 调优应先确定目标是吞吐、TTFT、ITL 还是并发容量，并用接近真实的输入/输出长度分布压测。参数名和默认值随版本变化，应以当前版本官方文档为准。

### 显存与上下文

- `gpu_memory_utilization`：提高后可给 KV Cache 更多显存、降低 preemption，但过高会给 CUDA graph 和临时 buffer 留不够空间而 OOM。
- `max_model_len`：按真实业务限制上下文，设得过大会减少可容纳的并发 KV Cache。
- `kv_cache_dtype`：硬件支持时使用 FP8 KV 可增加容量，但需验证精度。
- 量化、CPU offload 等可让模型装下，但可能增加计算或 PCIe 开销。

### 调度参数

- `max_num_seqs` 控制单次迭代最大序列数。提高可能增加吞吐和显存占用，也可能恶化单请求延迟。
- `max_num_batched_tokens` 控制每轮处理的 token 预算。在 V1 chunked prefill 下，较小值通常有利于 decode 的 ITL；较大值可提高 prefill 吞吐和 TTFT 表现，具体需压测。
- prefix caching 适合大量共享 system prompt/前缀的场景。
- speculative decoding 适合 decode 瓶颈，但收益取决于 draft 接受率和额外开销。

### 并行与部署

单卡放不下时优先 tensor parallel；跨节点还要考虑带宽。MoE 可评估 expert parallel，多副本流量可用 data parallel。并行度并非越高越好，小模型跨卡通信可能降低吞吐。

### 经验化调优流程

先固定模型、硬件和请求分布，记录 TTFT、ITL、tokens/s、P95/P99、KV 使用率和 preemption。逐项扫描 `max_num_batched_tokens`、`max_num_seqs` 与显存比例，画 Pareto 曲线。长短请求混合时分池或启用 chunked prefill；线上再配 admission control 和最大输出限制，避免少数超长请求拖垮全局。
