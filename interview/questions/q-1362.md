# 你觉得当前RAG的最大瓶颈在哪?你做过哪些改进来提升Recall

## 答案

当前 RAG 的最大瓶颈往往不是向量库，而是“正确证据的定义与治理”：文档解析错误、切分破坏条件、版本冲突、权限和元数据缺失，会让后续 Embedding、Rerank 和大模型都无从补救。

### 1. 先做错误归因

将未召回分成：

- 文档根本未入库；
- OCR/表格解析错误；
- 正确内容被切散；
- Query 与文档表达不一致；
- 精确实体被 Dense 忽略；
- metadata 过滤错；
- ANN 参数导致近邻漏失；
- 正确文档在候选中但排名低；
- 金标本身不完整。

每类解法不同。

### 2. 数据与切分改进

- 去重、版本和生效时间治理；
- 标题、章节路径注入；
- 表格按完整业务单元保留；
- parent-child chunk；
- 文档摘要和 hypothetical questions；
- 页码、国家、币种、通道、产品和权限 metadata；
- 增量更新与删除校验。

### 3. Query 改进

- 多轮指代消解；
- 拼写和业务同义词；
- 保留订单号、错误码等精确实体；
- Multi-query 覆盖不同表达；
- 复杂问题拆分；
- 原 query 与改写 query 同时召回；
- 实时问题路由到工具。

### 4. 多路召回

```text
BM25
+ Dense
+ Structured/SQL
+ Graph/Entity
```

用 RRF 或学习融合。BM25 对编号和术语强，Dense 对语义强，结构化查询处理时间、金额和实体关系。

### 5. 模型与索引

- 用业务黄金集选择 Embedding；
- hard negative 微调；
- 调 HNSW `efSearch`、IVF `nprobe` 等；
- 检查 cosine/dot 与 normalize 是否匹配；
- 提高初始 TopK 后再 Rerank；
- 多语言可用跨语言模型或按语言路由。

提高 ANN 搜索参数只能解决近似搜索漏失，不能修复语义和数据问题。

### 6. 项目表达

建议报告真实消融：

| 方案 | Recall@10 | P95 |
|------|-----------|-----|
| Dense baseline | 实测 | 实测 |
| + 标题/metadata | 实测 | 实测 |
| + BM25/RRF | 实测 | 实测 |
| + Query Rewrite | 实测 | 实测 |
| + Embedding FT | 实测 | 实测 |

数字必须来自自己的评测，不要复用模板。

### 7. 跨境支付

支付查询常同时包含语义和硬实体。“巴西 PIX 失败 E102”应保留 PIX/E102 做 BM25，并用 Dense 理解“失败原因”，再按巴西、通道和版本过滤。

**面试总结**：Recall 提升是数据、查询、检索器和索引的联合工程；先定位漏在哪一层，再优化。
