# 介绍一些Agent的实现框架;这些框架有什么区别;LangGraph适用于什么场景;LangGraph构建Agent的方式有哪几种

## 答案

常见 Agent 框架包括 LangChain/LangGraph、LlamaIndex、AutoGen、Microsoft Agent Framework、Semantic Kernel、Haystack，以及各云厂商 SDK。选型应看状态管理、数据/RAG、多 Agent、持久化和团队生态。

### 1. 框架区别

- **LangChain**：模型、工具、Retriever、structured output 等通用集成；
- **LangGraph**：有状态图运行时，支持分支、循环、checkpoint、streaming、human-in-the-loop；
- **LlamaIndex**：数据 connector、索引、Retriever、Query Engine、RAG 和 Workflow 强；
- **AutoGen**：消息驱动的单/多 Agent、团队协作和研究原型；
- **Microsoft Agent Framework**：AutoGen/Semantic Kernel 的后继方向，面向企业 Agent 与 workflow；
- **Semantic Kernel**：Microsoft 生态的插件、规划和企业集成；
- **Haystack**：检索管道和组件化 NLP/RAG。

固定、强约束、高风险流程也可以使用普通状态机或工作流引擎，不必为了 Agent 引入重框架。

### 2. LangGraph 适用场景

- 流程有显式状态、分支和循环；
- 需要长任务 checkpoint 与失败恢复；
- 工具调用后根据结果动态路由；
- 需要暂停等待人工审批；
- 多 Agent 以 subgraph 协作；
- 要求流式事件、trace 和可回放。

简单单轮 tool calling 用 LangChain `create_agent` 或直接模型 API 即可，不一定手写图。

### 3. LangGraph 的构建方式

**预构建 Agent**

使用 LangChain `create_agent`，其底层运行在 LangGraph 上。适合标准 tool-calling loop，开发最快。

**Graph API**

使用 `StateGraph`：

```text
定义 state schema
-> add_node
-> add_edge / conditional edge
-> compile(checkpointer=...)
```

适合需要精确控制节点、状态、路由和恢复的生产工作流。

**Functional API**

使用 `@entrypoint`、`@task` 等把普通函数组织为持久化 workflow，适合不想显式维护图结构的代码风格。

**Subgraph / Multi-Agent**

把专业 Agent 或工作流封装为子图，由 supervisor、router 或共享状态组合。只有角色、权限或上下文确实需要隔离时才拆多 Agent。

### 4. 生产注意

- State 只保存必要、可序列化数据；
- 工具服务端鉴权、幂等、超时和重试；
- checkpoint 不等于业务事实数据库；
- Prompt、模型、工具和 schema 版本绑定；
- 限制循环、步骤、token 和成本；
- 高风险动作显式人工审批；
- 框架外维护评测和可观测性。

### 5. 跨境支付场景

支付调查可用 `StateGraph` 编排订单、通道、账务和规则查询，在退款节点中断等待审批。订单状态来自业务系统，LangGraph checkpoint 只保存流程状态。

### 6. 面试总结

LangGraph 强在持久化有状态工作流；构建可分预构建 Agent、Graph API、Functional API 和 Subgraph/Multi-Agent。选框架的核心是控制与恢复需求，不是组件越多越好。
