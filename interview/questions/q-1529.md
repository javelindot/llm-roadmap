# LangChain、LlamaIndex和AutoGen这三个框架如何选择？

## 答案

三者现在能力有较多重叠，不能再简单说“LangChain 做链、LlamaIndex 做 RAG、AutoGen 做多 Agent”。应根据核心问题、运行时要求和既有生态选择。

### 1. LangChain / LangGraph

LangChain 提供模型、工具、Retriever、structured output 和中间件等通用组件；其 Agent 运行时构建在 LangGraph 上，适合有状态工作流、分支循环、checkpoint、streaming 和 human-in-the-loop。

适合：

- 多工具业务 Agent；
- 流程需要显式状态和恢复；
- 模型/工具供应商集成很多；
- 团队愿意使用 LangGraph 管理复杂工作流。

风险是抽象与版本变化较快，生产中应把框架封装在薄适配层内。

### 2. LlamaIndex

LlamaIndex 以“让私有数据可被 LLM 使用”为核心，文档 connector、解析、Node、索引、Retriever、Query Engine、RAG 和 Workflow 能力完整，也支持 Agent。

适合：

- 多数据源接入；
- 文档、索引和检索策略复杂；
- 快速构建知识问答、数据 Agent；
- 需要围绕 RAG 反复实验。

若核心只是固定 API 编排，可能不需要引入其完整数据抽象。

### 3. AutoGen

AutoGen 适合事件驱动、消息式的单 Agent/多 Agent 协作、团队模式和研究原型。它可以表达多个角色通过消息交换解决任务，但多 Agent 不应成为默认方案，否则会增加延迟、成本和调试难度。

截至 2026 年，Microsoft Agent Framework 已作为 AutoGen 与 Semantic Kernel 的直接后继进入公开预览，并提供迁移指南。新建长期项目应同时评估 Agent Framework；已有 AutoGen 系统则根据稳定性和迁移成本决定，不必为了新框架立刻重写。

### 4. 选择矩阵

| 核心需求 | 优先评估 |
|---|---|
| 复杂有状态工作流 | LangGraph |
| 数据接入、索引、RAG | LlamaIndex |
| 消息驱动多 Agent 原型 | AutoGen |
| Microsoft 新项目与企业集成 | Agent Framework |
| 固定、强约束流程 | 自研状态机/工作流引擎 |

框架选型还要实测 checkpoint、幂等、trace、权限、测试、升级、延迟和成本。关键业务规则与工具鉴权应放在框架外的服务端。

### 5. 跨境支付场景

知识库与文档检索重可用 LlamaIndex；复杂调查流程可用 LangGraph；多个专业 Agent 协作只有在单 Agent + 工具基线不足时才考虑 AutoGen/Agent Framework。资金动作必须由后端状态机与审批控制，不能由框架会话自治。

### 6. 面试总结

LangGraph 强在状态工作流，LlamaIndex 强在数据与 RAG，AutoGen 强在消息式多 Agent；新 Microsoft 技术栈还应评估 Agent Framework。最终看业务控制、可恢复性和维护成本，而不是框架名气。
