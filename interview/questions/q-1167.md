# 评估OneRec和RankMixer的ROI，并分析为什么RankMixer更优

## 答案

OneRec 和 RankMixer 解决的问题并不相同，不能仅凭论文指标直接判断谁更优：

- **OneRec**：尝试用一个生成式模型统一召回与排序，直接生成推荐列表；
- **RankMixer**：保留工业级联链路，重点扩展精排模型容量和 GPU 利用率。

如果问题限定在“现有工业推荐系统、严格延迟和高 QPS 下的短期投入产出”，RankMixer 往往更容易获得较高 ROI；如果目标是重构整个推荐范式、追求列表生成和长期能力上限，OneRec 的潜在收益更大，但迁移风险与成本也更高。

### 1. OneRec 的收益与成本

OneRec 使用 encoder-decoder 编码用户历史并生成视频列表，主要设计包括：

- Sparse MoE 扩大模型容量；
- session-wise generation 一次生成一组上下文连贯的结果；
- Iterative Preference Alignment；
- 使用 reward model、自困难负样本和 DPO 做偏好对齐。

它的收益是减少召回、排序目标之间的割裂，并可以直接优化列表级偏好。OneRec 论文报告在快手主场景上线后 watch-time 提升 1.6%。

其成本和风险包括：

- 自回归生成带来的解码延迟和 GPU 成本；
- semantic ID、合法 item 约束和新增 item 更新；
- 召回、排序、重排一起替换时，问题定位和回滚更困难；
- 需要重新适配广告、运营、内容安全和多样性约束；
- reward model 和 DPO 数据可能放大反馈偏差；
- 一次性迁移现有多阶段系统的工程成本很高。

因此 OneRec 的 1.6% 是该论文特定平台和实验设置下的结果，不能与其他论文的线上指标直接横向相减。

### 2. RankMixer 的收益与成本

RankMixer 面向工业精排，把传统低 GPU 利用率的手工特征交叉模块替换为统一、硬件友好的结构：

- 使用 multi-head token mixing 建模 token 间交互；
- 使用 per-token FFN 保留不同特征子空间建模；
- 通过高并行结构提升 GPU Model FLOPs Utilization；
- 进一步使用 Sparse MoE 扩大容量，并通过动态路由改善专家不均衡。

论文报告的工程结果包括：

- MFU 从 4.5% 提升到 45%；
- 参数量扩大 100 倍，同时保持大致相同的推理延迟；
- 1B Dense-Parameters 模型全流量部署且未增加 serving cost；
- 线上用户活跃天数提升 0.3%，总使用时长提升 1.08%。

RankMixer 的主要改造集中在排序模型内部，现有召回、特征、业务约束和实验平台大多可以复用，因此迁移范围更小、可灰度性更好。它仍然需要大规模训练、GPU 服务和特征 token 化改造，但通常不需要一次性重写整条推荐链路。

### 3. 为什么在工业 ROI 口径下 RankMixer 可能更优

ROI 不应只看业务 uplift，还要除以总增量成本：

```text
ROI =
增量业务价值 - 训练/服务/迁移/风险成本
──────────────────────────────────
              增量总成本
```

在现有级联系统中，RankMixer 更可能占优的原因是：

1. **改造边界小**：替换精排模型，不必同时替换召回与重排；
2. **硬件利用率高**：模型扩容主要转化为有效 GPU 计算，而不是低效小算子；
3. **延迟可控**：论文目标就是在工业延迟和 QPS 约束下扩展；
4. **可渐进上线**：可以与旧精排并行 shadow、灰度和快速回滚；
5. **复用现有治理**：权限、内容安全、运营和广告约束仍在原链路中；
6. **收益归因清楚**：通过替换单一阶段更容易做消融和容量规划。

这不代表 RankMixer 在推荐能力上永久优于 OneRec。它只是对“近期扩展精排容量”的问题更匹配；OneRec 对“统一生成推荐”的问题更匹配。

### 4. 正确的 ROI 评估方案

不能直接比较两篇论文的百分比，应在同一业务、流量和预算中设计三组：

```text
A：现有召回 + 现有精排
B：现有召回 + RankMixer 精排
C：OneRec 生成式推荐，或 OneRec 作为一路候选
```

统一比较：

- 业务：时长、留存、转化、长期价值和负反馈；
- 质量：Recall、NDCG、多样性、长尾覆盖和有效 item 率；
- 系统：P95/P99、QPS、GPU 小时、显存和单位请求成本；
- 工程：训练周期、索引更新、发布频率和故障恢复时间；
- 风险：无效生成、内容安全、反馈回路和回滚复杂度。

先用 OneRec 作为 challenger 或一路召回，可以把范式迁移风险拆小；RankMixer 则适合作为现有精排的直接 challenger。最终结论应使用企业内部 A/B 的增量净收益，而不是跨论文对比。

### 5. 跨境支付中的选型

支付产品推荐或通道路由具有强合规和可执行性约束，更适合先采用级联系统：

> 硬规则生成合法候选 → 可扩展排序模型比较成功率、成本和时效 → 确定性服务执行。

RankMixer 类架构可以用于候选排序，但需要证明模型容量带来的收益覆盖 GPU 成本。OneRec 类生成模型更适合理解商户需求、补充候选和生成解释，不应直接生成未经校验的资金通道或执行资金动作。
