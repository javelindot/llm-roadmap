# 提及使用DeepSpeed开展SFT训练，请说明DeepSpeed ZeRO Stage 1 - 3的不同之处，以及何种情况下使用FSDP更为合适。

## 答案

ZeRO 的核心是逐步切分数据并行中重复保存的模型状态。设数据并行度为 \(N\)：

| 阶段 | 分片内容 | 参数是否每卡完整 | 特点 |
|---|---|---|---|
| ZeRO-1 | 优化器状态 | 是 | 改动小，主要节省 Adam 状态 |
| ZeRO-2 | 优化器状态 + 梯度 | 是 | 进一步节省梯度，通信通常较可控 |
| ZeRO-3 | 优化器状态 + 梯度 + 参数 | 否 | 显存最低，但前后向需按层 all-gather 参数 |

ZeRO-2 可将优化器卸载到 CPU；ZeRO-3 还可卸载参数到 CPU/NVMe，但会受 PCIe、CPU 和存储带宽限制。官方定义可参考 [DeepSpeed ZeRO 文档](https://deepspeed.readthedocs.io/en/stable/zero3.html)。

### 如何选择 ZeRO

- 模型参数本身能放入单卡，优化器/梯度放不下：优先 ZeRO-1/2；
- 模型参数也放不入单卡：使用 ZeRO-3；
- 网络较慢或模型较小：ZeRO-3 的频繁 all-gather 可能不划算；
- LoRA 训练需实测，基座权重仍占显存，但可训练参数很少，ZeRO-2 与 ZeRO-3 的收益结构不同。

### 何时更适合 FSDP

FSDP 与 ZeRO-3 思想相近，按模块在前向/反向时 all-gather，之后 reshard。以下情况更适合：

- 项目原生使用 PyTorch，希望减少第三方训练引擎依赖；
- 需要与 `torch.compile`、DTensor/DeviceMesh、PyTorch checkpoint 体系集成；
- 需要自定义 auto-wrap、mixed precision 和 hybrid sharding；
- 团队希望沿 PyTorch 主线维护。

截至 2026 年，PyTorch 官方推荐关注基于 per-parameter DTensor sharding 的 [FSDP2 `fully_shard`](https://docs.pytorch.org/docs/stable/distributed.fsdp.fully_shard.html)，其可用性与 FSDP1 有差异。

DeepSpeed 更适合已深度使用其 Trainer 集成、CPU/NVMe offload、Pipeline Parallel 或成熟配置体系的项目。最终应按峰值显存、tokens/s、checkpoint 时间和故障恢复实测，而不是按库名选型。

### 面试总结

ZeRO-1 分优化器，ZeRO-2 再分梯度，ZeRO-3 再分参数。模型权重放不下单卡才真正需要 Stage 3。FSDP/FSDP2 更适合 PyTorch 原生与 DTensor 生态，DeepSpeed 在 offload 和完整训练引擎方面更成熟。
