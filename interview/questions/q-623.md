# 对 Agent 框架有了解吗？

## 答案

Agent 框架的核心价值是管理“模型决策、工具调用、状态流转、持久化与观测”，而不是简单封装一次 LLM API。

### 常见框架定位

- LangChain 提供模型、Prompt、Retriever、Tool 等大量组件；LangGraph 更强调有状态图、循环、检查点和人工介入，适合复杂工作流。
- LlamaIndex 强项是数据接入、索引和 RAG，也提供 Agent 与工作流能力。
- AutoGen 偏向多 Agent 对话与协作模式，适合实验角色分工和消息驱动流程。
- Semantic Kernel 与应用工程、插件和企业生态集成较紧密。
- 对流程固定、可靠性要求高的业务，也常直接使用工作流引擎加自研薄层，避免框架抽象过重。

具体 API 更新较快，面试中更重要的是说明选型维度和落地经验。

### 选型维度

1. 是否支持显式状态机、循环、条件分支和并行步骤。
2. 是否能持久化 checkpoint，并在进程失败后恢复。
3. 工具参数是否有 schema 校验，是否支持超时、重试、幂等和权限控制。
4. 是否支持 streaming、人工审批、多租户和长任务。
5. 是否提供 trace、token 或费用统计、回放和离线评测。
6. 与现有模型、向量库、队列和部署平台的集成成本。

### 生产落地原则

关键业务流程应尽量显式化，模型只负责适合其判断的节点；支付、删除、发信等高风险操作必须经过权限检查和人工确认。Agent 状态外置，工具调用可审计，步骤设置预算、超时与最大循环次数。上线前用固定任务集评估完成率、工具选择正确率、事实性、延迟和成本。

因此应先画出任务状态图和失败恢复路径，再选择能最少侵入现有系统的框架，而不是先选框架再把业务塞进 ReAct 循环。
