# 规则召回是怎么做的，用到哪些规则？

## 答案

规则召回是根据请求上下文和确定性业务条件直接生成候选。它不是简单的“人工置顶”，而是一条高可解释、低延迟、可审计的候选通道，尤其适合模型数据稀疏、强运营要求和强业务约束场景。

### 1. 常见规则类型

- **准入规则**：地域、年龄、权限、会员等级、设备或产品可用范围；
- **行为触发**：浏览某类内容、完成某一步骤、连续搜索某主题后召回相关 item；
- **关联与互补**：购买主体商品后召回配件、服务或下一流程；
- **时间规则**：节日、活动、生效时间、最近行为和时效窗口；
- **热门与趋势**：按地区、渠道、时间窗或人群返回热门候选；
- **运营规则**：活动白名单、保量、新品扶持和特定内容池；
- **负向规则**：已消费、已拒绝、频控、投诉、下架和风险候选过滤；
- **兜底规则**：其他通道无结果或超时时提供安全候选。

规则召回应区分“候选生成”和“硬过滤”。召回规则负责提供可能相关的候选；权限、下架、合规等硬规则对所有召回通道统一生效，不能只在规则通道内部检查。

### 2. 实现方式

规则可以配置成版本化 DSL：

```text
WHEN country = \"BR\"
 AND currency = \"BRL\"
 AND beneficiary_type = \"individual\"
 AND amount <= limit
THEN recall product_pool = \"br_local_payout\"
     reason = \"country_currency_match\"
```

在线流程通常是：

1. 从请求和特征服务读取上下文；
2. 使用决策树、规则引擎、倒排索引或 bitmap 匹配规则；
3. 从指定候选池读取 Top-K；
4. 为候选附加 `rule_id、version、reason、priority、expire_time`；
5. 合并去重后送入统一粗排；
6. 再执行全局硬过滤和频控。

大规模标签规则不应逐条扫描，可预编译为倒排索引或 bitmap；复杂条件可由决策树或 RETE 类规则引擎执行。高频规则和候选池可以缓存，但生效时间、紧急下线和版本切换必须实时可控。

### 3. 跨境支付中的规则召回

商户产品推荐可以根据以下条件召回：

- 商户注册地、KYB 状态、行业和风险等级；
- 收付款国家、币种、金额区间和交易走廊；
- 企业或个人收款人；
- 本地付款、SWIFT、卡收单、全球收款账户或 FX 产品能力；
- 预计时效、费用偏好和历史产品使用；
- 通道营业时间、维护状态和实时容量。

例如巴西企业商户需要向个人收款人支付 BRL，可以先按国家、币种、收款人类型和额度召回可用的本地付款产品，再由后续服务比较费用、预计到账时间和通道成功率。

以下条件不能只做“召回加分”，而必须作为硬过滤：

- 牌照和产品覆盖范围；
- 制裁与反洗钱限制；
- 商户 KYB/KYC 和账户状态；
- 单笔、日累计和监管额度；
- 通道实时不可用；
- 商户、账户与数据访问权限。

LLM 可以把商户自然语言需求解析成结构化条件，但不能自行生成或修改这些规则，最终判断必须由权威规则和业务服务完成。

### 4. 规则治理

每条规则需要负责人、适用范围、优先级、生效与失效时间、审批记录和回滚版本。发布前进行：

- 语法与字段校验；
- 与现有规则的冲突和覆盖检测；
- 历史流量回放，估算命中量和候选变化；
- shadow 或小流量灰度；
- 监控异常命中、空结果和候选激增。

当多条规则冲突时，通常遵循：

> 合规与安全 > 用户负反馈和频控 > 产品准入 > 运营策略 > 个性化偏好。

### 5. 评测指标

- 规则命中率、候选有效率和最终采纳率；
- 独占召回贡献、与其他通道重合率；
- 空结果率、误召回率和硬过滤淘汰率；
- 不同国家、币种和产品的覆盖；
- P95/P99、缓存命中和规则发布延迟；
- 规则冲突数、过期规则数和回滚次数；
- 下游产品开通率、支付成功率、净收益和风险指标。

规则召回的优势是稳定、可解释和冷启动友好；缺点是维护成本高、覆盖有限，容易形成规则堆积。因此它应与协同过滤、双塔和内容召回并行，并通过增量实验定期清理没有价值的规则。
