# lora的原理，lora会不会更新原来的权重，如果对embedding层也配置了lora会不会更新。

## 答案

LoRA 的核心假设是：下游任务所需的权重更新具有较低的内在秩，因此不必训练完整的权重矩阵。对线性层 $W\in\mathbb{R}^{d_{out}\times d_{in}}$，LoRA 写成：

```text
W' = W + ΔW
ΔW = (alpha / r) * B * A
```

其中：

- `A` 的形状为 `[r, d_in]`；
- `B` 的形状为 `[d_out, r]`；
- `r` 远小于 `d_in` 和 `d_out`；
- 原始权重 `W` 冻结，只训练 `A` 和 `B`。

前向计算为：

```text
h = W*x + (alpha / r) * B*(A*x)
```

常见初始化让其中一个低秩矩阵为零，使训练开始时 $\Delta W=0$，模型初始输出与基础模型一致。

### 1. LoRA 会不会更新原权重

**标准 LoRA 训练不会更新原始权重。** 基础参数设置为 `requires_grad=False`，优化器中只包含 Adapter 参数。需要区分三种情况：

1. **训练阶段**：`W` 不变，A/B 更新；
2. **未合并推理**：每次前向同时计算基础分支和 LoRA 分支；
3. **merge 后推理**：把 $\Delta W$ 加入 $W$ 生成一个合并权重，便于部署。

merge 改变的是导出的推理权重表示，不等于训练过程中对基础参数做了梯度更新。只要保留原 base revision 和 Adapter，就可以 unmerge 或重新加载。

### 2. Embedding 层配置 LoRA 会怎样

Embedding 可看作形状为 `[vocab_size, hidden_size]` 的查表矩阵 `E`。如果框架对 `nn.Embedding` 实现了 LoRA，逻辑仍然是：

```text
E' = E + ΔE
```

- 原始 embedding 权重 `E` 仍冻结；
- 训练的是 embedding 对应的低秩 Adapter 参数；
- 前向结果包含 $\Delta E$ 对被访问 token 的增量；
- merge 后才会把增量写入导出的合并矩阵。

所以“给 embedding 配 LoRA”不等于“解冻原 embedding”。是否真的如此，要检查具体框架是否将该模块包装为 LoRA Embedding。

### 3. 容易混淆的配置

以下配置会让结论不同：

- **直接解冻 embedding**：原始 $E$ 会收到梯度并更新，这已不是纯 LoRA；
- **`modules_to_save`**：某些 PEFT 框架会额外完整训练并保存指定模块；
- **新增 special token**：扩充词表后，新 token 的 embedding 可能需要单独训练；
- **trainable tokens**：只更新指定 token 行，与低秩更新不同；
- **tied embeddings**：输入 embedding 与 `lm_head` 权重共享时，挂载、合并和保存语义要按框架实现核验；
- **量化基座**：QLoRA 中基础量化权重仍冻结，梯度通过反量化计算流向 Adapter。

最可靠的检查方法是打印全部参数的 `requires_grad`，统计 trainable parameter，并确认优化器参数组：

```python
trainable = [
    name
    for name, param in model.named_parameters()
    if param.requires_grad
]
print("\n".join(trainable))
```

还应在一个训练 step 前后比较 base embedding、LoRA 参数和合并权重，确认只有预期参数发生变化。

### 4. 是否应该对 Embedding 加 LoRA

大多数领域微调先从 attention 的 `q_proj、k_proj、v_proj、o_proj` 和 FFN 的 `gate_proj、up_proj、down_proj` 开始。以下情况才重点评估 embedding：

- 新增了大量领域 token 或特殊控制 token；
- 语言、脚本或代码符号分布变化明显；
- 消融实验表明 attention/FFN LoRA 已不足；
- tokenizer 和 embedding 处理方案允许可靠保存与部署。

跨境支付中的币种代码、通道名和业务缩写通常可以由已有 subword 表示，不应仅因为存在行业术语就盲目更新 embedding。规则、费率、账户状态和制裁名单变化快，应放在检索和工具层，而不是写入 embedding 参数。
