# 大模型相关岗位的判断标准有哪些？

## 答案

判断一个岗位是否值得做，不能只看 JD 写了“LLM/Agent”。应从业务价值、技术深度、数据权限、评测闭环、工程责任和团队资源六方面核实。

### 1. 业务是否真实

- 用户是谁，解决什么高频问题；
- 当前是 Demo、试点还是生产；
- 核心 KPI 和错误成本；
- 模型能力是否真正必要；
- 项目有没有预算、负责人和上线时间。

如果答案始终是“探索一下”，风险较高。

### 2. 工作内容

- 是训练/后训练、RAG/Agent、推理平台还是 API 集成；
- 日常时间在数据、实验、工程、汇报中的比例；
- 是否拥有从问题定义到线上指标的责任；
- 是核心研发还是项目交付/驻场；
- 是否需要频繁做低复用 Demo。

### 3. 数据与评测

- 能否访问合法、代表性的业务数据；
- 是否有黄金集、自动 verifier 和线上反馈；
- 是否能做 A/B、shadow 和事故回放；
- bad case 能否转成数据和产品改进；
- 模型升级是否有稳定回归流程。

没有数据和评测的“调模型”通常很难积累能力。

### 4. 技术与基础设施

- 可用模型、GPU、推理平台和工具链；
- 是否允许做微调、RAG、Agent 或只调用固定 API；
- 代码质量、CI/CD、监控和安全要求；
- 技术决策权与跨团队协作；
- 是否有资深同事和评审机制。

### 5. 成长与产出

- 6 个月和 1 年成功标准；
- 能否形成可展示的代码、论文、专利或业务结果；
- 技能是否可迁移到其他公司；
- 是否长期维护同一闭环；
- 职级、薪资、工作地点和强度是否匹配。

### 6. 面试反问

> 这个岗位当前最重要的线上指标是什么？

> 最近一次模型迭代从 bad case 到上线用了多久？

> 团队当前最大的瓶颈是数据、模型、推理还是产品？

> 新人前三个月具体交付什么？

> 模型输出错误时，谁负责评测、回滚和事故复盘？

### 7. 红旗

- 只讲模型名，不讲用户和指标；
- 只做套壳 Demo，却要求“精通所有大模型技术”；
- 没有数据、评测、GPU或上线权限；
- 责任全在个人但无安全/工程支持；
- 岗位描述与面试官实际业务明显不一致。

### 8. 面试总结

好的大模型岗位应有真实业务、可用数据、可靠评测、上线闭环、明确责任和可迁移成长。判断时看实际工作系统，不看标题里的热门词。
