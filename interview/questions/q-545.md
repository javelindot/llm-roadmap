# 讲一下最近很火的onerec？你认为onerec对你工作最大的启发是啥？

## 答案

OneRec 指 2025 年公开的 [OneRec](https://arxiv.org/abs/2502.18965)，目标是用一个生成模型统一传统推荐系统中的召回与排序，而不是让生成模型只承担召回。

### 1. 核心方法

- 使用 Encoder-Decoder 编码用户历史，并逐步解码可能感兴趣的视频；
- 引入稀疏 MoE，在不同比例增加 FLOPs 的情况下扩大容量；
- 从逐 item 的 next-item prediction 改为 session-wise generation，一次生成具有上下文连贯性的推荐列表；
- 使用 Iterative Preference Alignment：先训练奖励模型模拟反馈，再构造偏好样本进行 DPO，迭代改善列表质量。

论文报告其在快手主场景部署后带来 1.6% watch-time 提升。这个数字属于论文特定实验，回答时不能直接外推到其他业务。

### 2. 它解决了什么

传统级联的召回、粗排、精排分别优化局部目标，候选截断后信息不可恢复，且模块间可能目标不一致。OneRec 尝试用统一生成目标和列表级反馈减少这种错位。

### 3. 局限与后续方向

- 超大 item 空间下的约束解码与延迟；
- 新 item、下架 item 和索引新鲜度；
- 列表去重、多样性、探索和商业规则约束；
- 奖励模型偏差及迭代偏好数据的反馈回路；
- 统一模型故障时的影响范围和回滚难度；
- 生成模型收益是否足以覆盖训练和服务成本。

未来更可能是生成式主干与确定性约束、缓存、轻量服务头结合，而非完全取消所有级联模块。

### 4. 对工作的启发

最大的启发不是“所有系统都改成生成式”，而是：

1. 训练目标应尽量接近最终列表或任务结果；
2. 统一建模能减少模块目标错位，但必须保留可验证约束；
3. 用户反馈需要迭代回流，同时防止曝光偏差被强化；
4. 评估必须覆盖效果、延迟、成本和系统风险。

迁移到跨境支付时，可用于推荐支付产品或下一步操作，但可用通道、费率、牌照和用户资格必须由实时系统约束，模型不能生成不存在或不合规的选项。

### 面试总结

OneRec 用 Encoder-Decoder、稀疏 MoE、session-wise generation 和迭代 DPO 尝试统一召回与排序。它的核心价值是减少级联目标错位，核心挑战则是大空间解码、新鲜度、反馈偏差和线上成本。
