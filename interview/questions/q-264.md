# 遇到的困难？如何解决？

## 答案

**一句话**：项目难点不要只说“技术难”，最好讲清楚遇到什么问题、为什么难、怎么定位、怎么解决、最后带来什么效果。

### 1. 常见困难

AI 客服或 RAG 项目常见难点有：

- 知识库质量不稳定，文档版本多、内容重复。
- 用户问题口语化，和文档表达不一致。
- RAG 召回不准，相关内容排不到前面。
- 模型会幻觉，尤其在检索不到答案时编造。
- 工具调用参数容易缺失或错误。
- 多系统数据口径不一致。
- 延迟和成本难控制。
- 评估标准不清晰，无法判断优化是否有效。

### 2. 解决思路

我会按链路拆解：

- 数据侧：清洗文档、去重、加元数据、版本管理。
- 检索侧：query rewrite、混合召回、rerank、元数据过滤。
- 生成侧：约束必须基于引用回答，不知道就说不知道。
- 工具侧：参数校验、权限校验、异常兜底。
- 评估侧：构建真实工单评测集，持续看准确率和转人工率。
- 运维侧：日志、trace、监控、灰度和回滚。

### 3. 面试总结

讲困难时最好选择一个最能体现你能力的点，比如“RAG 准确率提升”或“多系统状态不一致排查”。不要泛泛说“模型不稳定”，要说你如何用工程手段把不稳定变可控。

### 4. 结合跨境支付项目怎么说（面试口径）

> 我遇到比较大的困难是支付异常问题的答案容易不准。用户问“为什么不到账”，可能原因很多：通道处理中、银行审核、风控拦截、收款人信息错误、国家假期、合规限制。单纯靠知识库检索，模型可能只召回到一个通用 FAQ，回答就会很泛。
>
> 我的解决方式是把问题拆成两部分：先用工具查实时交易状态和错误码，再用 RAG 查对应规则。比如查到通道回执是收款人账户无效，就只检索账户无效相关的处理 SOP，而不是全量检索“不到账”。同时给知识 chunk 加国家、币种、通道、错误码这些元数据，让检索更精确。
>
> 后续我们用真实工单做评测，看回答准确率、人工改写率和转人工率。这个过程让我比较深地理解到，大模型项目不是调一个 prompt 就结束，关键是数据、工具、流程和评估一起闭环。

**钩子**：可以引到“复杂支付异常拆解、实时工具 + RAG、真实工单评测闭环”。
