# 3．了解 RAG 吗， GraphRAG 的做法

## 答案

RAG 通过检索外部知识为 LLM 提供证据。普通向量 RAG 擅长定位具体片段，但对“整个语料有哪些主题、实体之间有什么跨文档关系”这类全局问题较弱。GraphRAG 用图索引和社区摘要补足这一点。

### GraphRAG 离线构建

1. 解析、清洗和分块原始文档。
2. 用 LLM 或信息抽取模型识别实体、关系和关键声明，合并同义实体并保留来源。
3. 构建带权知识图谱，节点代表实体，边代表关系或共现。
4. 使用 Leiden 等社区发现算法得到层级社区。
5. 为每个社区预生成包含主要实体、关系、事件和证据的摘要。

图谱和社区摘要都必须保留到原文 chunk 的引用，否则答案无法核验。

### 查询方式

- Local Search：从 query 命中的实体出发，取邻居、关系、相关原文和社区上下文，适合“某实体与谁有关”。
- Global Search：检索或遍历相关社区摘要，对每个摘要生成 partial response，再 map-reduce 聚合，适合“语料的主要主题是什么”。
- 还可将图检索与向量/BM25 混合，根据 query 意图路由。

### 优缺点

优势是支持跨文档、多跳关系和全局总结，可解释性也较强。代价是实体关系抽取与社区摘要需要较高离线成本，图谱可能有别名、错误边和更新延迟；简单事实查询未必优于普通 RAG。

评估应分别覆盖实体检索、多跳问答和全局 sensemaking，并检查引用正确率。生产方案通常保留 hybrid RAG，把 GraphRAG 作为特定查询类型的增强，而非完全替代文本索引。
