# ReAct的应用场景在哪里？

## 答案

**一句话**：ReAct 适合需要模型边思考边调用工具的场景，把 Reasoning 和 Acting 结合起来完成多步任务。

### 核心回答
ReAct 的模式是：
1. Thought：分析下一步要做什么。
2. Action：调用工具。
3. Observation：读取工具结果。
4. 再继续 Thought/Action，直到完成任务。

适用场景：
- 需要查外部系统的问题。
- 多步骤排查。
- 需要根据中间结果动态决策。
- 复杂问答、数据分析、代码调试、客服工单处理。

不适合：
- 简单 FAQ。
- 高风险动作没有审批的场景。
- 工具结果不可靠且缺少校验的场景。

### 1. ReAct 和普通问答的区别

普通问答通常一次生成答案。ReAct 会在推理过程中决定是否调用工具，并根据工具返回结果继续推理。它适合答案依赖外部状态的任务。

### 2. ReAct 的风险

如果工具权限不控制好，ReAct 可能误调用高风险接口。生产里要做：

- 工具白名单。
- 参数校验。
- 幂等设计。
- 高危动作二次确认。
- 工具调用日志。
- 失败回退。

### 3. 面试总结

ReAct 的价值是让模型能边查边想，适合多步排查；但高风险业务里必须限制动作权限。

### 4. 结合跨境支付项目怎么说（面试口径）

> 跨境支付里 ReAct 很适合异常排查。比如用户问"这笔款为什么没到账"，Agent 不能只靠知识库回答，它要先查订单状态，再查通道回执，再查清算批次，再看风控是否拦截。
>
> 每一步工具返回结果后，再决定下一步查什么。最后输出原因和建议，比如是通道回执延迟、账号信息错误、清算批次未完成，还是风控拦截。
>
> 但高风险动作比如退款、冻结、拒付处理，我会让 ReAct 只生成建议，必须人工确认后执行。支付场景里 ReAct 适合查证据，不适合无审批地直接动资金。

**钩子**：ReAct 适合查证据和归因，高风险资金动作必须审批。
