# 怎么处理对话嵌套？

## 答案

> **对话嵌套 = 在一个对话流程中，用户中途切换到子任务、子任务完成后回到主任务**。处理方式靠**对话栈 + 状态切换**。
>
> **典型嵌套场景**：
>
> 场景一：
> - 用户：我想申请退款（主任务：退款）
> - 系统：请提供订单号
> - 用户：先帮我查下汇率（子任务：汇率查询）
> - 系统：当前 1 USD = 7.2 CNY
> - 用户：好的，订单号是 ABC（回到主任务）
> - 系统：已为您查到订单 ABC，请确认退款……
>
> 场景二：
> - 用户：我的商户怎么入网（主任务：KYB 流程）
> - 系统：第一步需要营业执照
> - 用户：营业执照要什么版本（**澄清子问题**）
> - 系统：（解答）
> - 用户：好的（**回到主任务**）
>
> **核心实现：对话栈**
>
> 用栈数据结构管理嵌套：
>
> ```python
> class DialogStack:
>     def __init__(self):
>         self.stack: list[DialogContext] = []
>
>     def push(self, ctx: DialogContext):
>         self.stack.append(ctx)
>
>     def pop(self) -> DialogContext:
>         return self.stack.pop()
>
>     def top(self) -> DialogContext:
>         return self.stack[-1] if self.stack else None
>
> class DialogContext(BaseModel):
>     intent: str  # "refund" / "rate_query" / ...
>     state: dict  # 该任务的进度
>     slots: dict  # 该任务的槽位
>     created_at: datetime
> ```
>
> **嵌套切换的判定**：
>
> 每轮用户输入后，让 LLM 做四选一：
> ```
> 1. continue：继续当前任务
> 2. push：进入新子任务（嵌套）
> 3. pop：完成子任务，回到上一层
> 4. switch：完全切换到新任务（不嵌套，丢弃当前）
> ```
>
> Prompt 示例：
> ```
> 当前对话栈：[退款（订单号待补）]
> 用户最新消息：先帮我查下汇率
>
> 请判断动作类型：
> - continue / push / pop / switch
> 输出 JSON。
> ```
>
> **状态切换流程**：
>
> ```python
> async def handle_user_message(stack: DialogStack, msg: str):
>     decision = await llm_classify_action(stack, msg)
>
>     if decision == "push":
>         new_ctx = DialogContext(intent=decision.new_intent, ...)
>         stack.push(new_ctx)
>         return await handle_intent(new_ctx, msg)
>
>     elif decision == "pop":
>         finished = stack.pop()
>         resume_ctx = stack.top()
>         return await resume(resume_ctx, with_result=finished.result)
>
>     elif decision == "switch":
>         stack.stack.clear()
>         new_ctx = DialogContext(intent=decision.new_intent, ...)
>         stack.push(new_ctx)
>
>     else:  # continue
>         return await handle_intent(stack.top(), msg)
> ```
>
> **回归提示语**：
>
> 子任务完成后，主动告诉用户回到主任务：
> > "汇率查询完成。您刚才在申请退款，请继续提供订单号。"
>
> 这是关键 UX——让用户知道嵌套关系。
>
> **几个工程细节**：
>
> 1. **栈深度限制**：最多 3 层，太深就转人工
>
> 2. **超时清理**：子任务挂起 5 分钟没回，自动 pop
>
> 3. **槽位独立**：每层 context 有自己的槽位，不共享
>
> 4. **持久化**：栈状态存 Redis，断线重连恢复
>
> 5. **可视化 debug**：trace 显示对话栈每轮变化
>
> **常见错误处理**：
>
> - **歧义嵌套**：用户说"那个怎么办"，不知道指哪层 → 主动澄清
> - **遗忘子任务**：栈里挂了个老 ctx 没人理 → 超时清理
> - **嵌套深度爆炸**：用户老切话题 → 限制深度
>
> **不嵌套的简化方案**：
>
> 对话系统也可以**禁用嵌套**——任何新意图都 switch，简化逻辑：
> - 优点：实现简单
> - 缺点：用户体验差（"我刚才不是在说退款吗"）
>
> 我们跨境支付**客服 RAG** 选了"**有限嵌套**"——常见的"主任务 + 一层澄清"支持，多层嵌套直接转人工。
>
> 一句话：**对话嵌套用栈管理 + LLM 判定切换动作**——嵌套深度有限、状态独立、回归时主动提示，是工业级处理方式。
