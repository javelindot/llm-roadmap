# FastAPI、WebSocket、Flask、Socket，了解吗？

## 答案

> 都用过，分享一下我的实战经验和场景选型。
>
> **概念关系**：
>
> | 层级 | 项目 | 说明 |
> |------|------|------|
> | 底层网络 | Socket | TCP/UDP 原始协议 |
> | 协议 | HTTP / WebSocket | 应用层 |
> | Python 框架 | Flask、FastAPI、Django、Tornado | 封装上面 |
>
> Socket 是底层、HTTP/WS 是协议、Flask/FastAPI 是上层框架。
>
> **逐个详细聊**：
>
> **1. Flask（Python 老牌轻量框架）**
> - 同步阻塞、WSGI
> - 上手简单、生态丰富
> - 我用过：早期内部小工具、POC、写运维脚本接口
> - **缺点**：单线程阻塞，不适合高并发 LLM 服务
>
> **2. FastAPI（现代主流）**
> - 异步、ASGI、基于 Starlette + Pydantic
> - 自动生成 OpenAPI / Swagger 文档
> - 强类型 + 数据校验
> - 性能远超 Flask
> - 我们生产**90% 服务用 FastAPI**：RAG 服务、Agent 服务、内部 API
>
> 典型用法：
> ```python
> from fastapi import FastAPI
> from pydantic import BaseModel
>
> app = FastAPI()
>
> class Query(BaseModel):
>     text: str
>
> @app.post("/chat")
> async def chat(q: Query):
>     answer = await rag_service.answer(q.text)
>     return {"answer": answer}
> ```
>
> 部署：`uvicorn main:app --workers 4 --host 0.0.0.0 --port 8000`
>
> **3. WebSocket（长连接、双向通信）**
> - 持久 TCP 连接，服务端能主动推
> - 适合：实时聊天、流式输出、推送通知
> - 我们用过：
>   - **客服流式回复**：LLM 生成每个 token 通过 WS 推前端
>   - **Agent 步骤通知**：长任务进度推送
>
> FastAPI 内置 WS：
> ```python
> @app.websocket("/ws/chat")
> async def ws_chat(ws: WebSocket):
>     await ws.accept()
>     async for token in llm.stream(query):
>         await ws.send_text(token)
> ```
>
> 实际生产里我们**更多用 SSE（Server-Sent Events）**，理由：
> - 单向流（从服务端推），LLM 场景够用
> - 用普通 HTTP，过 nginx / CDN 无障碍
> - WS 经常被中间代理切断
>
> **4. Socket（底层 TCP/UDP）**
> - 直接用 socket 库写网络程序
> - 我用过：写过 Redis 协议解析的客户端、TCP 长连接监听
> - **生产几乎不直接写 Socket**——用上层框架更稳
>
> **我们生产里的具体选型**：
>
> | 场景 | 用什么 | 原因 |
> |------|--------|------|
> | RAG 普通问答 | FastAPI HTTP | 简单可靠 |
> | 流式回复 | FastAPI + SSE | 比 WS 友好 |
> | Agent 长任务进度 | WS / SSE | 服务端推 |
> | 微服务间通信 | gRPC（不是 socket） | 性能好、类型安全 |
> | 内部工具 | FastAPI / Flask | 灵活 |
>
> **几个工程经验**：
>
> 1. **FastAPI + uvicorn workers** 是 Python 后端标配
>
> 2. **WS / SSE 都要做心跳**——nginx / 负载均衡器会切断空闲连接
>
> 3. **流式响应 + 反向代理**：nginx 要关 buffering（`proxy_buffering off`）
>
> 4. **asyncio 不能 block**——任何 sync 操作（DB、CPU 计算）要用 `run_in_executor` 包
>
> 5. **uvicorn 不要在 windows 用 gunicorn**——选 hypercorn 或 uvicorn 多 worker
>
> 6. **错误处理**：FastAPI 用 `Exception Handlers` 统一返回 JSON 错误格式
>
> **我们 LLM 服务的标准模板**：
> - FastAPI + uvicorn (4 workers per pod)
> - SSE 流式推 token
> - Pydantic 校验
> - OpenTelemetry 自动 trace
> - Sentry 异常上报
> - Prometheus metric
>
> 一句话：**FastAPI 是 Python 后端首选、WS/SSE 看推送方向、Socket 一般不直写**——LLM 服务标配是 FastAPI + SSE + uvicorn。

**🪝 顺嘴带出去的钩子**：
- "SSE vs WebSocket 在 LLM 流式输出的选择可以聊。"
- "FastAPI 高并发部署 nginx + uvicorn 的配置可以展开。"
- "Python asyncio 在 LLM 服务里怎么用得对。"
