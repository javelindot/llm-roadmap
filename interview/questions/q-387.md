# 怎么保存查询结果、上下文？

## 答案

> 分**短期上下文**和**长期记忆**两类，存储方案不同。详见 q-225 上下文压缩、q-341 记忆遗忘；这里讲存储和检索的具体实现。
>
> **短期上下文（会话内）**：
>
> 用 **Redis**，TTL 30 分钟：
>
> ```python
> class SessionContext(BaseModel):
>     conversation_id: str
>     messages: list[Message]  # 对话历史
>     intents: list[Intent]    # 槽位 / 意图栈
>     last_query: str
>     last_result: dict
>     updated_at: datetime
>
> async def save_context(ctx: SessionContext):
>     await redis.setex(
>         f"ctx:{ctx.conversation_id}",
>         1800,  # 30 分钟
>         ctx.model_dump_json()
>     )
>
> async def load_context(conversation_id: str) -> Optional[SessionContext]:
>     data = await redis.get(f"ctx:{conversation_id}")
>     return SessionContext.model_validate_json(data) if data else None
> ```
>
> 为什么用 Redis：
> - 单 K-V 操作微秒级
> - TTL 自动清理
> - 支持复杂数据结构
> - 多副本高可用
>
> **长期记忆（跨会话）**：
>
> 用 **数据库 + 向量库**组合：
>
> ```python
> class LongTermMemory(BaseModel):
>     user_id: str
>     memory_id: str
>     content: str         # 记忆内容（事实 / 偏好 / 上下文摘要）
>     memory_type: str     # "fact" / "preference" / "summary"
>     embedding: list[float]
>     importance: float    # 重要度 0-1
>     created_at: datetime
>     last_accessed: datetime
> ```
>
> 存两路：
> - PostgreSQL：结构化存
> - Milvus：向量检索（语义召回）
>
> 检索时：
> ```python
> async def recall_memory(user_id: str, query: str, k=5):
>     q_emb = await embed(query)
>     # 在该用户的记忆里向量检索
>     candidates = await milvus.search(
>         q_emb,
>         filter=f"user_id == '{user_id}'",
>         top_k=20
>     )
>     # 按重要度 + 时效性重排
>     return rerank_by_importance(candidates)[:k]
> ```
>
> **查询结果缓存**：
>
> 高频 query 答案直接 cache：
>
> ```python
> async def cached_rag(query: str, ttl=300):
>     cache_key = f"rag:{hash(normalize(query))}"
>     if cached := await redis.get(cache_key):
>         return cached
>
>     answer = await rag.answer(query)
>     await redis.setex(cache_key, ttl, answer)
>     return answer
> ```
>
> 注意：
> - **个性化 query 不能 cache**——含用户 ID 的不行
> - **时效性敏感不能 cache 长**——汇率、库存
> - **TTL 短**（5-15 分钟）
>
> **Agent 中间状态**：
>
> LangGraph 的 checkpoint：
>
> ```python
> from langgraph.checkpoint.redis import RedisSaver
>
> graph = StateGraph(State)
> # ... build graph
> compiled = graph.compile(checkpointer=RedisSaver(redis_client))
>
> # 每次执行 state 自动持久化
> async for state in compiled.astream({"input": query}, {"thread_id": "user_123"}):
>     pass
> ```
>
> 优点：
> - Agent 任务中断后可从最近 checkpoint 恢复
> - 不丢失中间结果
> - 调试可回放
>
> **trace 上下文**：
>
> 完整 LLM 调用记录入 **ClickHouse**：
>
> ```python
> async def log_llm_call(trace_id, query, context, response, metrics):
>     await clickhouse.insert(
>         "llm_traces",
>         {
>             "trace_id": trace_id,
>             "query": query,
>             "context": json.dumps(context),
>             "response": response,
>             "tokens": metrics.tokens,
>             "latency_ms": metrics.latency,
>             "timestamp": datetime.now()
>         }
>     )
> ```
>
> 用于：
> - 失败 case 复盘
> - 审计追溯
> - 数据回流（喂训练）
>
> **存储分层总结**：
>
> | 存储 | 用途 | TTL | 容量 |
> |------|------|-----|------|
> | **Redis** | 短期会话、缓存 | 30 分钟 | GB 级 |
> | **PostgreSQL** | 长期记忆、用户档案 | 永久 | 100 GB+ |
> | **Milvus** | 语义检索 | 永久 | TB 级 |
> | **ClickHouse** | trace 日志 | 6 个月 | TB-PB |
> | **OSS** | 原始数据归档 | 永久 | PB 级 |
>
> **几个工程经验**：
>
> 1. **不要全塞 Redis**——内存贵，只放热数据
>
> 2. **会话 ID 设计**：用户 ID + UUID，可追溯
>
> 3. **过期策略**：TTL + 主动清理（防 Redis 雪崩）
>
> 4. **多端同步**：用户多设备登录，会话独立
>
> 5. **隐私脱敏**：trace 入库前 PII 脱敏
>
> 6. **持久化策略**：重要状态多副本，临时缓存可丢
>
> 一句话：**查询结果 + 上下文要分层存储**——Redis 热、DB 持久、Milvus 语义、ClickHouse 审计、OSS 归档，按数据生命周期匹配存储介质。
