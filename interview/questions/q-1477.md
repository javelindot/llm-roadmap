# 上来就问我这题要考我什么，我答策略模式。后面就是自己写，没写完，但是面试官很看重细节（边界判断，特殊情况......这些才是他要考察的点）

## 答案

题干没有给出具体代码题，因此无法还原唯一实现。从复盘描述看，面试官考察的不只是“能否认出策略模式”，而是能否把变化点抽象出来，并把边界、异常和可测试性落实到代码。

### 1. 策略模式解决什么

当同一业务有多种可替换算法时，定义统一接口：

```python
from abc import ABC, abstractmethod


class Strategy(ABC):
    @abstractmethod
    def execute(self, context):
        pass


class StrategyA(Strategy):
    def execute(self, context):
        return "A"


class StrategyB(Strategy):
    def execute(self, context):
        return "B"
```

Context 依赖抽象而不是大量 `if/else`：

```python
class Context:
    def __init__(self, strategy):
        self.strategy = strategy

    def run(self, data):
        return self.strategy.execute(data)
```

### 2. 面试官更看重的细节

写之前先确认：

- 输入能否为空；
- 参数是否合法；
- 默认策略；
- 找不到策略如何处理；
- 是否线程安全；
- 策略是否有状态；
- 策略失败是否回退；
- 是否允许运行时切换；
- 返回值和异常约定；
- 相同请求是否幂等。

### 3. 避免过度设计

只有算法确实可替换、会扩展或需要独立测试时才用策略模式。若只有两个固定分支且不会变化，一个清晰的函数或 `if` 更简单。

### 4. 工厂与注册表

```python
STRATEGIES = {
    "a": StrategyA(),
    "b": StrategyB(),
}


def get_strategy(name):
    if name not in STRATEGIES:
        raise ValueError(f"unknown strategy: {name}")
    return STRATEGIES[name]
```

实际代码要考虑重复注册、配置错误和实例生命周期。

### 5. 测试清单

- 每个策略正常输入；
- 空值、最小值、最大值；
- 未知策略；
- 异常回退；
- 多次调用；
- 并发；
- 策略切换；
- side effect；
- 性能边界。

### 6. 跨境支付场景

支付通道路由可抽象不同策略：成本优先、时效优先、成功率优先。但制裁、牌照、币种和限额必须先做硬过滤，不能作为可随意替换的策略。

### 7. 面试总结

识别出策略模式只是第一步。真正考察的是：抽象是否恰当、接口是否稳定、边界是否完整、失败是否可控、测试是否覆盖。题干缺失时应补全需求后再编码。
