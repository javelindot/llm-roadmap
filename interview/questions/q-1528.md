# 多工具多Prompt下，消融实验怎么展开？

## 答案

多工具、多 Prompt 的消融不能把所有组合穷举。应先把系统拆成可开关的因子，使用分阶段实验和因子设计，区分主效应与交互效应。

### 1. 定义因子

Prompt 因子：

- system policy；
- planner prompt；
- tool description；
- reflection/verifier prompt；
- memory summary prompt；
- few-shot 示例。

工具因子：

- 工具是否可见；
- 单工具与组合工具；
- 参数 schema；
- 路由策略；
- 并行/串行；
- 超时、重试和 fallback。

还要冻结模型、数据、采样、知识库和后端版本，否则无法归因。

### 2. 分阶段实验

**第一阶段：单模块诊断**

在只需要一个工具的样本上，分别测工具选择、参数正确率和执行成功率；Prompt 则用固定工具结果测试计划与回答。

**第二阶段：单因素消融**

从完整系统逐个移除模块：

```text
Full
- memory
- reflection
- tool X
- few-shot
- query rewrite
```

获得主效应，但不能发现强交互。

**第三阶段：交互实验**

对预期相关的因子做 `2 x 2`：

```text
planner prompt on/off
tool descriptions rich/simple
```

工具很多时使用正交/分数因子设计，先筛选重要因子，再对少数组合全量测试。

### 3. 数据与执行控制

- 按任务类型、工具数量、难度和风险分层；
- 使用固定 case、固定后端快照和工具响应；
- 生成任务使用多随机种子，报告均值和置信区间；
- 同一 case 对不同版本配对比较；
- 测试顺序随机化，避免后端负载偏差；
- 保留真实故障、冲突工具和缺失参数案例。

### 4. 指标

- 端到端任务成功率；
- 工具选择/参数/调用顺序准确率；
- 计划步骤数、无效调用和循环率；
- 事实正确率和安全违规；
- P95、token、工具次数和成本；
- 各任务切片与严重错误。

除了最终结果，还要比较 trajectory，判断某模块是提高规划、工具还是仅改善表达。

### 5. 跨境支付场景

可把订单查询、通道查询、知识检索和退款规则工具分开消融。实验使用沙箱或历史快照，高风险写工具始终禁用。若移除某工具后 Agent 仍“回答成功”，要检查是否在编造订单状态。

### 6. 面试总结

先做模块级单因素消融，再对关键 Prompt × 工具做交互实验；组合过多时用分数因子设计。固定其他版本、按任务分层、重复多随机种子，并同时评端到端和轨迹指标。
