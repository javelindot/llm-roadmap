# 生成式推荐的 ROI问题

## 答案

生成式推荐的 ROI 不能只比较离线指标，而要衡量“相对现有级联系统带来的增量业务价值”能否覆盖训练、推理、数据、研发和迁移成本：

```text
ROI =
增量业务收益 - 增量总成本 - 风险预期损失
────────────────────────────────────
               增量总成本
```

### 1. 收益项

可能的收益包括：

- 统一部分召回、排序和重排逻辑，减少多模型目标不一致；
- 利用长行为序列和上下文生成更连贯的列表；
- 通过 semantic ID 或生成式检索增强长尾候选；
- 使用偏好优化直接学习列表级或长期反馈；
- 多场景共享模型和数据，减少重复建设；
- 新任务通过提示、微调或新增 token 更快接入。

收益最终应映射到可计价指标，如增量时长、转化、净 GMV、广告收入、留存或人工维护成本下降。离线 Recall、NDCG 或生成准确率只是中间指标。

### 2. 成本项

生成式推荐常被低估的成本包括：

- 大模型预训练、微调、偏好对齐和周期性重训；
- 自回归解码、beam search 和 KV cache 带来的推理成本；
- semantic ID 构建、版本迁移和新增 item 增量更新；
- 受约束解码、无效 ID 防护和候选可用性校验；
- GPU 容量、峰值 QPS、容灾、监控与在线特征服务；
- 与现有召回、精排、广告和运营系统并存的迁移成本；
- 生成结果不可控、反馈回路和错误曝光带来的风险损失。

不能用“一个生成模型替代多个模型”直接推导成本下降。即使模型数量减少，单次推理 FLOPs、显存和尾延迟也可能显著上升。

### 3. 建立可比较的实验口径

应在相同流量、候选可用性、延迟预算和业务约束下比较：

```text
Baseline：现有多路召回 + 粗排 + 精排 + 重排
Challenger：生成式召回，或生成式端到端推荐
```

离线评测包括：

- Recall@K、NDCG、HitRate、列表多样性和长尾覆盖；
- 有效 item 率、重复率、下架或越权候选率；
- P50/P95/P99、吞吐、单请求 GPU 时间和单位成本；
- 模型训练时间、数据新鲜度和新增 item 生效时延。

线上使用 shadow、灰度和 A/B 实验，观察主业务指标，同时设置延迟、投诉、负反馈、内容安全和成本守护指标。ROI 应使用增量收益而不是实验组总收益，并计算置信区间。

### 4. 分阶段落地更容易获得正 ROI

可以按风险从低到高演进：

1. 先用生成模型做离线特征、意图理解或内容表征；
2. 作为现有系统的一路召回，测量独占增量；
3. 用生成式模型做重排或列表生成，但保留合法候选白名单；
4. 在低风险场景尝试统一召回与排序；
5. 只有质量、成本和稳定性均满足门槛后再扩大流量。

OneRec 采用 encoder-decoder、Sparse MoE、session-wise generation 和迭代偏好对齐，论文报告在快手主场景取得 1.6% 的 watch-time 提升。这说明端到端生成式推荐可能产生业务增量，但该结果来自特定平台、数据和基础设施，不能直接当作其他公司的 ROI 预期。

### 5. 跨境支付中的 ROI 边界

跨境支付更适合先把生成模型用于：

- 理解商户的国家、币种、金额、时效和成本偏好；
- 从合法产品池中补充候选收款、付款或换汇方案；
- 生成产品对比和推荐理由；
- 辅助客服和销售发现商户需求。

资金路由、报价、制裁筛查、KYB、额度和风险决策必须由确定性服务执行。由于支付场景 QPS 可能低于内容推荐，但单次错误成本更高，ROI 公式中应显式加入错误承诺、错误路由、合规和人工复核成本。

是否值得上线，取决于单位增量收益：

```text
单位增量收益 =
新增开通/使用带来的净收入
- GPU 与工程成本
- 人工复核成本
- 风险和合规预期损失
```

生成式方案若只是让回答更自然，却没有提高产品开通、问题解决或支付成功，也没有降低人工成本，就不能证明具有正 ROI。
