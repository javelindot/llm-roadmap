# 你是否熟练掌握Python？若项目涉及跨语言开发需求，会采取怎样的解决方案？

## 答案

**一句话**：我熟悉 Python 在 AI 工程里的常见用法，包括数据处理、RAG、模型调用、评估脚本和服务封装；如果项目涉及跨语言，我一般用服务化接口、消息队列或统一协议把语言边界隔离开。

### 1. Python 掌握到什么程度

在 AI 项目里，Python 主要承担这些工作：

- 数据清洗、切分、抽取和格式转换。
- RAG 的文档解析、embedding、向量入库、检索评估。
- 模型推理、微调脚本、评测脚本。
- FastAPI/Flask 封装模型服务。
- 调用第三方 LLM API、向量库、OCR、PDF 解析组件。
- 写离线任务、定时任务、实验脚本和自动化工具。

我不会把 Python 只当脚本语言看，它在 AI 工程里更像“模型能力和业务系统之间的胶水层”。

### 2. 跨语言开发怎么做

如果主业务系统是 Java/Spring，而模型能力用 Python，更推荐把 Python 模块服务化，而不是把代码强行混在一起。

常见方案有：

- **HTTP/OpenAPI**：适合普通业务调用，调试方便。
- **gRPC**：适合高性能、强契约、内部服务调用。
- **消息队列**：适合异步任务，比如文档解析、向量构建、批量评估。
- **SDK 封装**：调用方只依赖统一 SDK，底层可以是 Python、Java 或其他服务。
- **任务调度平台**：离线训练、批处理、定时同步适合用调度系统管理。

接口层要统一处理超时、重试、幂等、traceId、错误码、版本号和安全认证，不能只把模型服务暴露出去就结束。

### 3. 工程注意点

跨语言最大的风险不是语法，而是契约和运维。

- 数据结构要统一，最好用 OpenAPI、Protobuf 或 JSON Schema 管理。
- 错误码要标准化，不能让 Python 异常直接透传给 Java 业务层。
- 关键链路要有 traceId，方便跨服务排查。
- 模型服务要有超时、熔断和降级，避免拖垮主交易流程。
- 版本要可灰度，比如 embedding 模型升级不能直接影响线上召回。

### 4. 面试总结

我对 Python 的定位是：在 AI 能力、数据处理、模型实验和推理服务方面效率很高；但核心交易系统更适合用稳定的工程语言承载。跨语言时，我会通过服务边界和协议边界来解耦，而不是让系统互相侵入。

### 5. 结合跨境支付项目怎么说（面试口径）

> 在跨境支付业务里，我更倾向于让 Java 负责核心交易链路，比如收款、付款、账务、清结算和状态流转；Python 负责 AI 相关能力，比如客服知识库、交易异常解释、文档解析、风控特征分析和模型评估。两边通过 HTTP 或 MQ 交互。
>
> 比如用户上传一份 KYB 材料，Java 系统负责接收文件、鉴权、落库和发起任务；Python 服务负责 OCR、PDF 解析、字段抽取和风险标签识别；处理完之后通过 MQ 回写结果。这样即使 Python 侧模型服务变慢，也不会直接阻塞支付主链路。
>
> 我会特别关注三个点：接口契约、超时降级和链路追踪。因为支付系统不能因为一个模型服务异常就影响用户付款，所以模型能力一定要做成可降级的旁路能力。

**钩子**：这个问题可以引到“AI 服务旁路化”“Python + Java 服务边界”“支付主链路稳定性”。
