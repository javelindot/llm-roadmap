# 怎么实现u-u-i的召回？

## 答案

U-U-I 召回是 User-User-Item 的协同过滤路径：

> 先找到与目标用户相似的用户，再把相似用户交互过、但目标用户尚未交互的 item 召回。

它的核心假设是“行为相似的用户，未来可能喜欢相似的 item”。

### 1. 构建用户与 item 的行为矩阵

先把点击、收藏、购买等行为映射为权重：

```text
r(u, i) = action_weight × time_decay × quality_weight
```

例如购买权重大于点击，最近行为高于很久以前的行为，无效订单、退款或强负反馈需要降权或排除。时间衰减可使用：

```text
time_decay = exp(-λ × time_gap)
```

由此得到稀疏的 User-Item 矩阵。

### 2. 计算用户相似度

最基础可以使用 cosine similarity：

```text
sim(u, v) =
Σi r(u,i)r(v,i)
────────────────────────────
sqrt(Σi r(u,i)²) sqrt(Σi r(v,i)²)
```

也可以使用 Jaccard、Pearson 或加权共现。热门 item 会让大量用户产生虚假相似，因此常用 IIF 惩罚：

```text
cooccur(u, v) += 1 / log(1 + user_count(i))
```

即越热门的 item，对用户相似度的贡献越小。还可以对短时间内共同发生的行为赋予更高权重，增强兴趣时效性。

工程上不会全量计算所有用户对。通常从 item 的倒排表出发，只为共同交互过 item 的用户累积共现，再为每个用户保留 Top-N 相似邻居。超大规模场景可以使用 MinHash/LSH、用户 embedding ANN 或按人群分桶减少计算量。

### 3. 从相似用户扩展 item

对目标用户 `u`，取相似用户集合 `N(u)`，候选 item 得分可写为：

```text
score(u, i) =
Σv∈N(u) sim(u,v) × r(v,i) × freshness(i)
```

然后完成：

- 删除目标用户已消费或明确负反馈的 item；
- 删除下架、无权限或不适用候选；
- 对同一用户的重复行为去重；
- 对热门 item 做降权，避免结果全部集中在头部；
- 设置每个邻居的最大贡献，防止单个重度用户控制结果；
- 与 ItemCF、双塔、内容和热门召回合并。

候选最好保留相似用户贡献、共同 item 和召回理由，便于调试。但面向用户展示解释时不能泄露其他用户身份或行为。

### 4. 样本和偏差处理

U-U-I 对隐式反馈非常敏感：

- 未点击可能是未曝光，不能直接视为负反馈；
- 机器人、批量操作和异常账户会污染共现；
- 热门 item 容易造成所有用户相似；
- 新用户和新 item 缺少行为，冷启动能力弱；
- 用户兴趣变化快时，长期邻居可能已经失效。

因此要做曝光偏差校正、异常行为过滤、热门惩罚和时间衰减，并用内容召回、地域热门或画像相似作为冷启动补充。

### 5. 跨境支付中的迁移方式

跨境支付更适合使用 Merchant-Merchant-Product，而不是未经授权跨商户共享终端用户交易：

> 目标商户 → 相似商户群组 → 群组常用的支付产品或能力。

相似度可以来自注册地、行业、交易走廊、币种结构、金额区间和已开通产品；候选可以是全球收款账户、本地付款、FX、卡或收单产品。

数据边界必须严格控制：

- 优先使用商户自己的第一方行为；
- 跨商户统计必须匿名聚合、满足最小群组规模和用途授权；
- 不使用银行卡号、收款人身份等敏感特征做跨租户近邻；
- 推荐结果仍要经过 KYB、牌照、额度、风险和产品准入校验；
- 不能展示“某个具体商户使用了某产品”之类的泄露性解释。

对于资金通道路由，也可以利用相似交易走廊的聚合成功经验生成候选，但实时通道状态、成本、风险和合规规则必须由确定性服务重新校验。

### 6. 评测

- Recall@K、HitRate@K、MRR、NDCG；
- 邻居覆盖率、item 覆盖率和长尾覆盖；
- 热门度分布、新用户和稀疏用户分桶表现；
- 与其他召回通道的重合率和独占贡献；
- 离线计算耗时、邻居表大小、在线 P95/P99；
- 下游转化、产品开通、支付成功和风险指标。

U-U-I 结构简单、可解释，适合行为相对稳定且用户规模可控的场景；当用户规模大、兴趣复杂或冷启动占比高时，应与双塔、内容召回和图模型组合使用。
