# 介绍一下常见的文本分块方法

## 答案

**一句话**：分块是 RAG 的"灵魂"——分得太大召回噪声多、太小语义碎片化；主流五种方法是**定长 / 滑窗 / 递归 / 语义 / 文档结构感知**，工业里常用"**结构感知 + 重叠滑窗 + 元数据**"组合拳。

### 1. 定长分块（Fixed-size）
- 按字符 / token 切等长块
- 简单粗暴，最常见
- 缺点：硬切可能切断句子和段落

### 2. 滑窗分块（Sliding Window）
- 定长 + 块间 overlap（如 50-100 token）
- 减少边界处语义丢失
- 同一句话可能出现在多个块里

### 3. 递归分块（Recursive Character）
- 优先按段落（`\n\n`）切，超长则按句号、再按词
- LangChain `RecursiveCharacterTextSplitter` 经典实现
- 平衡定长和语义边界

### 4. 语义分块（Semantic Chunking）
- 用 embedding 衡量相邻句子语义距离
- 距离突变处切块
- 块大小不固定，按语义自然停顿
- 代表：LlamaIndex SemanticSplitter

### 5. 文档结构感知分块（Structure-aware）
- 根据 Markdown 标题、章节、HTML tag 切
- 保留文档原始层级
- 每块附带 path（如 `第3章 > 3.2 退款政策`）
- 最适合知识库 / 文档型 RAG

### 6. 高级方法
| 方法 | 思路 | 用例 |
|------|------|------|
| **Proposition Chunking** | 一句话一个独立命题 | 精细 QA |
| **Late Chunking** | 先全文 embed 再切 | 保上下文 |
| **Agentic Chunking** | LLM 自己决定怎么切 | 高质量小规模 |
| **Multimodal Chunking** | 图文混合 | 多模态 RAG |

### 7. 工业经验值
- 块大小：**200-500 token**（中文）/ 256-512（英文）
- overlap：块大小的 **10-20%**
- 元数据：必带 `source、section、page、create_time`
- 嵌入前**清洗**：去 header/footer、统一格式

### 8. 怎么选
- 通用知识库 → **递归 + overlap**
- 结构化文档（手册、政策） → **结构感知 + 路径**
- 短问答型 → **小块（200 token）**
- 长推理型 → **大块（500-1000 token）+ rerank**
- 多模态 → **图文混合块 + VLM 描述**

### 9. 评测分块好坏
- 召回率（recall@k）
- 答案完整性（人工 spot check）
- 端到端 RAG accuracy
- 块大小分布、空块比例

### 10. 一句话总结
**没有最好的分块方法，只有最匹配文档结构的**——工业默认"递归 + 200-500 token + 10% overlap + 结构元数据"，再按场景微调。

## 结合项目谈（口语版 · 约 30 秒）

> 分块在我们跨境支付 RAG 里是**最反复迭代的环节**，因为不同文档类型最佳策略差很多。
>
> 我们的实战分类：
>
> **FAQ 类（5w 条）**：每条 FAQ 本身就是一个块，不切，直接整体 embed。这是最简单也最准的场景，召回率天花板。
>
> **政策手册类**：用 **结构感知分块**——按 Markdown 标题切，每个 H2/H3 一块，块大小限制 400 token、超出再递归切。元数据带"章节路径 + 国家 + 币种"，召回时这些元数据可以过滤。
>
> **对账单 / 表格类**：表格不切散，整张表当一块；表前后段落各带 100 token overlap 当上下文。
>
> **PDF 政策文档**：MinerU 解析为 Markdown 后，按 Markdown 结构切。
>
> 几个踩坑：
> - **块太小**（100 token 试过）→ 一个完整规则被切散，召回不全；
> - **块太大**（1000 token 试过）→ 噪声多、embedding 信号稀释；
> - **没 overlap** → 块边界处规则丢失，比如"退款时效"被切到了上一块的尾部，下一块只有"3-5 个工作日"无主语；
> - **没元数据** → 路径不清晰，answer 引用时不知道来源章节。
>
> 最终甜点：**400 token + 50 overlap + 结构元数据**，召回率 88%。
>
> 一句话：**分块策略要按文档类型分场景**，没有 one-size-fits-all。

**🪝 顺嘴带出去的钩子**：
- "结构感知分块在多语言政策文档上有特殊处理。"
- "表格分块的特殊策略可以聊。"
- "Late Chunking 我们也试过可以分享结果。"
