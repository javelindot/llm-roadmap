# 听说过MOE吗

## 答案

MoE 是 Mixture of Experts，核心思想是为模型提供多个 Expert，并由 Router/Gate 为每个 token 选择少量专家计算。这样可以扩大总参数容量，而每个 token 的激活计算量不必与总参数同比增长。

### 1. 基本结构

在 Transformer 中通常替换 FFN：

```text
router_logits = W_r * x
selected = TopK(router_logits)
y = sum_i gate_i * Expert_i(x)
```

例如总共有 128 个专家，每个 token 只激活 8 个。总参数很多，但 active parameters 更少。

### 2. 优势

- 增大模型容量；
- 不同比例的专家可学习不同模式；
- 相同计算预算下提升效果；
- 适合多语言、多领域和复杂任务；
- 推理 FLOPs 更接近激活参数而不是总参数。

### 3. 训练难点

**负载不均衡**

Router 可能把大量 token 路由到少数 Expert，导致热点和 expert collapse。常用 load-balance loss、router z-loss、capacity factor 或动态负载策略。

**Token Dropping**

Expert 容量不足时 token 可能丢弃或转到备选专家，影响质量。

**通信**

Expert Parallel 中 token 要通过 All-to-All 发往不同 GPU，跨节点带宽和拓扑非常重要。

**训练稳定性**

Router 梯度、专家初始化、数据分布和低精度都可能导致不稳定。

### 4. 推理难点

- 所有 Expert 权重仍需存储；
- 小 batch 下专家 GEMM 不饱和；
- 路由动态导致缓存和调度复杂；
- 多机通信可能抵消 FLOPs 优势；
- 总参数、激活参数和实际延迟不能混为一谈。

### 5. Dense 与 MoE

| 维度 | Dense | Sparse MoE |
|---|---|---|
| 每 token 激活 | 全部参数 | 少量 Expert |
| 总容量 | 与计算绑定 | 可扩大 |
| 部署 | 简单 | 路由与通信复杂 |
| 小批量延迟 | 通常稳定 | 可能利用率低 |
| 专家分工 | 无显式路由 | 可形成分工 |

### 6. 与 MMoE 的区别

推荐系统 MMoE 常对每个样本计算所有密集 Expert，再由不同任务 Gate 加权；大模型 Sparse MoE 通常只激活 Top-K Expert，目的是扩展模型容量和控制计算。

### 7. 跨境支付场景

MoE 可增强多语言和多领域理解，但支付场景是否采用应看延迟、部署成本和数据规模。国家规则和资金状态仍应通过工具和规则提供，不能指望某个专家承担最终合规责任。

### 8. 面试总结

MoE 用 Router 为 token 稀疏选择专家，实现“大总参数、小激活计算”。它的核心难点不在公式，而在负载均衡、All-to-All 通信、专家容量和线上部署。
