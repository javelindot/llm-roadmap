# 结合项目实际，深度思考并阐述自主抉择Agent与相关技术模式的完整实现思路？

## 答案

**一句话**：自主抉择 Agent 的核心不是让模型“想干什么就干什么”，而是在可控边界内，让它根据任务状态自主选择下一步：检索、调用工具、追问用户、交给子 Agent、结束输出或转人工。

### 1. 什么是自主抉择 Agent

普通链式流程通常是固定的：先检索、再生成、再返回。自主抉择 Agent 更像一个带状态的执行器，它会根据当前上下文决定下一步动作。

常见动作包括：

- 判断用户意图和任务类型。
- 选择是否需要 RAG 检索。
- 决定调用哪个工具或哪个业务系统。
- 判断是否需要拆分任务给 SubAgent。
- 根据工具返回结果判断是否继续追问、重试或结束。
- 遇到高风险动作时进入人工确认或保护节点。

但工程上一定要强调：自主不是无约束，Agent 的自由度必须被流程图、工具白名单、权限、状态机和风控规则限制住。

### 2. 完整实现思路

我会把它拆成几层：

第一层是**意图识别和任务路由**。先判断用户是咨询类、查询类、操作类、投诉类还是风控解释类，不同类型进入不同流程。

第二层是**状态管理**。用一个统一的 state 保存用户问题、已检索知识、工具结果、风险标记、当前步骤、失败次数和最终答案。LangGraph 这类框架的优势就在于可以把流程状态显式保存下来。

第三层是**工具选择**。工具不能直接全部暴露给模型，而是按任务类型绑定白名单。比如客服 Agent 可以查订单和知识库，但不能直接放款；风控 Agent 可以读取风险特征，但不能绕过审批。

第四层是**决策节点**。每次工具调用后，不是马上回答，而是让决策节点判断结果是否充分、是否冲突、是否需要二次检索、是否需要人工介入。

第五层是**保护节点**。涉及资金、合规、隐私、账户状态变更时，必须做权限校验、参数校验、幂等校验、审计记录和人工确认。

### 3. 相关技术模式

- **Plan-and-Execute**：先规划步骤，再逐步执行。
- **ReAct**：边思考边调用工具，根据观察结果继续决策。
- **Router Agent**：根据任务类型路由到不同子流程或子 Agent。
- **Supervisor + SubAgent**：主 Agent 负责调度，子 Agent 负责专项任务。
- **State Machine / Graph**：把 Agent 流程显式建模，方便恢复、回放和观测。
- **Human-in-the-loop**：高风险决策必须引入人工确认。

### 4. 面试总结

我理解自主抉择 Agent 的重点是“有限自主”。模型可以负责理解、规划和选择，但资金动作、合规结论、权限边界不能完全交给模型。真正可落地的 Agent，一定是模型决策能力和工程约束能力结合起来的系统。

### 5. 结合跨境支付项目怎么说（面试口径）

> 如果放到跨境支付业务里，我会把自主抉择 Agent 做成一个“支付问题处理助手”。用户问“为什么我的付款失败了”，Agent 不会固定查一个表就结束，而是先判断问题类型：是付款失败、到账慢、风控拦截、汇率问题还是通道处理中。然后它会按状态自主决定下一步，比如先查交易单，再查通道回执，再查风控命中原因，再查合规限制，最后生成解释。
>
> 这里面我会设置几个边界：第一，客服类 Agent 只能查询和解释，不能直接改资金状态；第二，风控结论只能展示可解释原因，不能泄露完整规则；第三，如果涉及重新发起付款、解除冻结、补材料审核，就必须进入人工确认。也就是说，Agent 可以自主判断“下一步查什么”，但不能自主决定“放不放款”。
>
> 这样做的好处是客服效率会提高很多，因为 Agent 能把订单、通道、风控、合规的信息串起来，不需要人工在多个后台来回查。同时它的每一步工具调用和判断都能记录下来，出了问题可以回放。

**钩子**：可以主动强调“有限自主、工具白名单、保护节点、可回放状态”，这些点能体现你不是只会讲 Agent 概念，而是知道生产系统怎么控风险。
