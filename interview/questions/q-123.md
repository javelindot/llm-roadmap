# RAG 如何提升索引数据的质量？

## 答案

**一句话**：索引质量决定 RAG 上限——治理路径分**清洗 → 分块 → 富化 → 评测 → 持续优化** 五步，每一步都有明确动作，少做一步都掉点。

### 1. 清洗（最基础但效果最大）
- 去重（hash + 近似去重 minhash）
- 去 header/footer/页码/版权声明
- 去广告、广告链接、乱码
- 统一编码、繁简、半全角
- 修复 OCR 错字（自定义词表 / LLM 校对）
- **质量过滤**：用规则或小模型打分丢低质量内容

### 2. 分块（详见 q-121）
- 结构感知 + 递归
- 块大小适中（200-500 token）
- overlap 10-20%
- 带 path / source 元数据

### 3. 富化（核心提升点）

**3.1 添加摘要**
- 给每块加 LLM 生成的摘要
- 检索时摘要 + 内容双 embed
- 用于"短问题对长文档"场景

**3.2 添加假设问题（HyDE 反向）**
- 给每块生成"这块能回答的几个问题"
- 把问题也放入索引
- 用户问题与"假设问题"匹配度更高

**3.3 添加关键词 / 实体**
- NER 抽取关键术语
- 与文本一起入库，配合 BM25 检索

**3.4 添加元数据**
- 创建时间、章节、来源、可信度、领域标签
- 用于过滤和加权

### 4. 多 Index 策略
- 按文档类型建多个 index
- 检索时先路由再查
- 提升精准度

### 5. Embedding 选型 + 微调
- 选适合领域的 embedding 模型（bge、jina、Qwen-Embedding）
- 用业务 (query, doc) 对做对比学习微调
- 微调后召回率提升 10-20%

### 6. 评测和监控

**离线评测**
- 召回率 @ k（k=5/10/20）
- MRR、NDCG
- 端到端 RAG accuracy
- 黄金集 100-1000 条

**线上监控**
- 召回结果点击率
- "答非所问"率
- 用户反馈

### 7. 持续优化（飞轮）
- 用户反馈数据回流
- 失败 case 定位（召回失败 vs 生成失败）
- 定期重训 embedding
- 知识库定期清理过期内容

### 8. 一句话总结
**索引质量 = 清洗 + 分块 + 富化 + 评测 + 持续优化**——每一步都重要，缺一就掉点；最大收益来自"清洗"和"富化"。

## 结合项目谈（口语版 · 约 30 秒）

> 我们跨境支付 RAG 索引质量的优化是个**持续工程**，分享几个关键收益。
>
> **第一轮：基础清洗**——50w 字政策文档原样入库，召回率只有 62%。清洗页眉页脚、版权声明、重复段落后涨到 72%（**+10**）。这是最便宜的提升。
>
> **第二轮：富化摘要**——给每块加 LLM 生成的"这块讲了什么"一句话摘要，summary + content 双 embed，召回率 78%（**+6**）。
>
> **第三轮：HyDE 反向富化**——给每块生成 3-5 个"这块能回答的问题"，问题也入库。用户问题和"假设问题"语义更近，召回率 82%（**+4**）。
>
> **第四轮：元数据 + 多 Index**——按"国家 / 币种 / 业务线"分 index，用户问题先 LLM 路由再检索。准确率 86%（**+4**）。
>
> **第五轮：embedding 微调**——5w 条业务 (query, doc) 对做 InfoNCE 微调 bge-large，召回率 88%（**+2**）。
>
> **第六轮：持续监控**——上线后每周收集"答非所问"case，回查是召回失败还是生成失败，针对性补数据。三个月降到 92%（**+4**）。
>
> 一句话：**RAG 索引优化是个滚雪球**——每一步都不大，加起来从 62% 到 92% 是质变。
