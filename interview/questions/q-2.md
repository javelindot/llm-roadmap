# SubAgent 是什么？

## 答案

**一句话**：SubAgent 是主 Agent 按需创建或调用的"专职子智能体"，它在独立上下文里完成某个边界清晰的子任务，最后把结果摘要返回给主 Agent，由主 Agent 继续决策和整合。

### 1. 是什么（What）
- 主 Agent：负责理解用户目标、拆任务、调度资源、汇总结果。
- SubAgent：负责一个更聚焦的子任务，比如代码审查、检索资料、跑测试、分析日志、生成文档。
- 它不是一个普通函数调用，而是一个**独立的 Agent 实例**：通常有自己的 system prompt、上下文窗口、工具权限、模型配置和执行过程。

可以把它理解成：

> 主 Agent 是项目负责人，SubAgent 是被临时拉来处理专项工作的专家。

### 2. 为什么需要 SubAgent

#### 2.1 上下文隔离
很多任务会产生大量中间信息，比如全文搜索、日志分析、测试输出、代码扫描。如果都塞进主对话，会污染主 Agent 的上下文。

SubAgent 可以在自己的上下文里消化这些信息，只把关键结论返回给主 Agent。

#### 2.2 专业化
不同任务需要不同的角色和标准：
- code-reviewer：只关注安全、可维护性、边界条件。
- test-runner：只负责运行测试、归因失败原因。
- researcher：只负责查资料、读文档、给出处。
- planner：只做方案拆解，不直接改代码。

通过给 SubAgent 单独的 prompt 和工具权限，可以让它更像一个专项角色，而不是让主 Agent 同时扮演所有角色。

#### 2.3 并行执行
如果多个子任务相互独立，可以并行交给多个 SubAgent：
- 一个分析 API 层；
- 一个分析数据库层；
- 一个分析前端交互；
- 最后主 Agent 汇总出整体结论。

这能提升复杂任务的吞吐，但也会增加调度、合并和 token 成本。

#### 2.4 权限隔离
SubAgent 可以被限制工具权限，例如只允许 `Read/Grep/Glob`，不允许 `Write/Edit/Bash`。这样适合做只读审查、资料检索、安全扫描等任务，降低误操作风险。

### 3. 怎么工作（How）

典型流程是：

1. 用户提出复杂目标。
2. 主 Agent 判断哪些部分适合委派。
3. 主 Agent 根据 SubAgent 的描述、能力和工具权限选择合适的 SubAgent。
4. 主 Agent 生成一段任务说明，把必要背景交给 SubAgent。
5. SubAgent 在独立上下文中执行任务，可以读文件、调用工具、分析结果。
6. SubAgent 返回最终结论或摘要。
7. 主 Agent 汇总多个 SubAgent 的结果，继续规划、执行或回复用户。

关键点：SubAgent 通常**不共享主 Agent 的完整对话历史**，只看到主 Agent 传给它的任务背景和它自己的系统提示。因此，任务交接是否清楚非常重要。

### 4. 常见配置

一个 SubAgent 通常会包含这些配置：

| 配置 | 作用 |
|------|------|
| `name` | 子智能体名称，例如 `code-reviewer` |
| `description` | 说明什么时候应该使用它，主 Agent 依赖这个字段做路由 |
| `prompt` / Markdown body | SubAgent 自己的系统提示，定义角色、目标和输出格式 |
| `tools` | 允许使用哪些工具，例如只读工具、命令行工具、编辑工具 |
| `model` | 使用哪个模型，可能继承主 Agent，也可能指定更快或更强的模型 |
| `memory` | 是否拥有持久记忆，用于积累项目经验 |
| `hooks` | 生命周期钩子，用于执行前后校验、权限控制或环境准备 |

### 5. 和相关概念的区别

| 概念 | 本质 | 适合场景 |
|------|------|----------|
| Tool | 一个确定性的外部能力或函数 | 查文件、跑命令、调 API、写入数据 |
| Skill | 可复用的指令、流程和资源包 | 把团队 SOP、领域知识沉淀给当前 Agent 使用 |
| SubAgent | 独立上下文的子 Agent | 处理复杂、独立、会产生大量中间信息的子任务 |
| Multi-Agent | 多个 Agent 的协作架构 | 需要角色分工、并行分析、互相审查的大任务 |

简单说：Tool 是"手"，Skill 是"方法书"，SubAgent 是"另一个带着方法和工具去干活的人"。

### 6. 什么时候适合用

适合：
- 搜索大量代码、文档、日志，但只需要结论。
- 多个模块可以独立分析。
- 想让某个角色按固定标准审查输出。
- 需要限制权限，比如只读审查。
- 主上下文很宝贵，不想被中间过程污染。

不适合：
- 任务很小，主 Agent 直接做更快。
- 需要和用户频繁来回确认。
- 子任务强依赖主上下文里的细节，但又没有被清楚传递。
- 多个阶段高度耦合，拆开后反而增加沟通成本。

### 7. 面试总结

SubAgent 的核心价值是**分工、隔离、专业化和并行化**。它让主 Agent 从"什么都自己做"变成"负责规划和调度"，把可独立处理的部分交给专门的子 Agent。工程上要重点关注三件事：任务边界是否清晰、上下文交接是否充分、工具权限是否最小化。

### 8. 结合跨境支付项目怎么说（面试口径）

如果面试官问我 SubAgent 是什么，我不会只说概念，我会结合支付业务理解：

> 我理解 SubAgent 不是为了把系统做复杂，而是把一个大 Agent 做不了、或者不适合全都放在一个上下文里的事情拆出去，让不同子 Agent 做专项判断。比如跨境支付里，一笔交易从收款、付款、KYC/AML、制裁名单校验、风控评分、汇率计算、通道路由到回执通知，每个环节关注点都不一样。如果都让一个 Agent 处理，它既要看用户画像，又要看交易流水，还要看合规规则和通道成本，上下文会很乱，权限也会过大。
>
> 更合理的方式是：主 Agent 负责编排交易流程和最终决策；风控 SubAgent 只负责分析设备指纹、历史交易、金额、国家和频次，输出风险等级和原因；合规 SubAgent 负责检查国家、币种、名单和监管限制；通道路由 SubAgent 根据成本、成功率、时效选择最优收付款通道。最后主 Agent 汇总这些结果，决定放行、拒绝、人工审核还是补充材料。
>
> 这样做的价值有三个：第一是专业化，每个子 Agent 只做自己擅长的判断；第二是隔离，风控和合规可以只开放只读权限，避免误操作；第三是可审计，每个 SubAgent 的判断原因都可以记录下来，方便后续排查和合规追踪。所以我理解 SubAgent 的本质，是把复杂业务流程里的专家角色工程化，让主 Agent 从执行者变成调度者。

回答时可以重点抛出三个钩子：**风控专项判断、权限隔离、审计追踪**。这三个点和支付业务强相关，也能体现你不是只懂概念，而是能把 Agent 架构落到真实业务流程里。
