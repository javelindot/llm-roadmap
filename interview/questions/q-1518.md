# 业务做啥?垂领大模型构建,KG+LLM

## 答案

垂直领域大模型的业务目标不是“训练一个行业模型”，而是让领域知识、实时业务数据和操作流程可被准确、安全地使用。KG+LLM 是其中一种组合：知识图谱提供结构化事实和关系，LLM 负责理解、推理、生成与交互。

### 1. 业务需求

常见场景包括：

- 领域知识问答与检索；
- 文档、合同、票据和报告理解；
- 实体识别、关系抽取与知识维护；
- 调查分析和证据归纳；
- 流程 Agent 与工具调用；
- 推荐、风控和决策辅助。

先确定用户、流程、错误成本和 KPI，再决定是否微调、RAG 或建图。

### 2. KG+LLM 架构

```text
文档/数据库/事件流
-> 实体对齐、关系抽取、版本与置信度
-> 知识图谱 + 向量库 + 搜索索引
-> Query 理解与实体链接
-> 图查询/路径检索 + 文本检索
-> Rerank 与证据融合
-> LLM 生成/工具调用
-> 引用、校验、审计
```

LLM 可辅助图谱构建和自然语言转图查询；KG 可限制实体、补充多跳关系并提供可追溯证据。

### 3. 模型构建路线

优先级通常是：

1. Prompt + 工具调用建立 baseline；
2. RAG/GraphRAG 接入可更新知识；
3. 用 SFT/PEFT 学习领域语言、格式和流程；
4. 只有数据和收益充分时才继续预训练；
5. 可验证任务再考虑 DPO/GRPO 等对齐。

频繁变化的事实更适合检索和工具，不应全部写入参数。

### 4. 跨境支付场景

图谱可连接：

```text
商户-受益所有人-账户-设备-IP
国家-币种-支付方式-通道-牌照
交易-订单-退款-拒付-结算
规则-司法辖区-生效时间
```

应用于商户尽调、风险关系发现、通道路由解释和客服调查。LLM 可以归纳证据、生成查询和解释路径，但制裁命中、余额、订单状态与资金动作必须由权威系统和确定性规则确认。

### 5. 评测与治理

- 问答正确率、引用支持率和无答案识别；
- 图查询执行准确率、实体链接 F1、多跳召回；
- 工具成功率、流程完成率和人工接管率；
- P95、成本和可用性；
- 权限泄漏、过期规则、幻觉和错误资金动作；
- 图谱版本、来源、时间和置信度审计。

### 6. 面试总结

垂域模型负责语言和任务能力，KG 负责结构化关系与可追溯事实，实时系统负责当前状态。真正的业务价值来自三者协同，而不是把所有知识塞进模型参数。
