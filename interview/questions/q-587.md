# 如果要在 GPU 资源有限的条件下同时提供推理和微调服务，如何做资源分配和任务调度以保证时延和吞吐？

## 答案

核心原则是把在线推理当作有明确 SLO 的高优先级服务，把微调当作可抢占、可暂停的弹性任务，而不是让两类负载直接竞争 GPU。

### 资源隔离与容量规划

- 先通过压测得到推理在不同并发、输入长度和输出长度下的显存、首 token 延迟、每 token 延迟与吞吐曲线。
- 为在线服务预留满足峰值或目标分位延迟的 GPU 容量，训练只能使用剩余配额。
- 支持 MIG 的设备可做硬隔离；也可以采用独立 GPU、容器显存配额或时间片，但软隔离更容易出现算力和显存带宽干扰。
- 在线模型可使用 INT8/INT4、PagedAttention、KV Cache 管理和连续批处理降低资源占用；微调优先使用 LoRA/QLoRA、梯度检查点和低精度优化器。

### 调度策略

建立两级队列：推理请求按 SLO、租户和到达时间调度，训练任务按优先级、资源需求和截止时间调度。推理队列超过阈值或 P95/P99 延迟恶化时，调度器应：

1. 停止接纳新的训练任务。
2. 降低训练进程的 GPU 配额或并行度。
3. 在安全点保存 checkpoint 并抢占训练。
4. 必要时扩容推理副本，或对低优先级请求做限流、降级和排队。

训练在低峰期恢复，checkpoint 需同时保存模型、优化器、学习率调度器、随机数状态和数据游标，保证可继续训练。

### 提升在线吞吐

- 使用动态或连续 batching，但设置最大等待时间，避免为了凑 batch 破坏延迟 SLO。
- 按 prompt 长度分桶，减少 padding；限制最大输入、输出 token，防止长请求独占资源。
- 将 prefill 和 decode 分离调度，或者对长 prefill 做 chunked prefill，降低对短请求的阻塞。
- 设置 admission control，根据可用 KV Cache 和预测生成长度决定是否接收请求，而不是等到 OOM。

### 监控与闭环

至少监控 P50/P95/P99 首 token 延迟、token 间延迟、吞吐、队列长度、KV Cache 命中和占用、GPU 利用率、显存、功耗、训练进度及抢占恢复时间。调度决策由这些指标触发，并通过压测验证在峰值流量下仍满足 SLO。若业务对延迟要求严格，最稳妥的方案仍是推理与训练物理隔离，共享只作为资源利用率优化手段。
