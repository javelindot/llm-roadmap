# Lora sft dpo

## 答案

LoRA、SFT、DPO 不在同一个维度：

- **LoRA** 是参数高效训练方式，决定“更新哪些参数”；
- **SFT** 是监督学习阶段，决定“用标准答案学习什么行为”；
- **DPO** 是偏好优化目标，决定“在两个回答之间更偏好哪一个”。

LoRA 可以承载 SFT，也可以承载 DPO；SFT 和 DPO 也都可以全量更新参数。

### 1. LoRA

对目标权重增加低秩更新：

```text
W' = W + ΔW
ΔW = (alpha / r) * B * A
```

基础权重冻结，只训练 A/B。它降低梯度、优化器状态、通信和 checkpoint 成本，便于按业务维护多个 Adapter，但并不改变训练数据的监督形式。

### 2. SFT

SFT 使用 `(instruction, response)` 高质量示范，通过 token-level cross entropy 最大化标准回答：

```text
L_SFT = -Σ_t log pi_theta(y_t | x, y_<t)
```

它适合学习：

- 指令格式和多轮对话；
- 业务术语和表达口径；
- 工具调用 Schema；
- 拒答、澄清和转人工示范；
- 结构化抽取与固定流程。

SFT 先把模型带入目标任务分布，是 DPO 常用的初始化阶段。

### 3. DPO

DPO 使用 `(prompt, chosen, rejected)` 偏好对，在不显式训练 Reward Model、也不做在线 PPO rollout 的情况下，让策略相对 reference model 更偏向 chosen：

```text
margin =
    log pi_theta(chosen) - log pi_ref(chosen)
    - log pi_theta(rejected) + log pi_ref(rejected)

L_DPO = -log sigmoid(beta * margin)
```

它适合优化多个“都能回答但质量不同”的行为，例如更准确、更简洁、更合规，或者正确拒绝越权请求。

### 4. 常见组合流程

```text
Base model
→ LoRA-SFT
→ 固定 SFT checkpoint 作为 reference
→ LoRA-DPO
→ 合并或独立加载 Adapter
→ 离线回归与灰度
```

实现上有两种常见选择：

1. 在 SFT Adapter 基础上继续做 DPO，并保留冻结副本作为 reference；
2. 合并 SFT Adapter 后建立新的 DPO Adapter。

无论哪种方式，都要固定 base model、tokenizer、chat template 和 reference revision。Reference 搭错会使 DPO 的概率比失去意义。

### 5. 跨境支付数据示例

**SFT 样本**

- 输入：商户询问付款失败原因，并提供错误码；
- 输出：查询工具、解释错误码、给出材料与下一步；
- 目标：学会流程、格式和工具参数。

**DPO 偏好对**

- chosen：状态与工具结果一致，不泄露内部风控规则，不承诺未确认到账时间；
- rejected：回答流畅但虚构费率、混淆退汇与拒付，或建议绕过 KYC；
- 目标：在可读回答中进一步选择准确、安全、可执行的行为。

动态费率、交易状态和国家规则不应靠 SFT/DPO 记忆，而由 RAG 与业务工具提供。

### 6. 训练与验收重点

- SFT 看 token loss、任务成功率、格式和基础能力回归；
- DPO 看 pair accuracy、reward margin、人工胜率、长度与拒答偏差；
- LoRA 看 target modules、rank、alpha、训练参数是否挂载正确；
- 三者都要评估多语言、事实性、安全、工具调用和灾难性遗忘；
- DPO 不能修复 SFT 未覆盖的基础能力，也不能替代硬权限和规则。
