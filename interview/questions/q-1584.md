# 解释一下Prompt-tuning

## 答案

Prompt Tuning 是一种 PEFT 方法：冻结整个预训练模型，只训练一小段连续的“软 Prompt”向量，并将其拼到输入 Embedding 前。

### 1. 原理

普通 hard prompt 是离散 token：

```text
"请判断情感：" + input
```

Prompt Tuning 学习：

```text
P in R^(m x d_model)
```

模型输入变为：

```text
[P_1, P_2, ..., P_m, Embedding(x_1), ..., Embedding(x_n)]
```

基础模型参数全部冻结，仅通过下游 loss 更新 `P`。

### 2. 参数量

若 soft prompt 长度为 `m`、隐藏维度为 `d_model`：

```text
trainable parameters = m * d_model
```

例如 `m=100、d_model=4096`：

```text
409,600 parameters
```

远小于数十亿参数基座。若使用额外 Prompt Encoder，参数会更多。

### 3. 与其他方法区别

**Hard Prompt**

人工或搜索离散文字，不训练模型参数。

**Prompt Tuning**

只在输入层训练连续虚拟 token。

**Prefix Tuning**

通常为每层 Attention 注入可训练的 prefix K/V，容量更强，也增加每层 KV。

**P-Tuning**

常用可训练编码器生成连续 Prompt，不同版本定义有差异。

**LoRA**

训练线性层的低秩权重增量，通常比 Prompt Tuning 有更强任务适配容量，但参数更多。

### 4. 优点

- 参数和 checkpoint 极小；
- 同一基座可维护大量任务 Prompt；
- 不修改基础模型权重；
- 训练和切换成本低；
- 大模型上效果可能接近全参微调。

### 5. 局限

- 占用上下文长度；
- 小模型和复杂生成任务可能弱于 LoRA；
- 对初始化、Prompt 长度和学习率敏感；
- Soft Prompt 不可直接解释为自然语言；
- 跨基座/版本不可直接迁移；
- 推理服务必须支持虚拟 embedding；
- 不适合写入频繁变化的事实。

### 6. 适用场景

- 大量轻量分类/生成任务；
- 每个租户一个小型任务适配；
- 显存非常有限；
- 基座已很强，只需改变任务行为；
- 需要快速实验。

若工具调用、复杂领域或多任务差距较大，通常优先比较 LoRA。

### 7. 跨境支付场景

可为工单分类、固定报表或语言风格训练软 Prompt，但订单、汇率和合规规则仍通过工具/RAG。不同商户 Prompt 也要做租户隔离和版本控制。

### 8. 面试总结

Prompt Tuning 冻结模型，只训练 `m*d_model` 个软提示参数。它比 LoRA 更轻，但容量和服务兼容性有限，适合强基座上的轻量任务适配。
