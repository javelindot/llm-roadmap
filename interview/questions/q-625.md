# 有没有用rl进行策略更新？

## 答案

应先如实说明有没有做过。没做过在线 RL，可以明确说使用的是 SFT/DPO 或规则优化，再说明如果采用 RL 会如何设计，不能把离线偏好训练冒充策略更新。

### 没有实际使用的回答

> 项目中没有用在线 RL 更新策略，主要采用 SFT 加离线偏好优化，因为任务有明确示范、数据量有限，而且真实业务不允许探索。我们用工具调用成功率和人工评审做评估。若后续在沙箱中积累了可验证轨迹，才考虑 RL。

### 使用 RL 的完整闭环

1. 定义状态、动作、终止条件和允许工具；
2. 在当前策略上采样轨迹；
3. 由规则、执行结果或 Reward Model 打分；
4. 计算 advantage，使用 PPO/GRPO 等更新策略；
5. 加 KL、clip、预算和安全约束；
6. 用独立测试集检查 reward hacking 和能力退化；
7. 影子运行、灰度、监控并支持回滚。

单步偏好任务通常先考虑 DPO；数学、代码、工具参数等有可验证结果的任务更适合 RL；真实资金和高风险权限不适合开放式探索。

### 跨境支付场景

可在沙箱中优化客服 Agent 的工具选择，奖励由“选对订单、参数合法、最终问题解决、调用成本”组成。转账、退款和制裁处理必须走确定性审批，策略只能提出建议，不能在真实账户上试错。

### 面试总结

先诚实区分 SFT、DPO 和在线 RL。真正的 RL 策略更新需要当前策略采样、奖励、advantage、受约束更新及独立评测；高风险业务只应在可验证沙箱中使用。
