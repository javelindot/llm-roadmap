# LLM中温度的意义

## 答案

Temperature 控制采样前 logits 分布的平坦程度：

```text
p_i(T) = exp(z_i / T) / Σ_j exp(z_j / T)
```

其中 `z_i` 是下一个 token 的 logit，`T>0`。

### 1. 不同温度的效果

- `T < 1`：放大 logit 差异，分布更尖锐，更倾向高概率 token；
- `T = 1`：保持原始 softmax 分布；
- `T > 1`：缩小 logit 差异，分布更平坦，低概率 token 更容易被采样；
- `T → 0`：趋近 argmax/greedy，但实现中通常直接使用 greedy，避免除零。

假设三个 token 的 logits 为 `[3, 2, 1]`，降低 T 会让第一个 token 的概率更接近 1；提高 T 会让三个概率更接近。

Temperature 不会改变模型知识，也不会让错误答案自动变正确。它只是改变从现有条件分布中如何采样。

### 2. 与 Top-k、Top-p 的关系

常见顺序是：

1. 对 logits 应用 temperature；
2. 使用 Top-k 或 Top-p 截断候选；
3. 归一化后采样。

- Top-k 只保留概率最高的 k 个 token；
- Top-p 保留累计概率达到 p 的最小候选集合；
- Temperature 调整集合内外的相对概率。

同时把温度设得很高、Top-p 又放得很宽，容易产生无关或不稳定输出；把温度设得很低且 Top-p 很小，则接近确定性输出。

### 3. 不同任务的选择

- 分类、路由、工具参数、结构化抽取：greedy 或低温度；
- RAG 事实回答：通常低温度，并以证据约束和校验为主；
- 摘要和改写：低到中等温度；
- 创意文案和多候选生成：可提高温度并配合 Top-p；
- self-consistency：有意使用非零温度生成多条推理路径，再由可验证规则聚合。

具体值依赖模型和服务实现，不能把 `0.7` 当成所有模型通用最佳值。

### 4. 评测和工程注意点

- 评测必须固定 temperature、Top-p、Top-k、seed 和解码版本；
- 非零温度要重复运行，报告均值、方差或 `pass@k`；
- temperature=0 也不保证跨硬件绝对逐 bit 一致，浮点归约和并发服务可能影响并列 logits；
- 结构化输出优先使用 Schema/约束解码，不能只靠降低温度；
- 模型升级后需要重新扫参数，旧模型最佳值不一定适用。

### 5. 跨境支付场景

订单状态、金额、币种、汇率和工具参数应使用 greedy/低温度及结构化校验；客服解释可以保留少量表达自由，但事实必须来自 RAG 或业务工具。退款、付款和解冻等资金动作不因温度低就变得安全，仍需鉴权、幂等、规则和审批。
