# 命名实体识别中如何处理实体嵌套问题？

## 答案

嵌套实体是一个实体 span 内部还包含另一个实体，普通 BIO 序列标注要求每个 token 只有一个标签，因此无法同时表示重叠边界。

例如：

```text
[中国银行北京分行]ORG
       [北京]LOC
```

同一组 token 同时属于 ORG 和 LOC，单层 BIO 无法完整编码。

### 1. Span-based 方法

枚举候选 `(start, end)`，对每个 span 分类：

```text
span_repr =
    start_embedding
    + end_embedding
    + width_embedding
    + pooled_inside_tokens
```

优点：

- 天然支持嵌套和重叠；
- 可以联合预测实体类型；
- 边界表示清晰。

缺点是候选数量近似 `O(n²)`。工程上限制最大实体长度、先预测边界或剪枝候选。

### 2. Start-End / GlobalPointer

对每种实体类型预测所有 token 对 `(start, end)` 的分数。只要多个 span 得分超过阈值，就能同时输出外层和内层实体。

优点是解码直接、适合嵌套 NER；需要通过 mask 排除 `end < start`、padding 和超长 span，并按类型校准阈值。

### 3. 多层序列标注

为不同嵌套层或实体类型建立多个 BIO head：

- 第一层预测外层实体；
- 第二层预测内层实体；
- 或每个实体类型一个独立序列标签器。

实现简单，但层数不固定时难扩展，不同 head 之间还可能产生冲突。

### 4. 基于超图或区域建模

Hypergraph、biaffine、table filling 等方法可以联合表示开始、结束和实体关系，适合结构复杂的嵌套实体，但训练与解码实现更复杂。

### 5. 生成式抽取

LLM 按 JSON 输出多个实体及边界：

```json
[
  {"text": "中国银行北京分行", "type": "ORG", "start": 0, "end": 8},
  {"text": "北京", "type": "LOC", "start": 4, "end": 6}
]
```

优点是灵活、少样本迁移方便；缺点是边界偏移、漏实体、重复和格式不稳定。生产中需要 Schema 约束、字符级回对齐和规则校验，不适合仅凭生成文本判断精确 span。

### 6. 数据和评测

标注规范要明确：

- 是否保留所有嵌套层；
- 相同边界不同类型是否允许；
- discontinuous entity 如何处理；
- 字符 offset 是闭区间还是半开区间；
- tokenizer subword 如何映射回原文。

评测使用严格 span-level Precision、Recall、F1，边界和类型都正确才算命中；同时分别报告 flat、nested、最长实体和实体类型分桶。

### 7. 跨境支付示例

商户材料中可能同时出现：

```text
[SUNRISE TRADING SINGAPORE PTE. LTD.]ORG
                 [SINGAPORE]GPE
```

或者地址内部包含国家、城市和银行机构。可以使用 GlobalPointer/span model 做高精度抽取，LLM 辅助处理新类型和低频样本。最终公司名、账户名和地址还要与 OCR、表单字段和业务规则交叉验证，不能仅以 NER 输出作为 KYC/KYB 结论。
