# rag 向量化的模型

## 答案

RAG 中的向量化模型通常是双塔或 Bi-Encoder：查询和文档分别编码成向量，通过 cosine similarity、dot product 等方式进行近邻检索。它的优势是文档向量可以离线计算，在线只需编码查询并搜索向量索引；代价是查询和文档没有逐 token 深度交互，排序精度通常不如 Cross-Encoder。

### 1. 常见模型路线

- **单向量 Dense Embedding**：每个 query 或 chunk 生成一个向量，检索速度快、存储和部署相对简单。E5、BGE、GTE、Jina Embeddings 等都属于常见候选；
- **稀疏语义向量**：保留词项维度和可解释性，对产品名、错误码、国家码等精确词更友好；
- **多向量或 Late Interaction**：如 ColBERTv2，一个文档保存多个 token 级向量，细粒度匹配能力更强，但索引体积和计算成本更高；
- **多模态 Embedding**：用于检索扫描凭证、流程图、表格截图等图文混合资料，但关键支付规则仍应尽量解析为可审计的文本或结构化数据。

BGE-M3 的特点是多语言、长文本，并统一支持 dense、sparse 和 multi-vector 表示；multilingual E5 适合多语言单向量检索。它们可以作为基线候选，但模型名称或公开榜单不能替代企业内部评测。

### 2. 跨境支付场景的选型标准

选型时至少评估以下维度：

1. **语言覆盖**：中文、英文以及本地语言问题能否检索到同一份规则；
2. **领域术语**：能否区分 payout、payin、chargeback、refund、recall，以及 SWIFT、SEPA、PIX 等专有词；
3. **精确标识符**：错误码、币种、国家码和通道名不能只依赖语义向量，应与 BM25 或字段过滤结合；
4. **长文本能力**：最大长度、截断方式和长文档表示是否适配规则手册；
5. **非对称检索**：短问题和长文档的输入格式、query instruction、pooling 方法是否与模型训练方式一致；
6. **工程成本**：向量维度、索引体积、吞吐、P95 延迟、显存、量化效果和部署许可证；
7. **数据边界**：能否私有化部署，日志和请求是否满足支付数据的合规要求。

例如用户问“巴西收款人 CPF 校验失败怎么处理”，Dense Embedding 负责匹配自然语言语义，BM25 保证 `CPF` 和错误码不丢失，元数据过滤限制产品、国家与生效版本，最后由 reranker 精排。只使用一个向量模型，很难同时把口语表达和精确业务字段都处理好。

### 3. 向量化和索引要保持版本一致

生产中常见问题不是模型本身差，而是索引与查询配置不一致：

- 文档和查询使用了不同模型版本；
- 一侧做了 L2 normalization，另一侧没有；
- E5 等模型需要的 query/document 前缀被遗漏；
- chunk 超过最大长度后静默截断，关键结论位于尾部；
- 模型升级后只重建了部分向量，索引中混入不同向量空间；
- 相似度函数与模型训练目标不匹配。

因此索引元数据应记录 `embedding_model、revision、dimension、normalization、distance_metric、chunk_version`。升级模型时采用影子索引或双索引回放，验证完成后再切流，不能直接覆盖线上索引。

### 4. 用业务黄金集评测，而不是只看公开榜单

评测集应从真实跨境支付问题中分层抽样，并由业务专家标注相关 chunk。至少覆盖：

- 多语言和中英混输；
- 同义表达与缩写；
- 错误码、币种、国家和通道精确匹配；
- 否定条件、例外条款和相似但不适用的规则；
- 长尾国家、低频产品和新版本文档；
- 具有迷惑性的困难负样本。

离线比较 Recall@K、MRR、NDCG、关键证据覆盖率，同时记录索引大小、QPS 和 P95/P99。还要把候选送入同一个 reranker 和生成器做端到端评测，因为 embedding 的 Top-K 略有提升，不一定会转化为最终答案质量提升。

推荐架构通常不是“选一个最强向量模型”，而是：

> BM25/字段检索 + Dense Retrieval + 结果融合 + Cross-Encoder Rerank。

向量模型负责扩大语义召回，精确检索和业务过滤保证边界，reranker 再判断 query 与候选是否真正匹配。
