# 为什么引入 RAG?在什么场景下 RAG 比纯 SFT 更有效?

## 答案

引入 RAG 是为了让模型在回答时读取外部、可更新、可授权和可引用的事实。SFT 主要改变模型行为与表达，不适合承载频繁变化、需要删除或必须追溯来源的业务知识。

### 1. RAG 解决的问题

- 知识更新无需重新训练；
- 私域文档按权限检索；
- 回答可附证据和引用；
- 降低仅凭参数记忆产生的幻觉；
- 支持长尾知识；
- 可删除、版本化和审计；
- 结合实时工具获取状态。

### 2. SFT 更擅长什么

- 回答格式和语气；
- 领域术语表达；
- 工具调用协议；
- 决策流程和拒答边界；
- 稳定的任务模式；
- 输入到输出映射。

SFT 学的是“如何做”，不适合当作可更新数据库。

### 3. RAG 比纯 SFT 更有效的场景

**知识频繁变化**

费率、国家规则、产品文档、通道错误码。

**知识量大且长尾**

数万份 SOP、合同、FAQ、拒付指南。

**需要证据**

合规、客服和审核回答必须引用来源。

**多租户与权限**

不同商户只能访问自己的文档。

**知识需要删除**

隐私、过期规则和客户数据可从索引移除。

**标注数据少但文档多**

没有大量问答对，却已有高质量业务文档。

### 4. SFT 可能更有效的场景

- 固定分类或抽取；
- 统一报告格式；
- 特定语气和品牌风格；
- 工具参数和流程稳定；
- 极低延迟、无需外部检索；
- 文档质量差但有高质量监督数据。

### 5. 最佳实践

通常组合：

```text
SFT/Prompt
-> 学会业务行为和工具协议

RAG/Tools
-> 提供当前事实
```

例如模型通过 SFT 学会“先查订单再解释”，订单工具提供状态，RAG 提供错误码规则。

### 6. 跨境支付

实时汇率、费率、名单和订单状态不能写入权重。RAG 用于文档知识，工具用于实时事实，规则/人工用于高风险决策。

**面试总结**：知识要可更新、可引用、可授权时选 RAG；行为要稳定一致时选 SFT；复杂生产系统通常两者结合。
