# 问简单介绍一下2中的技术，dpo sft rag等

## 答案

RAG、SFT 和 DPO 解决的是不同问题：

- **RAG**：让模型在推理时获取外部、可更新、可引用的知识；
- **SFT**：用标准示范教模型任务、格式和工具行为；
- **DPO**：用偏好对进一步优化回答风格、取舍和安全边界。

### 1. RAG

基本流程：

```text
文档解析与切分
 → Embedding/BM25 索引
 → Query 理解
 → 混合召回
 → Rerank
 → 证据约束生成
 → 引用与校验
```

适合经常变化的产品规则、国家要求、错误码、SOP 和内部知识。优点是知识可更新、可追溯；缺点是召回、版本和上下文噪声会影响答案。

### 2. SFT

使用 `(input, target output)` 或多轮 messages，通过交叉熵训练模型模仿高质量答案。

适合：

- 支付术语和任务格式；
- 意图和实体抽取；
- 工具调用；
- 追问、拒答和转人工；
- 稳定的客服表达。

SFT 不适合记忆实时汇率、订单状态和快速变化规则。

### 3. DPO

使用同一 Prompt 下的 `(chosen, rejected)`，直接提高模型对 chosen 的相对概率，并通过 reference model 控制策略漂移。

适合：

- 证据完整优于无依据承诺；
- 简洁可执行优于冗长模板；
- 正确拒答优于越权回答；
- 正确工具和参数优于相似但错误的调用。

DPO 对偏好数据质量、长度和风格偏差敏感。

### 4. 三者如何组合

```text
Base Model
 → SFT：学会任务和格式
 → DPO：学会质量取舍
 → RAG/Tools：运行时提供事实
 → Guardrails：权限、规则和执行安全
```

RAG 通常是推理架构，不是训练阶段；SFT/DPO 改模型参数。三者可以共同使用，不是互相替代。

### 5. 跨境支付示例

用户问“这笔 USD 付款为什么没到账”：

- SFT 让模型知道要解析订单号、币种和意图；
- RAG 检索对应国家、产品和通道的到账规则；
- Tool 查询真实订单、银行回执和清算状态；
- DPO 让模型偏好“基于证据说明并给下一步”，而不是猜测到账时间；
- 后端权限保证只能查询当前商户；
- 退款或重发付款仍需确定性工作流和审批。

### 6. 如何判断该用哪一个

- 知识过期或缺失：RAG；
- 格式、工具和稳定行为不会：SFT；
- 两个都能回答但质量取舍不好：DPO；
- 实时状态和计算：Tool/API；
- 权限、金额和资金动作：规则、状态机与人工审批。

评测分别看检索 Recall/NDCG、SFT 任务与格式正确率、DPO 人工偏好胜率，并用端到端任务成功率、faithfulness、安全和成本决定是否上线。
