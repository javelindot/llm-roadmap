# 介绍一下KV Cache?KV Cache优缺点

## 答案

**一句话**：KV Cache 是在自回归生成时缓存历史 token 的 Key/Value，避免每生成一个新 token 都重新计算所有历史注意力。

### 核心回答
Transformer 解码时，每个新 token 都要关注前面的 token。没有 KV Cache 时，历史 token 的 K/V 会被重复计算；有了 KV Cache，只需要算新 token 的 Q/K/V，并复用历史 K/V。

优点：
- 大幅降低解码计算量。
- 提升生成速度。
- 对长文本和高并发推理很关键。

缺点：
- 占用显存，且随 batch size、层数、head 数、hidden size、上下文长度线性增长。
- 长上下文和高并发时容易成为显存瓶颈。
- 多轮对话和流式生成需要管理 cache 生命周期。

优化方式：PagedAttention、MQA/GQA、量化 KV Cache、限制上下文、前缀缓存。

### 1. KV Cache 为什么能加速

自回归生成是一个 token 一个 token 生成的。第 `t` 个 token 生成时，需要关注前面所有 token。如果每一步都重新计算历史 token 的 K/V，计算会大量重复。

KV Cache 把历史 token 的 Key 和 Value 缓存下来，下一步只计算新 token 的 Q/K/V，然后复用历史缓存。这样生成阶段从"反复算整段上下文"变成"增量计算"。

### 2. 优缺点怎么答

优点：

- 首 token 之后的解码速度明显提升。
- 长文本生成和多轮对话更高效。
- 是流式输出和高并发推理的基础优化。

缺点：

- KV Cache 本身占显存。
- 上下文越长、并发越高，占用越大。
- 多轮对话如果不清理或压缩，缓存会持续膨胀。
- prefix cache、分页管理、请求调度会让系统更复杂。

### 3. 面试总结

KV Cache 是推理加速的核心机制，但它把计算压力换成了显存压力。线上系统要同时管延迟、并发和显存。

### 4. 结合跨境支付项目怎么说（面试口径）

> 在线支付助手最容易遇到 KV Cache 问题。比如客服或风控 Agent 同时处理很多商户对话，每个请求都带交易背景、规则片段和历史上下文，生成时 KV Cache 会快速吃显存。
>
> 我的做法不是简单加卡，而是先控制输入。RAG 只召回必要证据，历史对话做摘要，超长流水不要直接塞进上下文。推理框架上可以用 vLLM 的 PagedAttention 管理 KV Cache，对于固定系统提示、通用风控规则，还可以考虑 prefix cache。
>
> 所以 KV Cache 对我来说是线上推理优化必须关注的点：它能提升速度，但如果上下文治理不好，高并发时显存会先扛不住。

**钩子**：线上推理延迟和显存，很多时候要看 KV Cache 管理。
