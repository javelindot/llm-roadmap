# 槽位是怎么应用的？怎么更新的？

## 答案

> **槽位（Slot）** 是传统对话系统里"**结构化的信息单元**"，记录用户提到的关键字段。我们跨境支付 LLM Agent 时代还在用，但形态变了——从规则模板变成"**LLM 抽取 + 状态机维护**"。
>
> **槽位的作用**：
> - 把对话信息结构化
> - 跨轮次记住关键字段
> - 触发条件性动作（所有必填槽位填满才能执行）
>
> **典型例子（客服退款流程）**：
> ```python
> class RefundIntent(BaseModel):
>     order_id: Optional[str] = None
>     refund_reason: Optional[str] = None
>     amount: Optional[float] = None
>     contact_method: Optional[str] = None
>     # 必填：order_id + refund_reason
>     # 选填：amount（全额默认）+ contact_method
> ```
>
> **怎么应用**：
>
> **1. 槽位定义**
> 每个意图对应一组槽位，用 Pydantic 或类似 schema 定义。
>
> **2. 槽位抽取**
> 用户每轮输入后，让 LLM 从对话中抽取槽位值：
> ```python
> async def extract_slots(intent: RefundIntent, user_msg: str) -> RefundIntent:
>     prompt = f"""
>     已有信息：{intent.dict()}
>     用户最新消息：{user_msg}
>     请从消息中抽取以下字段（如果有）：
>     - order_id: 订单号
>     - refund_reason: 退款原因
>     - amount: 金额
>     - contact_method: 联系方式
>     输出 JSON，没提到的字段为 null。
>     """
>     update = await llm.invoke(prompt, response_format=RefundIntent)
>     return intent.model_copy(update=update.dict(exclude_none=True))
> ```
>
> **3. 槽位检查**
> 必填槽位都填了 → 触发动作；缺槽位 → 主动询问。
>
> **4. 主动询问**
> 缺哪个问哪个：
> ```
> 系统："请提供您要退款的订单号？"
> 用户："2026 0601 ABC"
> 系统："好的，订单号已记录。请问退款原因是什么？"
> ```
>
> **怎么更新**：
>
> **1. 增量更新（最常用）**
> 用户每轮回答 → 抽取新槽位 → 合并到已有 state（不覆盖已填的）。
>
> **2. 显式修改**
> 用户："不是那个订单，是 #20260602-XYZ" → 识别到修改意图，覆盖旧值。
>
> **3. 多轮校正**
> "金额应该是 100 不是 1000" → LLM 识别校正，更新对应槽位。
>
> **4. 槽位失效**
> 时间过久（>5 分钟）或换话题 → 清空旧槽位，重新开始。
>
> **存储**：
> 状态序列化到 Redis，key 是 `conversation_id`，TTL 30 分钟。
>
> **LLM 时代相比传统 NLU 的变化**：
>
> | 维度 | 传统 NLU | LLM 时代 |
> |------|---------|---------|
> | 抽取方式 | 正则 + NER | LLM 抽取 |
> | 槽位 schema | 复杂规则 | Pydantic 自动 |
> | 多意图切换 | 状态机硬切 | LLM 识别 |
> | 槽位校正 | 难做 | LLM 自然处理 |
> | 开发成本 | 高 | 低 |
>
> **我们的工程经验**：
> - **结构化输出**：用 Pydantic + LLM JSON mode 强制 schema
> - **必填校验**：业务规则在 Agent 层校验，缺就追问
> - **槽位与意图绑定**：意图切换时携带相关槽位、丢弃无关的
> - **trace 留底**：每次槽位变更入日志，方便复盘对话
> - **超时清理**：30 分钟无更新自动清空，防止状态污染
>
> 一句话：**槽位是"对话里的结构化状态"**，LLM 抽取代替了正则，但状态管理逻辑（增量更新、必填校验、超时清理）仍是工程核心。

- "多意图切换时的槽位继承策略可以展开。"
- "槽位状态可视化和 debug 可以讲讲。"
