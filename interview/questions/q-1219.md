# 讲lora微调，如果是全量参数sft需要多长时间？

## 答案

LoRA 在冻结基础权重的前提下，为目标矩阵学习一个低秩增量：

```text
W' = W + ΔW
ΔW = (alpha / r) * B * A
```

其中 $A\in\mathbb{R}^{r\times d_{in}}$、$B\in\mathbb{R}^{d_{out}\times r}$，且 $r$ 远小于输入、输出维度。训练时只保存 Adapter 的梯度和优化器状态，因此参数更新和通信成本远低于全量 SFT；基础模型仍要参与前向与反向传播，所以激活显存并不会按同样比例消失。

### 1. LoRA 与全量 SFT 的主要差异

| 维度 | LoRA | 全量参数 SFT |
|---|---|---|
| 更新参数 | 低秩 Adapter | 全部或绝大多数权重 |
| 梯度/优化器状态 | 只针对 Adapter | 覆盖全模型 |
| 通信量 | 较低 | 多卡场景显著更高 |
| Checkpoint | 通常 MB 到 GB 级 | 接近完整模型大小 |
| 任务容量 | 适合稳定的领域与行为适配 | 适合较大分布迁移或能力重塑 |
| 回滚与多租户 | Adapter 易切换 | 版本成本高 |

全量 SFT 并不一定在相同 token 数下比 LoRA 多很多计算量，因为二者都要完成大部分前向和反向算子；真正显著增加的是全参数梯度、优化器状态、参数同步、checkpoint I/O 和更复杂的分布式切分。

### 2. 不能脱离条件直接报训练时长

全量 SFT 的时间至少取决于：

- 模型参数量和网络结构；
- 训练 token 数、epoch、序列长度与 packing 效率；
- GPU 型号、数量、互联带宽和拓扑；
- BF16/FP16/FP8 等精度；
- FSDP、ZeRO、张量并行、流水线并行配置；
- FlashAttention、梯度检查点和激活重计算；
- global batch、梯度累积、数据加载和 checkpoint 频率；
- 实际 MFU、故障重启和评测开销。

核心估算公式是：

```text
T_train ≈ (N_tokens * N_epochs) / effective_tokens_per_second
          + T_eval + T_checkpoint + T_restart
```

其中 `effective tokens/s` 必须在目标硬件、目标序列长度和最终并行配置下实测，不能引用另一台机器的峰值吞吐。

### 3. 可执行的估时方法

1. 先固定模型 revision、数据量、最大长度和 global batch。
2. 用与正式训练相同的并行、精度和 checkpoint 配置预热。
3. 跑至少数百个稳定 step，排除编译、缓存和初始数据加载。
4. 统计端到端 tokens/s、峰值显存、MFU、通信等待和 I/O。
5. 按总 token 数外推，再加入定期评测、保存和合理故障余量。
6. 对 LoRA 和全量 SFT 使用相同 token budget 比较质量、时间和成本。

经典 AdamW 混合精度全量训练中，参数、梯度、主权重和一阶/二阶动量可带来每参数十余字节的训练状态，尚未包含激活；具体数值会随优化器、精度和分片方式变化。因此 7B 模型也可能需要多卡 ZeRO/FSDP，不能只按模型权重大小估显存。

### 4. 何时选择全量 SFT

如果只是让模型掌握跨境收付款术语、客服格式、交易查询工具和合规拒答，通常先用 LoRA 建立基线，并把实时费率、国家规则和交易状态放在 RAG 或工具中。只有在以下情况下才考虑全量 SFT：

- 高质量数据规模足够大，且 LoRA 容量已经通过 rank、target module 消融证明不足；
- 任务分布与基础模型差异很大；
- 需要同时改变大量语言、代码或推理行为；
- 训练基础设施、回归集和回滚能力能够承受全模型版本。

无论选择哪种方式，资金指令、账户权限和账务状态都不能通过微调直接授权，仍由确定性系统控制。
