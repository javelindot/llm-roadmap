# 项目中 PPT 生成的完整流程是什么？断点恢复功能如何实现？

## 答案

> 跨境支付里我们做过一个**月度业务报告 PPT 自动生成 Agent**，给运营和管理层用。完整流程和断点恢复我详细说一下。
>
> **完整流程**：
>
> ```
> 1. 用户输入主题
>    "生成 2026 年 5 月跨境业务报告"
>
> 2. 大纲生成
>    Agent → LLM → 输出章节结构（标题 + 要点 list）
>    e.g. ["业务概览", "交易趋势", "风控指标", "增长机会", "下月计划"]
>
> 3. 用户确认 / 编辑大纲（HITL）
>
> 4. 逐章数据查询
>    每个章节 → SQL Agent 查数据 → 生成图表数据
>
> 5. 逐章内容生成
>    数据 + 大纲 → LLM → 章节文本 + 关键洞察
>
> 6. 图表生成
>    matplotlib / Plotly 渲染 PNG，存 MinIO
>
> 7. 模板拼装
>    用 python-pptx 加载模板，按章节填充文字 + 图片
>
> 8. 后处理
>    - 排版优化
>    - 一致性检查（术语、口径）
>    - 公司 logo / 水印
>
> 9. 输出 + 通知
>    pptx 文件 → 上传 → 飞书机器人通知用户
> ```
>
> 一份完整 PPT 大概 30 页，端到端 5-10 分钟（吃 SQL 查询和图表生成的时间）。
>
> **断点恢复怎么实现**：
>
> 核心思想——**每一步产出都持久化到 Redis + 状态机记录进度**，崩了能从最近 checkpoint 恢复。
>
> **状态定义**：
> ```python
> class PPTJob(BaseModel):
>     job_id: str  # 唯一标识
>     user_id: str
>     topic: str
>     status: Literal["pending", "outlining", "querying", "generating",
>                     "rendering", "assembling", "done", "failed"]
>     outline: Optional[list[Section]] = None
>     section_data: dict[str, dict] = {}  # 已查到的数据
>     section_text: dict[str, str] = {}   # 已生成的文本
>     section_charts: dict[str, list[str]] = {}  # 已渲染的图表路径
>     current_section: Optional[str] = None
>     created_at: datetime
>     updated_at: datetime
> ```
>
> **关键节点持久化**：
> ```python
> async def generate_section(job: PPTJob, section: Section):
>     # 检查是否已完成（断点恢复时直接跳过）
>     if section.id in job.section_text:
>         return
>
>     # 查数据
>     data = await sql_agent.query(section.query)
>     job.section_data[section.id] = data
>     await redis_save(job)  # 持久化
>
>     # 生成文本
>     text = await llm.generate(prompt(section, data))
>     job.section_text[section.id] = text
>     await redis_save(job)  # 持久化
>
>     # 渲染图表
>     charts = await render_charts(data)
>     job.section_charts[section.id] = charts
>     await redis_save(job)
> ```
>
> **恢复入口**：
> ```python
> async def resume_or_start(job_id: str):
>     job = await redis_get(job_id)
>     if not job:
>         job = create_new_job()
>     # 从 job.status 推断从哪一步继续
>     if job.status == "outlining":
>         await do_outline(job)
>     # 各 section 单独检查 section_text 是否存在，存在就跳过
>     for section in job.outline:
>         await generate_section(job, section)
>     await assemble(job)
> ```
>
> **几个工程细节**：
> - **粒度选 section 级**——每个 section 几十秒，太细持久化开销大，太粗丢失多
> - **TTL 24 小时**——超时未恢复就清理
> - **idempotent**：同一 section 重跑结果一致（temperature=0）
> - **失败重试**：每个 section 失败重试 2 次，再失败标记 partial
> - **进度通知**：每完成一个 section 通过 SSE 推用户进度
>
> **触发恢复的场景**：
> - 服务重启（K8s 滚动更新）
> - LLM 服务挂导致中间失败
> - 用户主动暂停后继续
> - 流量切换到备机
>
> 上线后失败重启率从 8% 降到 < 1%，用户能"等几分钟回来发现还在跑"。
