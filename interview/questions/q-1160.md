# 召回在整个推荐链路中的作用和必要性？

## 答案

召回的作用是把百万或亿级候选压缩到几百或几千个高潜候选，供后续粗排和精排处理。它同时决定推荐系统的计算可行性和效果上限：

> 精排只能在召回结果中排序，真正相关的 item 没有被召回，后面的模型再强也无法补救。

### 1. 为什么不能直接全量精排

精排模型通常包含用户行为序列、交叉特征和多任务网络，对每个候选都执行一次推理。若每次请求对千万 item 全量打分，计算量、特征读取和网络开销都不可接受。

召回阶段使用倒排索引、协同过滤、双塔 ANN、图检索等低成本方法，将复杂度从“逐个计算全库”降低为近似检索或稀疏匹配，再把有限候选交给更重的模型。

### 2. 召回不仅追求相关性，还要保证覆盖

推荐需求通常包含多个目标，单一路径容易形成盲区。因此会并行使用：

- 协同过滤召回长期兴趣；
- 序列或 session 召回短期兴趣；
- 双塔召回个性化语义匹配；
- 内容召回解决新 item 和稀疏行为；
- 热门、地域和趋势召回提供基本覆盖；
- 规则和运营召回满足确定性业务需求；
- 探索通道增加新品、长尾和多样性。

召回层的任务是构建“高质量且有差异的候选池”，而不是提前完成最终排序。只追求热门 item 的命中率，会导致长尾和新品没有曝光，反馈回路越来越强。

### 3. 召回质量决定级联系统上限

召回有三类典型错误：

- **漏召回**：相关 item 未进入候选，造成不可恢复的损失；
- **噪声过多**：无关候选占满预算，增加粗排压力；
- **分布偏差**：候选过度集中在热门、旧内容或单一类目，限制精排的选择空间。

因此候选数不能凭经验固定。应绘制 Recall@K 与延迟、内存、后续模型吞吐的曲线，找到业务预算下的拐点，并为不同场景设置不同候选规模。

### 4. 跨境支付中的迁移应用

同样的级联思想可以用于向商户推荐支付产品或为付款生成候选通道：

```text
全量产品/通道
  → 准入与合规硬过滤
  → 多路候选召回
  → 成本、时效、成功率和风险排序
  → 确定性校验与执行
```

候选可以来自：

- 国家、币种和收款人类型匹配；
- 商户历史成功使用的产品或通道；
- 相似商户常用的收款、付款和换汇方案；
- 实时通道可用性和银行工作时间；
- 商户主动表达的成本、时效或到账方式偏好。

这里必须先执行牌照覆盖、KYB、制裁筛查、额度和通道硬限制。召回模型只能在合规候选池中提高覆盖，不能把“不准入但相似”的产品带回后续链路。最终资金路由还要由确定性服务完成幂等、报价锁定、风险校验和审计。

### 5. 如何评估召回是否有价值

- **相关性**：Recall@K、HitRate@K、MRR、NDCG；
- **覆盖性**：item 覆盖率、长尾覆盖、新品覆盖和类目分布；
- **通道贡献**：每路召回的独占命中率、重合率和被精排采纳率；
- **系统成本**：P95/P99、超时率、QPS、索引更新延迟；
- **下游价值**：CTR、转化率、产品开通率、支付成功率、净收益和风险指标。

判断一条召回通道是否必要，应看“移除该通道后的增量损失”，而不是只看它自己的命中量。若候选几乎都被其他通道覆盖、独占候选质量低且延迟高，就应降配或下线。
