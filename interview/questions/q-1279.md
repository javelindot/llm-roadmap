# 多模态向量维度规模多大？全量推涨后提升显著的根因分析结论是什么？

## 答案

多模态向量没有统一固定维度，必须同时说明使用的编码器、投影头和索引配置。维度是模型接口的一部分，不是越大效果越好；“全量推涨后提升显著”的根因也不能直接归为向量维度，需要通过消融和流量分桶验证。

### 1. 向量维度如何回答

CLIP 类模型将图像和文本投影到同一个语义空间。常见输出可能是 512、768、1024 维，但具体值由模型配置决定。例如原始 OpenAI CLIP 的不同骨干就有不同 `embed_dim`，不能只说“CLIP 向量是 512 维”。

项目中应报告：

```text
image encoder / text encoder 版本
projection dimension
是否 L2 normalize
相似度是 cosine 还是 inner product
向量 dtype
索引类型和压缩方式
全量向量数、分片数和副本数
```

若有 `M` 条向量、维度 `D`，仅原始 FP32 向量的存储近似为：

```text
memory ≈ M * D * 4 bytes
```

还要加 ANN 图结构、倒排表、ID、元数据和副本。维度翻倍会增加存储、带宽和距离计算，但不保证召回质量提升。

### 2. 全量上线后的根因分析框架

先确认提升不是实验伪差：

- 随机分流是否稳定；
- 上线是否同时重建了索引或更新了数据；
- 流量、节日和营销活动是否变化；
- 位置和曝光偏差是否一致；
- 新旧模型的候选量、过滤规则和 rerank 是否相同；
- 是否存在缓存预热或冷启动差异。

然后做受控消融：

```text
文本单模态
图像单模态
图文融合
旧向量 + 新索引
新向量 + 旧排序
新召回 + 固定 rerank
```

分别比较 Recall@K、MRR、NDCG、覆盖率、无结果率、长尾召回、延迟和业务指标，才能把收益归因到表示、覆盖、索引还是排序。

### 3. 常见的真实收益来源

多模态全量推开后，显著提升常见于：

- 文本描述缺失或质量差时，视觉信息补齐颜色、样式、版式等属性；
- OCR、截图和票据内容可以进入语义检索；
- 冷启动和长尾对象缺少行为数据，多模态表示提升覆盖；
- 用户查询与图片语义接近，但关键词无法匹配；
- 全量重算消除了旧向量、漏向量和版本不一致；
- 新召回增加候选多样性，后续 rerank 才有机会选中正确结果。

但这些只是候选原因，最终结论必须由分桶和消融支撑。例如“提升主要来自低文本覆盖样本，而非高维向量本身”就是可验证的结论；“模型更先进所以涨了”不是根因分析。

### 4. 跨境支付场景

多模态检索可用于 KYC 材料、银行回单、发票、错误截图和商户上传凭证：

- 文本向量覆盖 OCR 与字段语义；
- 图像向量覆盖版式、印章和视觉结构；
- 元数据过滤国家、证件类型、有效期和租户；
- rerank 结合字段一致性和文档规则。

涉及身份证件、账户和交易凭证时，要做加密、脱敏、访问控制、地域合规和审计。向量检索只能辅助查找相似材料，不能绕过 KYC、AML 或资金决策规则。

### 5. 上线验收

- 离线：Recall@K、NDCG、分语言和分文档类型；
- 在线：任务完成率、人工处理时长、无结果率；
- 系统：索引构建时间、P95/P99、内存与成本；
- 风险：跨租户召回、敏感信息泄露、错误材料匹配率；
- 运维：模型、向量、索引和元数据版本可追溯。
