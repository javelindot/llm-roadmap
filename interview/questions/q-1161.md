# 多路召回融合的动态权重设计，手动还是自适应调整？

## 答案

多路召回融合通常分两个阶段演进：

> 固定配额和规则权重建立可解释基线，再逐步升级为基于场景的动态配额或可学习融合。

直接上完全自适应策略风险较高，因为各路分数不在同一尺度、曝光数据存在偏差，模型还可能为了短期指标压制新品、长尾和高价值低频通道。

### 1. 先解决各路结果不可比的问题

BM25、协同过滤、双塔和规则召回的原始分数含义不同，不能直接相加。常用处理方式包括：

- 每路固定 Top-K 配额后合并去重；
- 对分数做分位数、z-score、min-max 或基于历史分布的校准；
- 使用 Reciprocal Rank Fusion，只依赖各路名次；
- 保留 `source、source_rank、source_score、multi_hit_count` 作为粗排特征；
- 同一 item 多路命中时聚合来源，而不是重复占用候选预算。

固定配额是可靠基线，例如每路先取一部分候选，并为热门兜底、新品探索和强规则设置最低配额。它易解释、易回滚，也便于测量每路增量。

### 2. 动态权重应依赖请求上下文

不同用户和场景需要不同召回结构：

- 新用户缺少行为，应提高内容、地域热门和冷启动通道权重；
- 老用户长期兴趣稳定，可提高协同过滤和双塔权重；
- session 中刚发生搜索或连续点击，应提高短期序列召回；
- 新 item 场景应增加内容与探索配额；
- 某通道超时率升高或索引过旧，应自动降权并由其他通道补位。

动态融合器可以使用轻量 LR、GBDT、MLP 或 gating network，输入用户活跃度、场景、请求时间、各路候选数、历史质量、延迟和多路命中特征，输出：

- 各路召回预算；
- 各路融合权重；
- 或候选级统一分数。

如果使用 softmax gate，可令权重和为 1；但对运营兜底、合规规则和新品探索仍应保留显式上下限，避免模型把必要通道权重压到零。

### 3. 训练自适应融合要处理曝光偏差

线上日志只包含旧策略展示过的候选。未展示 item 没有反馈，直接用点击标签训练会让新模型复制旧权重。应结合：

- 随机流量或受控探索收集更充分的曝光；
- IPS、SNIPS 或 Doubly Robust 等反事实校正；
- 按召回来源记录 propensity 和候选生成过程；
- 使用成熟标签窗口，避免退款、取消等延迟反馈污染目标；
- 对长尾、新 item 和高价值低频目标设置约束或辅助损失。

多臂老虎机可以用于小范围在线调节配额，但应在安全边界内探索，并设置止损、最小样本量和回滚机制。高风险业务不适合让 bandit 无约束修改硬规则。

### 4. 跨境支付中的动态融合

在商户产品推荐或候选支付通道生成中，可以并行使用：

- 国家、币种、收款人类型的规则召回；
- 商户历史成功通道召回；
- 相似商户产品使用召回；
- 双塔语义召回；
- 成本、时效和实时健康度召回。

动态权重可依据商户类型、交易走廊、金额、币种、紧急程度和通道状态调整。例如商户强调“最快到账”，时效通道权重提高；强调“成本最低”，成本通道权重提高；新商户缺少历史时，提高规则和画像相似召回。

但制裁筛查、牌照覆盖、KYB、额度和通道可用性属于硬约束，只能作为前置过滤或最终校验，不能被融合模型降低权重。模型负责在“合法且可执行”的候选中排序，而不是学习绕过规则。

### 5. 工程和评测

各路召回应并行执行，设置独立超时、熔断、最低返回数和降级策略。融合服务记录每个候选的来源、权重版本和去重过程，支持离线复现。

评测包括：

- 每路 Recall@K、独占命中率、覆盖率和重合率；
- 融合后的 NDCG、候选多样性和长尾覆盖；
- 动态权重在不同人群、国家和场景中的分布；
- 移除单路后的增量损失；
- P95/P99、超时补位率和单位请求成本；
- 线上转化、支付成功、净收益及风险守护指标。

因此不是简单选择“手动”或“自适应”。固定策略负责基线、边界和兜底，自适应模型负责在边界内做个性化资源分配，两者共同工作更稳健。
