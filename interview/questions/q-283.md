# 什么情况需要多个子智能体？

## 答案

> Multi-Agent 听起来很酷，但 90% 场景**单 Agent + Subagent 已经够用**，真正需要多个独立智能体的场景不多。我列下我们跨境支付实战里的判断标准。
>
> **需要多个智能体的几种情况**：
>
> **1. 上下文长度爆炸**
> 一个任务的上下文超过模型窗口，必须拆。
> - 例：分析 100 笔关联交易 + 写报告，30k+ token 塞不下
> - 方案：开 Subagent 负责子任务，主 Agent 只看汇总
>
> **2. 角色 / 视角对抗**
> 任务需要"多视角辩论"才有质量：
> - 例：辩论式生成（A 提出方案、B 反驳、C 仲裁）
> - 例：风控审核（执行 vs 监管 vs 合规三视角校验）
>
> **3. 专业领域分工**
> 不同 Agent 用不同专长 / base 模型：
> - 例：客服总管 Agent + 风控专属 Agent + 商户专属 Agent
> - 每个 Agent 有专属 RAG 知识库、专属 SFT 模型
>
> **4. 并行加速**
> 互相独立的子任务可以并行跑：
> - 例：同时拉链路 + 查黑名单 + 查关联，最后聚合
> - 单 Agent 串行 1 分钟，多 Agent 并行 20 秒
>
> **5. 权限隔离**
> 不同 Agent 有不同工具权限：
> - 例：只读查询 Agent vs 写操作 Agent（写操作必须 HITL）
> - 例：客户能调用的 Agent vs 客服能调用的 Agent
>
> **6. 长流程上下文污染**
> 主 Agent 上下文不被中间过程污染：
> - 例：研究任务，主 Agent 关注最终结论，让 Subagent 做信息收集
> - 这正是 Claude Code 的 Subagent 设计哲学
>
> **不需要多智能体的情况**：
>
> **1. 短链路单任务** → ReAct 单 Agent 就够
> **2. 共享上下文需求强** → 多 Agent 通信成本高，单 Agent 更简洁
> **3. 实时性要求高** → 多 Agent 协调有延迟，单 Agent 快
> **4. 一致性要求强** → 多 Agent 可能给冲突结论
>
> **我们跨境支付的实际使用**：
>
> **场景一：风控调查 Agent（主 Agent + 多个 Subagent）**
> - 主 Agent：调度 + 综合判断
> - Subagent A：拉链路（独立上下文，结果摘要回主）
> - Subagent B：查黑名单（同上）
> - Subagent C：写报告
> - 优势：主 Agent 上下文不被几十笔交易污染
>
> **场景二：多视角合规审查（Multi-Agent）**
> - Agent 执行：写一个 KYB 审核结论
> - Agent 监管：从合规角度挑刺
> - Agent 仲裁：综合判断
> - 优势：多视角发现单 Agent 看不到的盲区
> - 代价：成本翻 3 倍，只在高价值场景上
>
> **没上的场景**：
> - 客服对话：单 Agent + Skill 已够，上 Multi-Agent 反而慢
> - 简单分类：单 BERT / 小 LLM
> - PPT 生成：状态机驱动 + Subagent 隔离章节，不需要"多个独立 Agent"
>
> **几个工程经验**：
>
> 1. **Multi-Agent 协调成本高**——通信、状态同步、结果合并都不便宜
>
> 2. **Subagent ≠ Multi-Agent**——Subagent 是单 Agent 的辅助工具，不是平等角色
>
> 3. **失败模式更复杂**——多 Agent 之一挂了怎么办？要设计降级
>
> 4. **可观测性挑战**——分布式 trace、跨 Agent 调用链
>
> 5. **价值要明确**——上 Multi-Agent 必须能证明"单 Agent 真做不到"
>
> 一句话：**Multi-Agent 不是技术炫耀，是"单 Agent 实在够不着"才上**——上下文爆炸、角色对抗、专业分工、并行加速、权限隔离这五个真实需求才值得上。

**🪝 顺嘴带出去的钩子**：
- "Subagent 和 Multi-Agent 的边界可以聊。"
- "Multi-Agent 失败模式和降级策略可以展开。"
- "我们没推 Multi-Agent 的原因可以讲讲。"
