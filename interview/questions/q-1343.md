# SFT数据的构造方法

## 答案

SFT 数据构造的核心不是“拼成 instruction/input/output”，而是让数据覆盖真实任务、边界和失败分支，并保证标签、对话模板、loss mask、权限和评测切分正确。少量高质量数据通常优于大量重复合成数据。

### 1. 明确目标能力

先列任务矩阵：

- 指令遵循；
- 领域问答；
- 工具选择和参数；
- 多轮对话；
- 结构化输出；
- 拒答、澄清和转人工；
- 多语言和长上下文。

每种任务定义输入、期望输出、允许工具和评分标准。

### 2. 数据来源

- 专家人工编写；
- 真实历史工单和报告脱敏；
- 规则/SOP 自动转换；
- 强模型蒸馏和 Self-Instruct；
- RAG 文档合成 QA；
- 线上 bad case 修正；
- 工具执行轨迹；
- 对抗和安全样本。

合成数据必须经规则、执行器或人工抽检，不能把教师模型幻觉批量放大。

### 3. 数据格式

多轮 Chat 常表示为：

```json
{
  "messages": [
    {"role": "system", "content": "..."},
    {"role": "user", "content": "..."},
    {"role": "assistant", "tool_calls": [...]},
    {"role": "tool", "content": "..."},
    {"role": "assistant", "content": "..."}
  ]
}
```

必须使用与目标模型匹配的 chat template、special token 和 EOS。

### 4. Loss Mask

通常：

```text
system/user/padding -> -100
assistant output    -> target token
```

工具结果一般作为上下文，不要求模型背诵；tool-call 是否监督取决于任务设计。Packing 多样本时要隔离边界。

### 5. 质量治理

- 去重和近重复；
- 纠正事实、格式和角色错位；
- 过滤泄漏隐私、恶意指令和无答案样本；
- 统一术语、币种和日期格式；
- 控制长度与难度分布；
- 保留“信息不足时追问/拒答”样本；
- 多种正确表达，避免模型只会固定模板。

### 6. 数据配比与课程

- 通用指令防遗忘；
- 领域任务增强专业性；
- hard cases 和安全样本补长尾；
- 可先短、易、强格式，再加入长、多步和复杂工具轨迹；
- 依据评测和梯度贡献调整比例，不按文件大小自然混合。

### 7. 切分与评测

- 按商户、问题簇、文档和时间切分；
- 防止同一模板改写后跨 train/test；
- 单独保存冻结回归集；
- 按国家、语言、通道和风险等级分桶；
- 评估事实、工具、格式、安全与 OOD。

### 8. 跨境支付

交易、证件和商户数据必须脱敏并最小化使用。费率、汇率、名单和状态变化快，SFT 只训练查询和解释方式，实时事实由工具/RAG 提供。冻结、退款等写操作轨迹应在沙箱生成。

**面试总结**：SFT 数据的价值取决于“覆盖真实决策边界并可验证”，不是样本条数本身。
