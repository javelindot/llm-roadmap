# 🔥 如何估算 LLaMA-7B 模型推理时的显存占用？

## 答案

推理显存不能只算模型权重，至少要拆成：

```text
总显存 ≈ 权重 + KV Cache + 激活/临时张量 + 推理框架开销
```

### 1. 权重显存

LLaMA-7B 约有 70 亿参数：

| 精度 | 理论权重显存 |
|------|--------------|
| FP32 | 7B × 4 Byte ≈ 28 GB |
| FP16 / BF16 | 7B × 2 Byte ≈ 14 GB |
| INT8 | 7B × 1 Byte ≈ 7 GB |
| INT4 | 7B × 0.5 Byte ≈ 3.5 GB |

量化还会有 Scale、Zero Point 等额外开销，所以实际值会略高。

### 2. KV Cache

通用估算公式：

```text
KV bytes =
2 × layers × batch × sequence_length
  × num_kv_heads × head_dim × bytes_per_element
```

前面的 `2` 代表 K 和 V。

以原始 LLaMA-7B 为例：

- 32 层
- 32 个 KV 头
- head_dim = 128
- FP16 = 2 Byte

单个 Token 的 KV Cache 约为：

```text
2 × 32 × 32 × 128 × 2 Byte
= 524,288 Byte
≈ 0.5 MiB
```

因此 Batch=1、上下文 2048 Token 时约需 1 GiB；Batch=4 时约需 4 GiB。GQA/MQA 因 KV 头更少，会明显降低这部分显存。

### 3. 其他开销

- 当前层激活和 Logits
- GEMM / Attention Workspace
- CUDA Context 和内存碎片
- Prefill 阶段的临时张量
- 框架预留的显存池

是否使用 FlashAttention、PagedAttention，以及最大并发和最大序列长度，都会影响峰值。

### 4. 粗略结论

FP16 的 LLaMA-7B 在 Batch=1、约 2K 上下文下，通常需要在 14 GB 权重之外再预留若干 GB，实际部署常按 **16–18 GB 或更多**估算。若要提高并发或上下文长度，KV Cache 往往成为主要增量。

**注意**：这是推理显存。训练还需要梯度、优化器状态和更多激活，显存会大很多。
