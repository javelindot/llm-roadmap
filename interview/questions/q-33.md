# 如何微调出带有思维链的 LLM 推理模型？

## 答案

**一句话**：思维链推理模型的微调三板斧——**长 CoT 数据合成、SFT 冷启动、RL 拉上限**（R1-style）；难点不在训，在数据和奖励信号。

### 1. 主流路线（按 DeepSeek-R1 / OpenAI o1 范式总结）
1. **冷启动 SFT**：用少量高质量长 CoT 样本启动，让模型先学会"分步思考"的格式
2. **大规模 RL（RLVR / GRPO）**：用可验证奖励（数学正确性、代码通过率）做 RL，让 CoT 越来越深
3. **拒绝采样 + SFT**：从 RL 模型采样高质量 CoT，回炒一轮 SFT 让格式稳定
4. **通用对齐**：再做一轮 RLHF 让 CoT 模型也能闲聊

### 2. CoT 数据来源
| 来源 | 优点 | 缺点 |
|------|------|------|
| **GPT-4 / Claude 蒸馏** | 质量高 | 贵、有版权风险 |
| **人工标注** | 业务可控 | 慢、量小 |
| **R1 蒸馏** | 长 CoT 风格 | 受原模型偏好影响 |
| **MathPile / OpenR1** | 公开、量大 | 通用性强但领域弱 |
| **Reject Sampling** | 模型自己产 | 依赖一个不弱的 base |

### 3. 关键训练技巧
- **CoT 格式标记**：用 `<think>...</think><answer>...</answer>` 显式分段，方便 loss masking 和评估
- **长上下文支持**：CoT 动辄 4k-16k token，要确保 base 支持长上下文（YaRN / RoPE scaling）
- **Loss masking**：可以只对最终 answer 算 loss，也可以全 CoT 算 loss，效果差别大
- **温度策略**：训练时低 temperature，评估时 majority vote / self-consistency

### 4. RL 阶段的奖励设计
- **可验证奖励（Verifiable）**：数学题对就 +1、代码 pass 就 +1（最稳）
- **过程奖励（PRM）**：步骤级监督，但训 PRM 成本高
- **格式奖励**：CoT 长度、结构合规性
- **业务奖励**：业务领域规则匹配

### 5. 常见坑
- **过度思考**：CoT 越来越长，简单题也写一千字
- **格式塌陷**：训着训着没有 `<think>` 了
- **能力遗忘**：CoT 训得太狠，闲聊能力下降
- **数据污染**：CoT 数据里夹了答案

### 6. 一句话总结
**长 CoT SFT + 可验证 RL 是当前主流**，但**数据质量 > 训练技巧 > base 选择**——能拿到几万条高质量长 CoT 就成功一半。

## 结合项目谈（口语版 · 约 40 秒）

> 这块我们跨境支付里实际跑过一次——**风控调查 Agent 的推理链路**。
>
> 风控调查不是简单分类，而是要"读一段交易+商户信息，推断这是不是洗钱、为什么这么判"，本质就是长推理任务。我们一开始直接 prompt 一个通用 7B 模型，它推理短而浅，往往 2-3 步就给结论，证据链不完整。
>
> 后来我们走了简化版的 R1 流程：
>
> 一是**冷启动数据**——找资深风控专家标了 800 条样本，每条都是完整推理链（看到什么字段 → 联想到什么风险 → 怎么验证 → 结论），然后 GPT-4 扩到 5000 条做 SFT 冷启动；
>
> 二是**可验证奖励 RL**——风控的好处是有金标，历史结案的洗钱案件就是金标。我们让模型推理一段，结论对就 +1、推理引用了关键字段就 +0.3，跑了几轮 GRPO；
>
> 三是 **reject sampling 回炒** SFT 稳格式。
>
> 最后这个模型在我们内部测试集上证据召回率提升 40%、误报率降了 15%。当然踩了不少坑——一开始模型过度推理，简单洗钱案件写一千字，后来加了"CoT 长度惩罚"才控制住。

- "风控的可验证奖励怎么设计的可以展开。"
