# 直接让大模型生成 SQL？不是先生成 DSL，再解析成 SQL？

## 答案

> 我们生产里**"直接 SQL"和"DSL 中间层"两条路线都试过**，最终是**简单场景直接 SQL、复杂场景过 DSL**——不是非此即彼。
>
> **直接生成 SQL 的优点**：
> - 简单：prompt → SQL，少一层抽象
> - LLM 现在 SQL 能力强（GPT-4 / Qwen 都见过海量 SQL）
> - 灵活：能写复杂的 JOIN、子查询、窗口函数
>
> **直接生成 SQL 的痛点**：
> - **SQL 注入风险**：LLM 可能生成 `DROP TABLE users` 这种危险语句
> - **schema 暴露**：要把表结构全塞 prompt 才能写对 SQL
> - **方言碎片化**：MySQL / Postgres / Hive SQL 语法差异
> - **可控性差**：LLM 偶尔写错字段、JOIN 错
> - **审计难**：每条 SQL 都得单独审，不能复用规则
> - **优化器友好性**：LLM 写的 SQL 不一定高效
>
> **DSL 中间层方案**：
>
> 定义一个**受限的领域查询语言**，LLM 生成 DSL，框架解析成 SQL：
>
> ```yaml
> # 自定义 DSL（例）
> query:
>   table: orders
>   filters:
>     - field: status
>       op: eq
>       value: "completed"
>     - field: amount
>       op: gt
>       value: 10000
>   aggregations:
>     - field: amount
>       func: sum
>   group_by: [country]
>   order_by:
>     - field: total
>       direction: desc
>   limit: 10
> ```
>
> **DSL 方案的优点**：
> - **安全**：DSL 不允许危险操作（DROP / DELETE / UPDATE 默认禁用）
> - **可控**：字段、操作符、聚合函数白名单
> - **审计**：每个 DSL 节点都可校验、可日志
> - **方言无关**：DSL → MySQL / Postgres / Hive 用不同 backend
> - **易优化**：可以在 DSL 层加缓存、限流、改写
> - **schema 注入少**：只暴露相关字段
>
> **DSL 方案的缺点**：
> - **复杂查询难表达**：DSL 越简单越安全，但牺牲灵活性
> - **维护成本**：要维护 DSL 解析器、规则
> - **LLM 不一定见过你的 DSL**：要 few-shot 教
>
> **我们的选型**：
>
> **场景一：业务运营查数据（直接 SQL）**
> - 用户：内部运营 + 数据分析师
> - 权限：read-only 账号
> - schema：暴露部分授权表
> - 防护：SQL 解析器拦截 DROP / TRUNCATE，限制单查询返回行数
> - 模型：Qwen-14B + few-shot SQL 范例
>
> **场景二：客户问业务数据（DSL）**
> - 用户：B 端商户问"我上月跨境交易总额"
> - 权限：商户只能查自己的数据
> - DSL：自定义"商户报表 DSL"，固定查询模板 + 参数
> - 模型：Qwen-7B 生成 DSL，框架解析填参数
>
> **几个工程经验**：
>
> 1. **直接 SQL 也要包"安全壳"**——SQL 解析器 + 白名单 + DRY-RUN 校验
>
> 2. **DSL 不要太死**——留 escape hatch（高级用户可写 SQL），覆盖 99% 用 DSL，1% 用 SQL
>
> 3. **schema 设计要为 LLM 友好**——字段命名清晰、注释完整、没有歧义
>
> 4. **few-shot 比 prompt 重要**——给 LLM 看 10 个业务真实 SQL 例子，效果远超描述 schema
>
> 5. **离线评测必上**——黄金集 (问题, 期望 SQL) 对，每次代码改动跑回归
>
> 一句话：**直接 SQL 和 DSL 不是非此即彼**——业务复杂度低、用户可信用 SQL + 安全壳，业务复杂度高、用户不可信用 DSL + 模板。
