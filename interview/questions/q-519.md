# 长期记忆如何存储?如果历史记录量非常大,怎么优化查询效率?

## 答案

长期记忆不应只存成一堆向量。更合理的是按数据类型分层存储，并通过多阶段检索把少量相关记忆送入上下文。

### 1. 存储设计

| 内容 | 推荐存储 |
|------|----------|
| 用户偏好、实体属性 | 关系数据库 / KV |
| 对话事件、任务历史 | 事件表或日志库 |
| 文本摘要、经验片段 | 向量数据库 |
| 实体关系 | 知识图谱 |
| 原始长文档、附件 | 对象存储 |

每条记忆应带：

- `user/tenant_id`
- 类型与主题
- 时间、来源和版本
- 重要性与置信度
- 权限和过期时间
- 指向原始记录的引用

### 2. 写入前治理

- 判断是否值得长期保存。
- 去重和合并相似记忆。
- 将长对话提炼成事实、偏好、决策和未完成事项。
- 冲突信息保留版本和时间，不直接覆盖。
- 敏感信息脱敏、加密并支持删除。

### 3. 大规模查询优化

```text
用户与权限过滤
→ 时间 / 类型 / 主题过滤
→ BM25 + ANN 向量召回
→ 图关系补充
→ Rerank
→ 时间衰减、重要性和新鲜度加权
→ 注入 Top-K
```

### 4. 工程手段

- 按租户和时间分区，缩小扫描范围。
- 使用 HNSW、IVF 等 ANN 索引。
- 热数据与冷数据分层。
- 对旧事件做周期性摘要和层级索引。
- 缓存高频用户画像与最近记忆。
- 批量生成 Embedding，异步更新索引。
- 设置 TTL，清理低价值和过期记忆。

### 5. 质量控制

不能只评估检索相似度，还要看：

- 召回的记忆是否真的帮助完成任务；
- 是否召回过时或错误信息；
- 是否发生跨用户数据泄漏；
- 长期记忆是否让模型过度依赖历史偏好。

**面试总结**：海量长期记忆的核心是“结构化过滤先缩小范围，再做语义召回和重排”，而不是对全部历史做一次向量搜索。
