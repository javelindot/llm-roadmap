# DPO 如何解决回答过长的问题，除了正则

## 答案

DPO 出现回答变长，常见原因是偏好数据把“更完整”与“更多 token”混在一起，而 sequence log-prob、评审模型和人工标注又存在长度偏差。除了增加正则项，更有效的方法是从数据、目标函数、评审和推理约束四层处理。

### 1. 先修偏好数据

- chosen/rejected 尽量长度匹配，只保留质量差异；
- 同一答案制作精简版和冗长版，明确偏好精简但完整的版本；
- 增加“正确但重复、过度解释、未按字数要求”的 hard negative；
- 按任务设置长度区间，避免所有任务共用“越详细越好”；
- 分析标注者是否系统性偏好长答案。

如果数据本身把长回答标为 chosen，再强的算法也会学习长度。

### 2. 长度归一化的偏好目标

标准 DPO 使用整段 sequence log-prob，长度会影响量级。SimPO 使用平均 token log-prob 作为隐式 reward：

```text
reward(y | x) = beta / |y| * log pi_theta(y | x)
```

并加入目标 reward margin，使 chosen 比 rejected 至少高出一定幅度。平均化可以降低序列长度对 reward 尺度的直接影响，但不能消除数据和评审中的长度偏好。

也可使用 length-controlled DPO，在目标中显式控制 chosen/rejected 长度差，并用消融比较真实质量。

### 3. 把简洁性写进 rubric

标注和 AI Judge 应单独评价：

- 是否回答所有必要要点；
- 是否存在重复和无关背景；
- 能否在更少 token 下保持事实完整；
- 是否遵循“只给结论、三点内、200 字”等指令。

评审时随机候选顺序、隐藏长度统计，并用人工集检查 judge 是否天然偏好长答案。

### 4. SFT 与任务条件

- 在 SFT 中提供不同长度控制示范；
- 使用明确的 `brief/standard/detailed` 风格标签；
- 对短回答任务增加 EOS 学习和完整的短答案；
- 将“解释深度”作为输入条件，而不是训练一个固定长度策略；
- DPO 与少量 SFT loss 混合，保持基础格式和结束行为。

### 5. 推理阶段约束

- 设置合理 `max_tokens`，但不能粗暴截断正文；
- 使用 stop sequence 和结构化模板；
- 先生成要点计划，再按预算生成；
- 输出后做重复句检测和受约束压缩；
- 长度超限时重新生成，而不是直接切掉风险提示。

### 6. 如何验收

同时绘制 win rate 与平均输出长度，使用 length-controlled win rate，并按任务检查：

- 必要事实覆盖率；
- 每个正确答案的 token 成本；
- 重复率、废话率；
- 用户追问率；
- 工具与引用正确性。

跨境支付回答不能为了变短删掉适用条件、到账不确定性或合规提示。目标是“最短的完整答案”，而不是统一压缩 token。
