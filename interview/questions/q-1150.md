# rag 文档怎么切分

## 答案

RAG 文档切分的目标不是把文本机械地切成固定长度，而是构造“能够被准确召回、单独读得懂、又足以支撑回答”的最小证据单元。生产链路通常是：

> 文档解析 → 结构识别 → 语义切分 → 上下文补全 → 元数据挂载 → 索引与版本管理 → 离线评测。

### 1. 先按文档结构切，再处理长度

应先识别标题层级、段落、列表、表格、代码块、图片说明、页码和脚注，再决定 chunk 边界。不能先转成纯文本后每 500 token 截断，否则容易把规则的适用条件、例外条款和结论拆开。

跨境支付知识库中的不同文档应采用不同策略：

- **FAQ**：一个问答作为一个原子 chunk，保留标准问法、同义问法和适用产品；
- **规则与操作手册**：按“章节 → 小节 → 规则项”切分，超长小节再递归拆分；
- **错误码文档**：一个错误码或一组语义完整的关联错误码作为一个 chunk；
- **费率和限额表**：保留表头、币种、单位和生效时间，按国家、产品或行组拆分；
- **操作流程**：前置条件、步骤顺序、异常分支和完成条件尽量放在同一单元；
- **扫描件和 PDF**：保留页码、版面区域与 OCR 置信度，低质量页面进入人工复核队列。

例如“英国本地 GBP 付款的单笔限额”不能只切出数值，还必须带上产品类型、收款人类型、币种、限额单位、适用地区和生效版本。

### 2. 使用父子块兼顾召回精度和上下文完整性

小 chunk 更容易精确召回，但可能缺少上下文；大 chunk 信息完整，却会稀释向量并增加模型输入成本。常用方案是父子索引：

- 子块用于 embedding 和召回，例如一条规则、一行错误码说明；
- 父块保存完整小节、表格或流程；
- 命中子块后，根据问题类型扩展相邻块或回取父块；
- 多个子块属于同一父块时先合并去重，避免重复占用上下文。

如果一个小节高度依赖上文，还可以在入库时为 chunk 补充标题路径和短上下文，例如：

```text
文档：全球付款产品手册
章节：巴西 > BRL 本地付款 > 退汇处理
适用范围：企业收款人，版本 2026-05
正文：……
```

Contextual Retrieval 会在每个 chunk 前增加文档级语境；Late Chunking 则先用长上下文模型编码整篇文档，再对 token 表征做池化切分。二者都用于减少独立切块导致的上下文丢失，但是否采用必须由业务评测决定，不能因为方法新就直接替换生产方案。

### 3. overlap 只解决跨边界问题

固定重叠窗口可以避免一句话或一个列表刚好被边界切断，但过大的 overlap 会带来重复召回、索引膨胀和上下文浪费。更稳妥的做法是：

- 优先通过标题、段落、句子和表格边界切分；
- 只有长段落被迫拆分时才增加小范围 overlap；
- 对流程步骤和规则条件使用结构化扩展，而不是盲目复制前后 token；
- 在召回后按 `document_id + chunk_range` 做重叠去重。

### 4. chunk 必须携带业务与治理元数据

跨境支付规则具有国家、币种、通道和时间边界，正文相似不代表规则可以互换。每个 chunk 至少应携带：

> `chunk_id、document_id、title_path、country、currency、product、channel、customer_type、effective_time、version、language、ACL、source、page`

其中 `chunk_id` 要稳定可追踪；文档更新时按内容指纹增量新增、更新和删除，而不是每次全量重建后留下旧版本。检索时先执行租户、权限、生效时间等强过滤，再做语义排序，避免把过期费率或其他商户的内部材料送入模型。

### 5. 用端到端数据选择切分参数

不存在通用最佳的 `chunk_size`。应在真实问题集上对“结构切分、递归切分、父子块、上下文切分”等方案做网格实验，至少比较：

- **检索层**：Recall@K、MRR、NDCG、关键证据覆盖率；
- **上下文层**：Context Precision、Context Recall、重复率和碎片率；
- **生成层**：答案完整性、faithfulness、引用正确率和错误承诺率；
- **系统层**：索引体积、P95 延迟、输入 token 和单次成本。

评测集要覆盖多语言问法、精确错误码、长流程、表格查询、跨章节问题、旧版本干扰和权限隔离。最终选择的不是召回分数最高但成本失控的方案，而是在正确率、时延和维护复杂度之间满足业务目标的方案。
