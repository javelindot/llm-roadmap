# 文本很长，显卡数量不多，导致训练的batch_size太小，如何解决这个问题

## 答案

长文本导致单卡只能放很小的 micro-batch，应先区分目标：

- 需要更大的有效 batch 来稳定优化；
- 还是单条长序列本身已经放不下。

两类问题的解决方式不同。

### 1. 梯度累积扩大有效 Batch

```text
global_batch
= micro_batch_per_gpu
× gradient_accumulation_steps
× data_parallel_world_size
```

连续多个 micro-batch 累积梯度后再执行 optimizer step，可以在不增加峰值激活显存的情况下扩大有效 batch。

注意：

- loss 要按累积步数正确缩放；
- scheduler 按 optimizer step 而非 micro-step 更新；
- 梯度裁剪在累积完成后执行；
- 训练 token 数比样本数更适合衡量长短不一的数据；
- 梯度累积不降低单条序列的显存。

### 2. 减少无效 Token

- 按长度分桶，减少 padding；
- 使用 packing 将多个短样本填入同一序列；
- 使用正确 attention mask，避免跨样本泄漏；
- 动态 batch：按总 token 数而不是样本数装批；
- 截断无关上下文，采用结构化切片；
- 对超长文档使用分层编码、检索后训练或滑窗。

如果任务只需要局部证据，不应把整篇文档无条件放入模型。

### 3. 降低激活显存

- BF16/FP16 混合精度；
- Activation/Gradient Checkpointing，用重计算换显存；
- FlashAttention，避免物化完整注意力矩阵；
- 选择性 checkpoint，只重算高占用模块；
- Sequence/Context Parallel，将长序列激活切到多卡；
- Activation Offload，在可接受速度损失下移到 CPU。

长上下文中激活和 Attention 通常比优化器状态更早成为瓶颈，ZeRO-1/2 不会自动切分这些激活。

### 4. 降低参数与优化器显存

- LoRA/QLoRA，只训练少量低秩参数；
- 8-bit optimizer；
- DeepSpeed ZeRO/FSDP 分片状态；
- ZeRO-Offload/FSDP CPU Offload；
- 更小模型、蒸馏或冻结部分层。

全量微调不是默认答案。领域格式、工具调用和客服风格通常先验证 LoRA。

### 5. 优化小 Batch 稳定性

- 使用 AdamW 和合理的 `beta2、eps`；
- 降低学习率并设置 warmup；
- 梯度裁剪；
- 增加累积步数；
- 多随机种子检查方差；
- 监控梯度范数、loss spike 和有效 token；
- 小数据优先减少 epoch，防止过拟合。

Adam 不能解决单条序列 OOM，也不能等价替代大有效 batch。

### 6. 选型顺序

1. 按 token 分桶和 packing；
2. BF16 + FlashAttention；
3. Gradient Checkpointing；
4. 梯度累积达到目标有效 batch；
5. LoRA/QLoRA；
6. ZeRO/FSDP 和必要的 Offload；
7. 仍不足时再做 Context Parallel 或缩短序列。

每一步都要测 `tokens/s、峰值显存、P95 step time、收敛和任务指标`。省显存方案如果让训练时间或故障率不可接受，也不一定有正收益。

### 7. 跨境支付场景

支付规则、客服会话和调查报告很长，但训练任务通常只需要与答案相关的片段。先用结构化切分和检索缩小输入，再配合 packing、checkpointing 和 LoRA，往往比直接训练 128K 全上下文更经济。敏感数据还需在进入训练前脱敏和按租户隔离。
