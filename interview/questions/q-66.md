# MHA、MQA和GQA

## 答案

**一句话**：MHA、MQA、GQA 的区别在于 Query 头很多，但 Key/Value 头是否共享，从而影响推理速度和 KV Cache 大小。

### 核心回答
- **MHA（Multi-Head Attention）**：每个 query head 都有自己的 K/V head，表达能力强，但 KV Cache 大。
- **MQA（Multi-Query Attention）**：多个 Q head 共享一组 K/V，KV Cache 小、推理快，但表达可能受影响。
- **GQA（Grouped-Query Attention）**：折中方案，多个 Q head 分组共享 K/V，兼顾效果和效率。

对比：
- 训练/表达：MHA 通常最充分。
- 推理效率：MQA 最省，GQA 折中。
- 现代 LLM：很多使用 GQA，因为更适合线上长上下文和高并发。

### 1. 为什么会从 MHA 演进到 MQA/GQA

MHA 表达能力强，但每个 attention head 都有自己的 K/V，推理时 KV Cache 很大。大模型在线服务时，长上下文和高并发会让 KV Cache 成为主要瓶颈。

MQA 让多个 Query head 共享一组 K/V，显著减少 KV Cache，但压缩太强可能影响效果。GQA 是折中，把 Query head 分组，每组共享 K/V，在效果和推理效率之间平衡。

### 2. 对线上推理的影响

- KV Cache 更小。
- 同样显存下支持更长上下文或更高并发。
- 解码速度更好。
- 对服务成本影响明显。

这也是为什么现代 LLM 很多采用 GQA。

### 3. 面试总结

MHA、MQA、GQA 的核心区别是 K/V 是否共享。回答时要把它和 KV Cache、推理成本、高并发联系起来。

### 4. 结合跨境支付项目怎么说（面试口径）

> 在线支付助手里，我会关注 GQA/MQA，因为它直接影响推理显存和并发。比如客服 Agent 同时处理大量查单、拒付、风控解释请求，每个请求都带上下文，KV Cache 是主要显存开销之一。
>
> 用 GQA 的模型通常在效果和成本之间更平衡。同样一张卡，可能可以承载更多并发，或者支持更长的业务上下文。
>
> 所以业务选型时我不会只看模型榜单，也会压测长上下文、多并发下的延迟、显存和吞吐。支付场景响应慢会影响客服体验，显存爆了会影响稳定性。

**钩子**：模型结构会影响线上成本，GQA 对高并发长上下文很关键。
