# Langchain的了解程度及工作机制

## 答案

我对 LangChain 的理解是：它提供模型、消息、工具、结构化输出、检索和 Agent 的统一接口，适合快速接入不同模型与组件；当前复杂 Agent 的持久状态、分支和恢复主要由 LangGraph 承载。生产中应把核心业务逻辑保留在自己的服务层。

### 1. 核心抽象

- Chat Model：统一不同模型调用；
- Message：system/user/assistant/tool 消息；
- Prompt Template：参数化提示词；
- Tools：带名称、描述和参数 Schema 的函数；
- Structured Output：将模型输出约束为对象；
- Retriever/Vector Store：知识检索；
- Middleware/Callbacks：重试、日志、拦截和观测；
- Agent：模型在工具结果反馈下循环决策。

### 2. 典型工作机制

```text
用户输入
-> Prompt/Messages
-> LLM 判断回答或 Tool Call
-> 框架校验并执行工具
-> Tool Result 写回消息
-> LLM 继续推理
-> 达到结束条件
```

Agent 不是框架替模型做决策，而是框架负责循环、状态、工具执行和错误处理，模型负责生成下一步动作。

### 3. LangGraph

复杂工作流通常显式建图：

```text
State
-> Node
-> Conditional Edge
-> Checkpoint
-> Interrupt/Resume
```

适合多 Agent、人工审批、长任务、失败重试和断点恢复。当前 LangChain Agent 也建立在 LangGraph 之上。

### 4. RAG 链路

```text
Loader/Parser
-> Splitter
-> Embedding
-> Vector Store/Retriever
-> Rerank
-> Prompt
-> Model
```

生产系统不应被默认 chain 限制，文档解析、权限过滤、混合检索和评测通常需要自定义。

### 5. 优势

- 快速验证；
- 模型和工具接口统一；
- 生态丰富；
- LangGraph 支持状态、持久化和人机协作；
- LangSmith 可做 tracing 与评测。

### 6. 局限

- 抽象层多，排错需看最终 Prompt 和调用链；
- 版本演进快；
- 默认组件不等于生产最佳实践；
- 高性能路径和复杂权限仍需自研；
- Agent 自由度过高会增加不可控性。

### 7. 跨境支付

我会让 LangGraph 编排“意图识别 -> 权限 -> RAG/订单工具 -> 校验 -> 人工审批”，但金额计算、幂等、限额和写操作放在后端服务。每次 run 保存 trace、证据和工具参数，满足审计。
