# 如何调用不同ai模型的api，如何进行选择

## 答案

调用多个 AI 模型时，我会先做统一 Model Gateway/Adapter，再基于任务质量、延迟、成本、安全和可用性路由，而不是让业务代码直接绑定每家 SDK。

### 1. 统一接口

定义内部请求：

```json
{
  "model_alias": "reasoning-large",
  "messages": [],
  "tools": [],
  "response_schema": {},
  "temperature": 0.2,
  "max_tokens": 1024,
  "tenant_id": "...",
  "trace_id": "..."
}
```

Adapter 负责转换为不同厂商或自建模型的 API，包括：

- chat template；
- tool calling；
- multimodal input；
- streaming；
- structured output；
- error code；
- token usage。

### 2. 调用工程

- API key 放 Secret Manager；
- HTTPS；
- timeout；
- 有限重试和指数退避；
- 幂等键；
- rate limit；
- circuit breaker；
- 流式取消；
- 日志脱敏；
- 模型和 prompt 版本；
- usage/cost 统计；
- fallback。

不要记录完整敏感 prompt，尤其是支付和身份材料。

### 3. 模型选择维度

- 任务效果；
- 指令、工具和结构化输出能力；
- 文本/图像/音频模态；
- context 与有效长文本能力；
- TTFT/TPOT；
- 单 Token/请求成本；
- 并发与限流；
- 数据驻留和隐私；
- 许可证；
- 可用区域；
- SLA 和供应商风险。

### 4. 路由策略

```text
规则和小模型
-> 高频简单任务

中型模型
-> 一般问答与抽取

强推理模型
-> 复杂规划与调查

多模态模型
-> 图片/音频/视频
```

可以先规则路由，再用轻量 classifier/gate 动态选择。高风险任务不能只按成本最低路由。

### 5. 评测与上线

建立同一版本的 golden set，比较：

- 任务成功率；
- 事实和引用；
- tool call；
- JSON 合法率；
- 安全；
- 分语言/场景；
- P95 和成本。

离线通过后 shadow/A-B，避免用公开 benchmark 直接替代业务评测。

### 6. 跨境支付场景

PII 和交易数据优先路由到满足数据驻留、权限和审计要求的模型；外部模型只接收最小化、脱敏数据。订单和账务结论由内部工具提供，模型供应商切换不能改变业务规则。

### 7. 面试总结

多模型调用应通过统一网关解耦，选择依据是业务评测、延迟、成本、安全和合规。路由采用“简单任务小模型、复杂任务强模型、特殊模态专用模型”，并具备限流、降级、审计和回归。
