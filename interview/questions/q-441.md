# 🔥 为什么现在的LLM都是Decoder only的架构？

## 答案

先纠正一个表述：并不是所有大模型都是 Decoder-only，BERT 是 Encoder-only，T5 等是 Encoder-Decoder；但当前通用生成式 LLM 大多采用 Decoder-only。

### 为什么 Decoder-only 成为主流

1. **训练目标统一且数据容易获得**
   只需要对海量文本做 Next Token Prediction，不必为不同任务设计额外目标。

2. **生成任务天然匹配**
   对话、代码、推理和 Agent 都需要自回归生成，因果 Decoder 与推理形式一致。

3. **架构简单，容易扩展**
   只有一套主干，不需要协调 Encoder、Decoder 和 Cross-Attention，Scaling 和分布式训练更直接。

4. **Prompt 可以统一任务接口**
   输入、标签、示例和输出都串成一个 Token 序列，分类、抽取、翻译、问答可以统一为文本生成。

5. **In-context Learning 能力强**
   模型可以根据前文示例和指令，在不更新参数的情况下适应新任务。

6. **推理基础设施成熟**
   KV Cache、Continuous Batching、PagedAttention、Speculative Decoding 等优化都围绕自回归 Decoder 建立。

### 代价和局限

- 因果注意力不能像 Encoder 那样天然双向编码同一段输入。
- 做纯 Embedding、分类或检索表征时，Encoder 往往更高效。
- 生成是串行的，Decode 延迟较高。
- 输入和输出混在一个序列中，某些严格 Seq2Seq 任务不如 Encoder-Decoder 结构自然。

### 什么时候其他架构更合适

- 文本分类、NER、向量表征：Encoder-only。
- 输入输出结构明确的翻译、摘要：Encoder-Decoder 仍有优势。
- 多模态系统：常用视觉/音频 Encoder + LLM Decoder 的组合。

**面试总结**：Decoder-only 胜出主要因为它用最简单统一的目标吸收海量数据，并把理解、生成和任务适配统一成“根据上下文预测下一个 Token”。
