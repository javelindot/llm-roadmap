# 项目细节中的SFT数据合成、SFT/RL训练细节和奖励函数设计

## 答案

面试讲项目细节时，最好不要只说“用了 SFT 和 RL”，而要把数据、训练、奖励、评测和失败样例讲清楚。可以按下面结构回答。

### 1. SFT 数据怎么合成

SFT 数据的目标是让模型先学会任务格式、工具调用方式和基本业务知识。数据来源可以分几类：

- **真实日志改写**：从客服工单、商户咨询、运营问答中抽样，脱敏后改写成指令数据；
- **专家标注**：由业务、合规、风控同学给标准答案和不可回答边界；
- **规则生成**：根据支付状态机、错误码、材料清单自动生成样本；
- **教师模型合成**：用强模型生成候选答案，再用规则和人工做筛选；
- **拒答/安全样本**：例如用户要求绕过 KYC、修改账务、伪造交易证明时必须拒绝。

跨境支付场景可以构造：

```text
用户问题：为什么我的 USD 出款还没到账？
输入上下文：订单状态=pending，通道回执=beneficiary_address_missing
期望输出：解释原因、列出需补材料、说明 SLA、禁止承诺具体到账时间
```

### 2. 数据清洗和切分

我会重点说明这些步骤：

- PII 脱敏：姓名、证件号、银行卡、邮箱、手机号；
- 去重：避免训练集和评测集泄漏；
- 质量过滤：删除幻觉、过时规则、无来源结论；
- 格式统一：system、user、assistant、tool result 分层；
- 难例保留：失败支付、拒付、退款、风控拦截、跨币种结算；
- 切分方式：按商户、时间或问题类型切分，而不是随机打散，防止数据泄漏。

### 3. SFT 训练细节

训练目标一般是 next-token cross entropy，但只对 assistant 目标部分计算 loss，prompt、工具返回、系统指令通常 mask 掉。

```text
loss = CE(model(input_ids), labels)
labels[prompt_tokens] = -100
```

训练时要能讲出：

- 基座模型版本和参数量；
- 全参、LoRA 还是 QLoRA；
- 学习率、batch size、epoch、warmup、weight decay；
- 最大长度和截断策略；
- 是否使用 packing；
- 验证集 loss、任务准确率、安全拒答率；
- 最终选择 checkpoint 的依据。

如果自己没有真实数值，不要编造，可以说“项目中会记录这些字段，面试时我会按实验表复盘”。

### 4. RL 训练细节

RL 阶段一般用于优化 SFT 学不到的偏好：更准确的工具调用、更稳的格式、更好的推理链路、更少幻觉。常见流程：

1. 对同一 prompt 采样多个回答；
2. 用 reward model、规则 verifier 或人工偏好打分；
3. 用 PPO、GRPO、DPO/IPO 等方法优化；
4. 用 KL 约束模型不要偏离 SFT policy 太远；
5. 在 held-out 任务、红队样本和线上灰度中验证。

跨境支付里更适合用可验证奖励，例如：

```text
reward =
  2.0 * 状态解释正确
+ 1.5 * 工具调用参数正确
+ 1.0 * 补材料指引完整
+ 0.5 * 语气清晰
- 3.0 * 编造订单状态
- 5.0 * 越权承诺放款或退款
- 5.0 * 泄露敏感信息
```

### 5. 奖励函数设计原则

- **先硬约束再软偏好**：资金、合规、隐私错误直接强惩罚；
- **结果奖励和过程奖励结合**：既看最终回答，也看是否查了正确工具；
- **可执行验证优先**：订单号、金额、币种、状态能用程序校验就不用纯 LLM judge；
- **防止长度投机**：奖励要做长度归一化，避免模型靠长答案拿高分；
- **保留人工抽检**：尤其是合规、争议处理和高金额交易。

### 6. 监控指标

- SFT loss 和验证集任务准确率；
- RL 平均 reward、KL、entropy、clip fraction；
- 工具调用准确率；
- 幻觉率和越权承诺率；
- 高风险拒答率；
- 线上客服转人工率、平均处理时长、用户满意度；
- 分国家、币种、通道的 bad case。

### 7. 面试总结

这题要答出工程细节：SFT 解决“会不会按格式完成任务”，RL 解决“回答是否更符合业务偏好和安全约束”。跨境支付项目里，奖励函数不能只奖励用户满意，还必须把资金安全、合规边界、工具真实性和隐私保护放进硬约束。
