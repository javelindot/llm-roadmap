# LLMs 使用了哪些位置编码？

## 答案

**一句话**：主流 LLM 的位置编码方案分四代：**Sinusoidal → Learned → RoPE → RoPE + 外推（YaRN/NTK/PI）**——**RoPE 是事实标准**，长上下文场景配 YaRN 等外推方案。

### 1. 各家 LLM 对比表

| 模型 | 位置编码 | 训练长度 | 外推 |
|------|----------|---------|------|
| BERT | Learned Absolute | 512 | 无 |
| GPT-2 | Learned Absolute | 1024 | 无 |
| GPT-3 | Learned Absolute | 2048 | 无 |
| T5 | Relative Bias | 512 | 一般 |
| **Llama-1** | **RoPE** | 2k | PI |
| **Llama-2** | **RoPE** | 4k | NTK / PI |
| **Llama-3/3.1** | **RoPE（base=500k）** | 8k/128k | 原生 |
| **Llama-3.2** | **RoPE** | 128k | 原生 |
| **Mistral** | **RoPE + Sliding Window** | 32k | SWA |
| **Qwen-2** | **RoPE + YaRN** | 32k/128k | YaRN |
| **Qwen-3** | **RoPE** | 128k+ | 原生 |
| **GLM-4** | RoPE | 128k | 内嵌 |
| **DeepSeek-V3** | RoPE + YaRN | 128k | YaRN |
| BLOOM | ALiBi | 2k | 自动外推 |
| MPT | ALiBi | 8k+ | 自动外推 |

### 2. 为什么 RoPE 赢了
- **数学优雅**：旋转矩阵让相对位置自然出现在 attention 里
- **实现简单**：只在 Q/K 上做旋转，不改 attention 主流程
- **可外推**：搭配 NTK/YaRN 能拉到训练长度的 16-32 倍
- **效果好**：在 LM Eval 上稳定优于其他方案

### 3. 长上下文方案速览
- **PI**：线性插值，简单但损质量
- **NTK-aware**：调 base，平滑过渡
- **YaRN**：NTK + temperature scaling，目前最优
- **LongRoPE**：搜索最佳频率，128k+
- **ALiBi**：自带外推但学术上不如 RoPE

### 4. 工程现实
- 新项目无脑选 RoPE
- 长上下文用 RoPE + YaRN 或 base 调大
- 老项目兼容性问题用 ALiBi
- 极长场景配合 Flash-Attention + Ring Attention

### 5. 一句话总结
**RoPE 现在赢者通吃**，长上下文加 YaRN/NTK，几乎所有 2024+ 主流开源 LLM 都是这套组合。

## 结合项目谈（口语版 · 约 40 秒）

> 这道题和上一题（q-61）我会拼在一起讲，但侧重点不同——这道更偏"业界都用啥、为啥都用 RoPE"。
>
> 我们跨境支付训练用的 base 主要是 **Qwen-2.5-7B/14B 和 Qwen-3**，**全部 RoPE**。一开始我们没特别注意位置编码这件事，直到要做长上下文风控调查 Agent 才意识到 RoPE 配什么外推方案对效果差别巨大。
>
> 我们做过一个评测：用 Qwen-2.5-14B 默认（4k）vs Qwen-2.5-14B + YaRN（32k）跑同一组 20k 长度的风控工单，**YaRN 版本召回率高了 18 个点**。这个数字让我对位置编码的工程价值真正服气。
>
> 还有个细节：Llama-3 默认 RoPE base 从 10000 调到 500000，这其实就是把"长上下文友好性"做进了 base 模型。Qwen-3 也是类似思路。所以**RoPE 不是一个静态选择，base 频率这个超参影响巨大**。
>
> 我的总结：**RoPE 是 2024 后所有人都会用的事实标准**，问题不在选哪个，而在"怎么把它的长度上限拉到业务需求范围内"，YaRN 和 base 调整是两个关键旋钮。
