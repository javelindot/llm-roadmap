# 什么是流水线并行？优缺点？

## 答案

**一句话**：流水线并行（PP）就是把模型按"层"切到多张卡上、像工厂流水线一样接力跑，**适合跨节点低带宽场景**——但要解决"气泡"和"负载不均"。

### 1. 基本原理
- 模型按 layer 切成 N 段（stage），每段放一张/一组卡
- 前向：stage 1 算完传给 stage 2，stage 2 算完传给 stage 3 ……
- 反向：反过来传梯度

### 2. 主要变体
| 调度 | 特点 |
|------|------|
| **GPipe**（Naive PP） | 一批数据走完整个 forward 再 backward，气泡大 |
| **1F1B**（PipeDream） | 交替 forward / backward，减小气泡 |
| **Interleaved 1F1B**（Megatron-LM） | 一张卡跑多个 chunk，气泡更小 |
| **ZB（Zero Bubble）** | 最新研究，理论上零气泡 |

### 3. 优点
- **跨节点友好**：层间只传激活，通信量小，**只需 P2P 不需要 all-reduce**
- **显存均匀切分**：每张卡只放 1/N 模型，能跑超大模型
- **带宽要求低**：不像 TP 那样吃 NVLink
- **可扩展性强**：理论上加卡就能加层

### 4. 缺点
- **气泡（Bubble）**：流水线启动和结束阶段卡是空的，利用率下降
- **负载不均**：embedding 层和 LM head 通常更大，切分要小心
- **微批数量限制**：micro-batch 太小气泡大，太大显存吃紧
- **调试复杂**：调度逻辑、checkpoint 切分、recompute 都比 DP 难

### 5. 气泡公式
气泡占比 ≈ **(N-1) / (N + M-1)**（N 是 stage 数，M 是 micro-batch 数）
要降低气泡 → 增大 M，但 M 受显存限制

### 6. 何时选 PP
- 跨节点训练（带宽差）
- 模型大到单节点放不下
- 与 TP / DP 组合做 3D 并行

### 7. 一句话总结
**PP = 跨节点的省带宽利器**，但代价是"气泡"和"调度复杂"——单节点能解决的别上 PP。

## 结合项目谈（口语版 · 约 40 秒）

> 我们跨境支付训练主线在 7B-14B 量级，**单节点 8 卡 H100 + ZeRO-2 就够用，所以 PP 没真正上过生产**。但调研和小规模验证跑过，分享几个观察。
>
> 当时我们想给风控做一个 70B 大模型做最终决策审核，单节点 8×80G 装不下，自然就考虑跨节点。三种方案我们比过：
>
> - **纯 ZeRO-3 跨节点**：跨节点 all-gather 直接把网卡打满，慢得离谱；
> - **TP=8 + DP**：单节点 TP 没问题，跨节点 DP 还行，但 70B 单节点装不下；
> - **TP=8（节点内） + PP=4（节点间）**：节点内吃 NVLink 跑 TP，节点间用 100G IB 跑 PP，激活传输量很小，气泡用 micro-batch=32 压到 10% 以内，是最稳的方案。
>
> 但最后我们没真上 70B——业务侧评估下来 14B 已经够用，70B 推理成本不划算。
>
> 我对 PP 的理解是：**它是"被迫选项"，不是"优选"**。如果模型能塞下单节点，永远先 ZeRO/TP；模型塞不下了才上 PP，因为 PP 调度复杂、调试痛苦。

**🪝 顺嘴带出去的钩子**：
- "TP + PP + DP 3D 并行的拓扑选型经验可以聊聊。"
- "气泡怎么测怎么压可以展开。"
- "为什么我们最后没上 70B 这是个 ROI 决策可以讲讲。"
