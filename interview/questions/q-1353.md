# RAG的优化方法及超参数（chunk size/overlap）如何设置？是否使用rerank技术？

## 答案

RAG 优化应从解析、分块、索引、查询、召回、重排、生成和评测逐层定位。`chunk_size` 与 `overlap` 没有通用最佳值，必须根据文档结构、答案跨度、Embedding 模型和业务评测调参。

### 1. 分块优先级

优先按结构：

- 标题/章节；
- 自然段；
- FAQ 问答对；
- 表格和列表；
- 代码/规则块；
- 页码和图片说明。

固定 token 只作为超长节点的二次切分。每个 chunk 注入标题路径、文档名、页码、版本和业务 metadata。

### 2. Chunk Size

可从多个候选做实验，例如：

```text
256 / 512 / 800 / 1024 tokens
```

不是推荐固定值，而是搜索范围。

- 太小：语义和条件被切散，召回虽精确但证据不完整；
- 太大：Embedding 被多个主题稀释，检索和上下文成本上升；
- 表格、合同条款和错误处理流程应按完整业务单元保留。

可使用 parent-child：小 chunk 召回，大 parent 提供上下文。

### 3. Overlap

Overlap 用于防止边界断裂，可从：

```text
0% / 10% / 15% / 20%
```

做消融。

- 太小：跨边界句子丢失；
- 太大：重复召回、索引膨胀和上下文浪费；
- 结构切分良好时可少 overlap；
- 表格和 FAQ 通常不应机械重叠。

### 4. 检索优化

- Query Rewrite 和实体抽取；
- BM25 + Dense 混合召回；
- 国家、币种、通道、版本和权限过滤；
- Multi-query 只用于复杂问题；
- parent-child、multi-vector；
- 去重和上下文压缩。

### 5. 是否使用 Rerank

生产 RAG 通常使用 Cross-Encoder Reranker：

```text
BM25 Top30 + Dense Top30
-> Fusion Top30~50
-> Rerank Top5~10
```

具体 K 通过 Recall/Latency 曲线决定。Reranker 能改善排序，但无法找回第一阶段完全漏掉的文档。

### 6. 评测

- Retrieval：Recall@K、MRR、NDCG；
- Generation：Correctness、Faithfulness、Citation；
- System：P95、token 和成本；
- 消融：chunk/overlap、BM25、Rerank、metadata；
- 分桶：PDF、表格、语言、问题类型和文档版本。

### 7. 跨境支付

国家限制表、费率表和拒付规则不能按固定 512 token 硬切。应按国家/通道/规则单元切分，并带生效日期。实时状态和金额使用工具，不从文档 chunk 推断。

**面试总结**：先保证业务语义完整，再用评测选择长度；Rerank 是精排层，不是召回漏失的补丁。
