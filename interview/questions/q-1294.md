# 有没有尝试修改项目中的prompt？prompt的准确度如何评估？

## 答案

修改过 Prompt，但不会用“感觉回答更顺”作为验收。Prompt 是可版本化的业务配置，每次修改都要在固定数据集上比较任务指标、安全指标、延迟和成本，再通过 shadow 或灰度验证。

### 1. 修改的 Prompt 层次

**System Prompt**

- 角色和任务边界；
- 证据优先级；
- 不知道时如何处理；
- 敏感信息和高风险动作规则；
- 输出语言与格式。

**RAG Prompt**

- 明确只能基于给定证据回答；
- 要求逐条引用；
- 文档冲突时报告冲突和生效时间；
- 将用户问题、历史对话和检索证据清晰分隔；
- 防止文档中的指令覆盖系统规则。

**Tool Prompt / Description**

- 工具适用条件和禁用条件；
- 参数类型、枚举和必填字段；
- 无数据、超时和权限不足的处理；
- 避免描述重叠导致选错工具。

**Output Schema**

- 结论、依据、下一步；
- JSON schema 或固定字段；
- 不允许输出内部规则、完整卡号和敏感阈值。

### 2. Prompt 没有一个统一“准确率”

要按任务拆指标：

**分类/抽取**

- Accuracy、Precision、Recall、F1；
- slot exact match；
- 金额、币种、日期字段准确率。

**RAG**

- Context Recall/Precision；
- answer correctness；
- faithfulness；
- citation correctness；
- 无证据时的拒答率。

**Tool Calling**

- 工具选择准确率；
- 参数 exact match/schema 通过率；
- 执行成功率；
- 是否出现多余或危险调用。

**生成回答**

- point-based rubric；
- 双盲人工评审；
- 事实性、完整性、可读性；
- 人工改写率和转人工率。

**生产指标**

- P95/P99；
- token 和工具调用次数；
- 单任务成本；
- 投诉、安全事件和越权率。

### 3. 评测集怎么设计

- 从真实任务分层采样；
- 覆盖国家、币种、产品、语言和风险等级；
- 包含正常、边界、对抗、信息不足、工具失败；
- 锁定知识库、工具 mock、模型版本和解码参数；
- 按时间切分，避免未来信息和模板泄漏；
- 高风险集单独设上线阈值，不能被平均分掩盖。

每次评测保留输入、Prompt 版本、检索证据、工具 trace、输出和评分原因，才能复现。

### 4. 修改与实验方式

```text
收集 bad case
-> 归因是 Prompt、检索、工具、模型还是数据问题
-> 提出单一改动
-> 离线全量回归
-> 人工复核关键样本
-> shadow
-> 小流量 A/B
-> 监控并决定扩大或回滚
```

一次尽量只改一个主要因素，否则无法归因。Prompt 变短可能降低成本，却也可能删掉安全边界；必须同时看主指标和 guardrail。

### 5. LLM-as-a-Judge 的使用边界

LLM Judge 适合大规模初筛，但要：

- 给清晰 rubric 和参考答案；
- 隐藏候选名称，随机交换顺序；
- 检查 position、verbosity 和 self-preference bias；
- 用人工标注集测 Judge 与专家的一致性；
- 金额、币种、工具参数等确定性字段用程序规则评分；
- 高风险样本最终由业务专家复核。

### 6. 跨境支付 Prompt 的重点

例如付款失败解释：

- 订单状态以工具为准；
- 规则以当前有效文档为准；
- 不暴露风控阈值；
- 不承诺确定到账时间；
- 不建议绕过 KYC/AML；
- 信息不足先追问订单号；
- 写操作进入审批流程。

Prompt 优化的目标是让流程更可控，而不是让语言更像人。若问题来自检索漏召回或工具数据错误，继续堆 Prompt 不会真正解决。
