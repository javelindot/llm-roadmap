# 模型选型时是否做定性定量对比ab test,客观数据验证文心性能不好了吗

## 答案

模型选型必须做定性、定量和线上实验，但不能把某次主观体验概括为“文心性能不好”。合理结论只能是：在明确的模型版本、评测日期、Prompt、工具和业务数据集下，某个模型在哪些指标上优于或弱于其他候选。

### 1. 先定义选型目标

跨境支付知识助手的能力拆成：

- 产品、费率、到账时效等知识问答；
- 订单、账户、换汇等工具调用；
- 引用和事实一致性；
- JSON/表格等结构化输出；
- 中英及小语种；
- 多轮指代和长上下文；
- 合规拒答、越权防护和敏感信息处理；
- 延迟、稳定性、吞吐和成本。

不同场景权重不同。客服解释更看重准确和可读性，自动化工作流更看重工具参数、结构化输出与可恢复性，不能只用一个总分选模型。

### 2. 离线定量评测

建立经过业务专家审核的黄金集，并按国家、币种、产品、语言、难度和风险等级分层。所有候选使用相同：

- 系统 Prompt 与上下文预算；
- 知识库快照、召回结果和工具协议；
- 解码参数与最大输出长度；
- 重试策略；
- 评测时间窗口。

指标包括：

- Exact Match、F1 或任务完成率；
- point-based rubric 和双盲人工评分；
- 引用正确率、faithfulness；
- 工具选择、参数准确率、工具执行成功率；
- JSON schema 通过率；
- 安全违规率和应拒未拒率；
- P50/P95/P99、错误率、token 与成本。

生成模型存在随机性，每题应重复运行，报告均值、方差或置信区间；评审顺序随机化、隐藏模型名，减少品牌和位置偏差。

### 3. 定性误差分析

定量总分之后，对错误按类型归因：

- 知识缺失还是检索失败；
- 已检索到证据但模型没有遵循；
- 金额、币种或日期复制错误；
- 工具选错、参数漏填；
- 小语种理解偏差；
- 过度拒答或危险放行；
- 输出格式不稳定；
- 长上下文遗忘。

这一步会决定是换模型，还是应修复检索、Prompt、工具 schema 或数据质量。若所有模型都在同一批文档上失败，根因通常不应直接归到基座模型。

### 4. Shadow、灰度与 A/B

离线通过后先做 shadow：

- 复制真实请求给候选模型；
- 不把候选输出返回用户；
- 比较质量、延迟、成本和失败模式。

再做小流量灰度或随机 A/B。实验单位应按用户或商户固定分桶，避免同一会话跨模型污染。主要指标可以是问题解决率、转人工率、工具成功率，guardrail 包括投诉、安全事件、P99 和成本。

低频高风险任务不能只看统计显著性；金额、资金状态和权限错误要设置零容忍或极低阈值，并保留人工审核。

### 5. 如何形成客观结论

报告中写清：

```text
模型名称与精确版本
评测日期和数据集版本
样本量与分桶
Prompt、RAG、工具和解码设置
总体及分场景指标
置信区间与失败案例
延迟、稳定性和成本
最终选择及适用边界
```

闭源模型可能静默升级，同名模型在不同日期也可能表现不同，因此需要版本锁定、定期回归和回滚方案。只有当文心与其他模型在同一受控实验中多次出现稳定差异，才能说它在特定业务指标上不占优；不能把结论扩展成通用能力判断。
