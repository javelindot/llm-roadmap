# 查询转换（Query Transformations）

## 答案

**一句话**：查询转换 = **在检索前对用户原始 query 做加工**，弥补"用户提问方式 ≠ 知识库表达方式"——主流五种：**重写、扩展、分解、HyDE、Step-back**，是 RAG 准确率的关键杠杆。

### 1. 为什么需要查询转换
- 用户问得短而模糊（"退款要多久"）
- 知识库写得长而详细（"跨境信用卡退款时效为 7-15 个工作日……"）
- 直接 embed 距离可能远
- 多轮对话里用户用代词（"那个怎么办"）

### 2. 五种主流方法

**1) Query Rewriting（重写）**
- 让 LLM 把口语化 query 改写成规范表达
- "退款多久" → "跨境支付退款到账时效是多少天"

**2) Query Expansion（扩展）**
- 同义词扩展、术语扩展
- "退款" → "退款 / 退回 / 退付 / refund"
- 配合 BM25 检索更有效

**3) Query Decomposition（分解）**
- 复杂问题拆成子问题
- "我跨境付款被拒为啥？" → ["跨境支付有哪些常见拒绝原因", "我的订单状态显示什么", ...]
- 每个子问题分别检索后聚合

**4) HyDE（Hypothetical Document Embeddings）**
- 让 LLM 先**假设性回答** query
- 用这个假设答案的 embedding 去检索
- 答案 vs 文档的距离 < query vs 文档的距离

**5) Step-back Prompting（退一步）**
- 把具体问题抽象成上位概念
- "我用信用卡跨境付款被卡了" → "跨境信用卡支付的常见问题"
- 召回更宏观的相关内容，再做细化

### 3. 多查询融合（RAG Fusion）
- 生成 N 个变体 query
- 每个分别检索
- 用 Reciprocal Rank Fusion (RRF) 合并结果

### 4. 多轮对话的查询改写（Conversation Rewriting）
- 把对话历史 + 当前 query 拼成"独立 query"
- "那个怎么办" → "信用卡跨境退款被卡的情况怎么办"
- 必备步骤，否则检索完全失效

### 5. 工具组合
| 工具 | 推荐场景 |
|------|---------|
| 重写 + 扩展 | 短 query 通用 |
| 分解 | 多步推理问题 |
| HyDE | 答案 vs 问题语义差距大 |
| Step-back | 长尾问题、细节难召回 |
| 多查询 + RRF | 关键场景追求高召回 |

### 6. 工程经验
- 转换 = 多调一次 LLM = 延迟 + 成本
- 简单 query 跳过转换
- 用小模型做改写（7B 够用）
- 评测转换前 vs 转换后 recall

### 7. 一句话总结
**查询转换是把"用户语言"翻译成"文档语言"的桥**——五种方法按场景选，能让 RAG 召回率提 5-15 个点。

## 结合项目谈（口语版 · 约 30 秒）

> 查询转换在我们跨境支付 RAG 里是**召回率从 78% 到 88% 的关键 10 个点**，分享几个落地。
>
> **多轮对话改写**——这是必上的，没有它客服多轮 RAG 直接崩。我们用 Qwen-7B 小模型做改写器，把"那笔款呢""怎么办呀"这种带代词的 query 拼上对话历史改写成完整独立 query。成本几乎可以忽略，召回率提升 12 个点。
>
> **HyDE**——对"用户问得短、知识库写得长"场景特别有效。比如用户问"为啥被拒"，我们让 LLM 先生成一个假设回答"跨境支付拒绝可能因为风控规则触发……"，用这个假设回答的 embedding 检索，比直接用"为啥被拒"召回准确率高 8 个点。
>
> **Step-back**——长尾问题用。一些用户问得太具体，知识库里没原文，我们做 step-back 抽象成上位概念，召回相关章节再细化推理。
>
> **多查询 + RRF**——重点场景才上，每个 query 多调一次 LLM 多检索一次成本翻倍，我们只在风控调查这种高价值场景用。
>
> 一个工程细节：**转换 prompt 要让 LLM 别"过度脑补"**——一开始 HyDE 让 LLM 写很长，反而引入噪声。后来 prompt 改成"用 50 字内简短假设回答"，效果立刻好转。
>
> 一句话：**查询转换的 ROI 是 RAG 优化里最高的之一**，但要按场景选用，不能无脑全开。
