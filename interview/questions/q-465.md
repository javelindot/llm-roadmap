# 🔥 你是怎么设计agent的记忆系统?

## 答案

我会把 Agent 记忆设计成 **分层存储 + 受控写入 + 按需检索 + 生命周期治理**，而不是把全部聊天记录无限塞回上下文。

### 1. 记忆分层

| 类型 | 内容 | 典型存储 |
|------|------|----------|
| Working Memory | 当前任务状态、最近对话、工具结果 | 上下文窗口 / 状态存储 |
| Episodic Memory | 做过什么、结果如何、失败原因 | 事件表 + 向量索引 |
| Semantic Memory | 用户偏好、业务事实、稳定知识 | 结构化库 / 知识图谱 |
| Procedural Memory | SOP、工具规则、工作流 | Prompt、Skill、配置仓库 |

### 2. 写入策略

不是每轮都写长期记忆。写入前要判断：

- 是否具有未来复用价值；
- 是否与已有记忆重复或冲突；
- 置信度和来源是什么；
- 是否包含敏感数据；
- 需要保留多久。

可以用规则 + 小模型提取结构化记忆，再做去重、合并和版本控制。

### 3. 检索策略

```text
当前目标 + 用户问题
→ 元数据过滤
→ 关键词 / 向量 / 图关系混合召回
→ 时间衰减与重要性打分
→ Rerank
→ 只注入少量最相关记忆
```

检索结果要保留来源、时间和置信度，避免旧记忆覆盖当前事实。

### 4. 状态与记忆的边界

- “订单查询做到哪一步”是任务状态，应保证强一致和可恢复。
- “用户偏好英文回复”是长期记忆，可按需召回。
- 资金、权限等关键事实不能只存向量库，应以业务数据库为准。

### 5. 治理

- TTL、遗忘和用户删除能力。
- 多租户与行级权限。
- 敏感信息脱敏和加密。
- 冲突记忆合并与审计日志。
- 评测记忆命中率、误召回率和对任务成功率的提升。

**面试总结**：Agent 记忆系统的难点不是“存下来”，而是决定什么值得存、什么时候取、取出的内容是否仍然可信。
