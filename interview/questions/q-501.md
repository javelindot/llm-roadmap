# 有没有用到类似AutoGen或LangChain的框架？为什么选这个框架？

## 答案

这类问题应从项目需求解释选型，而不是回答“哪个框架最流行”。

### 参考回答

> 我使用或调研过 LangChain/LangGraph 和 AutoGen。简单 RAG 或工具调用时，我倾向使用轻量 SDK 加少量自研代码；当任务需要显式状态、分支、重试和人工审批时，会选择图式编排；只有角色协作本身是核心需求时，才引入多 Agent 对话框架。

### 1. LangChain / LangGraph

- LangChain 提供模型、Prompt、Retriever、Tool 等大量集成和通用抽象。
- LangGraph 更适合有状态的 Agent 工作流，可显式描述节点、边、循环、检查点和人工介入。

适合：

- RAG 与工具生态集成；
- 需要可控状态机和可恢复执行；
- 团队希望快速组合已有组件。

### 2. AutoGen 类多 Agent 框架

强调 Agent 之间的消息协作、角色分工和对话式编排，适合：

- Planner、Executor、Reviewer 等角色协作；
- 多专家讨论、互审和任务分派；
- 快速验证 Multi-Agent 交互方案。

### 3. 选型标准

- 工作流是否确定，还是需要开放式协作；
- 是否需要持久状态、恢复、重试和人工审批；
- 工具与模型集成是否丰富；
- Trace、评测、并发和部署能力；
- 抽象是否容易调试，是否产生框架锁定；
- 团队维护成本和版本升级风险。

### 4. 生产实践

生产环境中，我会把核心业务状态、权限和幂等放在自己的服务层，框架只负责编排和模型交互。高风险动作不会仅依赖 Agent 对话决定。

**面试总结**：框架的价值是减少编排与集成成本，但不能替代清晰的状态模型、可观测性和业务安全边界。
