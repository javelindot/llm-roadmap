# 10. 4.19 讲讲工业界RAG具体如何优化？

## 答案

工业界优化 RAG 的原则是先定位瓶颈，再做分层实验。不能把更大的 embedding、更高的 Top-K 和更强的 LLM 同时上线，否则即使效果变化也无法归因。

> 建立基线 → bad case 分类 → 单层优化与消融 → 影子流量 → 灰度 A/B → 反馈回流。

### 1. 先做错误归因

将失败样本分为：

- Query 理解或路由错误；
- 正确知识未入库；
- 解析和分块错误；
- 召回遗漏；
- 已召回但 rerank 排错；
- 上下文噪声或冲突；
- 模型未忠实使用证据；
- 实时状态本应走工具；
- 权限或业务流程错误。

每类错误对应不同优化手段。正确证据未召回时更换大模型没有意义；证据完整但回答错误时继续扩大 Top-K 只会增加噪声。

### 2. 数据和索引优化

- 建立可信数据源、owner、版本和生效时间；
- 清理页眉页脚、重复内容和过期文档；
- 对 OCR、表格和图文混排做解析质量门禁；
- 按业务语义和结构分块；
- parent-child chunk 保留局部检索与完整上下文；
- 添加标题、章节路径、国家、币种、产品、通道和权限元数据；
- 使用稳定 chunk ID 支持增量更新、删除和回滚。

Contextual Retrieval 的思路是为 chunk 补充其在文档中的局部上下文，再建立 embedding 和 BM25 索引，适合原始片段脱离章节后含义不完整的场景。上下文由离线流程生成，也需要版本和质量评测。

### 3. Query 和路由优化

- 多轮指代消解；
- 意图分类和多意图拆分；
- 支付术语、缩写和多语言归一化；
- 提取国家、币种、通道、错误码和时间；
- 知识、实时工具、图查询和人工服务分流；
- 低置信路由允许多库召回或追问。

Query 扩展、HyDE 和 RAG-Fusion 只用于召回缺口明显的复杂问题。对错误码、产品名和法规编号等精确查询，保留原词通常更重要。

### 4. 召回和重排优化

基础方案是：

> BM25 + dense retrieval → metadata filter → RRF → cross-encoder rerank → 去重与证据覆盖。

优化重点：

- 使用领域评测集选择 embedding，而不是只看通用榜单；
- 针对 query-positive-hard negative 做对比学习；
- 统计每路召回的独占正确结果和增量延迟；
- 通过 Recall@K 曲线确定候选规模；
- rerank 结合相关性、权威性、时效和适用范围；
- 动态 Top-K：简单问题少取，复杂多跳问题按证据覆盖取。

召回不到的内容，reranker 无法补救；召回通道没有独占贡献时应删除。

### 5. 上下文构造优化

- 合并相邻且同源的 chunk；
- 删除重复证据和无关模板文本；
- 按子问题组织证据，不按相似度简单拼接；
- 对冲突文档显式标记版本和生效时间；
- 保留标题、来源、页码和 chunk ID；
- 使用 token budget 确保必要条件和例外都被覆盖。

长上下文不是 RAG 的替代品。把大量文档直接塞给模型会增加 prefill 成本，并可能降低对关键证据的注意力。

### 6. 生成与事实校验

- Prompt 明确只基于证据回答；
- 工具事实与知识规则分区；
- 每个关键 claim 绑定引用；
- 金额、币种、日期、状态和错误码用程序校验；
- 无证据时拒答，缺参数时追问；
- 高风险问题使用受控模板和人工审批。

需要时可加入 claim-evidence verifier，但验证模型本身也要评测，不能形成“一个模型无条件相信另一个模型”的链条。

### 7. 延迟和成本优化

- BM25、向量和独立工具并行；
- embedding、rerank 和 LLM 批处理；
- 高频稳定 FAQ 使用带版本与权限的缓存；
- 小模型处理分类、改写和简单回答；
- 复杂问题才启用多查询、图检索或强模型；
- 控制最大上下文、最大步骤和工具预算；
- 流式输出改善首字时间；
- 超时回退到较轻检索链路或转人工。

优化同时观察 P95/P99 和单个成功解决成本，不能只看平均延迟或单次模型价格。

### 8. 评测与持续迭代

离线指标覆盖 Recall@K、NDCG、Context Precision/Recall、faithfulness、引用正确率和任务成功率；线上观察人工改写率、转人工率、重复咨询、投诉、延迟和成本。

每次变更做：

1. 固定回归集；
2. 分国家、语言、产品和难度评测；
3. 关键模块消融；
4. 影子流量；
5. 小流量灰度；
6. 安全和业务 guardrail；
7. 自动回滚。

bad case 进入统一平台，关联 query、召回候选、最终上下文、模型输出和各组件版本。工业 RAG 的长期优势来自可诊断的反馈闭环，而不是某个单点技巧。
