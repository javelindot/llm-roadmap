# 模型回答评估为何采用奇数次评判的方式？核心考量是什么？

## 答案

> **核心原因就一个——奇数次投票永远能出多数，不会平局。** 我展开讲下我们的实际考虑。
>
> **为什么需要多次评判**：
>
> LLM-as-judge 评估开放生成质量时存在三个不稳定：
> - **temperature 引入随机性**：同一个 query + answer，judge 模型每次打分可能不同
> - **judge 模型偏置**：偏好长答案、礼貌词、结构化输出
> - **prompt 敏感性**：评判 prompt 微调结果差异大
>
> 单次评判不可信，所以要**多次评判 + 投票/平均**。
>
> **为什么必须奇数次**：
>
> 二选一（pass / fail）或者 A/B 对比这种任务，**偶数次容易平局**——5 票里 3:2 能出结果，4 票里 2:2 就难判。
>
> 所以工业里：
> - 3 次：最便宜，能出多数
> - 5 次：更稳，主流选择
> - 7 次：极致追求，成本翻倍
>
> **我们的具体做法**：
>
> **客服回复满意度评估**（每条人工标 + LLM 多judge 对比）：
> - 用 Qwen-72B 当 judge
> - 同一条 (query, response) 跑 3 次
> - temperature=0.3（不能 0，否则 3 次结果一样没意义）
> - 多数票为最终结论
>
> **风控调查报告评估**（高价值，5 次）：
> - 3 个不同 prompt + 5 次 judge
> - 多数票 + 一致性指标
> - 一致性 < 0.6 转人工
>
> **DPO 偏好对生成**（蒸馏 GPT-4 时）：
> - GPT-4 跑 3 次选 chosen
> - 不一致的样本丢弃（避免噪声）
>
> **几个工程细节**：
>
> 1. **不是所有维度都要多次**——简单的格式合规性 1 次够；语义质量、事实性这种主观维度才上多次
>
> 2. **多次 ≠ 一定要不同 prompt**——同 prompt 不同 seed 也行，最好 prompt + seed 都换
>
> 3. **平局比例是质量指标**——3 次 judge 2:1 比例高（>30%），说明 judge 不可靠，要换更强的 judge 或重设计 prompt
>
> 4. **成本控制**：3 次 = 成本 3x，关键场景才上 5 次
>
> 5. **跨 judge 一致性比单 judge 多次更稳**——3 个不同模型 judge 1 次，比 1 个模型 judge 3 次更可信
>
> **我们的最佳实践**：
> ```
> 低成本场景：1 个 judge × 3 次 → 多数票
> 中成本场景：3 个 judge × 1 次 → 多数票
> 高成本场景：3 个 judge × 3 次 = 9 次 → 多数 + 一致性
> ```
>
> 一句话：**奇数次是避免平局的硬约束**，多次是缓解 LLM-judge 噪声的工程手段——多 judge 联用比单 judge 多次更有信号。
