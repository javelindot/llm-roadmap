# Agent 类型

## 答案

**一句话**：Agent 没有标准分类，常见从**架构、推理范式、交互模式、任务场景** 四个维度切，面试时挑两三个能讲深就行。

### 1. 按架构
- **Single Agent**：一个 LLM + 一组工具，自己规划自己执行。
- **Multi-Agent**：多个 Agent 协作，常见角色组合：
  - Planner（拆任务）+ Worker（干活）+ Critic（评估）
  - 主管 + 多个执行者（Manager-Workers）
  - 辩论式（Debater A vs B + Judge）

### 2. 按推理范式
| 范式 | 特点 | 适用 |
|------|------|------|
| **ReAct** | 思考→行动→观察 循环 | 简单工具调用、信息收集 |
| **Plan-and-Execute** | 先出完整计划再执行 | 流程可预期、要求可追溯 |
| **Reflexion** | 执行后反思修正 | 容易出错、可验证结果 |
| **Tree / Graph of Thought** | 多路径探索、回溯择优 | 数学、推理、规划复杂 |
| **CodeAct** | 直接生成代码作为动作 | 灵活但风险大 |

### 3. 按交互模式
- **全自主（Autonomous）**：自己跑到结束，不打扰人——适合低风险任务。
- **人在环（Human-in-the-Loop, HITL）**：关键节点停下来等人确认——金融、医疗必备。
- **协作型（Collaborative）**：多 Agent 之间或 Agent 与人之间持续对话。

### 4. 按场景
- **代码 Agent**：Claude Code、Cursor、Devin
- **研究 Agent**：信息收集、综述生成
- **浏览器 Agent**：操作 Web UI
- **客服 Agent**：对话 + 工单 + 知识库
- **风控/合规 Agent**：规则匹配 + 调查取证
- **数据分析 Agent**：NL2SQL + 可视化

### 5. 选型经验
- 任务**可预期、要可追溯** → Plan-and-Execute
- 任务**短平快、靠尝试** → ReAct
- 任务**容易错、可验证** → Reflexion
- 任务**复杂多角色** → Multi-Agent
- 任务**高风险** → 必须 HITL，无论哪种范式

### 6. 一句话总结
**没有最好的 Agent 类型，只有最匹配场景的**。能不能讲清楚"为什么这个场景选这个范式"，比记住所有分类都重要。

---

## 结合项目谈（口语版 · 约 40 秒）

> 我们跨境支付里这几类 Agent 其实都用了，不同场景挑不同范式。
>
> 风控调查我们用 **Plan-and-Execute**，因为合规要求每一步都得能追溯——为什么要查这个账户、查到了什么、怎么结论的，监管来查得拉得出完整链路。ReAct 那种"想到哪查到哪"的风格不行，没法回溯。
>
> 客服意图分类和工单初分流我们用 **ReAct**，简单调几个工具就出结果，要的是快。这种场景上 Plan-and-Execute 反而绕。
>
> 商户准入审核我们用了 **HITL**——KYB 文件查了一圈、风险标签打完后，必须停下来等运营按钮，Agent 不能自己批。这块跨过红线就是合规事故，宁可慢不能错。
>
> 还有一个比较有意思的是对账差异定位，我们做成了 **Reflexion**——Agent 先猜一个原因（汇率口径不一致 / 时间窗错位 / 漏单），然后跑验证脚本，验证失败再换假设，几轮下来基本能定位。这种场景特别适合反思范式。
>
> 多 Agent 我们也试过，让 Planner 拆任务、几个 Worker 并行查证据、Critic 把关，但是协调成本太高，最后退回 Single Agent + Subagent 隔离上下文的方案。

**🪝 顺嘴带出去的钩子**：
- "我们 Multi-Agent 试过没推下去，可以聊聊为什么 Single Agent + Subagent 更现实。"
- "金融场景 HITL 怎么设计能聊聊，这是个产品设计问题。"
- "Reflexion 在对账定位里的落地我们有个具体案例可以展开。"
