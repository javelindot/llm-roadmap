# 做AI RAG项目时的思路是怎样的？

## 答案

**一句话**：做 RAG 项目的思路是先定义业务问题和评估标准，再围绕数据、检索、生成、评估和上线闭环逐层建设。

### 核心回答
步骤：
1. 明确业务场景：客服问答、风控规则、内部知识助手等。
2. 明确指标：准确率、召回率、解决率、引用正确率、延迟。
3. 数据治理：文档解析、清洗、权限、版本管理。
4. 切片和索引：语义切块、元数据、向量库、关键词索引。
5. 检索优化：混合检索、多路召回、rerank。
6. 生成控制：prompt、引用、拒答、防幻觉。
7. 评估迭代：测试集、bad case、人工评审。
8. 工程上线：日志、监控、权限、灰度。

### 1. 为什么不能先选框架

RAG 项目如果一上来就选 LangChain、向量库或模型，很容易变成技术堆砌。正确顺序应该是先明确业务问题和评估标准，再倒推数据和检索方案。

### 2. 核心设计点

- 数据是否可信、是否更新。
- 文档结构是否复杂。
- 检索是否需要混合检索。
- 是否有权限隔离。
- 是否必须引用来源。
- 答不出来时是否允许拒答。
- 如何收集 bad case。

### 3. 面试总结

RAG 项目是数据工程、检索工程、生成控制和评估闭环的组合，不只是接一个向量库。

### 4. 结合跨境支付项目怎么说（面试口径）

> 我的思路不是先选框架，而是先问业务到底要解决什么。跨境支付里我们优先做的是内部规则查询和异常处理助手，因为这类问题文档多、更新快、人工查资料耗时。
>
> 然后我会把通道文档、合规政策、拒付 SOP 做结构化解析，按国家、币种、通道打元数据；检索用 BM25 + 向量 + rerank；生成时强制引用来源，不确定就拒答。
>
> 最后通过真实工单 bad case 迭代。比如发现某类国家限制老是答错，就回头看是解析问题、切片问题、召回问题还是 prompt 问题，而不是直接换大模型。

**钩子**：RAG 成败不在框架，而在数据治理、检索质量和评估闭环。
