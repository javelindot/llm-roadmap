# 问数据的输入输出格式如何保证大模型输出稳定的json做了哪些工作？

## 答案

稳定 JSON 输出要分输入契约、受约束生成、服务端验证和失败恢复四层，不能只在 Prompt 中写“请输出 JSON”。

### 定义契约

使用 JSON Schema/Pydantic 定义字段类型、必填项、枚举、长度和嵌套结构。输入先做类型与权限校验，Prompt 中提供字段语义和一个最小正确示例，避免同义字段。

### 生成约束

优先使用模型 API 的 structured output/function calling 或推理框架的 grammar/constrained decoding，让解码器只生成符合 schema 的 token。设置合理 max tokens 和停止条件，不要求模型输出 Markdown 代码围栏。

### 验证与修复

服务端解析 JSON 后再次做：

- schema 与业务规则校验；
- 枚举、ID、日期和数值范围检查；
- 权限与交叉字段一致性；
- 防止未知字段和注入内容。

解析失败时可返回精确错误，让模型限次 repair；仍失败则回退模板、重试其他模型或转人工。不能用脆弱正则从任意自然语言中“抠 JSON”作为主方案。

### 评测与观测

构建包含长文本、转义字符、空值、Unicode、缺字段和恶意输入的测试集，统计 parse rate、schema pass、业务正确率和重试率。记录模型、schema、Prompt 版本和原始输出。

JSON 语法正确不代表语义正确。涉及 SQL、工具或支付时，结构化参数还必须经过执行前验证和人工确认。
