# LoRA微调的框架用的什么？微调的时候是否遇到什么问题？比如收敛问题，指标不符合预期的问题？是怎么解决的？

## 答案

LoRA 我通常使用 Hugging Face `Transformers + PEFT + TRL`，工程化训练可用 LLaMA-Factory、ms-swift 或 Axolotl，分布式底层结合 Accelerate、DeepSpeed/FSDP。框架不是重点，重点是能解释数据、target modules、超参数和问题定位。

### 1. 典型配置

- 基座：选择与语言、上下文、许可证匹配的 Instruct/Base；
- target modules：常从 `q_proj/v_proj` 起步，再按任务尝试 Q/K/V/O 和 MLP；
- rank：`8/16/32/64` 做网格或逐步扩大；
- alpha：常与 rank 同量级或倍数关系；
- dropout：小数据时可加；
- optimizer：AdamW；
- 精度：BF16 优先，显存不足用 QLoRA 4-bit；
- 只对 assistant answer 计算 loss。

### 2. Loss 不下降

排查顺序：

1. 标签是否全被 mask 成 `-100`；
2. chat template、EOS 和 padding 是否正确；
3. LoRA 参数是否 `requires_grad=true`；
4. target module 名称是否匹配真实模型；
5. 学习率是否过小；
6. 量化层和 dtype 是否兼容；
7. 数据是否为空、截断或答案字段拼错。

先用几十条样本做 overfit test。小数据都拟合不了，通常是管线错误。

### 3. Loss 下降但指标不提升

可能原因：

- 训练数据与评测分布不一致；
- loss 被大量简单 token 主导；
- 模型学会格式但没学会事实；
- 标签噪声、重复和相互冲突；
- rank 太小或 target modules 太少；
- 过拟合训练措辞；
- 推理模板与训练模板不一致。

解决方式包括清洗去重、增加 hard cases、按任务加权、扩大 rank/模块、早停，以及把业务事实转为 RAG 或工具查询。

### 4. 训练不稳定

- 降低学习率；
- 增加 warmup；
- gradient clipping；
- 检查 BF16/FP16 和 loss scaling；
- 减少异常长样本；
- 使用 packing 时保证样本边界；
- 监控梯度范数、有效 token 数和不同任务 loss。

### 5. 跨境支付经验表达

例如微调工单路由和工具调用时，我会按国家、币种、通道和错误类型切分数据，测试集按时间切分防止规则版本泄漏。金额、费率、汇率、名单和订单状态不写死进模型权重，而是通过受控工具实时读取。

### 6. 当前与未来方向

当前 PEFT 已从基础 LoRA 扩展到 DoRA、LoRA+、rsLoRA、QLoRA 和多 Adapter 路由。未来更重要的是自动选择插入层、按任务动态 rank、Adapter 组合和与量化/推理引擎的一体化，而不是盲目提高 rank。
