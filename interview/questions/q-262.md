# 嵌入模型的选择标准？存储向量与检索向量，用的嵌入模型是否一致

## 答案

注：问问题时啰嗦一堆，把我绕晕了，回头整理才发现问题其实很简单

### 正式回答

嵌入模型选择主要看几个维度：

- 中文和领域语义效果。
- 向量维度和存储成本。
- 上下文长度。
- 检索速度和吞吐。
- 是否支持多语言。
- 是否支持稠密、稀疏、多向量混合。
- 和 rerank 模型的搭配效果。
- 是否能私有化部署。

存储向量和检索向量原则上必须使用同一个 embedding 模型，至少要在同一个向量空间里。否则入库向量和 query 向量不是一个分布，相似度没有意义。

如果要升级 embedding 模型，通常要重建索引，或者做双索引灰度：老模型继续服务，新模型重新入库，评测通过后切流。

### 结合跨境支付项目怎么说（面试口径）

> 在跨境支付 RAG 里，我会优先选择中文和金融支付术语效果好的 embedding 模型，同时看它对英文缩写的支持，比如 payout、chargeback、KYC、AML、SWIFT、SEPA 这些词。因为支付知识库经常中英文混杂。
>
> 入库和查询必须用同一个 embedding 模型。比如我用 BGE 模型把通道规则入库，查询时也要用 BGE 生成 query 向量，不能中途换成另一个模型直接查老索引。
>
> 如果要升级模型，我会做双索引灰度，用真实工单评估 recall@k、引用命中率和答案准确率。通过后再切换，避免一升级就导致客服答案召回异常。

**钩子**：可以引到“同一向量空间、embedding 升级重建索引、支付术语中英文混合”。
