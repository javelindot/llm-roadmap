# 训练模型时用了几张卡,有遇到什么异常中断问题吗?

## 答案

项目不是从零预训练基础模型，而是对 7B/14B 级模型做领域 SFT、LoRA/QLoRA 和工具调用训练。小规模验证通常使用 1–2 张卡，正式全量训练使用单机 8 卡；具体采用 DDP、DeepSpeed ZeRO-2/3 或参数高效微调，取决于模型、序列长度和是否训练全部参数。

卡数必须与 batch 计算一致：

```text
global_batch_size
= micro_batch_per_gpu
  * gradient_accumulation_steps
  * data_parallel_size
```

如果 8 张卡中还使用 TP=2，那么 data parallel size 只有 4，不能直接把 8 代入。报告训练规模时还要说明 GPU 型号、显存、最大序列长度、精度和 tokens/s，只报“8 卡”信息不足。

### 1. 遇到过的异常类型

**OOM**

- 某些样本异常长，平均长度正常但峰值 activation 超限；
- 动态 batch 中 token 数失控；
- checkpoint 或评测阶段额外保存 logits；
- ZeRO bucket、临时 buffer 和显存碎片。

处理方式：

- 记录每批最大长度和 token 数；
- 使用 token-based batching、截断和 sample packing；
- 开启 gradient checkpointing、FlashAttention；
- 降低 micro batch，用 gradient accumulation 保持 global batch；
- 必要时升级 ZeRO stage，而不是盲目清缓存。

**NCCL timeout 或进程卡死**

常见根因不是 NCCL 本身，而是某个 rank 先 OOM、DataLoader 卡住或执行了不同的 collective，其他 rank 最后才超时。

排查顺序：

1. 找最早报错的 rank，而不是只看最后的 timeout；
2. 对比各 rank 的 step、batch shape 和 collective；
3. 检查网络、GPU Xid、节点和容器日志；
4. 开启 NCCL/PyTorch 分布式调试日志；
5. 确保异常时所有 worker 一起退出并由作业系统重启。

**Loss NaN/Inf**

- FP16 overflow 或 loss scale 不合适；
- 脏样本、非法标签；
- 学习率过大；
- 梯度爆炸；
- 某个 rank 输入为空或 mask 错误。

处理时先定位首次出现异常的 step，保存 batch 摘要，检查 logits、loss、grad norm。硬件支持时优先 BF16，并结合 gradient clipping、warmup 和数据校验。

**节点抢占、容器重启和磁盘问题**

- Spot/共享集群节点被回收；
- checkpoint 写满磁盘；
- 对象存储上传未完成；
- 容器被 OOM killer 终止；
- DataLoader worker 句柄或共享内存耗尽。

### 2. Checkpoint 不能只保存模型权重

要无缝恢复训练，至少保存：

- model/sharded model state；
- optimizer state；
- LR scheduler；
- AMP scaler；
- global step、epoch；
- Python/NumPy/PyTorch/CUDA RNG state；
- sampler 或数据游标；
- DeepSpeed/FSDP 分片元数据；
- 代码、配置、数据和 tokenizer 版本。

如果只恢复权重，学习率、Adam 动量和数据顺序改变，loss 曲线可能跳变，严格来说不是续训。

### 3. 可靠保存策略

- 按 step 定期保存，关键阶段额外保存；
- 先写临时目录，成功后原子提交 manifest；
- 每个 rank 分片完成后再标记 checkpoint 可用；
- 本地快速盘保存后异步复制到远端；
- 保留最近 N 个和最佳若干 checkpoint；
- 定期实际演练 resume，不要等故障才验证；
- 使用独立的健康检查与作业重启策略。

跨境支付训练数据还要保证恢复后数据权限、脱敏版本和审计信息一致。故障恢复的目标不只是“任务重新跑起来”，还包括不重复消费错误数据、结果可追溯和评测可复现。
