# 对prompt工程的了解程度

## 答案

Prompt 工程不是“找到一句神奇提示词”，而是对任务、上下文、约束、工具和输出进行结构化设计，并用评测、版本和观测持续迭代。

### 1. 基本结构

高质量 Prompt 通常包含：

```text
Role/Goal
Context
Allowed evidence
Constraints
Steps or policy
Output schema
Examples
Failure behavior
```

系统指令应明确优先级，用户数据和工具内容要标记为不可信输入。

### 2. 常用方法

- Zero-shot；
- Few-shot；
- 分解任务；
- 结构化输出/JSON Schema；
- 提供 rubric；
- query rewrite；
- self-consistency；
- tool calling；
- retrieval-augmented prompt；
- 先计划再执行；
- 让模型在证据不足时拒答或追问。

是否展示长 CoT 要考虑安全、成本和可验证性，更推荐输出简洁依据和证据。

### 3. 上下文工程

Prompt 之外还要决定：

- 放哪些历史；
- 如何摘要；
- 检索哪些文档；
- 工具结果如何裁剪；
- 状态如何结构化；
- Token 预算如何分配；
- 哪些内容不应进入模型。

很多所谓 Prompt 问题其实是数据和上下文问题。

### 4. 评测驱动

- 建立 golden set 和 bad-case taxonomy；
- Prompt 版本化；
- 固定模型和解码参数；
- 比较任务成功、格式、事实、安全和成本；
- 防止只针对几个案例过拟合；
- shadow/A-B 后再上线。

### 5. 安全

- Prompt injection 隔离；
- 最小工具权限；
- 敏感信息脱敏；
- tool schema 校验；
- 不可逆动作确认；
- 输出后验证；
- 不把 system prompt 当唯一安全防线。

### 6. 什么时候需要训练

如果 Prompt 已很复杂但高频错误仍稳定存在，可以考虑：

- SFT；
- DPO/RL；
- 专用分类器；
- RAG/工具；
- 工作流重构；
- 更合适的模型。

不能无限向 Prompt 堆规则。

### 7. 跨境支付场景

Prompt 可约束“先查订单再解释、不能承诺未知到账时间、资金动作必须确认”，但真正的订单状态、权限和合规边界仍由系统实现。

### 8. 面试总结

我把 Prompt 工程看成可测试的软件配置：结构化、版本化、可观测，并与检索、工具、状态和安全共同设计。目标是稳定任务成功，不是追求措辞技巧。
