# 项目里的异步编程？实现并发？

## 答案

> Python 异步编程在 LLM 工程里**几乎是标配**——LLM 调用 IO 密集，async 能让单进程吃满并发。我们跨境支付**asyncio + FastAPI 全栈**，分享下经验。
>
> **为什么必须异步**：
>
> - LLM 调用是**IO 密集**（HTTP / gRPC 等待）
> - 同步 + 单线程：QPS 1（一次卡死等模型）
> - 异步：QPS 100+（同一时刻挂着 100 个等待）
> - 同步 + 多线程：GIL 限制 + 切换开销大
> - **异步是 Python 高并发 LLM 服务的唯一解**
>
> **核心概念**：
>
> ```python
> import asyncio
>
> async def async_func():
>     # await 让出执行权
>     result = await some_io_call()
>     return result
>
> async def main():
>     # 并发执行多个任务
>     results = await asyncio.gather(
>         async_func(),
>         async_func(),
>         async_func()
>     )
>
> asyncio.run(main())
> ```
>
> **我们 RAG 服务的并发场景**：
>
> **1. 多路检索并发**
> ```python
> async def hybrid_search(query: str):
>     vec_task = asyncio.create_task(milvus_search(query))
>     bm25_task = asyncio.create_task(es_search(query))
>     hyde_task = asyncio.create_task(hyde_search(query))
>     vec, bm25, hyde = await asyncio.gather(vec_task, bm25_task, hyde_task)
>     return rrf_fusion(vec, bm25, hyde)
> ```
> 串行 3 × 100ms = 300ms，并行 100ms，**省 200ms**。
>
> **2. Map-Reduce 调用 LLM**
> ```python
> async def summarize_long_doc(chunks: list[str]):
>     # 并发处理每个 chunk
>     summaries = await asyncio.gather(*[
>         llm.summarize(chunk) for chunk in chunks
>     ])
>     # 聚合
>     return await llm.combine(summaries)
> ```
>
> **3. Agent 工具并行调用**
> ```python
> async def gather_evidence(ticket):
>     order_task = asyncio.create_task(query_order(ticket.order_id))
>     kyc_task = asyncio.create_task(query_kyc(ticket.user_id))
>     risk_task = asyncio.create_task(query_risk_score(ticket.user_id))
>     order, kyc, risk = await asyncio.gather(order_task, kyc_task, risk_task)
>     return synthesize(order, kyc, risk)
> ```
>
> **4. 流式生成**
> ```python
> async def stream_chat(query: str):
>     async for token in llm.stream(query):
>         yield f"data: {token}\n\n"
> ```
>
> **几个工程要点**：
>
> **1. 不能调 sync 阻塞函数**
> ```python
> # 错的：会阻塞整个事件循环
> async def bad():
>     result = blocking_db_call()  # ❌
>
> # 对的：用 run_in_executor 隔离
> async def good():
>     loop = asyncio.get_event_loop()
>     result = await loop.run_in_executor(None, blocking_db_call)
> ```
>
> **2. 限流避免 LLM 过载**
> ```python
> sem = asyncio.Semaphore(10)  # 最多 10 个并发
> async def call_llm(query):
>     async with sem:
>         return await llm.invoke(query)
> ```
>
> **3. 超时控制**
> ```python
> try:
>     result = await asyncio.wait_for(slow_call(), timeout=5)
> except asyncio.TimeoutError:
>     return fallback
> ```
>
> **4. 异常处理**
> ```python
> results = await asyncio.gather(
>     task1(), task2(), task3(),
>     return_exceptions=True  # 一个失败不全挂
> )
> for r in results:
>     if isinstance(r, Exception):
>         logger.error(f"Task failed: {r}")
> ```
>
> **5. 异步上下文管理**
> ```python
> async with aiohttp.ClientSession() as session:
>     async with session.get(url) as response:
>         data = await response.json()
> ```
>
> **常用异步库**：
>
> - **HTTP**：aiohttp、httpx
> - **DB**：asyncpg、aiomysql、motor（Mongo）
> - **Redis**：aioredis、redis-py async
> - **Web 框架**：FastAPI、Starlette
> - **任务队列**：Celery（不是异步原生但常用）、arq
> - **消息队列**：aiokafka
>
> **几个踩坑经验**：
>
> 1. **不能 mix sync 和 async**——一旦混了，整个 chain 都要异步
>
> 2. **CPU 密集任务用 ProcessPoolExecutor**——asyncio 解决不了 CPU 密集
>
> 3. **Sentry / OpenTelemetry 要支持 async**——同步版本会丢 trace
>
> 4. **多个 event loop 别混**——Jupyter 里嵌套 asyncio 用 nest_asyncio
>
> 5. **测试用 pytest-asyncio**——异步代码测试套件
>
> **并发性能对比**：
>
> 我们 RAG 服务的实测：
> - 同步单进程：5 QPS
> - 异步单进程：80 QPS
> - 异步 + 4 worker：250 QPS
> - 异步 + 4 worker + 4 副本 K8s：1000 QPS
>
> **不适合异步的场景**：
>
> - CPU 密集计算（用多进程）
> - 简单脚本（不值得复杂化）
> - 团队不熟（学习成本）
>
> 一句话：**LLM 工程异步是标配**——asyncio + FastAPI + httpx 是 Python 后端高并发的金三角，配合限流、超时、异常处理才能稳定上生产。

**🪝 顺嘴带出去的钩子**：
- "asyncio + run_in_executor 处理同步阻塞可以聊。"
- "Semaphore 限流在 LLM 调用里的应用可以展开。"
- "FastAPI 异步 worker 配置可以讲讲。"
