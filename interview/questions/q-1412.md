# 如何优化系统支持更高并发？存储瓶颈和流量瓶颈？

## 答案

高并发优化先建立容量模型并定位瓶颈，不能一开始就盲目加机器。常用关系：

```text
并发数 ≈ QPS * 平均响应时间
```

应分别测入口、应用、缓存、数据库、消息队列、外部依赖和模型服务。

### 1. 流量层优化

- CDN 和边缘缓存；
- 负载均衡；
- 无状态服务水平扩容；
- 连接复用和连接池；
- 限流、熔断、超时和重试预算；
- 请求合并、批处理和异步化；
- 热点 key 隔离；
- 多级缓存；
- 削峰填谷和消息队列；
- 优先级队列与背压。

重试必须带指数退避和 jitter，否则故障时会形成重试风暴。

### 2. 存储瓶颈

常见信号：

- 数据库 CPU/IO 高；
- 慢查询；
- 锁等待；
- 连接池耗尽；
- 主从延迟；
- 缓存命中率下降；
- 热分区；
- 磁盘或网络带宽饱和。

优化方法：

- 建索引和重写 SQL；
- 减少 N+1 查询；
- 读写分离；
- 分库分表；
- 缓存热点数据；
- 批量读写；
- 异步落库；
- 合理的数据分区；
- 避免大事务；
- CQRS 或物化视图；
- 冷热数据分层。

支付账务不能为了吞吐牺牲一致性。余额、清算和幂等记录应使用事务、唯一约束、版本号或原子操作；缓存只能加速读取，不能作为权威账本。

### 3. 模型服务瓶颈

- continuous batching；
- prefix/KV cache；
- 量化；
- speculative decoding；
- 小模型路由；
- 请求按长度分桶；
- 限制最大上下文和输出；
- CPU/GPU 异构调度；
- 模型副本自动扩缩；
- RAG 检索并行与超时降级。

要区分 Prefill 和 Decode，分别看 TTFT、TPOT、tokens/s 和 KV Cache 使用率。

### 4. 一致性和降级

- 幂等 key；
- 去重表；
- Outbox/Inbox；
- 状态机和版本控制；
- 读写超时；
- 熔断外部通道；
- 非核心功能降级；
- 高风险请求保留强一致和人工兜底。

跨境支付中不能重复扣款或重复出款，客户端重试和消息消费必须统一使用业务幂等键。

### 5. 排查方法

```text
压测
-> 指标与 tracing 定位最慢环节
-> 建立单机容量和依赖容量
-> 优化最大瓶颈
-> 再压测
-> 做故障和降级演练
```

监控 QPS、并发、P95/P99、错误率、饱和度、队列长度、缓存命中率、DB 慢查询和依赖超时。

### 6. 面试总结

高并发优化要从流量、应用、存储、外部依赖和模型服务分层处理。流量瓶颈靠扩容、缓存、异步、限流和背压；存储瓶颈靠索引、分区、读写分离和批处理。支付核心链路同时要保证幂等、一致性和可回滚。
