# LLM 的对齐阶段分为 SFT 和 RLHF，可以跳过 SFT 直接 RLHF 吗？

## 答案

**一句话**：**理论上可以、工业实践基本不行**——除非你像 DeepSeek-R1-Zero 那样有海量可验证奖励 + 算力，否则跳过 SFT 会换来训练不稳、对话格式乱、KL 漂移和昂贵的算力账单。

### 1. 为什么默认要 SFT 先行
- **格式启动**：base 模型不知道"对话格式"长啥样，连"User:""Assistant:" 都不会用
- **冷启动收敛快**：SFT 是监督学习，几小时就能让模型像样地回答
- **降低 RL 的方差**：好的 SFT 起点让 PPO 探索空间收敛得快得多
- **KL 约束有意义**：RLHF 用 SFT 模型作 ref，KL 才有合理基线
- **业务规范注入**：客服话术、品牌口吻这类靠 SFT 注入最高效

### 2. 跳过 SFT 直接 RLHF 会怎么样
- 输出格式混乱（base 模型在做下一词预测，输出 prompt 风格的文本）
- RL 几乎无法收敛，因为策略空间太大、奖励太稀疏
- PPO 的 KL ref 是 base 模型，约束等于没约束
- 调参极困难，β、clip、learning rate 怎么调都飘
- 训练成本暴涨

### 3. 例外：DeepSeek-R1-Zero 的启示
R1-Zero 真的跳过了 SFT，从 base 直接 RL，证明**纯 RL 是可行的**——前提：
- 任务有**可验证奖励**（数学/代码）
- 算力管够（大规模采样）
- 可以接受格式 ugly（"a-ha moments" 但读起来不好）
- 后面还是要做 SFT 修复格式（R1 = R1-Zero + 冷启动 SFT + 多阶段 RL）

### 4. 何时可以激进点跳过/简化 SFT
- 业务领域**有强可验证信号**（数学、代码、规则匹配）
- 算力非常充裕
- 接受输出风格不那么"工业产品"
- 目标是做研究，不是上业务

### 5. 工业界共识路径
```
Base → SFT (1w-100w 条指令) → RM 训练 → DPO/PPO → 上线
```
其中 SFT 这一步**几乎所有商业模型都不跳**。

### 6. 一句话总结
**SFT 是 RLHF 的"地基"，跳了不是不能盖楼，而是要烧很多算力补地基**——工业界除非有海量可验证奖励，否则别试。

## 结合项目谈（口语版 · 约 40 秒）

> 这道题我们跨境支付实打实试过。当时算法实习生看了 R1-Zero 论文很兴奋，跟我提议"我们客服对齐能不能跳过 SFT 直接 DPO？反正都有偏好数据"。
>
> 我让他先做了个小实验：拿 Qwen-7B 的 base（不是 chat 版），直接喂 5000 条客服偏好对跑 DPO。结果第一个 batch 模型输出就崩了——base 模型在做下一词预测，给它一个客户问题，它接着续写"客户问题，下面是答案，答案是这样的，因为……"，完全不是对话格式。DPO 的偏好信号在这种格式下根本学不动。
>
> 我们的结论是：**SFT 在工业场景不可跳**，至少要做"格式注入 SFT"——5000 条标准对话样本，让 base 学会"User → Assistant"的结构，再上 DPO 才稳。
>
> R1-Zero 那种纯 RL 路线我们调研过，**前提是任务有强可验证奖励**（数学、代码对错明确）。客服回复这种主观偏好任务没有可验证信号，纯 RL 在我们这儿不现实。
>
> 所以我面试碰到这个题会说："理论上可以，工业上别试，除非你像 R1-Zero 那样烧得起算力且任务有可验证奖励。"

**🪝 顺嘴带出去的钩子**：
- "R1-Zero 在业务场景的可行性我们调研过，可以展开。"
- "格式注入 SFT 用多少数据够用我们有经验值。"
- "可验证奖励在风控场景是怎么设计的可以聊聊。"
