# 是否了解 OneRec

## 答案

OneRec 是快手提出的端到端生成式推荐方案，核心目标是用统一生成模型替代传统“召回、粗排、精排”的级联学习目标，让模型直接根据用户历史生成一组候选视频。

### 1. OneRec V1

OneRec V1 将推荐建模为自回归生成：

```text
用户历史行为 → Encoder
             → Decoder 逐步生成推荐 item 序列
```

主要设计包括：

- **Encoder-Decoder**：Encoder 编码用户历史，Decoder 生成视频标识；
- **Sparse MoE**：扩大参数容量，同时控制每个 token 激活的计算量；
- **Session-wise Generation**：一次生成具有上下文关系的一组 item，而不是逐个独立预测；
- **Iterative Preference Alignment**：使用 Reward Model 构造偏好数据，再通过 DPO 对齐用户偏好；
- **受约束候选**：生成结果仍需映射到真实、可展示的 item。

论文报告 OneRec 在快手主场景部署后 watch-time 提升 1.6%。这是论文特定流量、基线和基础设施下的线上结果，不能直接外推到其他平台。

### 2. 为什么要统一召回和排序

传统级联系统各阶段优化目标不同：

- 召回强调覆盖；
- 粗排强调低成本保正例；
- 精排强调最终业务目标；
- 手工规则再做列表组合。

阶段之间可能产生信息损失和目标不一致。OneRec 试图直接学习：

```text
P(item_1, item_2, ..., item_k | user_history)
```

这样前一个生成结果可以影响后一个结果，更自然地学习列表上下文和多样性。

### 3. OneRec-V2 的改进

截至 2026 年 6 月，公开的 OneRec-V2 技术报告使用 Lazy Decoder-Only Architecture，重点解决 V1 的计算分配问题。报告指出 V1 中 97.66% 的资源消耗在序列编码，而不是生成。

V2 的主要变化是：

- 移除 Encoder 瓶颈，采用 Lazy Decoder-Only；
- 报告总计算量降低 94%、训练资源降低 90%；
- 扩展到 8B 参数；
- 使用 Duration-Aware Reward Shaping；
- 使用 Adaptive Ratio Clipping 从真实用户交互中做偏好对齐。

论文报告 A/B 测试中 App Stay Time 提升 0.467%/0.741%。这些数字对应报告中的不同实验设置，应原样理解为论文结果，而不是通用收益承诺。

### 4. 关键难点

- item 如何编码为稳定、可生成的标识；
- 新增、删除和下架 item 如何及时生效；
- 如何保证生成的 item 合法且当前可展示；
- 自回归解码的 P95/P99 和 GPU 成本；
- 热门偏差、重复生成和列表多样性；
- Reward Model 是否准确模拟真实用户；
- 端到端替换后如何定位问题和快速回滚；
- 广告、运营、安全和业务硬规则如何接入。

因此生成式推荐不是“去掉所有工程规则”，而是将合法候选约束、实时过滤和业务治理放在生成模型周围。

### 5. 与传统系统的关系

更稳妥的落地路径是：

1. 用生成模型增强内容和用户表示；
2. 作为一路生成式召回，与传统多路召回并行；
3. 在合法候选集上做生成式重排；
4. 通过 shadow 和 A/B 验证质量、延迟、成本与稳定性；
5. 再决定是否扩大到端到端主链路。

传统系统在精确过滤、实时状态、低延迟和可解释治理方面仍有优势，OneRec 更擅长统一长序列和列表偏好。

### 6. 跨境支付中的适用边界

OneRec 的生成思想可用于从已通过准入的产品池中生成收款、付款、换汇或收单方案，但支付场景不应直接端到端生成资金路由：

- KYB、牌照、制裁、额度和通道状态先做硬过滤；
- 模型只能选择真实存在且允许使用的产品；
- 报价、风控、幂等和资金执行由确定性服务完成；
- 生成模型可以解释方案，但不能编造费率、时效和审批结果。
