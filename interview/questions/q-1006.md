# RAG怎么评估,指标有哪些

## 答案

RAG 评估不能只看最终回答是否“像对的”，需要把查询、召回、重排、上下文和生成拆开评估，再用线上业务指标验证端到端价值。

> 黄金集建设 → 分层离线评测 → 端到端回归 → 影子流量 → 线上 A/B 与 bad case 回流。

### 1. 先构建可诊断的黄金集

每条样本至少包含：

> `query、历史上下文、业务标签、相关文档/chunk、答案要点、引用、允许工具、拒答条件`

跨境支付评测集按国家、币种、产品、通道、语言、问题难度和风险等级分层，覆盖：

- 产品规则、KYC/KYB、手续费和结算周期；
- 错误码、付款失败、退款和拒付；
- 多轮指代、信息缺失和多意图问题；
- 过期文档、规则冲突和无答案问题；
- 越权查询、提示注入和敏感信息请求。

训练集、调参集和最终测试集按时间或业务实体隔离，防止相似 FAQ 或同一文档泄漏到测试集。

### 2. 查询理解与路由指标

- 意图分类 Accuracy、Macro-F1；
- 多标签意图的 Precision、Recall、F1；
- 实体抽取的 span/entity F1；
- Query 改写的指代恢复正确率；
- 知识库、工具和人工路由准确率；
- 应拆分问题的子问题覆盖率；
- 越权路由和错误工具路由率。

支付问题经常同时包含“查订单”和“问规则”，因此只评单标签准确率不够，还要评估多意图拆分是否完整。

### 3. 召回与重排指标

召回层重点看正确证据是否进入候选：

- Recall@K、HitRate@K；
- MRR：首个正确证据排得是否靠前；
- NDCG@K：多条相关证据的整体排序质量；
- Context Recall：答案需要的要点是否被上下文覆盖；
- 分市场、分语言和长尾实体的召回率；
- 各召回通道的独占贡献、重合率和延迟。

重排层重点看噪声是否被压下去：

- Precision@K、NDCG@K；
- Context Precision；
- 关键证据保留率；
- 过期、越权和跨国家规则进入上下文的比例。

Recall@K 提升但 Context Precision 下降，说明只是召回了更多噪声，端到端回答未必变好。

### 4. 生成质量指标

- **Faithfulness/Groundedness**：回答中的 claim 是否由证据支持；
- **Answer Correctness**：事实和业务结论是否正确；
- **Answer Relevance**：是否真正回答用户问题；
- **Completeness**：必要条件、例外和下一步是否完整；
- **Citation Accuracy**：引用是否真的支持对应结论；
- **拒答质量**：无证据时是否拒答，应答问题是否被误拒；
- **安全合规**：是否泄露敏感信息、错误承诺资金状态或暴露风控规则。

对于订单状态、金额和币种等结构化字段，优先使用程序规则比对；对于自然语言完整性，可使用人工评审或经过校准的 LLM judge。

### 5. 端到端和业务指标

端到端指标包括：

- 任务成功率和首次解决率 FCR；
- 人工改写率、转人工率、重复咨询率；
- 用户满意度、投诉率和错误承诺率；
- 工具调用成功率和最终业务状态正确率；
- 单个成功解决成本。

跨境支付存在延迟结果，退款到账、拒付和投诉可能数天后才能确认。因此要设置即时、短期和成熟标签窗口，不能只按会话结束时的满意度判断。

### 6. 性能与运维指标

- 首字时间、总响应时间、P95/P99；
- 检索、重排、模型和工具各阶段耗时；
- QPS、超时率、错误率和降级率；
- token、GPU、搜索和第三方 API 成本；
- 索引新鲜度、文档解析失败率和版本滞后；
- 缓存命中率与错误缓存率。

所有指标需要关联 `query_id、trace_id、知识版本、embedding/reranker/LLM 版本`，否则出现退化时无法定位。

### 7. 评测方法

自动评测适合持续回归，人工评测负责高风险和校准：

1. 规则检查结构化事实、权限和引用；
2. RAGAS、RAGChecker 等框架评估检索与生成；
3. LLM judge 扩大覆盖，但用业务专家标注集校准一致性；
4. 每次变更做分桶回归和消融实验；
5. 影子流量观察真实分布；
6. 线上 A/B 同时设置质量、风险、延迟和成本 guardrail。

评测报告不能只给一个总分。RAGChecker 和 RAGVUE 代表的趋势是做细粒度诊断：明确错误来自检索缺失、上下文噪声、引用错配还是生成不忠实，才能指导下一轮优化。
