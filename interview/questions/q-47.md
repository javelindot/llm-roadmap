# 该选什么微调方法，全参数微调 / LoRA？

## 答案

**一句话**：**全参拼上限、LoRA 拼性价比**——数据多、要改基础能力 → 全参；数据少、迭代频繁、显存紧 → LoRA / QLoRA。工业 80% 场景 LoRA 就够。

### 1. 全参数微调（Full FT）
- **改动**：所有参数都训
- **显存**：参数 × 2（FP16）+ 梯度 × 2 + Adam × 8 ≈ **参数 × 16** 字节
- **适用**：
  - 数据量大（>10w 条）
  - 要改基础能力（不只是风格）
  - 显存充裕
  - 模型规模不大（7B-13B 单机能跑）
- **风险**：灾难性遗忘，要混入 base 数据

### 2. LoRA（Low-Rank Adaptation）
- **改动**：冻 base，只在 attention/MLP 旁挂低秩矩阵 ΔW = A·B
- **可训参数**：通常 < 1% 原模型
- **显存**：base 占大头 + LoRA 几乎不占
- **适用**：
  - 数据少（1k - 10w）
  - 风格 / 任务对齐为主
  - 多任务并存（多个 LoRA 切换）
  - 频繁迭代

### 3. QLoRA
- **base 4bit 量化（NF4）** + **LoRA 16bit 训练**
- 7B 单卡 24G 跑得起来
- 14B 单卡 H100 80G 没问题
- 论文证明效果接近全参 LoRA

### 4. 其他 PEFT
| 方法 | 思路 | 何时用 |
|------|------|--------|
| **Prefix Tuning** | 在每层 attention 前加可训 prefix | 早期方法，效果一般 |
| **P-Tuning v2** | 类似 prefix，但用 deep prompt | 中文项目偶用 |
| **Adapter** | 每层插一个小 MLP | 早期方法 |
| **IA3** | 缩放向量，参数极少 | 资源极紧 |
| **DoRA** | 分解 LoRA 的方向和幅度 | LoRA 的改进版 |

### 5. 选型 decision tree
```
有 10w+ 高质量数据 + 充裕 GPU → 全参
要改基础能力（如长 CoT 推理） → 全参
普通任务对齐（风格、领域、客服） → LoRA
显存有限 + 单卡训 → QLoRA
多个任务共享 base → 多 LoRA 切换
```

### 6. 实战经验
- LoRA rank 选择：通常 16-64，太小欠拟合，太大易过拟合
- **LoRA target**：attention 的 q/k/v/o + MLP 的 up/gate/down 全加，效果最好
- LoRA alpha：建议 alpha = 2 × rank（论文经验）
- 训完合并：LoRA 可以 merge 回 base 一键合并

### 7. 一句话总结
**LoRA 是工业默认选项，全参留给"必须改基础能力"的场景**——能用 LoRA 就别全参，能用 QLoRA 就别 LoRA，能 merge 就 merge。

## 结合项目谈（口语版 · 约 40 秒）

> 我们跨境支付里几条线都做过微调，选型很明确：**风格/任务用 LoRA，基础能力用全参**。
>
> **客服回复**——典型 LoRA 场景。5w 条客服历史话术，目的是让模型说话像客服、知道我们品牌的语气，base 能力（中文、对话）保留。Qwen-7B + LoRA rank=32 + alpha=64，单卡 A100 跑 4 小时收敛。上线之后业务有需求改风格，**重训一晚上就能换一版**，全参根本做不到这个迭代速度。
>
> **风控交易分类**——这块我们用了**全参**。原因是数据量够（80w 条标注样本）、任务是"判断这笔交易像不像洗钱"，要让模型在底层学到"什么叫异常模式"，LoRA 改不动这种基础语义。全参 Qwen-7B + ZeRO-2，8 卡 H100 跑了两天，效果比 LoRA 高 12 个点。
>
> **多模态 KYB 识别**——我们用了 QLoRA，因为是新场景实验性质，数据 2w 条，单卡跑通验证效果，验证完之后再讨论是否上全参。
>
> 我们的经验：**LoRA 像换衣服，全参像整容**——风格调用 LoRA，基础能力靠全参。
