# 标准RAG有什么问题？

## 答案

标准 RAG 通常指“切片 → 向量检索 Top-K → 拼接 Prompt → LLM 生成”。它能快速落地，但在复杂业务中有明显局限。

### 1. 数据入库问题

- PDF、表格、图片解析错误。
- 固定长度切片破坏语义和层级。
- 重复、过期、冲突文档没有治理。
- 权限和版本元数据缺失。

### 2. 检索问题

- 纯向量检索不擅长精确数字、专有名词和错误码。
- 用户问题与文档表达差异大，Query 不匹配。
- 单路召回漏掉关键词或关系信息。
- Top-K 只按局部相似度，缺少全局、多跳证据。

### 3. 上下文问题

- 召回片段相关但不完整。
- 多个片段重复、顺序混乱。
- 无关上下文过多导致 Lost in the Middle。
- 长上下文增加成本，却不保证模型使用正确证据。

### 4. 生成问题

- 模型无视证据，按参数记忆回答。
- 证据冲突时没有版本与可信度判断。
- 引用与实际结论不对应。
- 没有答案时仍强行生成。

### 5. 工程问题

- 知识增量更新和删除不及时。
- 多租户权限泄漏。
- 缺少检索、生成分层评测。
- 延迟、Embedding 成本和索引容量不可控。

### 6. 常见改进

- 结构化/语义分块与父子索引。
- Query Rewrite、HyDE、Multi-query。
- BM25 + Vector 混合召回。
- Reranker、去重、上下文压缩。
- GraphRAG 或 Agentic RAG 支持多跳。
- 引用校验、拒答和 Corrective RAG。

**面试总结**：标准 RAG 最大的问题是把“召回相似文本”等同于“找到完整正确证据”，而生产系统需要数据治理、混合检索、重排和生成校验。
