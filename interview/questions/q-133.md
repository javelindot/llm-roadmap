# 什么是 Graph RAG？

## 答案

**一句话**：Graph RAG = **把知识库构建成"实体-关系"图谱，检索时同时利用向量近似和图谱关系导航**——擅长需要**多跳推理 / 跨文档关联**的场景，是 Vector RAG 的高级互补。

### 1. 普通 RAG 的局限
- 向量召回是"独立 chunk"
- 跨文档关联弱
- 多跳问题（"A 公司的子公司 B 的法人代表是谁"）召回不全
- 长尾实体的关系信息散落

### 2. Graph RAG 的核心思路
1. **离线建图**：用 LLM 从文档抽取实体和关系，构造 (头实体, 关系, 尾实体) 三元组
2. **存图谱**：Neo4j / NebulaGraph / 内存图
3. **检索时**：query → 找相关实体 → 在图上遍历 → 拉相关子图作为上下文
4. **生成时**：把子图 + 文本块一起喂 LLM

### 3. 微软 GraphRAG 流程（2024 开源）
- **抽取**：LLM 抽 entity + relation
- **聚类**：Leiden 算法分社区
- **摘要**：每个社区生成层级摘要（多粒度）
- **检索**：query → 找社区 → 拉摘要 + 邻居
- **生成**：local（精细）+ global（社区摘要）两路融合

### 4. 适合 Graph RAG 的场景
- 多跳推理：A→B→C 关系链
- 跨文档关联：同一实体在多文档出现
- 长文档全局问答："这部小说的主要人物关系网"
- 知识密集型领域：医疗、法律、金融
- 实体关系丰富的业务：组织架构、商品关系、客户关系

### 5. 不适合 Graph RAG 的场景
- 单文档单 chunk 就能答的简单 QA
- 没有明显实体关系的"叙述性"文档
- 实时性要求极高（建图慢）

### 6. Graph RAG 的痛点
- **建图成本极高**：用 LLM 抽取 100w 字文档可能要几千美金
- **图谱质量难保证**：LLM 抽错关系会污染
- **维护成本**：文档更新要同步更新图
- **延迟**：图遍历 + 多次 LLM 调用
- **可解释**：好（图谱本身可视）
- **工程复杂**：图数据库 + 向量库 + LLM 三套系统

### 7. 工程组合
- **Hybrid RAG**：Vector + Graph 双路检索 + 融合
- **Graph for Global、Vector for Local**：宏观问题查图、细节查向量
- **元数据图**：不抽细粒度图，只建文档间引用关系

### 8. 一句话总结
**Graph RAG = 关系驱动的检索增强**——对"多跳关联"问题碾压 Vector RAG，但成本和工程复杂度也高，是个**目标精确**的工具。

## 结合项目谈（口语版 · 约 30 秒）

> 我们跨境支付**Graph RAG 在风控调查里上过**，分享下实际收益和坑。
>
> 业务场景：洗钱常通过"链式关联"——A 商户 → B 收款方 → C 中间账户 → D 最终受益人。传统 Vector RAG 召回的是"独立文档片段"，发现不了这种链。我们想试 Graph RAG。
>
> **怎么建图**：每条交易、每个商户、每个账户、每个客户都是节点，交易关系、关联账户、共同收款方是边。这个图我们本来就有（业务数据库自然就是图结构），不需要从文档抽取，这是天然优势。
>
> **检索流程**：
> 1. 用户问"调查这笔交易"
> 2. 拉这笔交易的子图（2 跳邻居），约 20-50 个节点
> 3. 把子图序列化成文本 + 知识库相关政策（Vector RAG）一起喂给风控 LLM
> 4. LLM 综合判断
>
> **收益**：单链路 Vector RAG 准确率 78%，加 Graph RAG 后 88%（**+10**）——多跳关联场景效果显著。
>
> **痛点**：
> - 业务图能扩到 2 跳，扩到 3 跳就 1000+ 节点，超出 LLM 上下文；
> - 节点 / 边 类型多了之后子图序列化为文本特别长，要精心 prompt 设计；
> - 实时性：风控调查接受 1-2s 延迟，所以图遍历可以接受。
>
> 我们**没上 GraphRAG 文档抽取流程**——因为我们的图天然来自业务库，不需要 LLM 抽取。**只在文档 RAG 里用了 Vector，业务关系图用结构化数据**，混合方案。
>
> 一句话：**Graph RAG 对"实体关系丰富"的业务是质变**，但要先看你有没有现成图，没有的话从头建图成本极高。
