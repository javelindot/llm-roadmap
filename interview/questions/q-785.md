# 若Agent推理API需要低延迟响应，会从哪些方面开展系统级优化工作？

## 答案

先把 Agent 端到端延迟拆成排队、检索、工具、模型 prefill、decode 和多步循环，并分别监控 TTFT、ITL/TPOT、P95/P99 与任务完成率。只优化单次模型调用，可能对多步 Agent 总延迟帮助有限。

### 1. 减少不必要步骤

- 简单请求直接走规则或单次模型，不进入通用 ReAct；
- 固定高风险流程使用 workflow，减少模型反复规划；
- 独立检索和工具并行执行，有依赖的步骤才串行；
- 设置最大步骤、token 预算和早停；
- 用小模型做意图路由，复杂任务才升级大模型。

### 2. 上下文与检索

- 压缩 system prompt、工具 schema 和历史消息；
- 只暴露当前允许的 top-k 工具；
- 缓存 query embedding、热点检索和公共前缀；
- ANN 分片、metadata 预过滤、轻量召回后只对少量候选 rerank；
- 订单、费率等实时数据并行查询，设置超时与降级。

### 3. 模型服务

- 使用 vLLM/SGLang/TensorRT-LLM 类服务的 continuous batching、Paged KV 管理和优化 kernel；
- FlashAttention、量化权重/KV、CUDA Graph 与编译；
- prefix caching 复用公共 system/tool 前缀；
- speculative decoding 降低逐 token 延迟，但需按接受率和负载实测；
- Prefill/Decode 分离、Tensor/Expert Parallel 用于更大规模；
- 流式输出改善感知延迟。

截至 2026 年，[vLLM 官方文档](https://docs.vllm.ai/en/stable/)已列出 chunked prefill、prefix caching、多种 speculative decoding 和分离式 prefill/decode 等能力，但具体收益依赖硬件和流量。

### 4. 系统治理

连接池、异步 I/O、超时、熔断、背压和租户隔离；按优先级和 deadline 调度，避免长请求阻塞短请求。缓存键必须包含租户、权限和数据版本。

跨境支付场景中，低延迟不能绕过合规和资金校验。可先流式返回“正在查询”，但最终结论必须等待权威系统；超时则降级人工，不能编造结果。

### 面试总结

Agent 延迟优化先减少模型步骤，再优化检索、工具和模型服务。用 TTFT、逐 token 延迟、排队和端到端 P99 分层定位，并在缓存、并发、量化和 speculative decoding 之间按真实负载取舍。
