# RAG Fusion为什么使用RRF？

## 答案

**一句话**：RAG Fusion 使用 RRF 是因为不同查询返回的分数不可直接比较，而 RRF 基于排名融合，简单、鲁棒、无需校准分数。

### 核心回答
RRF（Reciprocal Rank Fusion）公式常见形式：

```text
score(d) = sum(1 / (k + rank_i(d)))
```

含义：
- 一个文档在多个检索列表里排名越靠前，最终分数越高。
- 不依赖不同检索器的原始分数。
- 能融合 BM25、向量检索、多 query 检索结果。

为什么适合 RAG Fusion：
- 多个改写 query 的分数尺度不同。
- 不同检索方式分数不可比。
- 排名比绝对分数更稳定。

### 1. 为什么不用直接加分

不同检索器或不同 query 的分数尺度不一致：

- BM25 分数和向量相似度不是一个量纲。
- 不同 query 的相似度分布也不同。
- 有些检索器分数高，不代表和另一个检索器的高分可比。

直接加权容易不稳定。RRF 只看排名，更鲁棒。

### 2. RRF 的直觉

如果一个文档在多个召回列表里都排得靠前，说明它大概率重要；如果只在某一路里偶然靠前，融合后优势会小一些。

RRF 的 `k` 用来平滑排名影响，避免第一名和后面名次差距过大。

### 3. 面试总结

RRF 是一种简单实用的多路召回融合方法，特别适合 RAG Fusion、混合检索和多 query 检索。

### 4. 结合跨境支付项目怎么说（面试口径）

> 支付 RAG 里我会用 RRF 融合多路召回，因为 BM25 的分数、向量相似度、不同 query 的分数不能直接加。比如"PIX 入账延迟"和"巴西收款没到账"两个 query 都把同一份通道规则排在前面，那这份文档就应该被提升。
>
> RRF 简单但很实用，尤其适合支付这种有大量专有名词和业务别名的场景。它先解决多路结果怎么合并的问题，后面再接 reranker 和国家/通道过滤，效果会更稳。
>
> 我会把它放在召回融合阶段，而不是替代 rerank。RRF 解决多路融合，reranker 解决最终相关性判断。

**钩子**：RRF 解决的是多路召回分数不可比的问题。
