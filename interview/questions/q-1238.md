# 开放问题：为什么现在大家都在关注于大模型的推理能力 reasoning

## 答案

大家关注 reasoning，是因为模型从“生成看起来合理的文本”走向“完成可验证的复杂任务”时，瓶颈越来越集中在问题分解、约束满足、工具使用和错误修正，而不只是记住更多知识。

### 1. 预训练知识不等于任务求解

模型可能知道公式、API 和规则，但面对新组合问题仍要：

- 提取条件；
- 分解子问题；
- 选择工具；
- 保持中间状态；
- 验证结果并回退。

这些能力决定数学、代码、Agent 和复杂业务流程能否真正完成。

### 2. 推理任务更容易获得可验证反馈

数学答案、代码单测、SQL 执行、工具返回和规则约束都能提供相对客观的 reward。相比“文案是否优美”，可验证任务更适合大规模自动训练和迭代，也推动了 RL、verifier 和 test-time compute 的发展。

### 3. 推理时扩展提供新的 scaling 轴

除了扩大参数和预训练 token，还可以在推理阶段：

- 采样多个候选并投票；
- 搜索不同解题路径；
- 使用 verifier 选择答案；
- 动态分配思考 token；
- 调用搜索、代码和业务工具。

这使计算资源可以按问题难度分配，而不是每个请求都使用同样成本。

### 4. Agent 需要长程决策

真实应用不只是一次问答。软件工程、研究、数据分析和业务运营都需要多步执行。推理能力影响规划、状态跟踪、异常处理和停止条件，是 Agent 从 Demo 走向可靠系统的核心瓶颈。

### 5. 商业价值更容易度量

复杂任务能否正确完成，可以映射到任务成功率、人工节省、处理时长和错误率。相比单纯聊天流畅度，reasoning 的增益更容易形成可验证的产品价值。

### 6. 不能把长思维链等同于推理

- 输出更长不代表逻辑更强；
- benchmark 可能污染或被特定训练策略过拟合；
- 自然语言 CoT 不一定等于模型真实内部计算；
- 复杂问题仍会自信地产生错误中间步骤；
- test-time compute 会增加延迟和成本；
- 开放世界任务缺少可靠 verifier。

因此评测应覆盖最终正确率、过程约束、校准、鲁棒性和成本，而不是只比较思考 token 数。

### 7. 跨境支付中的边界

支付场景需要推理来整合订单状态、通道回执、清算规则和商户权限，但金额计算、账务一致性、制裁筛查和资金审批必须由确定性程序完成。模型负责问题分解、证据组织和建议，工具与规则负责事实和动作。高风险领域追求的是“可验证推理”，不是让模型自由发挥更长的思维链。
