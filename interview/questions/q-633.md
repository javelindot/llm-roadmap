# 开放题：你在大模型训练中遇到过的困难，如何解决？

## 答案

这是项目深挖题，重点不是罗列所有困难，而是讲清楚“现象、定位、方案、验证和沉淀”。回答时应替换成自己的真实经历，不要虚构规模或收益。

### 推荐回答结构

先用一句话交代训练任务、模型规模、数据量、硬件和目标指标，然后选择一个最有技术含量的问题展开。例如训练不稳定或显存不足：

1. 现象：loss 突然升高、出现 NaN、吞吐下降或特定 step OOM，并给出发生频率和影响。
2. 假设：从数据、数值稳定性、优化器、分布式通信和硬件五个方向列可能原因。
3. 定位：保存异常 batch 与 checkpoint，对比梯度范数、激活、学习率、样本长度、各 rank 日志和 NCCL 状态。
4. 处理：先做止损，再做根因修复。
5. 验证：固定数据和随机种子做对照实验，比较稳定性、质量、吞吐与成本。

### 示例思路：训练出现 NaN

先确认是 loss、梯度还是参数最早出现非有限值。检查异常 batch 是否含空样本、超长序列或错误 label mask；记录各层激活和梯度范数；关闭部分优化逐项复现。常见修复包括降低峰值学习率、延长 warmup、梯度裁剪、使用 bf16 替代 fp16、对敏感算子保留 fp32，以及修复数据清洗和除零问题。

修复后不能只看“不再 NaN”，还要跑足够长的回归训练，确认收敛速度和下游指标没有退化。

### 示例思路：显存和吞吐

用 profiler 拆分参数、梯度、优化器状态、激活和通信开销，再选择 gradient checkpointing、FlashAttention、序列打包、ZeRO/FSDP、张量或流水线并行。调整后同时报告 tokens/s、MFU、峰值显存和验证集指标，避免用精度下降换取表面加速。

结尾说明沉淀，例如增加数据预检、训练健康指标、自动保存异常 batch、可恢复 checkpoint 和标准压测脚本。这样的回答能体现系统化排障能力，而不只是记住某个参数。
