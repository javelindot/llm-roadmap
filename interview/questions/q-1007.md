# RAG如果有噪声怎么办

## 答案

RAG 噪声不是一个单点问题，需要先判断噪声产生在哪一层：

> 数据源噪声 → 解析噪声 → 分块噪声 → 召回噪声 → 上下文噪声 → 生成噪声。

不同来源的处理方法不同，不能只靠提高相似度阈值解决。

### 1. 数据源和版本噪声

常见问题包括重复文档、过期规则、草稿与正式版混用、不同国家规则混库，以及低可信网页进入知识库。

处理方式：

- 数据源白名单和 owner 责任制；
- 内容哈希与 MinHash 去重；
- 保存版本、生效时间、失效时间和替代关系；
- 草稿、正式、废止状态分开；
- 删除或下架通过事件及时同步索引；
- 冲突文档不静默覆盖，保留冲突标记和人工审核。

跨境支付中，同一通道不同国家、币种和商户类型的规则可能不同，必须使用元数据隔离，不能依赖模型从文本中自行判断适用范围。

### 2. 解析和分块噪声

PDF 页眉页脚、目录、OCR 错字、合并单元格、脚注和图表错位都会污染索引。应针对文件类型建立解析质量检查：

- 去除模板页眉、页脚和重复导航；
- 对扫描件做 OCR 置信度过滤；
- 表格保留表头、单位和行列关系；
- 按标题、规则、流程和表格结构分块；
- 使用 parent-child chunk 保留完整上下文；
- 每块绑定来源页码和章节路径。

支付规则中的限制条件和例外不能被切到不同 chunk，否则模型可能只召回“支持”而漏掉“不适用”。

### 3. 查询和召回噪声

用户问题过短、多义或包含多个任务时，召回会扩散。先做：

- 多轮指代消解和 query 改写；
- 意图识别与子问题拆分；
- 国家、币种、产品、通道和错误码抽取；
- 知识查询与实时工具查询路由；
- 权限和 metadata 前置过滤。

检索采用 BM25 与向量召回互补，再用 RRF 融合。简单、精确的错误码问题不应无条件使用 HyDE 或大量 query 扩展，因为生成式改写可能反而引入错误术语。

### 4. 重排和上下文去噪

召回后进行：

- cross-encoder rerank；
- 相似 chunk 聚类和去重；
- 来源权威性、时效和适用范围加权；
- 规则冲突检测；
- 低相关阈值过滤；
- token 预算内的证据覆盖优化。

Top-K 不是越大越好。可以先召回较多候选，再把少量高质量证据放入上下文。对于“支持 USD”和“不支持 USD”这类语义相近但结论相反的文本，还要结合否定词、结构化字段和版本校验。

### 5. 生成阶段降低噪声影响

Prompt 将不同证据按来源和业务对象分区，并明确：

- 只根据当前有效证据回答；
- 工具返回的实时状态优先于知识库说明；
- 不得合并不同国家、币种或商户规则；
- 每个关键结论必须引用来源；
- 证据冲突、缺失或低置信时追问或转人工。

可在生成后做 claim-evidence 校验，检查金额、币种、日期、订单状态和引用是否一致。涉及退款、到账承诺和风控原因时，使用模板化受控表达。

### 6. 防止恶意噪声

外部网页、上传文档和工具返回都可能包含提示注入。它们只能作为不可信数据，不能改变系统指令、权限或调用新的工具。

同时防止：

- 恶意文档污染长期记忆；
- 用户通过隐藏文本诱导泄密；
- 工具返回伪造的后续操作指令；
- 跨租户内容被召回到当前会话。

### 7. 用评测定位而不是凭感觉调参

重点观察：

- Context Recall 和 Context Precision；
- 过期、越权、跨市场文档进入上下文的比例；
- rerank 前后 NDCG 与关键证据保留率；
- Faithfulness、引用正确率和冲突回答率；
- 无答案场景的正确拒答率；
- 分国家、币种、语言和问题类型的 bad case。

通过逐层消融确认收益，例如固定生成模型，只替换分块、召回或 rerank。噪声治理的目标不是让上下文最少，而是在覆盖必要证据的前提下，提高有效信息密度。
