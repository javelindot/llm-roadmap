# 上下文压缩

## 答案

> **上下文压缩 = 在不丢失关键信息的前提下，把超长上下文压成更小、更密的形式**——核心驱动是长上下文成本、延迟、KV cache 三重压力。
>
> **为什么需要压缩**：
> - **成本**：长 prompt 直接吃 token 费
> - **延迟**：prefill 时间 ∝ 上下文长度
> - **质量**：上下文越长越容易"丢中段"（lost-in-the-middle）
> - **KV cache 爆炸**：长上下文每 token 都要缓存 K/V
>
> **主流压缩方法**：
>
> **1. 摘要压缩（最经典）**
> 用小 LLM 把长文本压成摘要：
> - 多轮对话 → 滚动摘要
> - 长文档 → 章节摘要
> - 历史交易 → 阶段汇总
>
> 优点：通用、简单
> 缺点：摘要时丢细节
>
> **2. 关键词 / 实体提取**
> 用 NER 抽核心实体（人名、订单号、金额），只保留这些 + 关系。
>
> 优点：极度压缩
> 缺点：丢上下文语义
>
> **3. 选择性保留（RAG 风格）**
> 把长上下文当"知识库"，对当前 query 做检索，只保留 top-K 相关片段。
>
> 优点：聚焦相关内容
> 缺点：需要 query 已知
>
> **4. Token-level 压缩**
> - **LLMLingua / LongLLMLingua**：用小模型给每个 token 算重要性，丢低重要 token
> - **AutoCompressor**：模型自学习压缩成 soft token
>
> 优点：保留语义
> 缺点：质量受小模型影响
>
> **5. KV Cache 压缩**
> - **H2O**：丢低 attention 权重的 KV
> - **Scissorhands**：相似度去重
> - **MLA**：低秩压缩 KV（DeepSeek-V2）
>
> 优点：推理侧透明优化
> 缺点：质量损失
>
> **6. 上下文分级**
> 分"核心 / 重要 / 参考"三级：
> - 核心放原文（system prompt、当前问题）
> - 重要做摘要
> - 参考存外部库 + 检索调用
>
> **我们跨境支付里的实战**：
>
> **客服多轮对话压缩**
> - 历史超过 5 轮时，前面的对话用 Qwen-7B 滚动摘要
> - 最近 3 轮保留原文
> - 关键实体（订单号、金额）始终保留
>
> **风控调查 Agent 长上下文压缩**
> - 拉了 50+ 笔关联交易，全塞 prompt 会 30k token 起步
> - 用 LLMLingua 做 token-level 压缩（保留率 30%），质量基本不掉
> - 关键交易摘要 + 普通交易"压缩态" 混合
>
> **PPT 生成的章节状态压缩**
> - 已生成章节存到 Redis，主 prompt 只带"章节标题列表"
> - 当前章节生成时按需再 load
>
> **几个工程经验**：
> - 摘要压缩 ROI 最高，从这里开始
> - **关键信息不能丢**——订单号、金额、规则编号必须保留原文
> - 压缩过的内容要标记，方便审计追溯
> - 多轮场景上 hierarchical 压缩（摘要 + 摘要的摘要）
> - **不是所有场景都要压**——简单 query 别上 LLMLingua，反而引入延迟
>
> 上线后我们风控 Agent 平均上下文从 30k 降到 12k，成本 -50%、p95 延迟 -40%、质量基本不掉。
