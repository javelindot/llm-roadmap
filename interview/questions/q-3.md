# Agent 如何创建工具？

## 答案

**一句话**：工具的本质就是"**函数 + 一份能让模型读懂的说明书（JSON Schema）**"，创建路径分协议层、框架层、跨进程层三种。

### 1. 工具的三要素
- **name**：唯一标识，建议动词开头（如 `query_rate`、`freeze_account`）。
- **description**：决定能不能被模型命中——要写清楚"什么时候用、什么时候**不要用**、有什么副作用"。
- **parameters**：JSON Schema 定义入参，参数尽量原子化、避免深嵌套。

### 2. 三种创建路径
| 层级 | 代表方案 | 适用场景 |
|------|---------|---------|
| 协议层 | OpenAI Function Calling / Anthropic Tool Use | 自己写循环、追求轻量 |
| 框架层 | LangChain `@tool` / StructuredTool、LlamaIndex FunctionTool | 已经在用框架，想快速集成 |
| 跨进程 | MCP Server | 工具属于第三方系统、要跨语言/跨服务 |

### 3. 调用回路（ReAct 风格）
```
模型推理 → 输出 tool_use(name, args) → 框架执行函数 → 把 tool_result 回填上下文 → 模型继续推理
```

### 4. 设计原则（实战踩坑总结）
- **单一职责**：一个工具只做一件事，别做"万能查询"。
- **幂等优先**：重试不会重复扣账、重复发短信。
- **错误信息让模型看得懂**：返回 `{"error": "余额不足，最低需要 100 USD"}`，而不是 Python 堆栈。
- **副作用工具加 confirm**：写库、转账、发外部消息这种必须二次确认。
- **参数校验**：模型可能给奇怪入参，schema 校验比业务校验便宜。

### 5. 一句话总结
工具 = 函数 + Schema + 好描述。**写工具一半在写代码，一半在写 description**——description 写不清楚，模型该用不会用、不该用乱用。

## 结合项目谈（口语版 · 约 40 秒）

> 我们做跨境支付，给 Agent 配工具的时候其实分得挺细的。读类的就是查汇率、查 KYC 状态、查交易链路、查商户分级，这些走标准 function calling，返回值规整一点 Agent 自己就能消化。
>
> 写类的我们格外小心——你想嘛，万一 Agent 把别人账户冻了、或者发错对账单出去，那是真出事故。所以像"冻结账户""调起人工审核""触发监管报送"这种，全部包一层 confirm 工具：调用时先返回一个"待确认"的 token，业务侧二次校验通过才真的执行。
>
> 还有一个坑是 description——我们一开始 description 写得太干，"查询商户信息"，Agent 见着商户相关问题就调。后来改成"用于查询**已入网商户**的分级、限额、风控标签；**不要用于**查询入网中或被拒商户"，误调率直接降下来。
>
> 现在内部稳定下来后，我们把高频工具集打包成了一个内部 MCP Server，风控、客服、对账几个 Agent 都能共用。

**🪝 顺嘴带出去的钩子**：
- "工具调多了会发现 description 比代码还难写，我们专门做了一套 description 评测集，可以聊聊。"
- "MCP 接进来之后权限怎么管也是个事儿，我们设计了一套工具准入流程。"
- "对了我们还踩过 Agent 重试导致重复扣款的坑，幂等怎么做能聊聊。"
