# 本地部署龙虾有什么办法可以让它安全去跑？

## 答案

**一句话**：本地部署 Ollama 要安全运行，重点是网络隔离、模型来源、数据脱敏、权限控制、日志审计和资源限制。

### 核心回答
安全措施：
- **网络隔离**：默认只监听本机或内网，不暴露公网。
- **访问控制**：API 前加网关、鉴权、IP 白名单。
- **模型来源**：只拉可信模型，确认 license 和 checksum。
- **数据脱敏**：敏感字段如卡号、证件号、账户号先脱敏。
- **权限隔离**：本地模型只能读必要目录，工具调用要白名单。
- **日志审计**：记录请求、工具调用、异常和输出。
- **资源限制**：限制并发、上下文长度、GPU/CPU 内存。
- **提示注入防护**：RAG 文档和用户输入分区处理。

如果接企业系统，必须把模型服务放到统一权限和审计体系里。

### 1. 本地部署不等于天然安全

本地部署只是解决数据不出外部 API 的问题，但如果服务暴露、权限混乱、日志泄露，仍然会有安全风险。

### 2. 安全运行的关键点

我会重点做：

- 网络只开内网。
- API 加鉴权。
- 输入输出脱敏。
- 工具调用白名单。
- 高危动作审批。
- 日志审计。
- 模型和依赖来源可信。
- 资源隔离，防止被打爆。

### 3. 面试总结

本地部署要把模型服务当成企业内部系统治理，不能只是起一个端口。

### 4. 结合跨境支付项目怎么说（面试口径）

> 跨境支付数据很敏感，所以本地部署不是跑起来就完事。我会先保证 Ollama 服务不暴露公网，只在内网访问；接入前做鉴权和 IP 白名单。
>
> 用户输入里的卡号、证件号、账户号要脱敏；如果 Agent 能调工具，工具必须白名单，不能让模型随便读文件或调资金接口。
>
> 生产环境还要有日志审计，方便追查模型回答和工具调用链路。比如某次风控建议为什么给出、查了哪些系统、用了哪些规则，都应该能回放。

**钩子**：本地部署解决数据不出域，但权限、脱敏和审计仍然必须做。
