# 如何确保模型输出的格式正确？

## 答案

确保模型输出格式正确，应采用“生成时约束 + 生成后校验 + 有限修复 + 失败降级”，不能只在 Prompt 中写一句“请输出 JSON”。

> 定义 Schema → 约束解码 → 结构校验 → 业务语义校验 → 一次修复 → 降级或转人工。

### 1. 定义严格的输出契约

使用 JSON Schema、Pydantic 或 Protobuf 定义：

- 字段类型、必填项和默认值；
- enum、长度、数值范围和正则；
- `additionalProperties: false`；
- 数组长度和嵌套深度；
- 不同意图对应的 discriminated union；
- Schema 版本。

例如付款状态查询：

```json
{
  "intent": "payout_status_query",
  "confidence": 0.93,
  "entities": {
    "order_id": "P20260606001",
    "currency": "USD"
  },
  "route": "payout_query_tool",
  "needs_clarification": false
}
```

`tenant_id、merchant_id、user_id` 等可信身份字段由后端绑定，不能让模型自行生成。

### 2. 使用结构化输出或约束解码

优先使用模型服务提供的：

- JSON Schema structured output；
- Function/Tool Calling；
- JSON mode；
- CFG、正则或 grammar constrained decoding。

这些方法在 token 生成阶段屏蔽非法路径，比事后从 Markdown 中截取 JSON 更可靠。对分类和路由任务降低温度，禁止额外解释文字。

约束解码只能保证语法合法，不能保证值正确。例如 `currency: "USD"` 符合 Schema，但可能与用户问题不一致。

### 3. 程序做结构与语义双重校验

结构校验包括：

- JSON 是否可解析；
- 类型、enum、必填项和长度；
- 是否出现额外字段；
- Schema 版本是否匹配。

业务语义校验包括：

- 订单号是否来自用户输入或授权上下文；
- 订单是否属于当前商户；
- 国家和币种组合是否有效；
- route 是否与 intent 匹配；
- 工具参数是否完整；
- 高风险操作是否需要二次确认或人工审批。

真实订单状态、余额、汇率和权限必须查询权威系统，不能因格式正确就信任模型内容。

### 4. 有限修复与降级

如果只是结构错误，可以把精简后的校验错误返回模型修复一次：

```text
Validation failed:
- route must be one of [...]
- currency must be ISO-4217
Return only the corrected JSON.
```

不能无限重试。连续失败时根据风险：

- 回退规则分类器或模板；
- 返回 `unknown`；
- 追问缺失字段；
- 只展示搜索结果；
- 转人工处理。

涉及付款、退款、冻结或解冻时，即使 JSON 完全合法也不能直接执行，仍需鉴权、幂等、规则校验和审批。

### 5. Prompt、训练和接口设计

- Prompt 中提供字段定义、边界正例和反例；
- 相似 intent 使用 hard negative；
- 让工具参数尽量小而明确，避免一个巨型万能 Schema；
- 工具名称和字段使用稳定版本；
- SFT 时只对目标结构计算 assistant loss；
- 将历史格式 bad case 加入回归集；
- 对 Schema 变更做兼容或显式版本升级。

### 6. 评测与监控

- 一次 Schema 通过率；
- 修复后通过率和平均重试次数；
- 字段级准确率和缺失率；
- intent、route 和实体准确率；
- 工具调用成功率；
- 越权字段和错误资金动作拦截率；
- P95/P99、token 和失败降级率。

格式正确率和任务正确率必须分开统计。一个 100% 合法的 JSON 仍可能表达完全错误的业务决策。
