# 🔥 KV Cache是什么？为什么能极大地提升推理速度？

## 答案

**一句话**：KV Cache 是在自回归推理时，缓存每一层历史 Token 的 Key 和 Value。生成下一个 Token 时只计算新 Token 的 Q/K/V，并让新 Q 读取缓存的历史 K/V，避免重复计算整个前缀。

### 1. 为什么可以缓存

因果注意力中，历史 Token 的表示不会因为后面生成了新 Token 而改变。因此历史 K/V 在第一次算出后可以复用：

```text
Prefill：一次处理完整 Prompt，生成初始 KV Cache
Decode：每生成一个 Token，只追加这一 Token 的 K/V
```

### 2. 为什么能提速

没有 KV Cache 时，第 `t` 步需要重新把前 `t` 个 Token 通过所有 Transformer 层。使用缓存后：

- 历史 Token 的 K/V 投影不再重复计算。
- 新 Token 的 Query 直接与长度为 `t` 的缓存做注意力。
- 单步 Attention 从重新计算完整 `t × t` 矩阵，变为计算新 Query 对 `t` 个历史位置的权重。

因此 Decode 阶段的单步注意力计算可从关于序列长度的平方级降为线性级。整个生成过程仍会随输出长度增长，但重复计算大幅减少。

### 3. Cache 保存什么

典型形状近似为：

```text
[layers, batch, num_kv_heads, sequence_length, head_dim]
```

K 和 V 各一份，所以显存近似为：

```text
2 × layers × batch × seq_len
  × num_kv_heads × head_dim × bytes
```

### 4. 代价与优化

- 上下文和并发越大，KV Cache 占用越高。
- MQA/GQA 通过减少 KV 头数降低缓存。
- PagedAttention 用分页方式减少碎片并支持动态批处理。
- KV Cache 量化可以进一步省显存，但要评估精度影响。
- Prefix Cache 可以让多个请求复用相同系统 Prompt 的缓存。

### 5. 常见误区

KV Cache 主要优化 **Decode**，并不会让长 Prompt 的 Prefill 免费；新 Token 仍要读取全部历史 K/V，所以长上下文下 Decode 仍受显存带宽限制。

**面试总结**：KV Cache 用显存换计算，是大模型自回归推理能够实用化的核心优化之一。
