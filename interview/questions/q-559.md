# LLM的评估

## 答案

LLM 评估应分层进行，不能用一个通用 Benchmark 或一个 LLM-as-Judge 分数代表全部能力。

### 1. 基础能力评估

- 知识与问答
- 数学和逻辑推理
- 代码生成
- 多语言
- 长上下文
- 安全与偏见

公开 Benchmark 可用于横向参考，但要警惕数据污染和与真实业务分布不一致。

### 2. 业务任务评估

建立来自真实场景的 Golden Set，并明确 Rubric：

- 事实正确性
- 完整性
- 指令与格式遵循
- 拒答是否合理
- 业务规则是否正确

分类、抽取等任务优先使用确定性指标；开放生成任务结合人工和 Judge。

### 3. RAG 与 Agent 分层评估

RAG：

- Retrieval Recall@K、MRR、NDCG
- Answer Correctness、Faithfulness、Citation Accuracy

Agent：

- 任务完成率
- 工具选择和参数正确率
- 路径长度、成本、延迟
- 错误恢复和高风险动作误触发率

### 4. LLM-as-Judge

优点是规模大、成本低于人工；风险包括位置偏置、长度偏置、自我偏好和 Judge 与被测模型同源。

应使用：

- 明确评分 Rubric；
- 随机交换答案顺序；
- 多 Judge 或人工抽检；
- 用人工标注集校准 Judge 一致性。

### 5. 线上评估

- A/B 测试
- 用户任务完成率
- 点赞、追问、转人工和投诉
- 延迟、Token 成本和错误率

线上反馈要防止只优化短期点击而损害长期体验。

### 6. 回归机制

所有线上 Bad Case 应进入版本化评测集。每次模型、Prompt、检索或工具变更都要跑回归，并按场景查看分数。

**面试总结**：高质量评估是“离线分层指标 + 人工校准 + 线上业务指标 + 持续回归”的组合。
