# 有哪些PPO算法的调参经验？

## 答案

**一句话**：PPO 调参核心是控制更新幅度和奖励尺度，重点看 KL、clip range、学习率、reward 归一化、advantage 归一化、batch size、value loss 和生成长度。

### 1. 常调参数
- **learning rate**：通常要小，过大容易 KL 飙升、模型崩。
- **clip range**：限制策略更新幅度，太大不稳，太小学不动。
- **KL coefficient**：控制新模型不要偏离 reference model 太远。
- **reward normalization**：奖励尺度要稳定，否则 advantage 波动很大。
- **advantage normalization**：让不同 batch 的优势估计更稳定。
- **GAE 的 gamma/lambda**：控制长期回报和方差偏差平衡。
- **batch size / mini-batch / epoch**：太小噪声大，太大成本高且可能过拟合。
- **value loss coefficient**：critic 学不好会导致 advantage 不准。
- **entropy bonus**：防止模型过早收敛到单一模式。
- **response length**：要监控长度偏置，防止模型靠变长拿高 reward。

### 2. 监控指标

训练时不要只看 reward，还要看：
- KL 是否稳定。
- policy loss / value loss 是否异常。
- reward 是否突然暴涨。
- 生成长度是否变长或变短。
- 拒答率、重复率、格式遵循率是否变差。
- 离线 benchmark 和人工评估是否一致。

### 3. 面试总结

PPO 的调参目标是让模型"慢慢变好"，不是让 reward 一路暴涨。reward 暴涨但 KL 失控，往往是 reward hacking 或模型退化的信号。

## 结合项目谈（口语版 · 面试现场约 55 秒）

> 如果放到跨境支付的对齐场景，我会特别谨慎调 PPO。比如我们训练一个风控解释或商户审核助手，reward model 可能会偏好"看起来很合规、很谨慎"的回答。如果 PPO 推得太猛，模型可能什么都建议转人工，reward 很高，但业务上误杀率也很高，商户体验会变差。
>
> 所以我会重点监控几个指标：第一是 KL，不能让模型偏离 SFT 或 reference 太远；第二是拒绝率和转人工率，不能因为追求安全把正常交易都拦掉；第三是回答长度和模板化程度，防止模型靠写一堆合规套话刷 reward；第四是离线评估和真实工单复核的一致性。
>
> 参数上我会从小 learning rate、小 clip range、合理 KL penalty 开始，先让模型小步更新。支付场景宁愿慢一点，也不能出现线上策略突然变激进的情况。
