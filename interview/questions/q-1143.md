# 意图识别LLM输出格式不好怎么优化

## 答案

意图识别输出格式不稳定，不能只靠 Prompt 里写“请输出 JSON”。应使用：

> 明确标签体系 → JSON Schema/结构化输出 → 约束解码 → 程序校验 → 有限修复 → 失败降级。

### 1. 先把意图体系设计清楚

格式问题有时来自标签定义重叠。需要明确：

- 每个 intent 的定义、正例、反例和边界；
- 是否支持多意图；
- `unknown`、需追问和转人工标签；
- 必填实体和可选实体；
- 哪些字段由模型输出，哪些由后端注入。

例如“查询付款状态”和“付款失败归因”应分开；“退款”还要区分规则咨询、状态查询和发起退款。

### 2. 使用严格 Schema

```json
{
  "intents": [
    {
      "name": "payout_status_query",
      "confidence": 0.92,
      "entities": {
        "order_id": "P2026...",
        "currency": "USD"
      }
    }
  ],
  "needs_clarification": false,
  "clarification_question": null,
  "route": "order_tool"
}
```

Schema 中使用：

- intent 和 route 枚举；
- confidence 范围；
- 订单号、币种和国家格式；
- 必填字段；
- `additionalProperties: false`；
- 多意图数组长度上限；
- 使用 discriminated union 区分不同意图所需字段。

用户、商户、权限和租户 ID 由可信后端绑定，不允许模型生成。

### 3. 使用结构化输出或约束解码

优先使用模型/推理服务提供的 JSON Schema、grammar 或 function calling，让解码阶段只能生成合法结构。温度设置较低，关闭不必要的自由文本。

约束解码保证“语法合法”，不能保证“语义正确”。模型仍可能选择错误意图或填写错误订单号，因此必须继续做业务校验。

### 4. 程序校验和语义校验

使用 Pydantic、JSON Schema Validator 等校验：

- JSON 是否可解析；
- 类型、枚举和必填字段；
- 实体是否来自用户输入或已授权上下文；
- 订单是否属于当前商户；
- route 是否与 intent 对应；
- 高风险意图是否错误进入自动执行。

币种、国家码和错误码可通过字典校验；订单和商户关系需要查询业务系统，不能只做正则。

### 5. 有限修复与降级

若只是格式错误，可以将验证错误返回模型修复一次：

> `currency must be ISO-4217; route must be one of ...`

不能无限重试。连续失败时：

- 回退规则或小分类器；
- 返回 `unknown`；
- 向用户追问；
- 高风险场景转人工。

修复模型只接收必要字段和校验错误，避免再次携带全部敏感上下文。

### 6. Prompt 和样本优化

- 提供少量边界清晰的正反例；
- 加入多意图、缺字段和越权样本；
- 明确禁止输出解释文字和 Markdown；
- 对相似标签加入 hard negative；
- SFT 时确保 assistant loss 只覆盖目标结构；
- 对历史格式 bad case 做持续回归。

### 7. 评测

- JSON/Schema 一次通过率；
- 修复后通过率与平均重试次数；
- Intent Macro-F1、多意图 F1；
- 实体抽取 F1 和实体绑定正确率；
- route 准确率；
- `unknown`、追问和转人工准确率；
- 越权路由和错误资金动作率；
- P95 与 token 成本。

格式通过率不能替代业务准确率。一个 100% 合法的 JSON 仍可能包含错误意图，因此上线门槛必须同时包含结构、语义和安全指标。
