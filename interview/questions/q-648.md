# 4.如果是做生成式召回应该对模型结构怎么修改

## 答案

生成式召回不再只计算 query 与 item 的向量相似度，而是让模型根据用户或查询直接生成候选 item 的标识。因此模型结构、item 表示和解码都需要调整。

### Item 标识与模型输出

不能直接把百万级 item 全部当作普通分类头，参数和更新成本很高。常见做法是为 item 构造可生成的 semantic ID：先根据内容或协同信号聚类，将 item 映射为多级离散 token 序列。模型学习：

$$
P(id_1,id_2,\ldots,id_m\mid query,user\ history)
$$

结构可使用 encoder-decoder：encoder 编码 query、用户历史和上下文，decoder 自回归生成 item ID；也可用 decoder-only 将上下文和目标 ID 拼接训练。

### 训练目标

使用 next-token cross-entropy 训练真实点击或相关 item 的 ID 序列。一个 query 对应多个正样本时，可分别构造样本或使用集合式目标，并结合曝光偏差校正、样本权重和困难负例。为同时学习语义与协同关系，可加入对比学习或辅助排序损失。

### 受约束解码

解码时必须保证生成的是库中有效 item。可用 trie、有限状态约束或层级树，只开放当前前缀下合法的下一个 token，再通过 beam search 生成 top-k 候选。下架和新增 item 需要更新约束索引；semantic ID 设计还要减少碰撞并支持增量扩展。

### 工程权衡

生成式召回能统一多任务、利用序列依赖并生成多样候选，但存在解码延迟、热门偏置、无效 ID、语料更新和新 item 冷启动问题。生产中常与双塔、倒排和规则召回并行，由后续排序融合，而不是一次性替换全部召回链路。

评估除 Recall@K、NDCG 外，还应看有效 ID 率、覆盖率、新颖性、延迟和索引更新成本。
