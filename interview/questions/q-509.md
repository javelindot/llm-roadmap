# 你了解哪些生成式推荐的论文

## 答案

生成式推荐可以理解为：不再只对候选 item 做判别打分，而是把用户行为、item ID 或语义 ID 序列化，直接生成下一个 item 或一组推荐结果。可按技术演进介绍代表论文。

### 1. 统一任务与语言建模

- [P5](https://arxiv.org/abs/2203.13366)：把交互、评论和元数据转成文本，通过统一的 text-to-text 目标处理序列推荐、解释等任务。
- TALLRec、LLaRA 等工作探索指令微调通用 LLM，使其适配推荐任务，优势是语义和少样本能力，挑战是 ID 协同信号与推理成本。

### 2. Semantic ID 与生成式检索

- [TIGER](https://arxiv.org/abs/2305.05065)：将 item 编码成语义 codeword 序列，自回归生成目标 item 的 Semantic ID，兼顾检索和冷启动泛化。
- 后续工作重点研究 tokenizer/codebook、无效 ID 约束、热门偏差、增量 item 和生成延迟。

### 3. 大规模生成式推荐

- [HSTU](https://arxiv.org/abs/2402.17152)：将推荐作为序列转导任务，针对高基数和非平稳行为设计高效架构，并报告规模扩展规律。
- [OneRec](https://arxiv.org/abs/2502.18965)：尝试用生成模型统一召回与排序，进行 session-wise generation，并通过迭代偏好对齐优化列表质量。
- 2025 年后的 OneRec-Think、OpenOneRec 等方向进一步探索推理、通用知识和推荐能力的结合。
- 2026 年的[生产扩展研究](https://arxiv.org/abs/2605.23312)表明，扩大参数并不会让所有任务同等受益，服务延迟、缓存陈旧、训练成本与冷启动仍是关键约束。

### 4. 我的判断

生成式推荐的潜力在于统一建模、迁移和 scaling，但短期内未必完全替代“召回、粗排、精排、重排”级联。工业落地还要解决超大 item 空间的约束解码、在线新鲜度、列表多样性、延迟、可控性和 A/B 归因。

如果迁移到跨境支付，可将相似思想用于下一步操作或产品推荐，但额度、费率、合规资格必须由实时系统约束，生成模型不能产生不存在或用户无资格使用的支付产品。

### 面试总结

可按 P5 的任务统一、TIGER 的 Semantic ID、HSTU 的规模化序列建模、OneRec 的召回排序统一来回答。最后要指出生成式推荐的真正难点是受约束生成、item 新鲜度、延迟成本和线上可验证收益，而不只是把推荐写成语言模型。
