# 现在的大模型如何应用到搜索检索层面？

## 答案

LLM 可以参与搜索链路的查询理解、召回、排序和答案生成，但不应让生成模型替代整个检索系统。更稳妥的是分层架构。

### 查询理解

LLM 可做意图识别、实体抽取、拼写纠正、查询改写、同义词扩展和多轮上下文补全。复杂问题可以拆成多个子查询。改写时要保留原始约束，并设置回退机制，避免模型改变用户意图。

### 召回与排序

- 使用 embedding 模型将 query 和文档编码到向量空间，进行语义召回。
- 与 BM25、规则、知识图谱召回并行，融合多个候选集，避免纯向量检索漏掉精确词、编号和新实体。
- 使用 cross-encoder 或 LLM 对候选文档重排，综合相关性、时效性、权威性和个性化特征。
- LLM 还可离线生成文档摘要、主题标签、潜在查询和训练样本，改善索引及小型排序模型。

由于 LLM 重排成本高，通常只处理第一阶段召回的几十到几百个候选，并用蒸馏、缓存或小模型控制延迟。

### RAG 与搜索答案

检索结果可以作为上下文交给 LLM 合成答案，要求引用来源并将每个关键结论绑定到证据。复杂任务可让 Agent 迭代搜索、阅读和补充查询，但必须设置最大步骤、预算和停止条件。无可靠证据时应返回传统搜索结果或明确表示无法确认，不能用生成内容填补检索空缺。

### 评估

分阶段评估 Recall@K、MRR/nDCG、查询改写成功率、重排延迟，再评估最终答案的正确性、完整性、引用准确率和用户满意度。线上还要监控成本、P95/P99 延迟、幻觉、陈旧信息和权限泄漏。LLM 的最佳角色通常是增强语义理解和结果组织，同时由检索索引提供可验证、可更新的事实基础。
