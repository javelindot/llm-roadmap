# 向量召回，关键词召回，实体召回等不同召回链路召回的结果如何合并，哪一个优先级更高？

## 答案

不同召回链路的原始分数通常不可直接相加：BM25、cosine、实体匹配和图距离不在同一尺度。正确流程是：

> 强过滤 → 各路召回 → 通道内校准 → 去重 → 融合 → 业务约束 → Rerank。

### 1. 先做强过滤

租户、ACL、国家、币种、产品、生效时间等硬条件优先过滤。越权或过期内容不能因为向量分高进入候选。

### 2. 统一候选标识

不同链路可能召回同一文档的不同 chunk。用稳定的 `document_id/chunk_id/entity_id` 去重，保留：

- 各路 rank 和 score；
- 命中的 term/entity；
- 文档权威性和版本；
- 通道来源；
- 召回时间与过滤原因。

### 3. 常见融合方法

**RRF**

```text
RRF(d) = Σ_i 1 / (k + rank_i(d))
```

只依赖排名，不要求不同通道分数校准，鲁棒且易上线。

**归一化分数加权**

```text
score(d) = Σ_i w_i * normalized_score_i(d)
```

可以按 Query 类型动态设置权重，但 min-max、z-score 或 calibration 必须用真实数据验证。

**学习融合**

把 BM25、Dense、实体命中、权威性、时效和 Query 特征输入 LR、GBDT/LambdaMART 或神经排序模型。质量更高，但需要可靠标注和线上维护。

**集合策略**

按通道设置 quota，先保证实体/精确结果，再补语义和探索结果，适合强业务规则。

### 4. 哪一路优先

不存在固定全局优先级，应根据 Query 意图决定：

- 明确错误码、订单号、机构名：精确词项/实体优先；
- 口语、同义和多语言：Dense 权重更高；
- 实时订单与余额：结构化工具是权威源；
- 多跳关系：图召回；
- 无实体的探索问题：多路均衡。

硬事实源的优先级高于模型相关性分数。例如订单 API 的当前状态优先于知识文档中的一般说明。

### 5. Rerank

融合后对 Top-N 使用 Cross-Encoder，并加入：

- Query-Document 语义；
- 实体和否定匹配；
- 国家、币种、产品一致性；
- 权威性、版本和时效；
- 文档完整性；
- 多样性和重复惩罚。

Reranker 超时时可以降级为 RRF 结果，但不能绕过 ACL。

### 6. 如何评测

- 各通道 Recall@K 和独占贡献；
- 融合后 Recall、MRR、NDCG；
- 精确实体 Query 与语义 Query 分桶；
- 越权、过期和冲突文档比例；
- Rerank 前后关键证据保留率；
- 延迟、成本和降级率。

跨境支付中，错误码精确文档与“付款失败”语义文档可以同时保留；最终由当前订单、通道和规则条件确定答案，不能简单规定向量召回永远第一。
