# 哈希表的其他设计方案，除了使用数组+链表

## 答案

除数组+链表的 Separate Chaining 外，哈希表还有多种冲突处理和存储设计。选型取决于负载因子、删除、并发、内存和 cache locality。

### 1. 开放寻址

所有元素直接存放在数组中，冲突后继续探测空槽。

**线性探测**

```text
index_i = (h(key) + i) mod m
```

实现简单、cache locality 好，但容易 primary clustering。

**二次探测**

```text
index_i = h + c1*i + c2*i²
```

减轻连续聚集，但要合理选择容量和参数。

**双重哈希**

```text
index_i = h1(key) + i*h2(key)
```

探测分布更均匀。

删除一般使用 tombstone，否则会截断探测链。

### 2. Robin Hood Hashing

记录元素距理想桶的探测距离。插入时，若新元素“更委屈”，就与探测距离较短的旧元素交换：

```text
steal from the rich, give to the poor
```

可降低探测长度方差，查询尾延迟更稳定。删除可使用 backward-shift。

### 3. Cuckoo Hashing

每个 key 有多个候选位置。冲突时把旧元素踢到它的另一个位置：

```text
h1(key), h2(key)
```

查询通常只检查少数位置，最坏查询近似 O(1)，但插入可能形成循环，需要 rehash 或 stash。

### 4. Hopscotch Hashing

让元素保持在理想桶附近，通过邻域 bitmap 加速查询，兼顾开放寻址的 cache locality 和较高负载。

### 5. Perfect Hashing

对静态 key 集构造无冲突映射：

- 查询最坏 O(1)；
- 适合关键字、编译器表和固定字典；
- 不适合频繁动态插入。

Minimal Perfect Hash 还能将 n 个 key 映射到恰好 n 个位置，但通常不直接保存不存在 key 的判断信息。

### 6. 桶内其他结构

Separate Chaining 的桶不一定是链表，也可以是：

- 动态数组；
- 红黑树；
- B-tree；
- 小型开放寻址表。

Java 8 HashMap 在冲突严重且容量足够时将链表树化。

### 7. 并发设计

- 分段锁；
- 桶级锁；
- CAS/lock-free；
- immutable snapshot；
- RCU；
- 分片哈希表。

### 8. 面试总结

替代方案包括开放寻址、Robin Hood、Cuckoo、Hopscotch 和 Perfect Hashing。开放寻址 cache 友好，Cuckoo 查询稳定，Robin Hood 控制尾部探测，Perfect Hash 适合静态集合；没有一种对所有场景最优。
