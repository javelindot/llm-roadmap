# DPO 的偏好数据集长什么样子

## 答案

**一句话**：DPO 数据集本质就是**三元组 (prompt, chosen, rejected)**——一个 prompt 配一个"好回复"和一个"差回复"，告诉模型该往哪边偏。

### 1. 基本格式

```json
{
  "prompt": "客户问：我的跨境汇款到账没？",
  "chosen": "您好，已查询您的订单 #20260601-XXXX 当前状态为'清算中'，预计 24h 内到账。如有疑问可提供订单号继续查询。",
  "rejected": "请联系客服，谢谢。"
}
```

### 2. 扩展字段（生产环境常见）

```json
{
  "prompt": "...",
  "chosen": "...",
  "rejected": "...",
  "chosen_score": 0.92,
  "rejected_score": 0.31,
  "margin": 0.61,
  "source": "human-annotation",
  "scenario": "汇款查询",
  "language": "zh-CN",
  "annotator_ids": ["a01", "a02", "a03"],
  "confidence": "high"
}
```

### 3. 数据来源（重要！）
| 来源 | 成本 | 质量 | 规模 |
|------|------|------|------|
| **人工标注** | 高 | 高 | 小（千-万） |
| **模型采样 + RM 打分** | 中 | 中 | 大（十万+） |
| **GPT-4 / Claude 当 judge** | 中 | 高 | 大（蒸馏路径） |
| **规则生成** | 低 | 视场景 | 大 |
| **历史数据挖掘** | 低 | 视场景 | 视业务 |

### 4. 常见公开数据集
- **Anthropic HH-RLHF**：通用对话偏好（helpful + harmless）
- **UltraFeedback**：6w 条多模型多维度偏好
- **OpenAssistant**：开源对话偏好
- **PKU-SafeRLHF**：安全偏好
- **stanford-shp**：reddit 投票自然偏好
- **OpenBMB UltraInteract**：推理任务偏好

### 5. 数据质量关键点
- **chosen 和 rejected 长度尽量平衡**：差太多模型会学到"长 = 好"的 length bias
- **同一 prompt 多组对比比单组稳**：增加信号密度
- **覆盖多样性**：场景、风格、长度、难度都要均衡
- **避免 chosen / rejected 来源同一模型**：容易学到风格 shortcut
- **margin 信号**：偏好强度可以加权学

### 6. 数据预处理 checklist
- [ ] 去重（同 prompt 重复）
- [ ] 长度过滤（chosen / rejected 比例 0.5-2.0 区间）
- [ ] 语言一致性
- [ ] 敏感词过滤
- [ ] 格式标准化（去掉额外 markdown / 换行）
- [ ] Tokenizer 验证（确保能 tokenize 不报错）

### 7. 一句话总结
**(prompt, chosen, rejected) 三元组 + 多人标注 + 长度对齐 + 多样性覆盖**——DPO 数据集本身不复杂，但"好坏对得对不对"决定模型对齐效果的上限。

## 结合项目谈（口语版 · 约 40 秒）

> 这块我们跨境支付落地过一次完整的客服 DPO 数据集，分享一下细节。
>
> 我们做的是 **客服回复偏好数据集**，最终 5w 条，分布是 30% 人工标 + 70% GPT-4 judge 蒸馏。
>
> **prompt 来源**：从历史一年客服工单里挖的真实客户问题，覆盖汇款查询、商户准入、退款申诉、KYC 补件、汇率咨询五大类。
>
> **chosen 来源**：高满意度（4-5 星）的历史客服回复 + 我们 SFT 模型生成的 top 5 候选里人工挑出来最好的。
>
> **rejected 来源**：低满意度（1-2 星）的历史回复 + SFT 模型生成的明显差回复（太短、太模板、答非所问、客户标"无效"的）。
>
> 踩过最大的坑是**长度偏置**——我们一开始 chosen 平均长度 180 字、rejected 平均 60 字，训出来模型每条回复都贼长，运营吐槽"啰嗦"。后来我们做了 **length-balanced sampling**——按长度分桶配对，强行让 chosen 和 rejected 长度比例落在 0.7-1.5 之间，砍掉极端值，模型才学到真正的"语义偏好"而不是"长度偏好"。
>
> 还有一个习惯是 **每条样本带 scenario 标签**，方便后续按场景做 ablation——发现哪个场景效果差就定向补数据。
>
> 一句话经验：**DPO 数据集，质量比规模重要，长度平衡比多样性重要**。

**🪝 顺嘴带出去的钩子**：
- "length-balanced sampling 怎么做的我们有具体脚本可以聊。"
- "GPT-4 judge 蒸馏的 prompt 怎么设计能保证一致性可以展开。"
- "DPO 数据集每个场景多少样本够用我们做过 scaling law 实验。"
