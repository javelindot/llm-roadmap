# 什么是大模型 Agent

## 答案

**一句话**：以 LLM 做大脑、能感知环境、能规划任务、能调用工具改变外部状态、还能反思迭代的**自治系统**——本质区别是它能"动手"，不是只能"说话"。

### 1. 和传统聊天机器人的根本区别
| 维度 | Chatbot | Agent |
|------|---------|-------|
| 输出 | 文本 | 文本 + 动作 |
| 能否改变世界 | 不能 | 能（写库、调 API、发邮件、转账） |
| 上下文 | 单轮/多轮对话 | 任务级长程，可分解、可回滚 |
| 评测 | BLEU / 满意度 | 任务完成率、动作成功率 |

### 2. 四大核心能力（Lilian Weng 经典定义）
- **Planning（规划）**：把大目标拆成子任务，常见 ReAct、Plan-and-Execute、Tree of Thought。
- **Memory（记忆）**：短期（上下文窗口）+ 长期（向量库 / KV 存储）。
- **Tool Use（工具使用）**：通过 function calling 扩展能力边界。
- **Reflection（反思）**：对自己的产出做评估、修正，必要时重试。

### 3. 主流范式
- **ReAct**：思考-行动-观察循环，最朴素也最常用。
- **Plan-and-Execute**：先规划完整方案再执行，适合可预期任务。
- **Reflexion**：执行后引入 critic 评估，失败就重试。
- **Multi-Agent**：多个角色协作（Planner / Worker / Critic / 人）。
- **Tree / Graph of Thought**：多路径探索，回溯择优。

### 4. 业界形态
- **代码类**：Claude Code、Cursor、Devin、Aider
- **通用类**：Auto-GPT、BabyAGI、AutoGen、CrewAI
- **任务类**：浏览器 Agent（Browser-Use）、研究 Agent（Manus）

### 5. 工程化要解决的问题
- 上下文管理（压缩、KV 缓存）
- 工具调用稳定性（重试、幂等）
- 长程规划（防止跑偏、防止死循环）
- 成本控制（token 经济学）
- 可观测性（trace、replay、回放）

### 6. 一句话总结
**Agent = LLM × (工具 + 记忆 + 规划 + 反思)**。LLM 是大脑，工具是手脚，记忆是经验，规划和反思决定它是"无头苍蝇"还是"靠谱同事"。

## 结合项目谈（口语版 · 约 40 秒）

> 我自己对 Agent 的理解，就是它和聊天机器人最大的区别是"能动手"。聊天机器人就算回答得再花，最后还是话；Agent 是真能查库、调接口、改状态的。
>
> 我们跨境支付里第一个把它跑通的是风控调查场景。以前一个可疑交易过来，运营得手动拉链路、查关联账户、查商户历史、查黑名单、查同设备 IP，光信息收集就半小时。我们把这套流程做成一个 Agent——它自己会规划"先拉链路再查关联"、按需调工具、查完之后还能反思"证据够不够定性"、不够就再补查、最后生成一份结构化调查报告给人工复核。
>
> 上线之后单工单调查时间从 30 分钟降到 3 分钟左右，运营从"信息搬运工"变成"决策审核员"。但有些坑也踩了——比如 Agent 会"过度调查"，一个小工单查了二十几次接口，后来加了步数和 token 上限才稳。还有就是它会"误以为查完了"，所以反思环节我们专门加了一个 checklist 校验。
>
> 现在风控、客服、对账三个场景都跑着 Agent，整体看下来，Agent 不是来取代人的，是把人从重复信息收集里解放出来做判断。

**🪝 顺嘴带出去的钩子**：
- "Agent 步数失控和 token 失控我们踩过坑，可以聊聊怎么治。"
- "我们风控调查 Agent 怎么设计反思环节防止漏证据，这块可以展开。"
- "Agent 上线之后人机协作流程怎么改可以聊聊，这是个组织问题。"
