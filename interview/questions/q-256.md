# 业务和流程是怎么对应的？

## 答案

**一句话**：业务和流程的对应关系，核心是把真实业务步骤抽象成系统节点、状态和流转条件，每个节点有清晰输入、输出、责任和异常处理。

### 1. 怎么做业务流程映射

我一般会先梳理业务流程：

- 用户从哪里进入。
- 每一步谁负责。
- 需要哪些数据。
- 有哪些判断条件。
- 正常路径和异常路径分别是什么。
- 哪些节点需要人工介入。
- 哪些状态必须持久化和审计。

然后再映射成技术流程，比如 API、状态机、队列任务、Agent 节点或工作流节点。

### 2. 技术上怎么表达

常见方式有：

- 用状态机表达订单、工单、审核流转。
- 用责任链表达多个校验节点。
- 用工作流或 LangGraph 表达 Agent 多步骤流程。
- 用事件驱动表达异步通知和解耦。
- 用数据库状态表记录关键业务节点。

关键是业务状态和系统状态要一一对应，不能系统里只有一个“处理中”掩盖所有业务差异。

### 3. 注意点

- 每个节点要有明确的成功、失败、重试和超时处理。
- 外部系统回调可能延迟或乱序，要做幂等和版本校验。
- 业务状态要能让客服、运营、研发都看懂。
- 高风险节点要有审批和审计。

### 4. 结合跨境支付项目怎么说（面试口径）

> 在跨境支付里，一笔付款的业务流程可以映射成创建订单、参数校验、KYC/AML 校验、风控评估、通道路由、提交通道、通道处理中、银行处理、成功入账或失败退回。每个业务节点在系统里都应该有状态和日志对应。
>
> 比如“付款失败”不能只是一个失败状态，要能区分是参数错误、余额不足、通道拒绝、银行退回、风控拦截还是合规限制。因为客服解释、资金处理和后续补救动作都不一样。
>
> 如果用 Agent 做客服解释，我会把这个业务流程映射成图流程：先识别问题，再查订单状态，再根据状态决定查风控、通道还是知识库，最后输出解释。这样业务流程和技术流程是对齐的，不是模型自己乱猜。

**钩子**：可以引到“支付状态机、异常原因细分、Agent 图流程和业务流程对齐”。
