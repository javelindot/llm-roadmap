# 为什么你要用GRPO？GPRO结果比之前好多少？显存开销多大？训练一个Step需要多久？奖励函数如何设置的，为什么？有没有想过为什么一开始Reward出现大幅度震荡？GRPO是否一定有效，还有什么解决方法？

## 答案

这是一组项目真实性追问。回答时必须给自己的实验配置、基线和日志，不能套用论文数字。若没有亲自做过，应明确说做过方案验证或 Demo，不要虚构显卡、训练时长和提升幅度。

### 1. 为什么选择 GRPO

适合使用 GRPO 的前提：

- 同一个 Prompt 可以采样多个候选；
- 奖励能够可靠比较，例如数学答案、代码测试、工具执行结果；
- PPO 的 Critic 成本较高；
- 需要当前策略在线探索，而不只是模仿偏好数据。

若已有高质量 chosen/rejected 数据且奖励不可验证，DPO 通常更简单稳定；若只有少量金标，可先做 SFT 或 rejection sampling。

### 2. 结果如何汇报

建议现场给表格：

| 项目 | SFT 基线 | GRPO 后 | 变化 |
|------|----------|---------|------|
| 任务成功率 | 实测 | 实测 | 绝对百分点 |
| 工具参数准确率 | 实测 | 实测 | 绝对百分点 |
| 格式合法率 | 实测 | 实测 | 绝对百分点 |
| 通用回归集 | 实测 | 实测 | 是否退化 |
| 平均输出长度 | 实测 | 实测 | 防止靠变长刷分 |

不能只说 reward 上升，因为 reward 可能被模型钻空子。

### 3. 显存和 Step 时间

GRPO 省去 Critic，但仍需：

- 训练策略模型及优化器状态；
- Reference/Old policy 的 log-prob；
- 每个 Prompt 的 `G` 条 rollout；
- 奖励计算和模型权重同步。

Step 时间可以拆成：

```text
T_step
= T_rollout
  + T_reward
  + T_logprob
  + T_backward
  + T_sync
```

长 CoT 中通常 rollout 占主要成本。显存应报告模型大小、精度、LoRA/全参、ZeRO/FSDP、序列长度、group size、micro batch 和 GPU 型号，否则数字没有可比性。

### 4. 奖励函数

跨境支付工具 Agent 可设计：

```text
R
= 1.0 * task_success
 + 0.3 * tool_selection
 + 0.3 * argument_validity
 + 0.2 * evidence_consistency
 - 1.0 * unauthorized_action
 - 0.5 * fabricated_fact
 - 0.1 * redundant_call
```

权重必须通过消融实验确定。资金和合规错误应使用硬门槛或大惩罚，不能被话术质量抵消。

### 5. Reward 初期震荡原因

- 初始策略产生的候选质量差异过大；
- group size 小，均值和标准差噪声大；
- 学习率高、clip 宽、KL 系数低；
- 奖励各分量尺度不一致；
- verifier 有 bug 或超时被错误计分；
- Prompt 难度分布突然变化；
- rollout 与训练权重不同步；
- 超长回答截断，使正确样本被误判；
- 大量组全对/全错，真正有信号的 batch 比例波动。

处理方式是先校验 reward，再调优化器；不要一看到震荡就只降学习率。

### 6. GRPO 是否一定有效

不一定。以下情况不适合：

- 开放式客服无法可靠标量评分；
- 模型尚不会基本格式和工具协议；
- 奖励稀疏且可被投机；
- 采样预算太小；
- 目标主要是语气偏好而非探索。

替代方案包括高质量 SFT、DPO、蒸馏、rejection sampling、Best-of-N、过程监督、搜索加 verifier，以及直接改工作流和工具约束。

**面试总结**：这道题最重要的不是证明 GRPO 强，而是证明你知道何时用、怎么量、为什么震荡，以及何时停止用。
