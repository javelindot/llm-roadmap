# 文档的采集、清洗、处理具体是怎么做的？

## 答案

> 这是一个**文档 ETL pipeline**，分采集、解析、清洗、切块、富化、入库六阶段。
>
> **阶段 1：采集**
>
> 来源：
> - 内部文档系统（飞书 / 钉钉 / WIKI）— API 同步
> - 网盘上传（OSS）— webhook 触发
> - 法规网站爬取（白名单 + 合规）— Scrapy
> - 业务系统导出（数据库 / API）— ETL job
>
> 工程实现：
> ```python
> async def collect_documents():
>     sources = [
>         FeishuConnector(),
>         OSSConnector(),
>         CrawlerConnector(),
>         BusinessAPIConnector(),
>     ]
>     for source in sources:
>         async for doc in source.fetch_new():
>             await ingest_queue.put(doc)
> ```
>
> 关键设计：
> - **增量采集**：每个 source 记录 last_sync_time
> - **去重**：用 doc_hash 跳过已采集
> - **版本管理**：同一文档不同版本都留底
> - **失败重试**：消息队列保证 at-least-once
>
> **阶段 2：解析（按文件类型）**
>
> | 类型 | 工具 | 关键点 |
> |------|------|-------|
> | PDF（文本） | MinerU、Marker | 保留章节结构 |
> | PDF（扫描） | PaddleOCR + LLM 后处理 | 印章、手写 |
> | Word | python-docx | 标题层级 |
> | Excel | openpyxl | 表头 + 多 sheet |
> | Markdown | python-markdown | 结构感知 |
> | HTML | BeautifulSoup + readability | 去广告/导航 |
> | 图片 | Qwen-VL 描述生成 | 转文字（详见 q-179）|
>
> 输出统一格式：
> ```python
> class ParsedDoc(BaseModel):
>     doc_id: str
>     source: str
>     title: str
>     content: str  # Markdown 格式
>     metadata: dict
>     sections: list[Section]
>     images: list[Image]
>     tables: list[Table]
> ```
>
> **阶段 3：清洗**
>
> 五件套：
>
> 1. **去重**：完全相同段落用 hash 去重；近似重复用 minhash
> 2. **去无关内容**：页眉、页脚、版权、广告、签名
> 3. **修复 OCR 错字**：自定义业务词表 + LLM 校对
> 4. **统一编码**：UTF-8、繁简、半全角
> 5. **质量过滤**：规则或 LLM 给每段打分，丢低质量
>
> **阶段 4：切块（详见 q-121）**
>
> 我们的策略：
> - 结构感知（按 Markdown 标题切）
> - 块大小 400 token、overlap 50
> - 元数据带 path、source、language
> - 表格 / 公式不切散
>
> **阶段 5：富化（关键提升点，详见 q-123）**
>
> 给每块加：
> - LLM 生成的摘要
> - "这块能回答的 3-5 个问题"
> - 关键实体（NER 抽取）
> - 元数据标签（业务线、国家、币种）
>
> **阶段 6：入库 + 索引**
>
> - 向量入 Milvus（embed 后）
> - 关键词 / 元数据入 ES
> - 关系入 Neo4j（如果有）
> - 原文存 OSS（追溯用）
>
> **几个工程经验**：
>
> 1. **解析器要兜底**：MinerU 处理不了的文档（小众版面）回落到 PyMuPDF
>
> 2. **清洗规则可配置**：每个文档源对应一组清洗规则
>
> 3. **失败 case 留底**：解析失败的文档存到 dead-letter 队列，人工复盘
>
> 4. **版本化处理**：文档更新后老版本保留 7 天，方便回滚
>
> 5. **流水线监控**：每个阶段的吞吐、失败率、延迟
>
> 6. **大文档分批处理**：100MB+ PDF 拆成 chunks 异步处理
>
> 7. **embedding 模型版本固定**：换 embedding 模型必须重建索引
>
> **完整流水线示意**：
>
> ```python
> async def document_pipeline(doc_meta):
>     # 1. 采集
>     raw = await collector.fetch(doc_meta)
>     # 2. 解析
>     parsed = await parser.parse(raw, type=doc_meta.type)
>     # 3. 清洗
>     cleaned = await cleaner.clean(parsed)
>     # 4. 切块
>     chunks = await chunker.split(cleaned)
>     # 5. 富化
>     enriched_chunks = await enricher.enrich(chunks)
>     # 6. 入库
>     await vector_store.upsert(enriched_chunks)
>     await es.index(enriched_chunks)
>     # 7. 监控
>     emit_metrics(doc_meta, len(enriched_chunks))
> ```
>
> 一句话：**文档处理是 RAG 系统的"数据基础设施"**，采集 + 解析 + 清洗 + 切块 + 富化 + 入库六阶段每一阶段都有专门工具和规则，质量层层把关。
