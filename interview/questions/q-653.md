# 是先召回再精排，还是单次生成？

## 答案

在大规模推荐或检索系统中，通常仍是“召回后排序”，而不是让一个生成模型从全库一次性生成最终列表。是否单次生成取决于物品规模、延迟、更新频率和可控性要求。

### 召回加精排

第一阶段从百万或亿级物品中召回数百到数千候选，强调高召回率和低延迟；第二阶段使用更复杂特征预测相关性或业务目标，再重排多样性和规则。

优点是索引更新快、可并行扩展、每阶段可单独评估，且容易加入库存、权限、地域和运营约束。缺点是级联误差：召回漏掉的物品无法被精排恢复。

### 单次生成

生成式推荐可让模型直接生成 item semantic ID 或整条列表，能建模 item 间依赖和全局列表目标，也能统一多任务。但它面临：

- 大词表或 ID 编码与碰撞问题；
- 自回归解码延迟和 beam search 成本；
- 生成无效、下架或无权限 item；
- 新 item 更新和实时特征接入困难；
- 难以解释与插入强业务约束。

必须通过 trie 或约束解码保证 ID 合法，并处理索引增量更新。

### 推荐方案

生产中更稳妥的是混合架构：传统多路召回保证覆盖，生成式模型作为一路召回、query 改写或重排器；最终统一粗排、精排和约束重排。若物品空间小、目录稳定、任务允许较高延迟，才更适合端到端单次生成。

决策应基于 Recall@K、NDCG、端到端延迟、无效 ID 率、更新成本和线上收益的对照实验，而不是因为生成模型新就替换成熟漏斗。
