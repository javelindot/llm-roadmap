# 核心代码模式，写一下大模型预测 token 时， beam 树如何构建，如何得到最终的结果（ pytorch 简单实现了一下，主要是还是讲明白）

## 答案

Beam Search 每一步保留累计分数最高的 `B` 条部分序列。所有保留序列构成宽度受限的搜索树，遇到 EOS 的序列进入完成集合，最终按长度归一化后的累计 log-prob 选择结果。

### 1. 核心过程

假设 beam size 为 `B`：

1. 根节点是输入 prompt，分数为 0；
2. 对每条未完成 beam 计算下一个 token 的 log-prob；
3. 扩展候选并累加分数；
4. 从所有候选中保留全局 top `B`；
5. 重复直到全部结束或达到最大长度；
6. 用长度惩罚后得分选择最优完成序列。

必须使用 log-prob 相加，避免概率连乘下溢。若每条 beam 各取 top-B，之后仍要在全部 `B*B` 个候选中做全局 top-B。

### 2. PyTorch 简化实现

```python
import torch
import torch.nn.functional as F


@torch.no_grad()
def beam_search(
    model,
    input_ids,
    beam_size=4,
    max_new_tokens=32,
    eos_token_id=2,
    length_penalty=1.0,
):
    # 简化为 batch size = 1；生产实现应批量计算并使用 KV Cache。
    beams = [(input_ids, 0.0, False)]
    prompt_len = input_ids.size(1)

    for _ in range(max_new_tokens):
        candidates = []

        for tokens, score, finished in beams:
            if finished:
                candidates.append((tokens, score, True))
                continue

            logits = model(input_ids=tokens).logits[:, -1, :]
            log_probs = F.log_softmax(logits, dim=-1)
            values, indices = torch.topk(log_probs, beam_size, dim=-1)

            for value, token_id in zip(values[0], indices[0]):
                next_token = token_id.view(1, 1)
                next_tokens = torch.cat([tokens, next_token], dim=1)
                candidates.append(
                    (
                        next_tokens,
                        score + value.item(),
                        token_id.item() == eos_token_id,
                    )
                )

        def normalized_score(item):
            tokens, score, _ = item
            generated_len = max(tokens.size(1) - prompt_len, 1)
            return score / (generated_len ** length_penalty)

        beams = sorted(
            candidates, key=normalized_score, reverse=True
        )[:beam_size]

        if all(finished for _, _, finished in beams):
            break

    return max(beams, key=normalized_score)[0]
```

### 3. Beam 树示例

```text
<root>
├─ A (-0.2)
│  ├─ C (-0.5)
│  └─ D (-1.3)
└─ B (-0.4)
   ├─ E (-0.6)
   └─ F (-0.9)
```

若 `beam_size=2`，下一层是在 `AC、AD、BE、BF` 中全局保留累计分数最高的两条，而不是每个父节点保留两条。

### 4. 工程优化与局限

- 对所有 beam 做 batched forward；
- 使用 KV Cache，并在 beam 重排时同步重排缓存；
- EOS beam 不再扩展；
- 可加 no-repeat n-gram、最小长度和词表约束；
- 维护 `length_penalty`，否则偏好短序列；
- beam 越大，时间和 KV Cache 近似线性增加；
- 开放式生成中 Beam Search 容易低多样性，常改用 top-k/top-p；机器翻译、受约束生成更适合 beam。

### 5. 面试总结

Beam Search 的关键是“累计 log-prob、全局 top-B、EOS 管理、长度惩罚和 KV Cache 重排”。代码不必复杂，但必须讲清 beam 树如何扩展和剪枝。
