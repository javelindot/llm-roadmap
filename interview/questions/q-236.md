# 保护节点做了什么？怎么应对回退等异常情况？

## 答案

**一句话**：保护节点就是在 Agent 或业务流程里的关键风险位置做校验、兜底和拦截，防止模型误判、工具误调、状态回退或异常数据影响核心业务。

### 1. 保护节点做什么

常见保护节点包括：

- 权限校验：用户是否能查看或操作这笔数据。
- 参数校验：金额、币种、订单号、国家是否合法。
- 状态校验：当前状态是否允许进入下一步。
- 幂等校验：防止重复执行。
- 风险校验：高风险动作是否需要人工审批。
- 输出校验：模型答案是否包含敏感信息或错误承诺。
- 异常兜底：工具失败、超时、无数据时如何返回。

### 2. 怎么应对回退和异常

回退通常发生在流程执行失败、工具调用异常、状态不一致或用户补充信息后。

处理方式有：

- 用状态机限制可流转路径，不能随意回退。
- 每个节点记录输入、输出和状态版本。
- 失败时回到最近的安全节点，而不是从中间乱跳。
- 对外部回调做幂等和版本校验。
- 对模型多步调用设置最大步数和失败次数。
- 高风险异常直接转人工。

### 3. Agent 场景下的保护

Agent 可能出现工具选错、参数幻觉、重复调用等问题，所以保护节点要放在模型和真实业务工具之间。模型提出调用意图，保护节点校验通过后才执行。

### 4. 结合跨境支付项目怎么说（面试口径）

> 在跨境支付项目里，保护节点非常重要，因为很多动作都涉及资金和合规。比如 Agent 判断用户想“重新发起付款”，它不能直接调用付款接口，必须先过保护节点：校验商户权限、订单状态、金额币种、是否重复提交、是否命中风控、是否需要人工确认。
>
> 对回退场景也一样。比如一笔付款已经进入“通道处理中”，不能因为用户重新问了一次，系统就把状态回退到“待提交”。状态流转要有状态机和版本号，旧回调、重复回调都不能覆盖新状态。
>
> 在客服 Agent 里，保护节点还会检查输出。比如风控拦截时，可以告诉用户“需要补充材料或人工审核”，但不能把具体风控规则和阈值说出去。

**钩子**：可以引到“模型和真实工具之间必须有保护层、支付状态不可随意回退、敏感输出过滤”。
