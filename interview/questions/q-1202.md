# 你在项目中做过哪些相关性优化？有没有量化结果？

## 答案

相关性优化应按“查询理解、召回、重排、上下文和生成”分层，否则端到端指标变化后无法定位收益来源。

### 1. 查询理解

- 多轮指代消解和 Query Rewrite；
- 支付术语归一化，如 payout、付款、出款；
- 提取国家、币种、产品、通道、错误码和版本；
- 复杂问题拆成知识查询与实时订单查询；
- 无法判断时追问，不强行检索。

评估意图 Macro-F1、实体 F1、改写正确率和路由准确率。

### 2. 索引与切分

- 按标题、规则项、错误码和表格语义切分；
- 使用父子块，小块召回、父块补上下文；
- 为 chunk 增加标题路径和短语境；
- 携带国家、币种、产品、生效时间、版本和 ACL；
- 文档更新时增量删除旧版本。

指标包括关键证据覆盖率、碎片率、重复率和过期文档进入率。

### 3. 混合召回

- BM25 保证错误码、国家码和产品名精确命中；
- Dense Retrieval 覆盖口语、同义和多语言表达；
- FAQ、结构化数据和图检索作为补充通道；
- 使用 RRF 或校准分数融合；
- 权限、生效时间和租户过滤尽量前置。

召回层报告 Recall@K、MRR、各通道独占贡献和延迟。

### 4. Rerank 与证据过滤

- Cross-Encoder 处理细粒度语义；
- 使用同国家不同币种、同产品旧版本等 hard negative；
- 去除重复、越权、过期和冲突文档；
- 检查否定词、数值、版本和适用范围；
- 证据不足时追问或转人工。

报告 NDCG@K、Precision@K、关键证据保留率和错误规则进入 Top-K 的比例。

### 5. 生成相关性

- 上下文按“实时事实、规则证据、历史会话”分区；
- 限制模型只能基于证据回答；
- 输出状态、依据、建议和引用；
- Claim-level 校验引用是否支持结论；
- 与工具数据冲突时以权威业务系统为准。

报告 Answer Relevance、Completeness、Faithfulness、引用正确率和人工改写率。

### 6. 量化结果的归因口径

量化结果必须来自同一冻结测试集或互斥 A/B，并记录模型、索引和数据版本。完整报告至少包含：

```text
Recall@K：baseline → new
NDCG@K：baseline → new
Faithfulness：baseline → new
引用正确率：baseline → new
人工改写率：baseline → new
P95 与单次成本：baseline → new
```

通过逐项消融确认收益来自 Query Rewrite、Hybrid Retrieval、Reranker 还是文档治理。没有可追溯实验记录时不应虚构百分比；可以如实说明已完成的方法和评测口径，但尚未形成最终线上 A/B 结论。

### 7. 跨境支付的守护指标

即使相关性提高，也必须保证：

- 跨租户泄漏为 0；
- 过期费率和规则不进入回答；
- 不错误承诺到账、退款和审批结果；
- 高风险问题正确转人工；
- P95、超时和成本满足 SLA。

相关性优化的最终价值是提高一次解决率并降低错误回答，而不是单独追求向量相似度。
