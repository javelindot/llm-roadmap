# 问部署优化有什么经验（面试官希望设计算子内容，但我不会）

## 答案

没有亲自开发 CUDA/Triton 算子时，不应把“使用 FlashAttention 或 TensorRT-LLM”表述为“写过算子”。可以准确说明部署优化经验，并展示对算子瓶颈和优化路径的理解。

### 1. 先说明能力边界

可以明确：

> 我的主要经验在模型服务、量化、批处理、KV Cache、RAG 链路和性能分析，没有独立实现过生产级 CUDA Kernel；算子层做过调用、profiling、配置和结果验证。

随后讲清已能完成的工作，而不是停在“不会”。

### 2. 如何定位算子瓶颈

使用 PyTorch Profiler、Nsight Systems/Compute 或引擎 trace，检查：

- GEMM/attention/normalization 占比；
- GPU idle、Kernel launch gap；
- HBM 带宽与 Tensor Core 利用率；
- Host-Device copy；
- NCCL 通信；
- shape 是否过小或不规则；
- prefill 与 decode 的瓶颈差异。

Prefill 通常更 compute-bound，decode 更容易 memory-bound。不同阶段不能用同一优化结论。

### 3. 常见算子级优化

**算子融合**

- RMSNorm + residual；
- bias + activation；
- QKV projection；
- dequant + GEMM；
- sampling/logits processing。

减少中间张量和 HBM 读写，也减少 Kernel launch。

**高效 Attention**

- FlashAttention 的 tiling 和 online softmax；
- PagedAttention 管理非连续 KV block；
- GQA/MQA 减少 KV 读取；
- fused RoPE、KV write 和 attention；
- 长序列使用 chunked prefill。

**GEMM**

- 选择合适 layout、tile 和 Tensor Core 精度；
- 合并小矩阵为 grouped/batched GEMM；
- 保证维度和对齐适合 Kernel；
- 控制转置、contiguous copy 和 padding。

**运行时**

- CUDA Graph 减少 CPU launch overhead；
- pinned memory 与异步 copy；
- stream 并行；
- 预分配 workspace；
- 避免频繁动态 shape 和内存申请。

### 4. 如果要求设计一个融合算子

以 `RMSNorm + residual` 为例：

1. 输入按行分块；
2. 每个 block 计算平方和归约；
3. 得到 reciprocal RMS；
4. 在寄存器中完成归一化、scale 和 residual；
5. 一次写回输出；
6. 处理尾部、对齐和混合精度；
7. 与 PyTorch reference 做数值比对；
8. 按 shape、dtype 和 batch benchmark。

设计目标不是少写几行，而是减少全局内存读写和 launch；还要处理数值稳定、occupancy、寄存器压力与不同 shape。

### 5. 验证标准

- correctness：不同 shape/dtype、极值、随机输入；
- 数值误差：绝对/相对误差；
- 性能：warmup 后 P50/P95、吞吐；
- roofline：compute 还是 bandwidth bound；
- 端到端收益：单算子快不代表整条链路快；
- fallback：不支持的硬件或 shape 走稳定实现。

跨境支付生产环境更看重稳定、审计和回滚。算子优化只有在 profiler 证明它是瓶颈、且业务输出回归通过时才进入上线流程。
