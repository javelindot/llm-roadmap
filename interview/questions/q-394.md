# 工具白名单，每个子Agent单独绑定工具；

## 答案

**一句话**：工具白名单的核心是每个 SubAgent 只能看到和执行自己职责范围内的工具，避免模型越权调用、误操作或扩大风险面。

### 1. 为什么要工具白名单

如果所有工具都暴露给所有 Agent，会有几个风险：

- 模型可能选错工具。
- 查询类任务误调用写操作。
- 子 Agent 越权访问敏感数据。
- 工具描述太多影响模型选择。
- 出问题后责任边界不清楚。

所以要按角色绑定最小工具集。

### 2. 怎么设计

可以按 Agent 职责划分：

- 客服 Agent：查订单、查知识库、查工单。
- 风控 Agent：查风险标签、设备、规则命中大类。
- 合规 Agent：查国家限制、KYC/KYB 材料要求、制裁名单结果。
- 通道 Agent：查通道状态、错误码、到账时效。
- 运营 Agent：查统计报表和指标。

每个 Agent 只绑定自己的工具，并且工具内部还要做权限校验。

### 3. 工程实现

- Tool Registry 里给工具打标签和权限。
- Agent 初始化时只注入允许的工具定义。
- 调用前做二次校验。
- 写操作需要人工确认。
- 记录工具调用日志和 trace。

### 4. 结合跨境支付项目怎么说（面试口径）

> 在跨境支付项目里，我会非常严格地做工具白名单。比如客服 Agent 可以查询订单状态和知识库，但不能调用放款、退款、解冻这种资金操作；风控 SubAgent 可以读取风险命中大类，但不能修改风控规则；通道 SubAgent 可以查通道回执和错误码，但不能直接切换通道路由。
>
> 这样做有两个好处：第一是安全，模型就算判断错，也没有机会调用高风险工具；第二是效果更稳定，因为每个 SubAgent 看到的工具少，选择空间更清晰。
>
> 我还会在工具执行层做权限校验，因为不能只相信模型侧白名单。即使模型发起了调用，后端也要检查用户、角色、商户和订单是否匹配。

**钩子**：可以引到“最小权限原则、工具执行层二次校验、资金操作不能直接给模型”。
