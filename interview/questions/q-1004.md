# 项目:你在项目里有没有做过 RAG 里的"召回-过滤-生成"三段式 pipeline?能不能细讲一下?

## 答案

生产中的 RAG 可以拆成“召回、过滤、生成”三段，但在召回前还需要查询理解和权限控制，在生成后还需要验证、监控与反馈闭环。完整链路是：

> 查询理解与鉴权 → 多路召回 → 过滤和重排 → 受约束生成 → 结果校验与反馈。

### 1. 召回前：先判断应该查知识还是查实时系统

跨境支付问题不能全部交给知识库：

- 国家和币种支持范围、KYC/KYB 材料、结算周期、拒付规则、错误码说明等相对稳定的知识走 RAG；
- 订单状态、账户余额、实时汇率、通道回执、清算批次和风控审核状态走业务工具；
- 同时包含规则和实时状态的问题，采用 RAG 与工具查询并行或串行编排。

例如“这笔 USD 付款为什么没到账”，需要先从会话中解析商户、订单号和币种，再查订单、通道回执与清算状态，最后检索对应通道的到账规则。仅召回一篇“付款时效”文档无法回答这笔交易的真实状态。

查询理解阶段会完成：

- 多轮指代消解，把“那退款呢”改写为包含订单和业务对象的完整问题；
- 意图识别与任务拆分，区分收款、付款、换汇、退款、拒付和商户准入；
- 支付术语归一化，如 `payout、付款、出款`，`chargeback、拒付`；
- 提取国家、币种、通道、产品、错误码和文档版本等过滤条件；
- 校验用户、角色和商户的数据访问范围，避免跨租户检索。

### 2. 召回：优先保证正确证据进入候选集

索引侧按业务语义切分文档，不只按固定 token 截断。一条规则的适用国家、币种、限制条件和例外条款应保持完整；费率表、错误码表和流程图作为独立结构化单元处理。每个 chunk 携带：

> `tenant、business_line、country、currency、channel、document_type、effective_time、version、visibility`

在线召回采用多路策略：

1. **BM25/全文召回**：命中 SWIFT、SEPA、PIX、币种、国家码、错误码和产品名等精确术语。
2. **Dense Retrieval**：覆盖“款没到”“银行还没回”等口语化和跨语言表达。
3. **FAQ/历史案例召回**：处理高频标准问题，但案例必须脱敏并标明适用范围。
4. **图检索或结构化查询**：只在商户、账户、交易、通道之间存在多跳关系时使用。

权限和强业务条件尽量在检索阶段以前置 filter 执行。各路候选先去重，再使用 RRF 或经过校准的融合分数合并，保留 `source、chunk_id、version、recall_score` 供后续追踪。

### 3. 过滤与重排：从“相关”筛到“可用于回答”

召回结果相关，不代表可以直接进入模型。过滤层主要完成六类检查：

- **权限过滤**：删除当前角色和商户不可见的内部风控、通道或客户资料；
- **时效过滤**：只保留当前生效版本，过期费率和旧通道规则不能参与回答；
- **元数据一致性**：国家、币种、产品、通道必须与问题一致；
- **去重与冲突检测**：合并重复 chunk，识别不同版本或不同地区规则之间的冲突；
- **相关性重排**：使用 cross-encoder reranker 对 query 与候选进行精排；
- **证据充分性判断**：关键条件缺失时先追问或转人工，不强行生成结论。

重排不能只看语义相似度。支付规则中“支持 USD”与“不支持 USD”可能在 embedding 空间非常接近，因此还需要否定词、适用范围、规则版本和结构化字段校验。

### 4. 生成：答案必须被证据和业务状态约束

生成阶段将知识证据和工具结果分区放入上下文：

- 工具返回的订单、通道和账务状态作为当前事实；
- RAG 文档用于解释规则、原因和处理建议；
- 每条证据带来源、版本和生效时间；
- Prompt 明确禁止使用上下文之外的信息补全金额、状态、时效或风控原因。

输出采用结构化模板：

1. 当前可确认的状态；
2. 依据的业务数据和规则；
3. 建议的下一步；
4. 引用来源；
5. 是否需要转人工。

低置信度、证据冲突、涉及投诉或无法公开的风控原因时，系统返回受控说明并转人工。退款、重新付款、冻结和解冻等资金动作不由生成模型直接执行。

### 5. 可观测性、评测和降级

每次请求记录完整 trace：查询改写、路由结果、召回列表、过滤原因、最终证据、模型版本、工具调用和转人工原因。这样才能定位问题发生在召回、过滤还是生成阶段。

离线评测按模块拆开：

- 召回：Recall@K、MRR、正确 chunk 覆盖率；
- 重排：NDCG、Top-K precision、关键证据保留率；
- 生成：faithfulness、答案完整性、引用正确率、错误承诺率；
- 业务：问题解决率、人工改写率、转人工率、用户追问率；
- 系统：P95/P99、超时率、单次成本和降级率。

LLM judge 可用于扩大评测规模，但需要用支付业务专家标注的黄金集校准，不能单独作为上线依据。2026 年的 RAGVUE 等工作开始强调 claim-level faithfulness 和可诊断评测，方向是正确的：总分只能说明变好或变差，分阶段指标才能说明应该修哪一层。

工程降级包括：

- 向量服务失败时退回 BM25 和 FAQ；
- reranker 超时时使用融合召回排序；
- 模型超时时返回已确认的结构化状态或转人工；
- 工具与文档冲突时以权威业务系统为准，并记录知识更新任务。
