# Skills
## 引言

随着大语言模型技术的快速发展，单一工具调用已无法满足日益复杂的业务需求。传统的 Function Call 机制虽然解决了模型调用外部工具的基础问题，但在面对需要多步骤协调、复杂流程编排的任务时显得力不从心。**Skills 技术应运而生**，作为一种基于 FunctionCall 和 MCP 技术之上的高阶能力封装方案，它实现了大模型从 "会用工具" 到 "会完成复杂任务" 的核心升级。

Skills 技术的核心价值在于通过**模块化封装**将特定任务的方法论、执行逻辑与资源整合为可复用的单元，使 AI 能够按照预设流程稳定执行复杂任务。它通过将复杂任务分解为标准化的子步骤，结合动态上下文管理和专属规则约束，大幅提升了任务执行的可靠性和效率。研究显示，采用 Skills 技术的企业平均运营效率提升 35%，人力成本降低 40%，决策速度提升 60%。

本文将从技术架构、实现机制、执行流程和应用案例四个维度，全面剖析 Skills 技术的核心原理和实践价值，为大模型应用开发提供系统性的技术指导。

## 一、技术基础：FunctionCall 与 MCP 的底层支撑

### 1.1 FunctionCall 技术核心机制

FunctionCall 是大模型调用外部工具的**底层技术实现**，其本质是让模型能够主动发起函数调用。这项技术的核心创新在于将大模型从 "纯文本生成" 升级为 "结构化决策 + 格式生成"，通过预训练范式、Prompt 约束、结构化输出、执行链路闭环，实现模型感知外部工具、生成合规调用指令、交由外部系统执行、再把执行结果带回模型续写回答的全流程。

FunctionCall 的技术架构包含四个核心组件：**工具注册、模型能力、格式约束和结果解析**。在工具注册阶段，开发者需要向大模型注册可调用的函数列表，并提供标准化的元信息，包括函数功能描述（让模型判断何时调用该函数）和参数的结构化定义（包括参数名、类型、是否必填、参数描述）。

技术实现层面，FunctionCall 的工作原理基于 "**应用发起请求→模型决策→函数执行→结果反馈→生成最终回复**" 的闭环流程。当用户发起请求时，大模型根据用户需求和工具描述，判断是否需要调用工具。如果需要，模型会生成结构化的工具调用请求，其关键特征是返回结构化的 JSON 格式而非自然语言。



```
{

 "id": "chatcmpl-abc123",

 "object": "chat.completion",

 "choices": \[

   {

     "index": 0,

     "message": {

       "role": "assistant",

       "content": null,

       "tool\_calls": \[

         {

           "id": "call\_abc123",

           "type": "function",

           "function": {

             "name": "get\_weather",

             "arguments": "{\\"city\\": \\"北京\\", \\"date\\": \\"today\\"}"

           }

         }

       ]

     },

     "finish\_reason": "tool\_calls"

   }

 ]

}
```

这个 JSON 结构的核心字段包括：**tool\_calls**（当需要调用工具时，这里包含工具调用的信息）、**function.name**（要调用的工具名称）、**function.arguments**（工具的参数，JSON 字符串格式）。系统解析 LLM 生成的工具调用请求，执行对应的工具函数（如调用天气 API），获取工具执行结果后，再将结果返回给 LLM，让模型根据结果决定下一步行动（继续调用工具，或者生成最终回答）。

### 1.2 MCP 协议的标准化价值

MCP（**Model Context Protocol，模型上下文协议**）是由 Anthropic 公司在 2024 年底提出并开源的开放标准，现已捐赠给 Linux 基金会。MCP 致力于提供一个标准化的接口规范，让 Agent 能更方便、更统一地接入和使用各种工具，无论是 RAG 功能还是其他 Function Call 实现的工具。

MCP 的技术定位是为 LLM 应用提供**标准化接口**以连接和交互外部数据源和工具，其核心仍然基于 Function Call 技术。MCP 的设计理念是成为 "AI 应用的 USB-C 接口"，通过统一的协议规范让 AI 模型能够标准化地访问各种数据源和工具。

技术架构层面，MCP 采用**三层模型架构**：Host（用户直接交互的 AI 应用，负责接收任务并选择匹配的 MCP Client）、Client（Host 内置的连接器，与 Server 建立 1:1 通信，处理协议转换与异常重试）、Server（外部服务的适配层，将私有 API 抽象为 MCP 标准能力）。

MCP 定义了三种核心原语确保 AI 对外部能力的调用逻辑统一：



* **Tools（工具）**：AI 可主动调用的可执行函数，需明确参数与返回格式

* **Resources（资源）**：AI 可读取的静态 / 动态数据源，通过 URI 标识

* **Prompts（提示模板）**：用户触发的预定义指令模板，用于标准化交互

MCP 的传输机制支持三种主流方式：**stdio**（Client 以子进程启动 Server，通过标准输入输出通信，适用于本地工具）、**Streamable HTTP + SSE**（Client 发起 HTTP GET 建立 SSE 流，Server 推送消息，Client 通过 POST 响应，适用于跨网络轻量交互）、**WebSocket**（双向全双工通信，适用于高实时性场景）。

### 1.3 Skills 技术的定位与创新

在 FunctionCall 和 MCP 的技术基础之上，**Skills 技术**应运而生。Skills 是封装了特定外部能力的功能模块，可以理解为 "大模型的插件"。从本质上看，Function Calling 是 "功能"，MCP 是 "操作系统级协议"，而 Skills 则是基于这两者的上层应用。

Skills 的核心定义是将特定任务的**方法论、执行逻辑与资源封装成模块化单元**，使 AI 能够按照预设流程执行复杂任务。一个 Agent 可以包含多个 Skills，通过组合使用这些 Skills 来完成复杂任务。Skills 的本质是 "可复用、可组合的 AI 任务执行单元"，其架构设计围绕 "如何让 AI 高效、可靠地完成特定领域任务" 展开。

与传统技术相比，Skills 技术的核心创新在于**从工具调用升级为任务级意图匹配**。普通 Tool Call 的执行链路是：用户提问→模型选择 Tool→生成调用 JSON→外部执行；而 Skill 模式的核心变化，是将路由对象从 "细粒度工具" 升级为 "粗粒度任务解决方案"。

技术实现层面，Skills 的核心机制是通过一个固化的函数和参数，让模型去查找和加载固定的 skills 文档。Skills 完全依赖于 Function Calling 这个基础能力，它只是在 Function Calling 之上的一个巧妙应用：把 "加载文档" 这个操作封装成一个函数，然后让模型在需要时自动调用。

## 二、技术实现逻辑：从概念到代码的完整解析

### 2.1 Skills 核心架构设计

Skills 技术的核心架构采用**分层协同设计**，包含三个关键层次。顶层是指令（Instructions），以自然语言或结构化提示形式，清晰界定任务目标、约束条件与输出规范；中层是可执行代码（Code），将抽象意图转化为模型可调用的操作逻辑，如函数调用、API 路由或格式化模板；底层虽未显式编码，却通过接口契约隐含对执行环境的适配要求。

一个成熟的 Skill 会把复杂任务拆解为**固定、可复用的子步骤**，例如报告解读 Skill 的提取→解读→分析→建议，行程规划 Skill 的查地点→查天气→排日程。当框架接收到模型的 Skill 激活指令后，会自动按 subtasks 定义的顺序，引导模型分步执行，中间按需调度底层 Tool，最终汇总所有步骤结果。

Skills 采用**渐进式加载机制**有效管理 AI Agent 的上下文。激活技能时，框架首先加载技能指令，然后注入上下文，实现 "激活技能并加载到上下文" 的过程。具体实现代码如下：



```
def activate\_skill(skill\_id, context):

   """激活技能并加载到上下文"""

   # 加载技能指令

   instructions = skill\_loader.load\_instructions(skill\_id)

  

   # 注入上下文

   context.inject\_skill\_knowledge(instructions)

  

   # 加载相关资源

   resources = skill\_loader.load\_resources(skill\_id)

   for resource in resources:

       context.inject\_resource(resource)

  

   # 加载专属工具

   tools = skill\_loader.load\_tools(skill\_id)

   context.update\_available\_tools(tools)

  

   return context
```

这种设计的核心价值在于用预定义流程约束模型行为，减少无效决策与错误调用，同时让任务执行过程可监控、可回溯、可优化。

### 2.2 任务分解与工具链编排机制

Skills 技术的任务分解机制基于**Subtasks 的标准化流程执行**。Subtasks 是 Skill 的内部执行蓝图，也是 Skill 区别于零散 Tool 的核心标志。一个完整的 Skill 配置包含 name（技能名称）、description（适用场景描述）、subtasks（子任务列表）和 trigger\_scenarios（触发场景）等关键字段：



```
{

 "name": "解读血液检验报告",

 "description": "用户提供血常规报告时，完成异常识别、临床解释、综合分析与健康建议",

 "subtasks": \[

   "数据预处理与异常指标提取",

   "异常指标解读",

   "综合分析",

   "生成健康建议"

 ],

 "trigger\_scenarios": \[

   "用户上传体检报告",

   "用户询问指标异常原因",

   "用户请求解读化验单数据"

 ]

}
```

工具链编排机制支持**串行、条件判断、递归等多种方式**串联不同 Skill 有序执行。在真正的工程实践中，不会让单一 Skill 承担全部任务，而是遵循 "单一职责原则"，把复杂任务拆分成多个子任务，再通过串行、条件判断、递归等方式，串联不同 Skill 有序执行。

编排层的核心原理价值在于：**用预定义流程约束模型行为**，减少无效决策与错误调用，同时让任务执行过程可监控、可回溯、可优化。这种机制通过提升路由粒度，降低模型的决策复杂度，同时让任务执行更贴合业务逻辑。

### 2.3 状态管理与上下文传递机制

Skills 技术的状态管理机制依赖于**会话状态（session.state）**，这是一个包含键值对的集合（字典或 Map），用于存储智能体有效执行需要用到的信息，比如记录用户偏好、跟踪多轮流程中的步骤、收集信息等。

上下文传递机制采用**三级知识系统设计**，模型会按照加载顺序，依次触发披露加载元数据、完整指令，并根据 SKILL.md 指令中提到的对捆绑资源的依赖，进一步调用链接文件资源。这种渐进式披露机制的设计理念类似于一本组织良好的手册：目录告诉你章节内容，真正需要时再去读对应章节。

在上下文管理方面，Skills 模式的优势在于**信息互通**。所有能力说明都铺在同一张桌上，主 Agent 能看到所有中间结果，推理过程连贯。Skills 的做法是：把写 PPT 的能力说明、工具调用方式、注意事项，全都塞进主 Agent 的上下文中。主 Agent 通过上下文学会了这项技能，它可以自己来写 PPT。区别在上下文管理，上下文就是 AI 的记忆。

### 2.4 错误处理与重试机制

Skills 技术的错误处理机制采用**Agent 的核心决策逻辑**，让根动态变化的环境信息选择执行具体的行动或者对结果做出判断，并响应环境，通过多轮迭代重复执行上述步骤，直到完成目标。

工程实现上，整个系统可以拆分出**四大核心模块：推理、记忆、工具、行动**。在架构层面，这体现为多轮的 BasicReflection，包含多个 Generator 和 Reflector。

具体的错误处理策略包括：



* **自动重试机制**：当工具调用失败时，系统会根据预设的重试策略自动重试

* **错误分类处理**：根据错误类型（网络错误、参数错误、权限错误等）采取不同的处理方式

* **回滚机制**：当某个子任务失败时，系统可以回滚到之前的状态，重新执行或选择替代路径

* **人工介入接口**：对于无法自动处理的错误，提供人工介入的接口

### 2.5 与 FunctionCall/MCP 的集成实现

Skills 与 FunctionCall 的集成实现体现为**三层决策机制**。模型在 Skill 体系中，不再直接生成底层工具的调用参数，而是完成三层决策：理解用户的整体任务目标，而非局部需求；从注册的 Skill 列表中，匹配最适配的一个或多个技能；输出激活指定 Skill 的指令（可以是自然语言、特殊标记或结构化格式）。

Skills 与 MCP 的集成可以通过**网关模式**实现。智能体通过网关查询可用 Skills（GET /skills 或 GET /skills/{skill_id}），向网关发送技能调用请求（含技能 ID、参数、会话 ID），网关转发至对应 Skill 执行器（本地函数 / 插件），将结果回传给智能体，并记录审计日志。

网关维护全局会话，**Skill 与 MCP 调用共享上下文**（如用户 ID、历史参数）。这种设计实现了 Skills 与 MCP 的无缝集成，让系统能够在不同层次上灵活调用各种能力。

在技术架构中，这三者呈现明确的分层关系：



* **感知层**：Agent（智能主体）处于架构核心，覆盖 "感知层 - 决策层 - 记忆层 - 行动层" 全链路，负责接收任务、拆解目标、调用 RAG、Skills、MCP 等，执行反馈闭环

* **决策层**：Skills（知识封装）属于 Agent 决策层的能力模块，将领域知识、操作流程固化为可执行包，解决 "该怎么干" 的问题

* **执行层**：MCP（工具连接）属于 Agent 执行层的核心组件，定义 Agent 与外部工具（数据库、业务系统、网络工具等）的交互协议，解决 "能调用什么" 的问题

## 三、执行流程解析：流程图与时序图的可视化呈现

### 3.1 Skills 整体执行流程

Skills 的整体执行流程可以划分为**四个核心阶段：Discovery（发现）→ Activation（激活）→ Execution（执行）→ Feedback（反馈）**。这个流程体现了 Skills 技术从任务接收到结果输出的完整生命周期。

**第一阶段：技能发现（Discovery）**

在技能发现阶段，系统会在启动时扫描预定的文件夹路径，将所有可用技能的 name 和 description 提取出来。在 Claude/Claude Code 中，这些信息会注入到系统提示中，让模型一开始就 "知道" 有哪些技能选项，但只以很简短的形式呈现，所以对话上下文开销很小。

**第二阶段：技能激活（Activation）**

当用户的请求与某个技能的描述吻合时，模型会决定 "激活" 它。不同平台触发的方式略有不同：Claude 可以自动触发（隐式调用）——Claude 检测到用户请求涉及技能场景，就内部执行相应的读取操作；OpenCode/Codex CLI 则既支持隐式也支持显式调用，显式方式下用户可以在提示中明确要求使用某技能。

**第三阶段：技能执行（Execution）**

一旦决定使用某技能，模型 / 环境会将对应 SKILL.md 的完整内容注入到对话上下文中。实现上通常是让模型调用一个工具去读取文件。例如 Claude Code 里，技能触发实际上体现为 Claude 执行了一条 read file（或 open\_file）命令来打开技能名 / SKILL.md。

**第四阶段：结果反馈（Feedback）**

Skill 内容进入上下文后，模型即可依据其中的指导继续完成用户请求。如果 Skill 包含需要运行的脚本（例如 scripts / 内的代码），模型通常也会尝试执行。例如 Skill 正文可能指示："运行脚本 scripts/do\_X.py 完成数据处理"。Claude Code 或 ChatGPT 会在看到这提示后，在其工具环境中执行该脚本，获取结果再继续。

### 3.2 详细流程图设计

基于上述四个阶段，Skills 的详细执行流程可以用以下流程图表示：



```
开始

│

▼

用户发起任务请求

│

▼

框架注入所有Skill描述、绑定Tool与基础规则

│

▼

大模型完成意图理解

│

▼

匹配并输出目标Skill激活指令

│

▼

框架激活对应Skill

│

▼

│

▼

│   ├─ 模型生成内容/决策

│   └─ 框架调度Tool执行并回灌结果

│

▼

│

▼

```

这个流程的核心特征是**模型与框架的交替执行**，模型负责理解、选择、生成、总结，不执行任何外部 IO、计算与接口调用；真正的业务执行由底层 Tool 完成，流程调度由 Agent 框架负责，三者分工明确、边界清晰。

### 3.3 时序图精确描述

Skills 技术的时序图描述了多个对象之间的动态协作，包括发送消息、接收消息、处理消息、返回消息等，重在展示对象之间的交互顺序，尤其强调交互的时间顺序。



```
时序图：Skills技术执行流程

参与者：用户、大模型、框架、Skill、Tool

用户    ->> 大模型: 发起任务请求

大模型  ->> 框架:   意图理解，请求可用Skill列表

框架    ->> 大模型: 返回Skill元数据（name, description）

大模型  ->> 框架:   选择激活特定Skill

框架    ->> Skill:   加载SKILL.md内容

Skill   ->> 框架:   返回完整指令和资源

框架    ->> 大模型: 注入Skill上下文

大模型  ->> 框架:   根据subtasks生成执行计划

框架    ->> Tool:    调用底层工具执行子任务

Tool    ->> 框架:   返回工具执行结果

框架    ->> 大模型: 回灌工具执行结果

大模型  ->> 框架:   分析结果，决定下一步

...     ...         ... （循环执行直到任务完成）

大模型  ->> 用户:   返回最终结果
```

这个时序图展示了**真实任务中数据流的双向交互特性**，而非单向传递。标准时序如下：Agent 接收目标→首次调用 LLM 做意图理解与任务拆解；Agent 根据拆解结果，调度对应的 Agent Skills；获取外部信息与执行结果。

### 3.4 关键时间节点分析

根据实际测试数据，Skills 技术的执行时间主要包括以下几个关键节点：



1. **技能发现阶段**：通常在系统启动时完成，耗时极短（毫秒级）

2. **意图匹配阶段**：模型进行意图分类和技能匹配，通常需要 1-3 秒

3. **技能加载阶段**：读取 SKILL.md 文件并解析，通常需要 0.5-1 秒

4. **指令执行阶段**：根据 subtasks 执行具体操作，这是主要耗时阶段

5. **结果汇总阶段**：整理和格式化最终输出，通常需要 0.5-1 秒

整体而言，简单任务的执行时间通常在**2-5 秒内**完成，形成 "思考 - 执行 - 记忆" 的闭环。复杂任务的执行时间则取决于子任务的数量和复杂度。

### 3.5 资源消耗优化机制

Skills 技术在资源消耗方面具有显著优势。根据 GitHub 测试数据，复杂任务的 token 消耗可降低**73%**（从 8500 降至 2300 token / 次），响应速度提升**2.4 倍**（从 12 秒降至 5 秒 / 次），准确率提升**18%**。

这种优化主要通过以下机制实现：



1. **渐进式加载**：只加载当前任务需要的技能内容，避免一次性加载所有技能

2. **上下文回收**：技能加载的知识只在当前任务期间使用，任务完成后被回收，不会永久占用上下文

3. **智能路由**：通过提升路由粒度，降低模型的决策复杂度，减少无效的思考步骤

4. **缓存机制**：对常用技能和工具结果进行缓存，避免重复计算

## 五、技术优势与发展前景

### 5.1 核心技术优势

Skills 技术相比传统单工具调用具有五大核心优势：



1. **抽象层级提升**：从工具调用升级为任务级意图匹配，通过提升路由粒度，降低模型的决策复杂度，同时让任务执行更贴合业务逻辑

2. **资源效率优化**：token 消耗降低 73%，响应速度提升 2.4 倍，准确率提升 18%。对于频繁使用的场景，成本可降低**40-60%**

3. **开发效率提升**：开发者无需重复编写相同的 Function Call 调用逻辑，只需复用已封装的 Skill；同时，Skills 让 Agent 的决策更简单，Agent 只需关注 "用哪个 Skill 解决问题"，而非 "用哪个 Function Call 完成动作"

4. **标准化程度高**：采用统一的技能规范，支持跨平台使用。技能配置（一次性）约 1000 tokens，token 节省约**90%**

5. **可维护性强**：每个 Skill 都是独立的 Markdown 文件，要加新功能只需要写一本新的专家手册，不需要改大模型的代码或者重新训练，也不会影响其他功能。所有的决策和处理都在大模型内部完成，不会出现调度逻辑和模型理解脱节的问题

### 5.2 与传统方案的对比分析

Skills 技术与传统方案的对比如下：

### 5.3 技术局限性与挑战

尽管 Skills 技术具有诸多优势，但仍存在一些局限性：



1. **技能定义质量依赖**：如果 Skill 的手册写得不好，大模型的处理效果也会差。这要求技能开发者具备较高的专业水平和经验

2. **标准化程度有待提升**：目前 Skills 技术仍在快速发展中，不同平台的实现方式存在差异，需要进一步的标准化

3. **调试复杂度增加**：虽然开发效率提升，但当出现问题时，调试和定位问题的复杂度相对较高，需要对整个技能体系有深入理解

4. **资源管理挑战**：随着技能数量的增加，如何高效管理和组织大量技能成为一个挑战，需要建立完善的技能库管理系统

### 5.4 发展趋势与未来展望

Skills 技术的发展趋势主要体现在以下几个方面：



1. **生态建设**：随着 Anthropic 将技能规范开放，以及社区贡献大量技能范例，各平台有望出现技能市场或目录，方便共享和发现技能。未来可能出现类似 App Store 的平台，用户可以浏览各类技能，一键安装到自己的 AI 代理中

2. **智能化发现**：模型自动发现技能的能力可能出现 —— 例如模型遇到未掌握的新任务时，能联网检索相应的技能包并请求用户确认后加载。这类似于人不会的就去搜教程的思路

3. **多模态融合**：今天的 Skills 主要围绕文本和结构化数据处理，但随着多模态大模型的发展，多模态技能可能会出现。例如，一个图像处理技能，里面包含如何调用图像识别模型或绘图工具的说明

4. **自主进化能力**：Anthropic 展望未来，希望让 "Agents to create, edit, and evaluate Skills on their own"。也就是说，让 AI 代理在执行任务中学会把自己的新知识提炼成技能，下次遇到类似问题就能自己调用

5. **安全与权限管理**：随着技能赋予模型更多自主权，安全问题也不可忽视。未来的发展一方面会加强技能沙盒环境的安全隔离，另一方面会引入更细粒度的技能权限
